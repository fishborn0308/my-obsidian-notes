
## コースの目的と方法論

本コースは、Webアプリケーションのペネトレーションテスト（ペントesting）の基礎を解説するものです。一部の初期コンテンツは、経験豊富なテスター向けの復習として構成されています。上級者向けには、「Over the Wire - Natas War Games」（VMからのアクセスが推奨されます）を通じて、Webアプリケーションのペントestingに関する追加の実践演習を利用できます。本コースは、2つの完全なWebアプリケーションペントesting（1つは指導付き、もう1つはCTF形式）の実施を中心に構成されています。体系的なアプローチが重視されており、以下が含まれます：

*  **脅威モデリング：** 想定される攻撃者の攻撃（例：SQLインジェクション）に基づいてテストの優先順位を決定します。

*  **静的アプリケーションセキュリティテスト（SAST）：** コードレビューを行い、自動化ツールと手動調査を組み合わせて実施します。

*  **動的アプリケーションセキュリティテスト（DAST）：** 本コースの核心となる部分。ツールを用いて広範囲をカバーしつつ、論理的欠陥、不安全なデシリアライゼーション、XXE、DOMベースのXSSについては手動による分析を重視します。

*  **事前情報に基づく評価：** 「ゼロ知識」状態で作業するのではなく、アーキテクチャ図、ソースコード、APIの詳細を事前に提供してもらうことで、効率を最大化することを推奨します。

  

  

OWASP Web Security Testing Guide (WSTG) を主要な方法論とし、一貫性があり、再現可能で、監査可能なフレームワークを提供します。受講生には、そのチェックリストを使用し、必要に応じて詳細な観察事項を記録するための「備考」欄を追加することを推奨します。

  

**実践事項：**

* （オプション、経験豊富なユーザー向け）追加のWebアプリケーション侵入テスト練習として、**「Over the Wire - Natas War Games」**を調査してください。

*  **OWASP Web Security Testing Guide (WSTG) のチェックリストを活用**し、ペネトレーションテストの活動を整理・追跡してください。

  

## Webアプリケーションペネトレーションテスト用ツールキット

  

Webアプリケーションのペネトレーションテストに不可欠なツールキットには、攻撃プラットフォーム（オペレーティングシステム）、動的Webアプリケーションスキャナー、ブラウザ、およびインターセプトプロキシが含まれます。

*  **オペレーティングシステム（OS）：** どのOS（Windows、Mac、Linux）でも適しています。重要なのは使い慣れていることです。Linuxは仮想マシン内で効果的に使用できます。

*  **動的スキャナー：** 豊富な機能とコストパフォーマンスの高さから、Burp Suite Professionalを強く推奨します。ZAP（Zed Attack Proxy）は、無料のオープンソースであり、高性能な代替ツールです。

*  **ブラウザ：** Webアプリケーションとのやり取りや、クライアントサイドの脆弱性の検証に必要です。

*  **インターセプトプロキシ：** ブラウザとWebアプリケーションの間に配置され、リクエスト/レスポンスの傍受や変更を可能にします。Webサーバーになりすまし、証明書を生成することでHTTPSを処理します（これには、そのルート証明書をブラウザにインポートする必要があります）。

  

**ZAPとBurp Suite Professionalの主な機能：**

*  **ZAP (Zed Attack Proxy)：** 階層的な「サイト」ビュー、リクエスト/レスポンスペイン、「履歴」タブ、各種攻撃（スキャン、ファジング、スパイダー）用のコンテキストメニュー、およびアドオンによる拡張性を備えています。また、エンコード/デコードツールや手動リクエストエディタも含まれています。

*  **Burp Suite Professional:** スキャンや問題点を表示する「ダッシュボード」、評価範囲を定義する重要な「ターゲット（スコープ）」タブ（不正なスキャンを防止）、リソースを階層的に表示する「サイトマップ」、および「Intruder」（ファジング）や「Repeater」（単発リクエスト）などのツールへリクエストを送信するための包括的なコンテキストメニューを備えています。さらに、フィルタリング機能、リクエストを一時停止するための「Intercept」タブ、「HTTP History」、「Sequencer」（ランダム性テスト用）、「Decoder/Encoder」、「Comparer」（微妙な差異の検出用）、およびプラグイン（例：Active Scan++、IPRotate）用の「Extender」も含まれています。

  

**実践事項：**

*  **ペネトレーションテストツールでUser-Agentヘッダーを変更する**。これにより、Webアプリケーションファイアウォール（WAF）によるブロックを回避し、競合を明確に解消できます。

*  **ZAPとBurp Suite Professionalの両方のインターフェースと機能に慣れる**。特にコンテキストメニューと利用可能なアドオンに細心の注意を払ってください。

*  **Burpのスコープを効果的に設定・使用する方法を学ぶ**。これにより、スキャンが許可された範囲内に留まるようにします。

*  **Burpのフィルターをリセットまたはクリアすることを忘れないでください**。これにより、関連するリクエストが意図せず非表示になるのを防げます。

  

## HTTPプロトコルの概要

  

HTTPプロトコルは、1989年の誕生以来、大幅に進化してきました。

*  **HTTP 1.1:** バーチャルホスティング（1台のサーバー上で複数のWebアプリケーションを実行可能）を導入し、HTTPメソッドを拡張しました。

*  **HTTP 2.0 (2015):** ヘッダー圧縮と単一のTCP接続上での多重化を特徴とするバイナリプロトコル。

*  **HTTP 3.0 (QUICベース):** 転送にUDPを利用するバイナリプロトコル。採用は急速に進んでいるが、ツールのサポートはまだ広まっていない。

  

**HTTPリクエストとレスポンスの構造:**

*  **リクエスト:** メソッド、URI、プロトコルバージョンで始まり、ヘッダー（例：User-Agent、Referer、Cookie）が続く。

各行はキャリッジリターン・ラインフィード（CRLF）で終了し、ヘッダーと本文の間は空白行で区切られる。

*  **レスポンス：** プロトコルバージョン、ステータスコード、ステータスメッセージで始まり、その後にヘッダー（例：Set-Cookie）が続く。

*  **HTTPメソッド：** 「安全」（読み取り専用）または「非安全」（サーバーの状態を変更する）、「冪等」（リクエストを繰り返してもシステムの状態は変わらない）または「非冪等」（リクエストを繰り返すとシステムの状態が変わる。適切なトランザクションIDのないPOSTなど）に分類されます。

*  **GET:** パラメータはURIに含まれるため、ログ記録や情報漏洩のリスクがあります。**機密情報は決してGET経由で送信してはなりません。**

*  **POST:** パラメータはリクエストボディに含まれるため、機密データの保護に優れています。

*  **HEAD:** レスポンスヘッダーのみを取得し、効率的な情報収集に役立ちます。

*  **TRACE:** サーバーが認識した通りの正確なリクエストを返します。プロキシによる改変の検出には役立ちますが、クロスサイト追跡の実用的な悪用には適しません。 *  **OPTIONS:** サポートされているHTTPメソッドを一覧表示します。これを無効にすることで、隠蔽性による最低限のセキュリティが得られます。

  

**重要なヘッダーとセキュリティ:**

*  **User-Agent:** WAFによる検出を回避したり、競合を解消したりするために、ペネトレーションテストツールでは変更する必要があります。

*  **Referer:** アプリケーションが基本的なパス強制を行うために使用されることがあります。

*  **Cookies:** セッション識別子やその他のデータを保存します。必須のフラグには、Secure（HTTPSのみでの送信）、HTTPOnly（XSSを軽減するためにJavaScriptによるアクセスを防止）、SameSite（CSRFを軽減するためにクロスサイトでのCookie送信を防止）が含まれます。

*  **セキュリティ応答ヘッダー：**

*  **X-Frame-Options：** クリックジャッキング対策として、iframeへのページの埋め込みを防止します（オプション：DENY、SAMEORIGIN）。 *  **HTTP Strict Transport Security (HSTS):** 初回訪問後はHTTPSのみでのアクセスを強制します。推奨されるmax-ageは少なくとも1年です。

*  **Content Security Policy (CSP):** コンテンツの信頼できるソースを指定します。スクリプトの実行元を制限することでXSSを防止するために不可欠です。

  

**実施すべき事項:**

*  **GETリクエストのパラメータで機密情報を送信しないようにしてください。**

*  **機密性の高いクッキーには適切なクッキーフラグ**（Secure、HTTPOnly、SameSite）が設定されていることを確認する。

*  **クリックジャッキングを防ぐため、X-Frame-Optionsが適切に設定されているか**（DENYまたはSAMEORIGIN）を確認する。

*  **HTTPSを強制するため、max-ageを少なくとも1年に設定したHSTSを推奨する**。

*  **コンテンツセキュリティポリシー（CSP）の設定を入念に確認**し、信頼できないコンテンツ、特に悪意のあるスクリプトが読み込まれるのを防ぐ。

  

## 情報収集（ターゲットのプロファイリングとHTTPS設定）

  

このフェーズでは、Webアプリケーションのアーキテクチャ、コンポーネント、およびサービス設定の把握に焦点を当てます。

*  **ターゲットのプロファイリング：**

*  **Shodan：** インターネットに接続されたデバイスを検索するエンジンで、証明書データの収集やリダイレクトの特定に役立ちます。 *  **NMAP:** ポートスキャンを行い、公開されているサービスを特定するために使用します。サービスを正確に特定するため（例：非標準ポート上のWebサーバー）、-sV（サービスバージョン検出）または-A（アグレッシブスキャン）フラグを使用し、NSEスクリプト（例：http-robots.txt）を有効にすることが不可欠です。 *  **GoWitness:** 開いているポート上で検出されたWebアプリケーションのスクリーンショットを撮影し、侵入経路の概要を視覚的に把握できるツール。

*  **手動による照会:** クライアントに直接、アーキテクチャ図、Webサーバーの詳細、アプリケーションフレームワーク、CMS、データベース情報を尋ねることは非常に効率的です。

*  **HTTPS設定の評価:** 「暗号化の失敗」（OWASP Top 10、#2）で頻繁に問題となるため、極めて重要な領域です。

*  **プロトコル：** SSL v2/v3 および TLS v1.0/1.1 は無効化すべきです。TLS v1.2 は一般的に許容されますが、機密性の高いデータやトランザクションには TLS v1.3 が推奨されます。

*  **暗号スイート：** 暗号スイートの強度を評価します。自動ツールの評価を盲信せず、保護対象データの機密性に基づいて推奨事項を文脈に合わせて判断してください。例えば、CBCモードのAES 128ビットとSHA-1ハッシュの組み合わせは、しばしば「A」評価を受けるものの、パディングオラクル攻撃やSHA-1の衝突攻撃に対して脆弱である可能性があり、機密性の高いデータには不適切です。

*  **ツール：**

*  **NMAPのssl-enum-ciphers NSEスクリプト：** 利用可能なプロトコルと暗号スイートを列挙し、基本的な評価を提供します。 *  **testssl.sh:** プロトコル、暗号スイート、既知の脆弱性、ユーザーエージェントネゴシエーションなどを網羅した包括的なHTTPS設定分析に推奨されるBashスクリプト。分かりやすさを重視し、出力は色分けされています。

*  **Qualys SSL Labs:** 公開Webサイト向けのオンラインサービス。管理に役立つ評価（文字評価）と詳細レポートを提供します。

  

**実行すべき項目：**

*  **NMAPを-sV（サービスバージョン検出）または-A（アグレッシブスキャン）オプション付きで使用し**、ターゲットポート上で実行されているサービスとバージョンを正確に特定する。

*  **GoWitnessの利用を検討し**、さまざまなポート経由でアクセス可能なWebアプリケーションの概要を視覚的に把握する。

*  **常にクライアントからアプリケーションインフラの詳細**（Webサーバー、フレームワーク、CMS、データベース、サポートされているHTTPメソッド）を要求・活用し、評価の効率を高める。

*  **Webサーバー上でSSL v2/v3およびTLS v1.0/1.1が無効化されていることを確認する**。

*  **機密性の高いアプリケーションについては、TLS v1.3のみを有効にするよう提言する。**

*  **HTTPS設定に関する推奨事項を提示する際は、アプリケーションのデータ機密性や機能という文脈**の中でツールの出力を解釈する。

*  **testssl.sh**（または NMAP の ssl-enum-ciphers）を利用して、HTTPS 設定の徹底的な評価を行う。

*  **Docker コンテナを使用するラボ（例：Restaurant システムや Heartbleed システムに対する testssl.sh）では、ヒントセクションに記載されている特定の Docker スイッチ**を使用して、コンテナ間の通信と適切な DNS 解決を有効にする。

*  **実習で restaurant update または restaurant restart スクリプトを実行した後、アプリケーションとのやり取りを試みる前に 15～30 秒間待機し**、サービスが完全にオンラインになっていることを確認してください。
