### 不適切なシリアライゼーション解除

  

OWASP Top 10（2017年）以降、広く認知されている脆弱性である「不適切なシリアライゼーション」が紹介されました。この脆弱性は、アプリケーションがメモリ内でオブジェクトを再構築する前にシリアライズされたデータの検証を行わない場合に発生します。これにより、データの改ざんやリモートコード実行（RCE）につながる可能性があります。exec関数を持つライブラリを参照し、コマンドを指定することでJavaベースのRCEエクスプロイトを作成する方法として、YSO Serialツールが紹介されました。

  

**実行項目/タスク:**

  

*  **実習:** コンテナ化されたアプリケーションから secret.txt を読み取るために、不適切なシリアライズ解除の脆弱性を悪用する手順が示されました。

*  **起動:** Burp Suite と実習用の deserialization.sh Docker コンテナを起動します。

*  **ログイン:** 「Arthur's VHS Museum」にアクセスし、「Student」 / 「Security 542」でログインします。

*  **ペイロードの作成:** YSO Serial を使用して、/secret.txt を /opt/app-files/dummy.html にコピーする Java ペイロードを生成します。例: java -jar ysoserial.jar CommonsCollections5 『cp /secret.txt /opt/app-files/dummy.html』.

*  **ペイロードのエンコード:** 生成されたバイナリペイロードをBase64エンコードします。例: cat dsir.obj | base64.

*  **リクエストの送信:** Burpで/makeOrderへのリクエストをインターセプトし、Repeaterに送信します。メソッドをPOSTに変更し、Base64エンコードされたペイロードをobj=として追加し、**ペイロードの値のみ（obj=の部分は除く）をURLエンコード**します。

*  **シークレットの取得:** リクエスト送信後、VHSテープのページ（例：『市民ケーン』）に移動し、filenameパラメータをdummyに変更してsecret.txtの内容を取得します。

*  **クリーンアップ:** cleanup.shを実行してDockerコンテナを停止します。

  

### SQLインジェクション

  

SQLインジェクションは、持続的かつ一般的な脆弱性として議論されました。これは、アプリケーションが信頼できないユーザー入力をSQLクエリに直接組み込むことで発生し、攻撃者がデータベースステートメントを操作できるようにします。初期の発見には、データベースのエラーメッセージ（例：不均衡なシングルクォートによるもの）や、アプリケーションの動作変化の観察がしばしば含まれます。認証のバイパスやテーブルからの全レコードの抽出を行うための、' OR 1=1 -- のような同語反復の使用について説明されました。取り上げられた高度な手法には、コメントのブラックリストの回避（例：『 OR 』A『=』A' の使用）、UNION文を用いた他テーブルへのクエリ（列数の一致と互換性のあるデータ型が必要）、およびNULL値と組み合わせたFROMLESS SELECTによる列構造の特定などが含まれました。

  

**ブラインドSQLインジェクション**の手法についても解説されました：

*  **真偽ベース：** 真偽の異なるSQL条件に対するアプリケーションの応答の違いを観察する。

*  **時間ベース：** データベースの応答に遅延を誘発し、真偽を推測する。

*  **アウトオブバンド：** ネットワークリクエスト（DNS、HTTP）やデータベース固有の関数（例：MS SQLのXP_COMMANDSHELL）を介してデータを流出させる。

SQLインジェクションの影響は、データ窃取にとどまらず、ファイルシステムへのアクセス、OSコマンドの実行、またはWebシェルの展開にまで及ぶ可能性があることが指摘されました。SQLインジェクションのチートシート用リソースが提供された。

  

**実行すべき項目／タスク：**

  

*  **実習：** ユーザーはエラーベースのSQLインジェクション実習を行った。

*  **ペイロードの実験：** 様々なSQLインジェクションペイロードを試す。

*  **カラム検出：** ORDER BYまたはUNION SELECT NULLペイロードを使用して、元のクエリのカラム数を特定する。

*  **データベースクエリの検査：** information_schema.processlist からデータを取得し、アプリケーションの元のクエリや注入されたペイロードを含め、データベースによって処理された完全な SQL ステートメントを確認する。

  

### SQLMap

  

SQLインジェクションの発見と悪用を自動化する強力な Python ツールである SQLMap が紹介された。このツールは、様々なデータベースタイプやブラインドおよびノンブラインドのインジェクション手法をサポートしており、他のツールとの統合も可能である。主要なコマンドラインスイッチについて解説した。これには、保存された結果を消去する --purge、ファイルからHTTPリクエストを読み込む -r（推奨）、インターセプトプロキシを使用する --proxy、およびユーザーエージェントを偽装する --user-agent などが含まれる。SQLMapをBurpやZAPと統合することで、SQLMapのトラフィックを監視し、動的なセッション/CSRFトークンを管理することが可能になる。

  

**実行項目／タスク：**

  

*  **実習：** ZAPをインターセプトプロキシとして使用し、SQLMapを用いたSQLインジェクションの悪用を実践しました。

*  **手動による悪用：** 注文照会アプリケーションにおいて、SQLインジェクションを手動で発見し、悪用しました。

*  **クライアントサイド保護の回避：** ブラウザのJavaScriptを無効化し、クライアントサイドの入力検証を回避しました。

*  **自動検出：** ZAPのアクティブスキャナーを使用して、SQLインジェクションの脆弱性を確認します。

*  **SQLMapのUser-Agent偽装：** デフォルトのUser-AgentがブロックされたためにSQLMapが最初に失敗する様子を観察します。その後、**偽装されたUser-Agentを使用するようにSQLMapを設定し**、アプリケーションのフィルタを回避して脆弱性を正常に検出します。

*  **データの持ち出し：** SQLMapを使用してデータベースからユーザーのハッシュをダンプします。

*  **ハッシュの解読：** John the Ripperを使用して、ダンプされたハッシュを解読します。

  

### クロスサイトスクリプティング (XSS)

  

ドキュメントオブジェクトモデル (DOM) と JavaScript の基礎を復習し、JavaScript が DOM と相互作用し、操作できる能力に重点を置きました。かつては主要な脆弱性の一つであったXSSは、現在では「インジェクション」に分類されています。XSSは、アプリケーションがユーザー制御の入力をDOMに安全でない方法で組み込むことで発生し、攻撃者がHTMLやJavaScriptを注入することを可能にします。XSSに対する主な防御策として、入力のサニタイズではなく、**出力エンコーディング（<、>、"などの危険な記号をHTMLエンコードすること）**の重要性が強調されました。

  

**同一オリジンポリシー（SOP）**は、ブラウザの重要なセキュリティ制御として説明されました。これは、プロトコル、ホスト名、ポートが同一でない限り、あるDOMインスタンス内のJavaScriptが別のインスタンスと相互作用することを防止します。XSSは、悪意のあるスクリプトを被害者のDOMに_注入_することでこれを悪用し、同一オリジンの一部として扱わせます。

  

**XSSの影響の実演：** 単純なalert()ボックスの表示にとどまらず、影響力の大きい実演例としては、認証情報の入力を促すもの、セッションクッキーやJOT（HttpOnlyが設定されていない場合）の窃取、**BEEF（Browser Exploitation Framework）**のようなエクスプロイトフレームワークの使用などが挙げられます。BEEFは、コマンド＆コントロールサーバー（攻撃者側）と、被害者のブラウザに注入されるクライアントサイドの「フック」（JavaScriptのhook.js）を提供し、攻撃者がブラウザを制御できるようにします。

ステルス的なデータ流出手法（例：document.cookieを付加して攻撃者のサーバーを指す 

*  **実行：** コードの実行を実現する（例：<script>alert(42)</script>）。

*  **保存型XSS：** ログに記録されたパラメータにペイロードを注入し、そのログを表示することで、保存型XSSの脆弱性を悪用する。

*  **Cookie窃取：** <script>alert(document.cookie)</script> を使用してセッションCookieを窃取する実演を行う。

*  **BEEF ラボ：**

*  **セットアップ：** BEEF を起動し、Firefox（攻撃者）でコンソールにアクセスし、Chromium（被害者）で「cute bunny」ページを開いて BEEF フックを取得します。

*  **制御：** BEEF コンソールから、**さまざまな BEEF モジュールをスケジュールして実行**し、Chromium（被害者）ブラウザでの効果を観察します。このラボは、XSSインジェクションではなく、BEEFの仕組みに焦点を当てています。

  

### XSSの分類：リフレクテッド、ストアド、DOMベース

  

XSSの脆弱性は、以下の3つのタイプに分類されます：

*  **リフレクテッド（非永続型）：** ペイロードはリクエストに含まれており、サーバーのレスポンスに即座に反映され、ブラウザによって実行されます。保存されることはありません。

*  **保存型（Persistent）：** ペイロードはサーバー（例：データベース）に保存され、複数のユーザーによって取得され、レンダリング時に実行されます。悪用リクエスト内に直接ペイロードは含まれません。「アウトオブバンド」というサブカテゴリが存在し、ここではペイロードが非HTTPプロトコル（例：電子メール）を介してデータストアに送信されます。

*  **DOMベース：** 脆弱性は_クライアントサイド_のJavaScriptに存在します。JavaScriptが値（例：window.location.hash）を読み取り、適切なエンコードを行わずにDOMに導入することで、コードの実行につながります。重要な違いは、JavaScriptインタプリタが_すでに実行中_であり、ペイロードが既存のスレッドを制御下に置く点です。ペイロードがサーバーに到達しない可能性があるため、このタイプはサーバーサイドの防御によって検出するのが難しい場合があります。

*  **リフレクテッド/ストアド型 DOMベースXSS:** これらのサブカテゴリは、サーバーサイドのコードが脆弱なクライアントサイドJavaScriptを_書き込み_、それがDOMベースXSSを悪用する際に発生します。

  

**実行すべき項目/タスク:**

*  **発見:** パスワード生成ツール内の脆弱性を特定します。

*  **コード実行:** JavaScriptの実行を実現します。

*  **Cookieの窃取:** セッションCookieの窃取を実演します。

*  **BEEFフック:** BEEFフックを注入し、被害者のブラウザを制御下に置きます。
