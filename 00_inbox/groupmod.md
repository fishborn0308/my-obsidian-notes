# `groupmod` - 既存のユーザーグループを変更する

## 概要

`groupmod` (group modify) は、システムに既に存在するユーザーグループの**名前**や**GID (グループID)** を変更するためのコマンドです。グループの運用方針が変わった際に、グループ自体を作り直すことなく、その属性を修正するために使用します。

このコマンドの実行には、通常 `root` 権限が必要です。

(出自: `shadow-utils` または `passwd` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`groupadd`** | 新しいグループを**作成**します。 | - |
| **`groupdel`** | 既存のグループを**削除**します。 | - |
| **`usermod`** | **ユーザー**の属性（所属グループなど）を変更します。`groupmod` は**グループ**の属性を変更します。 | - |

## よく連携されるコマンド

`groupmod` でGIDを変更した後は、古いGIDを持つファイルの所有権を修正するために `find` や `chgrp` と連携させる必要があります。

### シナリオ例: GID変更後にファイルの所有権を修正する (インフラ構築・運用視点)

* **目的**: グループのGIDを変更した後、そのグループが所有していたファイルのグループ所有権も新しいGIDに追従させる。
* **連携コマンド**: `find`, `chgrp`
* **解説**: `groupmod -g` でGIDを変更すると、ファイルシステム上に古いGIDを持つファイルが残り、「孤立した」状態になります。`find` でこれらのファイルを検索し、`chgrp` でグループ所有者を（名前に基づいて）再設定することで、整合性を保ちます。
* **コマンド例**:

    ```bash
    # 'webdevs' グループのGIDを 2001 に変更 (旧GIDは1005だったと仮定)
    sudo groupmod -g 2001 webdevs

    # GID変更後、古いGID (1005) を持つファイルを検索して所有者を修正
    sudo find / -gid 1005 -exec chgrp -h webdevs {} +
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **属性変更** | |
| ⭐ `-n`, `--new-name <NEW_NAME>`| グループの**新しい名前**を指定します。 |
| ⭐ `-g`, `--gid <NEW_GID>` | グループの**新しいGID**を指定します。 |
| **その他** | |
| `-o`, `--non-unique` | `-g` と組み合わせて使用し、GIDの重複を許可します。通常は非推奨です。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: プロジェクト名の変更に伴い、関連するグループ名を変更する。
* **組み合わせ**: `sudo groupmod -n <new_group_name> <old_group_name>`
* **解説**: グループ名を実態に合わせて修正する、最も基本的な使い方です。
* **例**:

    ```bash
    # 'alpha-devs' グループの名前を 'ares-devs' に変更
    sudo groupmod -n ares-devs alpha-devs
    ```

### 2. ブルーチーム視点

* **タスク**: `auditd` やシステムのログを監視し、`groupmod` の不正な使用を検知する。
* **組み合わせ**: (監視ツール側での設定)
* **解説**: 攻撃者が権限の隠蔽や混乱を試みるために、正規のシステムグループのGIDを変更する可能性があります。`groupmod` の実行ログは、不審な権限操作の兆候となり得るため、監査対象とします。
* **例**:

    ```bash
    # ausearchでgroupmodの実行履歴がないか確認
    sudo ausearch -c groupmod
    ```

### 3. レッドチーム視点

* **タスク**: **永続化と隠蔽**のため、既存の一般グループのGIDを `0` (`root` グループのGID) に変更する。
* **組み合わせ**: `sudo groupmod -g 0 <some_group>`
* **解説**: 攻撃者がroot権限を奪取した後、`docker` や `adm` といった、管理者ユーザーが所属していても不自然ではないグループのGIDを `0` に変更します。これにより、そのグループは `root` グループと全く同じ権限を持つことになり、`root` という名前を使わずに特権を維持・利用できます。
* **例**:

    ```bash
    # 'adm' グループのGIDを0に変更して特権を隠蔽
    sudo groupmod -g 0 adm
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `groupmod: GID '<gid>' already exists`
    * **考えられる原因**: `-g` オプションで指定したGIDが、既に別のグループによって使用されています。
    * **解決策**: 別のGIDを指定するか、`-o` オプションを付けて重複を許可します（非推奨）。

## 環境変数と設定ファイル

* `groupmod` の動作に直接影響する環境変数は通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

### 関連する設定ファイル

* **`/etc/group`**, **`/etc/gshadow`**: `groupmod` はこれらのファイルを更新します。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `groupmod` 自体ではなく、**GID変更のリスク**や**特権GIDへの変更**がセキュリティリスクとなります。
* **悪用シナリオ**: レッドチームのシナリオで述べたように、一般グループのGIDを `0` (`root`) などの特権GIDに変更することは、深刻なセキュリティホールを作り出す行為です。

### GTFOBins / LOLBAS における利用例

`groupmod` はOSの構成を変更するコマンドであり、直接的なコード実行や権限昇格の機能を持たないため、GTFOBinsにはリストされていません。しかし、**永続化 (Persistence)** や**防御回避 (Defense Evasion)** の文脈で、攻撃環境を整えるために悪用される可能性があります。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 最小権限の原則に従い、`groupmod` の `sudo` 権限を一般ユーザーに与えない。
* **Detection (検知)**: `/etc/group` ファイルへの変更をファイル整合性監視(FIM)ツールで監視する。`auditd` で `groupmod` コマンドの実行を監視する。

## 注意点・補足

* **GID変更のリスク**: グループのGIDを変更すると、ファイルシステム上に古いGIDを持つファイルが残り、「孤立した」状態になります。これらのファイルの所有権を適切に再設定しないと、意図しないアクセス権の問題を引き起こす可能性があります。
