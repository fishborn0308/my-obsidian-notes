# Linuxにおける主要ログファイルと活用術

## 概要

Linuxシステムが生成するログは、システムの動作状況、アプリケーションのエラー、セキュリティイベントなどを記録しており、システムの監視、トラブルシューティング、そしてサイバーセキュリティにおける脅威検知と対応の基礎となります。ログは主に `/var/log` ディレクトリ配下にテキスト形式で保存されるか、`systemd-journald` によってバイナリ形式で一元管理されます。

(出自: `rsyslog`, `systemd` などのシステムコンポーネント)

## 類似コマンド/概念との差異

| 概念 | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`systemd-journald`** | モダンなLinuxの標準。ログを構造化されたバイナリ形式で一元管理し、`journalctl` で高速に検索・フィルタリングできます。 | `journalctl` |
| **`rsyslog`** | 旧来からの標準。テキストベースのログを `/var/log` 配下にファイルとして出力します。多くのシステムで `journald` と共存しています。 | - |
| **`auditd`** | セキュリティ監査に特化。システムコールレベルで「誰が、何を」したかを詳細に記録します。一般的なログより低レベルで詳細です。 | `auditd` (セキュリティ監査) |

## よく連携されるコマンド

ログファイルは非常に大きくなるため、`grep`, `tail`, `less`, `awk` などのテキスト処理コマンドと連携して分析するのが基本です。

### シナリオ例: SSHのブルートフォース攻撃を調査する (ブルーチーム視点)

* **目的**: 認証ログから、特定のIPアドレスからのログイン失敗の試行をリアルタイムで監視する。
* **連携コマンド**: `tail`, `grep`
* **解説**: `tail -f` で認証ログ (`/var/log/auth.log`) の追記をリアルタイムに監視し、その出力をパイプで `grep` に渡してキーワードでフィルタリングします。
* **コマンド例**:

```bash
# 認証ログをリアルタイム監視し、"Failed password" を含む行のみ表示
sudo tail -f /var/log/auth.log | grep "Failed password"
```

## 📜 主要なログファイルと場所

| ログファイル名 | 説明 |
| :--- | :--- |
| **システム/セキュリティログ** | |
| ⭐ `/var/log/auth.log` <br> `/var/log/secure` | (Debian系) / (RHEL系) **認証ログ**。SSHログイン、`sudo` の使用、ユーザー作成など、セキュリティ関連のイベント。 |
| ⭐ `/var/log/syslog` <br> `/var/log/messages`| (Debian系) / (RHEL系) **システムログ**。カーネル、デーモン、各種サービスからの一般的なメッセージ。 |
| `/var/log/dmesg` | 起動時のカーネルリングバッファのメッセージ。 |
| `/var/log/faillog` | ログイン失敗の記録。 |
| `/var/log/lastlog` | 各ユーザーの最終ログイン記録。 |
| **アプリケーションログ** | |
| ⭐ `/var/log/nginx/access.log` <br> `/var/log/nginx/error.log` | Nginxのアクセスログとエラーログ。 |
| `/var/log/httpd/access_log` <br> `/var/log/httpd/error_log` | Apacheのアクセスログとエラーログ。 |
| `/var/log/mysql/error.log` | MySQL/MariaDBのエラーログ。 |
| **その他** | |
| `/var/log/dpkg.log` <br> `/var/log/yum.log` | (Debian系) / (RHEL系) パッケージ管理の操作ログ。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: サービスが起動しない原因を調査する。
* **組み合わせ**: `journalctl -u <service> -e`
* **解説**: `-u` で特定のサービスに絞り込み、`-e` でログの末尾（最新のエラー）にジャンプします。これにより、起動失敗の原因となったエラーメッセージを素早く確認できます。
* **例**:

```bash
# Nginxサービスの起動失敗ログを確認
sudo journalctl -u nginx.service -e
```

### 2. ブルーチーム視点

* **タスク**: 不正アクセスの痕跡を調査する。
* **組み合わせ**: `grep "Failed password" /var/log/auth.log`
* **解説**: **インシデントレスポンスの基本**。認証ログからブルートフォース攻撃の兆候を探します。ここから攻撃元IPや標的となったユーザー名を特定し、`last` や `lastb` での調査に繋げます。
* **例**:

```bash
# ログイン失敗ログから攻撃元IPを抽出し、回数順にランキング
sudo grep "Failed password" /var/log/auth.log | awk '{print $(NF-3)}' | sort | uniq -c | sort -nr | head
```

### 3. レッドチーム視点

* **タスク**: **痕跡消去 (Anti-Forensics)**。自身の活動ログを削除・改ざんする。
* **組み合わせ**: `rm`, `sed`
* **解説**: 攻撃者は侵入後、自身のIPアドレスや実行したコマンドが記録されたログエントリーを削除しようと試みます。
* **例**:

```bash
# 自身のコマンド履歴ファイルを削除 (非常に一般的な痕跡消去)
rm ~/.bash_history

# 認証ログから自身のIPアドレスを含む行を削除 (より高度)
sudo sed -i '/198.51.100.10/d' /var/log/auth.log
```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **現象**: **ログファイルが空、または更新が止まっている。**
    * **考えられる原因**: `rsyslog` や `systemd-journald` といったロギングデーモンが停止しているか、設定エラーで起動に失敗しています。
    * **解決策**: `systemctl status rsyslog` や `systemctl status systemd-journald` でデーモンの状態を確認し、必要であれば `restart` します。

## 環境変数と設定ファイル

* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

### 関連する設定ファイル

* **`/etc/rsyslog.conf`, `/etc/rsyslog.d/`**: `rsyslog` の設定ファイル。ログの転送先などを定義します。
* **`/etc/systemd/journald.conf`**: `systemd-journald` の設定ファイル。ログの永続化などを設定します。
* **`/etc/logrotate.conf`, `/etc/logrotate.d/`**: ログローテーションの設定ファイル。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: ログファイル自体の**不適切なパーミッション設定**。
* **悪用シナリオ**: ログファイルが一般ユーザーから書き込み可能になっている場合、攻撃者は自身の活動記録を削除・改ざんできます。また、読み取り可能な場合、他のユーザーの活動やシステム設定などの機密情報を窃取できます。

### GTFOBins / LOLBAS における利用例

ログファイルは攻撃対象であり、直接的な権限昇格機能はないためGTFOBinsにはリストされていません。しかし、リストされている多くのコマンドが、ログファイルから情報を窃取したり、ログファイルを改ざんしたりするために利用されます。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: ログファイルのパーミッションを厳格に管理する (`640` or `600`)。
* **Detection (検知)**: **ログのリアルタイムな外部転送**が最も堅牢な対策です。`rsyslog` などを使い、ログを改ざん不可能なリモートのSIEMなどのログサーバーに転送します。FIMツールでログファイルの変更を監視します。

## 注意点・補足

* **時刻同期の重要性**: 正確なログ分析のためには、`NTP` などを使ってシステム全体の時刻を正確に同期させておくことが不可欠です。
