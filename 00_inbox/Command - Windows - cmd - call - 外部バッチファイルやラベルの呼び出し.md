# Command - Windows - cmd - call - 外部バッチファイルやラベルの呼び出し

## 概要

`call` コマンドは、現在のバッチプログラムを終了させることなく、別のバッチプログラムを呼び出したり、同一ファイル内のラベル（サブルーチン）を実行したりするために使用されます。呼び出された処理が終了すると、制御は元の `call` 行の次の行に戻ります。

(出自: `Windows 標準搭載 - cmd.exe 内蔵コマンド`)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 (あれば) |
| :--- | :--- | :--- |
| **直接実行** | `call` を付けずにバッチ名のみ書くと、呼び出し先が終了した時点でプロセス全体が終了し、元の親スクリプトに戻らない。 | 戻る必要がない場合 |
| `start` | 別のウィンドウ（プロセス）を立ち上げて実行する。呼び出し元の終了を待たずに並行処理が可能。 | 並行処理時 |
| `goto` | 指定したラベルへジャンプするが、戻ってくる仕組み（リターンスタック）を持たない。 | 単純な分岐 |

## よく連携されるコマンド

### シナリオ例: 共通処理の部品化 (モジュール化)

* **目的**: 複数のスクリプトで使う「ログ出力」や「環境チェック」を 1 つのファイルにまとめ、必要に応じて呼び出す。
* **連携コマンド**: `call`, `setlocal`, `exit /b`
* **解説**: サブルーチン内で環境変数を汚染しないよう `setlocal` を使い、`exit /b` で親に戻ります。
* **コマンド例**:
    ```cmd
    REM 外部の logger.bat を引数付きで呼び出す
    call logger.bat "INFO" "Processing started."
    ```



## スイッチ/オプション説明

| 引数 / 書式 | 説明 |
| :--- | :--- |
| `[drive:][path]filename` | 呼び出すバッチファイルの名前と場所を指定する。 |
| `arguments` | 呼び出し先のバッチファイルに渡す引数を指定する。 |
| ⭐ `:label` | **【重要】** 同一バッチファイル内の指定したラベルへジャンプし、サブルーチンとして実行する。 |

## よく使われる組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 複雑なインストーラを段階的に実行し、各ステップの完了を確認しながら進める。
* **例**:
    ```cmd
    @echo off
    call :CheckDiskSpace
    call :InstallApp
    echo All steps completed.
    pause
    exit /b

    :CheckDiskSpace
    echo Checking disk space...
    REM 処理内容
    exit /b

    :InstallApp
    echo Installing application...
    REM 処理内容
    exit /b
    ```

### 2. ブルーチーム視点

* **タスク**: 侵害調査（フォレンジック）スクリプトにおいて、収集カテゴリごとに処理を分割し、管理しやすくする。

### 3. レッドチーム視点 (難読化と検知回避)

* **タスク 1: Indirect Command Execution (間接実行による静的解析回避)**
    * **目的**: 実行したいコマンドを直接書かず、変数や引数を介して `call` することで、文字列ベースの検知を遅らせる。
    * **解説**: 攻撃者は `call %env_var%` のように記述し、実行時にのみコマンドが完成するように工作することがあります。
    * **コマンド例**:
        ```cmd
        set a=p
        set b=i
        set c=n
        set d=g
        call %a%%b%%c%%d% 127.0.0.1
        ```

* **タスク 2: バッチファイル内サブルーチンによるペイロード展開**
    * **目的**: 1つのバッチファイル内に複数の役割（デコーダー、ダウンローダー、実行器）を持たせ、`call` で順次制御する。

## エラーメッセージとトラブルシューティング

1.  **別のバッチを直接実行して戻ってこない**
    * **原因**: `call` を付け忘れている。
    * **解決策**: `call other_script.bat` のように `call` を明示する。

2.  **ラベルが見つからない**
    * **原因**: `:label` の指定ミス、またはラベル行の末尾に不要なスペースがある。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **難読化 (Obfuscation - T1027):**
    * `call` はコマンドの遅延展開や変数展開と組み合わされることで、非常に強力な難読化手段となります。例えば、`call set var=%%data:~1,1%%` のような二重パーセント表記を用いた複雑な文字列操作により、解析を困難にします。
* **引数の悪用**:
    * 外部から渡された引数をそのまま `call` に渡すようなスクリプトは、コマンドインジェクションの温床となる可能性があります。

### LOLBAS (Living Off The Land Binaries and Scripts) における利用例

* **機能**: `Arbitrary Command Execution`
* **特性**: `call` 自体は単なる内部機能ですが、悪意あるバッチファイルが環境を調査し、次のペイロードを呼び出す際の「接着剤」として機能します。

### 対応策・緩和策 (ブルーチーム視点)

* **Detection**: 
    * コマンドラインロギング（Event ID 4688）を有効にし、不自然に細切れにされた変数を `call` している挙動がないか監視する。
    * 多数の `call` を含むバッチファイル、特に一時ディレクトリ（`%TEMP%`）から実行されるものを重点的にスキャンする。

## 注意点・補足

* **exit /b の重要性**: サブルーチンから戻る際は必ず `exit /b` を使用してください。単なる `exit` を使うと、コマンドプロンプト（親プロセス）そのものが終了してしまいます。
* **二重展開**: `call` を使うと、変数を 2 回スキャン（展開）させることができます。これは動的な変数名の解決に非常に便利です（例: `call echo %%%varname%%%`）。
