# Command  - Linux - kill - プロセスにシグナルを送信する




## 概要

`kill` は、指定したプロセスID (PID) を持つプロセスに**シグナル**を送信するためのコマンドです。コマンド名から「終了させる」という意味合いが強いですが、実際にはプロセスの終了だけでなく、設定の再読み込みや、一時停止、再開など様々な目的のシグナルを送信できます。

デフォルトで送信されるシグナルは `SIGTERM` (15番) で、プロセスに穏やかな終了 (graceful shutdown) を促します。

(出自: `util-linux` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`pkill` / `killall`** | プロセスID (PID) ではなく、**プロセス名**やその他の属性を指定してシグナルを送信します。PIDを調べる手間が省けるため、より便利です。 | `pkill` (高機能) |
| **`jobs`, `fg`, `bg`**| **現在のシェル**から起動したジョブを管理します。`kill` は、システム上の**あらゆるプロセス**を（権限があれば）対象にできます。 | - |
| **`systemctl stop`**| `systemd` で管理されている**サービス**を停止します。`kill` よりも上位の抽象的な操作であり、サービスの停止にはこちらを使うのが標準です。 | `systemctl` |

## よく連携されるコマンド

`kill` は、操作対象のPIDを特定するために `ps` や `pgrep` と連携するのが一般的です。

### シナリオ例: 応答しなくなったプロセスを強制終了する (インフラ構築・運用視点)

* **目的**: フリーズしてしまったアプリケーションのプロセスIDを特定し、強制的に終了させる。
* **連携コマンド**: `ps`, `grep`, `kill`
* **解説**: `ps aux` でプロセスのリストを取得し、`grep` で対象のプロセスを絞り込み、その出力からPIDを特定します。最後に `kill -9` でそのPIDを指定して、プロセスを強制終了します。
* **コマンド例**:

```bash
# 1. 応答しない 'my-app' プロセスのPIDを探す
ps aux | grep 'my-app'
# -> user      12345  100.0  0.5 ... ./my-app

# 2. 特定したPID (12345) を強制終了する
sudo kill -9 12345
```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **シグナル指定** | |
| ⭐ `-<SIGNAL_NUMBER>` | `-9` や `-15` のように、シグナルを**数値**で指定します。これが最も一般的な使い方です。 |
| `-s <SIGNAL_NAME>` | `KILL` や `TERM` のように、シグナルを**名前**で指定します。 |
| **情報表示** | |
| `-l`, `--list` | 送信可能なシグナルの一覧を表示します。 |

### 主要なシグナル

| 番号 | シグナル名 | 説明 |
| :--- | :--- | :--- |
| **`1`** | `SIGHUP` | (Hangup) 多くのデーモンは、このシグナルを受け取ると**設定ファイルを再読み込み**します。 |
| **`9`** | `SIGKILL` | **強制終了**。プロセスはクリーンアップ処理を行う猶予なく、カーネルによって即座に終了させられます。最後の手段です。 |
| **`15`** | `SIGTERM` | (Terminate) **デフォルトのシグナル**。穏やかな終了を要求します。 |
| **`19`** | `SIGSTOP` | プロセスを**一時停止**させます (`Ctrl+Z` と似ていますが、プロセス側で無視できません)。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: Nginxの設定ファイルを、サービスを停止せずにリロードする。
* **組み合わせ**: `kill -1 <PID>` または `kill -HUP <PID>`
* **解説**: `nginx` のマスタープロセスに `SIGHUP` を送ると、サービスを完全に停止することなく、新しい設定ファイルを読み込んでワーカープロセスを再起動します。ダウンタイムなしで設定変更を適用できます。
* **例**:

```bash
# NginxのマスタープロセスのPIDを調べてHUPシグナルを送る
sudo kill -HUP $(cat /var/run/nginx.pid)
```

### 2. ブルーチーム視点

* **タスク**: 侵害されたホスト上で発見した悪意のあるプロセスを、メモリ情報を保全しつつ活動停止させる。
* **組み合わせ**: `kill -19 <suspicious_PID>` (`SIGSTOP`)
* **解説**: プロセスを `kill -9` で終了させるとメモリ上の証拠が失われます。`-19` でプロセスを「凍結」させることで、活動を停止させつつ、そのメモリイメージをダンプ (`gcore`) するなどして、後の詳細なフォレンジック分析のために状態を保全できます。
* **例**:

```bash
# 不審なプロセス (PID 12345) を凍結
sudo kill -19 12345
```

### 3. レッドチーム視点

* **タスク**: **防御回避 (Defense Evasion)**。侵入したシステム上で動作しているセキュリティーツールを無力化する。
* **組み合わせ**: `kill -9 <PID_of_security_agent>`
* **解説**: 攻撃者がroot権限を奪取した後、EDRのエージェント、監査デーモン (`auditd`) などのプロセスを特定し、`kill -9` で強制終了させます。これにより、その後の攻撃活動が監視・検知されるのを防ぎます。
* **例**:

```bash
# EDRエージェントのプロセスIDを調べて強制終了
sudo kill -9 $(pgrep edr-agent)
```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `kill: (<PID>) - No such process`
    * **考えられる原因**: 指定したPIDのプロセスは、コマンドを実行する時点では既に存在しません。
    * **解決策**: `ps aux` などで、プロセスがまだ存在するか再確認してください。

2. **エラーメッセージ例 2**: `kill: (<PID>) - Operation not permitted`
    * **考えられる原因**: 他のユーザーが所有するプロセスを終了させようとしましたが、`root` 権限がありません。
    * **解決策**: `sudo` を付けてコマンドを実行してください。

## 環境変数と設定ファイル

* `kill` の動作に直接影響する環境変数や設定ファイルは通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `kill` 自体ではなく、**`sudo` での実行権限の不適切な付与**がリスクとなります。
* **悪用シナリオ**: `kill` コマンドの実行権限（特にroot権限）を持つユーザーは、システム上のあらゆるプロセスを終了させることができます。これにより、意図的に、あるいは誤って重要なサービス（データベース、Webサーバー、sshdなど）を停止させ、深刻な**サービス妨害 (DoS)** を引き起こすことが可能です。

### GTFOBins / LOLBAS における利用例

* **機能**: **Privilege Escalation** (権限昇格)
* **解説**: もし `kill` コマンドにSUIDビットが設定されているか、ユーザーが `sudo kill` を実行できる権限を持っている場合、これを利用して権限昇格が可能です。
* **参照**: `https://gtfobins.github.io/gtfobins/kill/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 最小権限の原則に従い、一般ユーザーに `kill` の `sudo` 権限を安易に与えない。
* **Detection (検知)**: `auditd` を使い、`kill` システムコールの実行を監視する。特に、セキュリティエージェントなどの重要なプロセスに対する `SIGKILL` を検知した場合、アラートを発報する。

## 注意点・補足

* **`SIGTERM` vs `SIGKILL`**: まずは常に `kill <PID>` (`SIGTERM`) を試し、プロセスに正常終了する機会を与えてください。`kill -9` (`SIGKILL`) は、ファイルが保存されない、ソケットが正常に閉じられないなどの副作用の可能性があるため、最後の手段としてのみ使用すべきです。
