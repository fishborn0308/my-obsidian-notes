# Command - Linux - objdump - オブジェクトファイルの内容を表示・解析する

## 概要
`objdump` は、ELF 形式などのオブジェクトファイルや実行ファイルから情報を表示します。最大の特長は「逆アセンブル」機能であり、バイナリに含まれる機械語命令を CPU が解釈する命令セット（x86_64, ARM 等）として表示できます。



## 🛠 バイナリ解析ツールの棲み分け
| コマンド | 特徴 | 用途 |
| :--- | :--- | :--- |
| **`objdump`** | 汎用性が高く、逆アセンブルが強力。 | **コード論理の解析** |
| `readelf` | ELF ヘッダーの解析に特化。より正確。 | ヘッダー・セクション情報の確認 |
| `nm` | シンボル（関数名・変数名）の一覧表示。 | 関数名の特定 |
| `strings` | 埋め込まれた文字列を抽出。 | 秘密のメッセージやパスワード探し |

## 主要なオプション
| オプション | 説明 |
| :--- | :--- |
| ⭐️ `-d`, `--disassemble` | **最頻用。** 実行可能なセクション（.text等）を逆アセンブルする。 |
| ⭐️ `-D`, `--disassemble-all` | 全てのセクション（データセクション含む）を逆アセンブルする。 |
| ⭐️ `-S`, `--source` | 可能であれば、**ソースコードとアセンブリを混在**して表示する。 |
| ⭐️ `-x`, `--all-headers` | 全てのヘッダー情報（シンボル表、リロケーション情報等）を表示。 |
| ⭐️ `-s`, `--full-contents` | 各セクションの内容を 16 進数と ASCII でダンプする。 |
| `-M <intel/att>` | アセンブリの記法（Intel か AT&T）を指定する。 |

## よく使われるコマンド例
```bash
# 特定の関数（mainなど）の動きを確認する
objdump -d ./my_program | grep -A 20 "<main>:"

# ソースコードと突き合わせて解析する（デバッグ情報がある場合）
objdump -S ./my_program
````

## 業務シナリオ別活用法

### 1. インフラ構築・運用視点

- **タスク**: バイナリのアーキテクチャ確認。
    
- **解説**: 「このバイナリは x86_64 用か？ ARM 用か？」を、実行せずにヘッダー情報から特定します。
    
- **例**:
    
    Bash
    
    ```
    # ファイルヘッダーを表示し、アーキテクチャを特定
    objdump -f ./unknown_binary
    ```
    

### 2. ブルーチーム視点

- **タスク**: マルウェアの静的解析（Static Analysis）。
    
- **解説**: 実行すると危険な不明なバイナリを、安全な環境で `objdump` にかけ、どのようなシステムコール（ネットワーク通信、ファイル削除など）を呼び出そうとしているか、アセンブリから推測します。
    

### 3. レッドチーム視点

- **タスク**: 脆弱性の特定とエクスプロイト（攻撃コード）の開発。
    
- **解説**: バッファオーバーフローが可能な箇所を特定したり、ROP (Return Oriented Programming) ガジェットと呼ばれる、攻撃に利用可能な命令の断片をバイナリ内から探し出したりします。

## セキュリティに関する考慮事項

### 脆弱性とリスク

- **リバースエンジニアリング**: `objdump` が使えるということは、攻撃者もあなたのプログラムのアルゴリズムを解析できるということです。機密性の高いロジックをバイナリに含める場合は、難読化（Obfuscation）などの対策が必要になります。
    

### 対応策・緩和策 (ブルーチーム視点)

- **シンボルのストリップ**: 本番環境に配置するバイナリは `strip` コマンドであらかじめ関数名などの情報を削除しておくことで、`objdump` で解析された際のヒントを最小限に抑えられます。
