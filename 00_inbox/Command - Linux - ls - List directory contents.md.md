## 1. 概要
`ls` (list) は、指定されたディレクトリ内のファイルやディレクトリの情報をリスト表示するための、UNIX系OSにおいて最も基本的かつ頻用されるコマンドの一つです。

- **出自**: GNU `coreutils` パッケージに含まれる。
- **核心的役割**: ファイルシステムの構造を確認し、パーミッション、所有者、サイズ、タイムスタンプなどのメタデータを可視化すること。ペンテストにおける初期潜入後の挙動調査（Enumeration）の出発点となります。

## 2. 類似コマンド比較

| コマンド | 特徴・モダンさ | 主な用途 |
| :--- | :--- | :--- |
| **ls** | 標準的、ほぼ全ての環境にプリインストール。 | 汎用的な確認、スクリプト利用。 |
| **dir** | `ls` に似ているが、デフォルトで色が反映されない等、Windows風の挙動。 | 後方互換性維持。 |
| **vdir** | `ls -l` と同等の詳細表示をデフォルトで行う。 | 詳細情報の即時確認。 |
| **exa / eza** | **モダン (Rust製)**。アイコン表示やGitステータス表示に対応。 | インフラエンジニアのローカル環境、視認性重視。 |
| **find** | 検索に特化。ディレクトリを再帰的に深く掘り下げる際に強力。 | 特定条件のファイル抽出。 |

## 3. 連携例

`ls` はパイプラインを通じて他のコマンドと組み合わせることで、強力な調査ツールとなります。

- **サイズ順にソートして上位を表示**:
  `ls -lS | head -n 11` (ヘッダーを含めて上位10ファイルを確認)
- **特定の拡張子を持つファイルのパーミッションを確認**:
  `ls -l *.conf | grep "rw-r--r--"`
- **更新日時が新しい順に表示し、更新を監視する**:
  `watch -d 'ls -lt | head'`
- **ディレクトリのみを表示する**:
  `ls -d */` または `ls -l | grep "^d"`

## 4. 全オプション（主要なものは⭐️）

| オプション | 役割 | 備考 |
| :--- | :--- | :--- |
| **-a, --all** ⭐️ | ドットファイル（.）を含む全てのファイルを表示。 | 設定ファイルの発見に必須。 |
| **-l** ⭐️ | ロングフォーマットで詳細（権限, 所有者, サイズ等）を表示。 | パーミッション確認の基本。 |
| **-h, --human-readable** ⭐️ | サイズを KB, MB, GB など読みやすい形式で表示。 | `-l` と併用推奨。 |
| **-t** ⭐️ | ファイルの更新時刻順にソートして表示。 | 最近の活動を特定する。 |
| **-r, --reverse** | ソート順を逆にする。 | `-t` や `-S` と組み合わせて使用。 |
| **-S** ⭐️ | ファイルサイズが大きい順にソート。 | ログ肥大化の調査などに有用。 |
| **-R, --recursive** ⭐️ | ディレクトリ内を再帰的に表示。 | 全体構造の把握。 |
| **-i, --inode** | 各ファイルの i-node 番号を表示。 | ハードリンクの調査やファイル特定。 |
| **-d, --directory** ⭐️ | ディレクトリ自体を表示し、中身を表示しない。 | ディレクトリ自体の権限確認。 |
| **-F, --classify** | 種別識別子（/=@*）を末尾に付与。 | 実行ファイル（*）等の即時判別。 |
| **-u** | アクセス時刻（atime）でソート。 | フォレンジック調査で利用。 |
| **-c** | 状態変更時刻（ctime）でソート。 | 権限変更や移動の追跡。 |
| **--color[=WHEN]** | 表示に色を付ける（always/auto/never）。 | 視認性の向上。 |
| **-1** | 1行に1ファイルずつ表示。 | スクリプト処理での利用。 |

## 5. 業務シナリオ別詳細

### A. インフラ運用視点：構築・トラブルシューティング
インフラエンジニアにとって `ls` は、サービスが正常にデプロイされているか、あるいはログがストレージを圧迫していないかを確認するための第一選択肢です。

- **ログの肥大化調査**: 
  どのログファイルがディスク容量を消費しているか、サイズ順（`-S`）かつ読みやすい単位（`-h`）で特定します。
- **デプロイ確認**: 
  シンボリックリンクの向き（`->`）が正しいバージョンを指しているかを確認します。
- **権限の整合性チェック**: 
  Webサーバー（www-data等）が設定ファイルを読み取れるパーミッションになっているか、所有者が適切かを確認します。

### B. ブルーチーム視点：不審な挙動の検知・フォレンジック
防御側としては、標準的なファイル名に紛れた「異常」を嗅ぎ取るために `ls` を駆使します。

- **タイムスタンプ分析 (`--time`)**: 
  `atime`（アクセス時刻）、`ctime`（状態変更時刻）を比較し、攻撃者がファイルを編集・実行した痕跡を追跡します。
- **隠しファイルの徹底捜索 (`-a`)**: 
  `.bash_history` の削除や、`.config` 配下に置かれた不審なバイナリ、あるいは `...`（ドット3つ）といった不自然なディレクトリ名を探します。
- **不正な権限設定の発見**: 
  誰でも書き込み可能なディレクトリ（Sticky Bitが立っていない `/tmp` 等）の有無を確認します。

### C. レッドチーム視点：列挙 (Enumeration) ・権限昇格
攻撃側やペンテスターにとって、`ls` はターゲット内部の「お宝」を探すサーチライトです。

- **機密情報の探索**: 
  `.ssh/id_rsa`、`.aws/credentials`、`config.php`（DBパスワード）などの有無を、再帰的（`-R`）に調査します。
- **バイナリの探索 (SUID/SGID)**: 
  権限昇格に利用できる `s` ビットが立ったファイル（GTFOBinsに掲載されているようなバイナリ）を探します。
- **書き込み可能ディレクトリの特定**: 
  自身がペイロードを配置できる場所を探すため、パーミッションの `w` フラグに注目します。

## 6. 実務ログ例

### ① インフラ：ログファイルのサイズ調査

```bash
# /var/log 内のファイルをサイズが大きい順に表示
user@srv:~$ ls -lhS /var/log | head -n 5
total 1.2G
-rw-r----- 1 syslog adm  850M Feb 26 20:26 syslog
-rw-r----- 1 syslog adm  210M Feb 25 23:59 syslog.1
-rw-r----1 root   root 120M Feb 26 10:15 auth.log
-rw-rw---- 1 root   utmp  15M Feb 26 20:00 lastlog
````

### ② ブルーチーム：タイムスタンプによる追跡 (ctime)

```bash
# ファイルの内容ではなく、パーミッションや名前が変更された時刻(ctime)を表示
# 攻撃者が所有権を書き換えた直後などを想定
user@srv:~$ ls -lc --full-time /etc/passwd
-rw-r---- 1 root root 2652 2026-02-26 20:15:32.123456789 +0900 /etc/passwd
# ↑ OSインストール時と異なる不自然な変更時刻があれば警戒
```

user@target:~$ ls -laR /home/victim/ | grep -E "\.ssh|\.bash_history|\.env|config"
