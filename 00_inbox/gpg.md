# `gpg` - OpenPGP暗号化・署名ツール

## 概要

`gpg` (GNU Privacy Guard) は、OpenPGP標準のフリーな実装であり、データや通信の暗号化とデジタル署名を行うためのコマンドラインツールです。主に2つの目的で利用されます。

1. **機密性 (Confidentiality)**: 公開鍵暗号を用いてファイルを暗号化し、対応する秘密鍵を持つ特定の受信者だけが内容を読めるようにする。
2. **完全性 (Integrity) と認証 (Authentication)**: デジタル署名を作成し、メッセージが指定された送信者から送られ、途中で改ざんされていないことを保証する。

ソフトウェアの配布元検証、ファイルの安全な保管、メールの暗号化など、デジタルセキュリティのさまざまな場面で利用されます。

(出自: `gnupg` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`openssl`** | SSL/TLS、証明書管理などを含む巨大な暗号ツールキット。`gpg` はOpenPGP標準と「信用の輪」モデルに特化。 | `openssl` (TLS関連), `gpg` (ファイル/メッセージ署名) |
| **`age`** | モダンでシンプルなファイル暗号化ツール。`gpg` のような複雑な鍵管理は不要。単純なファイルの暗号化には `age` が手軽。 | `age` (手軽な暗号化) |

## よく連携されるコマンド

`gpg` は、`tar` などのアーカイバと連携し、複数のファイルをまとめて暗号化する際によく使われます。

### シナリオ例: ディレクトリをアーカイブ・暗号化する (レッドチーム視点)

* **目的**: 窃取した複数のファイルやディレクトリを、転送しやすいように1つのファイルにまとめ、かつ暗号化して内容を隠蔽する。
* **連携コマンド**: `tar`, `gpg`, パイプ (`|`)
* **解説**: `tar` でディレクトリをアーカイブ・圧縮し、その結果をパイプで `gpg` に渡します。`-c` オプションでパスワードベースの対称暗号化を行い、中間ファイルをディスクに残さずに処理を完結させます。
* **コマンド例**:

```bash
# /path/to/loot ディレクトリをtarで固め、パイプ経由でgpgで暗号化する
tar -czf - /path/to/loot | gpg -c -o loot.tar.gz.gpg
```

## オプション説明 (主要なアクション)

| オプション | 説明 |
| :--- | :--- |
| **鍵管理** | |
| ⭐ `--gen-key` | 新しい鍵ペア（公開鍵と秘密鍵）を対話形式で生成します。 |
| ⭐ `--list-keys` | 自分の公開鍵束（キーリング）にある公開鍵を一覧表示します。 |
| `--list-secret-keys`| 自分が持つ秘密鍵を一覧表示します。 |
| `--import <key_file>` | ファイルから公開鍵（または秘密鍵）をインポートします。 |
| `--export -a <key_id>`| 指定した `key_id` の公開鍵をASCII形式 (`-a`) でエクスポートします。 |
| **暗号化と復号** | |
| ⭐ `-e -r <recipient> <file>`| (Encrypt) 受信者 (`-r`) の公開鍵でファイルを**非対称暗号化**します。 |
| `-c <file>` | パスワードを使ってファイルを**対称暗号化**します。 |
| ⭐ `-d <encrypted_file>` | (Decrypt) 暗号化されたファイルを復号します。 |
| **署名と検証** | |
| ⭐ `-s <file>` | (Sign) ファイルにデジタル署名を行い、圧縮された署名付きファイル (`.gpg`) を生成します。 |
| `-b <file>` | (Detach-sign) 元のファイルとは別に、署名ファイル (`.sig`) を生成します。 |
| ⭐ `--verify <sig_file> [data]`| 署名を検証します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: ソフトウェア配布元の正当性を検証する。
* **組み合わせ**: `gpg --verify <package.asc> <package.tar.gz>`
* **解説**: ソフトウェアのリリースファイルとその署名ファイルを検証します。これにより、ファイルが本物の開発元からリリースされ、ダウンロード中に改ざんされていないことを確認できます。
* **例**:

    ```bash
    # 1. 開発者の公開鍵をインポート (初回のみ)
    gpg --recv-keys <開発者のキーID>
    # 2. 署名を検証
    gpg --verify apache-airflow-2.9.3.tar.gz.asc apache-airflow-2.9.3.tar.gz
    # -> "Good signature from ..." と表示されればOK
    ```

### 2. ブルーチーム視点

* **タスク**: 機密情報を含むインシデント報告書を、特定の相手にのみ安全に共有する。
* **組み合わせ**: `gpg -e -r <recipient_key_id> -o <encrypted_report> <original_report>`
* **解説**: `-e` (暗号化) と `-r` (受信者) を使い、ファイルを相手の公開鍵で暗号化します。これにより、たとえファイルが漏洩しても、秘密鍵を持つ相手以外は内容を読むことができません。
* **例**:

    ```bash
    # 同僚 (bob@example.com) の公開鍵で報告書を暗号化
    gpg -e -r bob@example.com -o report.docx.gpg report.docx
    ```

### 3. レッドチーム視点

* **タスク**: 窃取したデータを暗号化し、ネットワーク監視による検知を回避して持ち出す。
* **組み合わせ**: `tar ... | gpg -e -r <attacker_key_id> ...`
* **解説**: 窃取したデータを `tar` でアーカイブし、その結果をパイプで `gpg` に渡して、自身の公開鍵で暗号化します。これにより、ネットワーク上のDLPやIDSがファイルの中身をスキャンして検知するのを回避できます。
* **例**:

    ```bash
    # /etc と /home/user の中身を、攻撃者の鍵 (0xATT4CK3R) で暗号化
    tar -czf - /etc /home/user | gpg -e -r 0xATT4CK3R -o loot.tar.gz.gpg
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `gpg: decryption failed: No secret key`
    * **考えられる原因**: あなたの秘密鍵束に、そのファイルを復号するために必要な秘密鍵が存在しません。
    * **解決策**: 正しい受信者であるか確認してください。もし自分のためのファイルであれば、必要な秘密鍵を `gpg --import` でインポートする必要があります。

2. **メッセージ例 2**: `gpg: WARNING: This key is not certified with a trusted signature!`
    * **考えられる原因**: 使用した公開鍵が、あなた自身やあなたが信頼する第三者によって署名（信頼のお墨付き）されていません。
    * **解決策**: これはエラーではなく警告です。公開鍵の指紋 (fingerprint) を別の信頼できる経路で相手に確認し、正しければ `gpg --edit-key <key_id>` で `sign` コマンドを実行して、その鍵を明示的に信頼します。

## 環境変数と設定ファイル

* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

### 影響を与える環境変数

* **`GNUPGHOME`**: `gpg` がキーリングや設定ファイルを保存するディレクトリを指定します。デフォルトは `~/.gnupg` です。

### 関連する設定ファイル

* **`~/.gnupg/gpg.conf`**: `gpg` の挙動をカスタマイズするためのユーザー設定ファイル。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `gpg` 自体よりも、**秘密鍵の管理不備**が最大のリスクです。
* **悪用シナリオ**: 攻撃者はシステム侵入後、`~/.gnupg` ディレクトリを探し、パスフレーズで保護されていない秘密鍵や、推測しやすいパスフレーズの秘密鍵を窃取しようとします。秘密鍵が盗まれれば、そのユーザーになりすまして署名したり、暗号化された通信を復号したりすることが可能になります。

### GTFOBins / LOLBAS における利用例

* **機能**: **Command Execution**, **File Write/Read**
* **解説**: `gpg` は `--command-fd` や `--exec-command` のようなオプションを持っており、特定の操作中に外部コマンドを実行させることができます。もしユーザーが `sudo gpg` を実行できる権限を持っている場合、これらのオプションを悪用して任意のコマンドを root 権限で実行できる可能性があります。
* **参照**: `https://gtfobins.github.io/gtfobins/gpg/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **秘密鍵は必ず強力なパスフレーズで保護**し、物理的なセキュリティキー (YubiKeyなど) に保管することが理想です。`sudoers` の設定で、ユーザーに `gpg` のような高機能なツールの `sudo` 権限を安易に与えないでください。
* **Detection (検知)**: `auditd` で `~/.gnupg` ディレクトリへの不審なアクセスや、`gpg` コマンドの実行を監視する。

## 注意点・補足

* **信用の輪 (Web of Trust)**: OpenPGPは、中央集権的な認証局 (CA) に頼らず、ユーザー同士が互いの公開鍵を署名しあうことで信頼関係を構築する「信用の輪」というモデルを採用しています。
