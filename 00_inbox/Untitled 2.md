# 第2章：Nginx の攻撃・検知・防御

## 1. 本章の到達目標

本章の完了後、読者は以下の能力を習得する。

- Nginx のイベント駆動型アーキテクチャに起因するプロセスの振る舞いを理解し、異常な子プロセスの生成を特定できる。
    
- リバースプロキシ設定におけるヘッダ操作（HTTP Request Smuggling等）の攻撃原理を分析できる。
    
- Nginx 特有の設定ミス（Alias/Locationの不備）を突き、非公開ディレクトリへのアクセスを再現・検知できる。
    
- SOCアナリストとして、上流（Nginx）と下流（Appサーバ）のログを突き合わせ、攻撃の到達範囲を判定できる。
    

---

## 2. 技術概要

Nginx は、高並列処理に特化したオープンソースの Web サーバ / リバースプロキシである。

- **役割：** 静的ファイルの高速配信、ロードバランシング、SSL 終端、およびキャッシュサーバ。
    
- **想定構成：** 現代のインフラでは、Nginx をフロントエンド（リバースプロキシ）に置き、背後のリクエストを Apache や Gunicorn などのアプリケーションサーバへ振り分ける構成が一般的である。
    
- **使用レイヤ：** 主に L7（HTTP/HTTPS）だが、Stream モジュールにより L4（TCP/UDP）の負荷分散も担う。
    

---

## 3. アーキテクチャ詳細

### 3.1 プロセス構造：イベント駆動モデル

Apache（プロセス/スレッド割り当て型）と異なり、Nginx は少数のプロセスで大量のリクエストを処理する。

- **Master プロセス：** 設定の読み込み、ポートのバインド、Worker プロセスの管理を行う（`root` 権限）。
    
- **Worker プロセス：** 実際にクライアントのリクエストを処理する（`nginx` 等の非特権ユーザー権限）。
    
- **攻撃の視点：** 1つの Worker が複数の接続を保持するため、Worker プロセスが RCE（リモートコード実行）等でクラッシュまたは乗っ取られた場合、その Worker が処理していた他のセッションへの影響や情報漏洩のリスクを考慮する必要がある。
    

### 3.2 通信フロー（図解）

1. クライアントが Nginx（Port 443）に接続。
    
2. Nginx が TLS 復号を行い、`proxy_pass` 指令に基づき内部ネットワークの App サーバ（Port 8000）へ転送。
    
3. この際、Nginx は `proxy_set_header` を用いてクライアントの真の IP アドレスを後続に伝える。
    

### 3.3 設定ファイルとデータ保存場所

- **メイン設定：** `/etc/nginx/nginx.conf`
    
- **サイト別設定：** `/etc/nginx/conf.d/` または `/etc/nginx/sites-enabled/`
    
- **デフォルトルート：** `/usr/share/nginx/html/` または `/var/www/html/`
    

---

## 4. ログ構造

Nginx はデフォルトで 2 種類のログを出力する。

### 4.1 デフォルトログ

1. **access.log：** 処理された全リクエストを記録。
    
2. **error.log：** 設定エラー、接続タイムアウト、アップストリーム（背後のサーバ）からのエラー等を記録。
    

### 4.2 実際のログ例

**例1：リバースプロキシを介した不正アクセス（access.log）**

Plaintext

```
172.16.0.10 - - [19/Feb/2026:11:00:01 +0900] "GET /admin/config HTTP/1.1" 403 153 "-" "Mozilla/5.0" "192.168.50.22"
```

- **分析ポイント：** 172.16.0.10（プロキシ IP）からのリクエストだが、末尾の `192.168.50.22`（X-Forwarded-For）が真の攻撃元。403 によりアクセス拒否されていることがわかる。
    

**例2：アップストリームのタイムアウト（error.log）**

Plaintext

```
2026/02/19 11:05:22 [error] 1234#0: *56 upstream timed out (110: Connection timed out) while connecting to upstream, client: 192.168.50.22, server: example.com, request: "GET /api/v1/search?id=1' OR '1'='1' HTTP/1.1", upstream: "http://10.0.0.5:8000/api/v1/search?id=1' OR '1'='1'"
```

- **分析ポイント：** 攻撃者が SQL インジェクションを試行した結果、背後の DB/App サーバが過負荷またはクラッシュし、Nginx がタイムアウトを記録している。

## 5. 攻撃観点：なぜ Nginx が狙われるか

Nginx はインフラの「最前線」に位置するため、ここを突破またはバイパスできれば、内部ネットワークへの全権を得られる可能性がある。

### 5.1 設定ミスによるパス・トラバーサル (Off-by-slash)

`alias` 指令の記述ミスにより、意図しない親ディレクトリが露出する。

- **脆弱な設定例：**
    
    `location /static { alias /var/www/app/static/; }`
    
- **攻撃フロー：** 攻撃者が `/static../settings.py` にアクセスすると、`/var/www/app/settings.py` が読み取れてしまう。
    

### 5.2 HTTP Request Smuggling (HRS)

Nginx と背後のサーバ（Apache 等）で、`Content-Length` と `Transfer-Encoding` ヘッダの解釈に差異がある場合に発生する。攻撃者はリクエストを「隠し持ち」、後続の他ユーザーのリクエストを乗っ取る。

## 6. 検知観点

### 6.1 ログ確認ポイント

- **ステータスコード 499 (Client Closed Request):** Nginx 特有のコード。攻撃者がスキャナを高速で回し、応答を待たずに切断している際によく見られる。
    
- **ステータスコード 400 (Bad Request):** HTTP Request Smuggling などのプロトコル操作攻撃の失敗時。
    

### 6.2 エンドポイント視点（EDR/Auditd）

- **不審な子プロセスの生成：** Nginx の Worker プロセスが `sh`, `bash`, `python` 等を起動していないか。
    
- **一時ディレクトリの書き込み：** `/var/lib/nginx/tmp` 等に実行可能ファイルが配置されていないか。
    

### 6.3 実務での「誤検知」例

- **ヘルスチェックの大量ログ：** ロードバランサからの死活監視により 200 OK が数秒おきに記録され、攻撃ログが埋もれる。
    
- **対策：** ログ設定で `access_log off;` を特定のロケーション（`/health` 等）に適用し、ノイズを削減する。
    

---

## 7. ハードニング（防御策）

|**防御策**|**実装例**|**対応攻撃 / ATT&CK**|
|---|---|---|
|**バージョン隠蔽**|`server_tokens off;`|偵察 (T1592)|
|**バッファ制限**|`client_body_buffer_size 1k;`|DoS / オーバーフロー (T1499)|
|**セキュリティヘッダ**|`add_header X-Frame-Options DENY;`|クリックジャッキング|
|**アクセス制限**|`allow 10.0.0.0/24; deny all;`|未認可アクセス (T1078)|

## 8. 演習

### 8.1 攻撃再現（パス・トラバーサル）

1. ターゲットの Nginx で `/static` の alias 設定が不備であることを確認。
    
2. `curl http://target/static../etc/passwd` を実行。
    
3. ファイル内容が返ってくるか確認。
    

### 8.2 ログ確認手順

- `grep "static\.\." /var/log/nginx/access.log` を実行し、ドットが含まれる異常なリクエストを抽出する。
    

---

## 9. まとめ

Nginx は単なる Web サーバではなく、インフラの境界を制御するコンポーネントである。

- **ペンテスター**は、プロキシ背後のサーバとの「解釈の差異」や「パス解決の不備」を狙う。
    
- **SOCアナリスト**は、X-Forwarded-For を正確に追跡し、Nginx でブロックされたのか、背後のサーバまで攻撃が到達したのかを見極める必要がある。
    

---

## 10. ATT&CK対応

- **T1190 (Exploit Public-Facing Application):** Nginx 自体または背後のアプリの脆弱性悪用。
    
- **T1557.002 (HTTP Request Smuggling):** プロキシの解釈差異を利用した中間者攻撃。
    
- **T1595 (Active Scanning):** Nginx のエラーコードをトリガーにしたディレクトリ・ブルートフォース。
