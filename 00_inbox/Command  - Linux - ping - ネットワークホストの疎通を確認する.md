# Command  - Linux - ping - ネットワークホストの疎通を確認する

## 概要

`ping` は、指定したホスト（IPアドレスまたはドメイン名）がネットワーク的に**到達可能か**どうかをテストするための、最も基本的なネットワーク診断ツールです。`ping` は、**ICMP (Internet Control Message Protocol)** の「エコー要求 (echo request)」パケットをターゲットホストに送信し、そのホストからの「エコー応答 (echo reply)」を待ち受けます。

応答が返ってくることで、ホストが稼働していること、そしてそこまでのネットワーク経路に問題がないことを確認できます。また、応答時間 (**latency** or **RTT**) も表示されるため、ネットワークの遅延を測定するのにも利用されます。

(出自: `iputils-ping` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`traceroute` / `mtr`**| 宛先までの**ネットワーク経路**を調査します。`ping` は**最終的な宛先**との疎通のみを確認します。「繋がらない」ときに、どこが問題かを調べるのが `traceroute` や `mtr` です。 | `mtr` (詳細診断) |
| **`netcat -z`** | 特定の**TCP/UDPポート**の疎通を確認します。`ping` はICMPを利用し、ネットワーク層 (L3) での疎通を確認します。ファイアウォールが `ping` をブロックしていても、Webサーバーのポートは開いている、という状況はよくあります。 | - |
| **`arping`** | 同じLANセグメント内のホストに対し、ARP (L2) レベルで疎通を確認します。 | - |

## よく連携されるコマンド

`ping` は、スクリプト内でホストの死活監視を行う際、出力を `grep` でフィルタリングして使われることがあります。

### シナリオ例: Pingスイープによる生存ホスト探索 (レッドチーム視点)

* **目的**: ネットワーク内のどのホストがオンラインであるかを探索する。
* **連携コマンド**: `for`ループ、 `grep`
* **解説**: シェルの `for` ループを使ってサブネット内の全てのIPアドレスに対して `ping` を送信し、応答があった (`bytes from`) ホストのみを表示します。
* **コマンド例**:

```bash
# 192.168.1.0/24 ネットワーク内の生存ホストを探索する簡易スクリプト
for i in {1..254}; do ping -c 1 -W 1 192.168.1.$i | grep "bytes from"; done
```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **動作制御** | |
| ⭐ `-c <count>` | パケットを `<count>` 回送信した後に停止します。Linuxではデフォルトで無限に送信し続けます。 |
| ⭐ `-i <interval>` | パケットを送信する間隔を秒数で指定します。デフォルトは1秒です。 |
| `-W <timeout>`| 応答を待つ時間を秒数で指定します。 |
| **パケット設定** | |
| `-s <size>` | 送信するパケットのデータサイズをバイト単位で指定します。 |
| `-I <interface>`| 送信元のネットワークインターフェイスを指定します。（例: `-I eth1`） |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 「インターネットは使えるか？」という基本的な接続性を確認する。
* **組み合わせ**: `ping -c 5 8.8.8.8`
* **解説**: Google Public DNS (`8.8.8.8`) のような、常時稼働している安定したホストに `ping` を送信することで、サーバーからインターネットへの基本的な接続が確立されているかを確認します。`-c 5` で5回に制限します。
* **例**:

```bash
# google.com への疎通を5回だけ確認する
ping -c 5 google.com
```

### 2. ブルーチーム視点

* **タスク**: 既知の悪意のあるIPが、現在も活動中（オンライン）であるかを確認する。
* **組み合わせ**: `ping -c 5 <suspicious_ip>`
* **解説**: 脅威インテリジェンス情報から得られたマルウェアのC2サーバーのIPアドレスリストに対し、`ping` を実行して疎通を確認します。ただし、応答がないからといって活動していないとは限りません（ICMPをブロックしている可能性があるため）。
* **例**:

```bash
# C2サーバーの疑いがあるIPに疎通確認
ping -c 5 198.51.100.10
```

### 3. レッドチーム視点

* **タスク**: 侵入したネットワーク内の生存ホストを探索（Pingスイープ）する。
* **組み合わせ**: (スクリプト内) `ping -c 1 <target_ip>`
* **解説**: 上記「よく連携されるコマンド」のシナリオ例を参照。`ping` は、ネットワーク上のアクティブなホストを発見するための最も基本的な手段です。
* **例**:

		```bash
    # 単一ホストの生存確認

    ping -c 1 192.168.1.101

    ```


## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **メッセージ**: `100% packet loss` または `Request timed out`
		* **考えられる原因**:
				* 宛先ホストがシャットダウンしている。
				* 途中または宛先の**ファイアウォールがICMPパケットをブロックしている**。（非常によくある）
				* ネットワーク経路に問題がある。
		* **解決策**: 宛先ホストがICMPに応答しないように設定されている可能性を考慮します。`nc` など、別のツールでTCP/UDPポートへの接続を試みます。

2. **メッセージ**: `Destination Host Unreachable`
		* **考えられる原因**: 自分からゲートウェイまでの経路上にあるルーターが、宛先への経路を知らないと判断しています。
		* **解決策**: 自身のルーティングテーブル (`ip route show`) や、途中のルーターの設定を確認する必要があります。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `ping` 自体ではなく、それが利用するICMPプロトコルが悪用されることがあります。
* **悪用シナリオ**:
	* **偵察**: `ping` スイープは、ネットワーク上のアクティブなホストを発見するための最も基本的な手段です。
	* **サービス妨害攻撃 (DoS)**: 大量のICMPパケットを送りつける「Pingフラッド攻撃」は、古典的なDoS攻撃の一種です。

### GTFOBins / LOLBAS における利用例

* **機能**: **データ持ち出し (Data Exfiltration)**
* **解説**: `ping` の `-p <pattern>` オプションは、パケットのデータ部分に16進数の文字列を埋め込むことができます。攻撃者はこの機能を利用し、窃取したいファイルの内容を16進数に変換してICMPパケットに埋め込み、外部の自身のサーバーに `ping` を送信することで、隠密にデータを持ち出すことができます。
* **参照**: `https://gtfobins.github.io/gtfobins/ping/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 境界ファイアウォールで、外部からの不要なICMPエコー要求をブロックまたはレート制限する。
* **Detection (検知)**: 内部ネットワークから外部への大量のICMPパケットや、特定のホストからのPingスイープをネットワーク監視ツールで検知する。

## 注意点・補足

* **応答がない ≠ オフライン**: ファイアウォールによってICMPがブロックされていることは非常に一般的です。`ping` の応答がないからといって、そのホストがダウンしていると結論付けるのは早計です。
