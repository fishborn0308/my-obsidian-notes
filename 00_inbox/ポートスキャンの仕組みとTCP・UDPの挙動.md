
# ポートスキャンの仕組みとTCP/UDPの挙動
## 1. ネットワーク層の構造とプロトコルの違い

ポートスキャンを理解する基礎として、OSI参照モデルにおけるデータの流れと、トランスポート層の2大プロトコルの特性を把握する必要があります。

- **層間のデータの流れ:**
    
    アプリ層(L7) → トランスポート層(L4: TCP/UDP) → ネットワーク層(L3: IP) → データリンク層(L2) → 物理層(L1)
    
- **TCP (Transmission Control Protocol):**
    
    **接続型。** データの順序制御、再送制御を行い、信頼性の高い通信を提供します。
    
- **UDP (User Datagram Protocol):**
    
    **コネクションレス型。** 各パケットは独立しており、再送や順序制御は行いません。軽量ですが信頼性は上位アプリに依存します。
    

---

## 2. TCPヘッダーと制御ビット（フラグ）

TCPはヘッダー内の**制御ビット**を使用して通信の状態を管理します。ポートスキャンでは、これらのビットに対するターゲットの反応を利用してポートの状態を判定します。

|**フラグ**|**役割**|**スキャンにおける意味**|
|---|---|---|
|**SYN**|接続開始（同期）|セッション確立の最初の要求。|
|**ACK**|確認応答|以前のパケットを受け取ったことを示す。|
|**FIN**|接続終了|送信データの完了を伝え、正常終了を試みる。|
|**RST**|強制リセット|エラーなどで接続を即時切断する。**ポートが閉じているサイン**によく使われる。|
|**PSH**|プッシュ|データをバッファせず直ちにアプリへ渡す。|
|**URG**|緊急|緊急ポインタが有効な重要データ。|

## 3. TCPポートスキャンの仕組み

正規の接続プロセスである「**3ウェイハンドシェイク**」の挙動を利用します。

### スキャンの判定ケース

- **ポートが開いている場合:**
    
    `SYN`を送信 → ターゲットから`SYN-ACK`が返ってくる。
    
- **ポートが閉じている場合:**
    
    `SYN`を送信 → ターゲット（またはFW）から`RST-ACK`が返ってくる。すぐに「閉じている」と判断できるため、スキャンが高速に進みます。
    
- **フィルタリングされている場合:**
    
    `SYN`を送信 → **応答なし**（タイムアウト）。FWがパケットを黙ってドロップしているか、ICMP到達不能エラーが返る場合にこの状態になります。
    

---

## 4. UDPポートスキャンの仕組みと課題

UDPは「状態」を持たないため、TCPに比べてスキャンの信頼性が低く、速度も遅くなる傾向があります。

### スキャンの判定ケース

- **ポートが開いている可能性が高い:**
    
    UDPパケットを送信 → 何らかの**UDP応答**が返ってきた場合。
    
- **ポートが閉じている:**
    
    UDPパケットを送信 → **ICMPポート到達不能（Type 3, Code 3）**が返ってきた場合。
    
    > ※多くのOSではICMPの送信レート制限があるため、スキャン速度を落とす必要があります。
    
- **判定不能 (open|filtered):**
    
    UDPパケットを送信 → **応答なし**。
    
    「ポートが閉じているがFWがドロップした」のか、「ポートは開いているが、特定のデータ（ペイロード）がないので無視された」のか区別がつきません。
    

### 対策

Nmapなどのツールでは、特定のポート（DNS用など）に対して**プロトコル固有のペイロード**を送信することで、アプリからの応答を引き出し、精度を高める工夫がなされています。
