# `fg` - ジョブをフォアグラウンドで再開する

## 概要

`fg` (foreground) は、シェルのジョブ管理機能の一部であり、**バックグラウンドで実行中または一時停止 (suspend) 状態のジョブを、フォアグラウンドに切り替えて実行を再開させる**ためのシェル組み込みコマンドです。フォアグラウンドに切り替えることで、そのジョブは再びターミナルの標準入力を受け付けるようになり、対話的な操作や `Ctrl+C` による中断が可能になります。

(出自: `bash`, `zsh` などのシェルの組み込み機能)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`bg`** | 一時停止中のジョブを**バックグラウンド**で実行再開させます。`fg` はジョブを**フォアグラウンド**に移動させる対になるコマンドです。 | - |
| **`jobs`** | `fg` で操作対象とするジョブ番号 (`%1`など) を確認するためのコマンドです。 | - |
| **`Ctrl + Z`** | フォアグラウンドで実行中のプロセスを**一時停止**させます。`fg` や `bg` で再開するための前段階となる操作です。 | - |

## よく連携されるコマンド

`fg` は、`vi` などのエディタ作業を一時中断して別のコマンドを実行し、すぐに元の作業に戻る、という一連の流れで多用されます。

### シナリオ例: 設定ファイル編集中に別のコマンドを実行する (インフラ構築・運用視点)

* **目的**: 設定ファイルを `vi` で編集中に、別のファイルの内容を確認したり、コマンドの実行結果を確かめたくなった。
* **連携コマンド**: `vi`, `Ctrl+Z`, `grep` (または他のコマンド), `fg`
* **解説**: `vi` での編集作業を `Ctrl+Z` で一時停止し、必要な確認作業を行った後、`fg` を実行するだけで即座に `vi` の編集セッションに戻ることができます。
* **コマンド例**:

```bash
# 1. viで設定ファイルを編集中
vi /etc/nginx/nginx.conf

# 2. Ctrl+Z を押して一時停止
# [1]+  Stopped                 vi /etc/nginx/nginx.conf

# 3. 別の設定値を確認
grep "server_name" /etc/nginx/sites-available/default

# 4. fg で vi の編集に戻る
fg
```

## オプション説明 (`fg` の引数)

`fg` コマンド自体にオプションはありませんが、引数として操作対象の「ジョブスペック」を指定します。

| 引数 (ジョブスペック) | 説明 |
| :--- | :--- |
| **ジョブ指定** | |
| ⭐ (引数なし) | 最後に一時停止またはバックグラウンドに送られたジョブ (`%` または `%%` と同義) を対象とします。 |
| `%n` | `jobs` コマンドで表示されるジョブ番号 `n` のジョブを対象とします。（例: `fg %1`） |
| `%string` | コマンド名が `string` で始まるジョブを対象とします。（例: `fg %vi`） |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 複数のターミナルを開かずに、単一のシェルで「ちょっと寄り道してすぐ戻る」という作業を効率的に行う。
* **組み合わせ**: `fg` (引数なし)
* **解説**: 設定ファイルの編集中に一時停止し、別のコマンドを実行、その後 `fg` で編集に戻る、というワークフローは、サーバー管理者にとって基本的なテクニックです。
* **例**:

```bash
# 1. ログをlessで閲覧中に一時停止
less /var/log/syslog
# (Ctrl+Z)

# 2. ディスク容量を確認
df -h

# 3. 再びlessの画面に戻る
fg
```

### 2. ブルーチーム視点

* **タスク**: バックグラウンドで実行中の調査コマンドの制御を取り戻す。
* **組み合わせ**: "fg <job_id>"
* **解説**: 時間のかかるログ検索 (`grep`など) をバックグラウンドで実行していたが、その途中経過や最終結果を対話的に確認したくなった場合や、`Ctrl+C` で中断したくなった場合に使用します。
* **例**:

```bash
# 1. バックグラウンドで検索を開始
grep -r "198.51.100.10" /var/log/ > results.txt &

# 2. jobsでジョブ番号を確認 -> [1]
jobs

# 3. fg %1 でgrepプロセスをフォアグラウンドに持ってくる
fg %1
```

### 3. レッドチーム視点

* **タスク**: `tmux` や `screen` が使えない環境で、複数のリスナーやツールを単一のシェルで管理する。
* **組み合わせ**: `fg <job_spec>`
* **解説**: リバースシェル用のリスナー (`nc`) とペイロード配布用のWebサーバー (`python`) などを `Ctrl+Z` と `bg` で管理し、必要に応じて `fg` で特定のジョブを対話的に操作します。
* **例**:

```bash
# Netcatのリスナーを一時停止した後、再度フォアグラウンドで操作したい場合
# jobs -> [1]+ Stopped nc -lvnp 4444
fg %1
```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `bash: fg: current: no such job`
    * **考えられる原因**: フォアグラウンドに戻すべきジョブが存在しません。
    * **解決策**: `jobs` コマンドを実行して、操作対象のジョブが存在するか確認してください。

## 環境変数と設定ファイル

* `fg` はシェルの組み込み機能であり、その動作を直接制御する設定ファイルや環境変数は基本的にありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](OS%20%20-%20Linux%20-%20environment_and_config%20-%20環境変数と設定ファイル.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `fg` 自体はシェルの基本機能であり、脆弱性はありません。リスクは、フォアグラウンドで実行される**コマンドの内容**に依存します。
* **悪用シナリオ**: 攻撃者がバックグラウンドで不審なプロセス（リスナーなど）を実行しているのを管理者が発見した場合、`fg` でフォアグラウンドに戻すことで、そのプロセスがどのような出力をしているかを確認できます（ただし、対話的なマルウェアの場合は意図しない動作を引き起こすリスクもあります）。

### GTFOBins / LOLBAS における利用例

`fg` は基本的なシェル機能のため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: -
* **Detection (検知)**: ライブ調査中に `ps aux` などでプロセスリストを確認するだけでなく、不審なユーザーのシェルで `jobs` コマンドの実行を試み、そのセッション内で隠されているジョブがないかを確認することが有効です。

## 注意点・補足

* **対話性の回復**: `fg` の最大の利点は、ジョブの「対話性」を取り戻せることです。バックグラウンドジョブにはキーボードから入力 (`stdin`) を送れませんが、`fg` でフォアグラウンドに戻せば、再びキーボード入力や `Ctrl+C` での割り込みが可能になります。
