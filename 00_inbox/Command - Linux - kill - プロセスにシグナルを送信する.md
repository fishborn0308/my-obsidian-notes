# Command - Linux - kill - プロセスにシグナルを送信する

## 概要
`kill` は、指定したプロセス ID (PID) に対して特定のシグナルを送ります。デフォルトでは `SIGTERM` (15) を送り、プロセスを安全に終了させようと試みます。プロセスが応答しない場合には、より強力な `SIGKILL` (9) などを使用して強制終了させることも可能です。
(出自: シェルの組み込みコマンド、および /bin/kill バイナリ)



## 類似・関連コマンド
| コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `killall` | プロセス名（例: `nginx`）を指定して、一致する全プロセスを終了させる。 | 名前での一括操作 |
| `pkill` | パターンマッチング（部分一致）でプロセスを特定してシグナルを送る。 | 柔軟な条件指定 |
| `xkill` | GUI 環境で、クリックしたウィンドウのプロセスを終了させる。 | デスクトップ用 |
| `top / htop` | プロセス一覧を見ながら、インタラクティブにシグナルを送る。 | **目視確認しながらの終了** |

## 主要なシグナルの一覧
`kill -l` を実行することで、システムで使用可能なシグナルを確認できます。

| 番号 | 名前 | 説明 | 用途 |
| :--- | :--- | :--- | :--- |
| **1** | **SIGHUP** | ハングアップ。制御端末の終了、または設定の再読み込み。 | **設定の反映** |
| **2** | **SIGINT** | 割り込み（Interrupt）。`Ctrl+C` と同じ。 | 処理の中断 |
| **9** | **SIGKILL** | **強制終了**。プロセスはこれを無視できず、即座に終了する。 | **ハングアップ時の最終手段** |
| **15** | **SIGTERM** | **終了（デフォルト）**。クリーンアップ処理を行ってから終了する。 | **通常の終了推奨** |
| **18** | **SIGCONT** | 一時停止中のプロセスを再開させる。 | 処理の再開 |
| **19** | **SIGSTOP** | プロセスを一時停止させる（無視不可）。 | 処理のデバッグ等 |

## よく使われるオプションと構文
```bash
# 基本形: kill [シグナル] <PID>
````

|**オプション**|**説明**|
|---|---|
|⭐️ `-l`|利用可能なシグナルの一覧（名前と番号）を表示する。|
|⭐️ `-<signal>`|シグナルを番号または名前で指定する（例: `-9` または `-SIGKILL`）。|
|`-s <signal>`|`-<signal>` と同様、シグナルを明示的に指定する。|

## 業務シナリオ別活用法

### 1. インフラ構築・運用視点

- **タスク**: ゾンビプロセスや暴走プロセスの掃除。
    
- **解説**: CPU やメモリを異常に消費して応答しなくなったプロセスを、PID を特定して排除します。
    
- **例**:
    
    Bash
    
    ```
    # PID 1234 のプロセスを強制終了
    kill -9 1234
    ```
    

### 2. ブルーチーム視点

- **タスク**: 不審なプロセスの緊急停止。
    
- **解説**: マルウェアや不正なリバースシェルを検知した際、さらなる被害を防ぐために即座にプロセスを停止させます。その後、実行ファイルやネットワーク接続を調査します。
    

### 3. レッドチーム視点

- **タスク**: セキュリティ対策ソフト（EDR/AV）の無効化。
    
- **解説**: 権限昇格に成功した後、自分の活動を検知されないように監視プロセスを `SIGSTOP` で一時停止させたり、`kill -9` で終了させたりすることを試みます。
    

## エラーメッセージとトラブルシューティング

1. **「kill: (PID) - Operation not permitted」**:
    
    - **原因**: 自分以外のユーザー（特に root）が所有するプロセスにシグナルを送ろうとした。
        
    - **解決策**: `sudo` を使用するか、自分の所有するプロセスのみを対象にする。
        
2. **「kill: (PID) - No such process」**:
    
    - **原因**: 指定した PID が既に終了しているか、存在しない。
        
    - **解決策**: `ps` や `pgrep` で現在の正しい PID を再確認する。
        
3. **「kill -9 を打ってもプロセスが消えない」**:
    
    - **原因**: プロセスが「ゾンビ状態（Z）」であるか、カーネル内での I/O 待ち（D状態）でスリープしている。
        
    - **解決策**: ゾンビプロセスの場合は親プロセスを終了（または再起動）させる必要があります。D状態の場合は I/O の完了を待つしかありません。
        

## セキュリティに関する考慮事項

### 脆弱性とリスク

- **Denial of Service (DoS)**: 攻撃者が一般ユーザー権限でも実行可能な `kill` を悪用し、重要なユーザープロセスを終了させることで業務を妨害するリスクがあります。
    
- **Race Condition**: PID は使い回されるため、特定の PID を殺そうとした瞬間にそのプロセスが終了し、別の重要なプロセスが同じ PID で起動してしまった場合、誤って別プロセスを殺してしまう可能性があります。
    

### 対応策・緩和策 (ブルーチーム視点)

- **特権の分離**: 重要なサービス（Webサーバー等）は専用のユーザーで実行し、一般ユーザーからの `kill` シグナルを受け付けないようにします。
    
- **監視**: `auditd` などを用いて、誰がどのプロセスに対してシグナルを送ったかを記録します。
    

## 注意点・補足

- **SIGTERM の優先**: 可能な限り `kill -15` (SIGTERM) をまず使い、プロセスにデータを保存したり接続を閉じたりする時間を与えてください。いきなり `kill -9` を使うと、データ破損の原因になることがあります。
