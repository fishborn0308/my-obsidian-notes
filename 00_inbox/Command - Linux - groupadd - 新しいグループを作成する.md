# Command - Linux - groupadd - 新しいグループを作成する

## 概要
`groupadd` は、`/etc/group` ファイルに新しいエントリを追加し、システムに新しいグループを定義します。グループを作成することで、複数のユーザーに対して同一のファイルアクセス権限やリソース制限を一括で適用できるようになります。
(出自: `shadow-utils` パッケージに含まれる)



## 類似・関連コマンド
| コマンド | 役割 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `useradd` | 新しいユーザーを作成する（通常、同名のメイングループも作成される）。 | ユーザー作成 |
| `groupmod` | 既存のグループ名や GID（グループID）を変更する。 | グループ情報の修正 |
| `groupdel` | 既存のグループを削除する。 | 不要グループの整理 |
| `gpasswd` | グループのパスワード設定や、メンバーの管理を行う。 | メンバー追加・削除 |

## よく連携されるコマンド
### シナリオ例: 新規プロジェクト用共有ディレクトリの作成 (運用視点)
* **目的**: `devteam` というグループを作り、特定のディレクトリをそのメンバーだけで共有する。
* **手順**:
    1. グループ作成 (`groupadd`)
    2. ユーザーをグループに追加 (`usermod`)
    3. ディレクトリの所有グループ変更 (`chown`)
* **コマンド例**:
    ```bash
    # 1. 'devteam' グループを作成
    sudo groupadd devteam
    # 2. 既存のユーザー 'alice' を 'devteam' に追加
    sudo usermod -aG devteam alice
    # 3. ディレクトリの所有グループを 'devteam' にし、書き込み権限を付与
    sudo chown :devteam /var/www/project
    sudo chmod 775 /var/www/project
    ```

## オプション説明
※実行には **root 権限** が必須です。

| オプション | 説明 |
| :--- | :--- |
| ⭐️ `-g <GID>`, `--gid` | グループID（GID）を数値で直接指定する。指定しない場合は未使用の最小値が割り当てられる。 |
| ⭐️ `-r`, `--system` | **システムグループ**として作成する。通常、500または1000未満の小さい数値の GID が割り当てられる。 |
| `-f`, `--force` | グループが既に存在する場合にエラーを返さず終了する。また、GID重複時の挙動を制御する。 |
| `-p <password>` | 暗号化されたパスワードを直接指定する（非推奨）。 |

## よく使われる業務シナリオ

### 1. インフラ構築・運用視点
* **タスク**: 複数サーバー間での GID 整合性の確保。
* **解説**: NFS（ネットワークファイルシステム）等でファイルを共有する場合、サーバー間で GID が異なると権限エラーが発生します。そのため、ID を明示的に指定して作成します。
* **例**:
    ```bash
    # 全サーバーで共通の GID (9999) を持つグループを作成
    sudo groupadd -g 9999 shared_storage
    ```

### 2. ブルーチーム視点
* **タスク**: 異常なグループ追加の検知。
* **解説**: 攻撃者がバックドアとして「管理者権限に近いグループ（`sudo`, `wheel`, `docker` 等）」を自作したり、既存の重要グループを模倣した名前のグループを作成していないか監視します。
* **例**:
    ```bash
    # 最近追加されたグループを /etc/group で確認
    tail /etc/group
    ```

### 3. レッドチーム視点
* **タスク**: 権限昇格のための欺瞞（Deception）。
* **解説**: 管理者が誤って重要なファイルを配置してしまいそうな、それらしい名前のグループ（例: `backup_admin`）を作成し、そこに自分のアカウントを忍び込ませることで、将来的な情報窃取の足掛かりにします。

## エラーメッセージとトラブルシューティング

1.  **「groupadd: group 'testgroup' already exists」**:
    * **原因**: 既に同名のグループが存在する。
    * **解決策**: `grep testgroup /etc/group` で既存の情報を確認するか、`-f` オプションでエラーを無視する。

2.  **「groupadd: GID '1001' already exists」**:
    * **原因**: 指定した GID が既に他のグループで使用されている。
    * **解決策**: 別の GID を指定するか、`-o` オプション（非推奨）で重複を許容する。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例
* **docker グループの危険性**: `groupadd` で `docker` グループを作成し、そこに一般ユーザーを入れることは、そのユーザーに **root 権限を与えるのと同義** です（Docker コンテナ経由でホストのルートディレクトリをマウントできるため）。グループ設計は慎重に行う必要があります。

### 対応策・緩和策 (ブルーチーム視点)
* **Audit**: `auditd` を使用して、`/etc/group` への書き込みが発生した際にアラートを飛ばすように設定します。
* **Identity Management**: 大規模環境ではローカルの `groupadd` ではなく、LDAP や Active Directory による統合管理を行い、ローカルでの勝手なグループ作成を制限します。

## 注意点・補足
* **GID の範囲**: 一般的に 0-99 は予約済み、100-999 はシステム用、1000 以上が一般ユーザー用として使われます（ディストリビューションにより異なります）。
* **設定ファイル**: デフォルトの挙動は `/etc/login.defs` で定義されています。
