# `last` - 最近のログイン履歴を表示する

## 概要

`last` は、システムにログインしたユーザーの履歴を、新しいものから順に表示するコマンドです。この情報は、システムの利用状況の監査や、不正なログインがないかを確認する際に非常に重要です。`last` は、`/var/log/wtmp` というバイナリファイルに記録されたログイン・ログアウトのセッション情報を読み取って、人間が読みやすい形式で表示します。

(出自: `util-linux` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`who` / `w`** | **現在**ログインしているユーザーの一覧を表示します。`last` は**過去の**ログイン履歴を表示します。 | - |
| **`lastb`** | **失敗した**ログイン試行の履歴を表示します。ブルートフォース攻撃の調査などに使います。`last` は**成功した**ログインを表示します。 | - |
| **`journalctl -u sshd`**| `systemd` 環境で、SSHによるログインに関するより詳細なログ（鍵交換の情報など）を確認できます。 | `journalctl` (詳細分析) |

## よく連携されるコマンド

`last` の出力はテキスト形式なので、`grep` と連携して特定のユーザーやIPアドレスの活動を絞り込むのに便利です。

### シナリオ例: 特定ユーザーのログイン履歴を調査する (ブルーチーム視点)

* **目的**: 侵害された疑いのあるユーザーアカウント (`compromised-user`) のログイン履歴を全て洗い出す。
* **連携コマンド**: `grep`
* **解説**: `last` の出力結果をパイプで `grep` に渡し、ユーザー名でフィルタリングすることで、そのユーザーに関連するセッション情報だけを効率的に抽出できます。
* **コマンド例**:

```bash
# 全てのログイン履歴から 'compromised-user' の行だけを抽出
last | grep '^compromised-user'
```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **表示制御** | |
| ⭐ `-n <number>`, `-<number>`| 表示する行数を `<number>` に制限します。 |
| ⭐ `-i`, `--ip` | ホスト名ではなくIPアドレスで表示します。 |
| `-d`, `--dns` | (リモートログインの場合) IPアドレスをDNSで逆引きしてホスト名に変換して表示します。 |
| `-a`, `--hostlast` | 接続元ホスト名を最後の列に表示します。 |
| **フィルタリング** | |
| ⭐ `<username>` | 特定のユーザーのログイン履歴のみを表示します。 |
| ⭐ `reboot` | システムの再起動履歴 (`system boot`) のみを表示します。 |
| `-f <file>` | デフォルトの `/var/log/wtmp` ではなく、指定したファイルを読み込みます。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: システムが意図せず再起動していないか、再起動の履歴を確認する。
* **組み合わせ**: `last reboot`
* **解説**: システムの安定性を評価する指標として、または計画外の再起動が発生した場合の原因調査の第一歩として、再起動の履歴を一覧表示します。
* **例**:

    ```bash
    last reboot
    # reboot   system boot  5.15.0-101-gene  Mon Sep 22 09:10   still running
    # reboot   system boot  5.15.0-101-gene  Sun Sep 21 18:30 - 20:05  (01:35)
    # -> これで、9/22 09:10に再起動したことが分かる
    ```

### 2. ブルーチーム視点

* **タスク**: 不正アクセスの疑いがあるサーバーのログイン履歴を調査する。
* **組み合わせ**: `last -n 50 -i`
* **解説**: **インシデントレスポンスの基本操作**。直近50件のログイン履歴を、IPアドレス形式で確認します。見慣れないIPアドレスや、深夜などの不審な時間帯からのログインがないかを確認し、攻撃のタイムラインを構築します。
* **例**:

    ```bash
    # last と lastb を比較し、ブルートフォース攻撃が成功した痕跡がないか確認
    sudo lastb -i | head
    last -i | head
    ```

### 3. レッドチーム視点

* **タスク**: 侵入後の偵察として、システムの運用パターンを把握する。
* **組み合わせ**: `last`
* **解説**: 攻撃者は `last` を実行し、どの管理者アカウントが、どのIPアドレス（踏み台サーバーなど）から、どの時間帯にログインしてくるかを把握します。これにより、正規の管理者の活動に紛れて自身の活動を隠蔽したり、活動が手薄になる時間帯を狙ったりします。
* **例**:

    ```bash
    # ログイン履歴から管理者のIPや活動時間を偵察
    last
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **現象**: **`last` を実行しても何も表示されない、または非常に古いログしか表示されない。**
    * **考えられる原因**: `/var/log/wtmp` ファイルが存在しない、空である、または最近クリアされた可能性があります。
    * **解決策**: ログがローテーションされていないか (`/var/log/wtmp.1`など)、あるいは攻撃者によって消去された可能性を疑います。`ls -l /var/log/wtmp` でファイルの作成日時を確認するのも1つの手がかりです。

## 環境変数と設定ファイル

* `last` の動作に直接影響する環境変数や設定ファイルは通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

### 関連する設定ファイル

* **`/var/log/wtmp`**: 成功したログイン・ログアウトの履歴が記録されるバイナリファイル。
* **`/var/log/btmp`**: 失敗したログイン試行の履歴が記録されるバイナリファイル (`lastb` が使用)。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `last` 自体ではなく、**データソースである `wtmp` ファイルの改ざん可能性**がリスクです。
* **悪用シナリオ**: **痕跡消去 (Anti-Forensics)**。root権限を奪取した攻撃者は、`/var/log/wtmp` ファイルを削除または特定のツールで編集し、自身のログイン履歴を `last` コマンドから消し去ろうと試みます。

### GTFOBins / LOLBAS における利用例

`last` は情報表示のみを行うコマンドであり、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: `/var/log/wtmp` ファイルのパーミッションを厳格に管理し、不正な変更を防ぐ。
* **Detection (検知)**: 最も効果的な対策は、`wtmp` を含む全ての監査ログを、`rsyslog` などを通じて**リアルタイムで改ざん不可能なリモートのログサーバー (SIEMなど) に転送する**ことです。これにより、ホスト上でログが消去されても証拠が保全されます。

## 注意点・補足

* **`wtmp begins...`**: `last` の出力の最後にある `wtmp begins ...` という行は、現在 `last` が読み取っている `/var/log/wtmp` ファイルが、いつから記録を開始したかを示しています。
