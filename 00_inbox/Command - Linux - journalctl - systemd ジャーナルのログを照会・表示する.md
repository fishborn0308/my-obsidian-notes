# Command - Linux - journalctl - systemd ジャーナルのログを照会・表示する

## 概要
`journalctl` は、systemd のロギングサービスである `systemd-journald` が収集したログを表示するためのコマンドです。従来の `/var/log/messages` などのテキスト形式のログとは異なり、バイナリ形式で構造化されて保存されているため、時間指定やサービスごとの高速なフィルタリングが可能です。



## 類似・関連コマンド
| コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `tail -f /var/log/syslog` | 従来のテキストログ監視。systemd 非搭載の古いシステム用。 | 互換性重視 |
| `dmesg` | カーネルのリングバッファ（起動時のメッセージ等）のみを表示。 | カーネル・物理層の調査 |
| `logger` | 任意のメッセージをジャーナル（または syslog）に送信する。 | スクリプト内でのログ記録 |
| `journalctl` | サービス、ブート、時刻など、あらゆる条件で横断検索が可能。 | **現代のログ調査の決定版** |

## よく使われるオプション
※システム全体のログを見るには通常 **root 権限** が必要です。

| オプション | 説明 |
| :--- | :--- |
| ⭐️ `-f`, `--follow` | ログの末尾をリアルタイムで監視する（`tail -f` 相当）。 |
| ⭐️ `-u <UNIT>`, `--unit` | 特定のサービス（ユニット）のログのみを表示する。 |
| ⭐️ `-p <PRIO>`, `--priority` | ログレベル（debug, info, notice, warning, err, crit, alert, emerg）を指定。 |
| ⭐️ `-k`, `--dmesg` | カーネルメッセージのみを表示する。 |
| ⭐️ `-b`, `--boot` | 指定したブート時のログを表示する（`-b -1` で前回起動時など）。 |
| `--since`, `--until` | `2026-01-01 12:00:00` や `1h ago` などの形式で期間を指定。 |
| `-n <N>`, `--lines` | 直近の `N` 行のみを表示する。 |

## よく使われる業務シナリオ

### 1. インフラ構築・運用視点
* **タスク**: 起動に失敗したサービスのトラブルシューティング。
* **解説**: エラーメッセージを特定し、失敗の原因（設定ミス、依存関係など）を突き止めます。
* **例**:
    ```bash
    # nginx サービスのエラーログを直近 50 行、詳細表示
    sudo journalctl -u nginx -n 50 -e
    ```

### 2. ブルーチーム視点
* **タスク**: 特定の時間帯に発生した不審なアクセスの調査。
* **解説**: `--since` と `--until` を組み合わせて、インシデント発生時刻前後の全システムの挙動を時系列で追跡します。
* **例**:
    ```bash
    # 今日の午前9時から10時までの警告以上のログを表示
    sudo journalctl --since "09:00" --until "10:00" -p warning
    ```

### 3. レッドチーム視点
* **タスク**: 自分の活動痕跡（Footprints）の確認と消去の検討。
* **解説**: 自分の実行したコマンドやログインがどのように記録されているかを確認します。また、`journalctl --vacuum-time=1s` などを悪用して、ログを物理的に抹消しようと試みます（ただし強力な権限が必要）。

## 便利なテクニック：出力形式の変更
`journalctl` は `-o` オプションで出力形式を自由に変えられます。

* **`-o json-pretty`**: ログの全メタデータを JSON 形式で読みやすく表示。
* **`-o cat`**: タイムスタンプなどのヘッダーを省き、メッセージ本文のみを表示（スクリプト処理に便利）。
* **`-o short-iso`**: タイムスタンプを ISO 8601 形式（YYYY-MM-DD...）で表示。

## セキュリティに関する考慮事項

### 脆弱性とリスク
* **情報の露出**: ログには環境変数やデバッグ出力に含まれるパスワード、トークンなどが記録されることがあります。ジャーナルへのアクセス権限（`adm` や `systemd-journal` グループ）は最小限にすべきです。
* **ログの改ざん**: 攻撃者が root 権限を得た場合、バイナリログ自体を削除したり、一部を書き換えるリスクがあります。

### 対応策・緩和策 (ブルーチーム視点)
* **Forward Secure Sealing (FSS)**: ジャーナルにデジタル署名を施すことで、後からの改ざんを検知可能にします。
* **リモートログ転送**: ログをローカルだけでなく、即座に別のログ管理サーバー（SIEM等）へ転送し、証拠が消されないようにします。

## 注意点・補足
* **ディスク容量**: ジャーナルは設定 (`/etc/systemd/journald.conf`) に基づいてディスク容量を消費します。ディスクが一杯にならないよう、`SystemMaxUse` などの設定を確認しましょう。
* **永続化**: デフォルトで `/var/log/journal` ディレクトリが存在しないディストリビューションでは、再起動するとログが消える「一時モード」になっている場合があります。
