# Command - Linux - ip route - ルーティングテーブルの表示と管理

## 概要
`ip route` オブジェクトは、カーネル内の IPv4/IPv6 ルーティングテーブルを操作します。静的ルートの追加、デフォルトゲートウェイの設定、特定の宛先への経路検索などを行い、パケットがネットワーク内をどのように移動するかを完全に制御します。



## 主要サブコマンド
| サブコマンド | 略称 | 説明 |
| :--- | :--- | :--- |
| **`show`** | `sh`, `s` | ルーティングテーブルを表示する（デフォルト）。 |
| **`add`** | `a` | 新しい経路を追加する。 |
| **`del`** | `d` | 既存の経路を削除する。 |
| **`change` / `replace`** | | 既存の経路を修正、または置き換える。 |
| **`get`** | `g` | 指定した宛先への「実際の経路」をシミュレーションして表示する。 |
| **`flush`** | `f` | 条件に合う経路を一括削除する。 |

## コマンド例と構文

### 1. 経路の確認 (show)
現在システムが知っている「地図」を表示します。
```bash
# 全てのルーティングを表示
ip route show

# 特定のインターフェース（eth0）経由の経路のみ表示
ip route show dev eth0
````

### 2. ルートの追加と削除 (add / del)

外部ネットワークへの出口を指定します。

Bash

```
# デフォルトゲートウェイを設定する (全ての外向き通信の基本)
sudo ip route add default via 192.168.1.1 dev eth0

# 特定のネットワーク (10.0.0.0/8) への経路を追加する
sudo ip route add 10.0.0.0/8 via 192.168.1.254 dev eth0

# 追加した経路を削除する
sudo ip route del 10.0.0.0/8
```

### 3. 経路のシミュレーション (get) ⭐️超便利

「このパケットはどこを通るか？」をカーネルに直接問い合わせます。

Bash

```
# Google (8.8.8.8) へ行く際に使われるインターフェースと経路を確認
ip route get 8.8.8.8
```

## 出力項目の読み解き

- **`default`**: 宛先がどこにも該当しない場合に使われる「最後の手段（ゲートウェイ）」。
    
- **`via`**: パケットを次に渡すべきルーターの IP アドレス。
    
- **`dev`**: パケットを送り出す物理/仮想インターフェース。
    
- **`proto`**: 経路の出自（`kernel` は自動生成、`static` は手動、`boot` は起動時など）。
    
- **`scope`**: 経路の有効範囲（`link` は直接繋がっている、`global` はゲートウェイ経由など）。
    
- **`metric`**: 優先度。数値が小さいほど優先される。
    

## 業務シナリオ別活用法

### 1. インフラ構築・運用視点

- **タスク**: マルチホーム（複数 NIC）環境での通信制御。
    
- **解説**: 二つの異なるネットワークに接続しているサーバーで、社内システム用とインターネット用で出口を使い分けるために静的ルートを設定します。
    

### 2. ブルーチーム視点

- **タスク**: 不審なトラフィックのリダイレクト検知。
    
- **解説**: 攻撃者が勝手にデフォルトゲートウェイを書き換えて、全ての通信を自分のサーバー（中間者）経由にしていないか監視します。
    
- **例**: `ip route` の結果に、意図しない `via` や `metric` の変更がないかチェックします。
    

### 3. レッドチーム視点

- **タスク**: ピボッティング（踏み台攻撃）。
    
- **解説**: 侵入したサーバーが「内部の別ネットワーク」に繋がっている場合、自分の攻撃端末からの通信がその別ネットワークへ届くように、`ip route` を書き換えて通信の「道」を作ります。
    

## セキュリティに関する考慮事項

### 脆弱性とリスク

- **Routing Hijack**: ルーティングテーブルを不正に操作されると、パスワード等を含む全通信が攻撃者の元へ「寄り道」させられる（Man-in-the-Middle）リスクがあります。
    
- **Denial of Service (DoS)**: `ip route flush cache` や `default` 経路の削除により、サーバーをネットワークから完全に孤立させることができます。
    

### 対応策・緩和策 (ブルーチーム視点)

- **RP Filter (Reverse Path Filter)**: 不適切な経路から来たパケットを破棄するようカーネルパラメータを設定し、IP スプーフィングを防ぎます。
    
- **モニタリング**: `ip monitor route` コマンドを使用すると、ルーティングテーブルの変更をリアルタイムで監視・ログ記録できます。
    

## 注意点・補足

- **永続化**: `ip route` コマンドによる変更は**一時的**です。再起動後も有効にするには、`/etc/network/interfaces` や `Netplan` (Ubuntu)、`nmcli` (NetworkManager) での設定が必要です。
