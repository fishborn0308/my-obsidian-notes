# Command - Linux - killall - 名前を指定してプロセスを一括終了する

## 概要
`killall` は、指定した名前に一致するすべてのプロセスにシグナルを送信します。デフォルトでは `SIGTERM` (15) を送信します。PID を指定する `kill` とは対照的に、プロセス名（実行コマンド名）を指定するため、同一名の複数プロセスを一度に操作するのに適しています。
(出自: `psmisc` パッケージに含まれる)



## 類似・関連コマンド
| コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `kill` | PID を直接指定して操作する。 | 特定の1プロセスを狙い撃つ時 |
| `pkill` | 部分一致やユーザー名指定、拡張正規表現が使える。 | **より柔軟なパターンマッチング** |
| `pgrep` | 条件に合うプロセスの PID だけを表示する。 | スクリプト内での PID 取得 |
| `killall` | 正確なプロセス名での一括処理に特化。 | **名前が確定している複数プロセスの一掃** |

## よく連携されるコマンド
### シナリオ例: 特定のユーザーが動かしている特定のプロセスのみを終了 (運用視点)
* **目的**: 共有サーバーで、特定のユーザー `guest` が動かしている `python` プロセスだけをすべて終了させる。
* **解説**: `-u` オプションを使って対象を絞り込みます。
* **コマンド例**:
    ```bash
    # guest ユーザーの python プロセスだけを殺す
    sudo killall -u guest python
    ```

## 主要なオプション
| オプション | 説明 |
| :--- | :--- |
| ⭐️ `-e`, `--exact` | 非常に長いプロセス名の場合、完全一致するものだけを対象にする。 |
| ⭐️ `-i`, `--interactive` | プロセスを終了させる前に、一つずつ確認プロンプトを表示する。 |
| ⭐️ `-I`, `--ignore-case` | プロセス名の大文字・小文字を区別せずにマッチングさせる。 |
| ⭐️ `-u <USER>`, `--user` | 指定したユーザーが所有するプロセスのみを対象にする。 |
| ⭐️ `-v`, `--verbose` | シグナルが正しく送られたかどうかを表示する。 |
| `-w`, `--wait` | すべての対象プロセスが死ぬまで待機し、死んだ後にプロンプトを戻す。 |
| `-s`, `--signal` | `kill` と同様、送信するシグナルを指定する（例: `-s 9`）。 |

## 業務シナリオ別活用法

### 1. インフラ構築・運用視点
* **タスク**: サービスの強制再起動が効かない時のクリーンアップ。
* **解説**: `systemctl restart` でもゾンビのように残ってしまった古いプロセスを、名前を指定して一気に掃除します。
* **例**:
    ```bash
    # 残ってしまった nginx プロセスをすべて強制終了
    sudo killall -9 nginx
    ```

### 2. ブルーチーム視点
* **タスク**: 不正なマイニングプロセスの即時停止。
* **解説**: 侵入者がバックグラウンドで走らせている `xmrig` などのクリプトマイナーを、全コア分まとめて一瞬で停止させます。

### 3. レッドチーム視点
* **タスク**: 競合する攻撃ツールの排除。
* **解説**: 自分が侵入した環境で、他の攻撃者が既にツールを動かしている場合、それらを一括で停止してリソースを独占したり、自分の活動を隠したりします。

## エラーメッセージとトラブルシューティング

1.  **「killall: process_name: no process found」**:
    * **原因**: 指定した名前のプロセスが現在動いていない。または、名前にタイポがある。
    * **解決策**: `ps aux | grep process_name` で正しい実行バイナリ名を確認する。

2.  **「killall: (PID): Operation not permitted」**:
    * **原因**: 自分より高い権限（root など）のプロセスを殺そうとした。
    * **解決策**: `sudo` を付与して実行する。

## セキュリティに関する考慮事項

### 脆弱性とリスク
* **誤爆（False Positive）**: ⭐️ **重要**。`killall` は名前に基づくため、意図しない同名の重要プロセスまで殺してしまうリスクがあります。特に `/tmp/test` と `/home/user/test` が両方 `test` という名前で動いている場合、どちらも対象になります。
* **Race Condition**: `killall` がプロセス名を確認してからシグナルを送るまでのわずかな間に PID が入れ替わる可能性はゼロではありませんが、通常は `kill` ほど深刻ではありません。

### 対応策・緩和策 (ブルーチーム視点)
* **Prevention**: スクリプト内では可能な限り PID を直接扱うか、`-e` (完全一致) オプションを付けて安全性を高めます。
* **Logging**: 誰がどのプロセスを一括終了させたかを、`auditd` 等で監視対象にします。

## 注意点・補足
* **環境による挙動の違い**: Solaris などの一部の古い UNIX システムでは、`killall` コマンドは「システム上の**すべての**プロセスを殺す（シャットダウン用）」という非常に恐ろしい動作をします。Linux 以外の環境で打つときは、マニュアルを必ず確認しましょう。
