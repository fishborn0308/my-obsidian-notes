# Command - Linux - history - コマンドの実行履歴を表示・操作する

## 概要
`history` は、現在のセッションおよび過去のセッションで実行したコマンドのリストを表示します。メモリ上の「ヒストリバッファ」と、ファイル（通常は `~/.bash_history`）を同期させることで、作業の効率化や過去の操作の監査を可能にします。
(出自: シェルの組み込みコマンド (Built-in))



## 類似・関連コマンド
| コマンド | 役割 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `Ctrl+R` | インクリメンタルサーチ。過去のコマンドをキーワードで遡って検索する。 | **高速な再実行** |
| `!!` | 直前に実行したコマンドを再実行する（`sudo !!` などで多用）。 | 直前のミス修正 |
| `!n` | ヒストリ番号 `n` 番のコマンドを実行する。 | 番号指定の実行 |
| `fc` | 過去のコマンドをエディタで開いて編集し、再実行する。 | 複雑なコマンドの修正 |

## よく連携されるコマンド
### シナリオ例: 過去に打った複雑なパイプラインの再発見 (運用視点)
* **目的**: 数日前に実行した、`awk` や `sed` を駆使した複雑なログ解析コマンドをもう一度使いたい。
* **連携**: `grep`
* **解説**: 全履歴の中からキーワードを含むものだけを抽出します。
* **コマンド例**:
    ```bash
    # 過去の履歴から 'awk' を使ったものを検索
    history | grep "awk"
    ```

## オプションとサブコマンド
| オプション | 説明 |
| :--- | :--- |
| ⭐️ `(引数なし)` | 全履歴を表示する。 |
| ⭐️ `-c` | 現在のセッションのヒストリ（メモリ上）をすべて消去する。 |
| ⭐️ `-d <offset>` | 指定した行番号のヒストリを削除する。 |
| ⭐️ `-w` | 現在のメモリ上のヒストリをファイル（`~/.bash_history`）に書き込む。 |
| `-a` | 現在のセッションで追加された分だけをファイルに追記する。 |
| `-n` | まだ読み込まれていない履歴をファイルから読み込む。 |

## よく使われる環境変数（カスタマイズ）
* `HISTSIZE`: メモリ上に保存する履歴の数。
* `HISTFILESIZE`: ファイルに保存する履歴の数。
* `HISTTIMEFORMAT`: ⭐️ `"%F %T "` などを設定すると、実行日時も記録されるようになります。
* `HISTCONTROL`: `ignorespace` を設定すると、先頭にスペースを入れたコマンドを履歴に残さないようになります。

## よく使われる業務シナリオ

### 1. インフラ構築・運用視点
* **タスク**: 作業手順書の作成。
* **解説**: 実際にサーバーを構築した際のコマンドの流れを `history` で出力し、それをベースにドキュメントを作成します。
* **例**:
    ```bash
    # 作業ログとして履歴をファイルに保存
    history > setup_log.txt
    ```

### 2. ブルーチーム視点
* **タスク**: 侵害調査（ポストインシデント解析）。
* **解説**: 攻撃者が侵入後にどのような操作（バックドアの作成、データの窃取など）を行ったかを `~/.bash_history` から読み解きます。
* **例**:
    ```bash
    # 攻撃者の痕跡（wget, curl, chmod など）を履歴から探す
    cat ~/.bash_history | grep -E "wget|curl|chmod|nc"
    ```

### 3. レッドチーム視点
* **タスク**: 活動痕跡の消去（Anti-Forensics）。
* **解説**: 自分の操作が履歴に残らないよう、セッション開始時に `unset HISTFILE` を実行したり、撤退前に `history -c` や `history -w` を駆使して特定行を抹消します。

## エラーメッセージとトラブルシューティング

1.  **「コマンドを打ったのに履歴に残っていない」**:
    * **原因**: `HISTCONTROL=ignoredups`（重複を無視）や `ignorespace` が設定されている。または、複数の端末を開いていて、上書きされてしまった。
    * **解決策**: `shopt -s histappend` を設定して、複数セッションの履歴を追記型にする。

2.  **「日時が表示されない」**:
    * **原因**: デフォルトではコマンド文字列しか保存されないため。
    * **解決策**: `export HISTTIMEFORMAT='%F %T '` を `.bashrc` に追加する。

## セキュリティに関する考慮事項

### 脆弱性とリスク
* **Credential Leakage**: `mysql -u root -p'PASSWORD'` のようにパスワードを引数に含めて実行すると、履歴ファイルにパスワードが平文で保存されます。これは「情報漏洩」の典型的な原因です。

### 対応策・緩和策 (ブルーチーム視点)
* **Prevention**: コマンドの先頭に半角スペースを入れて実行する習慣を推奨する（`HISTCONTROL=ignorespace` 前提）。
* **Detection**: `auditd` などを使って、`~/.bash_history` の改ざんや削除を監視する。
* **Immutable Logs**: 履歴を削除できないよう、リモートの syslog サーバーにコマンド実行履歴をリアルタイムで転送する設定（`PROMPT_COMMAND` の利用など）を検討する。

## 注意点・補足
* **書き込みのタイミング**: 通常、メモリ上の履歴がファイルに書き込まれるのは「シェルを正常に終了した時」です。フリーズして強制終了した場合は、そのセッションの履歴が消えることがあります。
