# `docker ps` - コンテナを一覧表示する

## 概要

`docker ps` は、Dockerホスト上で稼働しているコンテナを一覧表示するコマンドです。デフォルトでは**現在実行中のコンテナのみ**が表示されます。コンテナのID、使用しているイメージ、現在の状態、ポートマッピング、コンテナ名などを素早く確認でき、日々のコンテナ管理やトラブルシューティングにおいて最も頻繁に使用されるコマンドの1つです。

(出自: `docker` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`docker images`** | ローカルに保存されている**イメージ**の一覧を表示します。`docker ps` は、そのイメージから作成された**コンテナ**（プロセスの実体）の一覧を表示します。 | - |
| **`docker stats`** | 実行中のコンテナの**リソース使用状況**(CPU、メモリなど)をリアルタイムで表示します。 | - |
| **`docker container ls`**| `docker ps` の**エイリアス(別名)**です。機能は全く同じです。 | - |
| **`ps` (Linuxコマンド)**| ホストOS上で実行されている**プロセス**を一覧表示します。`docker ps` はDockerによって隔離された**コンテナ**を一覧表示します。 | - |

## よく連携されるコマンド

`docker ps` は、コンテナIDや名前を特定し、その情報を `docker logs`, `docker exec`, `docker stop`, `docker rm` などの他のコマンドに渡すための起点となります。

### シナリオ例: 起動に失敗したコンテナのログを調査する (インフラ構築・運用視点)

* **目的**: 起動直後に終了してしまったコンテナの原因を調査する。
* **連携コマンド**: `docker logs`
* **解説**: まず `-a` オプションで停止中のコンテナも含めて表示し、`Exited` 状態のコンテナのIDまたは名前を特定します。次に、そのIDを `docker logs` に渡して、コンテナが終了する直前に出力したエラーメッセージを確認します。
* **コマンド例**:

    ```bash
    # 1. 停止中のコンテナも含めて一覧表示
    docker ps -a

    # 2. 起動に失敗したコンテナ (例: my-app) のログを確認
    docker logs my-app
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **表示対象** | |
| ⭐ `-a`, `--all` | **停止しているコンテナも含め**、すべてのコンテナを表示します。**非常に重要です**。 |
| **出力制御** | |
| ⭐ `-q`, `--quiet` | コンテナIDのみを表示します。他のコマンドへパイプする際に便利です。 |
| `--format <Go template>`| Go言語のテンプレートを使い、出力形式を自由にカスタマイズします。 |
| **フィルタリング** | |
| ⭐ `--filter <key>=<value>`| 特定の条件で表示するコンテナをフィルタリングします。（例: `--filter "status=exited"`） |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 起動直後に終了してしまったコンテナの原因を調査する。
* **組み合わせ**: `docker ps -a`
* **解説**: `docker run` を実行したのに `docker ps` に表示されない場合に `-a` を付けて実行すれば `Exited` 状態のコンテナが見つかります。そのIDを使って `docker logs` で原因を調査するのが一連の流れです。
* **例**:

    ```bash
    # 終了状態のコンテナIDを全て取得し、一括で削除する
    docker rm $(docker ps -a -q -f "status=exited")
    ```

### 2. ブルーチーム視点

* **タスク**: ホスト上の全コンテナのリストを取得し、監査する。
* **組み合わせ**: `docker ps -a --format "table {{.ID}}\t{{.Image}}\t{{.Status}}\t{{.Names}}"`
* **解説**: `--format` オプションで必要な情報だけを抽出し、監査レポートを作成します。`Exited` 状態のコンテナも含めることで、短時間だけ実行された不審なコンテナの痕跡がないかを確認します。

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **現象**: **`STATUS` カラムが `Exited (137)` となっている。**
    * **考えられる原因**: 終了コード137は、プロセスが `SIGKILL` で強制終了されたことを示します。多くの場合、コンテナがメモリ不足 (OOM Killer) によってホストOSに強制終了させられたことを意味します。
    * **解決策**: コンテナに割り当てるメモリを増やすか、アプリケーションのメモリ使用量を見直す必要があります。

## 環境変数と設定ファイル

* `docker ps` の動作に直接影響する環境変数は通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `docker ps` の出力は**情報漏洩**に繋がります。
* **悪用シナリオ**: コンテナ名、イメージ名、ポートマッピング情報は、攻撃者にとってシステムのアーキテクチャや脆弱性を推測するための重要なヒントとなります。

### GTFOBins / LOLBAS における利用例

`docker ps` は `docker` コマンドの一部として扱われます。`sudo docker` の権限があれば、任意の操作が可能です。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: Dockerソケットへのアクセス権限を厳格に管理する。
* **Detection (検知)**: `docker ps` の実行を `auditd` で監視し、不審なユーザーによる偵察活動がないかを確認する。

## 注意点・補足

* **コンテナ名**: `docker run` 時に `--name` オプションで分かりやすい名前を付けておくと、`docker ps` の出力が見やすくなり、コンテナの識別が容易になります。
