# `git merge` - ブランチの変更を統合する

## 概要

`git merge` コマンドは、指定されたブランチの変更履歴を、現在チェックアウトしているブランチ（カレントブランチ）に統合します。これにより、異なるブランチで並行して行われた作業を1つの履歴にまとめ、プロジェクトの最新状態を更新できます。マージには主に、履歴が直線的に進む**Fast-forwardマージ**と、分岐した履歴を統合するための**マージコミット**を作成する2つの方法があります。

(出自: `git` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`git rebase`** | `rebase` は履歴を**直線的**に書き換えます。`merge` は**マージコミット**を作成し、分岐・統合の履歴をそのまま残します。共有されたブランチには `merge` が安全です。 | `merge` (共有ブランチ) |
| **`git pull`** | `git pull` は `git fetch` と `git merge` の組み合わせです。リモートの変更を自動的に取得してマージします。 | - |
| **`git cherry-pick`**| **単一のコミット**だけを選択的に現在のブランチに適用します。`merge` はブランチ全体の変更を統合します。 | - |

## よく連携されるコマンド

`git merge` は、マージ前にリモートの最新状態を取得する `git fetch` や、マージ後の状態を確認する `git status` と連携して使われます。

### シナリオ例: リモートの更新を手動で安全に統合する (インフラ構築・運用視点)

* **目的**: `git pull` で自動マージされるのを避け、まずリモートの変更内容を確認してから手動で統合する。
* **連携コマンド**: `git fetch`, `git diff`, `git merge`
* **解説**: まず `git fetch` でリモートの変更を取得し、`git diff` でローカルブランチとの差分を確認します。安全だと判断したら、`git merge` でその変更を現在のローカルブランチに統合します。
* **コマンド例**:

    ```bash
    # このコマンドシーケンスで、リモートの変更を安全に取り込む
    # 1. リモートの最新情報を取得 (ローカル作業には影響なし)
    git fetch origin

    # 2. リモートの最新パッチ内容を確認
    git diff main origin/main

    # 3. 安全を確認後、リモートの変更をマージ
    git merge origin/main
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **マージ戦略** | |
| ⭐ `<branch>` | 現在のブランチにマージしたいブランチの名前。 |
| ⭐ `--no-ff` | Fast-forwardマージが可能な場合でも、**必ずマージコミットを作成**します。 |
| `--ff-only` | Fast-forwardマージが可能な場合にのみマージを実行します。 |
| `--squash` | マージするブランチの全コミットを、現在のブランチで**単一のコミットとして統合**します。 |
| **競合解決** | |
| ⭐ `--abort` | 競合発生などで中断されたマージ操作を中止し、マージ開始前の状態に戻します。 |
| `--continue`| 競合解決後に、中断されたマージ操作を再再開します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 新機能開発ブランチを `main` ブランチに統合する際に、その機能がいつ導入されたかを明確なマージコミットとして履歴に残したい。
* **組み合わせ**: `git merge --no-ff <branch>`
* **解説**: `--no-ff` を使うことで、Fast-forwardマージが可能な場合でも必ずマージコミットを作成します。これにより、機能開発の境界が明確になり、後から履歴を追跡する際に役立ちます。
* **例**:

    ```bash
    # mainブランチに切り替え
    git checkout main

    # featureブランチを、マージコミットを作成してマージ
    git merge --no-ff feature/new-ansible-role
    ```

### 2. ブルーチーム視点

* **タスク**: 複数の小さなセキュリティ修正コミットを含むブランチを `main` にマージする際、履歴を煩雑にせず、一連のパッチ適用を1つの論理的な単位として記録したい。
* **組み合わせ**: `git merge --squash <branch>`
* **解説**: `--squash` は、マージするブランチの全コミットを1つの変更セットにまとめ、ステージングします。その後、手動でコミットすることで、クリーンな単一コミットとして履歴に残せます。
* **例**:

    ```bash
    git checkout main
    git merge --squash security-hotfix-branch
    git commit -m "Security: Apply patches for CVE-202X-YYYY"
    ```

### 3. レッドチーム視点

* **タスク**: チーム内で共有する内部ツールセットのリポジトリで、他のメンバーが開発した新しいペイロード作成モジュールを自身の作業ブランチにマージする。
* **組み合わせ**: `git merge <remote-tracking-branch>`
* **解説**: `git fetch` で最新のリモート情報を取得した後、`origin/new-payload-module` のようなリモート追跡ブランチを自身の作業ブランチにマージし、新しいツールを統合します。

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `Automatic merge failed; fix conflicts and then commit the result.`
    * **考えられる原因**: **最もよくあるエラー**。マージしようとしている両方のブランチで、同じファイルの同じ行が異なる方法で変更されたため、Gitが自動的にどちらを採用すべきか判断できません。
    * **解決策**:
        1. `git status` で競合しているファイルを確認します。
        2. 競合ファイルをエディタで開き、`<<<<<<<`, `=======`, `>>>>>>>` のマーカーを参考に手動で編集・解決します。
        3. `git add <file>` で解決したファイルをステージングし、`git commit` でマージを完了させます。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **悪意のあるコードの混入 (Code Injection)**。
* **悪用シナリオ**: コードレビューが不十分な環境で、悪意のある機能ブランチ（バックドア、情報窃取コードなど）がメインブランチにマージされてしまうケース。特にマージ時の競合解決において、攻撃者が意図的に脆弱なコードを残す可能性があります。

### GTFOBins / LOLBAS における利用例

`git merge` は直接的なコード実行機能を持たないため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **プルリクエスト(Pull Request)プロセスを必須化**し、メインブランチへのマージは複数人によるレビューを経て承認されるようにする。GitHubなどで**ブランチ保護ルール**を設定する。
* **Detection (検知)**: CI/CDパイプラインに、静的アプリケーションセキュリティテスト(SAST)や依存関係スキャン(SCA)を組み込み、マージ前に脆弱性を自動的に検出する。

## 注意点・補足

* **競合の解決**: 競合が発生した場合は、焦らず `git status` で状況を確認し、丁寧に競合を解決することが重要です。VS Codeのマージエディタなどを活用すると、視覚的に分かりやすく解決できます。
