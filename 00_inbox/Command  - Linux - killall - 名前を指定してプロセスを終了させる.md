# Command  - Linux - killall - 名前を指定してプロセスを終了させる





## 概要

`killall` は、指定した**コマンド名**に一致する全てのプロセスに対して、シグナルを送信するコマンドです。プロセスID (PID) を1つずつ調べる必要がなく、`nginx` や `httpd` のように同じ名前で複数のワーカープロセスが起動している場合に、それらを一度にまとめて終了させることができます。

`kill` と同様、デフォルトでは穏やかな終了を促す `SIGTERM` (15番) シグナルを送信します。

(出自: `psmisc` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`kill`** | **プロセスID (PID)** を指定して、単一のプロセスにシグナルを送信します。`killall` は**プロセス名**で、該当する**全て**のプロセスを対象にします。 | - |
| **`pkill`** | `killall` と非常に似ていますが、より高機能です。正規表現やフルコマンドライン (`-f`) など、より柔軟な条件でプロセスを検索してシグナルを送信できます。 | `pkill` |
| **`systemctl stop`**| `systemd` で管理されている**サービス**を停止します。サービスの正規の停止方法であり、`killall` よりも安全です。`killall` は、サービスが応答しなくなった際の強制的な手段として使います。 | `systemctl` |

## よく連携されるコマンド

`killall` は、`ps` や `pgrep` でプロセスの存在を確認した後に、それらをまとめて停止させる、という流れで使われます。

### シナリオ例: 特定のユーザーが実行している全プロセスを停止する (ブルーチーム視点)

* **目的**: 侵害されたユーザーアカウントの活動を迅速に封じ込めるため、そのユーザーが実行している全てのプロセスを強制終了させる。
* **連携コマンド**: `ps`, `killall`
* **解説**: まず `ps -u <user>` でユーザーがどのようなプロセスを実行しているかを確認します。その後、`killall -u <user>` でそのユーザーに紐づく全プロセスを一度に停止させます。
* **コマンド例**:

```bash
# 1. 侵害されたユーザー 'attacker' が実行中のプロセスを確認
ps -u attacker

# 2. 'attacker' ユーザーの全プロセスを強制終了 (-9 SIGKILL)
sudo killall -9 -u attacker
```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **シグナル指定** | |
| ⭐ `-<SIGNAL>`, `-s <SIGNAL>`| 送信するシグナルを番号 (`-9`) または名前 (`-s KILL`) で指定します。 |
| **ターゲット指定** | |
| ⭐ `-u`, `--user <user>` | 指定したユーザーが所有するプロセスのみを対象とします。 |
| **対話/出力制御** | |
| `-i`, `--interactive` | プロセスを終了させる前に、対話形式で確認を求めます。 |
| `-v`, `--verbose` | シグナルが正常に送信されたかどうかなどの詳細を表示します。 |
| `-w`, `--wait` | 指定したプロセスが全て終了するまで待機します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 応答しなくなったアプリケーションのプロセスを全て強制終了する。
* **組み合わせ**: `sudo killall -9 <process_name>`
* **解説**: `systemctl restart` コマンドがハングアップし、サービスが正常に終了しない場合などに、関連する全プロセスを強制的に停止させてから、再度サービスを起動します。
* **例**:

```bash
# 応答しなくなった全てのhttpdプロセスを強制終了
sudo killall -9 httpd
```

### 2. ブルーチーム視点

* **タスク**: 侵害されたユーザーアカウントの活動を迅速に封じ込める。
* **組み合わせ**: `sudo killall -u <compromised_user>`
* **解説**: 特定のユーザーが起動した全てのリバースシェルやマイニングツールなどを、名前を特定せずとも一括で停止させ、攻撃者の活動を迅速に無力化します。
* **例**:

```bash
# 'compromised-user' の全プロセスを停止
sudo killall -u compromised-user
```

### 3. レッドチーム視点

* **タスク**: **サービス妨害 (DoS)** または **防御回避 (Defense Evasion)**。
* **組み合わせ**: `sudo killall <critical_service_or_agent>`
* **解説**: root権限を奪取した後、`sshd` や `httpd` などの重要なサービスを `killall` で停止させてサービス妨害を引き起こします。また、`edr-agent` のようなセキュリティ監視エージェントのプロセスを `killall -9` で強制終了させ、検知を逃れて活動します。
* **例**:

```bash
# セキュリティエージェントのプロセスを全て強制終了
sudo killall -9 edr-agent
```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `<process_name>: no process found`
    * **考えられる原因**: 指定した名前のプロセスが存在しないか、コマンドを実行したユーザーにそのプロセスを閲覧・操作する権限がありません。
    * **解決策**: `ps aux | grep <process_name>` でプロセスが存在するか、またどのユーザーで実行されているかを確認してください。他のユーザーのプロセスを対象にする場合は `sudo` が必要です。

## 環境変数と設定ファイル

* `killall` の動作に直接影響する環境変数や設定ファイルは通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `killall` 自体ではなく、**`sudo` での実行権限の不適切な付与**がリスクとなります。
* **悪用シナリオ**: `killall` は複数のプロセスを一度に停止させるため、誤って使用すると広範囲のサービス停止に繋がる可能性があります。攻撃者はこれを意図的なサービス妨害に利用します。

### GTFOBins / LOLBAS における利用例

`killall` はOSの構成を変更するコマンドであり、直接的なコード実行や権限昇格の機能を持たないため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 最小権限の原則に従い、一般ユーザーに `killall` の `sudo` 権限を安易に与えない。
* **Detection (検知)**: `auditd` を使い、`kill` システムコールの実行を監視する。特に、セキュリティエージェントなどの重要なプロセスに対する `SIGKILL` を検知した場合、アラートを発報する。

## 注意点・補足

* **`pkill` の利用**: `pkill` は `killall` よりも新しいコマンドで、より柔軟なプロセス選択が可能です。例えば、`pkill -f "python my_app.py"` のように、コマンド名だけでなく引数を含めたフルコマンドラインでプロセスを特定できます。特別な理由がない限り、現在では `pkill` の方が推奨されることが多いです。
