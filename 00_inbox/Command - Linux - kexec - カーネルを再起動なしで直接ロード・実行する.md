# Command - Linux - kexec - カーネルを再起動なしで直接ロード・実行する

## 概要
`kexec` は、現在実行中の Linux カーネルから、別のカーネル（あるいは同じカーネルの別バージョン）を直接ブートするための仕組みです。BIOS/UEFI やブートローダー (GRUB等) の処理を完全にスキップするため、物理サーバーの再起動時間を劇的に短縮します。
(出自: Linux カーネルのメインラインに含まれる機能)



## 類似・関連コマンド
| コマンド | 役割 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `reboot / shutdown` | 通常の再起動。ハードウェアの初期化からやり直す。 | 一般的な再起動 |
| `kdump` | `kexec` を利用して、クラッシュ時に解析用カーネルを起動する。 | 障害解析の自動化 |
| `grub-reboot` | 次回の起動時に使うカーネルを GRUB に指示する（再起動は必要）。 | 通常のカーネル更新 |
| `kexec` | 現在のメモリを新しいカーネルで上書きして即時ブートする。 | **高速再起動・緊急時** |

## よく使われるオプション
※実行には **root 権限** が必須です。

| オプション | 説明 |
| :--- | :--- |
| ⭐️ `-l <kernel>` | 新しいカーネルをメモリに「ロード」する。 |
| ⭐️ `-e`, `--exec` | ロード済みのカーネルを「実行（ブート）」する。 |
| ⭐️ `-f`, `--force` | 現在のサービス停止などを待たずに、強制的に新しいカーネルへ切り替える。 |
| ⭐️ `-p`, `--load-panic` | カーネルパニック時に備えて、クラッシュダンプ用のカーネルを予約する。 |
| `--initrd=<file>` | 初期 RAM ディスク（initrd/initramfs）を指定する。 |
| `--command-line=<str>` | 新しいカーネルに渡すブートパラメータを指定する。 |

## 業務シナリオ別活用法

### 1. インフラ構築・運用視点
* **タスク**: 物理サーバーの高速パッチ適用。
* **解説**: 何十テラバイトものメモリを搭載した大型サーバーは、BIOS のメモリチェックだけで数十分かかることがあります。`kexec` を使えば、OS レベルの再起動だけで済むため、メンテナンス時間を大幅に短縮できます。
* **例**:
    ```bash
    # 新しいカーネルをロードし、即座に切り替える
    sudo kexec -l /boot/vmlinuz-new --initrd=/boot/initrd.img-new --reuse-cmdline
    sudo kexec -e
    ```

### 2. ブルーチーム視点
* **タスク**: カーネルパニック原因の究明（kdump）。
* **解説**: システムがクラッシュした際、`kexec` によって「第2のカーネル」を起動し、死ぬ直前のメモリ内容をファイル（vmcore）に書き出します。これを解析することで、カーネルパンプの真犯人を特定します。

### 3. レッドチーム視点
* **タスク**: カーネルレベルの永続化とセキュアブートのバイパス。
* **解説**: root 権限を得た後、悪意あるコードを仕込んだ自作カーネルを `kexec` でロードします。正しく設定されていないシステムでは、これにより Secure Boot の検証を回避して、不正なカーネルを動作させ続けられる可能性があります。

## エラーメッセージとトラブルシューティング

1.  **「kexec_load failed: Operation not permitted」**:
    * **原因**: 権限不足、またはカーネル側で `kexec` が禁止されている。
    * **解決策**: `sudo` を使用する。または `sysctl kernel.kexec_load_disabled` が 1 になっていないか確認する。

2.  **「Invalid memory segment」**:
    * **原因**: ロードしようとしているカーネルや initrd のサイズ、配置場所がメモリマップと整合していない。
    * **解決策**: 現在のカーネルと互換性のあるパス、オプションを指定しているか再確認する。

## セキュリティに関する考慮事項

### 脆弱性とリスク
* **Kernel Hijacking**: 攻撃者が `kexec` を自由に使える状態になると、署名のない任意のカーネルをロードして実行されるリスクがあります。これは OS 全体の制御を完全に奪われることを意味します。

### 対応策・緩和策 (ブルーチーム視点)
* **Lockdown Mode**: カーネルの `lockdown` 機能を有効にし、`kexec` による未署名カーネルのロードを制限します。
* **Secure Boot**: UEFI の Secure Boot と連携し、`kexec` で起動するカーネルにも署名検証を強制する設定（`CONFIG_KEXEC_SIG`）を有効にします。

## 注意点・補足
* **デバイスの状態**: ハードウェアの初期化をスキップするため、一部の PCI デバイスなどが「前のカーネルが使っていた状態」のままになり、新しいカーネルで正しく初期化できない稀なケースがあります。
* **データの整合性**: `-f` (force) を使うと、ファイルシステムのアンマウントや同期（sync）を待たずにブートするため、データ破損のリスクがあります。通常はシステムのシャットダウンシーケンスの中で適切に呼び出されます。
