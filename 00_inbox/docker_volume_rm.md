# `docker volume rm` - 1つ以上のDockerボリュームを削除する

## 概要

`docker volume rm` は、指定した1つ以上のDockerボリュームを**完全に削除する**ためのコマンドです。ボリュームを削除すると、そのボリューム内に保存されていた**すべてのデータが恒久的に失われます**。このコマンドは、不要になったアプリケーションのデータを消去したり、テストで使用した一時的なボリュームをクリーンアップしたりする際に使用します。非常に破壊的な操作であるため、実行には細心の注意が必要です。

(出自: `docker` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`docker volume prune`** | **未使用の**（どのコンテナからも参照されていない）ボリュームを**一括で**削除します。`rm` は削除するボリュームを**個別に**指定します。 | `prune` (一括削除) |
| **`docker rm -v`** | コンテナを削除する際に、そのコンテナに紐付いていた**匿名ボリューム**も一緒に削除します。`docker volume rm` は主に**名前付きボリューム**を対象とします。 | - |

## よく連携されるコマンド

`docker volume rm` は、削除対象のボリュームを特定するために `docker volume ls` と連携するのが一般的なワークフローです。

### シナリオ例: 不要になったボリュームのクリーンアップ (インフラ構築・運用視点)

* **目的**: テストプロジェクトが終了し、不要になったボリュームを削除してディスク容量を解放する。
* **連携コマンド**: `docker volume ls`, `docker ps`
* **解説**: まず `docker volume ls` で不要なボリュームを特定します。次に、そのボリュームがどのコンテナからも使用されていないことを確認し、`docker volume rm` で削除します。
* **コマンド例**:

    ```bash
    # 1. 不要なボリューム 'old-project-db' を特定
    docker volume ls

    # 2. そのボリュームを使用しているコンテナがないことを確認
    docker ps -a --filter volume=old-project-db

    # 3. ボリュームを削除
    docker volume rm old-project-db
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **削除対象** | |
| ⭐ `<volume...>` | 削除するボリュームの名前を一つ以上、スペース区切りで指定します。 |
| **動作制御** | |
| `-f`, `--force` | 対話形式の確認プロンプトを表示せず、強制的にボリュームを削除します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: プロジェクト終了に伴い、不要になったアプリケーションのデータボリュームを削除する。
* **組み合わせ**: `docker volume rm <volume_name>`
* **解説**: 上記シナリオ例の通り、不要になったボリュームを個別に削除し、ディスク容量を解放します。

### 2. ブルーチーム視点

* **タスク**: 証拠保全が完了した後、マルウェアなどが格納されていた隔離済みのボリュームを完全に削除する。
* **組み合わせ**: `docker volume rm <quarantined_volume>`
* **解説**: インシデント調査完了後、マルウェアの残骸や攻撃者が残したファイルがシステムに残り続けないように、該当のボリュームを `rm` で確実に削除し、クリーンな状態に戻します。

### 3. レッドチーム視点

* **タスク**: **破壊活動 (Sabotage)** または**サービス妨害 (DoS)** の一環として、重要なデータを格納しているボリュームを削除する。
* **組み合わせ**: `docker volume rm <critical_data_volume>`
* **解説**: 攻撃者がシステムのroot権限を奪取した後、`prod-database-data` のような名前のボリュームを意図的に削除します。これにより、アプリケーションはデータを失い、サービス停止に追い込まれます。

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `Error response from daemon: remove <volume_name>: volume is in use by [<container_id>]`
    * **考えられる原因**: **最もよくあるエラー**。削除しようとしているボリュームが、まだ存在するコンテナ（実行中または停止中）によって使用されています。
    * **解決策**:
        1. エラーメッセージに表示されたコンテナIDを `docker rm` で先に削除します。（もしコンテナが実行中なら `docker stop` が先に必要です）
        2. その後、再度 `docker volume rm <volume_name>` を実行します。

## 環境変数と設定ファイル

* `docker volume rm` の動作に直接影響する環境変数は通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `docker volume rm` はDocker環境における**最も破壊的なコマンドの一つ**です。
* **悪用シナリオ**:
  * **データ損失**: バックアップがない状態で本番環境のデータボリュームを誤って削除すると、復旧不可能な大惨事となります。
  * **サービス妨害**: レッドチームのシナリオで述べたように、重要なデータボリュームを削除することは、深刻なサービス妨害攻撃となります。

### GTFOBins / LOLBAS における利用例

`docker volume rm` は `docker` コマンドの一部として扱われます。`sudo docker` の権限があれば、任意のボリュームを削除可能です。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: Dockerソケットへのアクセス権限を厳格に管理する。本番環境でのボリューム削除は、変更管理プロセスを通じて計画的に行う。
* **Detection (検知)**: `auditd` で `docker volume rm` コマンドの実行を監視し、計画外のボリューム削除がないかを確認する。

## 注意点・補足

* **安全装置**: Dockerが「使用中のボリューム」を削除できないようにしているのは、意図しないデータ損失を防ぐための重要な安全機能です。
