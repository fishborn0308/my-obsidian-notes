# Command  - Linux - bg - ジョブをバックグラウンドで再開する

## 概要

`bg` (background) は、シェルのジョブ管理機能の一部であり、**一時停止 (suspend) 状態のジョブをバックグラウンドで実行再開させる**ためのシェル組み込みコマンドです。時間のかかる処理をターミナルで開始したものの、その処理を中断せず、かつ同じターミナルで別の作業を続けたい場合に利用します。`fg`, `jobs`, `Ctrl+Z` と密接に関連しています。

(出自: `bash`, `zsh` などのシェルの組み込み機能)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`fg`** | バックグラウンドまたは一時停止中のジョブを**フォアグラウンド**に切り替えます。`bg` とは対になるコマンドです。 | - |
| **`&` (アンパサンド)**| コマンドの**実行開始時**に末尾につけることで、直接バックグラウンドジョブとして起動します。`bg` は**すでに一時停止しているジョブ**を後からバックグラウンドに移動させます。 | `&` (最初から) |
| **`jobs`** | `bg` や `fg` で操作対象とするジョブ番号 (`%1`など) を確認するためのコマンドです。 | - |
| **`nohup`** | ログアウトしてもプロセスが終了しないようにします。`bg` はログアウトするとジョブが終了します。`nohup command &` のように組み合わせて使います。| `tmux`/`screen` |

## よく連携されるコマンド

`bg` の利用は、一連のジョブ管理操作の中で行われます。

### シナリオ例: 時間のかかる処理をバックグラウンドに回す (インフラ構築・運用視点)

* **目的**: 時間のかかるファイルのコピー処理を開始した後、その処理をバックグラウンドで続行させつつ、別のコマンド操作を行いたい。
* **連携コマンド**: `Ctrl+Z`, `jobs`
* **解説**: フォアグラウンドで実行したコマンドを `Ctrl+Z` で一時停止し、`jobs` で状態を確認した後、`bg` でバックグラウンドで再開させる、という一連の流れです。
* **コマンド例**:

		```bash
    # 1. 時間のかかるコマンドを実行

    rsync -a /large/data/ /backup/

    # 2. Ctrl+Z を押して一時停止
    # [1]+  Stopped                 rsync -a /large/data/ /backup/

    # 3. bg コマンドでバックグラウンドで再開

    bg

    # [1]+ rsync -a /large/data/ /backup/ &

    # 4. これで別の作業が可能になる
    ```


## オプション説明

`bg` コマンド自体にオプションはありませんが、引数として操作対象の「ジョブスペック」を指定します。

| 引数 (ジョブスペック) | 説明 |
| :--- | :--- |
| **ジョブ指定** | |
| ⭐ (引数なし) | 最後に一時停止されたジョブ (`%` または `%%` と同義) を対象とします。 |
| `%n` | `jobs` コマンドで表示されるジョブ番号 `n` のジョブを対象とします。(例: `%1`) |
| `%string` | コマンド名が `string` で始まるジョブを対象とします。(例: `%sle`) |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 大規模なデータ処理中に、急遽別の設定ファイルを確認・編集する。
* **組み合わせ**: `bg`
* **解説**: サーバ管理者が単一のSSHセッションで複数の作業を効率的にこなすための基本的なテクニックです。ターミナルを多重化する `tmux` や `screen` が使えない環境でも役立ちます。
* **例**:

		```bash
    # 1. データ処理スクリプトを実行

    ./long-running-script.sh

    # 2. Ctrl+Z で一時停止
    # 3. bg でバックグラウンド実行に切り替え

    bg

    # 4. 別の作業を開始

    vi /etc/nginx/nginx.conf

    ```


### 2. ブルーチーム視点

* **タスク**: 時間のかかるログ検索をバックグラウンドで続行しつつ、並行して他の調査を行う。
* **組み合わせ**: `bg <job_id>`
* **解説**: インシデントレスポンスにおいて、時間のかかる調査と並行して、リアルタイムの状況分析を行う際に有効です。限られた時間とコンソールで効率的に調査を進めることができます。
* **例**:

		```bash
    # 1. 巨大なログファイルを検索

    grep "198.51.100.10" /var/log/secure* > suspicious_logins.txt

    # 2. Ctrl+Z で一時停止
    # 3. bg %1 でバックグラウンドで検索を続行

    bg %1

    # 4. ps aux や ss -anp などで他の調査を開始

    ps aux

    ```


### 3. レッドチーム視点

* **タスク**: 侵入したサーバで、リバースシェルのリスナーをバックグラウンドで稼働させつつ、偵察活動を行う。
* **組み合わせ**: `bg`
* **解説**: `tmux` や `screen` がインストールされていない環境での活動において、セッションを失うことなく複数のタスクを維持するための手段となります。
* **例**:

		```bash
    # 1. Netcatでリスナーを起動

    nc -lvnp 4444

    # 2. Ctrl+Z で一時停止
    # 3. bg でリスナーをバックグラウンドで稼働

    bg

    # 4. find / -perm -4000 2>/dev/null などで権限昇格の調査を開始

    find / -perm -4000 2>/dev/null

    ```


## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `bash: bg: job not found: %1`
		* **考えられる原因**: 指定したジョブ番号が存在しません。すでに終了しているか、番号を間違えています。
		* **解決策**: `jobs` コマンドを実行して、現在有効なジョブ番号を確認してください。

## 環境変数と設定ファイル

* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。
`bg` はシェルの組み込み機能であり、その動作を直接制御する設定ファイルや環境変数は基本的にありません。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `bg` 自体はシェルの基本機能であり、脆弱性はありません。
* **悪用シナリオ**: `bg` 自体は攻撃ツールではありませんが、攻撃者が活動を隠蔽・維持するための補助的なコマンドとして利用されます。リバースシェルのリスナーやバックドアのプロセスをバックグラウンドで動かし続けることで、攻撃の足がかりを維持します。

### GTFOBins / LOLBAS における利用例

`bg` は基本的なシェル機能のため、GTFOBinsにはリストされていません。しかし、リストされている多くのコマンド（`nc`, `socat` など）をバックグラウンドで実行する際に、補助的に利用されます。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 不要なユーザーにシェルアクセスを許可しない。
* **Detection (検知)**: ライブ調査中に `ps aux` などでプロセスリストを確認するだけでなく、不審なユーザーのシェルで `jobs` コマンドの実行を試み、そのセッション内で隠されているジョブがないかを確認することが有効です。

## 注意点・補足

* **出力の混在**: `bg` でバックグラウンドに送ったジョブは、標準出力と標準エラー出力を依然としてターミナルに送ってきます。これにより、フォアグラウンドで作業している最中に画面が乱れることがあります。
* **ログアウトとジョブの終了**: `bg` で実行したジョブは、そのシェルセッションに紐付いています。そのため、**シェルをログアウトすると、通常はそのジョブも終了します**。ログアウト後もプロセスを維持したい場合は、`nohup` や `disown` と組み合わせる必要があります。
