# Command - Linux - apt-get - パッケージの取得・インストール・更新を行う

## 概要
`apt-get` は、APT (Advanced Package Tool) ライブラリを利用してパッケージの管理を行うコマンドラインツールです。インターネット上のリポジトリから最新のパッケージ情報を取得し、依存関係を自動的に計算してインストールやアップグレードを行います。
(出自: Debian プロジェクト)



## 類似・関連コマンド
| コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `apt` | `apt-get` や `apt-cache` の主要機能を統合したモダンなツール。進捗バーが表示されるなど人間向け。 | **日常の対話操作** |
| `dpkg` | ローカルの `.deb` ファイルを直接操作する低レベルツール。依存関係の解決は行わない。 | ローカルパッケージ操作 |
| `apt-cache` | パッケージの検索や詳細情報の表示に特化したツール（`apt-get` はこれらを行わない）。 | パッケージ情報の検索 |
| `apt-get` | 出力が安定しており、歴史が長く信頼性が高い。 | **シェルスクリプト・自動化** |

## よく連携されるコマンド
### シナリオ例: システムの最新化と不要ファイルの掃除 (運用視点)
* **目的**: セキュリティパッチを適用し、不要になった依存パッケージを削除してディスクを節約する。
* **連携**: `&&` を使った連続実行。
* **解説**: 前の工程が成功したときのみ次へ進みます。
* **コマンド例**:
    ```bash
    # リポジトリ情報の更新、パッケージのアップグレード、不要パッケージの削除を一気に実行
    sudo apt-get update && sudo apt-get upgrade -y && sudo apt-get autoremove -y
    ```

## 主要なサブコマンド
※実行には原則として **root 権限** が必須です。

| サブコマンド | 説明 |
| :--- | :--- |
| ⭐️ `update` | リポジトリ（ソースリスト）から最新のパッケージリストを同期する。 |
| ⭐️ `upgrade` | インストール済みパッケージを最新バージョンに更新する（新しいパッケージの追加/削除は行わない）。 |
| ⭐️ `dist-upgrade` | 依存関係の変化に対応しつつ更新する。必要に応じてパッケージの追加・削除も行う。 |
| ⭐️ `install <pkg>` | 指定したパッケージをインストールする。 |
| ⭐️ `remove <pkg>` | パッケージを削除する（設定ファイルは残る）。 |
| ⭐️ `purge <pkg>` | 設定ファイルも含めてパッケージを完全に削除する。 |
| `autoremove` | 他のパッケージの依存関係としてインストールされ、現在は不要になったパッケージを削除する。 |
| `clean` | ダウンロードしたパッケージのアーカイブファイル（.deb）を削除してディスクを空ける。 |

## 主要なオプション
| オプション | 説明 |
| :--- | :--- |
| ⭐️ `-y`, `--yes` | 全ての確認プロンプトに自動的に "yes" と答える（自動化に必須）。 |
| `-d`, `--download-only` | インストールはせず、パッケージファイルのダウンロードのみ行う。 |
| `-s`, `--simulate` | 実際には何も変更せず、実行される内容のシミュレーションのみ行う。 |
| `--no-install-recommends` | 「推奨」されるパッケージをインストールせず、必須の依存関係のみ入れる（軽量化）。 |

## よく使われる業務シナリオ

### 1. インフラ構築・運用視点
* **タスク**: セキュリティアップデートの自動化。
* **解説**: 特定のパッケージ（例: `nginx`）だけを脆弱性対策のために更新します。
* **例**:
    ```bash
    # 特定のパッケージのみを最新にする
    sudo apt-get install --only-upgrade nginx
    ```

### 2. ブルーチーム視点
* **タスク**: 侵害されたシステムの調査と、不正なリポジトリの確認。
* **解説**: 攻撃者が不正なパッケージを流し込むために `/etc/apt/sources.list` に怪しいリポジトリを追加していないか、あるいは `apt-get` の履歴から怪しいツールが入れられていないかを確認します。
* **例**:
    ```bash
    # apt の実行履歴ログを確認
    cat /var/log/apt/history.log
    ```

### 3. レッドチーム視点
* **タスク**: 侵入後のツールセット（Recon用など）の迅速な展開。
* **解説**: ターゲット環境で利用可能な `apt-get` を使い、`nmap` や `socat`、`tcpdump` などを最小限のパッケージで導入します。
* **例**:
    ```bash
    # 推奨パッケージを省いて、必要なツールだけを最小構成でインストール
    sudo apt-get install -y --no-install-recommends nmap netcat-traditional
    ```

## エラーメッセージとトラブルシューティング

1.  **「E: Could not get lock /var/lib/dpkg/lock-frontend」**:
    * **原因**: 他のパッケージ管理プロセス（自動更新や別の apt 実行）が動いている。
    * **解決策**: プロセスの終了を待つか、`ps aux | grep apt` で確認。

2.  **「E: Unable to locate package ...」**:
    * **原因**: 指定したパッケージ名が間違っているか、`apt-get update` を忘れてリストが古い。
    * **解決策**: `apt-get update` を実行し、パッケージ名のタイポを確認。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例
* **Malicious Repositories**: 信頼できない第三者のリポジトリを追加すると、正規のパッケージを装ったマルウェア（バックドア入りバイナリ）を `apt-get upgrade` 時に自動で掴まされるリスクがあります。
* **Proxy 経由の攻撃**: `apt` の設定で不審なプロキシが設定されていると、通信内容の傍受や改ざんが行われる可能性があります。

### GTFOBins における利用例
`apt-get` に `sudo` 権限がある場合、特定のフラグを介して任意のシェルを起動したり、パッケージのインストールスクリプトを利用して root を奪取したりすることが可能です。
* **機能**: `Sudo`, `Interactive Shell`
* **コマンド例**:
    ```bash
    # sudo apt-get を使った特権昇格 (Changelogのページャーを悪用)
    sudo apt-get changelog apt
    # (ページャーが開いたら) !/bin/sh と入力して root シェルへ
    ```
* **参照**: `https://gtfobins.github.io/gtfobins/apt-get/`

### 対応策・緩和策 (ブルーチーム視点)
* **Prevention**: `/etc/apt/sources.list` および `sources.list.d/` の書き込み権限を root に限定し、変更を監視する。
* **Integrity**: 署名（GPG key）のないリポジトリからのインストールを許可しない。

## 注意点・補足
* **`apt` との使い分け**: インタラクティブな操作には `apt` を使い、CI/CD パイプラインやシェルスクリプトには、出力形式が安定している `apt-get` を使うのが業界のベストプラクティスです。
