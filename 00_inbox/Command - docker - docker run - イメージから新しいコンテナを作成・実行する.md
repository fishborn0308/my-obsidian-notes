# Command - docker - docker run - イメージから新しいコンテナを作成・実行する

## 概要

`docker run` は、指定したDockerイメージを基に新しいコンテナを作成し、その中でコマンドを実行するための中心的なコマンドです。このコマンド1つで、コンテナの作成、起動、そしてオプションによってはターミナルのアタッチまでを行います。Dockerのワークフローにおいて、アプリケーションやツールを実際に動かし始めるための起点となります。

`docker run` は内部的に `docker create`(作成)と `docker start`(起動)を組み合わせた動作をします。

(出自: `docker` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`docker start`** | **すでに作成済みで、停止している**コンテナを起動します。`run` は**新しいコンテナを作成して**起動する点が異なります。 | - |
| **`docker exec`** | **すでに実行中のコンテナ**の内部で、新しいコマンドを実行します。デバッグなどで後からコンテナに入る際に使います。 | - |
| **`docker compose up`**| `docker-compose.yml` に定義された**複数のコンテナ**をまとめて作成・起動します。`run` は原則として**単一のコンテナ**を対象とします。 | - |

## よく連携されるコマンド

`docker run` は、作成したコンテナの状態を確認する `docker ps` や、ログを確認する `docker logs` といったコマンドと一連のワークフローで連携します。

### シナリオ例: 開発用の一時的な環境を起動する (インフラ構築・運用視点)

* **目的**: ローカルマシンに開発環境をインストールすることなく、一時的なコンテナ内でアプリケーションの依存関係のインストール (`npm install`) やテストを行う。
* **連携コマンド**: `docker run` (単体)
* **解説**: `-it` で対話的に、`--rm` で終了後に自動削除されるコンテナを起動し、`-v` でローカルのソースコードをマウントします。
* **コマンド例**:

		```bash
		# このコマンドで、Node.jsの開発環境を一時的に立ち上げる

		docker run --rm -it -v $(pwd):/app -w /app node:18 bash

		```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **実行モード** | |
| ⭐ `-d`, `--detach` | **デタッチモード**。コンテナをバックグラウンドで実行し、コンテナIDを出力します。 |
| ⭐ `-it` | `-i` (interactive) と `-t` (tty) の組み合わせ。コンテナの標準入力と対話し、擬似ターミナルを割り当てます。 |
| ⭐ `--rm` | コンテナが終了した際に、自動的にコンテナを削除します。一時的なタスクの実行に非常に便利です。 |
| **命名とネットワーク** | |
| ⭐ `--name <name>` | コンテナに人間が識別しやすい名前を付けます。 |
| ⭐ `-p`, `--publish <host>:<cont>`| コンテナのポートをホストのポートに公開(マッピング)します。 |
| `--network <network>`| コンテナを接続するネットワークを指定します。 |
| **データ管理と環境** | |
| ⭐ `-v`, `--volume <src>:<dest>`| ホストのディレクトリや名前付きボリュームをコンテナ内にマウントします。データの永続化に必須です。 |
| ⭐ `-e`, `--env <KEY=VALUE>`| コンテナ内に環境変数を設定します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: Webサーバーをバックグラウンドで起動し、ポートを公開する。
* **組み合わせ**: `docker run -d -p 8080:80 --name my-nginx nginx:latest`
* **解説**: Nginxコンテナをバックグラウンド(`-d`)で起動し、ホストのポート8080をコンテナのポート80にマッピング(`-p`)する、最も基本的なパターンです。
* **例**: -

### 2. ブルーチーム視点

* **タスク**: 不審なイメージを安全なサンドボックス環境で実行し、静的解析を行う。
* **組み合わせ**: `docker run --rm --network none --cap-drop ALL <suspicious_image>`
* **解説**: ネットワークを無効化 (`--network none`) し、全てのカーネルケーパビリティを剥奪 (`--cap-drop ALL`) して、イメージを隔離された状態で実行します。これにより、安全にコンテナ内に入り (`exec`)、ファイルシステムを調査できます。
* **例**: -

### 3. レッドチーム視点

* **タスク**: **コンテナエスケープによる権限昇格**。
* **組み合わせ**: `docker run --rm -it -v /:/mnt/host --privileged ubuntu`
* **解説**: `--privileged` フラグで特権モードを有効にし、ホストのルートディレクトリ (`/`) をコンテナ内にマウントします。Dockerソケットへのアクセス権を奪取した場合、このコマンドでホストの全ファイルシステムにroot権限でアクセスでき、ホストを完全に掌握できます。

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `docker: Error response from daemon: ... Bind for 0.0.0.0:8080 failed: port is already allocated.`
		* **考えられる原因**: `-p` で指定したホスト側のポートが、すでに他のプロセスによって使用されています。
		* **解決策**: `sudo lsof -i :8080` などでポートを使用中のプロセスを特定・停止するか、`docker run` で別のホストポートを指定します。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **不適切なオプションの使用**。特に `--privileged` や `-v /:/host` のようなオプションは、コンテナの隔離を破壊し、ホストのセキュリティを著しく低下させます。
* **悪用シナリオ**: Dockerソケット (`/var/run/docker.sock`) へのアクセス権は、実質的にホストのroot権限と同じです。攻撃者はこの権限を悪用して、特権コンテナを起動し、ホストを乗っ取ります。

### GTFOBins / LOLBAS における利用例

* **機能**: **Privilege Escalation**
* **解説**: `sudo` ルールなどで `docker` コマンドの実行が許可されているユーザーは、`docker run` を使って即座にホストのrootシェルを取得できます。
* **参照**: `https://gtfobins.github.io/gtfobins/docker/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **`docker` グループへのユーザー追加は、そのユーザーにroot権限を与えることと実質的に同義とみなし、慎重に行う**。
* **Detection (検知)**: `auditd` でDockerソケットへのアクセスや、`docker run` コマンドの実行（特に `--privileged` フラグ付き）を監視する。

## 注意点・補足

* **`--rm` の活用**: 一時的なコマンド実行やテストには `--rm` を付ける習慣をつけると、不要なコンテナがホストに溜まるのを防げます。
