# `chmod` - ファイルやディレクトリのパーミッションを変更する

## 概要

`chmod` (change mode) は、ファイルやディレクトリのアクセス権限（パーミッション）を変更するためのコマンドです。Linux/Unixシステムでは、各ファイルに対して「**ユーザー (user)**」「**グループ (group)**」「**その他 (others)**」の3つのカテゴリごとに、「**読み取り (read)**」「**書き込み (write)**」「**実行 (execute)**」の権限が設定されています。`chmod` は、これらの権限を精密に制御することで、システムのセキュリティを維持する上で極めて重要な役割を果たします。

(出自: `coreutils` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`chown`** | ファイルの**所有者**を変更します。`chmod` は所有者が**何ができるか（権限）**を変更します。 | - |
| **`umask`** | **これから作成される**ファイルやディレクトリの**デフォルトのパーミッション**を決定します。`chmod` は**既存の**ファイルのパーミッションを変更します。 | - |

## よく連携されるコマンド

`chmod` は、スクリプトを実行可能にしたり、共有ディレクトリの権限を設定したりする際に、他のファイル操作コマンドと連携して使われます。

### シナリオ例: 共有ディレクトリの権限を安全に設定する (インフラ構築・運用視点)

* **目的**: 特定のグループのメンバーだけが読み書きできる共有ディレクトリを作成する。
* **連携コマンド**: `mkdir`, `chgrp`, `chmod`
* **解説**: ディレクトリを作成し、グループ所有者を設定した後、`chmod` で適切なパーミッション（この場合は `770`）と `SGID` ビット (`2`) を設定します。SGIDビットにより、このディレクトリ内で作成されたファイルは親ディレクトリのグループを継承するようになります。
* **コマンド例**:

    ```bash
    # このコマンドシーケンスで共有ディレクトリをセットアップする
    sudo mkdir /srv/shared
    sudo chgrp developers /srv/shared
    sudo chmod 2770 /srv/shared
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **モード指定** | |
| ⭐ `<mode>` | シンボリックモード (`u+x`) または八進数モード (`755`) で変更後のパーミッションを指定します。 |
| **ファイル/ディレクトリ操作** | |
| ⭐ `-R`, `--recursive` | ディレクトリとその中にある全てのファイル・サブディレクトリのパーミッションを**再帰的に**変更します。 |
| **その他** | |
| `--reference=<RFILE>`| パーミッションを直接指定する代わりに、参照ファイル `<RFILE>` と同じパーミッションに変更します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: シェルスクリプトに実行権限を付与する。
* **組み合わせ**: `chmod +x <script.sh>`
* **解説**: シンボリックモードを使い、スクリプトファイルに実行権限 (`x`) を付与する、最も基本的な操作です。
* **例**:

    ```bash
    # backup.sh というスクリプトを実行可能にする
    chmod +x backup.sh
    ./backup.sh
    ```

* **タスク**: SSHの秘密鍵ファイルのパーミッションを厳格にする。
* **組み合わせ**: `chmod 600 <key_file>`
* **解説**: 八進数モードを使い、SSHの秘密鍵を所有者のみが読み書きできるように設定します。これはSSH接続のための必須のセキュリティ設定です。
* **例**:

    ```bash
    # SSH秘密鍵のパーミッションを設定
    chmod 600 ~/.ssh/id_rsa
    ```

### 2. ブルーチーム視点

* **タスク**: マルウェアを無力化する。
* **組み合わせ**: `chmod 000 /path/to/malware`
* **解説**: インシデントレスポンス中に発見したマルウェアや不審な実行ファイルに対し、全ての権限を剥奪 (`000`) することで、そのファイルが誤って実行されたり、読み取られたりするのを防ぐ封じ込め措置として実行します。
* **例**:

    ```bash
    # 不審な実行ファイルの権限を全て剥奪
    sudo chmod 000 /tmp/evil.elf
    ```

### 3. レッドチーム視点

* **タスク**: 権限昇格のためのバックドアを作成する。
* **組み合わせ**: `chmod u+s <backdoor_binary>`
* **解説**: **SUID (Set User ID) ビット**を設定します。攻撃者がroot権限を奪取した後、`/bin/bash` のコピーなどにSUIDビットを設定することで、一般ユーザーに戻った後でも、そのバイナリを実行すればroot権限のシェルを再度取得できる永続化手法です。
* **例**:

    ```bash
    # bashのコピーにSUIDビットを設定する (root権限で実行)
    sudo cp /bin/bash /tmp/root-shell
    sudo chown root:root /tmp/root-shell
    sudo chmod u+s /tmp/root-shell
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `chmod: changing permissions of '<file>': Operation not permitted`
    * **考えられる原因**: ファイルのパーミッションを変更できるのは、そのファイルの所有者または `root` ユーザーのみです。
    * **解決策**: `sudo` を付けてコマンドを実行してください。

## 環境変数と設定ファイル

* `chmod` の動作に直接影響する環境変数や設定ファイルは通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **SUID/SGIDビット**。`root` が所有する実行ファイルにSUIDビットが設定されていると、一般ユーザーが実行しても `root` 権限で動作します。このプログラムに脆弱性があれば、即座にシステム全体の侵害に繋がります。
* **悪用シナリオ**: 攻撃者はシステムに侵入後、`find / -perm /4000` を実行してSUIDが設定されたファイルを探し、権限昇格の足がかりとします。

### GTFOBins / LOLBAS における利用例

`chmod` 自体はリストされていませんが、`chmod u+s` を使って**SUIDビットを設定する行為**が、GTFOBinsにリストされている多くの権限昇格攻撃の**前提条件**となります。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: `/tmp` のようなユーザーが書き込めるパーティションは `nosuid` オプション付きでマウントする。不要なSUIDビットは `chmod u-s <file>` で削除する。
* **Detection (検知)**: `find / -perm /6000` でSUID/SGIDが設定されたファイルを定期的に監査し、ベースラインと比較して意図しない変更がないか監視する。

## 注意点・補足

* **`-R` (再帰) オプションの注意**: `chmod -R 755 /somedir` は便利ですが、ディレクトリとファイルで適切なパーミッションは異なるため、意図しない権限を設定してしまう可能性があります。`find` コマンドと組み合わせて、ファイルとディレクトリを分けて設定する方が安全です。

  ```bash
  find /somedir -type d -exec chmod 755 {} \;  # ディレクトリは755
  find /somedir -type f -exec chmod 644 {} \;  # ファイルは644
