# `shutdown` - システムを安全に停止・再起動する

## 概要

`shutdown` は、システムを**安全に**シャットダウンまたは再起動するためのコマンドです。このコマンドを実行すると、まず `wall` コマンドを通じて、ログインしている全てのユーザーにシステムが停止する旨の警告メッセージが送信されます。その後、全てのプロセスに `SIGTERM` シグナルを送って穏やかな終了を促し、ファイルシステムの同期を取った上で、システムを停止・再起動します。

単に電源を落とすだけでなく、こうした丁寧な終了処理を行うことで、データの破損を防ぎ、システムをクリーンな状態で終了させることができます。

(出自: `systemd-sysv` または `sysvinit-utils` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`systemctl`** | `systemctl poweroff/reboot` はモダンな `systemd` 環境での推奨コマンド。`shutdown` は互換性のために残されており、内部的に `systemctl` を呼び出します。 | `systemctl` |
| **`reboot`, `poweroff`**| これらは `shutdown -r now` や `shutdown -P now` とほぼ同義のラッパーコマンドです。`shutdown` は**時間指定**や**警告メッセージ**の送信といった、より柔軟な機能を提供します。 | `shutdown` (時間指定時) |
| **`kill -9 1`** | **絶対にやってはいけない方法**。initプロセス (PID 1) を強制終了させ、カーネルパニックを引き起こし、システムを異常終了させます。 | - |

## よく連携されるコマンド

`shutdown` は単体で完結するコマンドですが、システムアップデートのような一連のメンテナンス作業の最終ステップとして実行されます。

### シナリオ例: 計画的なメンテナンス再起動 (インフラ構築・運用視点)

* **目的**: カーネルアップデートを適用した後、ログイン中のユーザーに通知し、15分後に安全にサーバーを再起動する。
* **連携コマンド**: `apt`/`yum`, `shutdown`
* **解説**: パッケージのアップグレード後、`shutdown -r` に時間とメッセージを指定して実行することで、他の管理者が作業中であっても安全に再起動プロセスを進めることができます。
* **コマンド例**:

    ```bash
    # (カーネルアップデートを実行した後...)
    # ログイン中の全ユーザーに通知し、15分後に再起動をスケジュール
    sudo shutdown -r +15 "Kernel update applied. System will reboot in 15 minutes."
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **時間指定** | |
| ⭐ `now` | **今すぐ**シャットダウンします。 |
| `+m` | **`m`分後**にシャットダウンします。（例: `+15` は15分後） |
| `hh:mm`| **指定した時刻**（24時間表記）にシャットダウンします。（例: `23:30`） |
| **アクション** | |
| ⭐ `-r`, `--reboot` | シャットダウン後にシステムを**再起動**します。 |
| ⭐ `-h`, `--halt` | システムを停止 (halt) します。最近のシステムでは電源オフ (`-P`) と同義です。 |
| `-P`, `--poweroff`| システムを停止し、**電源をオフ**にします。 |
| ⭐ `-c` | **予約されているシャットダウンをキャンセル**します。 |
| `-k` | 実際にシャットダウンはせず、警告メッセージの送信のみを行います（kidding）。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: サーバーのメンテナンスのために、今すぐシステムを再起動する。
* **組み合わせ**: `sudo shutdown -r now`
* **解説**: 最も一般的に使われる再起動コマンドです。
* **例**:

    ```bash
    # 今すぐシステムを完全に停止（電源オフ）
    sudo shutdown -h now

    # 10分後に予約した再起動をキャンセルする
    # (sudo shutdown -r +10 "...") を実行した後
    sudo shutdown -c
    ```

### 2. ブルーチーム視点

* **タスク**: 侵害されたホストを、これ以上悪用されないように完全に停止させる（最終的な封じ込め）。
* **組み合わせ**: `sudo shutdown -h now`
* **解説**: ネットワークから切り離すよりもさらに強力な封じ込め手段ですが、メモリ上にしか存在しない揮発性の証拠が失われるというトレードオフがあります。
* **例**:

    ```bash
    # ネットワーク上で被害を拡大させているホストを緊急停止
    sudo shutdown -h now
    ```

### 3. レッドチーム視点

* **タスク**: **サービス妨害 (Denial of Service, DoS) 攻撃**として、標的のサーバーをシャットダウンする。
* **組み合わせ**: `sudo shutdown -h now`
* **解説**: 攻撃者がroot権限を奪取した後、標的のビジネスに直接的な打撃を与えるために、本番稼働中のサーバーをシャットダウンします。これはシンプルながら非常に効果的な破壊活動です。
* **例**:

    ```bash
    # 標的のWebサーバーを停止させる
    sudo shutdown -h now
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `shutdown: Must be root.`
    * **考えられる原因**: `root` 権限なしで `shutdown` を実行しようとしました。
    * **解決策**: コマンドの前に `sudo` を付けて実行してください。

## 環境変数と設定ファイル

* `shutdown` の動作に直接影響する環境変数や設定ファイルは通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `shutdown` 自体ではなく、**`sudo` での実行権限の不適切な付与**がリスクとなります。
* **悪用シナリオ**: 攻撃者が `sudo shutdown` を実行できる権限を持っている場合、DoS攻撃に悪用されたり、GTFOBins に記載されているような手法で権限昇格に利用されたりする可能性があります。

### GTFOBins / LOLBAS における利用例

* **機能**: **Privilege Escalation** (権限昇格)
* **解説**: もしユーザーが `sudo shutdown` を実行できる権限を持っている場合、一部のシステムでは、シャットダウンプロセス中にシングルユーザーモードのようなメンテナンスシェルに移行することがあり、そこでrootプロンプトが得られる可能性があります。
* **参照**: `https://gtfobins.github.io/gtfobins/shutdown/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 最小権限の原則に従い、一般ユーザーに `shutdown` の `sudo` 権限を安易に与えない。データセンターなど物理アクセスが可能な場所では `Ctrl+Alt+Delete` キーによる再起動を無効化 (`sudo systemctl mask ctrl-alt-del.target`) する。
* **Detection (検知)**: `auditd` や `journalctl` で `shutdown` コマンドの実行を監視し、計画外のシャットダウンや再起動がないかを確認する。

## 注意点・補足

* **`systemctl` を推奨**: モダンな `systemd` 環境では、`shutdown` よりも `systemctl reboot` や `systemctl poweroff` を使う方が、`systemd` の状態管理とより密接に連携するため、挙動が明確です。
