# `journalctl` - systemdジャーナルログを照会・表示する

## 概要

`journalctl` は、**systemdジャーナル**に記録されたログを閲覧・照会するためのコマンドラインツールです。systemdジャーナルは、カーネル、システムサービス、アプリケーションなど、システム上のあらゆる場所から出力されるログを、一元的に収集・管理する仕組みです。ログは構造化されたバイナリ形式で保存されるため、単純なテキストファイルよりも高速な検索、強力なフィルタリング、信頼性の高い管理が可能です。

`journalctl` は、旧来の `/var/log/` 配下にある複数のテキストログファイル (`syslog`, `messages`, `auth.log`など) を横断的に、かつ統一されたインターフェイスで調査するためのモダンな標準ツールです。

(出自: `systemd` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`dmesg`** | カーネルのリングバッファメッセージを表示します。`journalctl -k` は `dmesg` と同じ情報を表示しますが、`journalctl` の方が時間指定などのフィルタリング機能が豊富です。 | `journalctl -k` |
| **`tail -f /var/log/...`**| 旧式の `rsyslog` 環境で、特定のテキストログファイルをリアルタイムで監視する方法です。`journalctl` は、システム全体の全てのログを対象に、より構造化されたクエリを実行できます。 | `journalctl` |

## よく連携されるコマンド

`journalctl` の豊富なフィルタ機能を使っても、最終的な絞り込みには `grep` との連携が有効です。

### シナリオ例: SSHのログイン失敗を調査する (ブルーチーム視点)

* **目的**: SSHサービス (`sshd`) のログの中から、"Failed password" という文字列を含む行だけを抽出する。
* **連携コマンド**: `grep`
* **解説**: まず `journalctl -u` で対象をsshdサービスに絞り込み、その出力の中からさらに `grep` でキーワード検索することで、調査したいイベントを効率的に見つけ出すことができます。
* **コマンド例**:

```bash
# sshdサービスのログから、ログイン失敗の記録を検索する
sudo journalctl -u sshd.service | grep "Failed password"
```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **表示制御** | |
| ⭐ `-f`, `--follow` | 新しいログが書き込まれるのをリアルタイムで追跡表示します (`tail -f` と同義)。 |
| `-e`, `--pager-end` | ページャの表示を即座に末尾にジャンプさせます。 |
| `-n <number>` | 直近の `<number>` 行を表示します (`tail -n` と同義)。 |
| `-o <format>` | 出力フォーマットを変更します。`short` (デフォルト), `verbose`, `json`, `json-pretty` など。 |
| **フィルタリング** | |
| ⭐ `-u <unit>` | 指定した `systemd` ユニット（サービス名など）のログのみを表示します。（例: `-u nginx.service`） |
| ⭐ `-k`, `--dmesg` | カーネルメッセージのみを表示します。 |
| `-b [<ID>]` | 特定の起動セッションのログを表示します。`-b` (現在), `-b -1` (前回), `-b -2` (前々回)... |
| `-p <priority>`| 指定した優先度以上のログを表示します。`0`(emerg)〜`7`(debug)、または `err`, `warning`, `info` などで指定。 |
| ⭐ `--since <time>`| 指定した時刻**以降**のログを表示します。"2025-09-22 10:00:00" や "1 hour ago" のように指定。 |
| `--until <time>` | 指定した時刻**以前**のログを表示します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 特定のサービスのログをリアルタイムで監視し、デバッグする。
* **組み合わせ**: `journalctl -u <service_name> -f`
* **解説**: Nginxやアプリケーションサーバーを再起動した際に、このコマンドでログを監視し、エラーが出力されずに正常に起動したかを確認する、最も代表的な使い方です。
* **例**:

```bash
# nginx.service のログをリアルタイムで表示
sudo journalctl -u nginx.service -f
```

### 2. ブルーチーム視点

* **タスク**: 過去1日分のSSHデーモンのログを調査し、不正ログインの試みがないか確認する。
* **組み合わせ**: `journalctl -u sshd --since "1 day ago"`
* **解説**: `--since` オプションで調査範囲を限定し、特定のサービス (`-u sshd`) のログを効率的に監査します。`"Failed password"` や `"Accepted publickey"` といったキーワードで `grep` を組み合わせることで、さらに詳細な分析が可能です。
* **例**:

```bash
# 過去1時間以内に "Failed" という単語を含む全ログを検索
sudo journalctl --since "1 hour ago" | grep "Failed"
```

### 3. レッドチーム視点

* **タスク**: 侵入後の偵察として、システムの活動状況を把握する。
* **組み合わせ**: `journalctl`
* **解説**: 攻撃者は侵入後 `journalctl` の内容を閲覧し、`cron` のログから定期実行ジョブを、`sshd` のログからログインユーザーや頻度を、アプリケーションのログからシステムの構成やバージョン情報を推測します。
* **痕跡消去**: root権限を奪取した攻撃者は、ジャーナルファイル (`/var/log/journal/` or `/run/log/journal/`) を直接削除・改ざんして、自身の侵入や活動の痕跡を消去しようと試みます。

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **現象**: **サーバーを再起動したら、それ以前のログが消えてしまった。**
    * **考えられる原因**: ジャーナルのログ保存が**永続化**されていません。デフォルトでは、ログは `/run/log/journal` （メモリ上）に保存され、再起動で消去されます。
    * **解決策**: `sudo mkdir -p /var/log/journal` を実行して、永続的な保存ディレクトリを作成します。ディレクトリが存在すれば、`systemd-journald` は自動的にそこにログを書き込むようになります。

## 環境変数と設定ファイル

* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

### 関連する設定ファイル

* **`/etc/systemd/journald.conf`**: `systemd-journald` デーモンの設定ファイル。ログの保存場所 (`Storage=`) や、最大ディスク使用量 (`SystemMaxUse=`) などを定義します。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `journalctl` 自体ではなく、それが呼び出すページャ (`less`) の機能が悪用される可能性があります。
* **悪用シナリオ**: レッドチームのシナリオで述べたように、攻撃者は `root` 権限を奪うとログファイルの削除・改ざんを試みます。

### GTFOBins / LOLBAS における利用例

* **機能**: **Privilege Escalation** (権限昇格)
* **解説**: `journalctl` は出力を `less` などのページャで表示します。もしユーザーが `sudo journalctl` を実行できる権限を持っている場合、ページャの画面で `!/bin/sh` と入力することで、ページャの機能（外部コマンド実行）を悪用し、**root権限のシェル**を起動できてしまいます。
* **参照**: `https://gtfobins.github.io/gtfobins/journalctl/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: `sudoers` の設定を厳格にし、一般ユーザーに `journalctl` の `sudo` 権限を安易に与えない。
* **Detection (検知)**: **ログのリアルタイムな外部転送**が最も堅牢な対策です。`systemd-journal-remote` などを使い、ジャーナルログを改ざん不可能なリモートのSIEMなどのログサーバーに転送します。

## 注意点・補足

* **ディスク使用量**: ジャーナルログは放置するとディスク容量を圧迫することがあります。`journalctl --disk-usage` で現在の使用量を確認し、`/etc/systemd/journald.conf` の `SystemMaxUse=` で上限サイズを設定することが推奨されます。
