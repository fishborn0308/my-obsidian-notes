# `docker start` - 停止しているコンテナを開始する

## 概要

`docker start` は、`docker stop` コマンドや、何らかの理由で終了して**停止 (`Exited`) 状態**になっているコンテナを、再度開始するためのコマンドです。このコマンドは新しいコンテナを作成するのではなく、**既存のコンテナを現在の状態で再開**します。複数のコンテナIDまたは名前をスペース区切りで指定することで、一度に複数のコンテナを開始できます。

(出自: `docker` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`docker run`** | **新しいコンテナを作成して**開始します。`start` は**既存の停止しているコンテナ**を開始する点が根本的に異なります。 | - |
| **`docker restart`** | 実行中のコンテナを**停止してから再度開始**（再起動）します。 | - |
| **`docker unpause`**| `docker pause` で**一時停止 (`Paused`) 状態**になっているコンテナのプロセスを再開します。`start` は**停止 (`Exited`) 状態**のコンテナを対象とします。 | - |

## よく連携されるコマンド

`docker start` は、`docker ps -a` で停止中のコンテナを確認した後に、そのコンテナを起動するという流れでよく使われます。

### シナリオ例: メンテナンス後にサービスを再開する (インフラ構築・運用視点)

* **目的**: サーバーメンテナンスで意図的に停止 (`docker stop`) したコンテナ群を、メンテナンス完了後に再度起動する。
* **連携コマンド**: `docker ps`, `docker start`
* **解説**: まず `docker ps -a` で停止中の対象コンテナ (`my-webapp`, `my-database`) の名前を確認し、`docker start` でそれらをまとめて起動します。
* **コマンド例**:

    ```bash
    # 1. 停止中のコンテナを確認
    docker ps -a

    # 2. メンテナンス完了後、コンテナをまとめて起動
    docker start my-webapp my-database
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **アタッチ** | |
| ⭐ `-a`, `--attach` | コンテナの標準出力/標準エラー出力にアタッチし、コンテナの起動時の出力をリアルタイムで確認します。 |
| `-i`, `--interactive` | コンテナの標準入力 (stdin) にアタッチします。`-a` と組み合わせて使われることがあります。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: ホストOSの再起動後、再起動ポリシーが設定されていないコンテナを手動で起動する。
* **組み合わせ**: `docker start <container_name_or_id>`
* **解説**: サーバーの再起動後、`docker ps -a` で `Exited` 状態のコンテナがないかを確認し、必要なものを手動で起動します。
* **例**:

    ```bash
    # メンテナンス後にDBコンテナとWebコンテナを起動
    docker start my-database my-webapp
    ```

### 2. ブルーチーム視点

* **タスク**: 不審なコンテナの起動シーケンスを直接監視する。
* **組み合わせ**: `docker start -a <suspicious_container_id>`
* **解説**: 停止状態の不審なコンテナを発見した場合、`-a` (`--attach`) を付けて起動します。これにより、コンテナが起動時にどのような処理を行い、どのようなログを出力するかをリアルタイムで捕捉し、マルウェアの挙動を分析する手がかりとします。

### 3. レッドチーム視点

* **タスク**: 侵入したホスト上で、普段は使われていないコンテナを起動し、攻撃の足がかりを探す。
* **組み合わせ**: `docker start <dormant_container>`
* **解説**: 攻撃者は `docker ps -a` でホスト上の全コンテナを偵察します。`admin-tools` や `backup-server` のような興味深い名前の停止中コンテナを発見した場合、`docker start` でそれを起動してみます。そのコンテナに脆弱性や特権的な設定があれば、新たな攻撃経路となる可能性があります。

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **現象**: **`docker start` を実行したが、すぐにコンテナが停止 (`Exited`) してしまう。**
    * **考えられる原因**: コンテナのメインプロセスが起動直後にエラーで終了しています。
    * **解決策**: `docker logs <container_id>` を実行して、コンテナがなぜ終了してしまったのか、エラーログを確認してください。

## 環境変数と設定ファイル

* `docker start` の動作に直接影響する環境変数は通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **休眠中のリスクの再有効化**。
* **悪用シナリオ**: 意図的に停止されていた、あるいは脆弱性が発見されたために停止措置が取られていたコンテナを誤って `start` してしまうと、セキュリティリスクを再びシステムに晒すことになります。

### GTFOBins / LOLBAS における利用例

`docker start` は `docker` コマンドの一部として扱われます。`sudo docker` の権限があれば、任意のコンテナを起動可能です。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: Dockerソケットへのアクセス権限を厳格に管理する。
* **Detection (検知)**: `auditd` で `docker start` コマンドの実行を監視し、計画外のコンテナ起動がないかを確認する。

## 注意点・補足

* **再起動ポリシーの活用**: 本番環境など、常に稼働させたいコンテナには、`docker run` 時に `--restart=always` を設定しておくことが推奨されます。これにより、手動で `docker start` を実行する必要がなくなります。
