# Command - Linux - kmod - カーネルモジュールを管理するツール群

## 概要
`kmod` は、Linux カーネルモジュールのロード、アンロード、表示、依存関係の解決を行うためのプログラムセットです。現代の Linux では、`lsmod` や `modprobe` といった個別のコマンドは、実際にはこの `kmod` バイナリへのシンボリックリンク（またはマルチコールバイナリとしての動作）となっています。



## 主要な関連コマンド（kmod の正体）
| コマンド | 役割 | 特徴 |
| :--- | :--- | :--- |
| **`lsmod`** | 現在ロードされているモジュールを表示。 | `/proc/modules` の内容を綺麗に見せる。 |
| **`modprobe`** | モジュールのロード/アンロード。 | **推奨。** 依存関係を自動で解決する。 |
| **`insmod`** | モジュールファイルを直接ロード。 | 依存関係を無視する。開発・デバッグ用。 |
| **`rmmod`** | モジュールを削除。 | 使用中のモジュールは削除できない。 |
| **`modinfo`** | モジュールの詳細情報を表示。 | 作者、ライセンス、パラメータ等を確認。 |

## よく使われるオプションと構文
`kmod` 自体を直接叩く機会は少ないですが、以下のようにサブコマンドとして機能します。

```bash
# 基本形: kmod <subcommand> [options]
````

|**サブコマンド**|**説明**|
|---|---|
|⭐️ `list`|`lsmod` と同等。ロード済みモジュールを表示。|
|⭐️ `static-nodes`|現在のカーネルで報告されているスタティックノードを表示。|
|⭐️ `help`|ヘルプを表示。|

## 業務シナリオ別活用法

### 1. インフラ構築・運用視点

- **タスク**: ネットワークカードやストレージコントローラのドライバ有効化。
    
- **解説**: OS インストール後に認識しなかったハードウェアを、適切なモジュールを `modprobe` することで使えるようにします。
    
- **例**:
    
    Bash
    
    ```
    # 仮想化環境で vmxnet3 ドライバが読み込まれているか確認
    lsmod | grep vmxnet3
    ```
    

### 2. ブルーチーム視点

- **タスク**: ルートキット（Rootkit）の潜伏調査。
    
- **解説**: 攻撃者は `kmod` を悪用して、自作の悪意あるモジュールをカーネルに挿入し、プロセスやファイルを隠蔽します。`lsmod` の結果と `/lib/modules/` 内の正規のリストを比較し、身に覚えのないモジュールがないか調査します。
    

### 3. レッドチーム視点

- **タスク**: カーネルレベルでの永続化（Persistence）。
    
- **解説**: root 権限奪取後、`insmod` を用いてバックドア機能を持つ LKM をロードします。これにより、ユーザーモードのツール（`ps` や `ls`）からは絶対に見えない形でシステムを制御し続けます。
    

## エラーメッセージとトラブルシューティング

1. **「modprobe: ERROR: could not insert 'module': Operation not permitted」**:
    
    - **原因**: 権限不足。カーネルの操作には root 権限が必須。
        
    - **解決策**: `sudo` を付けて実行する。
        
2. **「rmmod: ERROR: Module module is in use」**:
    
    - **原因**: 他のモジュールやプロセスがそのモジュールを使用しているため、削除するとシステムが不安定になる。
        
    - **解決策**: 使用しているプロセスを停止させるか、依存している他のモジュールを先にアンロードする。
        

## セキュリティに関する考慮事項

### 脆弱性とリスク

- **LKM Rootkit**: 一度悪意あるモジュールがロードされると、OS の「脳」であるカーネル自体が汚染されるため、検知は極めて困難になります。
    
- **Denial of Service (DoS)**: 不適切なパラメータでモジュールをロードしたり、重要なモジュールを無理やりアンロードしたりすると、システムが即座にパニック（Kernel Panic）を起こして停止します。
    

### 対応策・緩和策 (ブルーチーム視点)

- **Module Signing**: 信頼できる署名がないモジュールのロードをカーネルレベルで拒否するように設定します（Secure Boot との連携）。
    
- **Disabling Module Loading**: サーバーの稼働開始後に新しいモジュールを読み込む必要がない場合、`sysctl -w kernel.modules_disabled=1` でロード機能を完全にロックします。
    

## 注意点・補足

- **設定の永続化**: `modprobe` でロードしただけでは再起動後に元に戻ります。永続化するには `/etc/modules-load.d/` に設定ファイルを作成する必要があります。
