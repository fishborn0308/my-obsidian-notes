# Command - Linux - mkdir - 新しいディレクトリを作成する

## 概要
`mkdir` は、指定した名前に基づいて1つまたは複数の新しいディレクトリを作成します。デフォルトでは、親ディレクトリが存在しない場合はエラーになりますが、オプションを駆使することで深い階層構造をワンコマンドで構築可能です。



## 🛠 類似・関連コマンド
| コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `rmdir` | 「空の」ディレクトリを削除する。 | 安全な削除 |
| `install -d` | ディレクトリ作成と同時に、詳細な属性設定を行う。 | Makefile内での配置 |
| `mkdir -p` | 途中の階層も含めて、一気に作成する。 | **スクリプト・自動化** |

## 主要なオプション
| オプション | 説明 |
| :--- | :--- |
| ⭐️ `-p`, `--parents` | **最頻用。** 親ディレクトリも必要に応じて作成。既に存在してもエラーにしない。 |
| ⭐️ `-m <MODE>`, `--mode` | 作成時にパーミッションを指定する（例: `mkdir -m 700 dir`）。 |
| ⭐️ `-v`, `--verbose` | 作成したディレクトリごとにメッセージを表示する。 |
| `-Z` | SELinux のセキュリティコンテキストをデフォルトで設定する。 |

## よく使われる連携コマンド（ブレース展開）
### シナリオ例: 複雑な構造を1秒で作る
* **目的**: 開発プロジェクトの初期ディレクトリ構造（src, bin, docs, tests）を即座に作成。
* **解説**: シェルの「ブレース展開 `{}`」と組み合わせると、`mkdir` は真の力を発揮します。
* **コマンド例**:
    ```bash
    # プロジェクトフォルダ内に複数のサブフォルダを同時に作成
    mkdir -p project/{src,bin,docs,tests/unit}
    ```

## 業務シナリオ別活用法

### 1. インフラ構築・運用視点
* **タスク**: ログ保存用のディレクトリを、適切な権限で作成。
* **解説**: 後から `chmod` を打つのではなく、作成の瞬間に所有者以外アクセス禁止に設定し、セキュリティホールをゼロにします。
* **例**:
    ```bash
    # 自分(root)しか中身を見られないログフォルダを作成
    sudo mkdir -m 700 /var/log/my_secret_app
    ```

### 2. ブルーチーム視点
* **タスク**: ディレクトリ・トラバーサル攻撃の痕跡調査。
* **解説**: 攻撃者が不自然な名前のディレクトリ（例: `... ` や `.. `）を作成し、そこにツールを隠していないか、`ls -la` と組み合わせて監視します。

### 3. レッドチーム視点
* **タスク**: 隠蔽工作とランディングゾーンの確保。
* **解説**: `/tmp` などの共有領域に、一見するとシステムの一部に見えるような名前のディレクトリを作成し、攻撃ツールや盗み出したデータを一時保存します。
* **例**:
    ```bash
    # システムの一部を装った隠しディレクトリを作成
    mkdir -p /tmp/.system_cache_update_v2.1
    ```

## エラーメッセージとトラブルシューティング

1.  **「mkdir: cannot create directory ‘...’: File exists」**:
    * **原因**: 同名のディレクトリまたはファイルが既に存在する。
    * **解決策**: `-p` を使えば、ディレクトリが既にある場合は何もせず正常終了します。

2.  **「mkdir: cannot create directory ‘...’: No such file or directory」**:
    * **原因**: 存在しない親ディレクトリの中に子ディレクトリを作ろうとした。
    * **解決策**: `-p` オプションを付けて親階層も自動生成させます。

3.  **「mkdir: cannot create directory ‘...’: Permission denied」**:
    * **原因**: 書き込み権限がないディレクトリで作成を試みた。
    * **解決策**: `sudo` を使うか、権限のある場所（`~/` など）を選びます。

## セキュリティに関する考慮事項

### 脆弱性とリスク
* **Insecure Permissions**: `umask` の設定が緩い場合、`mkdir` で作ったディレクトリが「誰でも読み書き可能」になってしまい、機密情報が露出するリスクがあります。
* **Race Condition**: 共有ディレクトリ内でディレクトリを作成する際、作成の瞬間を狙ってシンボリックリンクを差し込まれる「シンボリックリンク攻撃」に注意が必要です。

### 対応策・緩和策 (ブルーチーム視点)
* **umask の徹底**: ユーザーの `umask` を `022` または `077` に設定し、デフォルトで安全な権限が設定されるように強制します。
