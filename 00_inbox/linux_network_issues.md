# 一般的なネットワークトラブルシューティング

## 概要

このドキュメントは、ネットワーク関連の一般的な問題に対するトラブルシューティングの手順と解決策をまとめたものです。インフラエンジニアの運用視点に加えて、ブルーチーム(防御側)とレッドチーム(攻撃側)それぞれの視点からの問題特定、分析、対策についても触れます。

## トラブルシューティングの基本フロー

どのようなネットワークトラブルにも共通して適用できる、基本的なトラブルシューティングの流れを説明します。

1. **問題の特定と情報収集**: 何が、いつから、どこで、どのように発生しているか。影響範囲は。
2. **原因の切り分け**: OSI参照モデルに基づいたレイヤーごとの切り分けや、物理層からアプリケーション層への確認。
3. **仮説の立案と検証**: 収集した情報から考えられる原因を特定し、仮説を立てて検証する。
4. **解決策の実施と確認**: 問題を解決し、再度発生しないか確認する。
5. **文書化と再発防止**: トラブルシューティングの過程と解決策を記録し、今後の対策に活かす。

## 1. 物理層/データリンク層のトラブル

ネットワークケーブルやNIC、L2スイッチなど、もっとも基本的な層での問題と対処法です。

### 1.1. リンクアップしない/通信が不安定

- **考えられる原因**:
  - ケーブルの断線、コネクタ不良
  - NICの故障、ドライバーーーの問題
  - スイッチポートの故障、設定ミス(シャットダウン、VLANミスなど)
- **確認方法**:
  - 目視確認(ケーブル、LEDランプ)
  - `ping` コマンドによる疎通確認
  - `ip link show` (Linux) / `ipconfig /all` (Windows) でNICの状態確認
  - スイッチのポートステータス確認
- **解決策**:
  - ケーブルの交換、再接続
  - NICのドライバーーー再インストール、交換
  - スイッチポートの有効化、VLAN設定の見直し
- **ブルーチーム視点**: ケーブルが意図的に切断されていないか、不正なデバイスが接続されていないかなど、物理セキュリティの観点からも確認。
- **レッドチーム視点**: 物理的なアクセスが可能な場合、意図的にケーブルを切断したり、不正なデバイス(タップ、スプーフィングデバイス)を接続したりするシナリオ。

### 1.2. MACアドレスの重複/MACフラッディング攻撃

- **考えられる原因**:
  - 設定ミスによるMACアドレスの重複
  - MACフラッディング攻撃(L2DoS攻撃)
- **確認方法**:
  - スイッチのMACアドレステーブル確認
  - ネットワークトラフィックの監視(不正なARPパケットなど)
- **解決策**:
  - 重複MACアドレスの特定と修正
  - ポートセキュリティの設定(MACアドレス制限)
  - 不正なデバイスの隔離
- **ブルーチーム視点**: IDS/IPSでのMACフラッディング検知、異常なMACアドレスの監視。
- **レッドチーム視点**: MACアドレススプーフィング、MACフラッディングツール(macofなど)を用いた攻撃。

## 2. ネットワーク層のトラブル

IPアドレス、ルーティングに関する問題と対処法です。

### 2.1. IPアドレスの競合/不正なIPアドレス設定

- **考えられる原因**:
  - DHCPサーバーの設定ミス、またはDHCPクライアントの不具合
  - 手動設定でのIPアドレス重複
  - 不正なIPアドレスを持ったデバイスの接続
- **確認方法**:
  - `ipconfig` / `ip addr show` でIPアドレス確認
  - DHCPサーバーのログ確認
  - ARPテーブルの確認
- **解決策**:
  - DHCP設定の見直し、DHCPリース情報のクリア
  - 手動設定IPアドレスの修正
  - 不正なデバイスの特定と排除
- **ブルーチーム視点**: 不正なDHCPサーバーの検知、IPアドレス管理台帳との突合、ARPキャッシュポイズニングの監視。
- **レッドチーム視点**: 不正なDHCPサーバーの構築(Rogue DHCP Server)、ARPキャッシュポイズニング(Man-in-the-Middle攻撃)。

### 2.2. ルーティングの問題(デフォルトゲートウェイ、ルーティングテーブル)

- **考えられる原因**:
  - デフォルトゲートウェイの設定ミス
  - ルーティングテーブルの不正なエントリ、ルーティングプロトコルの不具合
  - ファイアウォールでのルーティング阻止
- **確認方法**:
  - `ip route show` (Linux) / `route print` (Windows) でルーティングテーブル確認
  - `traceroute` / `tracert` で経路確認
  - ファイアウォールログの確認
- **解決策**:
  - デフォルトゲートウェイの修正
  - ルーティングテーブルの修正、ルーティングプロトコルの再起動
  - ファイアウォールルールの見直し
- **ブルーチーム視点**: 不正なルーティング情報の注入(BGPハイジャックなど)、ルーティングプロトコルの脆弱性悪用監視。
- **レッドチーム視点**: ルーティング情報の改ざん、ルーティングプロトコルの脆弱性(OSPF、EIGRPなど)悪用によるトラフィック誘導。

## 3. トランスポート層のトラブル

TCP/UDPポート、ファイアウォールに関する問題と対処法です。

### 3.1. ポートの疎通不可/接続拒否

- **考えられる原因**:
  - ファイアウォール(OS、ネットワーク)によるブロック
  - サービスが起動していない、またはListenしていない
  - ポート番号の誤り
- **確認方法**:
  - `telnet` / `nc` でポートの疎通確認
  - `netstat -tulnp` (Linux) / `netstat -ano` (Windows) でListenポート確認
  - ファイアウォールログの確認
- **解決策**:
  - ファイアウォールルールの追加・修正
  - サービスの起動、設定の見直し
  - 正しいポート番号の使用
- **ブルーチーム視点**: 不正なポートスキャン、意図しないポートの開放、ファイアウォール設定の定期的な監査。
- **レッドチーム視点**: ポートスキャン(nmapなど)による開放ポートの探索、ファイアウォール回避技術の利用。

### 3.2. セッションタイムアウト/接続数限界

- **考えられる原因**:
  - サーバー側のセッションタイムアウト設定
  - サーバーのリソース不足(メモリ、CPU、ファイルディスクリプター数など)
  - DDoS攻撃による接続過多
- **確認方法**:
  - サーバーのログ(Webサーバー、DBサーバーなど)
  - サーバーのリソース使用状況監視
  - ネットワークトラフィックの分析
- **解決策**:
  - タイムアウト設定の調整
  - リソースの増強、設定の最適化
  - DoS/DDoS対策(WAF、IPS、レートリミットなど)
- **ブルーチーム視点**: 異常な接続数の監視、DDoS攻撃の検知と緩和策の適用。
- **レッドチーム視点**: DoS/DDoS攻撃(SYN Flood、HTTP Floodなど)、スローロリス攻撃。

## 4. アプリケーション層のトラブル

DNS、HTTP/HTTPS、特定のアプリケーションプロトコルに関する問題と対処法です。

### 4.1. 名前解決の問題(DNS)

- **考えられる原因**:
  - DNSサーバーの設定ミス(参照先DNSサーバー、フォワーダーなど)
  - DNSキャッシュの汚染
  - ドメイン名の入力ミス、レコードの誤り
- **確認方法**:
  - `nslookup` / `dig` で名前解決確認
  - `/etc/resolv.conf` (Linux) / ネットワークアダプターの設定 (Windows) で参照DNSサーバー確認
  - DNSサーバーのログ確認
- **解決策**:
  - DNSサーバーの設定修正
  - DNSキャッシュのクリア
  - ドメイン名、レコードの修正
- **ブルーチーム視点**: DNSキャッシュポイズニング、DNSトンネリング、DNSリフレクション攻撃の監視と対策。
- **レッドチーム視点**: DNSスプーフィング、DNSリコン(Zone Transferなど)、DNSトンネリングによるデータ窃取。

### 4.2. Webサイトへのアクセス不可/HTTPS接続エラー

- **考えられる原因**:
  - Webサーバーの停止、設定ミス
  - SSL/TLS証明書の問題(期限切れ、不一致、信頼されていないCA)
  - プロキシサーバー、WAFによるブロック
- **確認方法**:
  - Webサーバーの稼働状態、ログ確認
  - ブラウザの開発者ツールでネットワークタブ、セキュリティタブを確認
  - `openssl s_client` でSSL/TLS接続テスト
- **解決策**:
  - Webサーバーの起動、設定修正
  - SSL/TLS証明書の更新、再発行、設定修正
  - プロキシ、WAFの設定見直し
- **ブルーチーム視点**: SSL/TLS脆弱性の監視(Heartbleed、POODLEなど)、Webアプリケーションの脆弱性診断、WAFルールの最適化。
- **レッドチーム視点**: Webアプリケーションの脆弱性(SQLインジェクション、XSSなど)悪用、SSL/TLSダウングレード攻撃、証明書関連のミスを利用した攻撃。

## 5. その他の共通トラブルシューティング

### 5.1. パケットロス/遅延

- **考えられる原因**:
  - ネットワーク帯域の輻輳
  - 不正なルーティング、ループ
  - デバイスの処理能力不足
  - 物理層の問題(ケーブル品質など)
- **確認方法**:
  - `ping -c <count>` でパケットロス率、RTT確認
  - `mtr` でパケットロス、遅延が発生しているホップの特定
  - ネットワーク機器のトラフィック統計、エラーカウンター確認
- **解決策**:
  - 帯域の増強、QoS設定
  - ルーティングの見直し、ループの解消
  - 機器のアップグレード、設定最適化
  - ケーブル交換
- **ブルーチーム視点**: 意図的なパケットドロップの検知、ネットワークトラフィックのベースラインからの異常値監視。
- **レッドチーム視点**: 帯域枯渇攻撃、ジャンクパケット送信によるネットワーク負荷増大。

### 5.2. ログと監視

- **ログの種類**: OSログ(syslog, Event Log)、アプリケーションログ、ネットワーク機器ログ(SNMP Trap, Syslog)、ファイアウォールログなど。
- **監視ツール**: Zabbix, Prometheus, ELK Stack, Splunk, CloudWatch, Azure Monitorなど。
- **ブルーチーム視点**: 不正アクセス試行、異常な通信、設定変更、エラーなどのログ監視とSIEM連携。
- **レッドチーム視点**: ログの改ざん、削除、ログに残さない攻撃手法の利用。
