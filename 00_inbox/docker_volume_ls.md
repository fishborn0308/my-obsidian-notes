# `docker volume ls` - Dockerボリュームを一覧表示する

## 概要

`docker volume ls` (または `docker volume list`) は、Dockerホスト上に存在する**名前付きボリューム**と**匿名ボリューム**を一覧表示するコマンドです。このコマンドを使うことで、どのボリュームが現在利用可能か、不要になったボリュームがないかなどを素早く確認できます。ボリュームの削除 (`docker volume rm`) や詳細調査 (`docker volume inspect`) を行う前の、現状把握のための基本的なコマンドです。

(出自: `docker` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`docker volume inspect`**| 特定のボリュームに関する**全ての詳細情報**（ホスト上のパスなど）をJSON形式で表示します。`ls` は**概要の一覧表示**、`inspect` は**詳細な調査**に使います。 | - |
| **`docker volume prune`** | **未使用の**ボリュームを一括で削除します。`ls` は一覧表示、`prune` は削除を行うコマンドです。 | `prune` (一括削除) |
| **`docker ps -a --volumes`**| **コンテナに紐付いている**ボリューム情報を表示します。`volume ls` は、どのコンテナにも紐付いていない未使用のボリュームも含め、**全てのボリューム**を一覧表示します。 | - |

## よく連携されるコマンド

`docker volume ls` は、その出力結果を `xargs` に渡し、`docker volume rm` で一括削除する、というクリーンアップのワークフローでよく連携されます。

### シナリオ例: dangling状態のボリュームを一括削除する (インフラ構築・運用視点)

* **目的**: どのコンテナからも参照されなくなった「dangling」（宙ぶらりん）状態のボリュームを全て削除して、ディスク容量を解放する。
* **連携コマンド**: `xargs`, `docker volume rm`
* **解説**: `docker volume ls` の `-f` (`--filter`) オプションでdangling状態のボリュームを絞り込み、`-q` でその名前だけを出力します。その結果を `xargs` に渡して `docker volume rm` で一括削除します。
* **コマンド例**:

    ```bash
    # dangling状態のボリューム名をリストアップし、xargsで一括削除
    docker volume ls -f "dangling=true" -q | xargs docker volume rm

    # 注: この操作は `docker volume prune` コマンドでより簡単に行えます。
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **出力制御** | |
| ⭐ `-q`, `--quiet` | ボリューム名のみを表示します。他のコマンドと組み合わせる際に便利です。 |
| **フィルタリング** | |
| ⭐ `--filter <key>=<value>`| 特定の条件で表示するボリュームをフィルタリングします。（例: `--filter "dangling=true"`, `--filter "name=my-app"`） |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: ホスト上に存在するボリュームを棚卸しし、不要なものを特定する。
* **組み合わせ**: `docker volume ls`
* **解説**: 定期的なメンテナンス作業として、プロジェクトが終了して不要になったボリュームや、テストで作成されたまま残っているボリュームがないかを確認します。
* **例**:

    ```bash
    # 現在の全ボリュームを確認
    docker volume ls
    ```

### 2. ブルーチーム視点

* **タスク**: 侵害されたホスト上のデータ永続化ポイントを把握する。
* **組み合わせ**: `docker volume ls`
* **解説**: インシデント調査の初期段階で、`suspicious-data` や `loot-storage` のような不審な名前のボリューム、あるいは大量の意図しない匿名ボリュームが存在しないかを確認します。これらは攻撃者がデータを永続化させたり、ツールを隠したりするために作成した可能性があります。

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **現象**: **意図しないランダムな名前のボリュームがたくさん表示される。**
    * **考えられる原因**: これらは**匿名ボリューム**です。`docker run -v /data ...` のように、ホスト側のパスやボリューム名を指定せずにコンテナ側のパスだけを指定してボリュームをマウントすると自動的に作成されます。
    * **解決策**: データの永続化には、管理しやすい**名前付きボリューム**を使用することが推奨されます。不要な匿名ボリュームは `docker volume prune` で一括削除できます。

## 環境変数と設定ファイル

* `docker volume ls` の動作に直接影響する環境変数や設定ファイルは通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **情報漏洩**。
* **悪用シナリオ**: ボリューム名は、そのボリュームにどのようなデータが格納されているかを推測させる重要な情報源です。`mysql-prod-backup` のような分かりやすい名前は、管理者にとっては便利ですが、攻撃者にとっても格好の標的となります。

### GTFOBins / LOLBAS における利用例

`docker volume ls` は `docker` コマンドの一部として扱われます。`sudo docker` の権限があれば、任意の操作が可能です。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: Dockerソケットへのアクセス権限を厳格に管理する。
* **Detection (検知)**: `auditd` で `docker volume ls` の実行を監視し、不審なユーザーによる偵察活動がないかを確認する。

## 注意点・補足

* **バインドマウントは表示されない**: このコマンドのリストには、`-v /host/path:/container/path` の形式でマウントされた**バインドマウントは表示されません**。バインドマウントはDockerが管理するボリュームではないためです。
