# Command  - Linux - du - ファイルやディレクトリのディスク使用量を表示する




## 概要

`du` (disk usage) は、指定したファイルやディレクトリが、ディスク上で実際にどれくらいの容量を消費しているかを計算し、表示するコマンドです。サブディレクトリを再帰的にたどり、それぞれの使用量と合計を表示します。サーバのディスク容量が逼迫している際に、`df` コマンドで「どのファイルシステムが一杯か」を確認した後、`du` を使って「**具体的にどのディレクトリやファイルが容量を消費しているのか**」を特定するのが一般的な使い方です。

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント |
| :--- | :--- |
| **`df`** | (disk free) **ファイルシステム（パーティション）全体**の使用状況を表示します。`du` は**ファイルやディレクトリ単位**の使用量を表示します。**「パーティション全体の空き容量は？」と聞くのが `df`**、**「このディレクトリのサイズは？」と聞くのが `du`** です。 |
| **`ls -l`**| `ls -l` が表示するのは個々の**ファイル**のサイズです。ディレクトリ自体のサイズは表示しますが、その**中に含まれるファイルの合計サイズ**は表示しません。ディレクトリの中身を含めた合計サイズを知るには `du` を使います。 |
| **`ncdu`** | `du` のncurses（テキストベースのUI）版。対話的にディレクトリをドリルダウンしながら、どのサブディレクトリが最も容量を消費しているかを視覚的に分析でき、非常に便利です。 |

## `du` コマンド出力の読み方

`du` の出力はシンプルで、各行の先頭にサイズ、次にファイル/ディレクトリ名が表示されます。

```bash

8       ./.config
12      ./.ssh
16      ./logs/old
20      ./logs
40      .

```

- `8`, `12`...: ディスク使用量（デフォルトはキロバイト単位）。

- `./.config`...: ディレクトリ名またはファイル名。
- `40 .`: 最後の行はカレントディレクトリ (`.`) の合計サイズ。

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **`-h`, `--human-readable`** | **最もよく使うオプション**。サイズを人間が読みやすい単位 (K, M, G) で表示します。 |
| **`-s`, `--summarize`** | サブディレクトリの内訳を表示せず、指定したディレクトリの**合計サイズのみ**を表示します。 |
| **`-c`, `--total`** | 最後に**総合計 (grand total)** を表示します。複数の引数を指定した場合に便利です。 |
| **`-a`, `--all`** | ディレクトリだけでなく、個々のファイルの使用量も表示します。 |
| **`--max-depth=<N>`**| 指定した階層の深さ (`N`) までのディレクトリのみを表示します。 |
| **`--exclude=<pattern>`**| 指定したパターンに一致するファイルやディレクトリを計算から除外します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

- **組み合わせ**: `du -sh * | sort -rh | head -n 10`
- **説明**: **最も代表的な使い方**。「カレントディレクトリ内で最も容量の大きいファイルやディレクトリをトップ10で表示する」ための黄金パターンです。
  - `du -sh *`: 各ファイル/ディレクトリの合計サイズ (`-s`) を読みやすい形式 (`-h`) で表示。
  - `sort -rh`: 数値順 (`-n`) で逆順（大きい順、 `-r`）にソート。`-h` は `sort` 側にも必要。
  - `head -n 10`: 上位10件を表示。
- **シナリオ**: `df -h` で `/var` が一杯になっていることを確認した後、`/var` に移動してこのコマンドを実行。`/var/log` や `/var/cache` など、どのサブディレクトリが原因かを即座に特定します。
- **例**:

```bash
# /var ディレクトリで、最も容量を食っているサブディレクトリを探す
cd /var
sudo du -sh * | sort -rh | head -n 5
```

### 2\. ブルーチーム視点

- **組み合わせ**: `du -h /tmp`
- **説明**: `/tmp` ディレクトリなど、攻撃者がファイルを一時的に配置する可能性のある場所のサイズを確認します。
- **シナリオ**: インシデント調査中に、`/tmp` やWebサーバのアップロードディレクトリのサイズが異常に大きいことを発見した場合、それは攻撃者がツールキットをアップロードしたり、窃取したデータを圧縮して一時保管（ステージング）したりしている痕跡である可能性があります。`du` で異常を検知し、次に `ls -l` で具体的なファイルを探します。

### 3\. レッドチーム視点

- **組み合わせ**: `du -sh /`
- **説明**: 侵入したシステムの偵察（Reconnaissance）の一環として、主要なディレクトリのサイズを把握します。
- **シナリオ**: 攻撃者は `du -sh /var`, `du -sh /home`, `du -sh /opt` などを実行し、どのディレクトリに重要なデータ（バックアップ、ユーザーデータ、アプリケーションなど）が大量に保存されていそうかを推測します。これにより、データ窃取の標的を効率的に絞り込むことができます。

## エラーメッセージとトラブルシューティング

1. **現象**: **`df` の使用量と `du` で計算した `/` の合計サイズが一致しない。**

      - **考えられる原因**: これは正常な動作です。主な原因は、プロセスがファイルを開いたまま、そのファイルが `rm` で削除された場合です。`du` は存在しないファイルをカウントしませんが、`df` はプロセスがファイルハンドルを掴んでいる限り、ディスクブロックが解放されないため、使用中としてカウントします。
      - **解決策**: `sudo lsof | grep deleted` コマンドで、削除済みのファイルを開き続けているプロセスを特定し、そのプロセスを再起動するとディスクスペースが解放されます。

2. **現象**: **実行中に `du: cannot read directory ...: Permission denied` というエラーが多数表示される。**

      - **考えられる原因**: コマンドを実行しているユーザーに、スキャン対象のディレクトリへの読み取り権限がありません。
      - **解決策**: `sudo du ...` のように `sudo` を付けて実行するのが一般的です。エラーメッセージだけを非表示にしたい場合は `du -sh * 2>/dev/null` のように標準エラー出力をリダイレクトします。

## セキュリティに関する考慮事項

`du` は読み取り専用のコマンドですが、その出力は攻撃者にとっても防御者にとっても有益な情報となります。

- **偵察への利用**: 攻撃者は `du` を使い、システム上で価値のあるデータがどこに集積しているかを効率的に探し出します。
- **サービス妨害攻撃の検知**: 攻撃者がログを溢れさせたり、巨大なファイルを生成したりしてディスクを一杯にするDoS攻撃を仕掛けた場合、`du` はその攻撃によって書き込まれたデータがどこにあるかを特定するための主要なツールとなります。

## 注意点・補足

- **`--max-depth=1` の活用**: ルートディレクトリ (`/`) のような巨大なディレクトリで `du -h` を実行すると、膨大な数のサブディレクトリが表示されてしまいます。`du -h --max-depth=1 /` のように深さを1に限定することで、`/` 直下のディレクトリごとの合計サイズを簡潔に表示でき、調査の効率が上がります。
- **ハードリンク**: ファイルが複数のハードリンクを持つ場合、`du` はデフォルトでは各リンクを個別にカウントするため、実際のディスク使用量よりも大きく表示されることがあります。
