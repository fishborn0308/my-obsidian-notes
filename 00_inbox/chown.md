# `chown` - ファイルやディレクトリの所有者を変更する

## 概要

`chown` (change owner) は、ファイルやディレクトリの**ユーザー所有者**と**グループ所有者**を変更するためのコマンドです。Linuxのセキュリティモデルでは、ファイルへのアクセス権は所有者に基づいて制御されるため、`chown` は適切な権限管理を行う上で不可欠なツールです。例えば、ユーザーのホームディレクトリはそのユーザーが所有し、WebサーバのコンテンツはWebサーバのプロセスが所有する、といった設定に使われます。

このコマンドの実行には、通常 `root` 権限が必要です。

(出自: `coreutils` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`chgrp`** | ファイルの**グループ所有者**のみを変更します。`chown` はユーザーとグループの両方を変更できる、より包括的なコマンドです。 | `chown` |
| **`chmod`** | ファイルの**パーミッション（権限モード）**を変更します。`chown` は「誰が所有者か」を変更するのに対し、`chmod` は「その所有者が何ができるか」を変更します。 | - |

## よく連携されるコマンド

`chown` は、新しいユーザーを作成した後や、アプリケーションをデプロイした際に、`useradd` や `cp` と連携して適切な所有権を設定するためによく使われます。

### シナリオ例: 新しいユーザーのホームディレクトリを設定する (インフラ構築・運用視点)

* **目的**: `useradd` で新しいユーザーを作成した後、そのユーザーのホームディレクトリの所有権を正しく設定する。
* **連携コマンド**: `useradd`, `chown`
* **解説**: ユーザーが自身のホームディレクトリ内でファイルを自由に作成・編集できるように、ディレクトリ全体の所有者をそのユーザーに設定します。
* **コマンド例**:

    ```bash
    # user1 を作成後、ホームディレクトリの所有権を再帰的に設定する
    sudo useradd user1
    sudo chown -R user1:user1 /home/user1
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **ファイル/ディレクトリ操作** | |
| ⭐ `-R`, `--recursive` | ディレクトリとその中にある全てのファイル・サブディレクトリの所有者を**再帰的に**変更します。 |
| **出力制御** | |
| `-v`, `--verbose` | 処理した全てのファイルについて、変更内容を詳細に表示します。 |
| `-c`, `--changes` | 実際に所有者が**変更された**ファイルについてのみ表示します。 |
| **その他** | |
| `--from=<owner>:<group>`| 現在の所有者が指定と一致する場合にのみ、所有者を変更します。 |
| `--reference=<RFILE>`| 所有者を直接指定する代わりに、参照ファイル `<RFILE>` と同じ所有者に変更します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: Webサーバのドキュメントルート以下の所有者を再帰的に設定する。
* **組み合わせ**: `sudo chown -R <user>:<group> <directory>`
* **解説**: Webサーバプロセス (`www-data`など) がコンテンツファイルを読み書きできるように、ディレクトリ全体の所有者をWebサーバの実行ユーザーに設定します。
* **例**:

    ```bash
    # /var/www/my-app ディレクトリ以下の所有者を再帰的に www-data に設定
    sudo chown -R www-data:www-data /var/www/my-app
    ```

### 2. ブルーチーム視点

* **タスク**: 不適切な所有者を持つファイルを特定し、正規の状態に修復する。
* **組み合わせ**: `find / -nouser -o -nogroup` と `chown`
* **解説**: ユーザーやグループが削除された後に残る「孤立したファイル」を `find` で検索し、`chown` で `root` などの安全な所有者に変更します。これは、システムの健全性を保ち、意図しない権限の問題を防ぐための修復作業です。
* **例**:

    ```bash
    # 所有者ユーザーが存在しないファイルを検索し、所有者をrootに変更する
    sudo find / -nouser -exec chown root {} \;
    ```

### 3. レッドチーム視点

* **タスク**: 権限昇格後、ファイルの所有権を自身が制御するユーザーに奪取する。
* **組み合わせ**: `sudo chown attacker:attacker /path/to/file`
* **解説**:
    1. **永続化**: 攻撃者がroot権限でバックドアを設置した後、そのファイルの所有者をシステムに存在する一般ユーザーに変更することで、`root` 所有のファイルとして目立つのを避け、発見されにくくします。
    2. **権限昇格への布石**: SUID化したいバイナリの所有権を `root` に変更 (`sudo chown root:root my_backdoor`) してから、`chmod u+s` でSUIDビットを付与し、権限昇格用のバックドアを作成します。

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `chown: invalid user: ‘<username>’`
    * **考えられる原因**: 指定したユーザー名またはグループ名がシステムに存在しません。
    * **解決策**: `getent passwd` や `getent group` コマンドで、システムに存在するユーザー・グループの一覧を確認し、正しい名前を指定してください。

## 環境変数と設定ファイル

* `chown` の動作に直接影響する環境変数は通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `chown` 自体ではなく、**不適切な所有権の設定**がセキュリティリスクとなります。
* **悪用シナリオ**: システムファイルを一般ユーザーが所有していると、システムの改ざんが可能になってしまいます。

### GTFOBins / LOLBAS における利用例

* **機能**: **Privilege Escalation** (権限昇格)
* **解説**: もしユーザーが `sudo chown` を実行できる場合、そのユーザーはシステム上の**あらゆるファイルの所有権を奪取**できます。例えば、`/bin/bash` の所有権を自分に変更した上でSUIDビットを立てて、root権限のシェルを取得することが可能です。
* **参照**: `https://gtfobins.github.io/gtfobins/chown/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 最小権限の原則を徹底し、ファイル所有権を適切に設定する。`sudoers` の設定を厳格にし、`chown` のような強力なコマンドに安易に `sudo` 権限を付与しない。
* **Detection (検知)**: ファイル整合性監視 (FIM) ツールで、重要なシステムファイルの所有者変更を監視する。

## 注意点・補足

* **再帰的変更の注意 (`-R`)**: `-R` オプションは強力ですが、システムの重要なディレクトリ (`/` や `/usr` など) に対して誤って実行すると、システムが起動しなくなるなど、致命的な問題を引き起こす可能性があります。実行前に対象パスを十分に確認してください。
