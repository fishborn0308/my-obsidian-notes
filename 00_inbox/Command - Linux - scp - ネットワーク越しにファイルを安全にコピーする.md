# Command - Linux - scp - ネットワーク越しにファイルを安全にコピーする

## 概要
`scp` は、SSH (Secure Shell) を利用して、ローカルホストとリモートホスト間、あるいはリモートホスト同士でファイルをコピーします。通信内容はすべて暗号化されるため、パスワードやファイルの中身がネットワーク上で盗聴される心配がありません。



## 🛠 ファイル転送ツールの比較
| コマンド | 特徴 | 弱点 |
| :--- | :--- | :--- |
| **`scp`** | **非常にシンプル、設定不要（SSHがあればOK）。** | 中断再開ができない、差分転送がない。 |
| `rsync` | 高機能、差分転送、同期が得意。 | 初回の設定やオプションがやや複雑。 |
| `sftp` | インタラクティブな操作が可能。 | 1つだけサッと送るにはコマンド数が多い。 |

## 基本的な構文
```bash
scp [オプション] [送り元] [送り先]
````

リモートを指定する場合は `ユーザー名@ホスト名:パス` の形式を使います。

## 主要なオプション

|**オプション**|**説明**|
|---|---|
|⭐️ `-r`|**再帰的。** ディレクトリを丸ごとコピーする。|
|⭐️ `-P <port>`|**要注意。** SSH のポート番号を指定する（大文字の P）。|
|⭐️ `-p`|タイムスタンプや権限（パーミッション）を維持してコピーする。|
|⭐️ `-C`|データを圧縮して転送する（低速回線で有効）。|
|⭐️ `-i <key>`|認証に使用する秘密鍵ファイルを指定する。|

## コマンド例

Bash

```
# --- 基本的な使用方法 ---

# 1. ローカルのファイルをサーバーへ送る
scp my_document.pdf user@remote-server.com:/home/user/

# 2. サーバーのディレクトリをローカルへ持ってくる（再帰的）
scp -r user@remote-server.com:/var/www/html ./backup/

# 3. SSH ポート番号が 22 以外（例: 2222）の場合
scp -P 2222 script.sh user@backup-server.local:/tmp/


# --- 踏み台サーバー（Bastion）を経由する場合 ---

# 4. 踏み台を 1 台経由してターゲットへ送る
# 順序: [踏み台] を経由して [ターゲット] へ
scp -J user_bastion@bastion_ip local_file user_target@target_ip:/path/

# 5. 【多段接続】複数の踏み台を順番に経由して送る
# 順序: [踏み台A] -> [踏み台B] -> [ターゲット]
# -J オプション内でカンマ区切りで「左から順に」経由する順に記述します
scp -J user_a@bastion_a,user_b@bastion_b local_file user_target@target_ip:/path/
```

## ⚠️ 2026年の常識：SCP の「中身」が変わった？

長年、`scp` は独自のプロトコルを使っていましたが、セキュリティ上の理由から最新の OpenSSH では内部的に **SFTP プロトコル** を使うように変更されています。これにより、古い `scp` にあった脆弱性が解消され、より安全になっています。

## 業務シナリオ別活用法

### 1. インフラ構築・運用視点

- **タスク**: 設定ファイルの一時的な配布。
    
- **解説**: 新しいサーバーをセットアップする際、手元の `.vimrc` や `nginx.conf` をサッと放り込むのに最適です。
    

### 2. ブルーチーム視点

- **タスク**: 証拠保全（フォレンジック用データの回収）。
    
- **解説**: 侵害の疑いがあるサーバーからログファイル（`/var/log/auth.log` 等）を手元の安全なマシンへ暗号化して吸い上げます。
    

### 3. レッドチーム視点

- **タスク**: ツールのアップロードとデータの持ち出し。
    
- **解説**: 侵入に成功した後、自身の攻撃用ツールをターゲットに送り込んだり、奪取したパスワードリストを自分のサーバーへ転送したりするために使用します。OS 標準ツールであることが多いため、不自然なツールの導入を疑われにくいという利点があります。

## セキュリティに関する考慮事項

### 脆弱性とリスク

- **中間者攻撃 (MITM)**: 初めて接続するサーバーのフィンガープリント（指紋）をろくに確認せず `yes` を押すと、攻撃者のマシンに接続させられている可能性があります。
    
- **秘密鍵の管理**: `-i` で指定する鍵ファイルを適切に保護（`chmod 600`）していないと、`scp` 自体が実行を拒否します。
    

### 対応策・緩和策

- **SSH 鍵認証の利用**: パスワード入力ではなく、公開鍵認証を利用することで、ブルートフォース攻撃を防ぎ、自動化スクリプトでも安全に運用できます。
