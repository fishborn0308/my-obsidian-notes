# 第0章：サーバセキュリティのための基礎技術整理

## 1. 本章の目的と位置づけ

本章は、ペネトレーションテスター（Red Team）およびSOCアナリスト（Blue Team）が共通して持つべき**「攻撃の足場」と「検知の境界線」**を定義するための基礎章である。

SEC504（GCIH）レベルの実務においては、個別の脆弱性知識（Exploit）以上に、パケットの挙動やログの構造を正しく解釈する能力が求められる。本章で整理するネットワーク、認証、OSログの知識は、後続のWebサーバ、DBサーバ、AD環境への攻撃・防御シナリオにおいて、説明を省略するための「前提知識」として機能する。

---

## 2. ネットワーク基礎：パケットレベルの可視化

攻撃者は通信の「隙間」を突き、SOCは通信の「痕跡」を追う。

### 2.1 TCP/IPと3-way Handshake

攻撃者がポートスキャンを行う際、OSやツールの設定によりスキャン手法（SYN, ACK, FIN, NULL等）を使い分ける。

- **攻撃の視点：** * **SYNスキャン（ハーフオープン）：** 3-way handshakeを完了させずにRSTを送ることで、ターゲット側にアプリケーションログを残さずにポートの開閉を確認する。
    
    - **バナーグラビング：** 確立されたコネクション（ESTABLISHED）を通じて、サービス名やバージョン情報を取得する。
        
- **検知の視点：** * 同一IPからの短時間における大量のSYNパケット、および宛先ポートの分散をIDS/IPSで捉える。
    

### 2.2 可視化ポイント（Tap vs Span）

- **TAP（Test Access Point）：** 物理的に回線を分岐。パケットのドロップがなく、高負荷時も正確なキャプチャが可能。
    
- **SPAN（Mirroring）：** スイッチのポートを複製。設定が容易だが、高負荷時にパケットが欠落するリスクがあり、フォレンジック精度に影響する。
    

> **実務での注意点：** > ペンテスターは、ターゲット環境にIDSが存在する場合、スキャン速度を落とす（T1/T2設定など）ことで、スレッショルド（閾値）ベースの検知を回避しようとする。SOCは、単一の閾値だけでなく、長期間のトレンド分析が必要となる。

---

## 3. HTTP / HTTPSの仕組み：アプリケーション層の攻防

Webサーバへの攻撃は、すべてこのプロトコルの作法に従って行われる。

### 3.1 リクエストとレスポンスの構造

- **Method：** GET/POSTだけでなく、PUT（ファイルアップロード）、OPTIONS（許可メソッドの調査）が悪用される。
    
- **Header：** `User-Agent`（攻撃ツールの特定）、`X-Forwarded-For`（プロキシ背後の攻撃元特定）、`Referer`（攻撃の起点確認）が重要。
    

### 3.2 可視性の違い

- **HTTP：** 全てが平文。WAFやIDSでペイロード（SQLi, XSS等）を直接検知可能。
    
- **HTTPS：** ペイロードが暗号化されるため、ネットワーク型IDSでは攻撃内容が見えない。ホスト型（Webサーバログ）またはSSL復号化装置（TLS Inspection）での解析が必須。
    

### 3.3 実務ログ例（Webサーバアクセスログ）

Plaintext

```
192.168.10.55 - - [19/Feb/2026:09:00:01 +0900] "POST /login.php HTTP/1.1" 200 534 "https://target.com/index.php" "Mozilla/5.0 (sqlmap/1.5.2)"
```

- **SOCの着眼点：** `User-Agent` に `sqlmap` が含まれている。これは典型的な自動攻撃ツールの痕跡である。
    

> **実務での注意点：**
> 
> 攻撃者は `User-Agent` を一般的なブラウザ（Chrome等）に偽装する。SOCは、ヘッダの「値」だけでなく、ヘッダの「出現順序」や「欠落」から異常を判断する必要がある。

## 4. TLSの基礎：暗号化通信の解読と限界

### 4.1 ハンドシェイクとメタデータ

暗号化されていても、ハンドシェイク中の `Client Hello` に含まれる **SNI (Server Name Indication)** は平文であるため、アクセス先のドメイン名は特定できる。

### 4.2 証明書と不審な接続

- **自己署名証明書（オレオレ証明書）：** C2（Command and Control）サーバとの通信によく見られる。
    
- **JA3フィンガープリント：** クライアント側のTLS実装の癖をハッシュ化したもの。特定のマルウェアやハッキングツールを特定する強力な指標となる。
    

> **実務での注意点：**
> 
> TLS 1.3ではハンドシェイクの多くが暗号化されるため、1.2以前に比べてネットワーク側での解析難易度が上がっている。エンドポイント（ホスト）側での可視化を強化すべきである。

---

## 5. 認証と認可：アイデンティティの侵害

「誰であるか（認証）」と「何ができるか（認可）」の混同は、深刻な脆弱性を招く。

### 5.1 方式とリスク

- **Basic認証：** Base64エンコードのみで、平文と同等。
    
- **Session ID：** 固定化（Session Fixation）や奪取（Session Hijacking）により、ログイン後の権限を丸ごと乗っ取られる。
    
- **JWT (JSON Web Token)：** 署名検証の不備により、権限（Role）を書き換えられるリスクがある。
    

### 5.2 侵害時の影響

認証情報が漏洩すると、攻撃者は「正当な利用者」として振る舞うため、脆弱性を突く攻撃ログ（403 Forbidden等）が発生せず、検知が極めて困難になる。

> **実務での注意点：**
> 
> ペンテストでは、認証バイパス後の「権限昇格（Horizontal/Vertical）」を狙う。SOCは、同一アカウントによる「異常な時間帯」「異常なソースIP」「異常なアクセス頻度」を監視対象とする。

---

## 6. プロセスと権限モデル：OS内部の挙動

攻撃者がサーバに侵入（Initial Access）した後、最初に行うのが権限確認である。

### 6.1 Linux vs Windows

- **Linux：** UID/GIDによる管理。`root` (UID 0) が全能。ファイルパーミッション（rwx）と `sudo` の設定不備が狙われる。
    
- **Windows：** SID（Security Identifier）による管理。`SYSTEM` 権限や `Built-in Administrator`。トークン操作による権限昇格が一般的。
    

### 6.2 権限昇格のトリガー

- **SUID設定ファイル：** 一般ユーザーが実行しても所有者の権限で動くバイナリの悪用。
    
- **サービス設定不備：** 実行バイナリの書き込み権限が一般ユーザーにある場合、悪意あるプログラムに差し替えられる。
    

---

## 7. ログの概念と発生箇所

ログは以下の3層で多層的に捉える必要がある。

1. **ネットワークログ：** (Firewall, IDS/IPS, Proxy) 通信の「経路」を記録。
    
2. **OSログ：** (Auth, Syslog, EventLog) 「誰が」「どのプロセスで」何をしたかを記録。
    
3. **アプリログ：** (Web Access, DB Audit) アプリケーション固有の「操作内容」を記録。
    

---

## 8. Linuxログ体系：調査の起点

Linuxサーバの調査において、`/var/log/` 配下の理解は必須である。

- **syslog / messages：** システム全般の挙動。
    
- **auth.log / secure：** ログイン試行（成功・失敗）、sudo実行履歴。
    
- **auditd：** ファイル操作やシステムコールレベルの詳細な監査ログ。
    

### 8.1 実務ログ例（不審なログイン失敗）

Plaintext

```
Feb 19 09:10:05 srv-web01 sshd[1234]: Failed password for invalid user admin from 10.0.0.50 port 54321 ssh2
```

- **攻撃の視点：** 辞書攻撃やブルートフォース。
    
- **SOCの着眼点：** 短時間での連続した失敗、および実在しないユーザー名（invalid user）での試行。
    

> **実務での注意点：**
> 
> 熟練した攻撃者は、侵入後に `history -c` やログファイルの直接編集（消去）を試みる。ログは必ずリアルタイムで外部（SIEM等）へ転送しておく必要がある。

## 9. Windowsイベントログ：侵害拡大の検知

Windows環境では、特定のイベントIDを監視することが侵害検知の鍵となる。

### 9.1 主要ログ種別

- **Security：** ログオン/ログオフ、権限行使。
    
- **System：** サービスの開始・停止、ドライバのロード。
    
- **Application：** アプリのエラーや動作。
    

### 9.2 重要イベントID (Security)

|**ID**|**内容**|**攻撃者の挙動**|
|---|---|---|
|**4624**|ログオン成功|侵入成功、横展開（Lateral Movement）|
|**4625**|ログオン失敗|パスワードスプレー、辞書攻撃|
|**4672**|特権の割り当て|管理者権限への昇格|
|**4688**|プロセスの作成|コマンド実行（Whoami, Net user等）|

> **実務での注意点：**
> 
> イベントID 4624における「ログオンタイプ」に注目せよ。タイプ3（ネットワーク）は共有フォルダやリモート実行、タイプ10はリモートデスクトップ（RDP）を示す。
