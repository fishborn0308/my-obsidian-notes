# `docker stop` - 実行中のコンテナを停止する

## 概要

`docker stop` は、実行中の1つ以上のコンテナを**穏やかに (gracefully)** 停止させるためのコマンドです。まず、コンテナのメインプロセス（PID 1）に `SIGTERM` シグナルを送信し、アプリケーションがクリーンアップ処理（例: データの保存、コネクションのクローズ）を行うための時間を与えます。指定した時間内（デフォルトは10秒）にプロセスが終了しない場合、次に `SIGKILL` シグナルを送信して強制的にプロセスを終了させます。

(出自: `docker` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`docker kill`** | `SIGTERM` を送らず、最初から**強制的に** `SIGKILL` シグナルを送信します。`stop` で停止できない場合に使いますが、データ破損のリスクがあります。 | `docker stop` |
| **`docker restart`** | コンテナを**停止してから再度開始**（再起動）します。 | - |
| **`docker rm -f`** | 実行中のコンテナを**強制的に停止し、かつ削除**まで一度に行います。 | - |
| **`docker pause`**| コンテナ内の全プロセスを**一時停止 (suspend)** します。`stop` はコンテナを**終了 (exit)** させます。 | - |

## よく連携されるコマンド

`docker stop` は、停止対象のコンテナIDを特定するために `docker ps` と連携するのが一般的なワークフローです。

### シナリオ例: 実行中の全コンテナを安全に停止する (インフラ構築・運用視点)

* **目的**: Dockerホストのメンテナンス（再起動など）の前に、実行中の全てのコンテナを安全に停止する。
* **連携コマンド**: `docker ps`, `xargs`
* **解説**: `docker ps -q` で実行中のコンテナIDのみをリストアップし、それを `xargs` に渡して `docker stop` で一括停止します。
* **コマンド例**:

    ```bash
    # 実行中の全てのコンテナIDを取得し、一括で停止する
    docker ps -q | xargs docker stop
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **動作制御** | |
| ⭐ `<container...>` | 停止するコンテナの名前またはIDを一つ以上、スペース区切りで指定します。 |
| ⭐ `-t`, `--time <seconds>` | `SIGTERM` を送信してから、`SIGKILL` を送信するまでの**待機時間（秒）**を指定します。デフォルトは10秒です。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: アプリケーションのメンテナンスや、ホストの計画的なシャットダウンの前に、コンテナを安全に停止させる。
* **組み合わせ**: `docker stop <container_name_or_id>`
* **解説**: 新しいバージョンをデプロイする前に古いコンテナを停止したり、データベースコンテナを安全にシャットダウンしたりする際の基本的な使い方です。
* **例**:

    ```bash
    # WebサーバーとDBサーバーを安全に停止
    docker stop my-webapp my-database
    ```

### 2. ブルーチーム視点

* **タスク**: **インシデントレスポンスにおける封じ込め**。
* **組み合わせ**: `docker stop <compromised_container_id>`
* **解説**: 侵害されたコンテナを即座にネットワークから切り離し、マルウェアの活動を停止させます。`docker rm` とは異なり、コンテナのファイルシステムはそのまま残るため、後のフォレンジック調査のために**状態を保全する**ことができます。
* **例**:

    ```bash
    # 侵害されたコンテナ "suspicious-wordpress" の活動を停止
    docker stop suspicious-wordpress
    ```

### 3. レッドチーム視点

* **タスク**: **サービス妨害 (Denial of Service, DoS) 攻撃**。
* **組み合わせ**: `docker stop <critical_service_container>`
* **解説**: 攻撃者がDockerホストの制御を奪取した後、標的のビジネスに影響を与えるために、`database` や `auth-service` のような重要なコンテナを `docker stop` で停止させます。

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `Error response from daemon: No such container: <container_name>`
    * **考えられる原因**: 指定した名前またはIDのコンテナが存在しません。
    * **解決策**: `docker ps -a` を実行して、存在するコンテナの名前とIDを再確認してください。

## 環境変数と設定ファイル

* `docker stop` の動作に直接影響する環境変数や設定ファイルは通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `docker stop` はサービスを停止させるコマンドであり、**不適切な権限管理**がリスクとなります。
* **悪用シナリオ**: `docker` コマンドの実行権限を持つユーザーは、`docker stop` を使って意図的に、あるいは誤って重要なサービスを停止させ、サービス妨害 (DoS) を引き起こす可能性があります。

### GTFOBins / LOLBAS における利用例

`docker stop` は `docker` コマンドの一部として扱われます。`sudo docker` の権限があれば、任意のコンテナを停止可能です。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: Dockerソケットへのアクセス権限を厳格に管理する。
* **Detection (検知)**: `auditd` で `docker stop` コマンドの実行を監視し、計画外のサービス停止がないかを確認する。

## 注意点・補足

* **`stop` vs `kill`**: 可能な限り `docker kill` ではなく `docker stop` を使用してください。`stop` はアプリケーションに終了準備の時間を与え、データの整合性を保つための安全な方法です。`kill` は最後の手段と考えましょう。
