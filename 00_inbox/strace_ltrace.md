# OS  - Linux - strace vs ltrace - プロセスの動作を追跡・分析する

## 概要

`strace` と `ltrace` は、プログラムが実行中に何を行っているかを、低レベルで詳細に追跡（トレース）するための動的解析ツールです。アプリケーションのデバッグや、マルウェアの挙動分析など、プロセス内部の動きを理解するために不可欠なコマンドです。

* **`strace` (System Call Trace)**: プロセスが行う**システムコール**（カーネルへの要求）と、プロセスが受信する**シグナル**を追跡します。ファイルアクセス、ネットワーク通信、プロセス操作など、**OSとのやり取り**を監視します。

* **`ltrace` (Library Call Trace)**: プロセスが行う**共有ライブラリ**へのコール（関数呼び出し）を追跡します。`printf` や `strcpy` など、**アプリケーションがライブラリ関数をどのように利用しているか**を監視します。

(出自: `strace` / `ltrace` パッケージに含まれる)

## 類似コマンドと差異

| ツール | 追跡対象 | 抽象度 | 主な用途 |
| :--- | :--- | :--- | :--- |
| ⭐ **`strace`** | **カーネル**へのシステムコール | **低レベル** | ファイルアクセス、パーミッション、ネットワーク接続の問題解決 |
| ⭐ **`ltrace`** | **ユーザー空間**の共有ライブラリコール | **高レベル** | アプリケーションのロジック、ライブラリ使用法のデバッグ |
| **`gdb`** | プログラムの実行全般 | 対話的 | プログラムのステップ実行、メモリ検査など詳細なデバッグ |

## よく連携されるコマンド

これらのツールの出力は非常に膨大になるため、`grep` でフィルタリングしたり、`-o` オプションでファイルに保存して `less` で閲覧するのが一般的です。

### シナリオ例: アプリケーションのファイルアクセスを調査する (インフラ構築・運用視点)

* **目的**: アプリケーションが起動時にどの設定ファイルを、どの順序で読み込もうとしているかを確認する。
* **連携コマンド**: `strace`, `grep`
* **解説**: `strace` でアプリケーションを実行し、その出力の中からファイルを開くシステムコール (`openat`) を `grep` でフィルタリングします。これにより、設定ファイルの読み込みパスや、読み込みに失敗しているファイル (`ENOENT`) を簡単に特定できます。
* **コマンド例**:

    ```bash
    # my-app の実行中に行われる "openat" システムコールのみを表示
    strace -f -e trace=openat ./my-app 2>&1 | grep "myapp.conf"
    ```

## オプション説明 (共通)

`strace` と `ltrace` は多くのオプションを共有しています。

| オプション | 説明 |
| :--- | :--- |
| **プロセス指定** | |
| ⭐ `-p <PID>` | 既に実行中のプロセスにアタッチして追跡を開始します。 |
| ⭐ `-f` | 子プロセスやスレッドも追跡します (`fork`, `clone`, `vfork`)。**非常に重要です**。 |
| **出力制御** | |
| ⭐ `-o <file>` | 出力を画面ではなく、指定したファイルに書き込みます。 |
| ⭐ `-e <expr>` (`strace`)| `trace=open,read` のように、追跡対象のシステムコールを限定します。 |
| `-c` | 実行終了時に、各コールにかかった時間やエラー回数の統計サマリーを表示します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク (strace)**: 「Permission denied」エラーの根本原因を究明する。
* **組み合わせ**: `strace -e trace=open,openat,access -f /path/to/app`
* **解説**: ファイルアクセスに関連するシステムコールに絞り、出力の中から `EACCES (Permission denied)` を返す行を見つけることで、どのファイルへの権限が不足しているかを正確に特定します。

* **タスク (ltrace)**: アプリケーションがクラッシュする原因を調査する。
* **組み合わせ**: `ltrace /path/to/application`
* **解説**: クラッシュ直前に呼び出されたライブラリコールを確認します。例えば、`strcpy` の引数に異常に長い文字列が渡されていれば、バッファオーバーフローを疑うことができます。

### 2. ブルーチーム視点

* **タスク**: **動的マルウェア解析**。サンドボックス環境で、不審なバイナリを実行してその挙動を監視する。
* **`strace` の利用**: `sudo strace -f -o strace.log ./malware` を実行し、ファイル操作 (`openat`, `unlink`)、ネットワーク活動 (`connect`)、プロセス操作 (`execve`) を記録して分析します。
* **`ltrace` の利用**: `sudo ltrace -f -o ltrace.log ./malware` を実行し、暗号化ライブラリ (`libcrypto.so`) や通信ライブラリ (`libcurl.so`) の利用状況から、マルウェアの高レベルな目的（ランサムウェアか、C2通信か）を推測します。

### 3. レッドチーム視点

* **タスク**: **脆弱性分析とリバースエンジニアリング**。攻撃対象のアプリケーションの内部動作を理解する。
* **`strace` の利用**: SUIDビットが設定されたプログラムを `strace` で実行し、ハードコードされたパスや、予測可能な `/tmp` ファイルの利用がないか（シンボリックリンク攻撃の可能性）などを調査します。
* **`ltrace` の利用**: `ltrace` を使い、`strcpy` や `printf` といった、バッファオーバーフローやフォーマットストリング脆弱性の原因となりうる危険な関数が使われていないかを調査します。

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `strace: attach: ptrace(PTRACE_ATTACH, ...): Operation not permitted`
    * **考えられる原因**: `root` 権限なしで、自身が所有していないプロセスにアタッチしようとしました。または、`kernel.yama.ptrace_scope` の設定により `ptrace` が制限されています。
    * **解決策**: `sudo strace ...` のように `sudo` を付けて実行してください。

## 環境変数と設定ファイル

* `strace`/`ltrace` の動作に直接影響する環境変数や設定ファイルは通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `strace`/`ltrace` 自体ではなく、その出力内容が**情報漏洩**に繋がります。
* **悪用シナリオ**: `strace`/`ltrace` の出力には、ファイルパス、ネットワークアドレス、読み書きされたデータ（パスワードやキーを含む可能性）など、機密情報が豊富に含まれる可能性があります。

### GTFOBins / LOLBAS における利用例

* **機能**: **File Write**, **Privilege Escalation**
* **解説**: もしユーザーが `sudo strace` (または `ltrace`) を実行できる権限を持っている場合、`-o` オプションを使って任意のファイルを上書きできます。
* **参照**: `https://gtfobins.github.io/gtfobins/strace/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: `sudoers` の設定を厳格にし、`strace` のようなデバッグツールに `sudo` 権限を安易に与えない。`kernel.yama.ptrace_scope` を `1` 以上に設定し、`ptrace` の利用を制限する。
* **Detection (検知)**: `auditd` を使って `strace` コマンドの実行や `ptrace` システムコールの呼び出しを監視する。

## 注意点・補足

* **パフォーマンスへの影響**: プロセスをトレースすると、実行速度は**著しく低下します**。これらはデバッグや分析のためのツールであり、本番環境での常時監視には向きません。
