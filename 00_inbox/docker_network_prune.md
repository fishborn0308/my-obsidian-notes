# `docker network prune` - 未使用のDockerネットワークを一括削除する

## 概要

`docker network prune` は、Dockerホスト上に存在する**未使用のネットワーク**をすべて一度にまとめて削除するためのコマンドです。ここで言う「未使用」とは、現在どのコンテナからも接続されていないカスタムネットワークを指します。`docker compose down` を実行した後や、コンテナを個別に削除した際に残ってしまった孤立ネットワークなどをクリーンアップし、`docker network ls` の一覧を整理整頓するのに便利です。

デフォルトでは、実行する前に削除対象のネットワークを一覧表示し、実行してよいか確認を求めてきます。

(出自: `docker` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`docker network rm`** | **個別の**ネットワークを指定して削除します。`prune` は条件に合うものを**一括で**削除します。 | `prune` (一括削除) |
| **`docker system prune`**| ネットワークだけでなく、停止中のコンテナ、danglingイメージなど、**すべての種類の未使用リソース**を一括で削除します。 | - |

## よく連携されるコマンド

`docker network prune` は、クリーンアップ前に `docker network ls` で現状を確認するという流れで使われます。

### シナリオ例: 定期的な環境クリーンアップ (インフラ構築・運用視点)

* **目的**: 開発サーバーなどで不要になったネットワークを定期的に一括削除する。
* **連携コマンド**: `docker network ls`
* **解説**: まず `docker network ls` で現在のネットワーク一覧を確認し、意図せず重要なネットワークが「未使用」状態になっていないかを確認します。その後、`docker network prune` を実行してクリーンアップします。
* **コマンド例**:

    ```bash
    # 1. 現在のネットワーク一覧を確認
    docker network ls

    # 2. 未使用ネットワークの一括クリーンアップを実行
    docker network prune
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **動作制御** | |
| ⭐ `-f`, `--force` | 対話形式の確認プロンプト (`Are you sure ...?`) を表示せず、強制的に削除を実行します。 |
| **フィルタリング** | |
| `--filter <key>=<value>`| 削除対象とするネットワークをフィルタリングします。（例: `--filter "until=24h"` (24時間以上前に作成されたもの)） |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: CI/CDのビルドサーバーで、ジョブ実行後に残った不要なネットワークを自動でクリーンアップする。
* **組み合わせ**: `docker network prune -f`
* **解説**: スクリプト内で `-f` (`--force`) オプションを使い、確認プロンプトなしで未使用ネットワークを削除します。これにより、CIジョブごとに作成されたネットワークがサーバーに溜まっていくのを防ぎます。
* **例**:

    ```bash
    # CI/CDパイプラインのクリーンアップステップに組み込む
    docker network prune -f
    ```

### 2. ブルーチーム視点

* **タスク**: インシデント対応後のクリーンアップ作業。
* **組み合わせ**: `docker network prune`
* **解説**: 侵害されたコンテナや、攻撃者が作成した可能性のある不審なコンテナをすべて削除した後、このコマンドを実行して、それらのコンテナだけが接続していた可能性のある孤立ネットワークも残さず削除します。

* **タスク**: 痕跡消去 (Anti-Forensics)。

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **現象**: **削除されると思っていたネットワークが `prune` の対象にならない。**
    * **考えられる原因**: そのネットワークに、まだコンテナが接続されています。コンテナは**停止 (`Exited`) 状態**であっても、ネットワークに接続されていると見なされます。
    * **解決策**: `docker network inspect <network_name>` を実行して、`Containers` セクションにコンテナが残っていないか確認します。残っている場合は、`docker rm` で先にコンテナを削除してから、再度 `prune` を実行してください。

## 環境変数と設定ファイル

* `docker network prune` の動作に直接影響する環境変数は通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **意図しないサービス妨害**。
* **悪用シナリオ**: `docker network prune` は基本的に安全ですが、運用フローによっては意図しないサービス妨害に繋がる可能性があります。例えば、「コンテナを一時的に全て削除し、すぐに再作成する」バッチ処理の途中で `prune` が実行されると、再作成されるコンテナが接続するはずだったネットワークが削除されてしまう可能性があります。

### GTFOBins / LOLBAS における利用例

`docker network prune` は `docker` コマンドの一部として扱われます。`sudo docker` の権限があれば、任意のネットワーク操作が可能です。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: Dockerリソースを削除できるコマンドの実行権限は慎重に管理する。
* **Detection (検知)**: `auditd` で `docker network prune` の実行を監視し、計画外のクリーンアップ操作がないかを確認する。

## 注意点・補足

* **安全な設計**: `prune` コマンドは、現在アクティブな（コンテナが接続されている）ネットワークには影響を与えないように設計されています。そのため、定期的なクリーンアップスクリプトに組み込んでも、稼働中のサービスに影響を与えることは通常ありません。
