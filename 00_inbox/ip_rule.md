# `ip rule` - ルーティングポリシーデータベースを管理する

## 概要

`ip rule` は、`ip` コマンドスイートの一部であり、**ルーティングポリシーデータベース (RPDB)** を操作するためのコマンドです。通常のルーティング (`ip route`) がパケットの**宛先IPアドレス**のみを見て、単一のメインルーティングテーブルから経路を決定するのに対し、`ip rule` を使うと、より複雑な**ポリシーベースルーティング (Policy-Based Routing)** を実現できます。

これにより、「**送信元IPアドレス**が `10.0.1.10` ならばテーブルAを使え」「ファイアウォールで**特定のマーク**が付与されたパケットは、テーブルBを使え」といった、柔軟な経路制御が可能になります。

(出自: `iproute2` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`ip route`** | **ルーティングテーブルの中身**（経路情報）を管理します。`ip rule` は、どのパケットを**どのルーティングテーブルで処理するかを決定する**ための、より上位の「交通整理のルール」を管理します。 | - |
| **`iptables` (MARK)**| ファイアウォールは、パケットに**マーク**を付けることができます。`ip rule` は、`fwmark` セレクタでこのマークを検知し、マークされたパケットを特定のルーティングテーブルに振り分けることができます。 | - |

## よく連携されるコマンド

`ip rule` は、`ip route` と `/etc/iproute2/rt_tables` ファイルと連携して、完全なポリシーベースルーティングを構成します。

### シナリオ例: ソースベースルーティングを設定する (インフラ構築・運用視点)

* **目的**: サーバーが2つのインターネット回線に接続されている場合に、特定の送信元IPからのパケットを、特定の回線から送信させる。
* **連携コマンド**: `ip route`, `/etc/iproute2/rt_tables`
* **解説**: まずカスタムのルーティングテーブルを定義し、`ip route` でそのテーブルに経路を追加します。最後に `ip rule` で、どのようなパケットをそのカスタムテーブルで処理するかのルールを追加します。
* **コマンド例**:

```bash
# 1. カスタムテーブルを定義 (/etc/iproute2/rt_tables に追記)
echo "100 isp2" | sudo tee -a /etc/iproute2/rt_tables

# 2. isp2テーブルに、ISP2のゲートウェイをデフォルトルートとして追加
sudo ip route add default via 192.168.2.1 dev eth1 table isp2

# 3. eth1のIPアドレス (192.168.2.100) から送信されるパケットは、isp2テーブルを使うようルールを追加
sudo ip rule add from 192.168.2.100/32 table isp2 priority 1000
```

## オプション説明 (`ip rule` のサブコマンド)

| オプション | 説明 |
| :--- | :--- |
| **表示** | |
| ⭐ `show` (or `list`, `ls`) | (デフォルト) ルーティングポリシーデータベースのルールを一覧表示します。 |
| **設定** | |
| ⭐ `add` | 新しいルールを追加します。 |
| `del` | 既存のルールを削除します。 |
| `flush` | 全てのルールを削除します。 |
| **ルールの要素** | |
| `from <prefix>` | 送信元IPアドレス（セレクタ）を指定します。 |
| `to <prefix>` | 宛先IPアドレス（セレクタ）を指定します。 |
| `fwmark <mark>`| ファイアウォールマーク（セレクタ）を指定します。 |
| `table <table>` | 使用するルーティングテーブル（アクション）を指定します。 |
| `priority <prio>` | ルールの優先度を指定します。数値が小さいほど先に評価されます。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: サーバーが複数のインターネット回線を持つ（マルチホーム）構成で、通信経路を制御する。
* **組み合わせ**: `ip rule add from <source_ip> table <table_name>`
* **解説**: 上記の「よく連携されるコマンド」シナリオの通り、ソースベースルーティングは `ip rule` の最も代表的なユースケースです。これにより、ISPから来た通信の戻りパケットも、同じISPから出ていくように制御し、非対称ルーティングを防ぎます。
* **例**:

```bash
# 現在のルーティングポリシーを確認
ip rule show
```

### 2. ブルーチーム視点

* **タスク**: ルーティングポリシーが改ざんされていないかを確認する。
* **組み合わせ**: `ip rule show`
* **解説**: インシデント調査対象のホストで `ip rule show` を実行し、見慣れないルールが追加されていないかを確認します。特に、優先度の高い（数値の小さい）ルールで、特定の通信が不審なルーティングテーブルに送られるように設定されている場合、攻撃者による通信のハイジャックや盗聴を強く疑います。
* **例**:

```bash
# ルーティングポリシーの一覧を表示して監査する
ip rule show
```

### 3. レッドチーム視点

* **タスク**: **隠密な中間者攻撃 (Stealthy Man-in-the-Middle)** を行う。
* **組み合わせ**: `sudo ip rule add from <target_ip> table <attacker_table>`
* **解説**: root権限を奪取した後、メインのルーティングテーブルを変更せずに、特定のサーバーからの通信だけを傍受します。攻撃者はカスタムテーブルを作成し、そこに自身のマシンをゲートウェイとするルートを追加します。そして `ip rule` で、標的サーバーからのパケットだけをそのカスタムテーブルに振り分けるルールを追加します。
* **例**:

```bash
# 10.0.0.50 からのパケットを、テーブル 'hijack' で処理するようにルールを追加
sudo ip rule add from 10.0.0.50/32 table hijack priority 500
```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **現象**: **ルールを追加したが、期待通りに動作しない。**
    * **考えられる原因**: ルールの優先度 (priority) が適切でなく、より優先度の高い別のルールに先にマッチしているか、ルールが指し示しているルーティングテーブルの中身が間違っています。
    * **解決策**: `ip rule show` でルールの順序を確認します。`ip route show table <table_name>` で、対象テーブルの経路設定が正しいかを確認してください。

## 環境変数と設定ファイル

* `ip rule` の動作に直接影響する環境変数は通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

### 関連する設定ファイル

* **`/etc/iproute2/rt_tables`**: カスタムルーティングテーブルの名前と番号をマッピングするためのファイル。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `ip rule` 自体ではなく、**ルーティングポリシーの改ざん**がリスクとなります。
* **悪用シナリオ**: root権限を持つ攻撃者は、`ip rule` を使ってルーティングポリシーを自由に書き換えることができます。これにより、システムの通信を自身の制御下にあるサーバーにリダイレクトし、盗聴や改ざんを行うことが可能になります。

### GTFOBins / LOLBAS における利用例

* **機能**: **Privilege Escalation**
* **解説**: もしユーザーが `sudo ip` を実行できる権限を持っている場合、`-f` オプションでスクリプトファイルを読み込ませる機能などを悪用して、任意のコマンドを実行することが可能です。
* **参照**: `https://gtfobins.github.io/gtfobins/ip/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 最小権限の原則に従い、一般ユーザーに `ip` コマンドの `sudo` 権限を安易に与えない。
* **Detection (検知)**: `auditd` を使って、`netlink` 経由でのネットワーク構成の変更を監視する。`ip rule` の出力を定期的に取得し、ベースラインと比較して意図しない変更がないかを確認する。

## 注意点・補足

* **設定の永続化**: `ip rule` コマンドで行った設定変更は、デフォルトでは**OSを再起動すると失われます**。設定を永続化させるには、`systemd-networkd` の `.network` ファイルや、旧来の `ifupdown` スクリプトなど、ディストリビューションのネットワーク管理設定にルールを記述する必要があります。
