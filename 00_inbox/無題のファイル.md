# Command - Linux - bg - 中断されたジョブをバックグラウンドで再開する

## 概要
`bg` コマンドは、`Ctrl+Z` などで一時停止（suspend）されたジョブを、バックグラウンド状態で実行再開させるためのシェル内蔵コマンドです。これにより、現在のターミナルセッションを占有することなく、別の作業を並行して行えるようになります。
(出自: `Bash / Zsh 等のシェル内蔵機能`)



## 類似コマンドと差異
| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `fg` | バックグラウンドや停止中のジョブを**フォアグラウンド**（最前面）に戻す。 | 再び対話操作が必要な時 |
| `&` (アンパサンド) | コマンド実行時に末尾に付け、最初からバックグラウンドで起動する。 | 実行前から時間がかかると分かっている時 |
| `jobs` | 現在のシェルセッションで管理されているジョブの一覧を表示する。 | ジョブ番号を確認する時 |
| `nohup` | ログアウト後もプロセスを維持する。`bg` はセッションが切れると終了する。 | 長時間のバッチ処理 |
| `screen / tmux` | ターミナルセッション自体を維持・多重化する。 | リモート作業の標準 |

## よく連携されるコマンド
### シナリオ例: 誤ってフォアグラウンドで開始したバックアップの退避 (運用視点)
* **目的**: ターミナルを占有してしまった重い処理を、実行を止めることなくバックグラウンドへ移す。
* **連携コマンド**: `Ctrl+Z`, `jobs`
* **解説**: `Ctrl+Z` で一度プロセスをサスペンド（一時停止）させ、`bg` で再開させます。
* **コマンド例**:
    ```bash
    # 1. 重い処理を開始（&を付け忘れた）
    tar -zcvf backup.tar.gz /large/directory
    # 2. [Ctrl+Z] を押して一時停止
    # [1]+  Stopped                 tar -zcvf backup.tar.gz /large/directory
    # 3. バックグラウンドで再開
    bg %1
    ```

## オプション説明
| オプション | 説明 |
| :--- | :--- |
| **基本操作** | |
| ⭐️ `bg` (引数なし) | 直前に停止したジョブ（現在のジョブ）をバックグラウンドで再開する。 |
| ⭐️ `bg %<job_id>` | 指定したジョブ番号（`jobs` で確認可能）のジョブをバックグラウンドで再開する。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点
* **タスク**: 大規模なログ転送やファイル圧縮の並列実行。
* **組み合わせ**: `Ctrl+Z` -> `bg`
* **解説**: 1つのターミナルで複数のバッチ処理を次々と開始し、すべてをバックグラウンドで回しながら自分は別の調査を続けます。
* **例**:
    ```bash
    # 処理Aを開始 -> Ctrl+Z -> bg
    # 処理Bを開始 -> Ctrl+Z -> bg
    # 状況確認
    jobs
    ```

### 2. ブルーチーム視点
* **タスク**: 調査中に誤って起動したリソース消費の激しいプロセスの管理。
* **組み合わせ**: `bg` 実行後の `disown`
* **解説**: 調査ツール（スキャナ等）をバックグラウンドへ移した後、誤ってターミナルを閉じてもプロセスが死なないように `disown` でシェル管理から切り離します。
* **例**:
    ```bash
    bg %1
    disown %1
    ```

### 3. レッドチーム視点
* **タスク**: ペネトレーションテスト中のリスナー管理や、簡易的な C2 接続の維持。
* **組み合わせ**: `nc -lvnp` 等との併用
* **解説**: Netcat でリバースシェルを待機している際、一度ジョブをバックグラウンドに追いやってから別のエクスプロイトコードを実行するなどのマルチタスクに使用します。
* **例**:
    ```bash
    # リスナーを待機させておく
    nc -lvnp 4444
    # (シェルが取れた後) Ctrl+Z でサスペンド
    bg
    # メインのターミナルに戻り、別の作業を行う
    ```

## エラーメッセージとトラブルシューティング

1.  **エラーメッセージ例 1**: `bash: bg: job 1 already in background`
    * **考えられる原因**: 指定したジョブは既にバックグラウンドで実行中である。
    * **解決策**: 何もしなくてよい。必要なら `jobs` で状態を確認する。

2.  **エラーメッセージ例 2**: `bash: bg: no such job`
    * **考えられる原因**: ジョブ番号が間違っている、またはプロセスが既に終了している。
    * **解決策**: `jobs` コマンドで有効なジョブ番号を確認する。

## 環境変数と設定ファイル
* `bg` 自体に影響を与える設定ファイルはありませんが、シェルの `set -m` (monitor mode) が有効である必要があります（対話型シェルでは通常デフォルトで有効）。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例
* **悪用シナリオ**: 攻撃者が侵入後、バックグラウンドでマイニングツールやスニッファを動かす際、`&` を使う代わりに `bg` で隠蔽（目視の回避）を試みることがあります。ただし、`ps` コマンド等からは隠せません。

### GTFOBins における利用例
`bg` 自体はバイナリではない（builtin）ため GTFOBins には登録されていませんが、ジョブ制御全般は **Post-Exploitation** での効率的な作業に不可欠な知識です。

### 対応策・緩和策 (ブルーチーム視点)
* **Detection (検知)**: ユーザーが不自然に多くのプロセスをサスペンド・バックグラウンド化していないか、`jobs` コマンドの履歴やプロセスの親子関係（PPIDがシェルになっているもの）を監視します。

## 注意点・補足
* **標準出力の回り込み**: `bg` で実行しても、そのプロセスが標準出力 (`stdout`) を持っている場合、ターミナルに文字が流れ続けてしまいます。これを防ぐには最初から `> /dev/null 2>&1` でリダイレクトしておく必要があります。
* **入力待ちでの停止**: バックグラウンドに移したプロセスがユーザー入力を求めた場合、そのジョブは `Stopped (tty input)` 状態で停止します。その場合は `fg` で戻して入力する必要があります。
