# Scanning and Initial Access

## Version Scanning with Nmap and Netcat

### Version Scanning

• Nmapが開放ポートを検出すると、そのポートに一般的に関連付けられるデフォルトサービスを表示します
− nmap-servicesファイルに基づく
− では他のサービスはどうでしょうか？
• 非標準ポート上のサービスはどうでしょうか？
− 例：TCP 90上のWebサーバーやTCP 3322上のsshd
• ターゲットがリッスンしているサービスは、どのサービスとプロトコルバージョンを使用しているのでしょうか？
Nmapのバージョンスキャンが答えを出します。

開いているポートのリストがある場合、実際にそれらのポートを使用しているサービスを特定する必要があります。
これを行う簡単な（かつ自動的な）方法は、単にポートに関連付けられた一般的なサービスを調べるだけです。
ポート番号とサービスの対応関係は複数の場所で確認できます。ほとんどのUNIXおよびLinuxシステムには、この形式の基礎情報が記載された/etc/servicesファイルが含まれています。より多くのポートと詳細情報はnmap-servicesファイルで入手可能です。このファイルには約6,400の一般的なサービスと、それらが使用する既知のポートが記載されています。Nmapはデフォルトで出力表示時にこのファイルを自動的に参照します。
インターネット番号割り当て機関（IANA）が管理する公式ポート割り当てリストも
https://www.iana.org/assignments/service-names-portnumbers/service-names-port-numbers.xhtml で参照可能です。
ただし、こうした一般的なリストを検索することは有用ですが、限界があります。よく知られたサービスが
そのよく知られたポート上にない場合もあるからです。例えば、これらのリストのいずれも含まれていない
奇妙なポートでリスニングしているサービスは何か？さらに、管理者が一般的なサービスを
通常とは異なるポートに設定した場合、例えばWebサーバーをTCPポート90でリスニングさせたり、
sshdをTCP 3322で設定したりした場合はどうなるでしょうか？一般的なサービスが一般的なポートを使用している場合でも、
そのサービスが実行しているバージョンや通信プロトコルのバージョン番号を把握することは有用です。
これらの疑問のそれぞれに対応するのが、バージョンスキャンと呼ばれるNmapの別の有用な機能です。

### Nmap Version Scanning Functionality

• –sV オプションで実行されるバージョンスキャン
− TCP 接続を確立し、6 秒間待機…サーバーのバナーがバージョン正規表現に一致した場合：完了！
− TCP/UDPポートへプローブ送信。応答を誘発する設計のデータを送信し
サービス種別を判定
• nmap-service-probesファイルに1,000以上のサービスフィンガープリント収録
− TCPポート経由でTLSハンドシェイクを試行。成功時はTLS接続経由でプローブ
（HTTPSはTLS上のHTTP、stelnetはTLS上のtelnet）
− RPCサービス使用の有無を判定するためNull RPCコマンドを発行
• Nmapがサービスを識別できない場合、Netcatを使用できることがある
6
OSフィンガープリンティング、バージョンスキャン、デフォルトスクリプトによるスクリプトスキャン、
tracerouteを実行するには-Aを使用（つまり-A = -O + -sV + -sC + --traceroute）

Nmapのバージョンスキャン機能を呼び出すには、-sVオプションを使用します。あるいは、-Aオプション付きで実行すると、OSフィンガープリンティング、バージョンスキャン、スクリプトスキャン、Nmapのトレースルートを実行します。Nmapの演算式では、-A = -O + -sV + -sC + --traceroute と見なせます。つまり、-Aオプション付きでNmapを実行することは、-O（OSフィンガープリンティング）、-sV（バージョンスキャン）、-sC
（「デフォルト」カテゴリのNmapスクリプティングエンジンスクリプトを実行）、--traceroute（Nmapのトレースルート機能を使用）
オプションを同時に実行するのと同じです。
Nmapのバージョンスキャン機能は、ターゲットのポートスキャンが完了した後に実行されます。
発見された各オープンポートについて、Nmapはそのポートをプローブし、何がリスニングしているかを特定しようと試みます。
TCPポートの場合、Nmapはスリーウェイハンドシェイクでポートに接続し、応答を待ちます。応答が接続経由で到着した場合、Nmapはその応答をnmap-service-probesファイルで照合します。一致が見つかった場合、Nmapはサービスに関する情報を出力します。確固たる一致が見つからない場合、Nmapはポートのさらなるプローブを開始します。
開いているTCPおよびUDPポートに対して、Nmapはnmap-service-probesファイルで定義された様々なパケットを送信してターゲットをプローブします。このファイルには1,000以上のシグネチャが含まれており、ネットワーク動作に基づく様々なターゲットサービスの特定に極めて有用です。Nmapはまた、開いているTCPポート上でSSLハンドシェイクを試行し、SSLがサポートされている場合、SSL接続を介してターゲットポートをプローブしバージョン情報を取得します。Nmapはさらに、リスニングポートに対してヌルリモートプロシージャコール（RPC）
コマンドを送信し、リスニングポートに対して、そのマシン上でRPCサービスが使用する動的ポートに関する詳細情報を提供するポートマッパーアプリケーションを発見したか、あるいは特定のRPCベースのサービスを発見したかを判断します。
--version-traceオプションを指定して実行すると、Nmapはバージョンプローブの各ステップを出力に表示し、ユーザーがターゲットサービスの特定方法をリアルタイムで把握できるようにします。

### Netcat for the Pen Tester

![[スクリーンショット 2026-03-03 14.12.55.png]]

• Netcat: 汎用TCP/UDPネットワークツール
− 多くのLinuxディストリビューションに組み込まれ、Windowsでも利用可能
− NmapにはNetcatの機能に加え、SSLや多重化など追加機能を備えたncatが含まれる
− socatはさらに多くの機能を持つが、Linuxシステムにインストールされていることは稀
• ncは標準入力(STDIN)を受け取り、ネットワーク経由で送信する
• ネットワークからデータを受信し、標準出力（STDOUT）に出力する
• Netcat自身からのメッセージは標準エラー（STDERR）に出力される

Netcatは、Linux/UNIXおよびWindows向けの汎用TCP/UDPネットワークツールであり、指定されたTCPまたはUDPポートへのデータ送信、または指定されたTCP/UDPポートでのデータ受信待機を行います。機能面では基本的にこれだけです。しかし、この基本的な機能により、Netcatはペネトレーションテスターや倫理的ハッカーが日常的に直面するあらゆる種類のネットワーク関連タスクに活用できます。Netcatは
様々な形態で利用可能です。本講座で扱う最も一般的な形態は、多くのLinuxディストリビューションに
デフォルトでインストールされているものです。Windows向けの優れたNetcatバージョンも存在し、
本講座ではこれについても解説・使用します。Nmap開発チームはNetcatの機能の大半をncatというツールに
再実装しており、SSL暗号化機能も含まれています。
Netcatは標準入力から受け取ったあらゆるデータをネットワーク経由で送信します。標準入力とはキーボード、ファイルからのリダイレクト（ncオプション<ファイルのように<で標準入力をリダイレクト）、または他のプログラムからのパイプ入力（プログラム | ncオプションのように|でパイプ）を指します。
Netcatがネットワークからデータを受信すると、それを標準出力に配置します。標準出力は画面、ファイルへのリダイレクト（ncオプション > ファイルのように標準出力をリダイレクトする>を使用）、または他のプログラムの標準入力への送信となります。Netcatの標準出力を他のプログラムの標準入力に送信するには、2つの方法があります。一つ目は、ncオプション | プログラムのように|記号を用いた単純なパイプ処理です。これによりNetcatの出力が
即座にプログラムへストリーミングされ、プログラムは直ちに実行されます。
もう一つの方法は、-e program オプション付きで Netcat を使用することです。これにより、Netcat は
接続が確立された後（TCP の場合）またはデータが到着した後（UDP の場合）にのみプログラムを実行します。
さらに、-e オプションは、Netcat がネットワーク上で受信した内容をプログラムの標準入力に渡すだけでなく、
プログラムの標準出力を Netcat 経由でネットワーク上に送信する効果もあります。Netcatの重要な特性の一つは
標準エラーの扱いにある。Netcatがネットワーク上で実行している内容に関連するメッセージは全て
標準エラーに出力される。この形式のNetcatコメントを読み取り対話することは
後述するように有用である。

### Netcat Command Flags

![[スクリーンショット 2026-03-03 14.14.22.png]]

Netcatで最も重要なコマンドラインオプションは以下の通りです。他にも（多くの）オプションは存在しますが、これらを理解しておけば、約95%のケースでNetcatの使用状況を診断するのに役立ちます。フォーマットは次の通りです：
nc [オプション] [ターゲットIP] [リモートポート]
targetIPは単に相手側のIPアドレスまたはドメイン名です。クライアントモードでは必須です（接続先をクライアントに通知する必要があるため）、リスンモードではオプションです。
• -n: 相手側マシンの名前に対してDNSルックアップを実行しない
• -v: 詳細モード。接続発生時などのメッセージを標準エラー出力に表示
• -l: リスンモード (デフォルトはクライアントモード)
• -p: ローカルポート (リスンモードでは待機ポート、クライアントモードでは送信パケットの送信元ポート)
• -e: 接続確立後に実行するプログラム。標準入力と標準出力をプログラムに接続
• -z: ゼロI/Oモード (データ送信せず、ペイロードなしのパケットを送信)
• -wN: 接続タイムアウト、N秒待機。このオプション付きNetcatクライアント/リスナーはN秒間接続を試行。
   接続が成立しない場合、Netcatは動作を停止。接続が成立した場合、Netcatはデータ送受信を行う。その後、
   標準入力が合計N秒間閉じられた時点でNetcatは動作を停止。
• -u: UDPモード（デフォルトはTCP）

### Some Netcat Uses: Banner Grabbing

![[スクリーンショット 2026-03-03 14.15.25.png]]

• Netcatを使用してサービス情報を取得するか、サービスバージョン情報を確認する
• 応答を得るために接続文字列を入力する必要がある場合があります

これらのコマンドフラグについて簡単に説明したので、ペネトレーションテスター向けのNetcatの実用的な活用例を見ていきましょう。
接続時にサービスが提示する接続文字列を収集するには、以下の構文でNetcatクライアントをターゲットサービスに接続するだけです：
$ nc targetIP remote_port
一部のサービスは接続直後に、サービス種別・バージョン番号・プロトコルを含むバナーを表示します。他のサービスでは、この情報を返すために特定の文字列入力が必要です。サービスによってはEnterキーを2回押すだけで応答が得られる場合もあります。ターゲットサービスがHTTPを扱う場合、以下のコマンドで接続文字列を取得できます：
HEAD / HTTP/1.0 (Enterキーを2回押す)
スライドのスクリーンショットでは、Netcatを使用してTCPポート22（Secure Shellで一般的に使用されるポート）の10.130.10.10に接続しました。接続後、何も要求せずにターゲットがSSHのバージョンを通知します。CTRL-Cを押してNetcatが接続を切断させます。次にNetcatでTCPポート25上の10.130.10.25に接続します。対象はMicrosoftメールサービス（Exchange）を実行中であることを通知します。
TCPポート80で10.130.10.10に接続した。直ちに何も表示されなかったため、HEAD / HTTP/1.0と入力し、Enter、Enterを押した。システムはNginxを実行していること、そのバージョン番号、および基盤となるオペレーティングシステムの種類を伝えた。これらの接続文字列は攻撃者を欺くために変更できるが、通常は真実を伝える。

### Automating Banner Grabbing

![[スクリーンショット 2026-03-03 14.16.19.png]]

もちろん、1台のターゲットマシンで1つのポートをテストするのは有用ですが、ターゲットポートの範囲に対してこれを自動化したい場合もあるでしょう。Netcatはこのような機能をサポートしており、remote_port(s)オプションでx-y形式で指定された数値範囲を受け付けます。この設定により、Netcatはポートyから接続を試み始め、1ずつ減算しながらポートxまで順に接続を試みます。-rフラグは指定範囲のポートをランダムに処理しますが、わずかにステルス性を高めたい場合のみ使用します。
以下のコマンドでポート範囲から接続文字列を収集できます：
$ echo "" | nc -nvw2 [ターゲットIP] [ポート範囲]
これにより標準出力に何も出力せず、Netcatクライアントへパイプします。何も出力しないことで標準入力の強制終了を促します。Netcatの待機オプション(-wN)は、標準入力に情報がない状態でも開いたポートをN秒間待機することを覚えておいてください。このecho ""を行わないと、Netcatクライアントは最初の開いたポートでフリーズし、キーボードからの標準入力を永遠に待ち続けることになります。そのため、標準入力を閉じるために意図的に何も出力しないのです。この動作はまもなく実習で確認できます。Netcatクライアント(nc)に何も出力せず、接続確立を可視化するため詳細出力(-v)を有効化。出力の煩雑化を防ぐため名前解決を無効化(-n)。ターゲットIPアドレスのターゲットポート範囲に対し、接続確立まで最大2秒待機または接続成立後2秒待機(-w2)します。スライドのスクリーンショットでは、10.130.10.10をスキャン対象に設定し、データが返ってこない興味深いリスニングポート（TCP 80）と、データが返ってくるポート（TCP 22、21）を発見したことが確認できます。
