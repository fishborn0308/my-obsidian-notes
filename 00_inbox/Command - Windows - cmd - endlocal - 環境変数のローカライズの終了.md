# Command - Windows - cmd - endlocal - 環境変数のローカライズの終了

## 概要

`endlocal` コマンドは、バッチファイル内で `setlocal` コマンドによって開始された環境変数の変更（ローカライズ）の有効範囲を終了させます。
`setlocal` 以降に行われた環境変数の追加や変更は、`endlocal` が実行された時点で破棄され、`setlocal` 実行前の状態に復元されます。これにより、スクリプト内の一時的な変数がシステム全体の環境に影響を与えるのを防ぎます。

(出自: `Windows 標準搭載 - cmd.exe 内蔵コマンド`)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 (あれば) |
| :--- | :--- | :--- |
| `setlocal` | 環境変数のローカライズを開始する。必ず `endlocal` とセットで使用する。 | セット利用 |
| `exit /b` | バッチファイルを終了する。実行時に暗黙の `endlocal` が行われる。 | 終了時 |
| `set` | 変数を直接操作する。`setlocal` なしで使うと環境変数は永続的に残る。 | - |

## よく連携されるコマンド

### シナリオ例: 一時的な環境パスの追加と自動復元 (インフラ構築・運用)

* **目的**: 特定のツールを実行するために一時的に `PATH` を追加するが、処理終了後は元の `PATH` に戻す。
* **連携コマンド**: `setlocal`, `set`, `endlocal`
* **解説**: `setlocal` で現在の環境を保護し、`endlocal` で元の状態に戻します。
* **コマンド例**:
    ```cmd
    @echo off
    REM 現在の環境を保存
    setlocal
    REM PATH を一時的に追加
    set PATH=%PATH%;C:\temp\custom_tools
    call my_tool.exe
    REM PATH の変更を破棄し、元の環境に復元
    endlocal
    ```

## スイッチ/オプション説明

`endlocal` にはスイッチやオプションはありません。

| 使用法 | 説明 |
| :--- | :--- |
| `endlocal` | `setlocal` によって開始された環境変数のローカライズを終了し、以前の状態を復元する。 |

## よく使われる組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 複雑な構築バッチ内で、複数のモジュール（子バッチ）を呼び出す際、各モジュールで定義された一時変数が親バッチやシステム全体に影響を与えないように隔離する。
* **解説**: モジュール化されたクリーンなコードを書くための標準的な作法です。

### 2. ブルーチーム視点

* **タスク**: 侵害調査（フォレンジック）スクリプトを配布・実行する際、ターゲット端末の環境設定（`PATH` やプロンプト表示など）を一切変更せずに調査を終える。
* **例**:
    ```cmd
    setlocal
    REM 調査に必要な一時的な変数を設定
    set LOG_DIR=C:\Windows\Temp\investigation
    dir /s > %LOG_DIR%\files.txt
    endlocal
    ```

### 3. レッドチーム視点 (隠蔽とクリーンアップ)

* **タスク 1: Artifact Cleanup (痕跡の最小化)**
    * **目的**: 攻撃用スクリプトの実行中に定義した多数の環境変数（例：C2サーバのIP、奪取したトークン、一時的なパス）を、スクリプト終了後に一括で消去する。
    * **解説**: `setlocal`/`endlocal` を使用することで、セッションが維持されている場合でも、不審な変数が環境内に残るのを防ぎ、管理者の目から隠蔽します。

* **タスク 2: Variable Scope Sandboxing (変数のサンドボックス化)**
    * **目的**: 難読化された複雑なコマンドを展開・実行する際、中間生成された不審な文字列をメモリ（環境変数空間）から即座に破棄する。

## エラーメッセージとトラブルシューティング

1.  **引数なしの実行**
    * **現象**: `endlocal` を単体で実行してもエラーにはならない。
    * **原因**: 対応する `setlocal` がない場合、`endlocal` は何もせず、エラーも返しません。

2.  **変数が意図せず消える**
    * **原因**: `endlocal` の後に変数を参照しようとしている。
    * **解決策**: `endlocal` を実行するとその中の変数はすべて消えます。値を親のスコープに引き継ぎたい場合は、`endlocal & set PARENT_VAR=%LOCAL_VAR%` のように、`endlocal` と同じ行で `set` を実行する特殊なテクニックが必要です。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **環境変数汚染の回避**: 
    * 攻撃者が正規の管理スクリプトを改ざんして `endlocal` を削除した場合、スクリプトが作成した不審な変数がシステムに残り続け、後のプロセスで予期せぬ動作を引き起こす可能性があります。

### LOLBAS における利用例

* **機能**: なし (内部制御コマンドのため、単体での悪用例は少ない)
* **特性**: バッチファイル内での実行制御に使用される標準機能です。

## 注意点・補足

* **ネスト（入れ子）構造**: `setlocal` と `endlocal` は最大 32 レベルまで入れ子にできます。各 `endlocal` は、直近の `setlocal` を終了させます。
* **暗黙の終了**: バッチファイルの終端に達した場合、実行されていない `setlocal` に対する `endlocal` が自動的に実行されます。しかし、明示的に記述することが推奨されます。
