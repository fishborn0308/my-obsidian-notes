# `docker compose logs` - Composeアプリケーションのログを表示する

## 概要

`docker compose logs` は、`docker-compose.yml` ファイルで定義されたすべてのサービス（コンテナ）から出力されるログを集約し、表示するためのコマンドです。各コンテナの標準出力 (stdout) と標準エラー出力 (stderr) に書き込まれた内容が、サービス名を接頭辞として付けて一覧表示されるため、複数のコンテナが連携して動作するアプリケーション全体の挙動を一度に把握できます。主に、アプリケーションのデバッグや動作確認、問題発生時の原因究明に利用されます。

(出自: `docker compose` の基本機能)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`docker logs`** | **単一の**コンテナのログを表示します。`docker compose logs` は**プロジェクト全体の複数コンテナ**のログを時系列でまとめて表示します。 | `docker compose logs` (Compose環境) |
| **`journalctl -u docker.service`**| Docker**デーモン自体**のログを表示します。`docker compose logs` は**コンテナ内で実行されているアプリケーション**のログを表示します。 | - |

## よく連携されるコマンド

`docker compose logs` は、アプリケーションを起動した直後に `docker compose ps` と連携して、サービスの起動状態とログを併せて確認する、という流れでよく使われます。

### シナリオ例: アプリケーション起動後の状態確認 (インフラ構築・運用視点)

* **目的**: `docker compose up -d` で起動したアプリケーションが、全サービス正常に立ち上がっているかを確認する。
* **連携コマンド**: `docker compose up`, `docker compose ps`
* **解説**: アプリケーションを起動した後、まず `docker compose ps` で全サービスの `State` が `Up` になっているかを確認します。もし `Exited` や `restarting` のサービスがあれば、`docker compose logs` でそのサービスのエラーログを調査します。
* **コマンド例**:

    ```bash
    # 1. アプリケーションを起動
    docker compose up -d

    # 2. 状態を確認
    docker compose ps

    # 3. (もし問題があれば) ログを確認
    docker compose logs
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **表示制御** | |
| ⭐ `-f`, `--follow` | ログの出力をリアルタイムで追跡します (`tail -f` と同様)。 |
| ⭐ `--tail <number>` | ログの末尾から指定した行数だけを表示します。（例: `--tail="100"`） |
| ⭐ `[service...]` | 引数としてサービス名を指定すると、そのサービスのログのみを表示します。 |
| `-t`, `--timestamps` | 各ログ行の先頭にタイムスタンプを付けて表示します。 |
| `--no-color` | サービスごとの色分けを無効にします。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 開発中のアプリケーションの動作をリアルタイムでデバッグする。
* **組み合わせ**: `docker compose logs -f [service_name]`
* **解説**: `-f` でログを追跡し、特定のサービスに絞ることで、アプリケーションにリクエストを送った際のログの流れを効率的に確認できます。
* **例**:

    ```bash
    # WebサーバーとDBのログをリアルタイムで監視する
    docker compose logs -f webapp db
    ```

### 2. ブルーチーム視点

* **タスク**: インシデント発生時に、アプリケーション全体のログを証拠として保全する。
* **組み合わせ**: `docker compose logs --timestamps > investigation.log`
* **解説**: 全てのサービスのログにタイムスタンプを付けて、調査用のファイルにリダイレクトします。タイムスタンプにより、攻撃のタイムラインを正確に再構築するための重要な情報となります。
* **例**:

    ```bash
    # 不審な挙動を示したECサイトのコンテナ群の全ログをタイムスタンプ付きでファイルに保存
    docker compose -f /opt/ecommerce-app/docker-compose.yml logs --timestamps > case-002_ecommerce.log
    ```

### 3. レッドチーム視点

* **タスク**: 侵入したホスト上で稼働しているComposeアプリケーションのログを偵察する。
* **組み合わせ**: `docker compose logs`
* **解説**: 攻撃者は、`docker ps` や `docker-compose.yml` ファイルからアプリケーションの存在を特定した後、そのログを閲覧します。ログにデバッグ情報、内部IPアドレス、ユーザー名、場合によってはパスワードやトークンなどが含まれていないかを探し、さらなる攻撃のヒントを得ます。

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **現象**: **コマンドを実行しても何もログが表示されない。**
    * **考えられる原因**:
        * アプリケーションが標準出力/標準エラー出力ではなく、ファイルに直接ログを書き込んでいる。
        * コンテナが正常に起動していない (`docker compose ps` で確認)。
    * **解決策**: アプリケーションのロギング設定を確認し、標準出力にログを出すように修正します。それができない場合は、`docker exec` でコンテナ内に入り、ログファイルの中身を直接確認します。

## 環境変数と設定ファイル

* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

### 関連する設定ファイル

* **`docker-compose.yaml`**: `logging` ディレクティブを使って、サービスのロギングドライバやオプションをカスタマイズできます。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **ログに含まれる機密情報**。
* **悪用シナリオ**: アプリケーションがデバッグ目的で、個人情報(PII)、セッショントークン、APIキーなどをログに出力している場合があります。`docker logs` を実行できる権限を持つ攻撃者は、これらの情報を窃取できます。

### GTFOBins / LOLBAS における利用例

`docker compose` は `docker` コマンドの一部として扱われます。`sudo docker` の権限があれば、任意の操作が可能です。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: アプリケーションコード内で、本番環境では機密情報をログに出力しないようにする。
* **Detection (検知)**: **ログの集中管理**。`fluentd` などのロギングドライバを使い、コンテナログを改ざん困難な中央ログサーバー (SIEMなど) にリアルタイムで転送する。

## 注意点・補足

* **ログの追跡終了**: `-f` (`--follow`) オプションでログを追跡している状態は、`Ctrl+C` を押すことで終了します。
