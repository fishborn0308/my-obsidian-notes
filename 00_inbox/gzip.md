# `gzip` - .gzファイルに圧縮する

## 概要

`gzip` (GNU Zip) は、ファイルを圧縮してディスク容量を節約するための、Linux/Unix環境で最も標準的かつ広く利用されているコマンドです。DEFLATEアルゴリズムを使用し、一般的にファイル名の末尾に `.gz` という拡張子を付与します。

デフォルトでは、圧縮に成功すると元のファイルは削除されます。

(出自: `gzip` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`bzip2`** | `gzip` よりも**高い圧縮率**を達成できますが、**圧縮・展開の速度は遅く**なります。ディスク容量の節約を最優先するなら `bzip2`、速度や互換性を重視するなら `gzip` が適しています。 | `gzip` (速度) / `bzip2` (圧縮率) |
| **`zip`** | 複数のファイルやディレクトリを一つのファイルにまとめる**アーカイバ**機能と**圧縮**機能を持ちます。`gzip` は原則として**単一のファイル**を圧縮するためのツールです。 | - |
| **`tar`** | ディレクトリを圧縮する場合、まず `tar` でアーカイブし、それを `gzip` で圧縮します (`.tar.gz`)。`tar czf ...` のように `tar` コマンドから直接利用するのが一般的です。 | `tar` (ディレクトリの場合) |

## よく連携されるコマンド

`gzip` は、他のコマンドの出力をパイプで受け取り、オンザフライで圧縮する際によく使われます。

### シナリオ例: データベースダンプを直接圧縮する (インフラ構築・運用視点)

* **目的**: 巨大なデータベースのダンプファイルを、中間ファイルをディスクに書き出すことなく、直接 `gzip` で圧縮してバックアップファイルを作成する。
* **連携コマンド**: `mysqldump` (または `pg_dump`), パイプ (`|`)
* **解説**: `mysqldump` の標準出力をパイプで `gzip` に渡します。`-c` オプションで結果を標準出力に送り、リダイレクションでファイルに保存することで、ディスクI/Oと容量を大幅に節約できます。
* **コマンド例**:

    ```bash
    # mysqldumpの出力を直接gzipで圧縮してバックアップファイルを作成
    mysqldump -u root -p my_database | gzip -c > backup-$(date +%F).sql.gz
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **基本操作** | |
| ⭐ `-d`, `--decompress` | 展開（解凍）します。これは `gunzip` コマンドと同じ動作です。 |
| ⭐ `-k`, `--keep` | 圧縮または展開後も、元のファイルを**削除せずに残します (keep)**。 |
| `-c`, `--stdout` | 圧縮または展開結果をファイルに書き出す代わりに、**標準出力 (stdout)** に出力します。 |
| **圧縮レベル** | |
| `-[1-9]` | 圧縮レベルを指定します。`-1` (`--fast`) が最も速く（低圧縮）、`-9` (`--best`) が最も遅く（高圧縮）なります。デフォルトは `-6` です。 |
| **その他** | |
| `-r`, `--recursive`| ディレクトリを指定すると、その中のファイルを再帰的に一つずつ圧縮します。 |
| `-f`, `--force` | 同名の出力ファイルが既に存在する場合でも、警告なしで**強制的に上書き**します。 |
| `-v`, `--verbose` | 実行中の詳細な情報（圧縮率など）を表示します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 巨大なログファイルを圧縮してディスク容量を節約する。
* **組み合わせ**: `gzip -9 -v <large_logfile>`
* **解説**: `-9` で最高の圧縮率を指定し、`-v` で圧縮結果を確認しながらファイルを圧縮します。`logrotate` スクリプトなどで頻繁に利用されます。
* **例**:

    ```bash
    # 巨大なアクセスログを最高圧縮レベルで圧縮
    gzip -9 -v /var/log/nginx/access.log
    # -> /var/log/nginx/access.log.gz が生成される
    ```

### 2. ブルーチーム視点

* **タスク**: 証拠ファイルを、元のファイルを保持したまま圧縮する。
* **組み合わせ**: `gzip -k <evidence_file>`
* **解説**: フォレンジック調査で収集した巨大なディスクイメージやメモリダンプを、長期保管やネットワーク経由での転送のために圧縮します。`-k` を使うことで、元の証拠ファイルのハッシュ値などを変えることなく、圧縮コピーを作成できます。
* **例**:

    ```bash
    # メモリダンプファイルを圧縮してコピーを作成
    gzip -k memory_dump.raw
    ```

### 3. レッドチーム視点

* **タスク**: 窃取したデータを圧縮して、持ち出し（Exfiltration）の準備をする。
* **組み合わせ**: `gzip <stolen_data>`
* **解説**: システムから盗み出した大量のテキストベースのデータ（ログ、ソースコードなど）を `gzip` で圧縮し、データサイズを小さくします。これにより、ネットワーク転送時間を短縮し、データ転送量を監視しているIDS/DLPによる検知を逃れやすくします。
* **例**:

    ```bash
    # 窃取したユーザーのホームディレクトリをtarで固めてからgzipで圧縮
    tar cf - /home/victim | gzip > victim_home.tar.gz
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `gzip: <dirname> is a directory -- ignored`
    * **考えられる原因**: `gzip` はデフォルトではディレクトリを圧縮しません。`-r` オプションはディレクトリ内のファイルを個別に圧縮しますが、ディレクトリを1つのファイルにまとめるわけではありません。
    * **解決策**: ディレクトリを1つのファイルとして圧縮したい場合は、`tar czf archive.tar.gz directory/` のように**`tar` コマンドと組み合わせる**のが正しい方法です。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **伸張爆弾 (Decompression Bomb)**。展開すると極端に巨大なファイルになるように細工された悪意のある圧縮ファイル。
* **悪用シナリオ**: 攻撃者は、マルウェアや攻撃用スクリプトを `gzip` で圧縮してシステムに持ち込むことがあります。これにより、単純なファイルシグネチャによるアンチウイルス検索を回避しようと試みることがあります。

### GTFOBins / LOLBAS における利用例

`gzip` はファイルの圧縮に特化しており、直接的なコード実行や権限昇格の機能を持たないため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 信頼できないソースから入手した圧縮ファイルは、隔離されたサンドボックス環境で安全に展開・調査する。
* **Detection (検知)**: 攻撃者がデータを圧縮して持ち出す（Exfiltration）兆候として、予期せぬ場所での `.gz` や `.tar.gz` ファイルの作成をファイル整合性監視(FIM)ツールで監視する。

## 注意点・補足

* **`z`系コマンド**: `.gz` ファイルを展開せずに直接操作するための一連の便利なコマンド (`zcat`, `zless`, `zgrep`など) があります。

  # 圧縮されたログファイルを展開せずに検索
  zgrep "ERROR" application.log.gz
