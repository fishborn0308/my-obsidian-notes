# Command - Linux - hostid - 現在のホストの数値識別子を表示する

## 概要

`hostid` は、現在のホストを識別するための 32 ビットの数値を 16 進数で表示します。この数値は、本来はネットワーク全体でユニーク（一意）であることが期待されていますが、現在の Linux 実装では通常、マシンの IP アドレスを元に生成されるか、特定のファイルに保存された値が使用されます。

(出自: `coreutils` パッケージに含まれる)

## 類似・関連コマンド

| コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `hostname` | システムの名前（文字列）を表示する。 | 一般的な識別 |
| `cat /etc/machine-id` | systemd 環境で生成される 128 ビットの非常に一意性の高い ID。 | **現代の個体識別** |
| `ip link` | MAC アドレスを表示する。ハードウェア固有の識別子。 | ネットワーク的な識別 |
| `hostid` | 伝統的な 32 ビットの数値 ID。古いアプリとの互換用。 | **レガシーアプリ対応** |

## よく連携されるコマンド

### シナリオ例: マシンごとの簡易ログ作成 (運用視点)

* **目的**: 複数のサーバーからログを集約する際、ファイル名にホスト名ではなく、固定の ID を付与したい。
* **解説**: `hostname` が変わる可能性がある環境で、より不変に近い ID を利用します。
* **コマンド例**:
    ```bash
    # 実行結果にホストIDを付与して保存
    echo "$(hostid) - Job completed" >> /shared/log/cluster.log
    ```

## オプション説明

`hostid` には、標準的なヘルプとバージョン表示以外の実用的なオプションは存在しません。引数なしで実行するのが基本です。

| 用法 | 説明 |
| :--- | :--- |
| ⭐️ `hostid` | 現在のホスト ID を表示する。 |
| `--help` | ヘルプを表示する。 |
| `--version` | バージョン情報を表示する。 |

## よく使われる業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 商用ソフトウェアのライセンスアクティベーション。
* **解説**: 一部の古い CAD ソフトやデータベースソフトウェアは、ライセンスを特定のサーバーに紐付ける際、`hostid` の値を要求することがあります。
* **例**: ライセンス申請フォームに「Host ID: 007f0101」のように記入します。

### 2. ブルーチーム視点

* **タスク**: ログ解析における送信元特定。
* **解説**: アプリケーションログの中に `hostid` が記録されている場合、たとえホスト名が書き換えられていても、どのサーバーから送られたデータかを特定するヒントになります。

### 3. レッドチーム視点

* **タスク**: サンドボックス検知（Anti-Sandboxing）。
* **解説**: 攻撃用コード（マルウェア）が実行された際、`hostid` が特定の既知のサンドボックス環境や仮想マシンのデフォルト値（例: `007f0101` など）と一致するかを確認し、解析環境であれば活動を停止する仕組みに悪用されます。

## ホスト ID の決まり方と変更方法

Linux において、`hostid` は以下の優先順位で決定されることが多いです。

1.  ファイル `/etc/hostid` が存在すれば、その中身（バイナリ 4 バイト）を使用。
2.  存在しなければ、`gethostbyname` 等で取得した IP アドレスをビット操作して生成。

* **手動で設定する場合**:
    特定の ID に固定したい場合は、Python 等を使ってバイナリファイルを生成します。
    ```bash
    # 例: hostid を 'deadbeef' に設定する (root権限)
    python3 -c 'import struct; open("/etc/hostid", "wb").write(struct.pack("I", 0xdeadbeef))'
    ```

## エラーメッセージとトラブルシューティング

1.  **「異なるマシンなのに同じ hostid が表示される」**:
    * **原因**: どちらのマシンも IP アドレスが `127.0.0.1` で解決されているか、OS のクローン作成時に `/etc/hostid` までコピーしてしまった。
    * **解決策**: `/etc/hostid` を削除して再生成させるか、個別に ID を書き込む。

## セキュリティに関する考慮事項

### 脆弱性とリスク

* **情報の推測**: `hostid` が IP アドレスから生成されている場合、攻撃者はその数値からマシンの（かつての、あるいは内部の）IP アドレスを逆算できる可能性があります。

### 対応策・緩和策 (ブルーチーム視点)

* **Identity Management**: 現代的なシステム管理では `hostid` を唯一の識別子として信頼せず、`machine-id` や証明書ベースの識別を優先します。

## 注意点・補足

* **一意性の欠如**: `hostid` は 32 ビット（42億通り）しかないため、大規模なクラウド環境などでは衝突（重複）する可能性があります。
* **エンディアン**: システムのアーキテクチャ（Big Endian / Little Endian）によって、IP アドレスから生成される数値の並びが変わる場合があります。
