# Command  - Linux - locate - データベースを使ってファイルを高速に検索する

## 概要

`locate` は、ファイル名の一部または全部を指定して、システム上のファイルやディレクトリを**非常に高速に**検索するコマンドです。この高速性は、実際のファイルシステムを毎回スキャンするのではなく、**事前に作成されたファイルパスのデータベース** (`mlocate.db`など) を検索することで実現されています。

このデータベースは、通常 `updatedb` コマンドによって、`cron` などを通じて1日に1回程度、自動的に更新されます。

(出自: `mlocate` または `plocate` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`find`** | `find` は**リアルタイムのファイルシステム**を検索するため常に最新ですが**時間がかかります**。`locate` は**データベース**を検索するため**非常に高速**ですが、結果が最新でない場合があります。「速さの`locate`、正確さの`find`」と覚えられます。 | `locate` (高速検索) / `find` (正確・高機能) |
| **`grep`** | **ファイルの中身**をパターンで検索します。`locate` は**ファイル名やパス**を検索します。 | - |
| **`which`** | 実行可能ファイルを環境変数 `PATH` の中から探します。`locate` はシステム全体のあらゆる種類のファイルを検索します。 | - |

## よく連携されるコマンド

`locate` のデータベースは `updatedb` コマンドによって更新されるため、この2つは密接に関連しています。

### シナリオ例: 新しくインストールしたファイルを探す (インフラ構築・運用視点)

* **目的**: パッケージをインストールした直後に、関連する設定ファイルがどこに配置されたかを探す。
* **連携コマンド**: `updatedb`
* **解説**: パッケージをインストールした直後は、`locate` のデータベースにはまだその情報が反映されていません。`sudo updatedb` を手動で実行してデータベースを更新することで、新しいファイルを `locate` で即座に見つけられるようになります。
* **コマンド例**:

```bash
# (例: my-app をインストールした後...)

# データベースを手動で更新
sudo updatedb

# my-app に関連する設定ファイルを探す
locate my-app.conf
```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **検索制御** | |
| ⭐ `-i`, `--ignore-case` | 大文字と小文字を区別せずに検索します。 |
| `-b`, `--basename` | パス名全体ではなく、ファイル名（ベース名）のみにマッチさせます。 |
| `-r`, `--regexp` | 検索パターンを基本的な正規表現として解釈します。 |
| **出力制御** | |
| ⭐ `-c`, `--count` | 一致したファイルの総数のみを表示します。 |
| `-e`, `--existing`| データベースには存在するが、実際のファイルシステム上では既に削除されているファイルは表示しません。 |
| **その他** | |
| `-S`, `--statistic` | データベースの統計情報（ファイル数、ディレクトリ数など）を表示します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 名前は分かっているが正確なパスが分からない設定ファイルなどを素早く見つける。
* **組み合わせ**: `locate <filename>`
* **解説**: 「`nginx.conf` ってどこにあったかな？」と思った際に `locate nginx.conf` を実行すると、`/etc/nginx/nginx.conf` のような候補が瞬時に表示されます。
* **例**:

```bash
# php.ini ファイルの場所を探す
locate php.ini
```

### 2. ブルーチーム視点

* **タスク**: 既知のマルウェアやツールのファイル名をシステム全体から高速に検索する。
* **組み合わせ**: `locate <malware_filename>`
* **解説**: インシデントレスポンスの初期調査で、`find` でフルスキャンをかける前に、まず `locate` を実行します。データベースに記録されていれば、瞬時にその存在とパスを特定でき、調査を迅速に進められます。ただし、データベース更新後に作成されたファイルは見つけられないという限界も理解しておく必要があります。
* **例**:

```bash
# Webシェルでよく使われる 'c99.php' が存在しないか高速にチェック
locate c99.php
```

### 3. レッドチーム視点

* **タスク**: **隠密な偵察 (Stealthy Reconnaissance)** を行い、システム上の興味深いファイルを探す。
* **組み合わせ**: `locate "*.conf" | grep -i "pass\|secret\|key"`
* **解説**: `find` よりもディスクI/Oをほとんど発生させないため、I/O監視による検知を避けつつ、システム上の設定ファイルやバックアップファイル、秘密鍵などを高速にリストアップできます。
* **例**:

```bash
# バックアップファイルやSQLダンプファイルがないか探す
locate "*.bak"
locate "*.sql"
```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **現象**: **ついさっき作成したファイルが `locate` で見つからない。**
		* **考えられる原因**: `locate` のデータベースがまだ更新されていないためです。データベースは通常、cronによって深夜などに一日1回更新されます。
		* **解決策**: `sudo updatedb` を手動で実行してデータベースを更新すれば、見つかるようになります。

## 環境変数と設定ファイル

* `locate` の動作に直接影響する環境変数は通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

### 関連する設定ファイル

* **`/etc/updatedb.conf`**: `updatedb` コマンドがデータベースを作成する際に、スキャンの対象から除外するパス（`/tmp`, `/mnt`など）やファイルシステムの種類を指定します。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `locate` 自体ではなく、そのデータベースが**情報漏洩**に繋がります。
* **悪用シナリオ**: 攻撃者は `locate` のデータベースを調べることで、`find` のような時間のかかるスキャンを行わずに、システム全体のファイル構成を瞬時に把握し、機密ファイルを探し出すことができます。

### GTFOBins / LOLBAS における利用例

`locate` は情報表示のみを行うコマンドであり、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: -
* **Detection (検知)**: 攻撃者は `find` よりも `locate` を好む可能性があるため、`locate` コマンド自体の実行を `auditd` で監視することが有効です。
* **調査上の注意**: `locate` は最新の状況を反映しないため、フォレンジック調査では必ず `find` によるリアルタイム検索も併用する必要があります。

## 注意点・補足

* **`plocate`**: `mlocate` のより高速な後継実装である `plocate` (Posting Locate) が、新しいディストリビューション（Ubuntu 22.04+など）で採用されています。ユーザーからは `locate` コマンドとして透過的に利用できます。
