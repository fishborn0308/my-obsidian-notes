# Command - Linux - cksum - ファイルのCRCチェックサムとバイト数を表示する

## 概要

`cksum` は、POSIX 標準に従ってファイルの CRC (Cyclic Redundancy Check: 巡回冗長検査) 値、バイト数、およびファイル名を出力します。主に、ネットワーク転送やコピーによってファイルの内容が意図せず破損していないかを、簡易的かつ高速に確認するために使用されます。

(出自: `coreutils` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `md5sum` | MD5アルゴリズムを使用。cksumより衝突耐性が高いが、現在はセキュリティ用途には不向き。 | 簡易的なハッシュ確認 |
| `sha256sum` | SHA-2アルゴリズムを使用。現代の標準であり、改ざん検知（セキュリティ）に必須。 | **セキュリティ・整合性保証** |
| `sum` | cksumよりさらに古いコマンド。アルゴリズムが非標準的な場合が多く、現在は非推奨。 | レガシー環境のみ |
| `cksum` (最新版) | 最新の GNU coreutils では、`-a` オプションで md5 や sha256 も扱える統合ツールへ進化。 | 汎用的な整合性チェック |

## よく連携されるコマンド

### シナリオ例: 複数ファイルの整合性一括チェック (運用視点)

* **目的**: バックアップディレクトリ内の全てのファイルが正しくコピーされたか確認する。
* **連携コマンド**: `find`, `diff`
* **解説**: 元の環境とコピー先の環境で `cksum` の結果をファイルに保存し、`diff` で比較します。
* **コマンド例**:
    ```bash
    # 全ファイルのチェックサムをリスト化
    find /path/to/data -type f | xargs cksum > baseline.txt
    # (別環境で同様に作成し) 比較
    diff baseline.txt backup.txt
    ```

## オプション説明

※伝統的な `cksum` にはオプションが殆どありませんが、現代の GNU 実装では以下の拡張がなされています。

| オプション | 説明 |
| :--- | :--- |
| **基本操作** | |
| ⭐️ `--help` | ヘルプを表示。 |
| ⭐️ `--version` | バージョン情報を表示。 |
| **拡張機能 (GNU coreutils 9.0+)** | |
| `-a`, `--algorithm` | 計算アルゴリズムを指定（crc, md5, sha1, sha256 等）。 |
| `--check` | チェックサムファイルから読み込み、整合性を検証する。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: リモートサーバへ転送した実行バイナリの破損確認。
* **組み合わせ**: 引数なし（デフォルト）
* **解説**: FTPや不安定なSCP転送後、ファイルサイズとCRCが一致するか確認し、転送ミスによる「謎のエラー」を未然に防ぎます。
* **例**:
    ```bash
    # 転送先で実行
    cksum downloaded_installer.sh
    # 出力例: 2843029487 154820 downloaded_installer.sh
    ```

### 2. ブルーチーム視点

* **タスク**: 変更が頻繁でない設定ファイルの「簡易的な」改ざん検知。
* **組み合わせ**: `cron` との連携
* **解説**: 負荷をかけたくない環境で、重要ファイルの変化を定期的にスクリプトでチェックします。ただし、意図的な改ざん（攻撃者によるハッシュ衝突攻撃）には弱いため、あくまで「不注意な変更」の検知用です。
* **例**:
    ```bash
    # 設定ファイルのハッシュを一時記録と比較
    [ "$(cksum /etc/ssh/sshd_config)" = "$(cat /var/lib/check/sshd_config.bak)" ] || echo "Warning: sshd_config changed"
    ```

### 3. レッドチーム視点

* **タスク**: ターゲット環境に存在するファイルのバージョン特定。
* **組み合わせ**: 既知のチェックサムリストとの照合
* **解説**: 特定の脆弱性がある古いライブラリやバイナリを見つけた際、そのチェックサムを計算し、既知のビルドバージョンと一致するか確認してエクスプロイトの成功率を高めます。
* **例**:
    ```bash
    # ターゲット上のバイナリを fingerprinting
    cksum /usr/lib/libvictim.so.1
    ```

## エラーメッセージとトラブルシューティング

1.  **エラーメッセージ例 1**: `cksum: <filename>: No such file or directory`
    * **考えられる原因**: ファイル名が間違っている、またはパスが通っていない。
    * **解決策**: 正しいパスを指定する。

2.  **チェックサムが毎回変わる**:
    * **考えられる原因**: ログファイルなどのように、バックグラウンドで常に書き込まれているファイルを指定している。
    * **解決策**: 書き込みを停止してから計算するか、スナップショットに対して実行する。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **衝突耐性の低さ**。`cksum` が使用する CRC アルゴリズムは、意図的に同じチェックサムを持つ異なるファイルを作成することが比較的容易です。
* **悪用シナリオ**: 攻撃者がバックドアを仕込んだバイナリを作成し、元のバイナリと同じ `cksum` 値になるようにパディング（ゴミデータ）を調整することで、簡易的な整合性チェックをパスさせる。

### GTFOBins における利用例

`cksum` に `sudo` 権限がある場合、本来読み取れないファイル（`/etc/shadow` 等）を読み取り、そのチェックサムを表示できます。内容そのものは見えませんが、オフラインで用意したパスワード辞書とハッシュ（またはチェックサム）を比較することで、中身を推測する足掛かりにされる可能性があります。

* **機能**: `File Read`, `Sudo`
* **参照**: `https://gtfobins.github.io/gtfobins/cksum/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: セキュリティが重要な場面では `cksum` ではなく、必ず `sha256sum` 以上を使用する。
* **Detection (検知)**: 重要なシステムバイナリに対して `cksum` だけに頼った運用をしていないか見直す。

## 注意点・補足

* **標準化**: `cksum` のアルゴリズムは POSIX で定義されていますが、古い `sum` コマンドは実装によって結果が異なるため、スクリプトの移植性を考えるなら `cksum` を選ぶべきです。
* **バイナリ/テキスト**: `cksum` はファイルをバイナリとして読み込むため、改行コードの違い（CRLF vs LF）でも結果が変わります。
