# Command - git - git add - 変更をステージングエリアに追加

## 概要

`git add` コマンドは、ワーキングディレクトリ（作業ディレクトリ）での変更内容を**ステージングエリア**（またはインデックス）に追加します。これは、次のコミットに含めたい変更を選択する第一歩です。Gitはステージングエリアにある変更のみをコミットします。

(出自: `git` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`git commit`** | `git add` は変更をコミットの「準備」をするコマンドです。`git commit` はステージングエリアにある変更を永続的にリポジトリの履歴に記録します。 | - |
| **`git restore`** | `git restore --staged <file>` はステージングエリアから変更を取り消します。`git add` は変更をコミットに含める方向、`git restore` は除外する方向です。 | `git restore --staged` |
| **`git rm`** | `git rm` はファイルを削除し、その削除をステージングエリアに記録します。 | - |

## よく連携されるコマンド

`git add` はGitワークフローの中核であり、変更の状態を確認する `git status` や、変更を記録する `git commit` と常に連携します。

### シナリオ例: 新規ファイルの追跡とコミット (インフラ構築・運用視点)

* **目的**: 新しく作成した設定ファイルやスクリプトをGitのバージョン管理下に置き、最初のコミットとして記録する。
* **連携コマンド**: `git status`, `git commit`
* **解説**: ファイルを作成した後、`git status`で未追跡の状態を確認し、`git add`でステージング、そして`git commit`で履歴に記録します。
* **コマンド例**:

		```bash
		# 新しい設定ファイルを作成

		touch new_server_config.conf

		# 未追跡ファイルとして表示されることを確認

		git status

		# ファイルをステージング

		git add new_server_config.conf

		# コミット対象としてステージされていることを確認

		git status

		# 変更をコミット

		git commit -m "Add new server configuration file"

		```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **ファイル指定** | |
| ⭐ `<pathspec>...` | ステージングするファイルやディレクトリを指定します。`git add file.txt dir/` のように複数指定可能。 |
| ⭐ `.` | 現在のディレクトリとそのサブディレクトリ以下の全ての新規・変更されたファイルをステージングします。 |
| **動作モード** | |
| ⭐ `-p`, `--patch` | 変更を「Hunk」(変更のまとまり)ごとにインタラクティブにステージングするか選択できます。 |
| `-A`, `--all` | 全ての変更（新規、変更、削除）をステージングします。 |
| `-u`, `--update` | 追跡中のファイルの変更と削除のみをステージングします（新規ファイルは含みません）。 |
| `-f`, `--force` | `.gitignore`で無視されているファイルも強制的にステージングします。 |
| `-n`, `--dry-run` | 実際にステージングせず、どのファイルがステージングされるかをシミュレーション表示します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 複数の設定ファイルを修正し、それら全てを一度にステージングしてデプロイ準備を整える。
* **組み合わせ**: `git add .` (または `git add -A`)
* **解説**: インフラ設定ファイルやデプロイメントスクリプトの修正後、全ての関連する変更を漏れなくコミットに含める際に使用します。
* **例**:

		```bash
		# 複数の設定ファイルを編集・作成・削除

		git add . # 全ての変更(新規・修正・削除)をステージング

		git commit -m "Prepare web server configuration for new deployment"

		```

## 2. ブルーチーム視点

* **タスク**: セキュリティスクリプトに実行権限を付与し、その変更もコミットに含める。
* **組み合わせ**: `git add <file>`
* **解説**: `chmod +x` で実行権限を付与した後、`git add` を実行すると、ファイルの内容だけでなく、パーミッションの変更もステージングされます。
* **例**:

		```bash
		# analysis_script.py を作成し、実行可能にする

		chmod +x analysis_script.py

		git add analysis_script.py # 実行権限の変更もステージングされる

		git commit -m "Add new malware analysis script with execute permission"

		```

## 3. レッドチーム視点

* **タスク**: 多数の変更を行った後、痕跡を隠す目的で、コミットしたい変更とそうでない変更を厳密に区別する。
* **組み合わせ**: `git add -p <file>`
* **解説**: `-p` (`--patch`) モードで、ファイル内の特定の変更ブロック（Hunk）だけを選択的にステージングします。これにより、クリーンなコミットに見せかけつつ、別の悪意ある変更はコミットせずに残しておく、といった操作が可能です。
* **例**:

		```bash
		# target_script.py を修正。機能追加とデバッグコードが混在。

		git add -p target_script.py

		# インタラクティブモードで、機能追加のHunkのみを 'y' で選択

		git commit -m "Add new feature to target_script.py"

		```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `fatal: pathspec '<filename>' did not match any files`
		* **考えられる原因**: 指定したファイル名やパスが間違っているか、ファイルが `.gitignore` で無視されています。
		* **解決策**: ファイル名やパスを確認します。無視されているファイルを強制的に追加したい場合は `-f` オプションを使用します。

## 環境変数と設定ファイル

* `git add` に直接影響する独自の環境変数はありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

### 関連する設定ファイル

* **`.gitignore`**: Gitに追跡させたくないファイルやディレクトリのパターンを記述します。`git add`はデフォルトでこのファイルに記載されたパターンを無視します。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **機密情報の誤コミット**。
* **悪用シナリオ**: `git add .` や `git add -A` を使用する際に、`.gitignore` の不備や確認不足により、機密情報（認証情報、APIキーなど）を誤ってステージングし、コミットしてしまうケースが頻繁に発生します。

### GTFOBins / LOLBAS における利用例

`git add` は直接的なコード実行機能を持たないため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **厳格な`.gitignore`の管理**。プロジェクト開始時に、認証情報ファイル、ログファイルなどを徹底的に`.gitignore`に含める。
* **Detection (検知)**:
	* **コミット前のレビュー徹底**: `git diff --staged` でステージングされている内容を必ず確認する習慣をつける。
	* **プリコミットフック**: `git-secrets` などのツールを `pre-commit` フックとして導入し、機密情報のコミットを自動的にブロックする。
	* **CI/CDでのスキャン**: CI/CDパイプラインに `trufflehog` などのシークレットスキャンツールを組み込む。

## 注意点・補足

* **ステージングエリアの役割**: `git add` した後のファイルにさらに変更を加えた場合、**再度 `git add` を実行しないと、最新の変更はコミットに含まれません**。ステージングエリアは、最後に `add` した時点のスナップショットを保持します。
