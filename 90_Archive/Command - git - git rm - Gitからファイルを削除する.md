# Command - git - git rm - Gitからファイルを削除する

## 概要

`git rm` コマンドは、Gitリポジトリからファイルやディレクトリを削除するために使用されます。単にファイルを削除するだけでなく、その削除を**ステージングエリアに記録**し、次のコミットでそのファイルがリポジトリから削除されるように準備します。これにより、Gitの追跡対象からファイルを安全かつ意図的に外すことができます。

(出自: `git` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`rm` (シェル)** | `rm`コマンドはファイルシステムから物理的にファイルを削除するだけです。`git rm`は**ファイルの物理的な削除とGitへの削除の記録を同時に行います**。 | `git rm` |
| **`git filter-repo`** | `git rm`が現在のコミットからファイルを削除するのに対し、`filter-repo`は**過去のコミット履歴からファイルを完全に削除**するために使用されます。 | - |
| **`.gitignore`** | `.gitignore`はファイルを**最初から追跡させない**ように指示します。`git rm --cached` は**すでに追跡されている**ファイルを追跡対象から外します。 | - |

## よく連携されるコマンド

`git rm`は、ファイル削除の前後でリポジトリの状態を確認する `git status` や、削除を記録する `git commit` と連携して使われます。

### シナリオ例: 不要なファイルの削除とコミット (インフラ構築・運用視点)

* **目的**: プロジェクトから不要になったファイルを削除し、その削除をGit履歴に記録する。
* **連携コマンド**: `git status`, `git commit`
* **解説**: まず`git status`で現在の状態を確認します。次に`git rm`でファイルを削除し、`git status`で削除がステージングされたことを確認します。最後に`git commit`で削除を履歴に永続化します。
* **コマンド例**:

		```bash
    # 1. 不要なファイルを削除し、その削除をステージング

    git rm old_script.sh

    # 2. 削除がステージングされたことを確認

    git status

    # 3. 削除をコミット

    git commit -m "chore: Remove old_script.sh as it's no longer needed"

    ```



## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **ファイル指定** | |
| ⭐ `<file>` | 削除するファイルの名前を指定します。 |
| **動作モード** | |
| ⭐ `--cached` | ファイルをGitの**ステージングエリアからのみ削除**し、**作業ディレクトリのファイルは保持**します。追跡をやめたい場合に非常に便利です。 |
| `-r` | ディレクトリを再帰的に削除します。 |
| ⭐ `-f`, `--force`| **強制的に削除**します。作業ディレクトリで変更がある場合でも警告なしに削除を実行します。 |
| **その他** | |
| `-n`, `--dry-run` | 実際にファイルを削除せず、何が削除されるかを表示します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 誤ってコミットされた機密情報を含むファイルを、Gitの追跡対象から外し、ローカルファイルは残したい。
* **組み合わせ**: `git rm --cached <secret-file>` → `echo "<secret-file>" >> .gitignore`
* **解説**: `--cached` を使ってGitの追跡のみを停止し、ファイル自体は残します。その後、将来的に再び追跡されないよう、そのファイルを `.gitignore` に追加します。
* **例**:

		```bash
    # .env ファイルをGitの追跡対象から外す

    git rm --cached .env

    # .env を .gitignore に追加

    echo ".env" >> .gitignore

    # 変更をコミット

    git add .gitignore

    git commit -m "Stop tracking .env file"

    ```



## 2. ブルーチーム視点

* **タスク**: 侵害されたシステムで、攻撃者によって追加された悪意のあるファイルをリポジトリから削除する。
* **組み合わせ**: `git rm <malicious-file>`
* **解説**: 調査の結果、Webシェル (`shell.php`など) がリポジトリに追加されていることが判明した場合、このコマンドでファイルを削除し、その削除をコミットして修復します。
* **例**:

		```bash
    # Webシェルを削除

    git rm shell.php

    git commit -m "Security: Remove malicious web shell"

    ```



## 3. レッドチーム視点

* **タスク**: 自身の活動の痕跡を隠蔽するために、一時的に作成したツールやログを削除する。
* **組み合わせ**: `git rm -f <evidence-file>`
* **解説**: `git add` される前に `rm` で削除するのが一般的ですが、もし誤ってステージングしてしまった場合、`git reset` でアンステージしてから `rm` するか、`git rm -f` で強制的に削除して痕跡を消します。

# 環境変数と設定ファイル

* `git rm` に直接影響する独自の環境変数はありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

## 関連する設定ファイル

* **`.gitignore`**: `git rm --cached` で追跡を停止したファイルをこのファイルに追加することが推奨されます。

# セキュリティに関する考慮事項

## 脆弱性と悪用事例

* **脆弱性**: **履歴に残る機密情報**。
* **悪用シナリオ**: `git rm` でファイルを削除しても、そのファイルは**過去のコミット履歴には残ったまま**です。攻撃者は `git log` をさかのぼることで、過去にコミットされて削除された機密情報を見つけ出すことができます。

## GTFOBins / LOLBAS における利用例

`git rm` は直接的なコード実行機能を持たないため、GTFOBinsにはリストされていません。

## 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 機密情報をGitリポジトリにコミットしないためのポリシーを徹底する。
* **Detection (検知)**:
	* 履歴から完全にファイルを削除する必要がある場合は、`git filter-repo` などの専門ツールを使用する。
	* CI/CDパイプラインに `TruffleHog` などのシークレットスキャンツールを導入し、履歴全体をスキャンする。

# 注意点・補足

* **コミットされていない変更**: ファイルに未コミットの変更がある場合、Gitはデフォルトで `git rm` の実行を拒否します。変更を破棄して削除したい場合は `-f` オプションを使用します。
