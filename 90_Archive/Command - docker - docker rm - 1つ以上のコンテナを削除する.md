# Command - docker - docker rm - 1つ以上のコンテナを削除する

## 概要

`docker rm` は、指定した1つ以上のコンテナをDockerホストから削除するためのコマンドです。コンテナを削除すると、そのコンテナの書き込み可能なファイルシステムのレイヤーと、関連するメタデータ（ログ、ネットワーク設定など）がすべて削除されます。デフォルトでは、**停止しているコンテナのみ**を削除できます。不要になったコンテナを削除することで、ディスク容量を解放し、`docker ps -a` の一覧を整理できます。

(出自: `docker` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`docker rmi`** | **イメージ**を削除します。`rm` は**コンテナ**を削除します。コンテナを削除しても、その元となったイメージは残ります。 | - |
| **`docker container prune`**| **停止しているすべてのコンテナ**を一括で削除します。`rm` は削除するコンテナを個別に指定します。 | `prune` (一括削除) |
| **`docker compose down`**| **Composeプロジェクト**に関連するコンテナ、ネットワークなどを一括で削除します。 | - |

## よく連携されるコマンド

`docker rm` は、`docker ps` で削除対象のコンテナIDを特定し、その結果を `xargs` やコマンド置換で渡して一括削除する、というワークフローでよく使われます。

### シナリオ例: 停止している全てのコンテナを一括削除する (インフラ構築・運用視点)

* **目的**: 開発環境などで不要になった、停止 (`Exited`) 状態のコンテナを全てクリーンアップする。
* **連携コマンド**: `docker ps`, `xargs`, `docker rm`
* **解説**: `docker ps -a -q -f "status=exited"` で終了状態のコンテナIDのみをリストアップし、それを `xargs` に渡して `docker rm` で一括削除します。
* **コマンド例**:

		```bash
    # 停止している全てのコンテナを一括で削除する

    docker ps -a -q -f "status=exited" | xargs docker rm

    # 注: この操作は `docker container prune` コマンドでより簡単に行えます。
    ```




## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **動作制御** | |
| ⭐ `<container...>` | 削除するコンテナの名前またはIDを一つ以上、スペース区切りで指定します。 |
| ⭐ `-f`, `--force` | **実行中のコンテナ**を `SIGKILL` で強制的に停止してから削除します。通常は `docker stop` してから `rm` するのが望ましいです。 |
| ⭐ `-v`, `--volumes` | コンテナに紐付いている**匿名ボリューム**も一緒に削除します。名前付きボリュームは削除されません。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 不要になった停止済みのコンテナを個別に削除する。
* **組み合わせ**: `docker rm <container_id_or_name>`
* **解説**: `docker ps -a` で確認した、テスト用に一時的に起動したコンテナや、古いバージョンのコンテナを削除してリソースを整理します。
* **例**:

		```bash
    # 停止している古いWebサーバーコンテナを削除

    docker rm old-webapp-container

    ```




## 2. ブルーチーム視点

* **タスク**: 証拠保全が完了した侵害コンテナを完全に削除し、無力化する。
* **組み合わせ**: `docker rm -v <compromised_container_id>`
* **解説**: インシデント対応において、マルウェアが活動していたコンテナを隔離・停止させ、証拠保全 (`docker commit` など) が完了した後、コンテナを完全に削除します。`-v` を付けることで、コンテナが作成したデータボリュームも同時に削除し、痕跡を消去します。

## 3. レッドチーム視点

* **タスク**: **痕跡消去 (Anti-Forensics)**。攻撃の踏み台として使用したコンテナを削除する。
* **組み合わせ**: `docker rm <attacker_tool_container_id>`
* **解説**: 攻撃者が、侵入したホスト上でスキャンツールなどを一時的にコンテナとして実行した場合、目的を達成した後にそのコンテナを `docker rm` で削除します。これにより、コンテナのファイルシステムがホストから消え、調査を困難にします。

# エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `Error response from daemon: You cannot remove a running container <container_id>. Stop the container before attempting removal or force remove`
		* **考えられる原因**: **最もよくあるエラー**。実行中のコンテナを削除しようとしました。
		* **解決策**:
				* 安全な方法: `docker stop <container_id>` でコンテナを停止してから、`docker rm <container_id>` を実行します。
				* 強制的な方法: `docker rm -f <container_id>` を実行します。

# 環境変数と設定ファイル

* `docker rm` の動作に直接影響する環境変数や設定ファイルは通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

# セキュリティに関する考慮事項

## 脆弱性と悪用事例

* **脆弱性**: `docker rm` は**破壊的なコマンド**です。
* **悪用シナリオ**: 攻撃者や内部の操作ミスにより、本番環境で稼働中の重要なコンテナが `docker rm -f` で削除されると、即座に**サービス妨害 (DoS)** に繋がります。また、インシデント発生時にブルーチームが調査を行う前にコンテナを `rm` されると、**証拠が破壊**されます。

## GTFOBins / LOLBAS における利用例

`docker rm` は `docker` コマンドの一部として扱われます。`sudo docker` の権限があれば、任意のコンテナを削除可能です。

## 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: Dockerソケットへのアクセス権限を厳格に管理する。本番環境でのコンテナ削除は、変更管理プロセスを通じて行う。
* **Detection (検知)**: `auditd` で `docker rm` コマンドの実行を監視し、計画外のコンテナ削除がないかを確認する。

# 注意点・補足

* **ボリュームの削除 (`-v`)**: `-v` オプションで削除されるのは、Dockerが自動で作成した**匿名ボリューム**のみです。`docker run -v my-volume:/data` のように作成した**名前付きボリューム**は、`docker rm -v` では削除されません。
