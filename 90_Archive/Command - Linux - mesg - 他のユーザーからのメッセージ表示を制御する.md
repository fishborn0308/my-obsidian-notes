# Command - Linux - mesg - 他のユーザーからのメッセージ表示を制御する

## 概要

`mesg` は、`write` コマンドや `talk` コマンドを使って他のユーザーが自分のターミナル画面にメッセージを表示することを許可するか、禁止するかを設定します。多くの Linux ディストリビューションではデフォルトで「許可（y）」に設定されています。

## 🛠 関連コマンド

| コマンド | 役割 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `write` | 特定のユーザーの画面にメッセージを直接送りつける。 | 1対1の簡易連絡 |
| `talk` | 画面を分割してリアルタイムでチャットを行う。 | 1対1の対話 |
| `wall` | ログイン中の**全ユーザー**に一斉メッセージを送る。 | 緊急メンテナンス通知 |
| `mesg` | **「受信側」**として、これらを拒否するか決める。 | **作業中の集中維持** |

## 基本的な使い方

引数なしで実行すると、現在の設定が表示されます。

| コマンド | 説明 |
| :--- | :--- |
| ⭐️ `mesg y` | メッセージの受信を**許可**する（is y）。 |
| ⭐️ `mesg n` | メッセージの受信を**禁止**する（is n）。 |

## 業務シナリオ別活用法

### 1. インフラ構築・運用視点

* **タスク**: 重要なログの監視中や、デモ中の画面保護。
* **解説**: 画面を凝視してデバッグしている最中に、他人の `write` メッセージが割り込んでログが読めなくなるのを防ぎます。
* **例**:
    ```bash
    # 集中モード：メッセージを遮断
    mesg n
    ```

### 2. ブルーチーム視点

* **タスク**: ターミナルを利用したソーシャルエンジニアリングの防止。
* **解説**: 攻撃者が `write` コマンドを使い、システム管理者を装って「パスワードを再入力してください」などの偽のプロンプトを表示させる「ターミナル・スプーフィング」を未然に防ぎます。

### 3. レッドチーム視点

* **タスク**: ターゲットの注意を逸らす（ディストラクション）。
* **解説**: 管理者が重要な作業を行っている最中に、大量のメッセージや偽の警告を `write` で送りつけ、判断力を鈍らせたり、ミスを誘発させたりすることを検討します（`mesg n` されていない場合に限ります）。

## 注意点・補足

* **root の特権**: あなたが `mesg n` を設定していても、**root ユーザー（管理者）からの `write` や `wall` によるメッセージは拒否できません**。管理者の声は常に届きます。
* **永続性**: `mesg` コマンドによる設定は、そのログインセッションの間だけ有効です。常に拒否したい場合は、`.bashrc` などの設定ファイルに記述しておく必要があります。
* **エラー**: `mesg n` のユーザーに `write` しようとすると、送信側に 「write: <user> has messages disabled」 というエラーが表示されます。

## セキュリティに関する考慮事項

### 脆弱性とリスク

* **情報の露出**: `mesg` の設定自体に大きな脆弱性はありませんが、`write` を許可していると、意図しないメッセージの受け取りによってターミナルのエスケープシーケンスが悪用され、画面の表示が壊されたり、特定の状況下でコマンドを実行させられたりするリスクが理論上存在します。

### 対応策・緩和策

* **基本は n**: サーバー上での作業において、他のユーザーと `write` で会話する必要がないのであれば、セキュリティと平穏のために `mesg n` をデフォルトにしておくのが賢明です。
