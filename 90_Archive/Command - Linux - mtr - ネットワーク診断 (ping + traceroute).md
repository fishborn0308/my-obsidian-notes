# Command - Linux - mtr - ネットワーク診断 (ping + traceroute)

## 概要

`mtr` は、パケットが目的のホストに到達するまでの全経路（ホップ）を継続的に調査し、各地点でのパケットロス率や遅延（レイテンシ）をリアルタイムで表示します。一時的な不調ではなく、継続的な統計を確認できるため、ボトルネックの特定に非常に強力です。

## 🛠 ネットワーク調査ツールの比較

| コマンド | 役割 | 弱点 |
| :--- | :--- | :--- |
| `ping` | 到達性と応答速度の確認。 | 経路のどこで遅れているか不明。 |
| `traceroute` | 経路のリストアップ。 | その瞬間の情報のみで、統計が取れない。 |
| **`mtr`** | **経路ごとの継続的な統計。** | インストールが必要な場合が多い。 |

## 主要なオプション

| オプション | 説明 |
| :--- | :--- |
| ⭐️ `-r`, `--report` | レポートモード。指定回数試行した結果をまとめて出力して終了する。 |
| ⭐️ `-c <count>` | 試行回数を指定（デフォルトは無制限）。 |
| ⭐️ `-i <seconds>` | パケット送信の間隔を指定（デフォルトは1秒）。 |
| ⭐️ `-w`, `--report-wide` | ホスト名を省略せずにフルで表示する。 |
| ⭐️ `-z`, `--aslookup` | 各ホップの AS 番号（自律システム番号）を表示する。 |
| `-n`, `--no-dns` | IP アドレスの逆引きを行わず、数字のまま表示する（高速）。 |

## 📊 出力の読み方（ここが重要！）

* **Loss%**: パケットロス率。ここが 0% でない場所がトラブルの原因。
* **Snt**: 送信済みパケット数。
* **Last**: 直近の遅延 (ms)。
* **Avg**: 平均遅延。
* **Best / Wrst**: 最小 / 最大遅延。
* **StDev**: 標準偏差。数値が大きいほど遅延が不安定。

## 業務シナリオ別活用法

### 1. インフラ構築・運用視点

* **タスク**: 「特定の拠点からだけ接続が遅い」というクレームの調査。
* **解説**: 自社ネットワーク内、ISP 間、あるいはクラウド事業者の入口、どこでパケットが落ちているかを一目で特定します。
* **例**:
    ```bash
    # Google DNS までの経路を AS 番号付きで調査
    mtr -z 8.8.8.8
    ```

### 2. ブルーチーム視点

* **タスク**: BGP ハイジャッキングや不審な経路変更の検知。
* **解説**: 普段通るはずのない国や ISP をパケットが経由していないか、`-z` オプションで AS 番号を確認しながら監視します。また、DDoS 攻撃を受けている際に、どのルートから負荷が来ているかの分析にも使います。

### 3. レッドチーム視点

* **タスク**: ターゲットネットワークのトポロジー（構造）把握。
* **解説**: 外部からターゲットへ `mtr` を実行し、経路上にあるファイアウォールや IPS（侵入防止システム）の位置を推測します。特定のホップで急にパケットが消える場合、そこにフィルタリングデバイスがある可能性が高いです。

## 注意点・補足

* **ICMP 制限**: 途中のルーターが ICMP パケット（ping）を制限している場合、特定のホップだけ `Loss 100%` と表示されることがありますが、その先のホップが正常なら問題ありません。
* **インストール**: `sudo apt install mtr` (Ubuntu/Debian) や `sudo dnf install mtr` (RHEL) で導入が必要です。

## セキュリティに関する考慮事項

### 脆弱性とリスク

* **情報の露出**: 内部ネットワークから外部へ `mtr` を許可していると、内部ルーターの IP アドレスや構造が外部に漏洩するヒントになります。

### 対応策・緩和策

* **ICMP 制御**: 境界ルーターやファイアウォールで、不必要な TTL 超過通知 (ICMP Type 11) を制限することで、外部からの経路探索を難読化できます。
