# Command - Windows - cmd - break - 拡張 CTRL+C チェックの設定・表示

## 概要

`break` は、MS-DOS システムにおいて「拡張 CTRL+C チェック」機能を有効または無効にするためのコマンドです。

本来は、プログラムがディスク I/O 以外の処理（計算中など）を行っている最中でも、`CTRL+C` による中断を受け付けるかどうかを制御するものでした。

現代の Windows (Windows 10/11 等) では、この設定は常に有効な状態に近いか、あるいは OS レベルで別の制御が行われているため、コマンドを実行しても**システムの挙動に影響を与えません。**

(出自: `Windows 標準搭載 - cmd.exe 内蔵コマンド (MS-DOS 互換)`)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 (あれば) |
| :--- | :--- | :--- |
| `CTRL+C` | 実際に実行中のプロセスに割り込み信号を送るキーボードショートカット。 | - |
| `taskkill` | 応答しなくなったプログラムを強制終了させる現代的な手法。 | `taskkill` |

## よく連携されるコマンド

現代のシナリオでは、他のコマンドと連携して実務的な効果を発揮することはありません。主に古いバッチファイル内での設定記述として見かけます。

### シナリオ例: レガシーバッチスクリプト内での記述 (インフラ運用視点)

* **目的**: 古い DOS ベースのアプリケーションを実行する際、ユーザーによる中断を確実に受け付けるようにする。
* **解説**: スクリプトの冒頭で `break on` を宣言することが慣習となっていました。
* **コマンド例**:
    ```cmd
    REM 拡張チェックを有効にする (現代のWindowsでは無視される)
    break on
    REM 何らかの処理を実行
    legacy_app.exe
    ```

## スイッチ/オプション説明

| スイッチ/オプション | 説明 |
| :--- | :--- |
| **引数なし** | 現在の `break` の設定状態（ON または OFF）を表示する。 |
| `on` | 拡張 CTRL+C チェックを有効にする。 |
| `off` | 拡張 CTRL+C チェックを無効にする。 |

## よく使われる組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 非常に古い産業用 PC や DOS 互換環境でのスクリプト動作確認。
* **解説**: 現代の Windows OS 上で実行しても `BREAK は ON です` または `BREAK は OFF です` と表示されるだけで、実害も実効もありません。
* **例**:
    ```cmd
    C:\> break
    BREAK は ON です
    ```

### 2. ブルーチーム視点

* **タスク**: レガシーシステムから移行したバッチファイルの監査。
* **解説**: 悪意のあるコードが含まれているわけではありませんが、古い慣習で残っている記述として識別します。

### 3. レッドチーム視点

* **タスク**: シェル脱出（Shell Escape）の試行。
* **解説**: 極めて稀なケースですが、制限されたシェル環境で `CTRL+C` の挙動を操作しようとする際の足掻きとして試されることがあります（ただし現代の Windows では効果なし）。

## エラーメッセージとトラブルシューティング

### よくあるエラーメッセージ

* **特になし**: どの引数を与えてもエラーになることは殆どなく、単に無視されるか状態が表示されるだけです。

## 環境変数と設定ファイル

* **`CONFIG.SYS`**: かつての MS-DOS 環境では、`BREAK=ON` という形式で起動時に設定されていました。現代の Windows には該当する設定ファイルはありません。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: 直接的な脆弱性は報告されていません。
* **悪用シナリオ**: ありません。このコマンド自体が現代のセキュリティ境界に影響を与えることはありません。

### LOLBAS における利用例

* **機能**: なし。`break` は LOLBAS プロジェクトの対象ではありません。

### 対応策・緩和策 (ブルーチーム視点)

* **対応**: 現代のシステムでは動作しないコマンドであるため、特に監視や制限を行う必要はありません。

## 注意点・補足

* **バッチファイルの `break`**: プログラミング言語の `for` ループを抜ける `break` 文と混同しないよう注意してください。cmd.exe のバッチファイルにはループを抜けるための `break` 予約語は存在せず、代わりに `goto` 等を使用します。
* **互換性**: Microsoft はこのコマンドを「旧バージョンとの互換性のためだけ」に保持しています。
