# Command - git - git diff - 変更内容の差分を表示する

## 概要

`git diff`コマンドは、Gitリポジトリ内の2つの異なる状態間の**差分**、すなわち変更内容を行単位で表示します。このコマンドは、作業ディレクトリ、ステージングエリア、コミット、ブランチなど、さまざまなGitオブジェクト間の変更を比較するために使用されます。これにより、どのコードが追加、削除、または変更されたかを詳細に把握できます。

(出自: `git` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`git status`** | `git status`は「何が変更されたか」をファイルリストで示します。`git diff`は**具体的な変更内容**を行レベルで表示します。 | - |
| **`git log -p`** | 各コミットの差分も合わせて表示しますが、コミットされた変更の履歴が対象です。`git diff`は、まだコミットされていない変更や、任意の2つのコミット間の差分を比較します。 | - |

## よく連携されるコマンド

`git diff`は、変更内容の確認が主目的であるため、変更を準備・管理する`git add`や`git commit`と密接に連携します。

### シナリオ例: コミット前の変更内容を確認する (共通)

* **目的**: コミットする前に、作業ディレクトリで行った変更が意図通りであるか、またはステージングエリアに追加した変更が正しいかを最終確認する。
* **連携コマンド**: `git add`, `git commit`
* **解説**: ファイルを変更した後、まず`git diff`で作業ディレクトリでの変更を確認します。次に、変更を`git add`でステージした後、`git diff --staged`でステージされた変更を確認します。
* **コマンド例**:

		```bash
    # 作業ディレクトリで行った、まだステージされていない変更を確認

    git diff

    # 特定のファイルをステージングエリアに追加

    git add src/main.py

    # ステージングエリアに追加された変更を確認

    git diff --staged

    # 確認後、変更をコミット

    git commit -m "feat: Add user authentication logic"

    ```





## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **比較対象** | |
| ⭐ `[引数なし]` | 作業ディレクトリにあるが、まだステージングされていない変更を表示します。 |
| ⭐ `--staged` / `--cached` | ステージングエリアにある変更を、最後のコミットと比較して表示します。 |
| ⭐ `<commit1> <commit2>` | 指定された2つのコミット間の差分を表示します。 |
| `<branch1>..<branch2>` | 2つのブランチの先端コミット間の差分を表示します。 |
| **出力形式** | |
| `--name-only` | 変更されたファイルの名前のみを表示します。 |
| `--stat` | 変更されたファイルの概要と、変更された行数の統計情報を表示します。 |
| `--color-words` | 行単位ではなく、単語単位での変更を色付きで表示します。 |
| **その他** | |
| `-w` | 空白の変更を無視して差分を表示します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 本番環境にデプロイする前に、前回のデプロイからの設定変更の全体像を把握したい。
* **組み合わせ**: `git diff --stat HEAD~1`
* **解説**: `--stat`で前回のコミットからの変更概要を素早く確認し、意図しない設定変更がないかを確認します。
* **例**:

		```bash
    # 最新のコミットで行われた変更の統計情報を表示

    git diff --stat HEAD~1

    ```





## 2. ブルーチーム視点

* **タスク**: 侵害調査において、Webアプリケーションコードにバックドアが注入されていないかを確認する。
* **組み合わせ**: `git diff <known-good-commit> HEAD -- *.php *.js`
* **解説**: 既知の安全な状態のコミットと現在のHEAD間の差分を、とくにスクリプトファイルに絞って確認することで、不正なコードの追加や改ざんを発見します。
* **例**:

		```bash
    # 既知の安全なコミットと現在のHEAD間で、PHPとJSファイルに絞った変更内容を確認

    git diff <known-good-commit> HEAD -- *.php *.js

    ```





## 3. レッドチーム視点

* **タスク**: ターゲットシステム上でコードを実行した後、その活動の痕跡がGitリポジトリに記録されていないかを確認する。
* **組み合わせ**: `git diff HEAD~1 HEAD --name-only -- *.log`
* **解説**: 自身の活動によって生成されたログファイルなどがGitに追跡されていないかを確認します。
* **例**:

		```bash
    # 直前のコミットと現在の作業ディレクトリ間で、生成されたログファイルがGitに追跡されていないことを確認

    git diff HEAD~1 HEAD --name-only -- *.log

    ```





## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `fatal: Not a git repository`
		* **考えられる原因**: 現在のディレクトリがGitリポジトリではありません。
		* **解決策**: 正しいGitリポジトリのディレクトリに移動してください。

* `git diff`に直接影響を与える独自の環境変数はありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `git diff` 自体に直接的な脆弱性は稀ですが、**出力内容**が**情報漏洩**に繋がる可能性があります。
* **悪用シナリオ**: 差分内に、誤ってコミットされたパスワードやAPIキーなどの機密情報が含まれている場合、`git diff` の出力を見ることでそれらが漏洩します。

### GTFOBins / LOLBAS における利用例

`git diff` は情報表示に特化したコマンドのため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **厳格なコードレビュー**。プルリクエストの差分を必ず確認し、機密情報の混入がないかチェックする。
* **Detection (検知)**: CI/CDパイプラインに `TruffleHog` などのシークレットスキャンツールを導入し、差分に機密情報が含まれていないか自動でスキャンする。

## 注意点・補足

* **出力形式の理解**: `git diff`の出力は、変更されたファイル名、変更タイプ、行番号、そして変更された行（`+`は追加、 `-`は削除）から構成されます。この形式を理解することが重要です。
