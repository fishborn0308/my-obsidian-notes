# Command - git - git switch - ブランチを切り替える

## 概要

`git switch` コマンドは、Git 2.23で導入された新しいコマンドで、ブランチの切り替え操作をより直感的かつ安全に行うために設計されました。以前の `git checkout` コマンドからブランチ切り替えの機能が分離され、ブランチの作成、切り替え、そして特定のコミットやタグへの移動をシンプルに実行できます。

(出自: `git` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`git checkout`** | `checkout` はブランチ切り替えとファイル復元の両方の機能を持っていました。`switch` は**ブランチの切り替えに特化**しており、意図が明確で安全です。 | `git switch` |

## よく連携されるコマンド

`git switch`は、ブランチベースの開発ワークフローにおいて、さまざまなGitコマンドと組み合わせて使用されます。

### シナリオ例: 新しい機能ブランチを作成し、そこに切り替える (共通)

* **目的**: 新しい機能開発を開始するために、`main`ブランチから新しいブランチを作成し、すぐにそのブランチに移動して作業を始める。
* **連携コマンド**: `git status`, `git branch`
* **解説**: `git switch -c`コマンドを使って、新しいブランチを作成し、同時にそのブランチへ切り替えます。
* **コマンド例**:

		```bash
    # main ブランチから新しい機能ブランチ 'feature/new-api' を作成し、切り替える

    git switch -c feature/new-api

    ```




## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **ブランチ操作** | |
| ⭐ `<branch-name>` | 指定した既存のブランチに切り替えます。 |
| ⭐ `-c <new-branch>` | 新しいブランチを作成し、そのブランチに切り替えます。 |
| `-C <new-branch>` | `-c` と似ていますが、同名のブランチが存在する場合、強制的に上書きします。 |
| ⭐ `-` | 直前に作業していたブランチに切り替えます。 |
| `--orphan <new-branch>` | 履歴を持たない**孤立した新しいブランチ**を作成します。 |
| **その他** | |
| `--detach <commit>` | 特定のコミットに切り替えて、「HEADを分離」状態にします。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: IaC (Infrastructure as Code) の変更を行う際、常に`main`ブランチから新しい設定ブランチを作成し、その中で作業を進める。
* **組み合わせ**: `git switch -c <config-branch> main`
* **解説**: これにより、本番環境への変更が他の開発作業と混ざることなく、独立した形で管理でき、レビューやデプロイのプロセスが明確になります。
* **例**:

		```bash
    # main から新しい設定変更ブランチを作成し、切り替える

    git switch -c feature/vpc-update

    ```




## 2. ブルーチーム視点

* **タスク**: インシデント発生時に、特定の時点のコードベースの状態を正確に調査する。
* **組み合わせ**: `git switch --detach <suspected-commit-id>`
* **解説**: `--detach`を使うことで、特定のコミット状態を「固定」し、その上で影響を与えずに分析を進められます。
* **例**:

		```bash
    # 特定のコミットID(例: abcd123)に移動し、その状態を調査

    git switch --detach abcd123

    ```




## 3. レッドチーム視点

* **タスク**: これまでのリポジトリ履歴とは完全に切り離された新しいマルウェア開発プロジェクトを開始したい。
* **組み合わせ**: `git switch --orphan <new-project-branch>`
* **解説**: `--orphan`を使用することで、以前のコミット履歴が意図せず含まれることを防ぎ、クリーンな状態から新しいツールの開発を始められます。
* **例**:

		```bash
    # 履歴を持たない新しい孤立ブランチ 'stealthy-payload' を作成

    git switch --orphan stealthy-payload

    ```




## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `error: you have uncommitted changes. Please commit or stash them.`
		* **考えられる原因**: 現在のブランチにコミットされていない変更がある状態で、別のブランチに切り替えようとしました。
		* **解決策**:
				1. 変更をコミットする: `git commit -m "WIP"`
				2. 変更を一時退避する: `git stash`
				3. 変更を破棄する: `git restore .`

## 環境変数と設定ファイル

* `git switch` に直接影響する独自の環境変数はありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `git switch` 自体に直接的な脆弱性は稀ですが、**不適切なブランチへの切り替え**が機密情報の露出に繋がる可能性があります。
* **悪用シナリオ**: 攻撃者が侵害したシステム上で`git switch`を使用し、特定のブランチ（例: 既知の脆弱性を含む古いブランチ）に切り替えることで、その環境をマルウェア開発などに利用する可能性があります。

### GTFOBins / LOLBAS における利用例

`git switch` は直接的なコード実行機能を持たないため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 主要なブランチに**ブランチ保護ルール**を設定し、レビューなしでのマージや強制プッシュを禁止する。
* **Detection (検知)**: Gitホスティングサービスの監査ログを監視し、不審なブランチの作成や切り替えがないかを確認する。

## 注意点・補足

* **HEADの分離状態 (`detached HEAD`)**: `git switch --detach <commit-id>` を使用すると、「HEADが分離された状態」になります。この状態でのコミットはどのブランチにも属さず、一時的な調査目的で使うべきです。
