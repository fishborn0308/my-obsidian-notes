# Command - Linux - nohup - ハングアップ信号を無視してコマンドを実行する

## 概要

`nohup` は、ターミナルが閉じられたり、ネットワークが切断されたりした際に送られる `SIGHUP` (Hangup) 信号を無視してコマンドを実行します。通常、バックグラウンド実行を意味する `&` と組み合わせて使用されます。

## 🛠 関連コマンド・機能の比較

| 手法 | 役割 / 特徴 | 推奨シーン |
| :--- | :--- | :--- |
| **`nohup`** | 信号を無視するだけ。シンプル。 | 単発の長時間スクリプト実行 |
| `&` | プロセスをバックグラウンドへ送る（SIGHUP は受ける）。 | ターミナルを開いたままの並行作業 |
| `disown` | 既に実行中のジョブを SIGHUP 対象から外す。 | 実行後に「あ、これ時間かかる」と気づいた時 |
| `screen` / `tmux` | 仮想端末を作成し、セッションごと維持する。 | 複数の窓を使い、後で再開したい時 |

## 基本的な使い方

```bash
# 基本形: nohup <コマンド> &
nohup ./long_running_script.sh &
````

### 📝 出力の行き先

`nohup` を使うと、標準出力と標準エラー出力の行き先が自動的に処理されます。

1. 出力をリダイレクトしていない場合、カレントディレクトリの **`nohup.out`** に書き込まれます。
2. カレントディレクトリに書けない場合は、`~/nohup.out` に書かれます。

## 業務シナリオ別活用法

### 1. インフラ構築・運用視点

- **タスク**: 数時間かかるデータベースのバックアップや、大規模なデータ移行。
- **解説**: 途中で Wi-Fi が切れてバックアップが失敗する悲劇を防ぎます。
- **例**:

    Bash

    ```
    # 出力を指定し、ログアウト後も実行を維持
    nohup rsync -avz /large_data/ backup_server:/data/ > transfer.log 2>&1 &
    ```

### 2. ブルーチーム視点

- **タスク**: サーバーの異常を監視するアドホックなスクリプトの実行。
- **解説**: 管理者が調査を終えてログアウトした後も、監視スクリプトを動かし続け、翌朝 `nohup.out` で結果を確認します。

### 3. レッドチーム視点

- **タスク**: 侵入後の隠密な情報収集や、持続的なスキャン。
- **解説**: 自分がログアウトした後も、ターゲットのネットワークをじわじわとスキャンするスクリプトを動かし続けるために使います。

## セキュリティに関する考慮事項

### 脆弱性とリスク

- **情報の露出 (Information Disclosure)**: ⭐️ **注意**。

    `nohup.out` には、実行したコマンドの出力（デバッグ情報やエラーログなど）がすべて記録されます。ここに機密情報（パスワードや API キー、内部 IP アドレスなど）が含まれていると、後から他のユーザーに読まれてしまうリスクがあります。

### 対応策・緩和策 (ブルーチーム視点)

- **リダイレクトの徹底**: 機密を扱う場合は `/dev/null` に捨てるか、適切な権限を設定したログファイルへ出力させます。
- **後片付け**: 処理が終わった後の `nohup.out` を放置せず、不要であれば削除します。

## 注意点・補足

- **`nohup` は「ログアウト」には強いが「再起動」には無力**: サーバー自体が再起動すればプロセスは死にます。再起動後も動かしたい場合は `systemd` などのサービス管理を使います。
- **優先度の変化**: 実装によっては、`nohup` で実行すると Nice 値（優先度）が少し下げられる（謙虚になる）ことがあります。
