# Command - Linux - auditctl - 監査システムのリアルタイム制御・ルール管理

## 概要

`auditctl` は、Linuxカーネルの監査サブシステムを制御するためのコマンドラインツールです。監査デーモン (`auditd`) の状態確認、ルールの動的な追加・削除、システムコールの監視設定などを行います。

(出自: `audit` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `auditd` | ログを収集・保存するデーモン。`auditctl` はこの動作を「制御」する。 | デーモン本体 |
| `ausearch` | 出力されたログを検索する。`auditctl` はログを出す「条件」を決める。 | ログ検索 |
| `sysctl` | カーネルパラメータ全般を操作するが、監査に特化したルール操作はできない。 | カーネル設定 |

## よく連携されるコマンド

### シナリオ例: ルールの永続化と検証 (運用視点)

* **目的**: `auditctl` でテストした一時的なルールを、再起動後も有効にする。
* **連携コマンド**: `augenrules`
* **解説**: `auditctl` で追加したルールは再起動で消失します。`/etc/audit/rules.d/` 下のファイルを編集した後、`augenrules` を実行して反映させます。
* **コマンド例**:
    ```bash
    # 現在の動的ルールをファイルから再生成して適用
    sudo augenrules --load
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **システム管理** | |
| ⭐️ `-s` | 監査システムの現在のステータス（有効か、エラー数、未処理ログ数など）を表示。 |
| ⭐️ `-l` | 現在ロードされているすべてのルールを一覧表示。 |
| `-e [0|1|2]` | 監査の有効化(1)/無効化(0)。`2` を指定すると設定がロック（不変）される。 |
| **ルール追加 (Watch)** | |
| `-w <path>` | 指定したファイルまたはディレクトリを監視対象にする。 |
| `-p [r|w|x|a]` | 監視する権限。r=read, w=write, x=execute, a=attribute (属性変更)。 |
| `-k <key>` | ログに付与する任意の識別キーワード（検索時に使用）。 |
| **ルール追加 (Syscall)** | |
| `-a <action>,<list>` | システムコールルールを追加。例: `always,exit` |
| `-S <syscall>` | 監視するシステムコール名または番号を指定（execve, open, connectなど）。 |
| **削除** | |
| `-W <path>` | 指定したパスの監視ルールを削除。 |
| `-D` | すべてのルールを削除する。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 特定の機密ディレクトリへの不正アクセス監視。
* **組み合わせ**: `-w /etc/ssh/ -p wa -k ssh_config_change`
* **解説**: SSH設定ディレクトリ全体を監視。誰かが設定を書き換えたり(w)、属性(a)を変えたりした際に即座にログを生成します。
* **例**:
    ```bash
    # SSH設定の変更を監視し、"ssh_config_change"というタグを付ける
    sudo auditctl -w /etc/ssh/ -p wa -k ssh_config_change
    ```

### 2. ブルーチーム視点

* **タスク**: リバースシェルや不審な通信プロセスの検知。
* **組み合わせ**: `-a always,exit -F arch=b64 -S connect -k network_conn`
* **解説**: ネットワーク接続（connectシステムコール）を監視します。通常の Web サーバであれば発生しない不自然な外部接続をログから追跡できます。
* **例**:
    ```bash
    # 64bitシステムでの外部接続試行をすべて記録
    sudo auditctl -a always,exit -F arch=b64 -S connect -k network_conn
    ```

### 3. レッドチーム視点

* **タスク**: 自身の活動が検知されるかどうかの事前チェック。
* **組み合わせ**: `-l`
* **解説**: 侵入後、どのファイルやディレクトリが監視されているかを `auditctl -l` で確認します。`/tmp` が監視されていないなら、そこで作業を進めるなどの判断材料にします。
* **例**:
    ```bash
    # 監査ルールを読み取り、監視の網を特定する
    auditctl -l
    ```

## エラーメッセージとトラブルシューティング

1.  **エラーメッセージ例 1**: `Error sending add rule data request (Invalid argument)`
    * **考えられる原因**: システムコールの名前が間違っている、またはアーキテクチャ (`-F arch=b64`) の指定が抜けている。
    * **解決策**: システムコール名が正しいか `ausyscall --dump` で確認し、適切なアーキテクチャを指定する。

2.  **エラーメッセージ例 2**: `The audit system is in immutable mode, no rule changes allowed`
    * **考えられる原因**: 監査システムが `-e 2`（不変モード）でロックされている。
    * **解決策**: OSを再起動するまでルール変更は不可。これはブルーチームによる堅牢化の成果。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `auditctl` 自体に脆弱性があるというより、**不適切な設定（監視漏れ）**が脆弱性となります。
* **悪用シナリオ**: 攻撃者が root 権限を得た後、`auditctl -D` を実行して全てのルールを消去し、自らの足跡を消す「ログ消去」に利用されます。

### GTFOBins / LOLBAS における利用例

`auditctl` はそれ自体でシェルを奪うものではありませんが、特権昇格後の**隠蔽（Defense Evasion）**において中心的な役割を果たします。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 監査ルールの最後に必ず `-e 2` を設定し、攻撃者が侵入後に `auditctl` でルールを消せないようにする。
* **Detection (検知)**: `auditctl` コマンド自体の実行（特に `-D` や `-e 0`）を監視し、実行されたら即座にアラートを上げる。

## 注意点・補足

* **ルール評価の順序**: ルールは上から順に評価されます。特定のディレクトリを除外したい場合は、監視ルールの前に除外ルールを書く必要があります。
* **再起動時の注意**: `auditctl` で入力したコマンドはメモリ上にしか残らないため、必ず `/etc/audit/rules.d/audit.rules` への反映をセットで行ってください。
