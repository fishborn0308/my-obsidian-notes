# Command - Linux - dmesg - カーネルのメッセージバッファを表示・制御する

## 概要

`dmesg` は、Linux カーネルが起動時や稼働中に生成するメッセージを表示します。これらのメッセージは「リングバッファ」と呼ばれるメモリ領域に保存されており、ストレージがマウントされる前の初期段階のログも確認できるのが特徴です。

(出自: `util-linux` パッケージに含まれる)

## 類似・関連コマンド

| コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `journalctl -k` | systemd環境でカーネルログを表示。永続化された過去のログも検索可能。 | 過去のログ調査 |
| `cat /var/log/kern.log` | ファイルとして保存されたカーネルログ。ディストリビューションに依存。 | ログファイル確認 |
| `dmesg` | 現在メモリ上にある最新のログを即座に表示。非常に高速。 | **リアルタイムの異常確認** |

## よく連携されるコマンド

### シナリオ例: USBデバイスやディスクの認識確認 (運用・保全視点)

* **目的**: 新しく接続したデバイスがどのデバイス名（`/dev/sdb` 等）で認識されたか特定する。
* **連携コマンド**: `tail`
* **解説**: 接続直後に実行し、末尾の数行を確認します。
* **コマンド例**:
    ```bash
    # デバイス接続直後に最新の数行を確認
    dmesg | tail -n 20
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **表示制御** | |
| ⭐️ `-T`, `--ctime` | タイムスタンプを人間が読みやすい形式（日付と時刻）で表示する。 |
| ⭐️ `-w`, `--follow` | 新しいメッセージをリアルタイムで待ち受け、表示し続ける。 |
| ⭐️ `-l <level>` | 指定したログレベル（err, warn, info, debug 等）のみを表示する。 |
| `-n <level>` | コンソールに出力するログのレベルを設定する。 |
| **制御 (要root)** | |
| `-C`, `--clear` | リングバッファの内容を消去する。 |
| `-c` | 内容を表示した後、バッファを消去する。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: OOM Killer（メモリ不足によるプロセス強制終了）の発生確認。
* **組み合わせ**: `dmesg -T | grep -i oom`
* **解説**: システムが重くなった際、カーネルがメモリ確保のためにどのプロセスを殺したかを特定します。
* **例**:
    ```bash
    # OOM Killer の履歴を時刻付きで検索
    sudo dmesg -T | grep -i "out of memory"
    ```

### 2. ブルーチーム視点

* **タスク**: セグメンテーション違反（Segmentation Fault）による攻撃試行の検知。
* **組み合わせ**: `dmesg -l err,warn`
* **解説**: バッファオーバーフロー攻撃などが失敗した際、ターゲットプロセスがクラッシュした記録がカーネルログに残ります。
* **例**:
    ```bash
    # プロセスの異常終了（segfault）を監視
    dmesg -w | grep -i segfault
    ```

### 3. レッドチーム視点

* **タスク**: 仮想化環境の検知（Anti-VM）およびカーネル脆弱性（Exploit）の成否確認。
* **組み合わせ**: `dmesg | grep -iE "vmware|virtualbox|qemu"`
* **解説**: 起動ログからハイパーバイザ固有のデバイス名を特定し、サンドボックス環境かどうかを判断します。また、カーネルエクスプロイト実行後の成功/失敗メッセージを確認します。
* **例**:
    ```bash
    # 仮想環境特有の文字列を検索
    dmesg | grep -i hypervisor
    ```

## エラーメッセージとトラブルシューティング

1.  **エラーメッセージ例 1**: `dmesg: read kernel buffer failed: Operation not permitted`
    * **原因**: 最近の Linux カーネル（特に Ubuntu 等）では、セキュリティ上の理由で一般ユーザーの `dmesg` 閲覧が制限されている（`kernel.dmesg_restrict = 1`）。
    * **解決策**: `sudo dmesg` を使用する。

2.  **タイムスタンプが [ 1234.5678] のような形式**:
    * **解決策**: これはシステム起動時からの経過秒数です。`-T` オプションを付けて「実時刻」に変換しましょう。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **情報漏洩 (ASLR回避)**: `dmesg` に出力されるメッセージには、カーネル内のメモリアドレスが含まれることがあります。これが一般ユーザーに露出していると、ASLR (Address Space Layout Randomization) をバイパスするためのヒントを攻撃者に与えてしまいます。

### GTFOBins における利用例

`dmesg` は読み取り専用の性質が強いため、直接的なシェル取得などには使われませんが、`sudo` 権限がある場合、本来見えない特権的な情報を収集する偵察ツールとして極めて優秀です。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: `/etc/sysctl.conf` に `kernel.dmesg_restrict = 1` を設定し、root 以外が閲覧できないようにする。
* **Detection (検知)**: 一般ユーザーが `dmesg` を繰り返し実行している場合、システム情報の収集を試みている可能性があるため、監視対象とする。

## 注意点・補足

* **バッファの上書き**: リングバッファはサイズが固定されているため、古いメッセージは新しいメッセージによって順次上書きされます。長期間のログが必要な場合は `/var/log/` 系のファイルを参照してください。
