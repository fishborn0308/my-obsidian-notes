# Command - Linux - rename - 複数のファイル名を一括で変更する

## 概要
`rename` は、指定したパターンに基づいて複数のファイル名を一括置換します。
**注意:** Linux には大きく分けて 2 種類の `rename` が存在します。
1.  **Perl版 (prename)**: Debian/Ubuntu系で一般的。正規表現（`s/old/new/`）が使えて非常に強力。
2.  **C版 (util-linux)**: RHEL/CentOS系で一般的。単純な文字列置換のみを行う。
※ここでは、より多機能で現場で重宝される **Perl版** を中心に解説します。



## 🛠 `mv` vs `rename`
| 項目 | `mv` | `rename` |
| :--- | :--- | :--- |
| **対象数** | 基本的に 1 つ。 | **大量のファイルを一括処理。** |
| **指定方法** | 移動先を直接指定。 | **置換ルール（パターン）を指定。** |
| **得意分野** | 名前の変更、場所の移動。 | 命名規則の一斉変更（拡張子変更など）。 |

## 主要なオプション
破壊的な変更を伴うため、以下のオプションの使いこなしが生存の鍵となります。

| オプション | 説明 |
| :--- | :--- |
| ⭐️ `-n`, `--dry-run` | **超重要。** 実際には変更せず、実行結果のシミュレーションのみ表示する。 |
| ⭐️ `-v`, `--verbose` | どのファイルがどう変更されたかを詳しく表示する。 |
| ⭐️ `-f`, `--force` | 同名のファイルが存在しても、上書きして強制実行する。 |

## 実戦コマンド例 (Perl版)
```bash
# 全ての .htm ファイルを .html に変換
rename 's/\.htm$/\.html/' *.htm

# ファイル名の中の「temp」を「final」に一括置換
rename 's/temp/final/' *

# 全てのファイル名を小文字に変換
rename 'y/A-Z/a-z/' *

# 先頭に今日の日付を付与する
rename 's/^/20260223_/' *.log
````

## 業務シナリオ別活用法

### 1. インフラ構築・運用視点

- **タスク**: ログファイルのローテーション後の整理。
    
- **解説**: アプリケーションが出力したバラバラな命名規則のログファイルを、監視システムが読み取りやすい形式（例：`app.log.1` → `2026-02-23_app.log`）に一括整形します。
    

### 2. ブルーチーム視点

- **タスク**: 隔離した不審ファイルの安全化。
    
- **解説**: 検知した不審な実行ファイル（`.exe` や `.sh`）を解析に回す際、誤実行を防ぐために一括で拡張子を `.suspect` や `.txt` に変更し、安全な状態で保存します。
    
- **例**: `rename 's/$/\.suspect/' *`
    

### 3. レッドチーム視点

- **タスク**: 痕跡の偽装・難読化。
    
- **解説**: 侵入後にアップロードした大量のツールやスクリプトを、一見すると無害なシステムファイルのように見せかける（例：`hack_tool` → `sys_module_X`）ために使用します。手動で一つずつ変えるよりも不自然なタイムスタンプのズレを抑えられる場合があります。

## 💡 プロの知恵：まずは -n から

`rename` は正規表現を書き間違えると、取り返しのつかない名前（あるいは全ファイルが同じ名前になって消滅するなど）になるリスクがあります。

**「いきなり実行しない。まずは `rename -n` で結果を愛でる」** これが鉄則です。

## セキュリティに関する考慮事項

### 脆弱性とリスク

- **意図しない上書き**: 正規表現が広範囲にマッチしすぎると、異なる複数のファイルが同じ名前に収束してしまい、データが上書きされて失われるリスクがあります。
    

### 対応策・緩和策

- **バックアップ**: 大規模なリネームの前には、`tar` などで現状を固めておくのが最も安全です。
    
- **環境の確認**: `rename --version` を叩き、その環境の `rename` が Perl版か C版か（引数の取り方が違うため）を必ず確認してください。
