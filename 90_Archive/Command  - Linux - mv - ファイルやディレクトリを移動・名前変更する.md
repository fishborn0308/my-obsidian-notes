# Command  - Linux - mv - ファイルやディレクトリを移動・名前変更する

## 概要

`mv` (move) は、ファイルやディレクトリを**移動**したり、**名前を変更**したりするためのコマンドです。`mv` の動作は、引数の数と種類によって決まります。

1. **名前の変更**: `mv <SOURCE> <DEST>`
		引数が2つで、`<DEST>` が既存のディレクトリで**ない**場合、`<SOURCE>` ファイル（またはディレクトリ）の名前が `<DEST>` に変更されます。

2. **移動**: `mv <SOURCE>... <DIRECTORY>`
		最後の引数が既存のディレクトリである場合、指定された全ての `<SOURCE>` ファイル（またはディレクトリ）が、その `<DIRECTORY>` の中に移動されます。

(出自: `coreutils` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`cp`** | ファイルを**コピー**します。元のファイルはそのまま残ります。`mv` はファイルを**移動**し、元のファイルはなくなります。 | - |
| **`rsync`** | ファイルやディレクトリの同期・転送を行う高機能なツール。ネットワーク越しの移動も可能です。単一ファイルシステム内での高速な移動や名前変更には `mv` が手軽です。 | `rsync` (ネットワーク越し) |
| **`rename`** | 複数のファイル名を、正規表現などを使って一括で変更することに特化したコマンド。`mv` よりも複雑な一括リネーム作業に適しています。 | `rename` (一括リネーム) |

## よく連携されるコマンド

`mv` はコマンドの出力を直接受け取ることは少ないですが、`find` などのコマンドと `xargs` を介して連携し、検索したファイルを一括で移動させることができます。

### シナリオ例: 特定のログファイルをアーカイブディレクトリに移動する (インフラ構築・運用視点)

* **目的**: 1週間以上前に更新された `.log` ファイルを全て検索し、アーカイブ用ディレクトリに移動する。
* **連携コマンド**: `find`, `xargs`, `mv`
* **解説**: `find` で条件に合うファイルを検索し、その結果を `xargs` に渡します。`xargs` は受け取ったファイルリストを `mv` コマンドの引数として効率的に処理します。
* **コマンド例**:

```bash
# /var/log 配下で7日以上前の.logファイルを /var/log/archive/ に移動
find /var/log -name "*.log" -mtime +7 -print0 | xargs -0 -I {} mv {} /var/log/archive/
```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **動作制御** | |
| ⭐ `-i`, `--interactive` | 移動先に同名のファイルが存在する場合、上書きしてよいか対話形式で確認を求めます。 |
| ⭐ `-f`, `--force` | `-i` の逆。確認プロンプトを表示せず、強制的に上書きします。 |
| ⭐ `-n`, `--no-clobber`| 移動先に同名のファイルが既に存在する場合、上書きせずに `mv` 操作をスキップします。 |
| `-u`, `--update` | 移動元のファイルが移動先のファイルより新しい場合、または移動先にファイルが存在しない場合にのみ移動を実行します。 |
| **出力制御** | |
| `-v`, `--verbose` | ファイルが移動または名前変更される際に、その詳細を表示します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 設定ファイルを編集する前に、元のファイルをバックアップとして退避させる。
* **組み合わせ**: `mv <file> <file>.bak`
* **解説**: ファイル名を変更することで、簡易的なバックアップを作成する最も基本的な使い方です。編集後に問題が発生した場合、`mv <file>.bak <file>` で即座に元に戻せます。
* **例**:

```bash
# nginx.conf を編集する前にバックアップ
sudo mv /etc/nginx/nginx.conf /etc/nginx/nginx.conf.old
```

### 2. ブルーチーム視点

* **タスク**: マルウェアを隔離し、無力化する（封じ込め）。
* **組み合わせ**: `sudo mv /path/to/malware /path/to/quarantine/`
* **解説**: `rm` で削除すると証拠が失われるため、代わりに `mv` を使って、マルウェアを隔離用の安全なディレクトリに移動させます。これにより、マルウェアを無力化しつつ、後の詳細なフォレンジック分析のためにファイルを保全できます。
* **例**:

```bash
# 不審なバイナリを隔離ディレクトリに移動
sudo mv /tmp/evil.elf /var/log/quarantine/
```

### 3. レッドチーム視点

* **タスク**: 悪意のあるスクリプトを、正規のファイル名に偽装する。
* **組み合わせ**: `mv backdoor.sh .profile`
* **解説**: **防御回避 (Defense Evasion)** と **永続化 (Persistence)**。自身が作成した悪意のあるスクリプトを、ユーザーのログイン時に自動的に実行される正規のファイル（`.profile`, `.bashrc`など）にリネームしたり、その内容を上書きしたりします。
* **例**:

```bash
# 攻撃用のスクリプトを、ログイン時に実行されるファイルにリネーム
mv /tmp/revshell.sh /home/user/.profile
```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `mv: cannot move '<file>' to '<dir>/<file>': Permission denied`
		* **考えられる原因**: 移動先のディレクトリへの書き込み権限がありません。
		* **解決策**: 移動先のディレクトリのパーミッションを確認し、必要であれば `sudo` を付けてコマンドを実行してください。

## 環境変数と設定ファイル

* `mv` の動作に直接影響する環境変数や設定ファイルは通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `mv` 自体ではなく、**ファイルの破壊と置換**がリスクとなります。同名のファイルが存在する場所にファイルを移動させると、デフォルトで警告なしに上書きします。
* **悪用シナリオ**: 攻撃者はこの挙動を悪用し、正規のプログラムや設定ファイルを、悪意のあるものにすり替えることができます。

### GTFOBins / LOLBAS における利用例

* **機能**: **Privilege Escalation** (権限昇格)
* **解説**: もしユーザーが `sudo mv` を実行できる場合、`/etc/shadow` や `/etc/sudoers` といった重要なファイルを、自身が作成したファイルで上書きできます。これにより、rootのパスワードを変更したり、自身に `sudo` 権限を付与したりして、root権限を奪取できます。
* **参照**: `https://gtfobins.github.io/gtfobins/mv/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 最小権限の原則に従い、`sudoers` の設定を厳格にし、一般ユーザーに `mv` の `sudo` 権限を安易に与えない。
* **Detection (検知)**: ファイル整合性監視(FIM)ツールで、重要なシステムファイルの変更や移動を監視する。

## 注意点・補足

* **ファイルシステム内/間の移動**: **同じファイルシステム内**でファイルを移動する場合、`mv` は実際のデータを動かさず、ポインタを書き換えるだけなので一瞬で完了します。**異なるファイルシステム間**で移動する場合は、`cp` と `rm` を実行するのと同じ動作になり、ファイルサイズに応じた時間がかかります。
