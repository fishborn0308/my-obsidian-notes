# Command - git - git fetch - リモートリポジトリの最新情報を取得する

## 概要

`git fetch` コマンドは、リモートリポジトリから**最新のコミット、ブランチ、タグなどのオブジェクトをダウンロード**します。しかし、ダウンロードした変更を**自動的にローカルのブランチにマージすることはありません**。

ダウンロードされた情報は、ローカルリポジトリ内の**リモート追跡ブランチ** (例: `origin/main`) に保存されます。これにより、現在の作業に影響を与えることなく、リモートの最新状態を確認できます。

(出自: `git` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`git pull`** | `git pull` は `git fetch` と `git merge` を組み合わせたものです。`fetch` は取得するだけですが、`pull` は取得して**自動的にマージ**します。 | `git fetch` (安全な確認) |
| **`git clone`** | リモートリポジトリを**はじめてローカルにコピーする**際に使用します。一度クローンした後は `fetch` や `pull` を使います。 | - |

## よく連携されるコマンド

`git fetch`は、リモートの状況を安全に確認し、その上で次のアクションを決定するために、多くのGitコマンドと連携して使用されます。

### シナリオ例: リモートの最新情報を確認してからマージする (インフラ構築・運用視点)

* **目的**: ローカルの作業に影響を与えずに、チームメンバーがリモートにプッシュした最新の変更内容を確認し、問題がなければ手動でマージする。
* **連携コマンド**: `git diff`, `git log`, `git merge`
* **解説**: まず `git fetch` でリモートの情報を取得します。次に `git diff` や `git log` でローカルブランチとリモート追跡ブランチの差分を確認し、安全だと判断したら `git merge` で統合します。
* **コマンド例**:

		```bash
    # 1. リモートの最新情報を取得(ローカル作業には影響なし)

    git fetch origin

    # 2. ローカルのmainブランチとリモートのorigin/mainの差分を確認

    git diff main origin/main

    # 3. 安全を確認後、リモートの変更をマージ

    git merge origin/main

    ```



## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **対象指定** | |
| ⭐ `<remote>` | フェッチするリモートリポジトリの名前(例: `origin`)。 |
| `<branch>` | 特定のブランチのみをフェッチします。 |
| `--all` | 設定されているすべてのリモートからフェッチします。 |
| **動作制御** | |
| ⭐ `-p`, `--prune` | リモートから削除されたブランチに対応するリモート追跡ブランチをローカルからも削除します。 |
| `--tags` | すべてのタグをフェッチします。 |
| `-n`, `--dry-run` | フェッチを実行せずに、何がフェッチされるかをシミュレーションして表示します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 古くなったリモート追跡ブランチを定期的にクリーンアップし、ローカルリポジトリを整理する。
* **組み合わせ**: `git fetch --prune origin`
* **解説**: `origin` リモートから最新情報を取得するとともに、リモート側で削除されたブランチに対応するローカルのリモート追跡ブランチも自動的に削除します。
* **例**:

		```bash

    git fetch --prune origin

    # From [https://github.com/your-username/your-repo-name](https://github.com/your-username/your-repo-name)
    #  x [deleted]         (none)     -> origin/old-feature-branch
    ```



## 2. ブルーチーム視点

* **タスク**: 複数のGitリポジトリを追跡し、新しいセキュリティパッチが公開されていないかを確認する。
* **組み合わせ**: `git fetch --all`
* **解説**: 設定されているすべてのリモート(例: `origin`と`upstream`など)から、最新情報を一括で取得します。
* **例**:

		```bash
    # 上流リポジトリを追加済みの場合

    git remote add upstream [https://github.com/upstream-project/repo.git](https://github.com/upstream-project/repo.git)

    # 全リモートから変更を取得

    git fetch --all

    ```



## 3. レッドチーム視点

* **タスク**: ターゲットシステムから外部のGitリポジトリにアクセスする際に、実際に変更を取得する前に、何が転送されるかを確認する。
* **組み合わせ**: `git fetch --dry-run origin main`
* **解説**: `--dry-run` を使って、実際にフェッチせずに、何がフェッチされるか（新しいコミットなど）をシミュレーション表示します。これにより、不必要な痕跡を残すのを避けることができます。
* **例**:

		```bash
    # 実際にはフェッチせずに、何が変更されるかを確認

    git fetch --dry-run origin main

    ```



## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `fatal: Authentication failed for 'https://github.com/...'`
		* **考えられる原因**: リモートリポジトリへの認証情報(ユーザー名、トークン、SSHキーなど)が間違っているか、権限が不足しています。
		* **解決策**: 認証情報を再確認します。SSHを使用している場合は、SSHキーが正しく設定され、ssh-agentに追加されているかを確認します。

## 環境変数と設定ファイル

* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: 古いバージョンのGitでは、悪意のあるリポジトリをフェッチした際に、特定のファイル名処理の欠陥などを突かれる脆弱性が存在しました。
* **悪用シナリオ**: 信頼できないソースからフェッチした場合、悪意のあるGitオブジェクトがローカルの `.git` ディレクトリにダウンロードされる可能性があります。

### GTFOBins / LOLBAS における利用例

`git fetch` は直接的なコード実行機能を持たないため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **常に最新のGitバージョンを使用する**。信頼できないソースからのリポジトリはフェッチしない。
* **Detection (検知)**: Gitホスティングサービスの監査ログを監視し、予期せぬリポジトリからのフェッチがないかを確認する。

## 注意点・補足

* `git fetch` は、ローカルのワーキングディレクトリや、現在チェックアウトしているブランチに**何も変更を加えない**ため、非常に安全に実行できます。
