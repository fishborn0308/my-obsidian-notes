# Command - Linux - hostname - システムのホスト名を表示・設定する

## 概要

`hostname` は、システムのホスト名を取得または設定するために使用されます。ホスト名は、ネットワーク内でコンピュータを識別するためのラベルであり、シェルプロンプトやログ記録、ネットワークプロトコルにおいて重要な役割を果たします。

(出自: `hostname` パッケージまたは `net-tools` 等に含まれる)

## 類似・関連コマンド

| コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `hostnamectl` | systemd 採用 OS における標準。静的・動的・外観用のホスト名を一括管理できる。 | **モダンな設定変更** |
| `dnsdomainname` | システムの DNS ドメイン名を表示する。 | ドメイン確認 |
| `uname -n` | システム情報表示コマンドの一部としてホスト名を表示する。 | スクリプト内での確認 |
| `domainname` | NIS/YP ドメイン名を表示・設定する。 | NIS環境 |

## よく連携されるコマンド

### シナリオ例: 自身のIPアドレスをサッと確認 (運用視点)

* **目的**: 自分のサーバーが現在どの IP を持っているか、設定ファイルを見ずに確認する。
* **解説**: `-I` (大文字) を使うと、ネットワークインターフェースに割り当てられた全ての IP を表示します。
* **コマンド例**:
    ```bash
    # 自身の IP アドレス一覧を表示
    hostname -I
    ```

## オプション説明

※ホスト名の「設定」には **root 権限** が必要です。

| オプション | 説明 |
| :--- | :--- |
| ⭐️ `(引数なし)` | 現在のホスト名を表示する。 |
| ⭐️ `-f`, `--fqdn` | 完全修飾ドメイン名 (Fully Qualified Domain Name) を表示する。 |
| ⭐️ `-i`, `--ip-address` | ホスト名に対応する IP アドレス（通常は /etc/hosts の定義）を表示する。 |
| ⭐️ `-I`, `--all-ip-addresses` | 全てのネットワークインターフェースの IP アドレスを表示する。 |
| ⭐️ `-s`, `--short` | ドメイン部分を除いた短いホスト名を表示する。 |
| `new_name` | (引数として渡す) ホスト名を一時的に `new_name` に変更する。 |

## よく使われる業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: サーバーの命名規則（Naming Convention）の適用。
* **解説**: 役割、場所、番号を組み合わせた名前にすることで、管理ミスを防ぎます。
* **例**:
    ```bash
    # 一時的にホスト名を変更 (再起動で戻る)
    sudo hostname web-prod-tokyo-01
    ```

### 2. ブルーチーム視点

* **タスク**: 内部情報の露出制限。
* **解説**: ホスト名に `db-admin-password-inside` のような露骨な名前を付けていると、ネットワークスキャンを受けた際に攻撃者のターゲットになりやすくなります。組織外に露出するホスト名は抽象的なものにするのが鉄則です。

### 3. レッドチーム視点

* **タスク**: 偵察（Reconnaissance）と信頼関係の悪用。
* **解説**: 侵入後、まず `hostname` を叩いてそのサーバーの役割を推測します。また、古いネットワーク環境では「ホスト名に基づいた信頼（rsh や古い NFS 等）」が残っている場合があり、ホスト名を偽装することで認証をバイパスできるケースを探ります。

## ホスト名の永続化（再起動後も有効にする方法）

`hostname` コマンドで変更した名前は、メモリ上の一時的なものです。永続化するには以下の方法をとります。

* **方法 1 (推奨)**: `sudo hostnamectl set-hostname <new_name>`
* **方法 2**: `/etc/hostname` ファイルを直接編集し、その後 `/etc/hosts` も整合性を合わせる。

## エラーメッセージとトラブルシューティング

1.  **「hostname: you must be root to change the host name」**:
    * **原因**: 権限不足。
    * **解決策**: `sudo` を付けて実行する。

2.  **「hostname: Name or service not known」**:
    * **原因**: `-f` 等で FQDN を出そうとした際、`/etc/hosts` や DNS で自分自身の名前を解決できなかった。
    * **解決策**: `/etc/hosts` に `127.0.0.1  hostname` の記述があるか確認する。

## セキュリティに関する考慮事項

### 脆弱性とリスク

* **情報の推測**: `jenkins-master` や `vault-server` というホスト名は、攻撃者に「どこに重要なデータがあるか」を教える地図になります。
* **DHCP 経由の汚染**: クライアント PC が自身のホスト名を DHCP サーバーに送り、それがそのまま内部 DNS に登録される設定になっている場合、悪意のあるホスト名（スクリプトを含む名前など）を送り込まれるリスクがあります。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention**: 外部公開するサーバーと内部管理用のホスト名を分け、外部には役割を特定させない名前を付ける。
* **Detection**: ホスト名の変更イベントをログで監視する。

## 注意点・補足

* **プロンプトへの反映**: ホスト名を変えても、現在開いているシェルのプロンプト（`user@oldname:~$`）は、一度ログアウトして入り直すまで更新されないことが多いです。
