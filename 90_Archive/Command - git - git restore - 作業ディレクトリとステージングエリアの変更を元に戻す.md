# Command - git - git restore - 作業ディレクトリとステージングエリアの変更を元に戻す

## 概要

`git restore` コマンドは、Gitの**作業ディレクトリ**(ワーキングツリー)や**ステージングエリア**(インデックス)にあるファイルの変更を、指定したコミットの状態に戻すために使われます。これにより、誤った変更を破棄したり、ステージングしたファイルをアンステージしたりできます。

このコマンドはGitバージョン2.23で導入され、以前は`git checkout`や`git reset`が担っていたファイル復元の一部機能を、より明確で安全な形で提供するようになりました。

(出自: `git` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`git checkout`** | 以前はファイルの変更を破棄するのに使われましたが、ブランチの切り替えにも使われるため機能が多岐にわたっていました。`restore` は**ファイル復元に特化**しており、意図がより明確です。 | `git restore` |
| ⭐ **`git reset`** | `git reset HEAD <file>` はステージングされたファイルを取り消せますが、コミット履歴を操作する強力なコマンドでもあります。`git restore --staged` はステージングの取り消しに特化しており、より安全です。 | `git restore --staged` |
| **`git clean`**| `restore` は追跡されているファイルの変更を元に戻しますが、`clean` は**追跡されていない**ファイルやディレクトリを削除します。 | - |

## よく連携されるコマンド

`git restore`は、変更の状態を確認するために `git status` や `git diff` と連携して使用されます。

### シナリオ例: 作業ディレクトリでの変更を破棄する (インフラ構築・運用視点)

* **目的**: 設定ファイルを編集している最中に誤った変更を加えてしまい、元の状態に戻したい。
* **連携コマンド**: `git status`, `git diff`
* **解説**: まず`git status`でファイルの変更状況を確認し、`git diff`で具体的な変更内容を把握します。その上で、変更を破棄するために`git restore`を使います。
* **コマンド例**:

		```bash
    # 1. 現在の変更状況を確認

    git status

    # 2. 変更内容を確認

    git diff network_config.conf

    # 3. network_config.conf の変更を破棄し、最後にコミットされた状態に戻す

    git restore network_config.conf

    ```





## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **対象指定** | |
| ⭐ `<file>` | 変更を元に戻す対象のファイル名を指定します。 |
| **操作モード** | |
| ⭐ `--staged` | 作業ディレクトリの変更ではなく、**ステージングエリア**からファイルを取り消します(アンステージします)。作業ディレクトリの変更は保持されます。 |
| `--worktree` | (デフォルト) 作業ディレクトリのファイルを復元することを明示的に指定します。 |
| ⭐ `-s`, `--source=<commit>` | ファイルを元に戻す参照元となるコミットを指定します。デフォルトは`HEAD`です。 |
| **その他** | |
| `-p`, `--patch` | 対話形式で変更を部分的に復元します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: IaC (Infrastructure as Code) の設定ファイルを誤って編集してしまい、稼働中の設定を壊す恐れがあるため、すぐに安全な状態に戻したい。
* **組み合わせ**: `git restore <config-file>`
* **解説**: 作業ディレクトリでの変更を完全に破棄し、最後にコミットされた状態に戻す、最も基本的な使い方です。
* **例**:

		```bash
    # nginx.conf の変更を破棄し、最後にコミットされた状態に戻す

    git restore nginx.conf

    ```





## 2. ブルーチーム視点

* **タスク**: 監査ログファイルや設定ファイルが改ざんされた可能性がある場合、特定の過去のコミット時点でのファイルの健全な状態を確認・比較する。
* **組み合わせ**: `git restore --source=<commit-id> <file>`
* **解説**: 現在のファイルを直接上書きするのではなく、`git show <commit-id>:<file> > <file>.old` のようにして過去のバージョンを別名で抽出し、`diff` で比較するのがより安全な調査手法です。
* **例**: -

## 3. レッドチーム視点

* **タスク**: 偵察段階で発見した認証情報などを誤ってステージングしてしまった場合に、コミットせずにステージング状態を解除する。
* **組み合わせ**: `git restore --staged <secret-file>`
* **解説**: これにより、不要な情報がリポジトリ履歴に残るのを防ぎます。
* **例**:

		```bash
    # 誤ってステージングしてしまった機密情報ファイル `creds.txt` をアンステージ

    git restore --staged creds.txt

    ```





## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `error: pathspec '<file>' did not match any file(s) known to git.`
		* **考えられる原因**: 指定したファイル名が間違っているか、そのファイルがGitによって追跡されていません。
		* **解決策**: ファイル名が正しいか確認します。追跡されていないファイルを削除したい場合は`rm`コマンドを使用します。

## 環境変数と設定ファイル

* `git restore`に直接影響を与える独自の環境変数はありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **情報漏洩の可能性**。
* **悪用シナリオ**: `git restore --source=<commit>` を使用して過去のコミットからファイルを復元する際、そのファイルに機密情報が含まれていた場合、意図せず現在の作業環境に機密情報が復元されてしまう可能性があります。

### GTFOBins / LOLBAS における利用例

`git restore` は主にファイルの復元に使われるため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: Gitリポジトリに機密情報をコミットしないためのポリシーを徹底する。
* **Detection (検知)**: `git log` や `git reflog` を使って、不審なファイル復元操作がないか履歴を監査する。

## 注意点・補足

* **未コミットの変更の喪失**: `git restore <file>`は、**コミットされていない作業ディレクトリの変更を完全に破棄**します。実行前には`git status`や`git diff`で変更内容を十分に確認してください。
* **ステージングエリアのみへの影響**: `git restore --staged <file>`は、ステージングエリアからファイルを取り消すだけで、**作業ディレクトリの変更はそのまま残ります**。
