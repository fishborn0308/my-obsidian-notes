# Command  - Linux - pwd - 現在のワーキングディレクトリを表示する

## 概要

`pwd` (print working directory) は、現在作業しているディレクトリの**絶対パス**を表示する、非常にシンプルで基本的なシェル組み込みコマンドです。ファイルシステムのどこにいるのか、自分の現在地を正確に把握するために使われます。複雑なディレクトリ構造を移動している際に、道に迷わないための「現在地」表示機能です。

(出自: `coreutils` パッケージに含まれるが、通常はシェルの組み込み機能が使われる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`$PWD` (変数)**| シェルが内部的に保持している現在のディレクトリパスを格納した変数です。`pwd -L` はこの変数の値を表示します。 | - |
| **`readlink -f .`**| カレントディレクトリ (`.`) の**物理的な**（正規の）パスを表示します。全てのシンボリックリンクを解決した後の最終的なパスが得られます。`pwd -P` と同じ結果になります。 | `pwd -P` |

## よく連携されるコマンド

`pwd` は、スクリプト内で現在のパスを変数に格納し、後続の処理の基準点として利用するためによく使われます。

### シナリオ例: スクリプトの実行パスを基準にする (インフラ構築・運用視点)

* **目的**: シェルスクリプトがどこから実行されても、スクリプト自身の場所を基準に他のファイルを読み込むようにする。
* **連携コマンド**: `dirname`, コマンド置換 (`$()`)
* **解説**: `$0` (スクリプト自身のパス) を `dirname` に渡してスクリプトのディレクトリを取得し、`cd` してから `pwd` で絶対パスを取得します。これにより、スクリプトの実行場所に関わらず、常にスクリプトと同じディレクトリを基準に動作させることができます。
* **コマンド例**:

```bash
# スクリプトの冒頭で、スクリプト自身のディレクトリに移動する定型句
cd "$(dirname "$0")"
SCRIPT_DIR=$(pwd)
echo "This script is located in: ${SCRIPT_DIR}"
```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **表示形式** | |
| ⭐ `-L`, `--logical` | (**デフォルト**) シンボリックリンクを解決せず、環境変数 `$PWD` に基づいた**論理的な**パスを表示します。 |
| ⭐ `-P`, `--physical`| 全てのシンボリックリンクを解決し、**物理的な**（実際の）パスを表示します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 危険なコマンドを実行する前に、現在地を最終確認する。
* **組み合わせ**: `pwd`
* **解説**: `rm -rf *` のような破壊的なコマンドを実行する前に `pwd` を実行し、誤ったディレクトリで操作しようとしていないかを確認するのは、基本的な安全策です。
* **例**:

```bash
# カレントディレクトリを確認してから、その中身を全て削除
pwd
# -> /tmp/safe-to-delete
rm -rf ./*
```

### 2. ブルーチーム視点

* **タスク**: 調査記録の正確性を担保する。
* **組み合わせ**: `pwd`
* **解説**: インシデント調査中に、不審なファイルやログを発見した場合、`pwd` を実行してそのディレクトリの完全なパスを取得し、報告書や証拠管理シートに正確に記録します。これにより、後から誰が見ても、どの場所で何が発見されたかを正確に再現できます。
* **例**:

```bash
# 証拠ファイルのあるディレクトリのフルパスを記録
pwd
```

### 3. レッドチーム視点

* **タスク**: 侵入後の状況確認 (Situational Awareness) のために、初期立ち位置を確認する。
* **組み合わせ**: `pwd`
* **解説**: 攻撃者はシステムに侵入後、`whoami`, `hostname` と共に `pwd` を実行し、自分がどのディレクトリに降り立ったかを確認します。Webアプリケーションの脆弱性を突いた場合は `/var/www/html`、ユーザーアカウントを奪取した場合は `/home/<username>` など、初期位置からシステムの構成や次の攻撃経路を推測します。
* **例**:

```bash
# 侵入直後に現在地を確認
pwd
```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **現象**: **`pwd: error retrieving current directory: ...`**
		* **考えられる原因**: 別のターミナルから、現在いるディレクトリが削除または移動されました。シェルはまだそのディレクトリに「いる」つもりですが、実体が存在しないためパスを取得できなくなっています。
		* **解決策**: `cd` や `cd ~` を実行して、存在するディレクトリに移動すれば解決します。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `pwd` は読み取り専用のツールであり、脆弱性を持つ可能性は低いです。
* **悪用シナリオ**: **情報漏洩**。`pwd` の出力は、ファイルシステムの構造や、プロジェクト名、ユーザー名 (`/home/<username>`) などを明らかにします。これは、攻撃者がシステムの構成を理解するための偵察情報となり得ます。

### GTFOBins / LOLBAS における利用例

`pwd` は基本的なシェル機能のため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: -
* **Detection (検知)**: `pwd` は基本的なコマンドのため検知は困難ですが、`auditd` で不審なユーザーによる一連の偵察コマンド（`whoami`, `hostname`, `pwd`, `ls`など）の実行を監視します。

## 注意点・補足

* **`pwd` vs `/bin/pwd`**: シェルのプロンプトで実行されるのは、通常、より高速なシェル組み込み版の `pwd` です。`/bin/pwd` という外部コマンドも存在します。
