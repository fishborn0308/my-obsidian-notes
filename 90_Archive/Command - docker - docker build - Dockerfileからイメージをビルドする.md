# Command - docker - docker build - Dockerfileからイメージをビルドする

## 概要

`docker build` は、`Dockerfile` という設計図と、「コンテキスト」と呼ばれるファイル群を基に、新しいDockerイメージを作成するコマンドです。コンテキストとは、ビルドプロセス中にイメージ内部にコピーされるファイルやディレクトリのセットで、通常は `Dockerfile` が置かれているディレクトリ (`.` で指定) を指します。このコマンドは、アプリケーションの実行環境をコード化し、再現性のある形でパッケージングするための中心的な役割を担います。

(出自: `docker` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`docker commit`** | 実行中のコンテナの**現在の状態**から新しいイメージを作成します。手順が不明確で再現性がないため、本番用のイメージ作成には**非推奨**です。 | `docker build` |
| **`docker compose build`**| `docker-compose.yml` に定義された複数のサービスのイメージをまとめてビルドします。内部的には各サービスに対して `docker build` を実行します。 | `docker compose` |

## よく連携されるコマンド

`docker build` は、ビルドしたイメージに名前を付け (`-t`)、そのイメージをレジストリにプッシュ (`docker push`) するまでの一連のワークフローで使われます。

### シナリオ例: イメージをビルドしてレジストリにプッシュする (インフラ構築・運用視点)

* **目的**: アプリケーションのソースコードからDockerイメージをビルドし、タグを付けて、共有レジストリにアップロードする。
* **連携コマンド**: `docker tag`, `docker push`
* **解説**: `docker build` でローカルイメージを作成した後、`docker tag` でレジストリ用の名前を付け、`docker push` でアップロードします。
* **コマンド例**:

	```bash
    # 1. 'my-app:1.0' という名前でイメージをビルド

    docker build -t my-app:1.0 .

    # 2. レジストリ (例: Docker Hub) の命名規則に合わせてタグ付け

    docker tag my-app:1.0 my-username/my-app:1.0

    # 3. レジストリにプッシュ

    docker push my-username/my-app:1.0

    ```


## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **ビルドコンテキスト** | |
| ⭐ `<PATH>` | ビルドコンテキストのパス (`.` や `~/project` など)。このパス以下のファイルがDockerデーモンに送られます。 |
| **タグ付けと命名** | |
| ⭐ `-t`, `--tag <name:tag>` | 作成されるイメージに `名前:タグ` の形式でタグを付けます。 |
| **ビルドプロセス制御** | |
| ⭐ `--build-arg <key=value>` | `Dockerfile` 内の `ARG` 命令に値を渡します。 |
| `-f`, `--file <Dockerfile>` | デフォルトの `Dockerfile` 以外の名前のファイルを使用します。 |
| `--no-cache` | ビルド時にレイヤーキャッシュを一切使用しません。 |
| `--secret id=<id>,src=<path>`| (BuildKit) ビルド中にのみ利用可能な機密ファイル（秘密鍵など）を安全にマウントします。 |
| `--progress=<type>` | ビルドの進捗表示形式を指定します。`plain` または `tty` (デフォルト)。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: アプリケーションのソースコードから、バージョン `1.0` のデプロイ用イメージを作成する。
* **組み合わせ**: `docker build -t <image_name>:<tag> .`
* **解説**: カレントディレクトリ (`.`) をコンテキストとして `Dockerfile` に基づいてイメージをビルドし、`-t` オプションで名前とタグを付ける、最も基本的な使い方です。
* **例**:

	```bash
    # カレントディレクトリのDockerfileを基に、my-web-app:1.0 というイメージをビルド

    docker build -t my-web-app:1.0 .

    ```


### 2. ブルーチーム視点

* **タスク**: 脆弱性スキャンを行うために、提供された `Dockerfile` からイメージを再現ビルドする。
* **組み合わせ**: `docker build .` → `trivy image <image_name>`
* **解説**: `Dockerfile` からイメージをビルドした後、`Trivy` や `Dive` といったツールでイメージを分析します。CI/CDパイプラインに組み込み、既知の脆弱性 (CVE) を持つOSパッケージやライブラリが含まれていないかをスキャンします。
* **例**:

	```bash
    # イメージをビルド

    docker build -t my-app-for-scan .

    # 脆弱性をスキャン

    trivy image my-app-for-scan

    ```


### 3. レッドチーム視点

* **タスク**: ターゲットの公開リポジトリにある `Dockerfile` を偵察し、攻撃の足がかりを探す。
* **組み合わせ**: (Dockerfileの分析)
* **解説**: 攻撃者は `docker build` を直接使うより、まず `Dockerfile` 自体を分析します。古いベースイメージ (`FROM ubuntu:18.04`)、不セキュアな `RUN` 命令 (`chmod 777 ...`)、ハードコードされた認証情報 (`ENV PASSWORD=...`) などは、ビルドされたイメージの攻略法を探るための重要なヒントとなります。

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `failed to solve: failed to read dockerfile: open /path/to/project/Dockerfile: no such file or directory`
		* **考えられる原因**: `docker build` を実行したディレクトリに `Dockerfile` という名前のファイルが存在しません。
		* **解決策**: 正しいディレクトリに `cd` するか、`-f` オプションで `Dockerfile` のパスを明示的に指定してください。

2. **エラーメッセージ例 1**: `The command '/bin/sh -c ...' returned a non-zero code: 1`
		* **考えられる原因**: `Dockerfile` 内の `RUN` 命令が失敗しました。パッケージが見つからない (`apt-get update`忘れ)、ネットワークの問題、スクリプトのエラーなどが考えられます。
		* **解決策**: エラーメッセージの直前の出力を確認し、`Dockerfile` の該当行を修正します。

## 環境変数と設定ファイル

* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

### 関連する設定ファイル

* ⭐ **`Dockerfile`**: ビルド手順を定義する最も重要なファイル。
* ⭐ **`.dockerignore`**: ビルドコンテキストに**含めない**ファイルやディレクトリを指定します。`.git` や `node_modules`, `.env` ファイルなどを除外することで、ビルドの高速化と、**機密情報がイメージ内に混入するのを防ぎます**。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **サプライチェーン攻撃**。`FROM` で指定するベースイメージが悪意のあるものだったり、`RUN` で信頼できないソースからスクリプトをダウンロード・実行したりすると、イメージ全体が侵害される可能性があります。
* **悪用シナリオ**: 攻撃者が Docker Hub に公式イメージと似た名前（タイポスクワッティング）の悪意のあるイメージをアップロードし、開発者がそれを誤って `FROM` で指定してしまう。

### GTFOBins / LOLBAS における利用例

`docker build` 自体はリストされていませんが、Dockerソケット (`/var/run/docker.sock`) へのアクセス権を持つユーザーは、実質的にホストのroot権限を持っています。`docker build` はその権限を行使する手段の1つになり得ます。

* **機能**: **Privilege Escalation**
* **解説**: Dockerグループに所属するユーザーが、ホストのルートディレクトリをマウントするような `Dockerfile` を作成し、そのイメージを実行することで、ホストの全ファイルにアクセスできます。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 最小の公式ベースイメージ (`alpine`, `distroless`) を利用する。**非rootユーザーで実行する** (`USER` 命令)。**マルチステージビルド**を活用し、最終イメージに不要なツールを含めない。`.dockerignore` を必ず利用する。
* **Detection (検知)**: `Trivy` などの脆弱性スキャンツールをCI/CDパイプラインに組み込み、ビルドされたイメージをスキャンする。

## 注意点・補足

* **ビルドコンテキスト**: `docker build .` の `.` は、カレントディレクトリ以下の全ファイルをDockerデーモンに送信することを意味します。`.dockerignore` の活用が不可欠です。
