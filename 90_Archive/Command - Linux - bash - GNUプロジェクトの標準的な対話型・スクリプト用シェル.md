# Command - Linux - bash - GNUプロジェクトの標準的な対話型・スクリプト用シェル

## 概要

`bash` は、GNU プロジェクトによって開発された Unix シェルであり、現在のほとんどの Linux ディストリビューションでデフォルトのログインシェルとして採用されています。コマンドの実行、スクリプトによる自動化、およびシステム制御のための強力なインターフェースを提供します。

(出自: `bash` パッケージに含まれる。Bourne shell (sh) の上位互換)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `sh` | 古典的な Bourne Shell。機能が制限されており、軽量。スクリプトの移植性を重視する場合に使用。 | 最小構成のスクリプト |
| `dash` | Debian/Ubuntu で `/bin/sh` として使われる高速なシェル。対話的機能が乏しい。 | 起動スクリプト |
| `zsh` | 高度な補完機能とプラグインシステムを持つ。macOS のデフォルト。 | 開発者の対話環境 |
| `rbash` | 機能を制限した Restricted Bash。ディレクトリ移動や特定のコマンド実行を制限する。 | セキュアな作業環境 |

## よく連携されるコマンド

### シナリオ例: デバッグモードでのスクリプト実行 (運用・構築視点)

* **目的**: 作成した自動化スクリプトがどこで止まっているか、変数がどう展開されているかを確認する。
* **連携コマンド**: `cat`, `echo`
* **解説**: `-x` オプションを付けて実行することで、実行されるすべてのコマンドをトレース表示します。
* **コマンド例**:
    ```bash
    # スクリプトの内容を展開しながら実行
    bash -x ./setup_server.sh
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **実行制御** | |
| ⭐️ `-c <string>` | 引数として渡された文字列をコマンドとして実行する。 |
| ⭐️ `-i` | 対話モード（Interactive）で実行する。リバースシェルで重要。 |
| ⭐️ `-p` | 特権モード。実効UIDと実際のUIDが異なる場合、実効UIDを維持して起動する（SUID悪用時に必須）。 |
| `-x` | 実行されるコマンドとその引数を標準エラー出力に表示（デバッグ用）。 |
| `-r`, `--restricted` | 制限付きシェル (`rbash`) として起動する。 |
| **環境設定回避** | |
| `--noprofile` | システム全体およびユーザー個別のプロファイルファイルを読み込まない。 |
| `--norc` | 対話モード時、`~/.bashrc` を読み込まない（攻撃の痕跡を最小化）。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 他のコマンドの出力結果をその場でループ処理する。
* **組み合わせ**: `bash -c`
* **解説**: パイプで渡されたリストに対して、複雑な処理をワンライナーで実行します。
* **例**:
    ```bash
    # 特定の条件に合うファイル全てに対して、所有者を変更しつつログを残す
    find /tmp -name "*.log" | xargs -I{} bash -c 'chown root {}; echo "Changed {}" >> /var/log/audit.txt'
    ```

### 2. ブルーチーム視点

* **タスク**: ログイン履歴と実行コマンドの整合性確認。
* **組み合わせ**: `~/.bash_history` の解析
* **解説**: 攻撃者が `HISTCONTROL` を操作して履歴を残さないようにしていないか、不自然なコマンド（`base64`, `nc`, `curl | bash` 等）がないか調査します。
* **例**:
    ```bash
    # タイムスタンプを含めた履歴の確認
    export HISTTIMEFORMAT="%F %T "
    history | tail -n 50
    ```

### 3. レッドチーム視点

* **タスク**: 権限昇格後の安定したシェルの取得（TTYの昇格）。
* **組み合わせ**: `bash -i`
* **解説**: シンプルなリバースシェルでは `vi` や `top` が使えないため、Python 等を併用して完全な対話型シェルにアップグレードします。
* **例**:
    ```bash
    # Netcat 等で繋がった後の TTY 昇格
    python3 -c 'import pty; pty.spawn("/bin/bash")'
    # または SUID ビットが立った bash を使った特権維持
    ./bash -p
    ```

## エラーメッセージとトラブルシューティング

1.  **エラーメッセージ例 1**: `bash: ./script.sh: /bin/bash^M: bad interpreter: No such file or directory`
    * **考えられる原因**: Windows 環境で作成されたスクリプトの改行コード（CRLF）が混入している。
    * **解決策**: `sed -i 's/\r$//' script.sh` または `dos2unix` コマンドで改行コードを LF に変換する。

2.  **エラーメッセージ例 2**: `bash: fork: retry: No child processes`
    * **考えられる原因**: プロセス数の上限（ulimit）に達した。フォーク爆弾（`:(){ :|:& };:`）等の影響。
    * **解決策**: プロセス数を制限するか、異常なプロセスを kill する。

## 環境変数と設定ファイル

### 影響を与える環境変数

* **`PATH`**: 実行ファイルを検索するディレクトリリスト。
* **`PS1`**: プロンプトの表示形式。
* **`HISTFILE`**: コマンド履歴が保存されるファイルパス（デフォルトは `~/.bash_history`）。

### 関連する設定ファイル

* **`/etc/profile`**: 全ユーザー共通のログイン時の設定。
* **`~/.bash_profile` / `~/.bashrc`**: ユーザー個別の設定。攻撃者がエイリアスを仕込む（Persistence）ターゲットになりやすい。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **Shellshock (CVE-2014-6271)**。環境変数を介して任意のコードが実行される。
* **悪用シナリオ**:
    * SUID 権限がついた `bash` バイナリを放置していると、一般ユーザーから `-p` オプションで root を奪取される。
    * 制限付きシェル (`rbash`) を、`vi` の `:set shell=/bin/bash` や `python` 経由でバイパスする（Escape Restricted Shell）。

### GTFOBins における利用例

`bash` に SUID が設定されている、あるいは `sudo` 許可がある場合、即座に特権シェルが得られます。

* **機能**: `Shell`, `Sudo`, `SUID`
* **参照**: `https://gtfobins.github.io/gtfobins/bash/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 一般ユーザーに不要な `sudo bash` 権限を与えない。`/bin/sh` が `bash` を指している場合、脆弱な古いバージョンでないか確認。
* **Detection (検知)**: プロセス監視において `bash -i` や `bash -c '.../tcp/...'` といったリバースシェルの特徴的なパターンをアラート対象にする。

## 注意点・補足

* **スクリプトの先頭行（Shebang）**: `#!/bin/bash` を明記することで、実行時に使用されるシェルを固定し、移植性を高めます。
* **コマンドの終了ステータス**: `$?` で直前のコマンドが成功（0）したか失敗（それ以外）かを確認する癖をつけることが、セキュアな運用に繋がります。
