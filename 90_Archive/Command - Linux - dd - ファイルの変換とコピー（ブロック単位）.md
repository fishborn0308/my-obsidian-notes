# Command - Linux - dd - ファイルの変換とコピー（ブロック単位）

## 概要

`dd` は、入力（if: Input File）から出力（of: Output File）へデータをブロック単位でコピーします。通常の `cp` と異なり、ディスクのブートセクタ（MBR）や未割当領域を含む「物理的なデータの並び」をそのまま複製できるのが最大の特徴です。

(出自: `coreutils` パッケージに含まれる。Unixの黎明期から存在する)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `cp` | ファイルシステム上の「ファイル」をコピー。メタデータや構造は維持しない。 | 通常のファイル複製 |
| `cat` | ストリームとして出力。ブロックサイズの制御などはできない。 | テキストやパイプ処理 |
| `rsync` | ファイル単位の同期。差分転送が得意。 | バックアップ・同期 |
| `dd` | セクタ・ブロック単位のコピー。ファイルシステムを無視した複製が可能。 | **ディスクイメージ作成・低層操作** |

## よく連携されるコマンド

### シナリオ例: ネットワーク経由のディスクイメージ転送 (運用・保全視点)

* **目的**: 物理ディスクの内容を、別のサーバーへネットワーク越しに直接バックアップする。
* **連携コマンド**: `ssh`, `gzip`
* **解説**: 標準入出力をパイプで繋ぐことで、巨大なディスクイメージをローカルに保存せず転送できます。
* **コマンド例**:
    ```bash
    # /dev/sda を圧縮しつつリモートサーバへ転送
    sudo dd if=/dev/sda | gzip -1 - | ssh user@remote "dd of=sda_backup.img.gz"
    ```

## オプション説明 (書式が独特: `key=value`)

| キー | 説明 |
| :--- | :--- |
| ⭐️ `if=<file>` | **入力**ファイル（またはデバイス）のパス。 |
| ⭐️ `of=<file>` | **出力**ファイル（またはデバイス）のパス。 |
| ⭐️ `bs=<bytes>` | 1回に読み書きする**ブロックサイズ**（例: `4M`, `1G`）。転送速度に直結。 |
| `count=<n>` | コピーするブロックの数。 |
| `skip=<n>` | 入力開始時に指定ブロック数分をスキップする。 |
| `seek=<n>` | 出力開始時に指定ブロック数分をスキップする。 |
| ⭐️ `status=progress` | 実行中に転送速度や残り時間を表示する（モダンな実装）。 |
| `conv=<options>` | `noerror` (エラーを無視), `sync` (足りない分を0埋め) などの変換オプション。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: ISOイメージからブート可能な USB メモリを作成。
* **組み合わせ**: `bs=4M status=progress`
* **解説**: OSのインストールメディアを物理メディアへ書き込みます。
* **例**:
    ```bash
    # USBメモリ(/dev/sdb)にOSイメージを書き込む
    sudo dd if=ubuntu-24.04.iso of=/dev/sdb bs=4M status=progress oflag=sync
    ```

### 2. ブルーチーム視点

* **タスク**: 侵害されたマシンのメモリダンプやディスクイメージの取得（保全）。
* **組み合わせ**: `conv=noerror,sync`
* **解説**: ディスクに物理的な不良セクタがあっても、エラーで止まらずに最後までイメージ化し、欠損部分は 0 で埋めて構造を維持します。
* **例**:
    ```bash
    # 証拠ディスクをエラー無視でイメージ化
    sudo dd if=/dev/sda of=/mnt/external/evidence.img conv=noerror,sync
    ```

### 3. レッドチーム視点

* **タスク**: ディスクワイプ（証跡抹消）または MBR 破壊による DoS。
* **組み合わせ**: `if=/dev/zero` または `if=/dev/urandom`
* **解説**: ディスクの先頭（ブートローダーがある場所）を 0 で上書きして起動不能にしたり、全体を乱数で埋めてファイル復元を不可能にします。
* **例**:
    ```bash
    # MBR (先頭512バイト) を破壊して再起動不能にする
    sudo dd if=/dev/zero of=/dev/sda bs=512 count=1
    ```

## エラーメッセージとトラブルシューティング

1.  **「No space left on device」**:
    * **原因**: 出力先のディスクやパーティションがいっぱいになった。
    * **解決策**: 出力先の空き容量を確認。または `count` を指定して必要な分だけコピーする。

2.  **実行中に進捗が分からず不安**:
    * **対策1**: `status=progress` を付ける。
    * **対策2**: 古い `dd` の場合、別のターミナルから `kill -USR1 $(pgrep ^dd$)` を送ると、その時点の統計が表示されます。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **悪用シナリオ**: 攻撃者が `sudo dd` 権限を持っている場合、`/etc/shadow` を読み取るだけでなく、`/dev/mem`（物理メモリ）をダンプしてパスワードや暗号鍵を抽出したり、システムのカーネルバイナリを直接書き換えて Rootkit を埋め込んだりすることが可能です。

### GTFOBins における利用例

`dd` が `sudo` で実行可能な場合、任意のファイルの読み書きが自由自在です。

* **機能**: `Sudo`, `File Read`, `File Write`
* **コマンド例**:
    ```bash
    # sudo dd を使って /etc/shadow の内容を読み取る
    sudo dd if=/etc/shadow
    ```
* **参照**: `https://gtfobins.github.io/gtfobins/dd/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 一般ユーザーに `/dev/sd*` や `/dev/mem` へのアクセス権を与えない。`sudoers` で `dd` を無制限に許可しない。
* **Detection (検知)**: デバイスファイルに対する異常な `dd` 実行や、`/dev/zero` を使った大量書き込みを監視する。

## 注意点・補足

* **`of=` の指定ミス**: `/dev/sda` (ディスク全体) と `/dev/sda1` (第1パーティション) を間違えるだけでシステムが壊れます。実行前に **3回** は確認しましょう。
* **パフォーマンス**: `bs=512` (デフォルト) は非常に遅いです。現代のディスクでは `bs=1M` や `bs=4M` を指定するのが一般的です。
