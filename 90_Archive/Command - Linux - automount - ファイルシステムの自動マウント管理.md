# Command - Linux - automount - ファイルシステムの自動マウント管理

## 概要

`automount` は、`autofs` サービスで使用されるデーモンプログラムであり、設定されたマップファイルに従って、ディレクトリへのアクセスが発生した瞬間にファイルシステムを自動的にマウントします。一定時間アクセスがない場合は自動的にアンマウントを行い、システムリソースとネットワーク帯域を節約します。

(出自: `autofs` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `mount` | 手動でマウントを行う。OS起動時に固定でマウントする場合は `/etc/fstab` を使用する。 | 一時的・固定マウント |
| `automount (autofs)` | アクセス時にのみマウント。サーバがダウンしていても起動を阻害せず、管理が柔軟。 | 大規模な共有ディレクトリ |
| `systemd-mount` | systemd ユニットとしてマウントを管理。現代的な代替手段の一つ。 | systemd環境での自動化 |

## よく連携されるコマンド

### シナリオ例: NFS共有のオンデマンド利用 (運用視点)

* **目的**: 必要な時だけリモートのNFSサーバを `/mnt/nfs` にマウントする。
* **連携コマンド**: `showmount`, `nfs-utils`
* **解説**: `showmount` でエクスポート状況を確認し、`autofs` のマップファイル（`/etc/auto.misc` 等）に定義します。
* **コマンド例**:
    ```bash
    # リモート共有の確認
    showmount -e 192.168.1.100
    # automount の動作確認（デバッグモード）
    sudo automount -f -v
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **起動・デバッグ** | |
| ⭐️ `-f`, `--foreground` | デーモン化せずフォアグラウンドで実行する。デバッグに有用。 |
| ⭐️ `-v`, `--verbose` | 詳細な動作ログを出力する。マウント/アンマウントの成否を確認できる。 |
| `-d`, `--debug` | デバッグ情報を出力する。詳細なトラブルシューティングに使用。 |
| **管理** | |
| `-p <pidfile>` | デーモンのプロセスIDを記録するファイルを指定。 |
| `-t <seconds>` | タイムアウト時間を指定（デフォルトは通常300秒）。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 大規模環境でのホームディレクトリ管理。
* **組み合わせ**: `automount` + LDAP/AD マップ
* **解説**: 数百人のユーザーがいる場合、全てのホームディレクトリを起動時に `fstab` でマウントすると起動が極端に遅くなります。`automount` を使い、ログインしたユーザーの分だけを動的にマウントします。
* **例**:
    ```bash
    # 設定反映後のサービス再起動
    sudo systemctl restart autofs
    ```

### 2. ブルーチーム視点

* **タスク**: 不審なリモートマウント試行の検知。
* **組み合わせ**: `automount -v` ログの監視
* **解説**: 攻撃者がネットワーク内で `autofs` の設定を書き換え、自身の管理する不正な共有サーバへトラフィックを誘導（窃取）していないかログを確認します。
* **例**:
    ```bash
    # マウントログの常時監視
    tail -f /var/log/syslog | grep automount
    ```

### 3. レッドチーム視点

* **タスク**: ネットワーク共有を介した情報収集と権限昇格。
* **組み合わせ**: `cat /etc/auto.master`
* **解説**: `/etc/auto.*` ファイルを読み取ることで、通常は隠されているファイルサーバの IP アドレスや共有パスを特定できます。また、NFS の `no_root_squash` が設定されている共有が自動マウント対象であれば、root 権限奪取の足掛かりになります。
* **例**:
    ```bash
    # マウント設定からターゲットサーバを特定
    cat /etc/auto.master
    cat /etc/auto.misc
    ```

## エラーメッセージとトラブルシューティング

1.  **エラーメッセージ例 1**: `automount[PID]: mount(nfs): nfs: mount failure server:path on /mnt/dir`
    * **考えられる原因**: サーバ側でのエクスポート設定ミス、ネットワーク断、またはファイアウォールによる遮断。
    * **解決策**: `rpcinfo -p <server_ip>` でサービス稼働を確認し、手動で `mount` を試みて詳細なエラーを確認する。

2.  **エラーメッセージ例 2**: `dir not empty`
    * **考えられる原因**: マウントポイントとなるディレクトリに既にファイルが存在している。
    * **解決策**: ディレクトリを空にするか、別のマウントポイントを指定する。

## 環境変数と設定ファイル

### 関連する設定ファイル

* **`/etc/auto.master`**: メインの設定ファイル。どのディレクトリをどのマップファイルで管理するかを定義。
* **`/etc/sysconfig/autofs`**: タイムアウトやログレベルなどのグローバル設定（ディストリビューションによりパスが異なる）。
* **`/etc/auto.misc` / `/etc/auto.home`**: 個別のマウント条件を記述するマップファイル。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: 不適切なマップファイルのパーミッション。一般ユーザーが書き込み可能な場合、任意のパスに悪意あるデバイスをマウントさせられる可能性があります。
* **悪用シナリオ**: 攻撃者が `/etc/auto.master` に自身のスクリプトを実行させるような設定を差し込み、管理者がアクセスした瞬間にコードを実行させる（間接的な実行）。

### GTFOBins / LOLBAS における利用例

`automount` 自体は登録されていませんが、これによってマウントされる **NFS共有の設定（no_root_squash）** は、権限昇格の定番ルートです。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: マップファイルの所有者を `root:root` にし、パーミッションを `644` 以下に制限する。NFS側では必ず `root_squash` を有効にする。
* **Detection (検知)**: `/etc/auto.*` ファイルの変更を `auditd` で監視し、予期せぬマウントポイントの追加を警告する。

## 注意点・補足

* **ゴーストディレクトリ**: `browse_mode = yes` に設定すると、アクセス前でもディレクトリが見えるようになりますが、セキュリティ上は `no` にして「隠しておく」のが一般的です。
* **アンマウントの失敗**: プロセスがマウント内のファイルを開いている（またはカレントディレクトリにしている）場合、自動アンマウントは失敗します。
