# Command - Linux - readelf - ELF ファイルの情報を表示する

## 概要

`readelf` は、一つまたは複数の ELF 形式のオブジェクトファイルに関する情報を表示します。`objdump` と似ていますが、`readelf` は BFD ライブラリ（バイナリ記述ライブラリ）に依存せず、独自に ELF 構造を解析するため、より詳細で正確な ELF 固有の情報を提供できるのが強みです。

## 🛠 `readelf` vs `objdump`

| 特徴 | `readelf` | `objdump` |
| :--- | :--- | :--- |
| **依存関係** | 独立（ELF 専門）。 | BFD ライブラリに依存。 |
| **得意分野** | セクション/プログラムヘッダの構造。 | 逆アセンブル（機械語の解読）。 |
| **正確性** | ELF の仕様に極めて忠実。 | 多様なバイナリ形式に対応。 |

## 主要なオプション

| オプション | 説明 |
| :--- | :--- |
| ⭐️ `-h`, `--file-header` | **最頻用。** ELF ヘッダを表示（マジックナンバー、アーキテクチャ等）。 |
| ⭐️ `-l`, `--program-headers` | プログラムヘッダ（メモリロード時のセグメント情報）を表示。 |
| ⭐️ `-S`, `--section-headers` | セクションヘッダ（コード、データ、シンボル情報の場所）を表示。 |
| ⭐️ `-s`, `--symbols` | シンボルテーブル（関数名や変数名の一覧）を表示。 |
| ⭐️ `-d`, `--dynamic` | 動的セクション（必要な共有ライブラリ .so の情報など）を表示。 |
| ⭐️ `-a`, `--all` | **全部盛り。** 上記の情報をすべて表示する。 |

## コマンド例

```bash
# 実行ファイルの基本情報を確認（32bit/64bitの判別などに便利）
readelf -h /bin/ls

# そのプログラムが依存している共有ライブラリを確認
readelf -d /bin/ls | grep NEEDED

# 関数名（シンボル）の一覧を表示して中身を推測する
readelf -s my_program
````

## 業務シナリオ別活用法

### 1. インフラ構築・運用視点

- **タスク**: ライブラリの依存関係トラブルシューティング。
- **解説**: 「ファイルはあるのに起動しない」という時、`readelf -d` で要求されている `.so` ファイル名を確認し、システム内のパスが正しいか、アーキテクチャ（x86_64 か AArch64 か等）が一致しているかを `readelf -h` で突き止めます。

### 2. ブルーチーム視点

- **タスク**: 不審なバイナリの初期解析（マルウェア解析）。
- **解説**:
    - **パッカーの検知**: セクション名が異常だったり、シンボルが極端に少なかったりする場合、難読化されている可能性があります。
    - **セキュリティ機構の確認**: スタック保護（Stack Canary）や実行不可スタック（NX bit）が有効かどうかをヘッダーから読み取ります。

### 3. レッドチーム視点

- **タスク**: エクスプロイト（攻撃コード）の開発。
- **解説**:
    - **GOT/PLT 攻撃**: 動的リンクの仕組みを利用した攻撃のために、関数のアドレスが解決される場所を `readelf` で特定します。
    - **情報収集**: シンボルテーブルから、開発者が消し忘れたデバッグ用の関数や秘密の機能がないかを探ります。

## セキュリティに関する考慮事項

### 脆弱性とリスク

- **情報の露出**: 実行ファイルにシンボル情報（`-s`）が残っていると、攻撃者にプログラムの内部構造を教えることになります。

### 対応策・緩和策 (開発者向け)

- **`strip` コマンドの実行**: 本番環境にデプロイする前に `strip --strip-all` を実行して、不要なシンボル情報を削除し、`readelf` で覗かれても解析を困難にする（リバースエンジニアリング耐性を高める）のが定石です。
