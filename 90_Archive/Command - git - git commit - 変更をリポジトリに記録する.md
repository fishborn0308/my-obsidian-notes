# Command - git - git commit - 変更をリポジトリに記録する

## 概要

`git commit` は、`git add` によって**ステージングエリア**に追加された変更のスナップショットを、ローカルリポジトリの履歴として永続的に記録するためのコマンドです。各コミットには、一意のハッシュIDと、誰が・いつ・なぜ変更したかを示すコミットメッセージが付与されます。

コミットは、プロジェクトの歴史におけるセーブポイントのようなものであり、バージョン管理の中核をなす操作です。

(出自: `git` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`git add`** | `git add` は変更をコミットの「準備段階」であるステージングエリアに追加します。`git commit` は、その準備された変更を実際に履歴として**記録**します。 | - |
| **`git reset`** | コミットを取り消したり、履歴を巻き戻したりします。`commit` が履歴を**進める**のに対し、`reset` は履歴を**戻す**操作です。 | - |
| **`git revert`** | 既存のコミットを**取り消すための新しいコミットを作成**します。履歴を書き換えないため、共有リポジトリでの変更の取り消しに適しています。 | `git revert` (共有リポジトリ) |

## よく連携されるコマンド

`git commit` は、変更を準備する `git add` と、コミット後に履歴を確認する `git log` と一連のワークフローで連携します。

### シナリオ例: 変更を作成し、コミットとして記録する (インフラ構築・運用視点)

* **目的**: 新しい設定ファイルを作成し、その変更をリポジトリの履歴として記録する。
* **連携コマンド**: `git status`, `git add`
* **解説**: ファイルを変更した後、`git status` で状態を確認し、`git add` でステージング、最後に `git commit` で記録、というGitの最も基本的なワークフローです。
* **コマンド例**:

		```bash
    # 1. 変更内容を確認

    git status

    # 2. 変更をステージング

    git add new_playbook.yml

    # 3. コミットメッセージを付けてコミット

    git commit -m "feat: Add new Ansible playbook for web server setup"

    ```



## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **メッセージ指定** | |
| ⭐ `-m <message>`, `--message=<message>`| コミットメッセージをコマンドラインで直接指定します。 |
| `-F <file>` | コミットメッセージをファイルから読み込みます。 |
| **ステージングとコミット** | |
| ⭐ `-a`, `--all` | 追跡中のファイルの変更を**自動的にステージングして**からコミットします。**新規ファイルは含まれない**ため注意が必要です。 |
| **コミットの修正** | |
| ⭐ `--amend` | **直前のコミットを修正**します。ステージングエリアの内容を直前のコミットに追加し、コミットメッセージも編集できます。 |
| **その他** | |
| `-v`, `--verbose` | コミットメッセージエディタに、コミットされる変更の差分（diff）を表示します。 |
| `--no-verify` | `pre-commit` フックなどの検証をスキップしてコミットします。 |
| `-S`, `--gpg-sign`| コミットにGPG署名を追加します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 直前のコミットに、含め忘れたファイルの追加や、メッセージのタイポ修正を行う。
* **組み合わせ**: `git add <forgotten_file>` → `git commit --amend`
* **解説**: プッシュする前に、直前のコミットをクリーンな状態に修正します。`--amend` は新しいコミットを作成するのではなく、直前のコミットを書き換えます。
* **例**:

		```bash
    # (コミット後に、files/template.j2 を追加し忘れたことに気づく)

    git add files/template.j2

    git commit --amend --no-edit # --no-editでメッセージは変更しない

    ```



## 2. ブルーチーム視点

* **タスク**: セキュリティインシデントの対応記録を、GPG署名付きでコミットする。
* **組み合わせ**: `git commit -S -m "<message>"`
* **解説**: `-S` オプションでコミットにGPG署名を追加します。これにより、そのコミットが本当にその本人によって作成され、改ざんされていないことを暗号学的に保証できます。インシデント対応の記録など、監査証跡としての信頼性が求められる場面で重要です。
* **例**:

		```bash
    # 調査結果を署名付きでコミット

    git commit -S -m "docs: Add initial incident report for case #123"

    ```

# エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `Changes not staged for commit`
		* **考えられる原因**: コミットしようとしましたが、ステージングエリアに何も変更がありません。
		* **解決策**: `git add <file>` を実行して、コミットに含めたい変更をステージングしてください。

2. **エラーメッセージ例 1**: `Please tell me who you are.`
		* **考えられる原因**: Gitのユーザー名とメールアドレスが設定されていません。
		* **解決策**: `git config --global user.name "Your Name"` と `git config --global user.email "you@example.com"` を実行して設定してください。

# セキュリティに関する考慮事項

## 脆弱性と悪用事例

* **脆弱性**: **機密情報のコミット**。
* **悪用シナリオ**: 開発者が誤ってパスワード、APIキー、秘密鍵などの機密情報をファイルに記述し、そのままコミットしてしまうケースが後を絶ちません。一度コミットされ、公開リポジトリにプッシュされると、その情報の流出は取り消しが非常に困難になります。

## GTFOBins / LOLBAS における利用例

`git commit` は直接的なコード実行機能を持たないため、GTFOBinsにはリストされていません。しかし、`git` 自体はリストされており、`pre-commit` フックなどを悪用して権限昇格に繋がる可能性があります。

## 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **`.gitignore` を徹底**し、機密情報を含む可能性のあるファイル（`.env`など）を追跡対象から除外する。**プリコミットフック** (`git-secrets`など) を導入し、機密情報が含まれていないかコミット前に自動でチェックする。
* **Detection (検知)**: CI/CDパイプラインに `trufflehog` などのシークレットスキャンツールを組み込み、リポジトリにプッシュされた内容を監視する。

# 注意点・補足

* **コミットメッセージの重要性**: コミットメッセージは、未来の自分やチームメンバーにとっての貴重なドキュメントです。「なぜ」その変更を行ったのかを明確に記述する習慣をつけましょう。
