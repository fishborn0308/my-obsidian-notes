# Command  - Linux - dig - DNSクエリユーティリティ

## 概要

`dig` (Domain Information Groper) は、DNS (Domain Name System) サーバーに問い合わせを行い、その応答を詳細に表示するための強力なコマンドラインツールです。単にIPアドレスを調べるだけでなく、MXレコード（メールサーバー）、NSレコード（ネームサーバー）、TXTレコードなど、様々な種類のDNSレコードを取得できます。その出力は構造化されており、スクリプトでの処理が容易なため、ネットワーク管理者やセキュリティ専門家によるDNSのトラブルシューティングや調査に広く利用されています。

(出自: `bind-utils` (RHEL系) or `dnsutils` (Debian系) パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`nslookup`** | 古くからあるDNSクエリツール。`dig` は出力がより詳細で構造化されており、TTLなどの情報も表示するため、プロフェッショナルな現場では `dig` の方が好まれます。 | `dig` |
| **`host`** | Aレコードなどをシンプルに問い合わせるためのコマンド。手早くIPアドレスを知りたいだけなら `host`、詳細に分析したいなら `dig` と使い分けます。 | - |
| **`getent hosts`** | DNSだけでなく、`/etc/hosts` ファイルなど、システムのネームサービス全体を参照して名前解決を行います。`dig` はDNSサーバーに直接問い合わせます。 | - |

## よく連携されるコマンド

`dig` の出力から特定の情報を抜き出すために、`grep` や `awk` などのテキスト処理ツールと連携することがあります。

### シナリオ例: 特定ドメインのSPFレコードを確認する (インフラ構築・運用視点)

* **目的**: メールの到達性問題を調査するため、ドメインのSPFレコード（送信ドメイン認証）が正しく設定されているかを確認する。
* **連携コマンド**: `grep`
* **解説**: `dig` でTXTレコードを問い合わせ、その結果の中から `spf1` を含む行だけを `grep` でフィルタリングします。
* **コマンド例**:

```bash
# google.com のTXTレコードを問い合わせ、SPFレコードのみを表示
dig google.com TXT +short | grep "spf1"
```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **クエリ制御** | |
| ⭐ `[record_type]` | `A` (デフォルト), `AAAA`, `MX`, `NS`, `TXT`, `SOA`, `ANY` など、問い合わせるレコードタイプを指定します。 |
| ⭐ `@<server>` | 問い合わせ先のDNSサーバーを明示的に指定します。（例: `@8.8.8.8`） |
| `-x <ip_address>` | IPアドレスの**逆引き** (PTRレコードの問い合わせ) を行います。 |
| `axfr` | **ゾーン転送**を試みます。 |
| **出力制御** | |
| ⭐ `+short` | 問い合わせ結果の値だけを簡潔に表示します。 |
| `+trace` | ルートDNSサーバーから始まる名前解決の階層的な問い合わせ経路を全て表示します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 新しく設定したDNSレコードが、外部のDNSサーバーから正しく引けるか確認する。
* **組み合わせ**: `dig @<dns_server> <domain>`
* **解説**: `@` を使い、システムのデフォルトリゾルバではなく、Google Public DNS (`8.8.8.8`) などの外部DNSサーバーに直接問い合わせることで、設定がインターネット全体に反映されているかを確認します。
* **例**:

```bash
# GoogleのDNSサーバーに [www.example.com](https://www.example.com) を問い合わせる
dig @8.8.8.8 [www.example.com](https://www.example.com)
```

### 2. ブルーチーム視点

* **タスク**: DNSハイジャックやキャッシュ汚染の疑いを調査する。
* **組み合わせ**: `dig +trace <domain>`
* **解説**: `+trace` を使い、ルートDNSサーバーから始まる正規の名前解決経路を追跡します。もし、途中で見慣れないDNSサーバーを経由していたり、最終的な応答が期待と異なっていたりすれば、DNS関連の攻撃を受けている可能性を示唆します。
* **例**:

```bash
# ルートサーバーからの委任経路を追跡し、正当性を確認
dig +trace www.nic.ad.jp
```

### 3. レッドチーム視点

* **タスク**: 標的組織のDNSサーバーから内部情報を窃取する。
* **組み合わせ**: `dig axfr @<target_ns_server> <target_domain>`
* **解説**: **DNSゾーン転送**を試みます。DNSサーバーがゾーン転送を許可するように誤って設定されている場合、そのドメインに登録されている全てのDNSレコード（サブドメイン、内部ホストのIPアドレスなど）を一度に窃取でき、攻撃対象の洗い出しに極めて有効です。
* **例**:

```bash
# ns1.example.comサーバーに対してexample.comドメインのゾーン転送を試みる
dig axfr @ns1.example.com example.com
```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **ステータス**: `status: NXDOMAIN`
		* **考えられる原因**: "Non-Existent Domain"。問い合わせたドメイン名やホスト名は存在しません。
		* **解決策**: ドメイン名のタイプミスがないか確認してください。

2. **ステータス**: `status: SERVFAIL`
		* **考えられる原因**: "Server Failure"。DNSサーバーが応答の生成に失敗しました。サーバー自体の問題や、上位サーバーへの問い合わせ失敗などが考えられます。
		* **解決策**: 別のDNSサーバー (`@8.8.8.8`など) を指定して再度試み、問題の切り分けを行います。

## 環境変数と設定ファイル

* `dig` の動作に直接影響する環境変数は通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](OS%20%20-%20Linux%20-%20environment_and_config%20-%20環境変数と設定ファイル.md) を参照。

### 関連する設定ファイル

* **`/etc/resolv.conf`**: `dig` が `@<server>` オプションなしで実行された場合に、問い合わせ先として使用するデフォルトのDNSリゾルバが記述されています。
* **`~/.digrc`**: ユーザーのホームディレクトリにこのファイルを置くと、`dig` のデフォルトオプションを定義できます。（例: `+short`）

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `dig` 自体ではなく、DNSプロトコルの仕組みやサーバーの設定不備が悪用されます。
* **悪用シナリオ**: `dig` は、攻撃者が**情報収集（偵察）**フェーズで最も重要なツールの1つです。`NS`, `MX` レコードからインフラをマッピングし、`axfr` (ゾーン転送) でサブドメインを列挙します。

### GTFOBins / LOLBAS における利用例

`dig` はネットワーク越しの情報収集ツールであり、ローカルシステム上でのファイル操作や権限昇格には直接利用できないため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 権威DNSサーバーの設定で、ゾーン転送 (`AXFR`) を許可するIPアドレスを、セカンダリDNSサーバーなどの信頼できるホストのみに厳格に制限する。
* **Detection (検知)**: 内部から外部への異常なDNSクエリ（特に `ANY` や `AXFR` タイプのリクエスト）を監視することで、偵察活動を検知できる可能性があります。

## 注意点・補足

* **キャッシュの影響**: `dig` の結果に表示される `TTL` (Time To Live) は、そのレコードがキャッシュに保持される残り時間（秒）を示します。DNS設定を変更した際は、このTTLが切れるまで古い情報が返され続けることがあります。
