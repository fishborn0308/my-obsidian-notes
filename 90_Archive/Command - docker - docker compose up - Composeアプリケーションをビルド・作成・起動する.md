# Command - docker - docker compose up - Composeアプリケーションをビルド・作成・起動する

## 概要

`docker compose up` は、`docker-compose.yml` ファイルに定義されたマルチコンテナアプリケーションを**起動するための中心的なコマンド**です。このコマンド1つで、以下の処理を必要に応じて実行します。

1. **イメージのビルド**: `build` ディレクティブで指定されたサービスのDockerイメージをビルドします。
2. **ネットワーク・ボリュームの作成**: デフォルトのネットワークや名前付きボリュームを作成します。
3. **コンテナの作成**: サービスの定義に基づいてコンテナを作成します。
4. **コンテナの起動**: 作成したコンテナを起動します。

デフォルトでは、コマンドを実行したターミナルにすべてのコンテナのログを集約して表示します（フォアグラウンドモード）。

(出自: `docker compose` の基本機能)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`docker compose start`** | **すでに作成済みで、停止している**コンテナを起動します。`up` のようにコンテナを(再)作成したりはしません。 | `up` (初回/更新時) |
| **`docker run`** | **単一**のコンテナをイメージから作成・実行します。`docker compose up` は**複数**のコンテナを協調させて起動します。 | - |
| **`docker compose run`** | サービスで定義されたコンテナを**ワンショット**で実行します。主に、データベースのマイグレーションなど、一時的なタスクに使います。 | - |

## よく連携されるコマンド

`docker compose up` は、起動後に状態を確認する `docker compose ps` や、ログを確認する `docker compose logs` と連携して使われます。

### シナリオ例: アプリケーションスタックの起動と確認 (インフラ構築・運用視点)

* **目的**: アプリケーションを構成する全サービスをバックグラウンドで起動し、正常に稼働しているかを確認する。
* **連携コマンド**: `docker compose ps`, `docker compose logs`
* **解説**: `-d` オプションでバックグラウンド起動した後、`ps` で全サービスが `Up` 状態かを確認し、問題があれば `logs` で原因を調査するのが一連の流れです。
* **コマンド例**:

		```bash
    # 1. アプリケーションをバックグラウンドで起動

    docker compose up -d

    # 2. 起動状態を確認

    docker compose ps

    # 3. (必要であれば) ログを確認

    docker compose logs webapp

    ```


## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **実行モード** | |
| ⭐ `-d`, `--detach` | **デタッチモード**。コンテナをバックグラウンドで起動し、ターミナルを解放します。サーバー環境での利用に必須です。 |
| ⭐ `--build` | イメージが存在していても、**強制的にビルドを再実行**します。`Dockerfile` やソースコードを変更した際に使用します。 |
| `--force-recreate` | 設定やイメージに変更がなくても、コンテナを一度削除してから再作成します。 |
| **その他** | |
| `--scale <service>=<num>`| 指定したサービスのコンテナを `<num>` 個にスケールさせて起動します。 |
| `--no-deps`| リンクされたサービスのコンテナを起動しません。 |
| `--abort-on-container-exit`| いずれか一つのコンテナが停止した時点で、すべてのコンテナを停止します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: ソースコードの変更を反映させるため、イメージを再ビルドして環境を立ち上げ直す。
* **組み合わせ**: `docker compose up --build --force-recreate -d`
* **解説**: アプリケーションのコードや `Dockerfile` を更新した場合に、古いキャッシュやコンテナが残ることを避け、クリーンな状態で環境を立ち上げ直します。
* **例**:

		```bash
    # 変更を確実に反映させてアプリケーションを起動

    docker compose up --build -d

    ```


### 2. ブルーチーム視点

* **タスク**: サンドボックス環境で、不審な `docker-compose.yml` ファイルを起動して挙動を分析する。
* **組み合わせ**: `docker compose -f <suspicious.yml> up`
* **解説**: 調査対象のファイルを `-f` で指定し、フォアグラウンドモードで起動します。ログをリアルタイムで監視することで、アプリケーションがどのようなネットワーク接続を試み、どのような処理を行っているかを動的に解析します。
* **例**:

		```bash
    # サンドボックス内で、不審なComposeアプリケーションを起動して挙動を監視

    docker compose -f /mnt/malware/docker-compose.yml up

    ```


### 3. レッドチーム視点

* **タスク**: 侵入したホスト上で、攻撃用のインフラを迅速に展開する。
* **組み合わせ**: `docker compose -f <attack.yml> up -d`
* **解説**: C2サーバー、内部スキャン用のプロキシ、データベースなどの攻撃ツール一式を定義したカスタムの `docker-compose.yml` を使い、バックグラウンドで攻撃インフラを一括で起動します。

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `Error response from daemon: driver failed programming external connectivity ...: Bind for 0.0.0.0:8080 failed: port is already allocated`
		* **考えられる原因**: `docker-compose.yml` で指定したホスト側のポート(この例では8080)が、すでに他のプロセスによって使用されています。
		* **解決策**: `sudo lsof -i :8080` や `sudo ss -tulpn | grep 8080` でポートを使用中のプロセスを特定し、停止させるか、`docker-compose.yml` のポートマッピングを変更します。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **信頼できないイメージ/Composeファイルの実行**。
* **悪用シナリオ**: インターネット上で見つけた信頼性の低い `docker-compose.yml` を安易に実行すると、悪意のあるイメージを `pull` したり、ホストの機密ディレクトリをコンテナにマウントしたりする記述が含まれている可能性があります。

### GTFOBins / LOLBAS における利用例

`docker compose` は `docker` コマンドの一部として扱われます。`sudo docker` の権限があれば、任意の操作が可能です。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: `docker-compose.yml` の `ports` ディレクティブで、`127.0.0.1:8080:80` のようにローカルホストにのみバインドし、意図しない外部公開を防ぐ。
* **Detection (検知)**: `docker compose` の実行ログやDockerデーモンのAPIコールを監視し、不審なイメージの `pull` やコンテナの作成がないかを確認する。

## 注意点・補足

* **コンテナの再作成**: `docker-compose.yml` の内容や、イメージが更新された後に `docker compose up` を実行すると、Composeは変更があったサービスについてコンテナを自動的に停止・削除し、新しい設定で再作成します。この挙動を理解しておくことが重要です。
