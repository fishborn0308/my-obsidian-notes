# Command  - Linux - dmidecode - DMIテーブルからハードウェア情報を表示する

## 概要

`dmidecode` は、システムの **DMI (Desktop Management Interface)** テーブル（SMBIOSとも呼ばれる）の内容を読み取り、人間が理解しやすい形式で表示するコマンドです。DMIテーブルには、システムのBIOS/UEFIファームウェアによって記録された、マザーボード、CPU、メモリ、シャーシ（筐体）、BIOSバージョン、シリアル番号など、非常に詳細なハードウェア構成情報が格納されています。このコマンドは、物理サーバーや仮想マシンの正確なスペックをOS上から確認するために利用され、**実行にはroot権限が必要**です。

(出自: `dmidecode` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`lshw`** | `dmidecode` よりも広範なハードウェア（PCI, USBなど）の情報を、カーネルが認識している状態に基づいて表示します。`dmidecode` は**ファームウェアが提供する静的な情報**に特化しています。 | `lshw` (全体像) |
| **`lspci`, `lsusb`, `lscpu`**| それぞれPCIデバイス、USBデバイス、CPUといった**特定のコンポーネント**に特化した情報を表示します。`dmidecode` はシステム全体の構成情報を表示します。 | - |
| **`/proc` ファイルシステム**| カーネルが認識している現在のハードウェアの状態を動的に表示します。`dmidecode` は、ファームウェアが記録した静的なハードウェア仕様を表示します。 | - |

## よく連携されるコマンド

`dmidecode` の出力は非常に冗長なため、`grep` と連携して必要な情報だけをフィルタリングするのが一般的です。

### シナリオ例: メモリスロットの情報を確認する (インフラ構築・運用視点)

* **目的**: サーバーに搭載されているメモリの詳細と、空きスロットの数を確認する。
* **連携コマンド**: `grep`
* **解説**: `dmidecode -t memory` の出力は全メモリスロットの情報を含むため長大です。`grep` を使って `Size:` や `Locator:` といったキーワードでフィルタリングすることで、必要な情報だけを簡潔に表示できます。
* **コマンド例**:

```bash
# メモリに関する情報から、サイズとスロット名だけを抽出
sudo dmidecode -t memory | grep -E "Size:|Locator:"
```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **フィルタリング** | |
| ⭐ `-t`, `--type <keyword>` | `bios`, `system`, `memory` のように、表示したい情報の**タイプ（クラス）**を指定します。 |
| ⭐ `-s`, `--string <keyword>` | `system-serial-number`, `bios-version` のように、特定の**文字列情報**だけをピンポイントで取得します。 |
| **その他** | |
| `-q`, `--quiet` | 一部の情報（デコードエラーなど）を抑制して表示します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: サーバーのシリアル番号や製品名を特定する。
* **組み合わせ**: `sudo dmidecode -s <keyword>`
* **解説**: `-s` (`--string`) オプションを使い、システムのシリアル番号やモデル名だけをピンポイントで取得します。資産管理やハードウェア保守の際に不可欠です。
* **例**:

```bash
# システムのシリアル番号を取得
sudo dmidecode -s system-serial-number

# サーバーの製品名を表示
sudo dmidecode -s system-product-name
```

### 2. ブルーチーム視点

* **タスク**: 調査対象ホストのハードウェアプロファイルを正確に記録する。
* **組み合わせ**: `dmidecode -t system`
* **解説**: インシデントレスポンスの初期調査で、対象システムの正確なハードウェアプロファイル（メーカー、モデル、BIOSバージョン）を把握します。この情報を元に、特定のファームウェアに存在する既知の脆弱性（例: UEFIルートキットに悪用される脆弱性）の影響を受ける可能性があるかを調査します。
* **例**:

```bash
# システムのメーカー、モデル、シリアル番号、UUIDなどを確認
sudo dmidecode -t system
```

### 3. レッドチーム視点

* **タスク**: 自身が活動している環境が物理マシンか仮想マシンかを特定する。
* **組み合わせ**: `dmidecode -s system-product-name`
* **解説**: 侵入後の偵察の一環として、このコマンドで環境を把握します。`VMware Virtual Platform` や `KVM` といった出力が得られれば、仮想マシン上であることが確定し、VMエスケープ攻撃の可能性を探ります。物理サーバーのモデル名が分かれば、そのモデル固有の管理インターフェイス（iDRACなど）の脆弱性を突くといった次の攻撃計画を立てられます。
* **例**:

```bash
# 仮想環境かどうかを判定
sudo dmidecode -s system-product-name
```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `/dev/mem: Permission denied`
		* **考えられる原因**: `dmidecode` はDMIテーブルを読み取るために `/dev/mem` などの低レベルなデバイスにアクセスする必要があるため、`root` 権限で実行されていません。
		* **解決策**: コマンドの前に `sudo` を付けて実行してください。

## 環境変数と設定ファイル

* `dmidecode` の動作に直接影響する環境変数や設定ファイルは通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](OS%20%20-%20Linux%20-%20environment_and_config%20-%20環境変数と設定ファイル.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `dmidecode` 自体は読み取り専用ツールであり、脆弱性を持つ可能性は低いです。
* **悪用シナリオ**: `dmidecode` の出力は、システムのハードウェアに関する極めて詳細な情報を含みます。シリアル番号、資産管理タグ、メーカー、モデル名などは、攻撃者が標的のシステムを正確に特定し、物理的な攻撃や、ハードウェア固有の脆弱性を利用した攻撃を計画するための重要な**情報漏洩**に繋がります。

### GTFOBins / LOLBAS における利用例

`dmidecode` は情報表示のみを行うコマンドであり、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 最小権限の原則に従い、不要なユーザーに `sudo` での `dmidecode` 実行権限を与えない。
* **Detection (検知)**: `auditd` などで `dmidecode` コマンドの実行を監視し、不審な偵察活動がないかを確認する。

## 注意点・補足

* **情報の正確性**: `dmidecode` が表示する情報は、あくまでBIOS/UEFIのDMIテーブルに書き込まれている内容に依存します。ハードウェアベンダーが正しい情報を書き込んでいなければ、不正確な情報が表示されることもあります。
