# Command - git - git remote - リモートリポジトリの管理

## 概要

`git remote` コマンドは、ローカルGitリポジトリが対話する**リモートリポジトリ**（たとえば、GitHub、GitLabなどのサーバー上のリポジトリ）を管理するために使用されます。このコマンドを使うことで、リモートリポジトリの追加、削除、名前変更、表示、URLの変更などが行えます。通常、`git clone`でリポジトリを複製すると、自動的に`origin`という名前のリモートが設定されます。

(出自: `git` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`git fetch`** | `fetch` は登録されたリモートからデータを**取得**します。`remote` はそのリモート自体の**登録や管理**を行います。 | - |
| **`git push`** | `push` はローカルの変更を登録されたリモートに**送信**します。 | - |
| **`git pull`** | `pull` はリモートからデータを**取得してマージ**します。 | - |

## よく連携されるコマンド

`git remote` は、リモートリポジトリとのやり取りを始める前や、その設定を変更する際に使われるため、その後に続く`git fetch`、`git pull`、`git push`といったコマンドと密接に連携します。

### シナリオ例: 既存のローカルリポジトリに新しいリモートを追加する (インフラ構築・運用視点)

* **目的**: 既存のローカルリポジトリを、新たなリモートサーバー（GitLabなど）と同期させたい。
* **連携コマンド**: `git push`
* **解説**: `git remote add`で新しいリモートの名前とURLを登録し、`git push`でローカルの変更を新しいリモートに送信します。
* **コマンド例**:

		```bash
    # 新しいリモートリポジトリ 'gitlab' を追加

    git remote add gitlab [https://gitlab.com/your-org/your-project.git](https://gitlab.com/your-org/your-project.git)

    # 新しいリモートが追加されたことを確認

    git remote -v

    # 新しいリモートにローカルブランチをプッシュ

    git push -u gitlab main

    ```



## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **表示と管理** | |
| ⭐ `-v`, `--verbose` | リモートの名前とそのURLを詳細に表示します。 |
| ⭐ `add <name> <url>` | `<name>` という名前で `<url>` のリモートリポジトリを追加します。 |
| ⭐ `rm <name>` | 指定した名前のリモートリポジトリを削除します。 |
| `rename <old> <new>` | 既存のリモートの名前を変更します。 |
| `set-url <name> <url>`| 指定したリモートのURLを変更します。 |
| `prune <name>` | リモートに存在しないローカルのリモート追跡ブランチを削除します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: オープンソースプロジェクトのフォーク元リポジトリ (`upstream`) を追加し、本家の最新の改善を取り込めるようにする。
* **組み合わせ**: `git remote add upstream <forked-repo-url>`
* **解説**: 自社でフォークしたIaCツールなどを、本家プロジェクトの最新状態から乖離しないように同期するために使用します。
* **例**:

		```bash
    # 本家のリポジトリを 'upstream' として追加

    git remote add upstream [https://github.com/original-project/iac-tool.git](https://github.com/original-project/iac-tool.git)

    # 全リモートから変更を取得

    git fetch --all

    ```



## 2. ブルーチーム視点

* **タスク**: 侵害されたシステムから取得したGitリポジトリを、安全なフォレンジックサーバーに転送して分析する。
* **組み合わせ**: `git remote add forensic-server <server-url>` → `git push --all`
* **解説**: 既存のリモート設定を確認後、安全なフォレンジックサーバーに新しいリモートを追加し、調査中のリポジトリの全履歴をプッシュして、詳細なオフライン分析を行います。
* **例**:

		```bash
    # フォレンジック分析サーバーを新しいリモートとして追加

    git remote add forensic-server ssh://forensic-user@192.168.1.100/var/git/compromised.git

    # リポジトリのすべてのブランチをフォレンジックサーバーにプッシュ

    git push forensic-server --all

    ```



## 3. レッドチーム視点

* **タスク**: 既存のリモート設定を操作し、マルウェアやペイロードのデリバリーに利用できるバックドア的なリモートを追加する。
* **組み合わせ**: `git remote set-url origin <fake-repo-url>`

# エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `fatal: remote <name> already exists.`
		* **考えられる原因**: 指定した名前のリモートがすでに存在します。
		* **解決策**: 別の名前を使用するか、既存のリモートを `git remote rm <name>` で削除してから再度追加します。

* **`.git/config`**: `git remote add` などのコマンドは、このリポジトリローカルな設定ファイルを直接編集します。

# セキュリティに関する考慮事項

## 脆弱性と悪用事例

* **脆弱性**: **悪意のあるリモートの追加**と**リモートURLの改ざん**。
* **悪用シナリオ**: 攻撃者が内部システムに侵入した場合、既存のGitリポジトリに悪意のあるリモートリポジトリを追加したり、URLを改ざんしたりする可能性があります。これにより、開発者が意図せずマルウェアを含むコードをフェッチしたり、機密情報を悪意のあるサーバーにプッシュしたりするリスクが生じます。

## GTFOBins / LOLBAS における利用例

`git remote` は直接的なコード実行機能を持たないため、GTFOBinsにはリストされていません。

## 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: Gitリポジトリへのアクセス権限を厳格に管理し、信頼できるユーザーのみがリモート設定を変更できるようにする。
* **Detection (検知)**: CI/CDパイプラインやホスト型IDS/EDRを使用して、`.git/config` ファイルへの予期せぬ変更や改ざんを監視する。

# 注意点・補足

* **`origin`という慣習**: `git clone` を実行すると、複製元のリモートリポジトリはデフォルトで `origin` という名前で登録されます。これは単なる慣習であり、変更も可能です。
