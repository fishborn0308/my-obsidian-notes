# Command - Linux - ln - ファイルやディレクトリへのリンクを作成する

## 概要

`ln` は、既存のファイルに対して別の名前（リンク）を作成します。リンクには **「ハードリンク」** と **「シンボリックリンク（ソフトリンク）」** の2種類があり、用途によって使い分ける必要があります。

## 🔗 ハードリンク vs シンボリックリンク

| 特徴 | ハードリンク | シンボリックリンク (`-s`) |
| :--- | :--- | :--- |
| **仕組み** | 実体（i-node）を直接共有する。 | 元のパス（名前）を指し示す。 |
| **実体の削除** | 片方を消してもデータは残る。 | 元を消すとリンクは「壊れる」。 |
| **ファイルシステム** | 同一パーティション内のみ。 | 異なるパーティションも OK。 |
| **ディレクトリ** | 原則不可。 | 可能。 |
| **見た目** | 元のファイルと見分けがつかない。 | ショートカットアイコン（`l`属性）。 |

## 主要なオプション

| オプション | 説明 |
| :--- | :--- |
| ⭐️ `-s`, `--symbolic` | **最頻用。** シンボリックリンクを作成する。 |
| ⭐️ `-f`, `--force` | リンク先と同名のファイルがある場合、上書きして作成する。 |
| ⭐️ `-v`, `--verbose` | 作成したリンクの結果を表示する。 |
| `-i`, `--interactive` | 上書き時に確認プロンプトを表示する。 |
| `-n`, `--no-dereference` | リンク先がディレクトリへのシンボリックリンクである場合、そのリンク自体を対象とする。 |

## コマンド例と構文

```bash
# 基本形: ln [オプション] <ターゲット（元）> <リンク名>
````

### 1. シンボリックリンクの作成 (最頻用)

設定ファイルやライブラリのバージョン管理でよく使われます。

Bash

```
# /etc/nginx/sites-available/default を current_config という名前でリンク
ln -s /etc/nginx/sites-available/default current_config
```

### 2. ハードリンクの作成

バックアップ時などに、容量を節約しつつ「別名」を作りたい場合に使います。

Bash

```
# file1.txt のハードリンク file2.txt を作成
ln file1.txt file2.txt
```

## 業務シナリオ別活用法

### 1. インフラ構築・運用視点

- **タスク**: ソフトウェアのバージョン切り替え。
- **解説**: `/opt/app_v1.0` と `/opt/app_v2.0` があるとき、シンボリックリンク `/opt/app_latest` を作成し、中身を切り替えるだけで、アプリケーションの設定を変えずにバージョンアップを実現します。

### 2. ブルーチーム視点

- **タスク**: シンボリックリンク攻撃の検知。
- **解説**: `/tmp` などの誰でも書き込めるディレクトリに、`/etc/shadow` などの重要ファイルへのシンボリックリンクが作成されていないか監視します。特権プログラムがリンクを辿って重要ファイルを上書きしてしまうリスクを防ぎます。

### 3. レッドチーム視点

- **タスク**: 競合状態（Race Condition）の悪用。
- **解説**: プログラムが一時ファイルを作成する直前に、そのパスを重要ファイルへのシンボリックリンクに差し替える（TOCTOU 攻撃）ことで、特権で重要ファイルを読み書きさせようと試みます。

## エラーメッセージとトラブルシューティング

1. **「ln: failed to create symbolic link '...': File exists」**:
    
    - **原因**: 指定したリンク名が既に存在する。
    - **解決策**: `-f` オプションで上書きするか、既存のファイルを削除する。
        
2. **「ln: failed to create hard link ...: Invalid cross-device link」**:
    
    - **原因**: 異なるディスク（パーティション）を跨いでハードリンクを作ろうとした。
    - **解決策**: シンボリックリンク (`-s`) を使用する。

## セキュリティに関する考慮事項

### 脆弱性とリスク

- **Dangling Symlinks**: リンク先が削除された「壊れたリンク」は、後に同じ名前で悪意あるファイルが置かれた際に、予期せぬ動作を引き起こす原因（踏み台）になることがあります。

### 対応策・緩和策 (ブルーチーム視点)

- **Linux Kernel 保護**: `/proc/sys/fs/protected_symlinks` を `1` に設定することで、自分以外が所有するディレクトリでの不審なシンボリックリンクの追跡を制限できます。
