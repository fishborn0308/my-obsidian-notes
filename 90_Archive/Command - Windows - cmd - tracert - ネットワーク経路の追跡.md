# Command - Windows - cmd - tracert - ネットワーク経路の追跡

## 概要

`tracert` (Trace Route) は、宛先へのパケットが通過するパスを特定するための診断ツールです。

IP パケットの **TTL (Time To Live)** 値を 1 ずつ増やしながら送信し、各ルーターから返される ICMP "Time Exceeded" メッセージを利用して、経由するホスト（ホップ）をリストアップします。

(出自: `Windows 標準搭載 - TCP/IP ユーティリティ`)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 (あれば) |
| :--- | :--- | :--- |
| `ping` | 宛先への疎通確認のみ。経路は不明。 | 疎通確認 |
| `pathping` | `tracert` と `ping` のハイブリッド。経路ごとのパケット損失率を計算できる。 | 詳細な品質調査 |
| `traceroute` | Linux/Unix 系。デフォルトで UDP パケットを使用する点に注意（Windows は ICMP）。 | - |
| `mtr` | Linux 等で利用される、継続的に経路情報を更新・表示する高機能ツール。 | 常時監視 |

## よく連携されるコマンド

### シナリオ例: 外部への通信経路に異常がないかの確認 (インフラ運用視点)

* **目的**: 外部サービス（例: クラウドストレージ）への接続が遅い際、自社ネットワーク内かプロバイダー側か、どの区間で遅延が発生しているかを特定する。
* **連携コマンド**: `ipconfig`, `tracert`, `nslookup`
* **解説**: まず `nslookup` で IP を特定し、`tracert` でホップごとの応答時間を確認します。
* **コマンド例**:
    ```cmd
    REM 特定のドメインへの経路を名前解決なし（高速）で確認
    tracert -d [www.google.com](https://www.google.com)
    ```

## スイッチ/オプション説明

| スイッチ | 説明 |
| :--- | :--- |
| ⭐ `-d` | **【推奨】** 各ホップの IP アドレスをホスト名に解決しない。実行速度が劇的に向上する。 |
| `-h max_hops` | ターゲットを検索する際の最大ホップ数を指定（デフォルトは 30）。 |
| `-j host-list` | 緩やかなソース ルートを指定（IPv4 のみ）。 |
| `-w timeout` | 各応答を待機するタイムアウト時間をミリ秒単位で指定。 |
| `-R` | 往復のパスを追跡する（IPv6 のみ）。 |
| `-S srcaddr` | 使用する送信元アドレスを指定（IPv6 のみ）。 |
| `-4` | IPv4 の使用を強制する。 |
| `-6` | IPv6 の使用を強制する。 |

## よく使われる組み合わせと業務シナリオ

### 1. インフラ構築・運用視点 (Docker/VMware環境)

* **タスク**: 仮想マシンから外部へ通信する際、意図したゲートウェイ（仮想ブリッジやNAT）を経由しているか確認する。
* **解説**: 最初の 1, 2 ホップ目が仮想マシンのデフォルトゲートウェイや物理ホストの IP であるかを確認します。
* **例**:
    ```cmd
    REM ローカルネットワークの出口を確認するために実行
    tracert -d 8.8.8.8
    ```

### 2. ブルーチーム視点

* **タスク**: VPN 接続がアクティブな際、社内リソースへの通信が正しく VPN トンネルを通っているか（スプリットトンネルの設定不備がないか）を検証する。
* **解説**: 通信経路の最初のホップが VPN ゲートウェイの IP になっているかを確認します。
* **例**:
    ```cmd
    REM 社内サーバーへの経路を調査
    tracert -d 10.0.0.50
    ```

### 3. レッドチーム視点

* **タスク**: 内部偵察 (Internal Reconnaissance)。ターゲットネットワークのセグメンテーション（ルーターの存在やファイアウォールの配置）を把握する。
* **解説**: ホップの途中でタイムアウト（`* * *`）が発生する箇所は、ICMP をブロックしているファイアウォールの存在を示唆します。
* **例**:
    ```cmd
    REM 内部ネットワークのトポロジーを把握するために実行
    tracert -d 172.16.0.1
    ```

```cmd

  1    <1 ms    <1 ms    <1 ms  192.168.1.1       <- デフォルトゲートウェイ
  2     2 ms     1 ms     1 ms  10.0.0.1          <- 組織内ルーター
  3    12 ms    11 ms    11 ms  203.0.113.1       <- ISPの入り口
  4     * * * 要求がタイムアウトしました。 <- FW等でICMPを遮断
  5    15 ms    14 ms    15 ms  8.8.8.8           <- 到着

```

## エラーメッセージとトラブルシューティング

1. **エラーメッセージ例 1**: `要求がタイムアウトしました。` (記号: `* * *`)
    
    - **考えられる原因**: そのホップにあるルーターが ICMP Time Exceeded パケットを送信しない設定になっている。またはネットワークが混雑している。
    - **解決策**: 経路上のセキュリティポリシーによる制限であることが多いため、次のホップが表示されていれば無視して問題ありません。
        
2. **エラーメッセージ例 2**: `宛先ホストに到達できません。`
    
    - **考えられる原因**: 自身のルーティングテーブルに適切なルートがない、または物理的な切断。
    - **解決策**: `ipconfig` や `route print` で自身のネットワーク設定を確認してください。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

- **情報漏洩 (Reconnaissance)**: 外部から `tracert` を許可していると、組織内部のネットワーク構成（ルーターの IP や段数）を攻撃者に容易に把握されてしまいます。

### 対応策・緩和策 (ブルーチーム視点)

- **Prevention (予防)**: 境界ファイアウォールで、外部からの ICMP Time Exceeded パケットや、内部からの送信に対する応答を制限する。
- **Detection (検知)**: 短時間に異なる宛先に対して複数の `tracert` が実行される振る舞いを監視する。

## 注意点・補足

- **名前解決のラグ**: `-d` スイッチを付けない場合、各ホップで DNS 逆引きを試みるため、非常に時間がかかります。通常は `-d` の使用を強く推奨します。
- **非対称ルーティング**: 行きの経路と帰りの経路が異なる場合、`tracert` で見えているのは「行き」の経路のみであることに注意が必要です。
