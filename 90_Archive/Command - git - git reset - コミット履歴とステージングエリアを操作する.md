# Command - git - git reset - コミット履歴とステージングエリアを操作する

## 概要

`git reset`コマンドは、Gitの**コミット履歴**、**ステージングエリア**(インデックス)、そして**作業ディレクトリ**(ワーキングツリー)を操作するための強力なコマンドです。このコマンドを使うと、過去の特定の状態にリポジトリを戻したり、コミットをやり直したり、ステージングされた変更を取り消したりできます。

`git reset`は非常に柔軟性がある一方で、その強力さゆえに誤って使うとコミット履歴を失う可能性もあるため、慎重な理解と使用が求められます。

(出自: `git` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`git restore`** | ファイルの変更を元に戻すことに特化しています。`git reset HEAD <file>` の機能は `git restore --staged <file>` の方が意図が明確で安全です。 | `git restore` (ファイル復元) |
| ⭐ **`git revert`** | 既存のコミットを**取り消す新しいコミットを作成**します。履歴を消去しないため、共有リポジトリでプッシュ済みのコミットを取り消す場合に安全です。 | `git revert` (共有リポジトリ) |
| **`git checkout`** | 主にブランチの切り替えに使われます。`reset` は履歴自体を**巻き戻す**ことができます。 | - |

## よく連携されるコマンド

`git reset`は、操作後にリポジトリの状態を確認する `git status` や `git log` と連携して使われることが多いです。

### シナリオ例: 直前のコミットをやり直す (インフラ構築・運用視点)

* **目的**: 直前のコミットに含めるべきファイルを忘れたため、コミットをやり直したい。
* **連携コマンド**: `git status`, `git add`, `git commit`
* **解説**: `--soft` オプションでコミットのみを取り消し、変更内容はステージングされた状態のまま保持します。その後、ファイルを追加して再度コミットします。
* **コマンド例**:

		```bash
    # 1. 直前のコミットを取り消し、変更内容はステージングされたままにする

    git reset --soft HEAD~1

    # 2. 含め忘れたファイルを追加でステージング

    git add forgotten_file.conf

    # 3. 再度コミットを実行

    git commit -m "feat: Add new server config with all files"

    ```



## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **リセットモード** | |
| ⭐ `--soft <commit>` | **コミット履歴のみをリセット**します。ステージングエリアと作業ディレクトリの変更はそのまま残ります。 |
| ⭐ `--mixed <commit>`| (デフォルト) **コミット履歴とステージングエリアをリセット**します。作業ディレクトリの変更は残ります。 |
| ⭐ `--hard <commit>` | **(危険)** **コミット履歴、ステージングエリア、作業ディレクトリの全てをリセット**します。未コミットの変更も全て失われます。 |
| **対象指定** | |
| `<commit>` | リセットの基準となるコミットを指定します。`HEAD~1` (1つ前) やコミットハッシュなど。 |
| `<file>` | `<file>` のみをステージングエリアから取り消します (作業ディレクトリの変更は残ります)。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 誤ったコミットをしたが、まだリモートにはプッシュしていない。そのコミットを完全に消し去りたい。
* **組み合わせ**: `git reset --hard HEAD~1`
* **解説**: ローカルリポジトリ限定で、直前のコミットとその変更内容を完全に破棄します。**作業ディレクトリの変更も消えるため、実行には細心の注意が必要です**。

### 2. ブルーチーム視点

* **タスク**: 不審なコミットを取り消し、その変更内容を分析のために作業ディレクトリに残したい。
* **組み合わせ**: `git reset --mixed HEAD~<n>`
* **解説**: `--mixed` オプションにより、ファイルはそのまま残るため、内容を確認したり修正したりする余地があります。
* **例**:

		```bash
    # 最後の2つのコミットを取り消し、変更内容を作業ディレクトリに残す

    git reset --mixed HEAD~2

    # 作業ディレクトリに残った変更内容を精査・分析

    git diff

    ```



## 3. レッドチーム視点

* **タスク**: 偵察中に誤って機密情報をコミットしてしまったが、まだプッシュしていない。コミットだけをなかったことにし、変更内容は残して編集し直したい。
* **組み合わせ**: `git reset --soft HEAD~1`
* **解説**: コミットだけを取り消し、変更内容はステージングされた状態として保持します。これにより、機密情報がコミット履歴に残るのを防ぎます。

1. **エラーメッセージ例 1**: `fatal: ambiguous argument 'HEAD~': unknown revision or path not in the working tree.`
		* **考えられる原因**: 指定したコミット参照が正しくないか、リポジトリが新しく、指定した数の親コミットが存在しません。
		* **解決策**: `git log` でコミット履歴を確認し、正しいコミット参照やハッシュ値を指定します。

# セキュリティに関する考慮事項

## 脆弱性と悪用事例

* **脆弱性**: **履歴の改ざん**。`git reset` はコミット履歴を書き換えることができます。
* **悪用シナリオ**: 攻撃者が `git reset` で自身の活動に関するコミット（マルウェアの追加など）を削除し、`git push --force` でリモートに反映させることで、活動痕跡を隠蔽しようとする可能性があります。

## GTFOBins / LOLBAS における利用例

`git reset` は直接的なコード実行機能を持たないため、GTFOBinsにはリストされていません。

## 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **強制プッシュ (`git push --force`) をブランチ保護ルールで禁止する**。これが履歴改ざんに対する最も効果的な対策です。
* **Detection (検知)**: Gitホスティングサービスの監査ログを監視し、不審な強制プッシュがないかを確認する。

# 注意点・補足

* **`--hard` の危険性**: `--hard` オプションは、未コミットの変更も含め、作業ディレクトリのファイルを完全に上書きします。この変更は基本的に `git reflog` を使わない限り元に戻せないため、実行前には必ず `git status` で状況を確認してください。
