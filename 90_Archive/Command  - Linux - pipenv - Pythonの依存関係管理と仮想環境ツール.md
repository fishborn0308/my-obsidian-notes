# Command  - Linux - pipenv - Pythonの依存関係管理と仮想環境ツール

## 概要

`pipenv` は、Pythonプロジェクトの依存関係管理と仮想環境の管理を自動化し、統合するためのツールです。「`pip` (パッケージインストーラ) + `venv` (仮想環境) + `requirements.txt` (依存関係ファイル)」という従来のワークフローを、よりシンプルで確実なものにすることを目指しています。

`pipenv` は主に2つのファイルを生成して依存関係を管理します。

* **`Pipfile`**: `requirements.txt` の代替。人間が読みやすい形式で、本番用 (`[packages]`) と開発用 (`[dev-packages]`) の依存関係を分けて記述します。
* **`Pipfile.lock`**: **決定論的ビルド**を実現するためのロックファイル。全てのパッケージ（サブ依存関係も含む）の正確なバージョンとハッシュ値を記録します。

(出自: `pipenv` パッケージ (PyPI経由でインストール))

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`pip` + `venv`** | これらはPythonに標準で組み込まれている低レベルなツールです。`pipenv` はこれらのツールを内部で利用し、仮想環境の作成・有効化とパッケージ管理を一つのインターフェイスで自動化します。 | `pipenv` (プロジェクト管理) |
| **`Poetry`** | `pipenv` と同様のモダンなPython依存関係管理ツール。依存関係解決のアルゴリズムがより高速で堅牢であると評価されることが多いです。 | `Poetry` (ライブラリ開発) |
| **`conda`** | Python以外のライブラリ（C/C++やRなど）も扱える、より強力な環境マネージャ。主にデータサイエンス分野で利用されます。 | - |

## よく連携されるコマンド

`pipenv` は、仮想環境内でスクリプトを実行するための `pipenv run` という形で、様々なコマンドと連携します。

### シナリオ例: cronジョブからスクリプトを実行する (インフラ構築・運用視点)

* **目的**: cronジョブやsystemdのサービスユニットから、アプリケーションを正しい依存関係の元で起動する。
* **連携コマンド**: `python`
* **解説**: `pipenv shell` で仮想環境に入ることなく、`pipenv run` を使うことで、正しい仮想環境内のPythonインタプリタでスクリプトを実行できます。
* **コマンド例 (`crontab -e` の記述)**:

		```bash
		# 毎時0分に、/path/to/project ディレクトリで my_task.py を実行

		0 * * * * cd /path/to/project && /usr/local/bin/pipenv run python my_task.py

		```

## オプション説明 (`pipenv` のサブコマンド)

| オプション | 説明 |
| :--- | :--- |
| **パッケージ管理** | |
| ⭐ `install [<package>]`| `Pipfile` に基づいて依存関係をインストール、または新しいパッケージを追加します。 |
| ⭐ `install --dev <package>`| `pytest` などの開発時にのみ必要なパッケージをインストールします。 |
| `uninstall <package>`| パッケージをアンインストールします。 |
| **環境操作** | |
| ⭐ `shell` | プロジェクト用に自動作成された仮想環境を有効化し、新しいシェルを起動します。 |
| ⭐ `run <command>` | 仮想環境内で `<command>` を実行します。 |
| **依存関係管理** | |
| `lock` | `Pipfile` を基に `Pipfile.lock` を生成・更新します。 |
| `graph` | インストールされているパッケージの依存関係ツリーを表示します。 |
| `check` | `Pipfile` に記載のパッケージに、既知のセキュリティ脆弱性がないかを検査します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: CI/CDパイプラインや本番環境で、開発環境と全く同じ依存関係を再現する。
* **組み合わせ**: `pipenv install --system --deploy`
* **解説**: `Pipfile.lock` が `Pipfile` と同期していることを厳密にチェックし、仮想環境ではなくシステムのPython環境（通常はコンテナ内）にパッケージをインストールします。
* **例 (Dockerfile内)**:

```Dockerfile
COPY Pipfile Pipfile.lock ./
RUN pip install pipenv && pipenv install --system --deploy
```

### 2. ブルーチーム視点

* **タスク**: プロジェクトの依存関係に既知の脆弱性がないかスキャンする。
* **組み合わせ**: `pipenv check`
* **解説**: CI/CDパイプラインにこのコマンドを組み込むことで、`Pipfile.lock` にリストされているパッケージに既知のセキュリティ脆弱性 (CVE) が含まれていないかを自動的に検査し、安全でないコードがデプロイされるのを防ぎます。
* **例**:

```bash
# プロジェクトの脆弱性をチェック
pipenv check
```

### 3. レッドチーム視点

* **タスク**: 侵入したホストで `Pipfile` を発見し、アプリケーションの技術スタックを偵察する。
* **組み合わせ**: `cat Pipfile`
* **解説**: 攻撃者は `Pipfile` の `[packages]` セクションを見るだけで、そのアプリケーションがどのWebフレームワーク（`django`, `flask`）、データベースクライアント（`psycopg2`）などを使用しているかを正確に把握できます。この情報を元に、特定のバージョンのライブラリに存在する既知の脆弱性を探して攻撃します。
* **例**:

```bash
# アプリケーションの依存関係を偵察
cat /var/www/my-app/Pipfile
```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `bash: pipenv: command not found`
		* **考えられる原因**: `pipenv` がインストールされていません。
		* **解決策**: `pip install --user pipenv` や `sudo apt install pipenv` などでインストールしてください。

2. **現象**: **`pipenv lock` または `pipenv install` が非常に遅い。**
		* **考えられる原因**: `pipenv` の依存関係解決プロセスは、複雑な依存ツリーを持つプロジェクトでは時間がかかることがあります。
		* **解決策**: まずは待ちます。解決しない場合は、`pipenv lock --clear` でキャッシュをクリアしたりすると改善することがあります。

## 環境変数と設定ファイル

* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

### 関連する設定ファイル

* **`Pipfile`**: プロジェクトの直接的な依存関係を定義します。
* **`Pipfile.lock`**: 全ての依存関係の正確なバージョンとハッシュ値を記録し、決定論的ビルドを保証します。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **サプライチェーン攻撃**。`pip` と同様に、PyPIにおけるタイポスクワッティングのリスクは `pipenv` にも存在します。
* **悪用シナリオ**: 悪意のあるパッケージをインストールしてしまうと、`pipenv lock` などのコマンド実行時に任意のコードが実行される可能性があります。

### GTFOBins / LOLBAS における利用例

`pipenv` は `pip` をラップするツールであり、`sudo pipenv` が許可されている場合、`pip` と同様の権限昇格が可能です。

* **機能**: **Privilege Escalation**
* **解説**: `sudo pipenv run` を使って、任意のコマンドをroot権限で実行できます。
* **参照**: `pip` のGTFOBinsエントリが参考になります: `https://gtfobins.github.io/gtfobins/pip/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: `pipenv check` をCI/CDパイプラインに組み込む。信頼できないパッケージはインストールしない。
* **Detection (検知)**: `Pipfile.lock` の変更をコードレビューで監査し、意図しないパッケージが追加されていないか確認する。

## 注意点・補足

* **`Pipfile.lock` の重要性**: `Pipfile.lock` は、依存関係のハッシュ値を記録することで、**中間者攻撃などによるパッケージの改ざんを防ぐ**重要な役割を果たします。`git` などでこのファイルを必ずバージョン管理し、`pipenv install --deploy` でlockファイルとの一致を強制することがセキュリティ上不可欠です。
