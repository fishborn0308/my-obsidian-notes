# Command - Linux - route - IPルーティングテーブルを管理する (旧)

## 概要

`route` は、LinuxカーネルのIPルーティングテーブルを表示・操作するためのコマンドです。どのネットワーク宛のパケットを、どのゲートウェイ（ルーター）を経由して送信するかという経路情報を管理します。

**⚠️ 重要**: `route` は `net-tools` パッケージに含まれる**レガシー（旧式）**なコマンドです。多くのモダンなLinuxディストリビューションではデフォルトでインストールされておらず、代わりに `iproute2` パッケージに含まれる **`ip route` コマンドの使用が強く推奨されます**。

(出自: `net-tools` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`ip route`** | `route` のモダンな後継。`ip route` は、ポリシーベースルーティングや複数のルーティングテーブルの管理など、遥かに高機能で柔軟な構文を持ちます。 | `ip route` |
| **`netstat -r`** | `route` コマンドとほぼ同じ形式でルーティングテーブルを表示するための旧世代のコマンドです。 | `ip route` |

## よく連携されるコマンド

`route` でルートを追加・変更した後は、`ping` や `traceroute` を使って、その経路が意図通りに機能しているかを確認するのが一般的です。

### シナリオ例: 静的ルート追加後の疎通確認 (インフラ構築・運用視点)

* **目的**: 別のネットワークセグメントへの静的ルートを追加した後、実際にそのネットワーク上のホストと通信できるかを確認する。
* **連携コマンド**: `ping`
* **解説**: `route add` で新しい経路を設定した後、その経路の先にあるホストに対して `ping` を実行し、双方向の通信が確立できているかをテストします。
* **コマンド例**:

		```bash
		# 1. 10.10.20.0/24 への静的ルートを追加

		sudo route add -net 10.10.20.0 netmask 255.255.255.0 gw 192.168.1.254

		# 2. 追加したルートの先にあるサーバー (10.10.20.10) への疎通を確認

		ping -c 4 10.10.20.10

		```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **表示** | |
| ⭐ `-n`, `--numeric` | ホスト名を解決せず、IPアドレスやポート番号を**数値**のまま表示します。 |
| **ルート操作** | |
| ⭐ `add` | 新しいルートを追加します。`-net` (ネットワーク宛) または `-host` (単一ホスト宛) を指定します。 |
| `del` | 既存のルートを削除します。 |
| **ルート指定** | |
| `gw <ip>` | ゲートウェイのIPアドレスを指定します。 |
| `netmask <netmask>`| ネットワークのネットマスクを指定します。 |
| `dev <interface>` | インターフェイスを指定します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 現在のルーティングテーブル（特にデフォルトゲートウェイ）を確認する。
* **組み合わせ**: `route -n`
* **解説**: サーバーが外部ネットワークと通信できない場合に、まずこのコマンドでデフォルトゲートウェイ (`Destination` が `0.0.0.0`) が正しく設定されているかを確認します。
* **例**:

		```bash
		# 現在のルーティングテーブルを数値表示で確認

		route -n

		```

## 2. ブルーチーム視点

* **タスク**: ルーティングテーブルが改ざんされていないかを確認する。
* **組み合わせ**: `route -n`
* **解説**: インシデント調査対象のホストで `route -n` を実行し、デフォルトゲートウェイが本来のものから、不審なIPアドレス（攻撃者のマシンなど）に変更されていないかを確認します。これは中間者攻撃の兆候である可能性があります。
* **例**:

		```bash
		# ゲートウェイが意図しないIPに変更されていないか監査する

		route -n

		```

## 3. レッドチーム視点

* **タスク**: 侵入後の偵察としてネットワークトポロジーを把握し、中間者攻撃を仕掛ける。
* **組み合わせ**: `route -n`, `sudo route add/del ...`
* **解説**: 攻撃者は `route -n` の出力からデフォルトゲートウェイや他の内部ネットワークの存在を把握し、横展開の計画を立てます。root権限を奪取した後は、デフォルトゲートウェイを自身が制御するマシンに変更し、通信を盗聴します。
* **例**:

		```bash
		# 既存のデフォルトゲートウェイを削除

		sudo route del default

		# 攻撃者のマシン (192.168.1.200) を新しいゲートウェイに設定

		sudo route add default gw 192.168.1.200

		```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `SIOCADDRT: Network is unreachable`
		* **考えられる原因**: ルートを追加しようとしましたが、`gw` で指定したゲートウェイが、どのインターフェイスからも直接到達できないネットワークにあります。
		* **解決策**: ゲートウェイのIPアドレスが正しいか、また、そのゲートウェイが所属するネットワークセグメントにホストが正しく接続されているかを確認してください。

## 環境変数と設定ファイル

* `route` の動作に直接影響する環境変数は通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

### 関連する設定ファイル

* **RHEL/CentOS**: `/etc/sysconfig/network-scripts/route-<iface>`
* **Debian/Ubuntu**: `/etc/network/interfaces` (`up route add ...` のように記述)

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `route` 自体ではなく、**ルーティングテーブルの改ざん**がリスクとなります。
* **悪用シナリオ**: root権限を持つ攻撃者は、ルーティングテーブルを自由に書き換えることができます。これにより、通信の盗聴（中間者攻撃）、特定のサーバーへの通信を妨害（ブラックホール化）することが可能になります。

### GTFOBins / LOLBAS における利用例

`route` はOSの構成を変更するコマンドであり、直接的なコード実行や権限昇格の機能を持たないため、GTFOBinsにはリストされていません。しかし、`sudo` 権限で実行できる場合、レッドチームのシナリオで述べたような攻撃に悪用されます。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 最小権限の原則に従い、一般ユーザーに `route` の `sudo` 権限を安易に与えない。
* **Detection (検知)**: `auditd` を使って、`netlink` 経由でのネットワーク構成の変更を監視する。`route -n` の出力を定期的に取得し、ベースラインと比較して意図しない変更がないかを確認する。

## 注意点・補足

* **設定の永続化**: `route` コマンドで行った設定変更は、OSを再起動すると失われます。設定を永続化させるには、ディストリビューション固有のネットワーク設定ファイルに記述する必要があります。
