# Command - git - git rebase - コミット履歴を整理・再構築する

## 概要

`git rebase` コマンドは、Gitの**コミット履歴を整理し、再構築する**ための強力なツールです。マージとは異なり、新しいコミットを作成するのではなく、現在のブランチの変更を別のブランチの最新のコミットの上に「再適用」することで、あたかも最初からそのブランチから開発を始めたかのような**線形の履歴**を作成します。これにより、コミット履歴がクリーンになり、読みやすくなります。

(出自: `git` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`git merge`** | `merge` はマージコミットを作成し、分岐・結合の履歴を明確に残します。`rebase` は履歴を**線形に書き換え**ます。**共有されたブランチには `merge` が安全**です。 | `merge` (共有ブランチ) |
| **`git commit --amend`** | 直前のコミットの内容を修正します。`git rebase -i` の `reword` や `squash` は、複数のコミットにわたって同様の操作が可能です。 | - |
| **`git cherry-pick`**| 他のブランチから**単一のコミット**だけを選択して適用します。 | - |

## よく連携されるコマンド

`git rebase` は、リモートの最新状態を取得する `git fetch` や、履歴を書き換えた後に強制的にプッシュする `git push --force-with-lease` と連携して使われます。

### シナリオ例: 機能ブランチをメインブランチの最新状態に更新する (インフラ構築・運用視点)

* **目的**: 開発中の機能ブランチを、最新の `main` ブランチの変更に基づいて更新し、履歴を線形に保つ。
* **連携コマンド**: `git fetch`, `git rebase`
* **解説**: まず `git fetch` でリモートの最新情報を取得します。その後、`git rebase` を使って、現在の機能ブランチのコミットを `main` の最新コミットの上に再適用します。
* **コマンド例**:

    ```bash
    # 1. リモートの最新情報を取得
    git fetch origin

    # 2. 現在のブランチを origin/main の上にリベース
    git rebase origin/main
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **基本操作** | |
| ⭐ `<base-commit>` | 現在のブランチのコミットを再適用する基準となるコミットやブランチを指定します。（例: `main`） |
| ⭐ `-i`, `--interactive` | **インタラクティブモード**でリベースを実行します。コミットの編集、統合 (`squash`)、削除 (`drop`) などが可能です。 |
| **競合解決** | |
| ⭐ `--abort` | リベース中に中断したい場合に、リベース開始前の状態に戻します。 |
| ⭐ `--continue` | コンフリクト解決後、停止していたリベース操作を再開します。 |
| `--skip` | 現在リベース中のコミットをスキップします（そのコミットの変更は破棄されます）。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 本番環境へのマージ前に、機能ブランチの不要なコミット（試行錯誤など）を1つにまとめる。
* **組み合わせ**: `git rebase -i main`
* **解説**: `-i` (`interactive`) を使って、`main` ブランチとの差分コミットを整理します。エディタが開き、`pick` を `squash` や `fixup` に変更してコミットをまとめ、クリーンな履歴を作成します。
* **例**:

    ```bash
    # mainブランチとの差分をインタラクティブにリベース
    git rebase -i main
    ```

### 2. ブルーチーム視点

* **タスク**: 過去のコミット履歴から、誤ってコミットされた機密情報（パスワードなど）を恒久的に削除する。
* **組み合わせ**: `git rebase -i <commit-hash>^`
* **解説**: **非常に危険な操作**ですが、`rebase -i` で問題のコミットを `edit` し、ファイルから機密情報を削除して `git commit --amend` を行い、`git rebase --continue` することで履歴を書き換えます。
* **例**: -

### 3. レッドチーム視点

* **タスク**: **痕跡を隠蔽**するため、自身の攻撃活動に関するコミット履歴を整理・偽装する。
* **組み合わせ**: `git rebase -i HEAD~N`
* **解説**: 複数の試行錯誤のコミットを1つのクリーンなコミットにまとめたり、`edit` や `reword` でコミットメッセージを変更したりして、監査ログとしての履歴の追跡を困難にします。
* **例**:

    ```bash
    # 直近の5つのコミットをインタラクティブに整理
    git rebase -i HEAD~5
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `error: could not apply <commit-hash>...` (コンフリクト発生)
    * **考えられる原因**: リベース中にコミットの再適用時に**コンフリクト（競合）**が発生しました。
    * **解決策**:
        1. `git status` で競合ファイルを確認し、手動で編集・解決します。
        2. `git add <file>` で解決したファイルをステージングします。
        3. `git rebase --continue` を実行してリベースを続行します。
        4. 中断したい場合は `git rebase --abort` を実行します。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **履歴改ざんの容易性**。
* **悪用シナリオ**: 攻撃者がコミット履歴を書き換えることで、以前に挿入したバックドアなどの痕跡を隠蔽しようとする可能性があります。

### GTFOBins / LOLBAS における利用例

`git rebase` は直接的なコード実行機能を持たないため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **強制プッシュ (`git push --force`) をブランチ保護ルールで禁止する**。リベースによる履歴の書き換えは、強制プッシュとセットでなければリモートに反映できないため、これが最も効果的な対策です。
* **Detection (検知)**: Gitホスティングサービスの監査ログを監視し、不審な強制プッシュがないかを確認する。

## 注意点・補足

* **共有ブランチでの使用は避ける**: **すでにリモートにプッシュされた共有ブランチ**に対して `rebase` を実行すると、他の開発者の履歴と衝突し、大きな混乱を引き起こします。`rebase` は、まだプッシュしていない**ローカルのプライベートなブランチ**を整理する場合にのみ使用すべきです。
