# Command - Linux - df - ファイルシステムのディスク空き容量を表示する

## 概要

`df` は、システム上の各ファイルシステムが、現在どの程度の容量を消費し、どの程度の空きがあるかを一覧表示します。物理ディスクだけでなく、仮想的なファイルシステムやネットワークマウント（NFS等）の情報も網羅的に確認できます。

(出自: `coreutils` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `du` | **ディレクトリ単位**の容量を計算する。どのファイルが重いかを探す用。 | 特定の巨大ファイル探し |
| `lsblk` | ブロックデバイスの物理的なツリー構造を表示する。容量より「繋がり」重視。 | ディスク構成の確認 |
| `fdisk -l` | パーティションテーブルの詳細を表示。ファイルシステム内蔵のデータは見ない。 | パーティション操作時 |
| `df` | **ファイルシステム単位**の全体像を表示。マウントポイントごとの空き状況に特化。 | **全体の空き容量確認** |

## よく連携されるコマンド

### シナリオ例: 容量不足の早期発見と特定 (運用視点)

* **目的**: 全体の空き容量を確認し、溢れそうな場合は原因のディレクトリを特定する。
* **連携コマンド**: `du`, `sort`
* **解説**: `df` で「どこが」溢れているかを確認し、`du` で「何が」原因かを探ります。
* **コマンド例**:
    ```bash
    # 1. まず全体を確認
    df -h
    # 2. 溢れているマウントポイント（例: /var）の中を調査
    sudo du -sh /var/* | sort -rh | head -n 10
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **表示制御** | |
| ⭐️ `-h`, `--human-readable` | 1K, 234M, 2G のように、読みやすい単位（1024単位）で表示する。 |
| `-H`, `--si` | 1000単位の計算で表示する（商用ストレージのカタログスペックに近い値）。 |
| ⭐️ `-T`, `--print-type` | ファイルシステムのタイプ（ext4, xfs, tmpfs 等）を表示する。 |
| ⭐️ `-i`, `--inodes` | 容量ではなく、**iノードの使用状況**を表示する（ファイル数が多すぎるトラブルに有効）。 |
| **フィルタリング** | |
| `-t <type>` | 指定したファイルシステムタイプのみ表示する。 |
| `-x <type>` | 指定したタイプを**除外**して表示する（tmpfs 等を除外したい時に便利）。 |
| `-a`, `--all` | 0ブロックのダミーファイルシステム（proc, sysfs等）も含めて全て表示。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: スクリプトによるディスク容量監視。
* **組み合わせ**: `--output` (GNU版のみ)
* **解説**: パーセンテージだけを抽出して、しきい値（例: 90%）を超えたらアラートを出す処理を自動化します。
* **例**:
    ```bash
    # ルートディレクトリの使用率だけを取得
    df --output=pcent / | tail -n 1 | tr -dc '0-9'
    ```

### 2. ブルーチーム視点

* **タスク**: 予期せぬ外部メディアやネットワークドライブのマウント検知。
* **組み合わせ**: `df -T`
* **解説**: `nfs` や `cifs` など、本来の構成にないファイルシステムがマウントされていないかチェックし、情報の持ち出し（Exfiltration）の経路を探ります。
* **例**:
    ```bash
    # 現在の全マウント状況とFSタイプを詳細に表示
    df -Th
    ```

### 3. レッドチーム視点

* **タスク**: 侵入後の内部偵察（書き込み可能なパーティションや重要データの保管場所特定）。
* **組み合わせ**: `df -h`
* **解説**: `/home`, `/var/www`, `/backup` など、データが溜まっていそうな場所や、マウントされている外部ストレージを特定します。
* **例**:
    ```bash
    # 全てのデバイスを人間が読める形式で表示
    df -h
    ```

## エラーメッセージとトラブルシューティング

1.  **「ファイルは消したのに `df` の空き容量が増えない」**:
    * **考えられる原因**: 削除したファイルを、まだどこかのプロセスが「開いたまま (Opened)」にしている。
    * **解決策**: `lsof | grep deleted` で犯人のプロセスを特定し、再起動または kill する。

2.  **「容量には空きがあるのに、ファイルの作成に失敗する (No space left on device)」**:
    * **考えられる原因**: **iノードの枯渇**。小さなファイルが大量にあり、管理領域がいっぱいになっている。
    * **解決策**: `df -i` で確認。不要なファイルを削除するか、iノードを増やして再フォーマット（通常は削除）。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **悪用シナリオ**: 攻撃者が `/tmp` や `/var/tmp` に巨大なファイルを書き込み続け、ディスクを 100% にすることでシステムをクラッシュ（DoS）させる。この兆候を `df` の定期監視で捉える必要があります。

### GTFOBins における利用例

`df` 自体はファイルを読み書きする機能がないため、GTFOBins には登録されていません。しかし、特権で `df` を実行すると、未マウントのボリュームや、通常ユーザーからは見えない特殊なマウントポイントの存在が露呈する可能性があります。

## 注意点・補足

* **単位の混同**: デフォルトは 1KB 単位ですが、POSIX標準では 512B 単位になることもあるため、常に `-h` を使うのが安全です。
* **リモートマウントの罠**: NFSサーバがダウンしているときに `df` を実行すると、応答待ちでコマンドがフリーズすることがあります。
