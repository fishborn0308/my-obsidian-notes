# Command - Linux - rsync - ファイルやディレクトリの高速な差分同期

## 概要

`rsync` は、ローカルまたはリモート（ネットワーク経由）間でファイルやディレクトリを同期します。独自の「デルタ転送アルゴリズム」により、変更された部分のみを送信するため、2回目以降の転送が劇的に速いのが特徴です。また、権限、所有者、タイムスタンプを維持したままコピー（ミラーリング）する能力に長けています。

## 🛠 ファイル転送ツールの比較

| コマンド | 特徴 | 弱点 |
| :--- | :--- | :--- |
| `cp` | ローカル専用、シンプル。 | リモート不可、差分転送なし。 |
| `scp` | SSH経由の暗号化転送。 | 常に全件転送。中断後の再開に弱い。 |
| **`rsync`** | **差分転送、属性維持、圧縮。** | オプションが多く、初見では複雑。 |

## 主要なオプション（鉄板の組み合わせ）

`rsync` は **`-avz`** の 3 つをセットで使うのが基本中の基本です。

| オプション | 説明 |
| :--- | :--- |
| ⭐️ `-a`, `--archive` | **超重要。** 階層構造、所有者、権限、タイムスタンプをすべて維持（再帰的）。 |
| ⭐️ `-v`, `--verbose` | 実行中に転送ファイル名を表示する。 |
| ⭐️ `-z`, `--compress` | 転送データを圧縮して帯域を節約する。 |
| ⭐️ `--delete` | **要注意。** 送信元にないファイルを送信先から削除する（完全同期）。 |
| ⭐️ `-n`, `--dry-run` | **必須。** 実際にはコピーせず、「何が起きるか」のシミュレーションのみ行う。 |
| ⭐️ `-e ssh` | 通信経路を SSH で暗号化する（現代ではデフォルトに近い）。 |
| `--partial` | 転送が中断されても、途中まで送ったファイルを残す（再開が速くなる）。 |

## ⚠️ 初心者が必ずハマる「スラッシュ `/`」の罠

`rsync` では、送り主のパスの最後に `/` を付けるかどうかで挙動が激変します。

* `rsync -a /src /dest` → `/dest/**src**/` というフォルダが作られる。
* `rsync -a /src/ /dest` → `/src/` の**中身だけ**が `/dest/` に展開される。

## 業務シナリオ別活用法

### 1. インフラ構築・運用視点

* **タスク**: サーバーのバックアップとミラーリング。
* **解説**: 定期的に実行することで、データのバックアップを最小限の負荷で作成します。
* **例**:
    ```bash
    # /data を /backup に完全同期（送信元にないものは削除）
    # 怖いのでまずは --dry-run で確認
    rsync -avz --delete --dry-run /data/ /backup/
    ```

### 2. ブルーチーム視点

* **タスク**: 証跡（ログ）のセキュアな収集。
* **解説**: 複数のサーバーからログをログ集約サーバーへ転送します。SSH 経由 (`-e ssh`) で実行することで、盗聴のリスクを排除します。

### 3. レッドチーム視点

* **タスク**: データの大量持ち出し (Exfiltration)。
* **解説**: 侵入後、機密データを外部の C2 サーバーへ送る際に使用します。`rsync` は帯域制限 (`--bwlimit`) が可能なため、管理者に「不自然なネットワーク負荷」を気づかれないよう、少しずつデータを盗み出すのに適しています。

## セキュリティに関する考慮事項

### 脆弱性とリスク

* **rsync デーモンモード**: `rsync --daemon` で立ち上げている場合、認証設定が甘いと認証なしでファイルが読み書きされてしまう深刻な脆弱性になります。現代では特別な理由がない限り SSH 経由で使うべきです。
* **意図しない削除**: `--delete` を指定した状態でパスを間違えると、バックアップ先のデータをすべて吹き飛ばすリスクがあります。

### 対応策・緩和策 (ブルーチーム視点)

* **SSH 鍵の制限**: `authorized_keys` 内で `command="rsync ..."` を指定することで、その鍵を使って実行できるコマンドを `rsync` のみに制限（制限付きシェル）します。
