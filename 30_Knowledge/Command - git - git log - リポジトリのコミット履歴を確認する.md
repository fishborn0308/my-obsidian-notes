# Command - git - git log - リポジトリのコミット履歴を確認する

## 概要

`git log`コマンドは、Gitリポジトリのコミット履歴を新しい順に表示します。これにより、プロジェクトの変更履歴、誰がいつ何を変更したのか、どのブランチで変更が行われたのかといった詳細を確認できます。

(出自: `git` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`git status`** | 現在の「作業状態」を示します。`git log` は過去の「コミット履歴」を表示します。 | - |
| **`git diff`** | ファイルやコミット間の**具体的な変更内容（差分）**を表示します。`git log` でコミットを特定し、`git diff` でその詳細を見る、という連携で使われます。 | - |
| **`git reflog`** | `HEAD` の移動履歴を記録します。`git log` では見えなくなったコミット（`reset`で削除したものなど）も追跡でき、**失われたコミットを復元する**際に強力です。 | `reflog` (復元時) |
| **`git show`** | **単一のコミット**の詳細情報と変更内容（差分）を表示します。`git log -p` に似ていますが、一つのコミットに焦点を当てます。 | - |

## よく連携されるコマンド

`git log` の出力はパイプ (`|`) を通じて `grep` に渡され、特定のキーワードを含むコミットを検索する際によく連携されます。

### シナリオ例: 特定のファイルに対する変更履歴を詳細に確認する (インフラ構築・運用視点)

* **目的**: 設定ファイル (`nginx.conf`) がいつ、誰によって、どのように変更されたかを、コミットメッセージと実際の差分と共に確認する。
* **連携コマンド**: `git log` (`-p` オプション)
* **解説**: `-p` (`--patch`) オプションと対象ファイルのパスを指定することで、そのファイルに対するコミットの差分も同時に表示されます。
* **コマンド例**:

    ```bash
    # path/to/nginx.conf ファイルに加えられた変更の履歴と差分を表示
    git log -p -- path/to/nginx.conf
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **表示形式** | |
| ⭐ `-p`, `--patch`| 各コミットの差分(変更内容)も表示します。 |
| ⭐ `--oneline`| 各コミットを1行で簡潔に表示します。 |
| `--stat`| 各コミットで変更されたファイルの統計情報を表示します。 |
| `--graph`| コミット履歴をASCIIアートのグラフ形式で表示します。 |
| **フィルタリング** | |
| ⭐ `-n <count>` | 最新の `<count>` 個のコミットのみを表示します。 |
| ⭐ `--author="<pattern>"` | 指定した作者のコミットのみを表示します。 |
| `--grep="<pattern>"`| コミットメッセージが指定したパターンにマッチするコミットのみを表示します。 |
| ⭐ `-S"<string>"`| 指定した文字列がコードの**追加または削除**で変更されたコミットのみを表示します（ピックアックス検索）。 |
| `--since`, `--until`| 指定した期間内のコミットのみを表示します。（例: `--since="2 weeks ago"`） |
| `<path>...`| 指定したファイルまたはディレクトリに関連するコミットのみを表示します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: リポジトリのブランチ戦略やリリース履歴の全体像を迅速に把握する。
* **組み合わせ**: `git log --oneline --graph --decorate --all`
* **解説**: 全ブランチの履歴を1行形式で、グラフとブランチ名・タグ名付きで表示します。緊急パッチ適用後の履歴確認などに有効です。
* **例**: -

### 2. ブルーチーム視点

* **タスク**: Webアプリケーションの脆弱性調査で、バックドアがいつ、どのようにしてコードベースに混入したかを特定する。
* **組み合わせ**: `git log -p -S"eval(" --all`
* **解説**: `-S` オプションで、`eval()` のような危険な関数が追加または削除された全てのコミットとその差分を表示し、悪意のあるコードの混入を追跡します。
* **例**:

    ```bash
    # コミットメッセージに特定のCVE IDが含まれるものを検索
    git log --grep="CVE-2023-XXXX"
    ```

### 3. レッドチーム視点

* **タスク**: 標的の公開リポジトリから、過去に誤ってコミットされた機密情報（APIキーなど）を探す。
* **組み合わせ**: `git log -p --reverse -- path/to/sensitive/config.yml`
* **解説**: `--reverse` で古い順に表示し、初期のセキュリティ対策が不十分だったコミットを狙って、機密設定ファイルの変更履歴を差分付きで精査します。
* **例**:

    ```bash
    # 全コミットメッセージから "password" という文字列を検索
    git log --all | grep -i "password"
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `fatal: your current branch 'master' does not have any commits yet`
    * **考えられる原因**: `git init` したばかりで、まだ一度もコミットが行われていません。
    * **解決策**: `git add` と `git commit` で最初のコミットを作成してください。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **情報漏洩**。
* **悪用シナリオ**: 公開リポジトリの履歴に、過去に誤ってコミットされたAPIキー、パスワード、秘密鍵などが含まれている場合、攻撃者は `git log -p` や `-S` を使ってこれらを容易に抽出できます。**すでに削除されたファイルや修正された内容であっても、履歴に残っている限り参照可能です**。

### GTFOBins / LOLBAS における利用例

`git log` は情報表示に特化したコマンドのため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **コミット前の厳格なレビュー**。機密情報がコミットされないようにコードレビュープロセスを強化する。**`.gitignore` の徹底**。
* **Detection (検知)**: CI/CDパイプラインに `TruffleHog` などのシークレットスキャンツールを導入し、コミット履歴を定期的にスキャンする。

## 注意点・補足

* **出力のページング**: `git log` の出力は、デフォルトで `less` コマンドにパイプされます。`q` キーで終了できます。
