# Command  - Linux - ps - 実行中のプロセスを表示する

## 概要

`ps` (process status) は、コマンドを実行した**時点**でシステム上で実行されているプロセスに関する情報の**スナップショット**を表示するコマンドです。「どのようなプロセスが動いているか？」「プロセスのID (PID) は何か？」といった情報を確認するための、基本的なシステム管理ツールです。`ps` にはオプションの指定方法に歴史的な経緯から複数のスタイル（BSD形式とSystem V形式）があります。

(出自: `procps-ng` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`top` / `htop`** | **リアルタイム**で更新され続ける、対話的なプロセスビューアです。`ps` は実行した瞬間の**静的な**一覧を表示します。 | `htop` (対話的監視) |
| **`pstree`** | プロセスの親子関係を**ツリー形式**で視覚的に分かりやすく表示します。 | - |
| **`pgrep` / `pkill`**| プロセス名などの条件でプロセスを**検索** (`pgrep`) したり、検索したプロセスにシグナルを**送信** (`pkill`) したりすることに特化しています。 | `pgrep` (PID検索) |

## よく連携されるコマンド

`ps` の出力は非常に多いため、`grep` と連携して特定のプロセス情報をフィルタリングするのが最も一般的な使い方です。

### シナリオ例: 特定のプロセスが実行中かを確認する (インフラ構築・運用視点)

* **目的**: システムで実行中の全プロセスの中から、`nginx` という名前のプロセスだけを表示し、そのPIDを特定する。
* **連携コマンド**: `grep`
* **解説**: `ps aux` で全プロセスのリストを出力し、その結果をパイプで `grep` に渡して `nginx` という文字列を含む行だけをフィルタリングします。
* **コマンド例**:

```bash
# psコマンドの出力からnginxプロセスだけをフィルタリング
ps aux | grep 'nginx'
```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **BSD形式 (ハイフンなし)** | |
| ⭐ `aux` | `a`: 全ユーザーのプロセス, `u`: ユーザー指向フォーマット, `x`: 端末に紐付かないプロセス。**最もよく使われる組み合わせ**。 |
| `ww` | `aux` と併用し、コマンドライン引数の表示幅を広げ、省略されないようにします。 |
| **System V形式 (ハイフンあり)** | |
| ⭐ `-ef` | `-e`: 全てのプロセス, `-f`: フルフォーマット。`aux` と並んでよく使われます。 |
| `--forest` | プロセスの親子関係をツリー形式（アスキーアート）で表示します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: サービスが実行中かを確認し、そのPIDを取得する。
* **組み合わせ**: `ps aux | grep <process_name>`
* **解説**: サービスを起動した後、このコマンドでプロセスが正しく起動しているかを確認します。応答しなくなったプロセスを `kill` するためにPIDを特定するのにも使います。
* **例**:

```bash
# httpdプロセスのPIDを調べる
ps aux | grep '[h]ttpd'
# -> [h]ttpd のようにブラケットを使うと、grep自身のプロセスが結果に含まれるのを防げる
```

### 2. ブルーチーム視点

* **タスク**: 侵害されたホスト上で、悪意のあるプロセスがいないかをライブで調査する。
* **組み合わせ**: `ps auxww`
* **解説**: **ライブフォレンジックの基本**。`ww` でコマンドラインの引数が**省略されないように**表示し、不審な実行パスや引数がないかを確認します。
* **例**:

```bash
# 全プロセスのフルコマンドラインを表示し、/tmp から実行されているものがないか確認
ps auxww | grep "/tmp/"
```

### 3. レッドチーム視点

* **タスク**: 侵入後の偵察として、システムで何が稼働しているかを完全に把握する。
* **組み合わせ**: `ps aux`
* **解説**: 攻撃者はシステム侵入後、即座に `ps aux` を実行し、EDRのエージェントやアンチウイルス、稼働中のサービス、他のユーザーセッションなどを確認し、次の攻撃計画を立てます。
* **例**:

```bash
# 実行中のプロセス一覧からセキュリティ製品を探す
ps aux | grep -i "sentinel\|falcon\|defender"
```

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **深刻な情報漏洩**。`ps` の出力、特にフルフォーマットや `ww` 付きの出力は、**コマンドライン引数**を明らかにします。
* **悪用シナリオ**: `mysql -u admin -p'MyS3cr3tP@ssw0rd'` のように、パスワードやAPIキーを引数に直接渡してスクリプトを実行している場合、`ps` コマンドを実行できる全てのユーザーにその機密情報が閲覧可能になってしまいます。

### GTFOBins / LOLBAS における利用例

* **機能**: **Privilege Escalation** (権限昇格)
* **解説**: もし `ps` コマンドにSUIDビットが設定されている（重大な設定ミス）場合、特定の環境変数を操作して `ps` がページャ（`less`など）を呼び出すように仕向け、ページャのシェルエスケープ機能を使ってroot権限を奪取する、といった高度な攻撃が可能です。
* **参照**: `https://gtfobins.github.io/gtfobins/ps/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **パスワードなどの機密情報をコマンドライン引数で渡さない**という開発プラクティスを徹底する。
* **Detection (検知)**: `auditd` で `ps` コマンドの実行を監視する。EDRは通常、不審な引数を持つプロセスの実行を検知します。

## 注意点・補足

* **`ps aux` vs `ps -ef`**: どちらも「全プロセスを表示する」という目的は同じですが、表示される情報が異なります。`aux` はCPU/メモリ使用率などリソース中心、`-ef` はUID/PPIDなどプロセス関係中心の情報を提供します。どちらかに慣れて、一貫して使うのが良いでしょう。
