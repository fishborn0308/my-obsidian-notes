# Command - Linux - dpkg - Debian パッケージマネージャの低レベルツール

## 概要
`dpkg` は、Debian 形式（.deb）のパッケージをインストール、ビルド、削除、管理するための低レベルのツールです。`apt` と異なり、依存関係の自動解決（インターネットからの不足パッケージのダウンロード）は行いませんが、システムにインストールされたパッケージの詳細な状態を直接操作・照会できます。
(出自: Debian プロジェクトによって開発)



## 類似・関連コマンド
| コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `apt / apt-get` | 高レベルツール。リポジトリからのダウンロードや依存関係の解決を自動で行う。 | **通常のパッケージ管理** |
| `gdebi` | ローカルの .deb をインストールする際、依存関係も同時に解決してくれるツール。 | ローカルファイルのインストール |
| `rpm` | RHEL/CentOS 系における `dpkg` に相当するツール。 | Red Hat 系 OS |

## よく連携されるコマンド
### シナリオ例: インストール済みファイルの整合性確認 (ブルーチーム視点)
* **目的**: システムファイルが改ざんされていないか、パッケージインストール時の状態と比較する。
* **連携コマンド**: `grep`
* **解説**: `-V` (verify) オプションを使い、MD5チェックサムの不一致を確認します。
* **コマンド例**:
    ```bash
    # 全パッケージの整合性をチェックし、変更があったファイルを表示
    sudo dpkg --verify
    ```

## オプション説明
※インストールや削除には **root 権限** が必須です。

| オプション | 説明 |
| :--- | :--- |
| **インストール・削除** | |
| ⭐️ `-i`, `--install <.deb>` | 指定したパッケージファイルをインストールする。 |
| ⭐️ `-r`, `--remove <name>` | パッケージを削除する（設定ファイルは残る）。 |
| ⭐️ `-P`, `--purge <name>` | 設定ファイルも含めてパッケージを完全に削除する。 |
| **情報照会** | |
| ⭐️ `-l`, `--list [<pattern>]` | インストール済みパッケージの一覧を表示する。 |
| ⭐️ `-L`, `--listfiles <name>` | 特定のパッケージによってインストールされたファイル一覧を表示する。 |
| ⭐️ `-s`, `--status <name>` | パッケージの詳細なステータス（バージョン、依存関係等）を表示する。 |
| ⭐️ `-S`, `--search <pattern>` | 指定したファイルがどのパッケージに含まれているか検索する。 |
| `-I`, `--info <.deb>` | 未インストールの .deb ファイルの情報を表示する。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点
* **タスク**: ネットワーク不通環境（エアギャップ環境）でのツール導入。
* **組み合わせ**: `dpkg -i`
* **解説**: 事前にダウンロードしておいた `.deb` ファイルを直接インストールします。
* **例**:
    ```bash
    # ローカルのパッケージをインストール
    sudo dpkg -i google-chrome-stable_current_amd64.deb
    # もし依存関係エラーが出た場合は、後で apt を使い修復
    sudo apt  --fix-broken install
    ```

### 2. ブルーチーム視点
* **タスク**: 不審な実行ファイルの出所特定。
* **組み合わせ**: `dpkg -S`
* **解説**: 調査中に見つけたバイナリが、公式のパッケージ由来のものか、それとも攻撃者が勝手に設置したものかを判別します。
* **例**:
    ```bash
    # /usr/bin/ls がどのパッケージ由来か確認
    dpkg -S /usr/bin/ls
    # 結果: coreutils: /usr/bin/ls (正常)
    ```

### 3. レッドチーム視点
* **タスク**: インストール済みソフトウェアの脆弱性調査。
* **組み合わせ**: `dpkg -l`
* **解説**: ターゲット環境にインストールされている全パッケージとそのバージョンを取得し、既知の脆弱性（CVE）がないか照合します。
* **例**:
    ```bash
    # インストール済みパッケージ一覧をファイルに保存して持ち出し
    dpkg -l > installed_packages.txt
    ```

## エラーメッセージとトラブルシューティング

1.  **エラーメッセージ例 1**: `dpkg: error: dpkg status database is locked by another process`
    * **原因**: `apt` や他の `dpkg` プロセスが実行中であり、データベースをロックしている。
    * **解決策**: 他の実行中プロセスが終わるのを待つか、異常終了している場合はロックファイルを削除する（※慎重に）。

2.  **エラーメッセージ例 2**: `dependency problems - leaving unconfigured`
    * **原因**: インストールしようとしているパッケージが必要とする別のパッケージが足りない。
    * **解決策**: `sudo apt --fix-broken install` を実行して依存関係を自動解決させる。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例
* **悪意あるパッケージ**: 攻撃者が `postinst`（インストール後に実行されるスクリプト）にバックドアを仕込んだ `.deb` ファイルを「便利なツール」として配布し、ユーザーが `dpkg -i` を実行した瞬間に root 権限でコードを実行させる。
* **情報の露出**: `dpkg -l` にはシステム構成が克明に記録されており、攻撃者にとっての「攻略マップ」になります。

### GTFOBins における利用例
`dpkg` に `sudo` 権限がある場合、対話的なシェルを起動したり、任意のファイルを読み書きすることが可能です。
* **機能**: `Sudo`, `Interactive Shell`
* **コマンド例**:
    ```bash
    # sudo dpkg を使って root シェルを取得（ページャー経由で !/bin/sh）
    sudo dpkg -l
    # (出力が多い場合にページャーが起動したら) !/bin/sh
    ```
* **参照**: `https://gtfobins.github.io/gtfobins/dpkg/`

### 対応策・緩和策 (ブルーチーム視点)
* **Prevention (予防)**: 公式リポジトリ以外の `.deb` ファイルを安易にインストールしない。
* **Detection (検知)**: `dpkg` による不自然なパッケージインストールを `auditd` や SIEM で監視する。

## 注意点・補足
* **ステータス文字**: `dpkg -l` の左端にある 2文字（例: `ii`）には意味があります。
    * 1文字目: 希望する状態（`i`=install）
    * 2文字目: 現在の状態（`i`=installed, `c`=config-files only）
