# Sec504 day1

## 1. ケーススタディの概要

-   **被害組織**: アルゴス社（政府向けチップ製造メーカー）。
-   **攻撃者**: 「グリーン・ペンギン」（民間企業を装った国家支援のAPTグループ）。
-   **発覚**: 火曜午前9時、CRMサーバー（顧客管理システム）で不審なプロセスが2つ発見される。

## 2. 攻撃の手口（キルチェーン）

攻撃者は、組織の防御の隙を突いて段階的に深部へ侵入しています。

1.  **偵察と侵入**: ポートスキャンで公開CRMサーバーを発見。Webアプリの既知の脆弱性（SSRF、コマンドインジェクション）を悪用して侵入。
2.  **永続化**: 性質の異なる2つのマルウェア（バックドアとビーコン通信型）を設置。片方が消されても操作を継続可能にする。
3.  **横展開（ピボット）**: CRMサーバーが「インターネット」と「社内LAN」の両方に接続（デュアルホーム）されていたため、そこを踏み台に内部へ侵入。
4.  **権限昇格**: パスワードハッシュをダンプ・解析。使い回されていたパスワードを利用し、データベースやドメインコントローラー（DC）へアクセス。

## 3. インシデント対応における「11のミス（NMS）」

現場で陥りがちな失敗が具体的に示されています。特筆すべきは以下の点です。

-   **表層的な対応**: 不審なプロセスを「強制終了」しただけで、侵入経路の調査や他の端末への影響確認を行わなかった。
-   **監視の不備**: ファイアウォールのログ無視、脅威情報の未活用、内部ネットワークの監視不足。
-   **衛生管理の欠如**: 脆弱性の放置（コスト優先）、パスワードの使い回し、弱いパスワード。
-   **スコーピングの失敗**: 感染範囲の特定（スコーピング）を怠り、DCまで汚染されていることに気づかなかった。

## 4. インシデント対応のプロセスと課題

-   **PICERL（ピクラル）**: SANSが提唱する世界的に有名な6段階プロセス（準備、識別、封じ込め、根絶、復旧、教訓）。
-   **プロセスの誤解**: 実際は直線的ではなく、**無限ループ**のように準備段階へ戻る動的なプロセスが必要。
-   **根本原因**: ログがないことによる「可視化の欠如」や、教訓を次に活かさない体質が不完全な封じ込めを招く。

## 5. 重要な用語の整理

| **用語** | **説明** |
| --- | --- |
| **APT** | 国家支援などを受ける高度で執拗な攻撃者。 |
| **アトリビューション** | 攻撃の主体（犯人）を特定すること。 |
| **ピボット** | 侵入した端末を踏み台にして、内部ネットワークへ移動すること。 |
| **IOC** | 侵害指標。マルウェアの特徴や攻撃者のIPなど、侵害の痕跡。 |
| **スレッドハンティング** | 侵害が発覚する前に、IOC等を用いて能動的に脅威を探すこと。 |

## 6. 動的なインシデント対応（DAIR）の考え方

従来の直線的な「PICERL（ピクラル）」モデルではなく、より現実的で動的な「DAIR」が推奨されています。

-   **トリアージ**: 戦場医療の概念を応用。リソースが限られているため、最も被害の大きい端末や重要な資産を優先して対応する。
-   **意思決定**: 現場の判断だけでなく、権限を持つ意思決定者と相談しながら、ケースバイケースで「封じ込め」や「復旧」の順序を決める。
-   **ゴール設定**: 最終的な「アウトカム（復旧、犯人逮捕など）」を決め、そこから逆算してアクティビティを計画する。

## 7. 準備段階（最重要フェーズ）

インシデントが発生する前の「備え」が成否を分けます。

-   **資産の特定**: 何がビジネスにとって最重要か（宝の持ち腐れにならないよう）を経営層と明確にする。
-   **ログ設定の不備**: Windowsのデフォルト設定では不十分（プロセス作成ログがない等）。正しいログ設定を自動化ツールなどで導入すべき。
-   **ルートキット対策**: OSやハードウェア（ファームウェア）まで感染する「ルートキット」の場合、駆除は困難。ハードウェアの処分や再構築が最も安全な対策となる。
-   **オフラインバックアップ**: 最近のランサムウェアはオンラインバックアップも暗号化するため、物理的に切り離された（オフライン）バックアップが不可欠。

## 8. 検知とスコーピング

-   **検知のレイヤー**: ネットワーク（IDS/IPS）、ホスト（イベントログ）、第三者からの通報など。内部で先に気づく体制が理想。
-   **スコーピング（被害範囲の特定）**:
		-   **「マルウェアはゴキブリと同じ」**: 1台見つけたら他にも潜んでいると考えるべき。
		-   **IOC（侵害指標）**: YARA（ファイル/メモリ）、Sigma（ログ）、Snort（ネットワーク）といった世界標準のルールを用いて全社をスキャンする。
-   **注目ツール**: **Velociraptor（ベロキラプトル）**。エージェント型で、全端末から迅速にフォレンジック調査やスキャンができる強力なオープンソースツール。

## 9. 封じ込め（Containment）

状況に応じて「すぐ止めるか、泳がせるか」を判断します。

-   **即時封じ込め**: ランサムウェアのような自己拡散型（ワーム型）は、一刻も早くネットワーク遮断やプロセス停止を行う。
-   **Watch and Learn（静観）**: 高度なAPT攻撃の場合、すぐに手を出すと攻撃者に気づかれ、証拠隠滅やさらなる攻撃を招く。全容解明（スコーピング）を待ってから一斉に駆除するのが理想。
-   **注意点**: シャットダウンするとメモリ上の証拠が消えるため、ネットワーク隔離にとどめて「メモリーダンプ」を優先する。

## 10. 根絶と復旧

-   **根絶**: マルウェア削除、バックアップからの復元、バックドアアカウントの削除。
-   **脆弱性診断**: 侵入口となった脆弱性以外にも穴があるはずなので、再発防止のためにペネトレーションテスト等を実施する。

### 10.1. 今回追加された重要キーワード

| **用語** | **内容** |
| --- | --- |
| **トリアージ** | 限られた人員で対応優先順位を決めること。 |
| **Velociraptor** | 高度な調査を可能にするオープンソースのIR/フォレンジックツール。 |
| **オフラインバックアップ** | 物理的に切断されたバックアップ。ランサムウェア対策の要。 |
| **ルートキット** | OSの深部に潜む、検知困難な高度なマルウェア。 |
| **DNSシンコーリング** | 悪意あるドメインへの接続を、安全な偽のIPに誘導して遮断する技術。 |

## 11. 復旧（Recovery）と改善（Remediation）

インシデント後の「元の状態に戻す」作業と「二度と起こさないための強化」についてです。

-   **復旧の戦略**: 理想はシステムの再構築。即時再構築が難しい場合は、IP制限などの「時間稼ぎ」を行いながら新サーバーを構築する。監視が容易な夜間や週末の作業が推奨される。
-   **改善（リメディエーション）**: 根本原因を解消すること。
		-   **人を責めない**: 弱いパスワードを設定した個人を責めるのではなく、それを許容した「ポリシー」を改善する。
		-   **継続的な監視**: 攻撃者は一度侵入した組織を「弱い」と見なし、3ヶ月〜半年は再攻撃を仕掛けてくる。作成したIOCを活用し、Velociraptor等で自動監視を続ける。
-   **事後報告（レポート作成）**: 経営層向け（ビジネス言語）と技術者向けの2部構成にする。予算獲得のチャンスだが、対策の導入には現場の抵抗も多いため、数ヶ月後のレビューミーティングで定着を確認する。

## 12. ライブ調査（Live Response）の基礎

証拠保全（フルフォレンジック）の前に、コストを抑えて「火事か、餃子を焼いている煙か」を判断する調査です。

-   **PowerShellの活用**: Windows標準の強力なツール。
		-   **オブジェクト指向**: Linux（テキスト転送）と違い、データを「オブジェクト」として受け渡すため、特定の項目（プロセス名など）を抽出するのが非常に容易。
		-   **リモート調査**: 資格情報があれば、自席からリモート端末に対してコマンドを実行できる。
-   **WMI / CIM**: `Get-CimInstance` を使い、Windowsの深い管理情報（Win32_Processなど）にアクセスする。これにより、標準のコマンドでは見えない「コマンドライン引数」や「親プロセスID」を調査できる。

## 13. 「悪」を見つけるプロセスのチェックポイント

異常なプロセスを見抜くための「Find Evil」の視点です。

-   **親子関係の矛盾**:
		-   例：`lsass.exe`（認証プロセス）は通常、子プロセスを起動しない。もしここから `cmd.exe` が動いていれば、ほぼ100%侵害されている。
-   **実行環境の異常**:
		-   ランダムなファイル名。
		-   異常なパス（Tempフォルダなど）からの起動。
		-   Base64で難読化されたコマンドライン引数。
		-   デジタル署名やメタデータの欠如。
		-   現代の64bit環境における不自然な32bitプロセス。

## 14. 便利な調査ツール

-   **CyberChef**: イギリスのGCHQ（政府通信本部）が開発。Base64のデコードや難読化解除をドラッグ＆ドロップで行える「サイバー界の多機能ナイフ」。
-   **SANS Find Evil ポスター**: Windowsシステムプロセスの「あるべき姿」がまとめられたリファレンス。

```PowerShell
# 親プロセスIDと起動コマンドラインを確認する（不審な実行の特定）
Get-CimInstance Win32_Process | Select-Object ProcessId, Name, ParentProcessId, CommandLine
```

## 15. ネットワーク調査（ネットワーク・フォレンジック）

パケットキャプチャ（PCAP）やログから、攻撃者の足跡を辿ります。

-   **3つの課題**:
		-   **アクセス**: 安価な機器ではログを書き出す機能がない。
		-   **確信性**: 重要なデータが保存されない。
		-   **可視性**: デフォルトで通信が暗号化（HTTPS）されているため、中身が見えない。
-   **メタデータの重要性**: 中身が見えなくても、送信元/先のIP・ポート・通信量などの「メタデータ」だけで、C2通信やデータ流出の推測は可能です。

### 15.1. 主要ツール

-   **TCPDump**: 基本的なパケットキャプチャツール。
		-   **Tips**: 調査時は `-n` オプション（名前解決をしない）が必須。攻撃者のDNSに接続してしまい、調査がバレるのを防ぐためです。
-   **T-Shark**: Wiresharkのコマンドライン版。巨大なPCAPファイルを解析する際、GUIのようにクラッシュせず、高度なフィルタリングが可能です。

## 16. Webプロキシ（Squid等）の解析

暗号化通信が主流の現代において、内部からの通信を監視する要となります。

-   **TLSの複合**: 社内プロキシを経由させ、ルート証明書を配布することで通信を複合し、中身を検査します。
-   **Squidログの注意点**: タイムスタンプが「UNIX時間」であるため、解析には人間が読める形式への変換が必要です。
-   **ビーコン通信の特定**: 一定の間隔で外部（C2サーバー）へ送られる「ビーコン通信」を特定することが、APT攻撃発見の鍵となります。

## 17. PowerShellによる高度なライブ調査

システム標準機能をフル活用した、攻撃者の「永続化」手法の特定方法です。

-   **レジストリ操作**: `Get-ChildItem` 等を使い、レジストリをファイルシステムのように操作して、自動起動設定に潜むマルウェアを探します。
-   **スケジュール済みタスク**: `Get-ScheduledTask` で確認。特に、ファイルを作成しない「ファイルレス・マルウェア」がPowerShellの `IEX`（Invoke-Expression）コマンドを悪用してメモリ上で動くケースを警戒すべきです。
-   **差分解析（ベースライン比較）**:
		-   **正常時の状態（ベースライン）**をテキスト保存しておき、有事の**現在（Current）**と比較することで、追加された不審なサービスや停止されたセキュリティ機能を瞬時に特定します。

## 18. インシデント対応の「自動化」と「継続」

-   **Sysinternalsツールの活用**: `Process Explorer`（タスクマネージャーの強化版）や `Autoruns`（永続化設定の一括確認）など、Microsoft公式の強力な調査ツール群を使いこなすことが推奨されます。
-   **事後対応（レビュー）**: 対策を導入して終わりではなく、3ヶ月後に「本当に運用できているか」をレビューするまでがコンサルタント/アナリストの役割です。

## 19. ネットワーク・フォレンジックの実践

同じ被害シナリオ（Polishementis社）を多角的に分析するプロセスです。

-   **C2サーバーの特定**: Webプロキシ（Squid）のログから、`www1-googleanalytics.com` のような「正規サイトに似せたドメイン」への不審な通信を見抜く。
-   **タイムゾーンの罠**: 調査時は必ず **UTC（協定世界時）** を基準にする。夏時間の有無で1時間のズレが生じると、他のログとのタイムラインが一致せず、調査が破綻するため。
-   **ビーコン通信の抽出**: `findbeacons.py` 等のツールで、定期的に発生する微弱な通信を特定し、感染端末（スコーピング）を洗い出す。

## 20. メモリ・フォレンジック（Volatility 3）

最近の攻撃の半分を占める「ファイルレス・マルウェア（ハードディスクに実体を持たない）」に対抗する手段です。

-   **Volatilityの活用**: プロセスの親子関係を可視化する（`pslist`, `pstree`）。
		-   **重要**: `lsass.exe` から `nc.exe`（Netcat）が起動している等、本来あり得ない親子関係は侵害の決定打。
-   **ハッシュダンプ**: メモリ上の認証プロセスからパスワードハッシュを抽出。攻撃者がオフラインで解析するのと同じ手法を、調査側も証拠収集のために行う。
-   **注意点**: メモリダンプはシステムが動いている間に取るため、100%完全なデータではない。レジストリ等の静的なデータは、メモリからではなくハードディスクのマスターデータ（Hiveファイル）から取得したほうが信頼性が高い。

## 21. マルウェア解析の基本アプローチ

高度なリバースエンジニアリング（アセンブリ読解）ができなくても、以下のステップで対応可能です。

1.  **OSINT（公開情報調査）**: まずは検体のハッシュ値をGoogleやVirusTotalで検索。既知のマルウェアなら、アンチウイルスベンダーのレポートを読むだけで全容がわかる。
2.  **動的解析（サンドボックス）**: `Any.run` や `Hybrid Analysis` 等の安全な環境で実際に動かし、どのような通信・ファイル操作を行うか観察する。
3.  **静的解析**: `Strings` コマンドで実行ファイル内の文字列を抽出。C2サーバーのURLや、エラーメッセージ、埋め込まれたIPアドレスが見つかることが多い。

> ⚠️ **注意：機密情報の流出**
>
> 標的型攻撃の検体をVirusTotal等の公開サイトにアップロードすると、攻撃者に「調査されている」とバレるだけでなく、検体に含まれる自社の機密情報が世界中に公開されるリスクがある。

## 22. インシデント対応の加速：生成AI（ChatGPT等）の活用

AIは、調査の「平均解決時間（MTTR）」を大幅に短縮する強力な味方になります。

-   **プロンプトエンジニアリングのコツ**:
		-   **役割を定義する**: 「あなたは熟練のインシデント・レスポンダーです」と役割を指定する。
		-   **反復と深掘り**: 一度の回答で諦めず、詳細や形式（JSONやテーブル形式など）を指定してしつこく聞き直す。
		-   **ハルシネーション（幻覚）に注意**: 存在しない脆弱性やツールを捏造することがあるため、常に人間によるダブルチェックが必要。

### 22.1. 具体的なユースケース

| **用語** | **AIにさせること** | **メリット** |
| --- | --- | --- |
| **コード解析** | 不審なC#や難読化されたPHPを読み解かせる。 | 数時間かかる解析が数十秒で終わる（精度は8割程度でも十分有用）。 |
| **スクリプト作成** | 「特定のログを探すPowerShellを作って」と依頼。 | プログラミングが苦手でも高度な自動化が可能になる。 |
| **報告書作成** | 技術的な詳細を「経営層向け（CISO向け）」に要約させる。 | 相手に合わせた言葉選びを自動化できる。 |

## 23. 実務での注意点：プライバシーとリスク

-   **情報の流出**: 未知のマルウェアや自社の機密情報を公開AIにアップロードすると、攻撃者に調査を察知されたり、データが一般に漏洩したりするリスクがある。
-   **Google NotebookLM の活用**: 自社のPDFやドキュメントを「ソース」として限定して読み込ませることで、情報の捏造（ハルシネーション）を防ぎつつ、正確な要約が可能。

## 24. 明日への備え：Linux/PowerShell ブートキャンプ
