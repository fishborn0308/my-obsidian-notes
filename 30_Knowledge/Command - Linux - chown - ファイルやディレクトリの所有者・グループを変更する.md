# Command - Linux - chown - ファイルやディレクトリの所有者・グループを変更する

## 概要

`chown` は、ファイルやディレクトリの所有ユーザー、および所有グループを変更するためのコマンドです。Linuxのアクセス制御（DAC: Discretionary Access Control）の根幹を成す所有権（Ownership）を管理します。

(出自: `coreutils` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `chgrp` | 所有グループのみを変更する。 | グループのみの変更 |
| `chmod` | アクセス権限（rwx）を変更する。所有者は変えない。 | 権限の変更 |
| `chown` | ユーザーとグループを同時に変更できるため、現在は `chgrp` よりも多用される。 | 総合的な所有権管理 |

## よく連携されるコマンド

### シナリオ例: Dockerボリュームマウント後の権限一括修正 (運用視点)

* **目的**: ホスト側からマウントしたディレクトリの所有者が `root` になっている場合、コンテナ内のアプリケーションユーザーに変更する。
* **連携コマンド**: `id`, `find`
* **解説**: 特定のUID/GIDを持つユーザーに再帰的に所有権を譲渡します。
* **コマンド例**:
    ```bash
    # 現在のディレクトリ以下の全ファイルを www-data ユーザーとグループに変更
    sudo chown -R www-data:www-data ./html
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **基本操作** | |
| ⭐️ `-R`, `--recursive` | ディレクトリとその中身を再帰的に変更する。 |
| ⭐️ `-v`, `--verbose` | 実行した処理の内容を詳細に表示する。 |
| `-c`, `--changes` | 実際に変更が行われた場合のみ表示する。 |
| **リンク・参照** | |
| `-h`, `--no-dereference` | シンボリックリンク自体の所有権を変更する（リンク先は変えない）。 |
| `--reference=<RFILE>` | 指定したファイル（RFILE）と同じ所有者・グループに設定する。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 新規サービスのデプロイにおけるログディレクトリのセットアップ。
* **組み合わせ**: `chown -R <user>:<group>`
* **解説**: アプリケーションがログを書き込めるよう、専用の実行ユーザーに所有権を渡します。
* **例**:
    ```bash
    # ログディレクトリの所有権を nginx ユーザーに変更
    sudo chown -R nginx:adm /var/log/my_app/
    ```

### 2. ブルーチーム視点

* **タスク**: 権限昇格の痕跡調査（所有者が不自然に変更されたファイルの特定）。
* **組み合わせ**: `find` との連携
* **解説**: 本来 `root` であるべきバイナリや設定ファイルの所有者が、一般ユーザーに変更されていないかスキャンします。
* **例**:
    ```bash
    # /etc 配下で所有者が root 以外になっているファイルをリストアップ
    find /etc ! -user root -ls
    ```

### 3. レッドチーム視点

* **タスク**: `sudo chown` を利用した特権昇格。
* **組み合わせ**: `sudo chown $(id -u):$(id -g) /etc/shadow`
* **解説**: もし `sudoers` 設定のミスで `chown` の実行が許可されている場合、`/etc/shadow` などの特権ファイルの所有権を自分に変更し、中身を直接書き換えて root 奪取を行います。
* **例**:
    ```bash
    # 特権ファイルの所有権を自分(current user)に変更
    sudo chown $(whoami) /etc/shadow
    # その後、直接パスワードハッシュを書き換える
    ```

## エラーメッセージとトラブルシューティング

1.  **エラーメッセージ例 1**: `chown: changing ownership of '...': Operation not permitted`
    * **考えられる原因**: 一般ユーザーが自分のファイルの所有者を他人に渡そうとしている（Linuxでは root のみが「所有権の譲渡」を行えます）。
    * **解決策**: `sudo` を使用して実行する。

2.  **エラーメッセージ例 2**: `chown: invalid user: 'nonexistent_user'`
    * **考えられる原因**: 指定したユーザー名がシステムに存在しない。
    * **解決策**: `/etc/passwd` を確認するか、UID（数値）で直接指定する。

## 環境変数と設定ファイル

* `chown` 自体に設定ファイルはありませんが、LDAPやActive Directory連携環境（`sssd`等）では、ネットワーク上のユーザー名を解決できない場合にエラーが発生することがあります。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: 不適切な `sudoers` 設定（ワイルドカードによる `chown` の許可など）。
* **悪用シナリオ**: 攻撃者が `sudo chown` を使って、自身の作成した偽のライブラリや設定ファイルの所有者を `root` に変え、システムサービスに読み込ませる。

### GTFOBins における利用例

`chown` は `sudo` 権限がある場合、任意のファイル（バイナリ、設定ファイル、シャドウファイル）の所有権を奪うことができるため、実質的にシステムの全権を握るのと同義です。

* **機能**: `Sudo`
* **コマンド例**:
    ```bash
    # 自身の権限で /etc/shadow を読めるようにする
    sudo chown $(whoami) /etc/shadow
    ```
* **参照**: `https://gtfobins.github.io/gtfobins/chown/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: `sudoers` において、`chown` を一般ユーザーに許可しない。許可が必要な場合は対象ディレクトリを厳格に制限する。
* **Detection (検知)**: `/etc/`, `/bin/`, `/sbin/` 内のファイルに対する所有権変更を `auditd` で常時監視する。

## 注意点・補足

* **シンボリックリンクの罠**: デフォルトでは `chown` はリンク先のファイルの所有者を変えます。リンク自体を変えたい場合は `-h` を忘れずに。
* **数値指定**: ユーザー名だけでなく UID:GID (例: `1000:1000`) での指定が可能です。スクリプト等で環境に依存したくない場合に有効です。
