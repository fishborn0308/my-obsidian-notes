# Command - Linux - export - 環境変数を設定して子プロセスへ継承させる

## 概要

`export` は、現在のシェルで定義された変数を「環境変数」としてマークします。これにより、そのシェルから実行される子プロセスは、その変数の値を受け継ぐことができるようになります。単に `VAR=VAL` と書くだけではそのシェル内（シェル変数）に留まりますが、`export` を使うことでシステム全体（そのセッション下）へ影響を広げられます。

(出自: シェルの組み込みコマンド (Built-in))

## 類似・関連コマンド

| コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `VAR=VAL` | シェル変数の定義。現在のシェル内でのみ有効で、子プロセスには引き継がれない。 | 一時的な計算やフラグ |
| `env` | 一時的に環境変数を変えて特定のコマンドを1回だけ実行する。 | 使い捨ての環境変更 |
| `set` | シェル変数と環境変数の両方を表示する。 | 全ての状態確認 |
| `export` | 現在のシェルおよび今後起動する全ての子プロセスに変数を波及させる。 | **セッション全体の環境構築** |

## よく連携されるコマンド

### シナリオ例: 新しいツールのパスを通す (運用・構築視点)

* **目的**: 自分でインストールしたプログラム（例: `/opt/mytool/bin`）を、フルパスを打たずに実行できるようにする。
* **連携**: `$PATH` 変数
* **解説**: 既存の `$PATH` に新しいディレクトリを追加し、`export` で有効化します。
* **コマンド例**:
    ```bash
    # PATHに新しいディレクトリを追加してエクスポート
    export PATH=$PATH:/opt/mytool/bin
    ```

## オプションと用法

`export` は引数なしで実行すると、現在エクスポートされている全ての変数一覧を表示します。

| 用法 | 説明 |
| :--- | :--- |
| ⭐️ `export VAR=VALUE` | 変数を定義し、同時にエクスポートする。 |
| ⭐️ `export VAR` | 既に定義されているシェル変数をエクスポート対象にする。 |
| `-p` | 現在エクスポートされている変数を、再定義可能な形式（`export VAR="VAL"`）で一覧表示する。 |
| `-n` | エクスポート属性を取り消す（シェル変数に戻す）。 |
| `-f` | 関数をエクスポートする（子プロセスでもその関数が使えるようになる）。 |

## よく使われる業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: プロキシサーバ経由でのインターネット接続設定。
* **解説**: `curl` や `apt` など、多くのコマンドが参照する標準的な環境変数を設定します。
* **例**:
    ```bash
    # HTTPプロキシを設定
    export http_proxy="[http://proxy.example.com:8080](http://proxy.example.com:8080)"
    export https_proxy="[http://proxy.example.com:8080](http://proxy.example.com:8080)"
    ```

### 2. ブルーチーム視点

* **タスク**: シェル履歴 (`.bash_history`) からの機密情報流出の防止。
* **解説**: `HISTCONTROL` 変数を設定することで、コマンドの先頭にスペースを入れた場合に履歴に残さないように制御できます。
* **例**:
    ```bash
    # 先頭にスペースを入れたコマンドを履歴に保存しない設定
    export HISTCONTROL=ignorespace
    # (実行時)  mysql -u root -p'password'  <-- 先頭にスペース
    ```

### 3. レッドチーム視点

* **タスク**: 共有ライブラリの乗っ取りによる特権昇格の準備。
* **解説**: `LD_PRELOAD` 変数をエクスポートしておくことで、その後に実行する全てのプログラム（特に `sudo` 経由のもの等）に自身のコードを注入しようと試みます。
* **例**:
    ```bash
    # 悪意あるライブラリを優先的に読み込ませる設定
    export LD_PRELOAD=/tmp/evil.so
    ```

## エラーメッセージとトラブルシューティング

1.  **「子プロセスで変数が消えている」**:
    * **原因**: `VAR=VAL` とだけ書いて、`export` し忘れている。
    * **解決策**: `export VAR` を実行するか、定義時に `export VAR=VAL` と書く。

2.  **「再起動したら設定が消えた」**:
    * **原因**: `export` は現在のセッションのみ有効なため。
    * **解決策**: ログイン時に自動で実行されるよう、`~/.bashrc` や `~/.zshrc` に記述する。

## セキュリティに関する考慮事項

### 脆弱性とリスク

* **情報漏洩**: `export` された変数は、`/proc/<PID>/environ` ファイルを通じて（適切な権限があれば）外部から読み取ることができます。APIキーなどを環境変数に入れたままにすると、メモリダンプや特定の脆弱性から漏洩するリスクがあります。
* **Shellshock (CVE-2014-6271)**: かつて `export -f` による関数のエクスポート機能を悪用し、環境変数の末尾に不正なコードを仕込むことで OS コマンドを実行させる致命的な脆弱性が存在しました。

### GTFOBins における利用例

`export` 自体は組み込みコマンドですが、`sudo` 設定（`env_keep` 等）が不適切な場合、攻撃者は `export` を通じて環境変数を操作し、実行ファイルの検索パスを偽装したり、ライブラリをインジェクションしたりして root を奪取します。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 秘密情報は環境変数ではなく、ファイル（権限 600）やシークレット管理サービスから読み込むように設計する。
* **Detection (検知)**: 不自然な `LD_PRELOAD` や `PYTHONPATH` のエクスポートを `auditd` や EDR で監視する。

## 注意点・補足

* **永続化**: `export` した内容はログアウトすると消えます。恒久的に設定したい場合は、設定ファイル（`.bashrc` 等）に書いた後、`source ~/.bashrc` で反映させるのが定石です。
* **大文字が慣習**: 環境変数は、シェル変数と区別しやすくするために `PATH`, `LANG`, `EDITOR` のように大文字で書くのが一般的です。
