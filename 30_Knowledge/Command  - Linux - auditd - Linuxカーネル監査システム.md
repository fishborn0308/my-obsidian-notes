# Command  - Linux - auditd - Linuxカーネル監査システム

## 概要

`auditd` は、Linuxカーネルの監査機能からのイベントをディスクに書き出すユーザー空間のデーモンです。カーネルと密に連携し、システムコールレベルで「**いつ誰が何をどうしたか**」を極めて詳細に記録できます。特定のファイルへのアクセス、コマンドの実行、ネットワーク接続、システムコールの呼び出しなど、セキュリティに関連するあらゆるイベントを追跡・記録するために設計されています。記録されたログは、インシデント調査、コンプライアンス準拠、不正アクセスの検知に不可欠な情報源となります。

(出自: `audit` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`syslog` (rsyslog)** | `syslog` は主に**アプリケーション**が出力するログを集約します。`auditd` は**カーネルレベル**のイベントを記録するため、より低レイヤーで詳細な情報を得られます。 | (両方目的が違う) |
| **eBPFベースのツール** | (例: Falco) eBPFもカーネルイベントをフックするモダンな技術で、より高パフォーマンスな場合があります。`auditd` はLinuxに標準で組み込まれている伝統的な監査フレームワークです。 | - |
| **SELinux / AppArmor**| これらは操作を**防ぐ (Enforce)** ことを目的とする強制アクセス制御です。`auditd` は操作を**記録する (Log)** ことが目的であり、連携して動作します。 | - |

## よく連携されるコマンド

`auditd` のログは非常に詳細なため、`ausearch` や `aureport` といった専用ツールや、`grep` などを組み合わせて分析するのが一般的です。

### シナリオ例: SELinuxの拒否ログを分かりやすく表示する (ブルーチーム視点)

* **目的**: SELinuxによってアクセスが拒否されたイベント (AVC denial) を、人間が読みやすい形式で表示し、トラブルシューティングに役立てる。
* **連携コマンド**: `ausearch`, `audit2why`
* **解説**: `ausearch` でAVC (Access Vector Cache) 関連のメッセージを検索し、その結果をパイプで `audit2why` に渡すことで、なぜアクセスが拒否されたのか、その理由と解決策のヒントを表示させます。
* **コマンド例**:

```bash
# 直近のSELinux拒否ログを検索し、原因と対策を表示する

ausearch -m AVC -ts recent | audit2why

```

## オプション説明 (`auditctl` - ルール管理)

| オプション | 説明 |
| :--- | :--- |
| **ルール管理** | |
| ⭐️ `-l` | 現在ロードされているルールを一覧表示します。 |
| `-D` | 全てのルールを削除します。 |
| **ファイル監視** | |
| ⭐️ `-w <path>` | (watch) ファイルやディレクトリへのアクセスを監視するルールを追加します。 |
| `-p <perms>`| `-w` と組み合わせ、監視する権限を指定します (`r`読取, `w`書込, `x`実行, `a`属性変更)。 |
| `-k <key>` | (key) ルールに検索用のキー（任意の文字列）を設定します。ログ検索時に非常に便利です。 |
| **システムコール監視** | |
| `-a <list,action>`| (append) システムコールのルールを追加します。例: `-a exit,always` (システムコール終了時に常に監査)。 |
| `-S <syscall>`| 監視するシステムコール名を指定します。例: `-S execve` (プログラム実行)。 |
| `-F <field=value>`| ルールを適用する条件を指定します。例: `-F arch=b64` (64bitアーキテクチャ)。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 重要な設定ファイル (`/etc/sudoers`) への全ての変更を記録する。
* **組み合わせ**: `auditctl -w <path> -p wa -k <key>`
* **解説**: `-w` でファイルを指定し、`-p wa` で書き込みと属性変更を監視対象とします。`-k` で検索用のキーを付けておくことで、後のログ追跡が容易になります。
* **例**:

```bash
# /etc/sudoers への書き込み(w)と属性変更(a)を監視し、"sudoers_change"というキーを付ける

sudo auditctl -w /etc/sudoers -p wa -k sudoers_change

```

### 2. ブルーチーム視点

* **タスク**: 権限昇格や攻撃に使われやすいコマンド (`nc`など) の実行を全て記録する。
* **組み合わせ**: `-a always,exit -F path=... -F perm=x -k ...`
* **解説**: `execve` システムコールの終了 (`exit`) 時に、常に (`always`) ルールを評価します。`-F path=` で実行ファイルのパスを、`-F perm=x` で実行権限を指定することで、そのコマンドの実行イベントを正確に捕捉します。
* **例**:

```bash
# custom.rulesに追記後、ルールを再読み込み。/usr/bin/nc の実行を監視する。

sudo augenrules --load

# ログ検索

sudo ausearch -k suspicious_execution -i

```

### 3. レッドチーム視点

* **タスク**: 攻撃対象サーバの監査設定を調査し、可能であれば無力化して検知を回避する。
* **組み合わせ**: `auditctl -l`, `systemctl stop auditd`
* **解説**: 攻撃者は権限昇格後、まず `auditctl -l` でどのような操作が監視されているかを把握し、監視を避ける手法を選択します。最終的には `auditd` デーモン自体を停止させ、以降の活動の痕跡が記録されるのを防ぎます。
* **例**:

```bash
# auditdの稼働状況とルールの確認

systemctl status auditd

auditctl -l

# auditdを停止 (root権限が必要)

sudo systemctl stop auditd

sudo auditctl -e 0

```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **現象**: **OS再起動後に `auditctl` で追加したルールが消えてしまう。**
		* **考えられる原因**: `auditctl` で追加したルールは、実行中のカーネルにのみ適用され、永続化されません。
		* **解決策**: ルールは `/etc/audit/rules.d/` ディレクトリ配下に `.rules` 拡張子を持つファイルとして記述します。OS起動時に `augenrules` スクリプトがこのディレクトリのファイルを読み込み、ルールを自動的に適用します。

## 環境変数と設定ファイル

* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

### 関連する設定ファイル

* **`/etc/audit/auditd.conf`**: `auditd` デーモン自体の設定ファイル。ログファイルの場所、サイズ、ローテーション方法などを定義します。
* **`/etc/audit/rules.d/`**: 監査ルールを記述したファイルを置くディレクトリ。この中の `.rules` ファイルが起動時に読み込まれます。
* **`/var/log/audit/audit.log`**: デフォルトの監査ログファイル。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `auditd` 自体よりも、それが監視するシステムコールの脆弱性が悪用されることが多いです。
* **悪用シナリオ**: 攻撃者は権限昇格後、最優先で `auditd` を停止または無効化しようと試みます。これにより、以降の攻撃活動（ラテラルムーブメント、データ窃取など）の痕跡が記録されるのを防ぎます。

### GTFOBins / LOLBAS における利用例

`auditd` は防御側のツールであり、GTFOBins にはリストされていません。むしろ、GTFOBins にリストされているような**環境寄生型攻撃を検知するための最も強力なツール**の1つです。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: `auditctl -e 2` (immutable mode) を設定し、再起動するまでルールの変更を不可能にする。監査ログをリアルタイムでSIEMなどの外部ログサーバに転送し、改ざんを防ぐ。
* **Detection (検知)**: `auditd` サービスが停止した場合にアラートを発報する。`ausearch` や `aureport` を使って定期的にログをレビューし、異常なイベントがないかを確認する。

## 注意点・補足

* **パフォーマンス**: 監査ルールが多すぎると、システムのパフォーマンスに影響を与える可能性があります。「何を監視する必要があるか」を明確にし、必要最小限のルールから始めることが重要です。
