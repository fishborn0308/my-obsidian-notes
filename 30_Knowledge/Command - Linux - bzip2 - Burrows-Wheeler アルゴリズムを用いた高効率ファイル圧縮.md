# Command - Linux - bzip2 - Burrows-Wheeler アルゴリズムを用いた高効率ファイル圧縮

## 概要

`bzip2` は、Burrows-Wheeler ブロックソートテキスト圧縮アルゴリズムとハフマン符号化を使用してファイルを圧縮するツールです。従来の `gzip` よりも高い圧縮率を実現しますが、その分、圧縮・展開に要する CPU 負荷と時間は大きくなります。

(出自: `bzip2` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `gzip` | 圧縮・展開が非常に高速。圧縮率は bzip2 より低い。標準ログ（logrotate）で多用。 | 速度優先 / 頻繁なアクセス |
| `xz` | 現代で最高峰の圧縮率を誇る。圧縮時の負荷が非常に高く、時間がかかる。 | 圧縮率最優先 / 長期保管 |
| `bzip2` | `gzip` と `xz` の中間の特性。多くの Linux ディストリビューションで標準採用。 | バランス重視 |
| `zip` | Windows との互換性が高い。複数ファイルを1つにまとめる（アーカイブ）機能を持つ。 | OS間での受け渡し |

## よく連携されるコマンド

### シナリオ例: 複数ディレクトリの圧縮アーカイブ作成 (運用視点)

* **目的**: 多数のファイルを1つのファイルにまとめつつ、`bzip2` で圧縮する。
* **連携コマンド**: `tar`
* **解説**: `tar` コマンドの `-j` オプションは内部的に `bzip2` を呼び出します。
* **コマンド例**:
    ```bash
    # /var/log ディレクトリを bzip2 形式で固める
    sudo tar -cjvf logs_backup.tar.bz2 /var/log
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **基本操作** | |
| ⭐️ `-d`, `--decompress` | 圧縮ファイルを展開（解凍）する。 |
| ⭐️ `-k`, `--keep` | 圧縮・展開後に元のファイルを削除せず保持する（デフォルトは削除される）。 |
| `-z`, `--compress` | ファイルを圧縮する（デフォルト動作）。 |
| `-c`, `--stdout` | 結果を標準出力に書き出す（パイプ処理に有用）。 |
| **圧縮効率設定** | |
| `-1` .. `-9` | 圧縮レベルを指定。`-1` が最速（低圧縮）、`-9` が最高圧縮（デフォルト）。 |
| **整合性・情報** | |
| `-t`, `--test` | 圧縮ファイルの整合性をチェックする。 |
| `-v`, `--verbose` | 圧縮率などの詳細情報を表示する。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: ディスク容量が逼迫している環境での古いバックアップの再圧縮。
* **組み合わせ**: `bzcat` または `bzip2 -dc`
* **解説**: 圧縮ファイルを展開せずに中身を確認したり、パイプで別の処理に渡したりします。
* **例**:
    ```bash
    # 圧縮されたログファイルの中から特定のキーワードを検索 (bzgrep)
    bzgrep "ERROR" backup_log.bz2
    ```

### 2. ブルーチーム視点

* **タスク**: インシデント調査中の証拠保全。
* **組み合わせ**: `-k -9`
* **解説**: 元の証拠ファイル（メモリダンプやディスクイメージ）を維持しつつ、最高圧縮でコピーを作成して分析環境へ転送します。
* **例**:
    ```bash
    # 元ファイルを残して最高圧縮
    bzip2 -k9 evidence.img
    ```

### 3. レッドチーム視点

* **タスク**: 窃取した大量のドキュメントの隠密転送。
* **組み合わせ**: `tar` + `bzip2` + `split`
* **解説**: 大容量データを圧縮してパケットサイズを減らし、さらに `split` で分割することで、IDS/IPS による大きなファイル転送の検知を回避します。
* **例**:
    ```bash
    # データを固めて圧縮し、10MBずつに分割
    tar -cj /home/user/documents | split -b 10M - out_part_
    ```

## エラーメッセージとトラブルシューティング

1.  **エラーメッセージ例 1**: `bzip2: Input file <file> already has .bz2 suffix.`
    * **考えられる原因**: 既に圧縮済みのファイルを再度圧縮しようとした。
    * **解決策**: 二重圧縮は効率が悪いため避ける。どうしても行う場合はリダイレクト等を使用する。

2.  **エラーメッセージ例 2**: `bzip2: I/O or other error, terminating.`
    * **考えられる原因**: ディスク容量不足、またはパーミッション不足。
    * **解決策**: `df -h` で空き容量を確認し、書き込み権限があるかチェックする。

## 環境変数と設定ファイル

* `bzip2` は特定の環境変数を持ちませんが、パイプ処理において `BZIP2` という環境変数にデフォルトオプションを設定できる実装もあります（例: `export BZIP2="-9"`）。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: 過去には解凍時のバッファオーバーフロー（CVE-2010-0405）などが報告されています。古いシステムでの解凍作業には注意が必要です。
* **悪用シナリオ**: **解凍爆弾（Zip Bomb / Bzip2 Bomb）**。極めて高い圧縮率を利用し、数KBのファイルを解凍すると数百GBになり、ディスクをパンクさせる攻撃。

### GTFOBins における利用例

`bzip2` に `sudo` 権限がある場合、本来読み取れないファイルを「圧縮して標準出力に吐き出す」ことで内容を窃取できます。

* **機能**: `File Read`, `Sudo`
* **コマンド例**:
    ```bash
    # sudo を使って /etc/shadow を読み取る
    LFILE=/etc/shadow
    sudo bzip2 -c "$LFILE" | bzip2 -d
    ```
* **参照**: `https://gtfobins.github.io/gtfobins/bzip2/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 解凍作業は、クォータ制限（ディスク使用量制限）が設定された隔離ディレクトリで行う。
* **Detection (検知)**: CPU 使用率の異常なスパイクや、短時間での急激なディスク消費を監視し、解凍爆弾の兆候を捉える。

## 注意点・補足

* **元ファイルの消失**: デフォルトでは圧縮後に元のファイルが消えるため、不慣れなうちは `-k` (keep) オプションを付ける習慣を推奨します。
* **並列処理**: 標準の `bzip2` はシングルスレッドですが、マルチコアを活かせる `pbzip2` という互換コマンドも存在します。
