# OS  - Linux - filesystem_hierarchy_standard - ファイルシステム階層標準 (FHS) - Linuxのディレクトリ構造

## 概要

**ファイルシステム階層標準 (Filesystem Hierarchy Standard, FHS)** は、Linuxやその他のUnix系OSにおける主要なディレクトリの構成と、それぞれのディレクトリにどのような種類のファイルを配置すべきかを定めた標準的な規約です。FHSに従うことで、ソフトウェアは予測可能な場所にファイルやライブラリを配置でき、ユーザーや管理者はディストリビューションが異なっても、目的のファイルを一貫した場所に見つけ出すことができます。これは、システムの管理性、移植性、セキュリティを保つ上で非常に重要な「地図」の役割を果たします。

(出自: Linux Foundation)

## 類似コマンドと差異

| 概念 | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **Windowsのディレクトリ構造** | `C:\Program Files` や `C:\Users` のようなWindowsの構造とは根本的に異なります。Linuxでは全てのパスが単一のルート (`/`) から始まります。 | - |

## よく連携されるコマンド

FHSの知識は、ファイルシステムを操作するほぼ全てのコマンドと連携します。

### シナリオ例: ログファイルの場所を特定し、内容を確認する (インフラ構築・運用視点)

* **目的**: Nginxでエラーが発生した際に、FHSの知識を元にログファイルの場所を推測し、内容を確認する。
* **連携コマンド**: `cd`, `ls`, `tail`
* **解説**: FHSでは、可変データ、特にログは `/var/log` に保存されると定められています。この知識があれば、`cd /var/log/nginx` のように目的のディレクトリに直接移動し、`tail` でログを確認できます。
* **コマンド例**:

```bash
# FHSの知識に基づき、Nginxのログディレクトリに移動
cd /var/log/nginx

# エラーログの最新100行をリアルタイムで監視
tail -n 100 -f error.log
```

## 🗺️ 主要なディレクトリとその役割

### ルートディレクトリ (`/`)

全てのファイルとディレクトリの起点となる最上位のディレクトリ。

| ディレクトリ | 説明 |
| :--- | :--- |
| ⭐ **`/etc`** | システム全体に関わる**設定ファイル**が格納される中心的なディレクトリ。 |
| ⭐ **`/var`** | ログ (`/var/log`)、キャッシュ (`/var/cache`) など、**動的に変化する可変データ**を格納するディレクトリ。 |
| ⭐ **`/home`** | 一般ユーザーの**ホームディレクトリ**が作成される場所（例: `/home/user1`）。 |
| **`/usr`** | システム全体で共有される読み取り専用のデータ。基本的なコマンド、ライブラリ、ドキュメントなどが含まれます。 |
| **`/bin`** | **(→ `/usr/bin`)** ユーザーが利用する**基本的なコマンド**（`ls`, `cp`, `cat`など）。 |
| **`/sbin`** | **(→ `/usr/sbin`)** システム管理者が利用する**基本的なシステム管理用コマンド**（`reboot`, `fdisk`, `ip`など）。 |
| **`/root`** | `root`ユーザーのホームディレクトリ。 |
| **`/lib`**, **`/lib64`**| **(→ `/usr/lib`)** `/bin` や `/sbin` のコマンドが依存する**基本的な共有ライブラリ**。 |
| **`/tmp`** | 全てのユーザーが書き込み可能な**一時ファイル**用ディレクトリ。OSの再起動で内容は消去されるのが一般的です。 |
| **`/opt`** | サードパーティ製の**追加アプリケーションパッケージ**のインストール場所。 |
| **`/boot`** | Linuxカーネルやブートローダーなど、**システムの起動に必要なファイル**。 |
| **`/dev`** | 物理デバイス、仮想デバイスを表す**デバイスファイル**が置かれる場所。 |
| **`/proc`**, **`/sys`**| カーネルやプロセスの情報をファイルとして提供する**仮想ファイルシステム**。 |

## 各視点からの関わり

### 1. インフラ構築・運用視点

* **タスク**: システムのトラブルシューティング。
* **解説**: FHSはシステム管理者のための地図そのものです。「アプリケーションがエラーを吐いている」→ `/var/log/` を調べる。「ディスクが一杯だ」→ `/var` や `/home` を `du` で調査する、といったように、問題解決の起点となります。

### 2. ブルーチーム視点

* **タスク**: 不審なファイルの設置場所を特定する。
* **解説**: FHSは「正常な状態（ベースライン）」を定義します。逸脱は異常の兆候です。`/tmp`, `/dev/shm`, `/var/tmp` といった誰でも書き込めるディレクトリは、攻撃者がツールやペイロードを置く最初の場所としてよく使われます。また、`/etc/cron.d/` や `/etc/systemd/system/` は永続化の標的となります。

### 3. レッドチーム視点

* **タスク**: 侵入後の偵察と永続化経路の探索。
* **解説**: FHSは攻撃者にとっての「宝の地図」です。`/etc/` を漁って設定ファイル内のパスワードを探したり、`/home/` を探索してユーザーの秘密鍵を盗んだりします。また、`/etc/profile.d/` などの場所にバックドアを仕掛けて永続化を図ります。

## エラーメッセージとトラブルシューティング

* FHSは概念であり、直接的なエラーメッセージはありません。トラブルシューティングは、この規約に基づいてどこに問題があるファイルが存在するかを探すことから始まります。
* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

## 環境変数と設定ファイル

* FHSの知識は、`PATH` や `HOME` といった基本的な環境変数がなぜその値になっているかを理解する助けになります。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](OS%20%20-%20Linux%20-%20environment_and_config%20-%20環境変数と設定ファイル.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: FHS自体ではなく、各ディレクトリの**不適切なパーミッション設定**が脆弱性となります。
* **悪用シナリオ**: `/tmp` のような誰でも書き込めるディレクトリに `noexec` マウントオプションが設定されていない場合、攻撃者はそこにマルウェアを配置して実行することが容易になります。

### GTFOBins / LOLBAS における利用例

FHSは概念であり、GTFOBinsにはリストされていません。しかし、GTFOBinsにリストされている多くの攻撃は、FHSで定められたディレクトリ（`/bin`, `/usr/bin`, `/etc`など）に存在する正規のバイナリを悪用します。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: `/tmp` などのパーティションに `noexec`, `nosuid`, `nodev` といったマウントオプションを設定する。
* **Detection (検知)**: ファイル整合性監視 (FIM) ツールを使い、FHSに基づき `/bin`, `/sbin`, `/etc` などの重要なディレクトリにあるファイルの変更（改ざん）を検知する。

## 注意点・補足

* **シンボリックリンク**: モダンなLinuxディストリビューションでは、階層構造を簡素化するため、`/bin` は `/usr/bin` へ、`/sbin` は `/usr/sbin` へのシンボリックリンクになっていることが多くなっています。
