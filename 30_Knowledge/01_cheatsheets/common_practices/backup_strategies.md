# `backup_strategies` - 堅牢なデータ保護のためのバックアップ戦略

## 概要

バックアップ戦略は、データの喪失を防ぎ、システムの復旧能力を確保するための計画と実装です。障害発生時のダウンタイムを最小限に抑え、事業継続性を確保する上で不可欠な要素です。堅牢なバックアップ戦略は、**3-2-1ルール**（3つのコピー、2種類のメディア、1つはオフサイト）を基本とし、**RTO（目標復旧時間）**と**RPO（目標復旧時点）**という2つの主要な指標に基づいて設計されます。

(出自: ITインフラにおける一般的なベストプラクティス)

## 類似概念との差異

| 類似概念 | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| **レプリケーション** | データをリアルタイムまたはニアリアルタイムで別の場所に複製する技術。**高可用性 (HA)** を実現します。バックアップは特定の**時点**にデータを戻すためのものであり、目的が異なります。 | - |
| **スナップショット** | 特定の時点の仮想マシンやボリュームの状態を記録する機能。単体ではバックアップではありませんが、整合性の取れたバックアップを取得するための起点として非常に重要です。 | - |

## よく連携されるコマンド/ツール

バックアップ戦略は、特定のコマンドというより、複数のツールと概念を連携させて実現します。

### シナリオ例: `rsync` と `cron` による日次バックアップ (インフラ構築・運用視点)

* **目的**: 毎日深夜に、Webサーバーのコンテンツディレクトリをバックアップサーバーに差分同期する。
* **連携コマンド**: `rsync`, `cron`
* **解説**: `rsync` は効率的な差分同期を行い、`cron` はその処理を定期的に自動実行します。これはLinuxにおける最も基本的なファイルベースバックアップの実装です。
* **コマンド例 (`/etc/cron.d/backup` などに記述)**:

    ```bash
    # 毎日午前3時30分に /var/www/html の内容を backup-server に同期
    30 3 * * * root rsync -a --delete /var/www/html/ backup-user@backup-server:/backups/www/
    ```

## バックアップの種類

| オプション | 説明 |
| :--- | :--- |
| **取得方法** | |
| ⭐ **フルバックアップ** | 全てのデータをバックアップ。復元は簡単だが、時間と容量を消費します。 |
| **差分バックアップ** | 最後に実行された**フルバックアップ**からの変更点のみをバックアップ。 |
| **増分バックアップ** | 最後に実行された**いずれかのバックアップ**からの変更点のみをバックアップ。最も効率的だが、復元が複雑になります。 |
| **メディア** | |
| **ディスク** | NASやDASなど。高速な読み書きが可能。 |
| **テープ** | LTOなど。大容量、低コスト、オフライン保管に適しています。 |
| ⭐ **クラウド** | AWS S3, Azure Blob Storageなど。オフサイト保管と高い耐久性を実現します。 |

## 1\. バックアップの基本原則と目的

バックアップ戦略を理解するための基礎的な概念と、なぜバックアップが必要なのかを説明します。

### 1.1. バックアップの重要性

* **データ損失からの保護**: ハードウェア故障、ソフトウェア障害、人為的ミス、自然災害、サイバー攻撃など。
* **事業継続性の確保 (BCP)**: 障害発生時のダウンタイムを最小限に抑え、サービスを迅速に復旧させる。
* **法的・規制要件の遵守**: データ保持期間に関する法的義務(GDPR, HIPAAなど)への対応。
* **監査とフォレンジック**: 過去のデータが必要となる場合への備え。

### 1.2. 主要な指標 (RTO / RPO)

* **RTO (Recovery Time Objective - 目標復旧時間)**: 災害や障害発生後、システムを許容されるレベルまで復旧させるのにかけられる目標時間。
* **RPO (Recovery Point Objective - 目標復旧時点)**: 災害や障害発生時に、どれだけのデータ損失を許容できるかの目標時点。
* **これらの指標がバックアップ戦略に与える影響**。

### 1.3. バックアップの3-2-1ルール

バックアップのベストプラクティスとして広く認識されているルール。

* **3**: データを**3つ**以上のコピーで保持する(オリジナルを含む)。
* **2**: 異なる**2種類**以上のメディアにバックアップする(例: ディスクとテープ、オンプレミスとクラウド)。
* **1**: バックアップの**1つ**はオフサイトに保管する(地理的に離れた場所)。

## 2\. バックアップの種類と方式

データの取得方法と保存方法に基づいたさまざまなバックアップ方式を説明します。

### 2.1. バックアップの種類 (取得方法)

* **フルバックアップ (Full Backup)**:
  * 全てのデータをバックアップ。
  * **利点**: 復元が最も簡単で高速。
  * **欠点**: 時間とストレージ容量を最も消費。
* **差分バックアップ (Differential Backup)**:
  * 最後に実行された**フルバックアップ**からの変更点のみをバックアップ。
  * **利点**: フルバックアップより高速で容量が少ない。復元にはフルバックアップと最新の差分バックアップが必要。
  * **欠点**: 累積されるため、世代が進むと容量が増える。
* **増分バックアップ (Incremental Backup)**:
  * 最後に実行された**いずれかのバックアップ**(フルまたは増分)からの変更点のみをバックアップ。
  * **利点**: 最も高速で容量が少ない。
  * **欠点**: 復元にはフルバックアップと全ての増分バックアップが必要なため、最も時間がかかり複雑。
* **スナップショット (Snapshot)**:
  * 特定の時点の仮想マシンやボリュームの状態を記録。データのコピーではないため、単体ではバックアップではないが、バックアップの起点として重要。

### 2.2. バックアップの方式 (保存方法/アプローチ)

* **ファイルベースバックアップ**: 特定のファイルやディレクトリを対象。
* **イメージバックアップ**: OS、アプリケーション、データを全て含むディスク全体をイメージとしてバックアップ。ベアメタル復元 (Bare Metal Recovery) に適している。
* **データベースバックアップ**: データベース固有のツールや機能(例: SQL Serverのトランザクションログバックアップ、MySQLのmysqldump)を利用。
* **SaaSバックアップ**: Microsoft 365, Salesforceなどのクラウドサービスデータのバックアップ。
* **レプリケーション (Replication)**: データのリアルタイムまたはニアリアルタイムでの同期。バックアップとは異なるが、高可用性やDR対策の一環として併用される。

## 3\. バックアップメディアと保存先

バックアップデーターをどこに保存するか、その種類と特性を説明します。

* **ローカルストレージ**: 内蔵HDD/SSD、外付けHDD。
* **ネットワークストレージ**: NAS (Network Attached Storage), SAN (Storage Area Network)。
* **テープストレージ**: LTO (Linear Tape-Open) など。大容量、低コスト、長期保存、オフライン保管に適している。
* **クラウドストレージ**:
  * **IaaSベース**: AWS S3, Azure Blob Storage, Google Cloud Storageなど。
  * **バックアップサービス**: AWS Backup, Azure Backup, Veeam Cloud Connectなど。
  * **利点**: スケーラビリティ、オフサイト保管、耐障害性。
  * **欠点**: データ転送コスト、復元時間、セキュリティ(アクセス制御)。

## 4\. バックアップ戦略の策定と運用

具体的なバックアップ戦略をどのように計画・実行・管理するかを説明します。

### 4.1. 戦略策定のステップ

1. **重要データの特定**: 何をバックアップするのか？　(OS, アプリケーション、 ユーザーデータ、 設定ファイルなど)
2. **RTO/RPOの決定**: どれくらいの時間とデータ損失を許容できるのか？
3. **バックアップ対象と頻度の決定**: フル、差分、増分をどのように組み合わせるか。
4. **保存先の選定**: 3-2-1ルールに基づいたメディアと場所。
5. **バックアップソフトウェア/ツールの選定**: OS標準機能、商用ソフトウェア、OSSツール。
6. **復旧手順の定義**: 障害発生時の具体的な復旧手順(ドキュメント化)。
7. **テスト計画**: 定期的な復旧テストの実施。

### 4.2. 運用管理のポイント

* **自動化**: スケジュールされたバックアップによる人為的ミスの削減。
* **監視とアラート**: バックアップの成功/失敗、容量超過などを監視し、異常があれば通知。
* **ログ管理**: バックアップジョブのログを定期的に確認。
* **定期的な復旧テスト**: 実際にデータを復元できるかを確認することが最も重要。
* **ドキュメント化**: バックアップ計画、手順、担当者を明確にする。

## 5\. 各視点からのバックアップ戦略

インフラ運用、ブルーチーム、レッドチームそれぞれの視点からバックアップ戦略における考慮事項と実践を詳述します。

### 5.1. インフラ構築・運用視点

* **バックアップ対象の選定**: OSイメージ、DBデータ、Webコンテンツ、設定ファイルなど。
* **バックアップツールの選定**:
  * **Linux**: `rsync`, `tar`, `dump/restore`, `dd`, LVMスナップショット、 Bacula, ZFSなど。
  * **Windows**: Windows Server Backup, VSS (Volume Shadow Copy Service), Hyper-Vスナップショット。
  * **仮想化環境**: Veeam Backup & Replication (VMware/Hyper-V), Acronis Cyber Backup。
  * **クラウド**: AWS Backup, Azure Backup, Google Cloud Backup and DR。
* **自動化とスクリプト**: Cron/タスクスケジューラでの自動実行、PowerShell/Bashスクリプトによるカスタマイズ。
* **パフォーマンスへの影響**: バックアップ実行時のシステム負荷、ネットワーク帯域の考慮。
* **データ整合性**: DBのコールドバックアップ/ホットバックアップ、VSSの利用。
* **コマンド例**:

    ```bash
    # Linuxでのrsyncによるファイルバックアップ例
    rsync -avz --delete /source/path/ /backup/destination/

    # Windows PowerShellでのファイルコピーバックアップ例
    Copy-Item -Path "C:\SourceData" -Destination "D:\Backup" -Recurse -Force
    ```

### 5.2. ブルーチーム視点

* **改ざん防止とデータの完全性**:
  * **不変性 (Immutability)**: バックアップデーターが一度書き込まれたら変更・削除できないようにする(WORM: Write Once Read Many)。クラウドストレージのオブジェクトロック機能など。
  * **ハッシュ値による検証**: バックアップデーターの整合性を定期的にチェック。
  * **分離 (Air Gap)**: ネットワークから物理的に分離されたオフラインバックアップの重要性(ランサムウェア対策)。
* **バックアップからのマルウェア除去**: 感染したシステムから復旧する際、バックアップデーター自体がマルウェアを含んでいないかの確認とクリーンな状態への復元。
* **認証とアクセス制御**: バックアップシステムへの厳格なアクセス制御(多要素認証の導入など)。
* **ログと監査**: バックアップジョブの実行、成功、失敗、復元操作などのログを監視。
* **リカバリープランのセキュリティ**: 復旧手順中にセキュリティリスクが生じないかを確認。
* **セキュリティに関する考慮事項**:
  * バックアップデーターを暗号化して保存する。
  * バックアップシステム自体の脆弱性管理とパッチ適用。
  * バックアップ用の認証情報を厳重に管理。
* **対応策・緩和策 (ブルーチーム視点)**:
  * \*\*Immutable Backup (不変バックアップ)\*\*の導入。
  * \*\*オフラインバックアップ (Air-gapped Backup)\*\*の活用。
  * \*\*多要素認証 (MFA)\*\*によるバックアップシステムへのアクセス保護。
  * **バックアップデーターの定期的なマルウェアスキャン**。
  * **最小特権の原則**に基づいたバックアップアカウントの権限設定。

### 5.3. レッドチーム視点

* **バックアップシステムへの侵入**:
  * バックアップシステム自体がセキュリティ対策が手薄になりがちであるため、攻撃対象となる可能性。
  * バックアップソフトウェアの既知の脆弱性を悪用。
  * バックアップサーバへの認証情報の窃取。
* **バックアップデーターの悪用**:
  * バックアップデーターから機密情報を窃取(設定ファイル、データベースダンプ、個人情報など)。
  * 復元プロセスを悪用したバックドアの設置やシステム改ざん。
* **バックアップの無効化/破壊**:
  * ランサムウェアなどによるバックアップデーターの暗号化・削除。
  * ディザスターリカバリー能力の妨害。
* **LOLBAS/GTFOBinsにおける関連性**:
  * 特定のバックアップツールやユーティリティが、情報収集や永続化に悪用される可能性。
  * 例: `VSSAdmin` (Windows) を使ったシャドウコピーの操作(削除など)。
* **悪用された事例**:
  * ランサムウェア攻撃におけるバックアップの破壊事例。
  * バックアップからの機密情報窃取事例。
* **対応策・緩和策 (ブルーチーム視点)**:
  * 前述の「改ざん防止とデータの完全性」の項目を徹底する。
  * バックアップシステムを他のシステムから**論理的・物理的に隔離**する。
  * バックアップシステムの脆弱性診断とペネトレーションテスト。
  * バックアップアカウントに**昇格権限を与えない**(必要な時だけ一時的に付与するなど)。

## 6\. 注意点・補足

* **バックアップは復旧できてはじめて意味がある**: 定期的な復旧テストの実施が何よりも重要です。
* **テスト環境での復旧演習**: 本番環境に影響を与えないテスト環境での復旧訓練。
* **バックアップの暗号化**: 保存中のデータと転送中のデータ両方の暗号化を検討する。
* **法的要件の確認**: データ保持期間やプライバシーに関する地域の法律や業界規制を確認する。
* **人材育成**: バックアップと復旧を担当するスタッフの継続的なトレーニング。

## 7\. よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 仮想化基盤上のWebサーバーとDBサーバーの整合性を保ったバックアップを取得する。
* **組み合わせ**: LVMスナップショット + `tar` または `rsync`
* **解説**: まずLVMスナップショットで静止点を確保し、そのスナップショットに対してバックアップ処理を実行します。これにより、バックアップ中にファイルが変更されることによるデータの不整合を防ぎます。
* **例**:

    ```bash
    # (スクリプト内での実行を想定)
    # 1. LVMスナップショットを作成
    lvcreate --size 1G --snapshot --name web_snapshot /dev/vg01/web_data
    # 2. スナップショットをマウントしてバックアップ
    mount /dev/vg01/web_snapshot /mnt/snapshot
    tar czf /backups/web-$(date +%F).tar.gz /mnt/snapshot
    # 3. 後処理
    umount /mnt/snapshot
    lvremove -f /dev/vg01/web_snapshot
    ```

### 2. ブルーチーム視点

* **タスク**: **ランサムウェア攻撃からの復旧**を想定したバックアップ戦略を設計・監査する。
* **組み合わせ**: **3-2-1ルール** + **不変（Immutable）バックアップ** + **オフライン（Air-Gapped）バックアップ**
* **解説**: ランサムウェアはバックアップデータ自体の暗号化や削除を試みます。これに対抗するため、一度書き込まれたら変更・削除できない**不変ストレージ**（S3オブジェクトロックなど）や、ネットワークから物理的に隔離された**オフラインバックアップ**（テープなど）を組み合わせることが極めて重要です。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: バックアップシステム自体の脆弱性、管理コンソールへの弱い認証、バックアップデータへの不適切なアクセス権設定。
* **悪用シナリオ**: レッドチームのシナリオで述べたように、ランサムウェア攻撃の最終段階としてバックアップの破壊が行われます。また、バックアップデータから平文のパスワードや設定ファイルなどの機密情報を窃取する攻撃も一般的です。

### GTFOBins / LOLBAS における利用例

バックアップ戦略は概念のため、GTFOBinsにはリストされていません。しかし、リストされている `tar`, `rsync`, `dd` などのコマンドは、バックアップとデータ窃取の両方に利用されます。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **バックアップシステムを本番環境からネットワーク的に隔離する**。バックアップ管理サーバーへのアクセスを多要素認証(MFA)で保護する。バックアップデータは必ず暗号化する。
* **Detection (検知)**: バックアップサーバーへの異常なアクセスや、大量のデータ削除、バックアップジョブの失敗を監視し、アラートを発報する。

## 注意点・補足

* **バックアップは復旧できてはじめて意味がある**: **定期的な復旧テスト**が、バックアップ戦略全体の中で最も重要かつ見過ごされがちなプロセスです。
