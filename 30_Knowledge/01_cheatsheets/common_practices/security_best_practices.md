# `security_best_practices` - 堅牢なシステムのための基本原則

## 概要

このドキュメントは、OSやプラットフォームを問わず、ITシステムをセキュアに保つための普遍的な**セキュリティベストプラクティス**をまとめたものです。これらの原則は、インフラの設計・構築から日々の運用、インシデント対応、そして攻撃者の視点に至るまで、あらゆる場面で考慮されるべき基本的な考え方です。

(出自: 一般的な情報セキュリティの原則とフレームワーク)

## 類似概念との差異

| 概念 | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| **コンプライアンス** | 法規制や業界標準（例: GDPR, PCIDSS）に準拠すること。ベストプラクティスは、コンプライアンスを満たすための技術的な基盤となります。 | - |
| **ハードニング** | OSやアプリケーションの設定を強化し、不要な機能を無効化して攻撃対象領域を減らす具体的な作業。ベストプラクティスの実践の一環です。 | - |

## 🛡️ 主要なベストプラクティス

### ⭐ 1. 最小権限の原則 (Principle of Least Privilege)

* **概要**: ユーザーやプロセスに、その業務を遂行するために**必要最小限のアクセス権限のみ**を与えるべきという原則。
* **実践**:
  * 日常業務には管理者アカウントを使わず、必要な時のみ `sudo` や「管理者として実行」で権限を昇格させる。
  * ファイルパーミッションを適切に設定し、不要なユーザーからのアクセスを制限する。
  * `sudoers` やグループポリシーで、実行可能なコマンドを厳密に定義する。

### ⭐ 2. ログの取得、監視、および保全

* **概要**: システムやアプリケーションの活動を継続的にログに記録し、それらを監視、分析することで、セキュリティイベントや異常を早期に検知し、インシデント対応に活用すること。
* **実践**:
  * **詳細なログ取得**: OSのシステムログ、認証ログ、PowerShellのスクリプトブロックロギング、Sysmonなどを有効化する。
  * **集中ログ管理 (SIEM)**: ログを中央のログサーバーに転送し、一元的に分析・監視する。
  * **ログの保全**: ログの改ざんや削除を防ぐために、ログファイルを保護し、オフサイトにバックアップする。

### ⭐ 3. 定期的なアップデートと脆弱性管理

* **概要**: OS、アプリケーション、各種ライブラリなどを常に最新の状態に保ち、既知のセキュリティ脆弱性への対策を講じること。
* **実践**:
  * **パッチ管理**: セキュリティパッチがリリースされたら、テスト後、速やかに適用する。
  * **脆弱性スキャン**: 定期的に脆弱性スキャンツールを実行し、システムの脆弱性を検出する。
  * **攻撃対象領域の最小化**: 不要なサービスやポートは停止・閉鎖する。

### 4. 強固な認証と多要素認証 (MFA)

* **概要**: ユーザー認証の強度を高め、不正なログイン試行から保護すること。
* **実践**:
  * 複雑なパスワードポリシーを強制する。
  * **多要素認証 (MFA)** を導入し、パスワード以外の認証要素を要求する。
  * パスワードはソルト付きの強力なハッシュアルゴリズムで保存する。

### 5. バックアップと復旧計画

* **概要**: データ損失に備え、定期的なバックアップと、迅速な復旧のための計画を策定・テストすること。
* **実践**:
  * **3-2-1ルール**: 3つのコピー、2種類のメディア、1つはオフサイト。
  * **不変・オフラインバックアップ**: ランサムウェア対策として、バックアップデータの改ざんを防ぐ。
  * **定期的なリストアテスト**: バックアップが確実に機能することを検証する。

## 1. アクセス制御と最小権限の原則 (Principle of Least Privilege)

* **概要**: ユーザーやプロセスに、その業務を遂行するために必要最小限のアクセス権限のみを与えるべきという原則。
* **実践**:
  * **ユーザーアカウント**: 日常業務には管理者権限を使わず、非特権ユーザーアカウントを使用する。管理者権限は必要な時のみ`sudo` (Linux/macOS) や「管理者として実行」(Windows) を用いて一時的に昇格させる。
  * **ファイル/ディレクトリ**: ファイルやディレクトリのパーミッションを適切に設定し、不必要なユーザーやグループからの書き込み・実行・読み取りを制限する。特にシステムファイルや設定ファイルは厳重に保護する。
  * **ネットワークサービス**: 必要なポートのみを開放し、不要なサービスは停止する。ファイアウォール(`ufw`, `firewalld`, Windows Defender Firewallなど)でアクセス制御を厳密に行う。
  * **`sudoers` / グループポリシー**: `sudoers`ファイルやWindowsのグループポリシーで、特権を付与するユーザーと実行可能なコマンドを厳密に定義し、ワイルドカードの使用は避ける。

## 2. 入力値の検証とサニタイズ

* **概要**: 外部からの入力(ユーザー入力、ファイルからの読み込み、ネットワーク通信など)は常に信頼せず、悪意のあるデータが含まれていないか検証し、安全な形式に変換(サニタイズ)すること。
* **実践**:
  * **シェルインジェクション対策**: スクリプトでコマンドの引数にユーザー入力を使用する場合、`bash -c "..."` や `Invoke-Expression` などで直接評価する前に、入力値をホワイトリスト方式で検証したり、引用符で囲むなどの対策を講じる。
  * **SQLインジェクション対策**: データベース操作を行う際、プリペアドステートメントを使用する。
  * **XSS (Cross-Site Scripting) 対策**: Webアプリケーション開発において、ユーザー表示前にHTMLエンティティに変換するなど。

## 3. ログの取得、監視、および保全

* **概要**: システムやアプリケーションの活動を継続的にログに記録し、それらを監視、分析することで、セキュリティイベントや異常を早期に検知し、インシデント対応に活用すること。
* **実践**:
  * **詳細なログ取得**: OSのシステムログ(`syslog`, Windowsイベントログ)、認証ログ、アプリケーションログなど、重要なログを可能な限り詳細に取得する。PowerShellのスクリプトブロックロギング、Sysmon(Windows)なども活用する。
  * **集中ログ管理 (SIEM)**: ログを集中管理システム(ELK Stack, Splunk, Graylogなど)に転送し、一元的に分析・監視する。これにより、複数のシステムにまたがる攻撃の痕跡を関連付けて検知できる。
  * **リアルタイム監視とアラート**: 不審なイベント(例: 複数回のログイン失敗、特権昇格の試行、異常なプロセス起動、大量のデータ転送)を検知した場合に、即座にアラートを発生させる仕組みを構築する。
  * **ログの保全**: ログの改ざんや削除を防ぐために、ログファイルを保護(Immutable属性、Linuxの`auditd`、Sysmonなど)し、定期的にオフサイトにバックアップする。攻撃者は自身の足跡を消そうとするため、ログの完全性は非常に重要。

## 4. 定期的なアップデートと脆弱性管理

* **概要**: OS、アプリケーション、各種ライブラリなどを常に最新の状態に保ち、既知のセキュリティ脆弱性への対策を講じること。
* **実践**:
  * **パッチ管理**: セキュリティパッチがリリースされたら、テスト環境で検証後、速やかに本番環境に適用する。
  * **脆弱性スキャン**: 定期的に脆弱性スキャンツール(Nessus, OpenVAS, Qualysなど)やSAST/DASTツール(アプリケーション向け)を実行し、システムやアプリケーションの脆弱性を検出する。
  * **不要なサービスの停止と削除**: 利用していないサービスや機能は無効化するか、完全に削除する。攻撃対象領域(Attack Surface)を最小化する。

## 5. 強固な認証と多要素認証 (MFA)

* **概要**: ユーザー認証の強度を高め、不正なログイン試行から保護すること。
* **実践**:
  * **複雑なパスワード**: 推測されにくい、長く複雑なパスワードの使用を強制するポリシーを導入する。
  * **多要素認証 (MFA)**: パスワードだけでなく、スマートフォンアプリ、ハードウェアトークン、生体認証など、複数の認証要素を組み合わせることで、アカウントの乗っ取りリスクを大幅に低減する。
  * **パスワードハッシュの安全な保存**: パスワードを保存する際は、ソルトとストレッチングを伴う強力なハッシュアルゴリズム(例: bcrypt, Argon2)を使用する。

## 6. バックアップと復旧計画

* **概要**: システムやデータの喪失(ハードウェア故障、サイバー攻撃、人為的ミスなど)に備え、定期的なバックアップと、迅速な復旧のための計画を策定・テストすること。
* **実践**:
  * **バックアップの種類**: フルバックアップ、増分バックアップ、差分バックアップを組み合わせ、RPO(目標復旧時点)とRTO(目標復旧時間)を満たすように計画する。
  * **バックアップ先の保護**: バックアップデーターを暗号化し、オフサイトやオフラインに保存することで、ランサムウェアなどによる同時暗号化や削除を防ぐ。
  * **定期的なリストアテスト**: バックアップが確実に機能することを検証するため、定期的にデータを実際に復旧させるテストを行う。

## 7. ネットワークセキュリティ

* **概要**: ネットワークの境界防御、セグメンテーション、暗号化などを用いて、通信経路やネットワークリソースを保護すること。
* **実践**:
  * **ファイアウォール**: 適切なファイアウォール(境界型、ホスト型)を配置し、不要な通信をブロックする。
  * **ネットワークセグメンテーション**: ネットワークを論理的に分割し、異なるセキュリティ要件を持つシステム間(例: DMZ, 内部ネットワーク、管理ネットワーク)の通信を厳しく制御する。
  * **VPN (Virtual Private Network)**: リモートアクセスや拠点間接続にはVPNを使用し、通信を暗号化する。
  * **IDS/IPS (侵入検知/防御システム)**: 不審なネットワークトラフィックや既知の攻撃パターンを検知・ブロックするシステムを導入する。

## 8. 各視点からの関わり

### 1. インフラ構築・運用視点

* **タスク**: 上記のベストプラクティスを、システムの設計、構築、日々の運用の全てのフェーズで実践する。
* **解説**: 安全な `sshd_config` の設定、`sudoers` の最小権限での設定、定期的な `apt update && apt upgrade` の実行、`fail2ban` の導入、ログのローテーションと監視など、このチートシート集で解説されている多くのコマンドが、これらのベストプラクティスを実装するための具体的な手段となります。

### 2. ブルーチーム視点

* **タスク**: これらのベストプラクティスが**遵守されているかを監査**し、逸脱を検知する。そして、逸脱が引き起こす**インシデントに対応**する。
* **解説**: ブルーチームは、ログ監視を通じて異常を検知し (`Failed password` の急増など)、FIMツールで設定ファイルの改ざんを発見し、脆弱性スキャンでパッチが適用されていないシステムを特定します。ベストプラクティスは、検知・対応活動の判断基準となります。

### 3. レッドチーム視点

* **タスク**: これらのベストプラクティスの**「穴」を探し出し、悪用する**。
* **解説**: レッドチームは、最小権限の原則が徹底されていない (`sudo -l` で強力な権限を発見)、パッチが適用されていない (`uname -a` で古いカーネルを発見)、ログ監視が不十分（痕跡を消去できる）、MFAが導入されていないアカウントを探すなど、ベストプラクティスが実践されていない箇所を突いて侵入を試みます。
