# `git push` - ローカルのコミットをリモートへ送信

## 概要

`git push`コマンドは、ローカルリポジトリで作成または更新されたコミット（および関連するブランチやタグ）を、リモートリポジトリへ送信します。これにより、ローカルで行った作業が共有され、他の開発者やデプロイシステムがその変更にアクセスできるようになります。Gitにおける協調作業の基本であり、ローカルの変更を公開する最終ステップです。

(出自: `git` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| ⭐ **`git pull`** | `git pull`はリモートの変更をローカルに**取得**します。`git push`はローカルの変更をリモートへ**送信**します。 | - |
| **`git fetch`** | `git fetch`はリモートの最新情報を取得しますが、ローカルの作業ブランチには変更を適用しません。 | - |

## よく連携されるコマンド

`git push` は、ローカルでの作業をコミットした後に、その成果を共有するために `git commit` と連携して使われます。

### シナリオ例: ローカルでの変更をリモートに共有する (インフラ構築・運用視点)

* **目的**: ローカルで完了した機能開発やバグ修正のコミットを、リモートリポジトリにアップロードし、他のメンバーと共有する。
* **連携コマンド**: `git add`, `git commit`
* **解説**: 変更を加え、`git add`でステージングし、`git commit`でコミットします。その後に`git push`を実行して、これらのコミットをリモートに反映させます。
* **コマンド例**:

    ```bash
    # 変更をステージ
    git add .

    # 変更をコミット
    git commit -m "feat: Add user profile management"

    # コミットされた変更をリモートリポジトリにプッシュ
    git push origin main
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **基本操作** | |
| ⭐ `<remote> <branch>` | 指定したリモートリポジトリの指定したブランチにプッシュします。（例:`git push origin main`） |
| ⭐ `-u`, `--set-upstream` | ローカルブランチとリモートブランチの追跡関係を設定します。最初のプッシュ時によく使用されます。 |
| **強制プッシュ** | |
| ⭐ `-f`, `--force` | **(危険)** リモートリポジトリの履歴を強制的に上書きします。 |
| `--force-with-lease` | `--force` と同様ですが、リモートブランチが把握している状態から変更されていない場合にのみ成功する、より安全な強制プッシュです。 |
| **その他** | |
| `--tags` | ローカルに存在するすべてのタグをリモートにプッシュします。 |
| `-d`, `--delete <branch>` | リモートリポジトリから指定したブランチを削除します。 |
| `--all` | ローカルに存在するすべてのブランチをリモートにプッシュします。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 緊急のホットフィックスで、ローカルでリベースした履歴をリモートに反映させたいが、他のメンバーの変更を壊したくない。
* **組み合わせ**: `git push --force-with-lease origin hotfix/config-issue`
* **解説**: `--force-with-lease` を使用して、他の変更を誤って上書きしないように安全に強制プッシュします。
* **例**: -

### 2. ブルーチーム視点

* **タスク**: 緊急の脆弱性パッチを、CI/CDパイプラインのテストをスキップして迅速にデプロイしたい。
* **組み合わせ**: `git push --no-verify origin main`
* **解説**: `--no-verify` を使用して pre-push フックなどを一時的にスキップし、迅速に修正をデプロイします。
* **例**:

    ```bash
    # セキュリティパッチの緊急適用のため、フックをスキップしてプッシュ
    git push --no-verify origin main
    ```

### 3. レッドチーム視点

* **タスク**: 侵害したリポジトリにバックドアを仕込み、強制的にリモートの履歴を上書きする。
* **組み合わせ**: `git push --force origin main`
* **解説**: 攻撃者は `--force` を使い、リモートの履歴を問答無用で上書きし、不正なコミットを強制的に反映させることがあります。

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `error: failed to push some refs to '<remote-url>'`
    * **考えられる原因**: 他の誰かが先にプッシュしており、ローカルとリモートで履歴が分岐しています。
    * **解決策**: まず `git pull` を実行してリモートの変更をローカルに取り込み、マージまたはリベースで履歴を統合してから再度 `git push` を試みます。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **機密情報の漏洩**。
* **悪用シナリオ**: 開発者が誤ってパスワードやAPIキーをコミットし、`git push` で公開リポジトリに送信してしまう事例が後を絶ちません。一度プッシュされた機密情報は、履歴から削除しない限り残り続けます。

### GTFOBins / LOLBAS における利用例

`git push` は直接的なコード実行機能を持たないため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: `pre-commit` フック (`git-secrets`など) を利用して、コミット前に機密情報が含まれていないか自動でチェックする。**ブランチ保護ルール**を有効にし、`main` ブランチへの強制プッシュを禁止する。
* **Detection (検知)**: CI/CDパイプラインに `TruffleHog` などのシークレットスキャンツールを導入し、プッシュされた内容を監視する。

## 注意点・補足

* **`--force`の危険性**: `--force`オプションは、他の開発者の作業を破壊する可能性があるため、共同作業を行うリポジトリでは**絶対に必要な場合以外は使用を避けるべきです**。
