# `git stash` - 作業中の変更を一時的に保存する

## 概要

`git stash`コマンドは、現在作業中のブランチにコミットしたくないが、まだ破棄したくもない変更（ステージング済みおよび未ステージングの変更）を一時的に保存するためのコマンドです。これにより、作業ディレクトリをクリーンな状態に戻し、別のブランチに切り替えたり、緊急の修正を行ったりすることが可能になります。保存された変更は**スタッシュ**(stash)としてリストに積まれ、必要なときにいつでもあとで適用できます。

(出自: `git` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| ⭐ **`git commit`** | `commit`は作業を履歴として**永続的に保存**するのに対し、`stash`は作業を**一時的に保存**します。 | - |
| **`git reset`** | `reset`はコミット履歴やステージングエリアの状態を元に戻すコマンドで、変更を**破棄**する際に使います。`stash`は変更を**退避**させるだけで、破棄はしません。 | - |

## よく連携されるコマンド

`git stash`は、開発作業中のコンテキストスイッチをスムーズにするために、さまざまなGitコマンドと連携して利用されます。

### シナリオ例: 緊急のバグ修正に対応する (インフラ構築・運用視点)

* **目的**: 現在開発中の機能を中断し、緊急のバグ修正ブランチに切り替える必要があるが、作業中の変更はまだコミットしたくない場合に、それらの変更を一時的に保存する。
* **連携コマンド**: `git checkout`, `git stash pop`
* **解説**: 作業途中の変更を`git stash`で退避させ、クリーンな状態でバグ修正ブランチに切り替えます。修正が完了した後、元のブランチに戻り、`git stash pop`で退避した変更を適用し作業を再開します。
* **コマンド例**:

    ```bash
    # 作業中の変更をメッセージ付きで一時保存
    git stash push -m "WIP: Feature X development"

    # 緊急のバグ修正ブランチに切り替え
    git checkout hotfix/critical-bug

    # (バグ修正作業...)

    # 元のブランチに戻る
    git checkout feature/X

    # 退避した変更を適用し、スタッシュから削除
    git stash pop
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **スタッシュ操作** | |
| ⭐ `push [-m <message>]` | 現在の変更をメッセージ付きでスタッシュに保存します。 |
| ⭐ `list` | 現在保存されているすべてのスタッシュのリストを表示します。 |
| ⭐ `pop [<stash>]` | 最新の（または指定した）スタッシュを適用し、そのスタッシュをリストから削除します。 |
| `apply [<stash>]` | 最新の（または指定した）スタッシュを適用しますが、リストからは削除しません。 |
| `drop [<stash>]` | 最新の（または指定した）スタッシュをリストから削除します（適用はしません）。 |
| `clear` | すべてのスタッシュをリストから削除します。**元に戻せないので注意が必要です**。 |
| `show [-p] [<stash>]` | スタッシュの内容を簡潔に（`-p`で差分形式で）表示します。 |
| **対象の拡張** | |
| ⭐ `-u`, `--include-untracked` | トラッキングされていないファイルもスタッシュに含めて保存します。 |
| `-a`, `--all` | `.gitignore`で無視されているファイルも含め、全ての変更をスタッシュに含めて保存します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: IaC (Infrastructure as Code) の変更中に、緊急で別の設定変更が必要になった。
* **組み合わせ**: `git stash push -m "Config changes before hotfix"`, `git stash pop`
* **解説**: 作業途中の変更を一時的に退避させ、修正完了後に安全に戻すために`git stash`が役立ちます。スタッシュにメッセージを付けることで、何の変更を退避したのかを明確にします。
* **例**: -

### 2. ブルーチーム視点

* **タスク**: インシデント調査のため、一時的に作成した解析スクリプトや生成されたログファイルをコミットせずに退避させたい。
* **組み合わせ**: `git stash push -u -m "Temporary analysis script"`
* **解説**: `-u` (`--include-untracked`) を使うことで、Gitで追跡されていない新しいファイルも含めて一時的に退避させ、迅速に作業コンテキストを切り替えられます。
* **例**:

    ```bash
    # 解析スクリプトの変更と、その実行で生成された一時的なログファイルを一緒にスタッシュ
    git stash push -u -m "Temporary analysis script and generated logs"
    ```

### 3. レッドチーム視点

* **タスク**: 侵害したシステム上で、作業中の痕跡（トラッキングされていないツールやペイロードなど）を一時的に隠蔽する。
* **組み合わせ**: `git stash push --all`
* **解説**: 作業ディレクトリ内の全ての変更（untracked/ignoredファイル含む）をスタッシュに保存し、そのあとで削除する可能性があります。
* **例**: -

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `error: Your local changes to the following files would be overwritten by merge:`
    * **考えられる原因**: `git stash pop` しようとしたときに、現在のブランチの同じファイルに未コミットの変更があり、スタッシュを適用するとそれらの変更が上書きされてしまうためです。
    * **解決策**: 現在の未コミットの変更をコミットするか、別のスタッシュに保存してから、再度 `git stash pop` を試みます。

## 環境変数と設定ファイル

* `git stash` に直接影響を与える独自の環境変数はありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **機密情報の残存**。
* **悪用シナリオ**: 開発者が誤って機密情報（パスワード、APIキーなど）をファイルに書き込んだ場合、それらのファイルが `git stash -u` などでスタッシュに保存されることがあります。そのリポジトリが後に共有された場合、スタッシュ内の機密情報が漏洩するリスクがあります。

### GTFOBins / LOLBAS における利用例

`git stash` は直接的なコード実行機能を持たないため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: コード中に機密情報を直接記述しない。`.gitignore` を適切に設定し、機密情報が含まれる可能性のあるファイルが誤ってスタッシュに含まれないようにする。
* **Detection (検知)**: 共有リポジトリで、重要な情報がスタッシュ内に残されていないか、定期的に監査を行う仕組みを検討する。

## 注意点・補足

* **コミットされていない変更のみ**: `git stash`は、コミットされていない変更のみを対象とします。
* **メッセージの重要性**: `git stash push -m "message"` のようにメッセージを付けることで、複数のスタッシュがある場合にどれを適用すべきか迷わずに済みます。
