# `git clone` - リモートリポジトリをローカルに複製する

## 概要

`git clone` コマンドは、リモートGitリポジトリの完全なコピーをローカルマシンに作成するために使用されます。このコマンドは、リポジトリの全履歴（すべてのバージョン、ブランチ、タグなど）と、ローカルで作業を開始するために必要なすべての設定ファイル(`.git`ディレクトリ)を取得します。新しいプロジェクトを開始する際や、既存のプロジェクトに参加する際に、最初に実行するGitコマンドの1つです。

(出自: `git` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| ⭐ **`git init`** | **新しい空の**リポジトリをローカルで作成します。`git clone` は**既存のリモート**リポジトリを複製します。 | - |
| **`git pull` / `fetch`** | **既存のローカルリポジトリ**をリモートの最新状態に更新します。`clone` はゼロから新しいローカルリポジトリを作成します。 | - |

## よく連携されるコマンド

`git clone` は、新しいプロジェクトの開始点となるため、その後に続くさまざまなGitコマンドと連携します。

### シナリオ例: リモートリポジトリをローカルに取得し、開発を開始する (インフラ構築・運用視点)

* **目的**: GitHubやGitLabなどのリモートリポジトリからプロジェクトのコードベースをローカルマシンにダウンロードし、開発作業を始められる状態にする。
* **連携コマンド**: `cd`, `git status`, `git branch`
* **解説**: `git clone` を実行すると、リモートリポジトリのコピーが指定したディレクトリに作成されます。その後、そのディレクトリに移動し、作業を開始します。
* **コマンド例**:

    ```bash
    # リモートリポジトリをカレントディレクトリにクローンする
    git clone [https://github.com/your-org/your-project.git](https://github.com/your-org/your-project.git)

    # クローンしたディレクトリに移動
    cd your-project

    # 現在のブランチなどを確認
    git status
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **基本操作** | |
| ⭐ `<repository_url>` | クローンしたいリモートリポジトリのURL。 |
| ⭐ `-b <branch>` | 指定したブランチのみをチェックアウトしてクローンします。 |
| **履歴の最適化** | |
| ⭐ `--depth <N>` | リポジトリの履歴を最新の `N` 件のみに制限してクローンします（**シャロークローン**）。 |
| `--single-branch` | `--branch` で指定したブランチのみをクローンし、他のブランチは無視します。 |
| **その他** | |
| `--recursive` | サブモジュールが含まれている場合、それらも同時にクローンします。 |
| `--bare` | 作業ディレクトリなしで、Gitリポジトリ (`.git`の中身のみ) をクローンします。サーバー間のミラーリングなどに使います。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: CI/CDパイプライン上で、デプロイに必要なコードのみを最小限の履歴で取得する。
* **組み合わせ**: `git clone --depth 1 --single-branch -b <branch-name> <repo-url>`
* **解説**: `--depth 1` で最新のコミットのみを取得することで、クローン時間を短縮し、サーバーのディスク使用量を抑えます。IaCリポジトリのデプロイなど、最新の状態だけが必要な場合に非常に有効です。
* **例**:

    ```bash
    # release/production ブランチの最新コミットのみをクローン
    git clone --depth 1 --single-branch -b release/production [https://github.com/my-infra/my-iac.git](https://github.com/my-infra/my-iac.git)
    ```

### 2. ブルーチーム視点

* **タスク**: 悪意のある可能性のあるスクリプトが自動実行されることを防ぎつつ、リポジトリの履歴を安全に調査する。
* **組み合わせ**: `git clone --no-checkout <repo-url>`
* **解説**: `--no-checkout` を使うことで、作業ディレクトリにファイルを展開せず、`.git` ディレクトリ内のオブジェクトと履歴のみを安全にダウンロードします。これにより、リポジトリに含まれる可能性のある悪意のあるフックなどの自動実行を防ぎます。
* **例**:

    ```bash
    # ファイルを展開せずにリポジトリをクローン
    git clone --no-checkout [https://github.com/suspicious/tool-kit.git](https://github.com/suspicious/tool-kit.git)
    cd tool-kit
    # git log や git cat-file などで安全に内部を調査
    ```

### 3. レッドチーム視点

* **タスク**: 攻撃に必要なツールキットやエクスプロイトコードを、依存関係も含めて完全に取得する。
* **組み合わせ**: `git clone --recursive <repo-url>`
* **解説**: 攻撃ツールは、他のGitリポジトリをサブモジュールとして依存していることがよくあります。`--recursive` を使うことで、これらのツールキットを完全な形で取得し、すぐに使用できる状態にします。
* **例**:

    ```bash
    # 脆弱性検証用のWebアプリケーションリポジトリを、サブモジュールも含めてクローン
    git clone --recursive [https://github.com/vulnerable/web-app.git](https://github.com/vulnerable/web-app.git)
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `fatal: repository '<repo-url>' not found`
    * **考えられる原因**: 指定したリポジトリのURLが間違っている、リポジトリが存在しない、またはアクセス権限がありません。
    * **解決策**: URLが正しいか、リポジトリがプライベートでないか、認証情報（SSHキー、Personal Access Tokenなど）が正しく設定されているか確認します。

2. **エラーメッセージ例 2**: `fatal: destination path '<directory-name>' already exists and is not an empty directory.`
    * **考えられる原因**: クローン先のディレクトリがすでに存在し、空ではありません。
    * **解決策**: 別のディレクトリ名を指定するか、既存のディレクトリの内容を削除してから再度クローンします。

## 環境変数と設定ファイル

* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

### 関連する設定ファイル

* **`~/.gitconfig`**: グローバル設定ファイル。`proxy` 設定などが `clone` に影響する可能性があります。
* **`~/.ssh/config`**: SSHプロトコルでクローンする際の、ホストごとの設定が影響します。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: 古いバージョンのGitでは、悪意のあるリポジトリをクローンした際に、`post-checkout` などのクライアントサイドフックが自動的に実行されてしまう脆弱性が存在しました (CVE-2014-9390など)。
* **悪用シナリオ**: 攻撃者が侵害したGitリポジトリにマルウェアをコミットし、開発者がそれをクローンすると、悪意のあるコードがローカルマシンにダウンロードされます。

### GTFOBins / LOLBAS における利用例

`git clone` はLOLBASやGTFOBinsにはリストされていません。これは、このコマンドが直接的なペイロード実行や特権昇格に用いられるものではないためです。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **常に最新のGitバージョンを使用する**。信頼できないソースからのリポジトリは安易にクローンしない。
* **Detection (検知)**: Gitホスティングサービスの監査ログを監視し、予期せぬリポジトリのクローンやアクセスがないかを確認する。

## 注意点・補足

* **SSH vs HTTPS**: リモートリポジトリのクローンには、主にHTTPSとSSHの2つのプロトコルが使用されます。HTTPSは設定が簡単ですが、認証情報の入力が求められることがあります。SSHはSSHキーペアの設定が必要ですが、一度設定すればパスワード入力なしで利用でき、自動化に適しています。
