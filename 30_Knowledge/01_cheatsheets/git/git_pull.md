# `git pull` - リモートリポジトリの変更をローカルに取り込む

## 概要

`git pull` コマンドは、リモートリポジトリの変更をローカルリポジトリに取り込むために使用します。これは、実質的に以下の2つの操作を組み合わせたものです。

1. **`git fetch`**: リモートリポジトリから最新の変更をダウンロードします。
2. **`git merge`**: `git fetch`で取得した変更を、現在のローカルブランチに統合（マージ）します。

つまり、「リモートの変更を取得して、すぐに自分のブランチに反映させる」コマンドです。

(出自: `git` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| ⭐ **`git fetch`** | `git fetch` はリモートの変更を**取得するだけ**で、現在のローカルブランチには影響を与えません。`git pull` は取得した変更を自動的にマージします。まず変更内容を確認してから統合したい場合は `git fetch` が安全です。 | `git fetch` (安全な確認) |
| **`git merge`** | `git merge` は、**すでにローカルに存在する**ブランチの変更を統合します。`git pull` はリモートから取得した変更をマージします。 | - |

## よく連携されるコマンド

`git pull` は、ローカルに未コミットの変更がある場合に競合を避けるため、`git stash` と連携することがあります。

### シナリオ例: 作業中の変更を退避させてからプルする (インフラ構築・運用視点)

* **目的**: 監視スクリプトを修正している途中で、緊急のセキュリティパッチがリモートからプッシュされた。自分の作業を失わずに最新のパッチを取り込みたい。
* **連携コマンド**: `git stash`
* **解説**: ローカルにコミットされていない変更がある状態で `git pull` を実行すると、競合が発生する可能性があります。安全策として、一時的に変更を退避 (`git stash`) してからプルし、あとで退避した変更を元に戻します (`git stash pop`)。
* **コマンド例**:

    ```bash
    # 未コミットの変更を一時退避
    git stash save "Working on new log parser"

    # リモートの変更をプル
    git pull origin main

    # 退避した変更を元に戻す
    git stash pop
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **基本操作** | |
| ⭐ `<remote> <branch>` | プルするリモートリポジトリ名とブランチ名を指定します。（例: `origin main`） |
| **マージ戦略** | |
| ⭐ `--rebase` | マージではなく**リベース**を使用します。これにより、ローカルのコミット履歴がリモートの最新コミットの上に再適用され、履歴が直線的になります。 |
| `--ff-only` | マージがファストフォワード（履歴が直線的に進む）である場合にのみプルを実行します。 |
| **その他** | |
| `--no-commit` | 変更をマージしますが、自動的にコミットしません。 |
| `-v`, `--verbose` | より詳細な情報を出力します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 朝、作業を始める前に、リモートの `main` ブランチの最新のインフラ変更を取り込んでおく。
* **組み合わせ**: `git pull`
* **解説**: 日常的な作業で、メインブランチの最新状態をサッと取得したい場合に最も使われます。
* **例**:

    ```bash
    git checkout main
    git pull
    ```

### 2. ブルーチーム視点

* **タスク**: 意図しないマージコミットの発生を防ぎ、常にクリーンで直線的な履歴を維持したい。
* **組み合わせ**: `git pull --ff-only`
* **解説**: リモートの変更をファストフォワードマージできる場合にのみプルします。もしマージコミットが必要な競合状態になる場合は、プルを中断します。自動デプロイメントパイプラインなどで、予期せぬ競合による失敗を防ぐためにも利用されます。
* **例**:

    ```bash
    git pull --ff-only origin main
    ```

### 3. レッドチーム視点

* **タスク**: チーム内で共有されているツールリポジトリを最新の状態に保つ。
* **組み合わせ**: `git pull origin <tool_branch>`
* **解説**: チーム内の別のメンバーが、特定のブランチに新しいツールやスクリプトをプッシュした場合、`git pull` でその変更を自身の作業ブランチに統合します。
* **例**:

    ```bash
    # 自身の作業ブランチに、リモートの 'dev_tools_update' ブランチの変更を取り込む
    git pull origin dev_tools_update
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `error: Your local changes to the following files would be overwritten by merge:`
    * **考えられる原因**: ローカルのワーキングディレクトリに未コミットの変更があり、それらがリモートからプルする変更と競合しています。
    * **解決策**:
        * 現在の変更をコミットする: `git commit -m "WIP"`
        * 現在の変更を一時退避する: `git stash`
        * 現在の変更を破棄する: `git restore .` (慎重に)

2. **エラーメッセージ例 2**: `Merge conflict in <file>`
    * **考えられる原因**: ローカルの変更とリモートの変更が同じファイルの同じ箇所を変更しており、Gitが自動的にマージできませんでした。
    * **解決策**:
        1. `git status` で競合しているファイルを確認します。
        2. 競合ファイルをエディタで開き、`<<<<<<<`, `=======`, `>>>>>>>` のマーカーを参考に手動で編集・解決します。
        3. `git add <file>` で解決したファイルをステージングし、`git commit` でマージを完了させます。

## 環境変数と設定ファイル

* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

### 関連する設定ファイル

* **`.git/config`**: `pull.rebase` を `true` に設定すると、`git pull` のデフォルト動作がリベースになります。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **悪意のあるコードの混入**。
* **悪用シナリオ**: 信頼できないソースからプルしたリポジトリ、または侵害されたリモートリポジトリから、悪意のあるコードやバックドア、マルウェアがローカルシステムにダウンロードされ実行されてしまう可能性があります。

### GTFOBins / LOLBAS における利用例

`git pull` は直接的なコード実行機能を持たないため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 信頼できるソースからのプルのみを行う。Gitのバージョンを常に最新に保つ。
* **Detection (検知)**: プル後にビルドやスクリプトが自動実行される環境では、CI/CDパイプラインにセキュリティスキャン（静的解析、依存関係スキャンなど）を組み込む。

## 注意点・補足

* `git pull` は便利ですが、競合が発生しやすいコマンドです。競合を避けたい場合は、まず `git fetch` でリモートの変更を確認し、その後手動で `git merge` または `git rebase` を行うのがより安全なワークフローです。
