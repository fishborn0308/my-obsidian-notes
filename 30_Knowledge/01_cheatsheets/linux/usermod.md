# `usermod` - 既存のユーザーアカウントを変更する

## 概要

`usermod` (user modify) は、既に存在するユーザーアカウントの属性（所属グループ、ログインシェル、ホームディレクトリ、ユーザー名、UIDなど）を変更するためのコマンドです。ユーザーに新しい権限（グループ）を付与したり、ログインを無効化したりと、日々のユーザー管理業務において中心的な役割を果たします。

このコマンドの実行には `root` 権限が必要です。

(出自: `shadow-utils` または `passwd` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| ⭐ **`useradd`** | **新しい**ユーザーアカウントを作成します。 | - |
| **`userdel`** | ユーザーアカウントを**削除**します。 | - |
| **`gpasswd`** | `-a <user> <group>` でユーザーをグループに追加するなど、グループ管理に特化しています。`usermod` はより汎用的です。 | - |

## よく連携されるコマンド

`usermod` でグループに追加した後、`id` コマンドで変更が正しく適用されたかを確認するのが一般的なワークフローです。

### シナリオ例: ユーザーをグループに追加し、確認する (インフラ構築・運用視点)

* **目的**: 開発者 `devuser` がDockerコマンドを実行できるように、`docker` グループに追加し、その結果を確認する。
* **連携コマンド**: `id`
* **解説**: `usermod -aG` でユーザーをグループに追加します。`-a` (`--append`) を忘れると他のグループから削除されてしまうため注意が必要です。変更後、ユーザーが再ログインすると新しいグループが有効になりますが、`id` コマンドで即座に定義を確認できます。
* **コマンド例**:

    ```bash
    # ユーザー 'devuser' を docker グループに追加
    sudo usermod -aG docker devuser

    # 変更が反映されたか確認 (groups=... の中にdockerが含まれているか)
    id devuser
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **グループ所属** | |
| ⭐ `-G <GROUPS>` | ユーザーが所属する**セカンダリ（追加）グループ**をカンマ区切りで指定します。 |
| ⭐ `-a`, `--append` | `-G` と組み合わせて使用し、ユーザーをグループに**追加**します。**これを付けないとグループが上書きされるため注意**。 |
| `-g <GROUP>` | ユーザーの**プライマリグループ**を変更します。 |
| **ユーザー属性** | |
| ⭐ `-s <SHELL>` | ユーザーのログインシェルを変更します。`/sbin/nologin` に設定してログインを無効化するためによく使います。 |
| `-l <NEW_NAME>` | ユーザーのログイン名を変更します。 |
| `-d <DIR>` | ユーザーのホームディレクトリを新しいパスに変更します。`-m` オプションを付けると中身も移動します。 |
| **アカウント状態** | |
| ⭐ `-L`, `--lock` | ユーザーのアカウントをロックします (`passwd -l` と同義)。 |
| ⭐ `-U`, `--unlock` | ロックされたアカウントを解除します (`passwd -u` と同義)。 |
| `-u <UID>` | ユーザーのUIDを変更します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 既存の一般ユーザーに管理者権限を付与する。
* **組み合わせ**: `sudo usermod -aG <sudo_group> <username>`
* **解説**: ユーザーを `sudo` または `wheel` グループに追加することで、`sudo` コマンドの実行を許可します。
* **例**:

    ```bash
    # ユーザー 'devuser' を sudo グループに追加 (Debian/Ubuntu系)
    sudo usermod -aG sudo devuser

    # ユーザー 'devuser' を wheel グループに追加 (RHEL/CentOS系)
    sudo usermod -aG wheel devuser
    ```

### 2. ブルーチーム視点

* **タスク**: 侵害されたアカウントの権限を剥奪し、無力化する（封じ込め）。
* **組み合わせ**: `sudo usermod -G <new_group_list> <compromised_user>`
* **解説**: 侵害されたユーザーから、`-a` を付けずに `-G` を使って所属グループを一般グループのみに上書きすることで、`sudo` や `docker` などの全ての特権グループから一度に削除し、権限を剥奪します。
* **例**:

    ```bash
    # user1 の所属グループを 'users' のみに限定する
    sudo usermod -G users user1
    ```

### 3. レッドチーム視点

* **タスク**: **隠密な権限昇格**として、一般ユーザーのUIDを `0` (rootのUID) に変更する。
* **組み合わせ**: `sudo usermod -u 0 <compromised_user>`
* **解説**: 攻撃者は、ユーザー名を `root` に変えるのではなく、一般ユーザー (`devuser`など) のUIDを `0` に変更します。`/etc/passwd` を一見しただけでは普通のユーザーに見えますが、このアカウントは `root` と全く同じ権限を持つため、検知されにくい強力なバックドアとなります。
* **例**:

    ```bash
    # 'devuser' のUIDを0に変更
    sudo usermod -u 0 devuser
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **現象**: **ユーザーをグループに追加したが、権限が反映されない。**
    * **考えられる原因**: グループメンバーシップの変更は、**新しいログインセッション**を開始したときに初めて有効になります。
    * **解決策**: 対象のユーザーに、一度ログアウトしてから再度ログインするように依頼してください。`su - <username>` でも新しいセッションを開始できます。

## 環境変数と設定ファイル

* `usermod` の動作に直接影響する環境変数は通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

### 関連する設定ファイル

* **`/etc/passwd`, `/etc/shadow`, `/etc/group`**: `usermod` はこれらのファイルを更新します。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `usermod` 自体ではなく、**不適切な権限付与**がリスクとなります。
* **悪用シナリオ**: 攻撃者がroot権限を奪取した後、`usermod -aG sudo` で自身が制御するアカウントを特権グループに追加したり、`usermod -u 0` で隠しrootアカウントを作成したりして、永続化を図ります。

### GTFOBins / LOLBAS における利用例

* **機能**: **Privilege Escalation** (権限昇格)
* **解説**: もしユーザーが `sudo usermod` を実行できる権限を持っている場合、自身を `sudo` グループに追加 (`-aG`) することで、完全な管理者権限を奪取できます。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 最小権限の原則に従い、ユーザーを不要な特権グループに追加しない。`sudoers` の設定を厳格にする。
* **Detection (検知)**: `/etc/group` ファイルへの変更をFIMツールで監視する。`auditd` で `usermod` コマンドの実行を監視し、意図しない権限昇格がないかを確認する。

## 注意点・補足

* **`-aG` と `-G` の違い**: ユーザーを新しいグループに**追加**したい場合は `-aG`、所属グループを**完全に置き換えたい**場合は `-G` を使います。この違いを理解しないと、意図せず権限を剥奪してしまう可能性があります。
