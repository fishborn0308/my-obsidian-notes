# `su` - ユーザーを切り替える

## 概要

`su` (substitute user または switch user) は、現在のターミナルセッションで、別のユーザーアカウントに切り替えるためのコマンドです。ユーザー名を指定しない場合は、デフォルトで **`root` (スーパーユーザー)** に切り替わろうとします。

切り替え先のユーザーのパスワードを入力することで認証が行われます。主に、一般ユーザーから `root` になってシステム全体の管理作業を行うためや、特定のサービスアカウントになりきってアプリケーションの動作確認を行うために利用されてきました。

(出自: `shadow-utils` または `util-linux` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| ⭐ **`sudo`** | `sudo` は**自身のパスワード**を使って、許可されたコマンドを一時的に別ユーザーとして実行します。`su` は**相手のパスワード**が必要です。監査と権限の最小化の観点から `sudo` が圧倒的に優れています。 | `sudo` |
| **`login`** | 新しいログインセッションを開始します。通常は物理コンソールから利用されます。`su` は既存のセッション内でユーザーを切り替えます。 | - |
| **`runuser`** | `su` と似ていますが、主にスクリプト内から、シェルを起動せずに特定のユーザーでコマンドを実行するために使われます。 | - |

## よく連携されるコマンド

`su` は、ログの確認や権限昇格の試みの調査において、`grep` や `journalctl` と連携して使われます（防御側の視点）。

### シナリオ例: `su`の実行履歴を監査ログから調査する (ブルーチーム視点)

* **目的**: システム上で`su`コマンドがいつ、誰によって実行されたかを確認する。
* **連携コマンド**: `grep`, `journalctl`
* **解説**: `su` の実行はセキュリティ上重要なイベントであり、監査ログに記録されます。`grep` で "su:" というキーワードを検索することで、`su` の成功・失敗の履歴を追跡できます。
* **コマンド例**:

    ```bash
    # Debian/Ubuntu系の認証ログからsuの実行履歴を検索
    sudo grep "su:" /var/log/auth.log

    # systemd環境でジャーナルから検索
    sudo journalctl | grep "session opened for user root"
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **シェル環境** | |
| ⭐ `-`, `-l`, `--login` | **ログインシェル**としてセッションを開始します。環境変数を完全に切り替え先のユーザーのものにリセットします。**`su -` の形式が強く推奨されます**。 |
| **コマンド実行** | |
| ⭐ `-c <command>` | 新しいシェルを起動せず、指定したコマンドだけをそのユーザーとして実行して終了します。 |
| `-s <shell>` | 指定したシェルを起動します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: `root`ユーザーの完全な環境で管理作業を行う。
* **組み合わせ**: `su -`
* **解説**: `sudo` が設定されていない古いシステムや、レスキューモードで作業する際に、`root` のパスワードを入力してスーパーユーザーになります。`-` を付けることで、`root` の `PATH` やプロファイルが正しく読み込まれます。
* **例**:

    ```bash
    # rootユーザーに切り替え
    su -
    ```

### 2. ブルーチーム視点

* **タスク**: `su`コマンドの実行履歴を監査ログから調査し、権限昇格の試みがないか確認する。
* **組み合わせ**: `grep "su:" /var/log/auth.log`
* **解説**: `su: PAM authentication failure` ログの大量発生は、攻撃者がパスワードをブルートフォースしている兆候を示します。`pam:session opened for user root` は権限昇格が成功したことを意味します。
* **例**:

    ```bash
    # suの認証失敗ログを検索
    sudo grep "su: PAM authentication failure" /var/log/auth.log
    ```

### 3. レッドチーム視点

* **タスク**: 窃取したパスワードを使い、権限昇格または横展開を行う。
* **組み合わせ**: `su - <username>`
* **解説**: 攻撃者はシステム侵入後、何らかの方法で `root` や他のユーザーのパスワードを入手した場合、`su` を使ってそのユーザーになりすまします。`root` になれればシステムを完全に掌握でき、他の一般ユーザーになれれば、そのユーザーがアクセスできるファイルを探索します。
* **例**:

    ```bash
    # 窃取したrootのパスワードでrootになる
    su -
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `su: Authentication failure`
    * **考えられる原因**: 間違ったパスワードを入力しました。
    * **解決策**: 正しいパスワードを再入力してください。

2. **現象**: **`su` は成功したが、元のユーザーの `PATH` が引き継がれてコマンドが見つからない。**
    * **考えられる原因**: `-` を付けずに `su` を実行したため、非ログインシェルとなり、環境変数が完全にリセットされませんでした。
    * **解決策**: `exit` して、**`su -`** を使用してください。

## 環境変数と設定ファイル

* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `su` 自体ではなく、**`root`パスワードの共有**が最大のリスクです。
* **悪用シナリオ**: `su` を使って複数人で管理作業を行う場合、`root` のパスワードを共有する必要が生じます。これは「誰が何をしたか」の追跡を困難にし、セキュリティレベルを著しく低下させます。

### GTFOBins / LOLBAS における利用例

* **機能**: **Privilege Escalation**
* **解説**: もしユーザーが `sudo su` を実行できる権限を持っている場合、それは実質的にパスワードなしでrootシェルを取得できる権-限と同じです。これは `sudo` の設定ミスとしてよく見られます。
* **参照**: `https://gtfobins.github.io/gtfobins/su/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **`sudo` を全面的に利用し、`root` のパスワードは無効化（または誰も知らない状態に）する**。`wheel` グループなどを利用し、`su` で `root` になれるユーザーを制限する。
* **Detection (検知)**: `su` の実行、特に認証失敗を監査ログで厳重に監視する。

## 注意点・補足

* **`sudo` の利用を最優先**: チームでサーバーを管理する近代的な環境では、全ての管理作業は `sudo` 経由で行うのが標準的なベストプラクティスです。`su` は、シングルユーザーモードでの修復作業など、限定的な状況でのみ使用すべきです。
