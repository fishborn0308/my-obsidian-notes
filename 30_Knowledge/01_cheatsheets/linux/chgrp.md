# `chgrp` - ファイルやディレクトリのグループ所有者を変更する

## 概要

`chgrp` (change group) は、ファイルやディレクトリの**グループ所有者**を変更するためのコマンドです。Linuxのパーミッションモデルでは、各ファイルに「ユーザー所有者」「グループ所有者」「その他」の3つのカテゴリで権限が設定されます。`chgrp` はこのうち「グループ所有者」を変更することで、特定のグループに所属するユーザー群に対して、ファイルのアクセス権をまとめて管理できるようにします。

通常、この操作はファイルの所有者または`root`ユーザーのみが実行できます。

(出自: `coreutils` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| **`chown`** | ユーザー所有者とグループ所有者の**両方**を変更できます。`chown :group <file>` のようにグループだけでも変更可能で、より多機能です。 | `chown` |
| **`chmod`** | ファイルの**パーミッション（権限モード）**を変更します。「誰が所有者か」ではなく「その所有者が何ができるか」を変更します。 | - |

## よく連携されるコマンド

`chgrp` は、共有ディレクトリの権限を設定する際に、`mkdir` や `chmod` と連携して使われることがよくあります。

### シナリオ例: 共有ディレクトリの権限を設定する (インフラ構築・運用視点)

* **目的**: 複数の開発者が書き込める共有ディレクトリを作成する。
* **連携コマンド**: `mkdir`, `chgrp`, `chmod`
* **解説**: まずディレクトリを作成し、`chgrp` でグループ所有者を開発者グループに設定します。最後に `chmod` でSGIDビットとグループ書き込み権限を付与することで、そのディレクトリ内で作成されるファイルは自動的に開発者グループに所属し、グループメンバーは書き込みができるようになります。
* **コマンド例**:

    ```bash
    # このコマンドシーケンスで共有ディレクトリをセットアップする
    sudo mkdir /srv/shared-project
    sudo chgrp developers /srv/shared-project
    sudo chmod 2775 /srv/shared-project # SGIDビット(2) + rwxrwxr-x (775)
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **ファイル/ディレクトリ操作** | |
| ⭐ `-R`, `--recursive` | ディレクトリとその中にある全てのファイル・サブディレクトリのグループを**再帰的に**変更します。 |
| **出力制御** | |
| `-v`, `--verbose` | 処理した全てのファイルについて、変更内容を詳細に表示します。 |
| `-c`, `--changes` | 実際にグループが**変更された**ファイルについてのみ表示します。 |
| **その他** | |
| `--reference=<RFILE>` | グループ名を直接指定する代わりに、参照ファイル `<RFILE>` と同じグループに変更します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: Webサーバのドキュメントルート以下のグループ所有者を再帰的に設定する。
* **組み合わせ**: `sudo chgrp -R <group_name> <directory>`
* **解説**: 開発者がコンテンツをアップロードした後、Webサーバプロセス (`www-data`など) がそれらのファイルを読み取れるように、ディレクトリ全体のグループを `www-data` に設定します。
* **例**:

    ```bash
    # /var/www/html ディレクトリ以下の全ファイルのグループを www-data に変更
    sudo chgrp -R www-data /var/www/html
    ```

### 2. ブルーチーム視点

* **タスク**: 不審なグループが所有するファイルを正規の状態に修復する。
* **組み合わせ**: `find / -group <unusual_group> -exec sudo chgrp <safe_group> {} +`
* **解説**: システム監査中に、攻撃者が作成した可能性のある不審なグループ (`hackergroup` など) を発見した場合、`find` でそのグループに所属するファイルを全て検索し、`chgrp` を使ってそれらのグループを `root` など安全なものに戻し、無力化します。
* **例**:

    ```bash
    # GID 1337 (不審なグループ) が所有するファイルを全て探し、グループをrootに変更
    sudo find / -gid 1337 -exec chgrp root {} +
    ```

### 3. レッドチーム視点

* **タスク**: 機密ファイルへのアクセス経路を確保するためにグループ所有者を変更する。
* **組み合わせ**: `sudo chgrp <controlled_group> /etc/shadow`
* **解説**: 権限昇格後、`/etc/shadow` (パスワードハッシュファイル) のグループを、自身が制御する別のユーザーが所属するグループ (`adm` や `docker` など) に変更します。`/etc/shadow` のパーミッションが `640` であれば、`adm` グループのメンバーとしてファイルを読み取ることが可能になり、より隠密な情報窃取に繋がります。
* **例**:

    ```bash
    # /etc/shadow のグループを 'adm' に変更
    sudo chgrp adm /etc/shadow
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `chgrp: invalid group: ‘<group_name>’`
    * **考えられる原因**: 指定したグループ名がシステムに存在しません。
    * **解決策**: `getent group` や `cat /etc/group` コマンドで、システムに存在するグループの一覧を確認し、正しいグループ名を指定してください。

## 環境変数と設定ファイル

* `chgrp` の動作に直接影響する環境変数は通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `chgrp` 自体ではなく、**不適切なグループ所有権の設定**がセキュリティリスクとなります。
* **悪用シナリオ**: システムファイルや設定ファイルのグループ所有権を、攻撃者が制御するグループに変更されると、設定の改ざんや機密情報の窃取に繋がる可能性があります。

### GTFOBins / LOLBAS における利用例

`chgrp` は、`chown` と同様のコンテキストで権限昇格に悪用される可能性があります。

* **機能**: **Privilege Escalation**
* **解説**: もしユーザーが `sudo chgrp` を実行できる権限を持っている場合、`/etc/shadow` のような機密ファイルのグループを自身が所属するグループに変更し、パーミッションによってはそのファイルを読み取ることが可能になります。
* **参照**: `chown` のGTFOBinsエントリが参考になります: `https://gtfobins.github.io/gtfobins/chown/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: ファイルやディレクトリのグループ所有権を最小権限の原則にしたがって設定する。`sudoers` の設定を厳格にし、`chgrp` のようなコマンドに安易に `sudo` 権限を付与しない。
* **Detection (検知)**: ファイル整合性監視 (FIM) ツールで、重要なシステムファイルの所有者変更を監視する。

## 注意点・補足

* **`chown`による代替**: `chown :<group_name> <file>` のように、`chown` コマンドでグループ所有者だけを変更できます。多くの管理者は、`chown` の方が多機能であるため好んで使う傾向にあります。
* **再帰的変更の注意**: `-R` オプションは非常に強力ですが、システムの重要なディレクトリ（`/etc` や `/usr` など）に対して誤って実行すると、システムのパーミッションが破壊され、動作しなくなる可能性があります。実行前には、対象のパスが正しいことを十分に確認してください。
