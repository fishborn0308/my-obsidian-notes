# `dmesg` - カーネルのリングバッファメッセージを表示する

## 概要

`dmesg` (display message) は、Linux**カーネルのリングバッファ**に記録されたメッセージを表示するためのコマンドです。リングバッファとは、カーネルが自身の動作状況（ハードウェアの認識、ドライバのロード、エラー、警告など）を記録しておくためのメモリ上の領域です。`dmesg` は、特にシステムの起動プロセス中や、新しいハードウェアを接続した際に何が起きているかを低レベルで確認するための、最も基本的なトラブルシューティングツールの1つです。

(出自: `util-linux` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| **`journalctl -k`** | `systemd` 環境で `dmesg` とほぼ同じ情報を表示します。`journalctl` は、より高機能で、永続化されたログの中からフィルタリングや時系列での検索が可能です。 | `journalctl -k` |
| **`tail /var/log/syslog`** | カーネルメッセージだけでなく、アプリケーションなどを含むシステム全体のログを表示します。`dmesg` はカーネルメッセージに特化しています。 | - |
| **`/var/log/dmesg`** | システム起動時の `dmesg` の内容が保存されている静的なファイルです。`dmesg` コマンドは現在のメモリ上のリングバッファの内容を動的に表示します。 | `dmesg` (現在の情報) |

## よく連携されるコマンド

`dmesg` の出力は非常に多いため、`grep` や `less` といったテキスト処理コマンドと連携して、目的の情報を絞り込むのが一般的です。

### シナリオ例: ハードウェア関連のトラブルシューティング (インフラ構築・運用視点)

* **目的**: サーバーに接続したUSBディスクが認識されない原因を調査する。
* **連携コマンド**: `grep`
* **解説**: `dmesg` の全出力の中から、`grep` を使って "usb" というキーワードを含む行だけをフィルタリングします。これにより、USBデバイスの認識、ドライバのロード、エラーに関するメッセージだけを効率的に確認できます。
* **コマンド例**:

    ```bash
    # USBに関連するカーネルメッセージのみをフィルタリングして表示
    dmesg | grep -i "usb"
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **表示形式** | |
| ⭐ `-H`, `--human` | 人間が読みやすい形式で出力します。メッセージのレベルに応じて色付けされ、タイムスタンプも見やすくなります。 |
| ⭐ `-T`, `--ctime` | タイムスタンプを標準的な日時の形式（例: `Fri Sep 22 10:30:00 2025`）で表示します。 |
| **動作制御** | |
| ⭐ `-w`, `--follow` | 新しいメッセージがバッファに追加されるのを待ち受け、リアルタイムで表示し続けます (`tail -f` と同様)。 |
| `-l`, `--level <list>` | 表示するメッセージのレベルを絞り込みます。`<list>` には `err`, `warn`, `info`, `debug` などを指定します。 |
| `-c`, `--read-clear` | リングバッファの内容を全て表示した後、バッファをクリアします。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 起動後から現在までの全てのカーネルメッセージを、人間が読めるタイムスタンプ付きで確認する。
* **組み合わせ**: `dmesg -T | less`
* **解説**: `-T` でタイムスタンプを読みやすく変換し、その長い出力を `less` に渡してページごとに閲覧します。ハードウェア認識の失敗や、ディスクI/Oエラー (`I/O error`) などの調査に不可欠です。
* **例**:

    ```bash
    # SCSIディスク(sda)に関するエラーがないか確認
    dmesg -T | grep -i "sda"
    ```

### 2. ブルーチーム視点

* **タスク**: システム上で発生するカーネルレベルのイベントをリアルタイムで監視する。
* **組み合わせ**: `dmesg -wH`
* **解説**: 攻撃者の活動によって引き起こされる可能性のあるカーネルレベルのイベントをライブで監視します。例えば、ネットワークインターフェイスがプロミスキャスモード (`promiscuous mode`) に移行した場合や、不審なカーネルモジュールがロードされた際の記録を発見するのに役立ちます。
* **例**:

    ```bash
    # カーネルメッセージをリアルタイムで監視し、不審なイベントがないか確認
    sudo dmesg -wH
    ```

### 3. レッドチーム視点

* **タスク**: 侵入したホストのハードウェア構成や環境を偵察する。
* **組み合わせ**: `dmesg`
* **解説**: `dmesg` の出力を精査し、仮想化環境 (`VMware`, `KVM`など) の特定、ハードウェア情報、そしてカーネルバージョンを把握します。これらの情報は、VMエスケープや、カーネルの脆弱性を利用した権限昇格エクスプロイトを選択するための重要な手がかりとなります。
* **例**:

    ```bash
    # 仮想マシン上で実行されているかどうかを確認
    dmesg | grep -i "hypervisor"
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `dmesg: read kernel buffer failed: Operation not permitted`
    * **考えられる原因**: セキュリティが強化された一部のシステムでは、一般ユーザーがカーネルのログバッファにアクセスすることが制限されています (`dmesg_restrict=1`)。
    * **解決策**: `sudo dmesg` のように、`sudo` を付けて実行してください。

## 環境変数と設定ファイル

* `dmesg` の動作に直接影響する環境変数や設定ファイルは通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `dmesg` 自体ではなく、その出力内容が**情報漏洩**に繋がります。
* **悪用シナリオ**: `dmesg` の出力は、システムの内部構成に関する非常に詳細な情報（カーネルバージョン、ハードウェア構成、メモリマップなど）を含みます。これらの情報は、攻撃者が脆弱性を探し、エクスプロイトを調整するための重要なヒントとなります。

### GTFOBins / LOLBAS における利用例

`dmesg` は情報表示のみを行うコマンドであり、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: `/etc/sysctl.conf` などで `kernel.dmesg_restrict = 1` を設定し、一般ユーザーによる `dmesg` の実行を制限する。
* **Detection (検知)**: `auditd` を使って `dmesg` コマンドの実行を監視する。`dmesg` の出力を定期的にSIEMに転送し、ベースラインと比較して異常なカーネルメッセージがないか分析する。

## 注意点・補足

* **リングバッファ**: `dmesg` が表示するメッセージは、カーネル内の固定サイズの「リングバッファ」に保存されます。新しいメッセージが来ると古いメッセージは上書きされて消えていきます。起動から長時間経過したシステムでは、起動初期のメッセージは `dmesg` コマンドでは見えなくなっていることがあります。
