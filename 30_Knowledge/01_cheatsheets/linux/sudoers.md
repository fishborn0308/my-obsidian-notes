# `sudoers` - `sudo`の権限設定ファイル

## 概要

`/etc/sudoers` は、`sudo` コマンドの動作を制御するポリシーファイルです。このファイルには、**どのユーザー**が、**どのホスト**から、**誰として** (`root`など)、**どのコマンド**を実行できるかが詳細に定義されています。

システムのセキュリティを維持するためには、このファイルを「**最小権限の原則**」に従って、慎重に設定・管理することが極めて重要です。

**⚠️ 最重要**: `/etc/sudoers` ファイルを編集する際は、決して `vi` や `nano` のような通常のテキストエディタで直接編集してはいけません。必ず、構文チェック機能を持つ専用のコマンド **`visudo`** を使用してください。

(出自: `sudo` パッケージの管理対象)

## 類似コマンドと差異

| 概念 | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| **`su` / rootログイン** | `su` やrootでの直接ログインは、rootのパスワードを知っていれば**全ての権限**を行使できます。`sudoers` は、**コマンド単位**で実行権限を詳細に委譲する、より安全で監査可能な仕組みです。 | `sudoers` |

## よく連携されるコマンド

`sudoers` ファイルを安全に編集するため、`visudo` コマンドが必須の連携ツールとなります。

### シナリオ例: 安全なsudoersファイルの編集フロー (インフラ構築・運用視点)

* **目的**: 構文ミスでシステムから締め出されるリスクを避けつつ、`sudoers` ファイルを安全に編集する。
* **連携コマンド**: `visudo`
* **解説**: `visudo` は、保存時にファイルの構文チェックを行い、エラーがあれば保存を中止してくれます。これにより、構文ミスで `sudo` が使えなくなるという最悪の事態を防ぎます。
* **コマンド例**:

    ```bash
    # このコマンドでsudoersファイルを安全に編集する
    sudo visudo

    # (エディタが起動するので、設定を編集して保存・終了する)
    # もし構文エラーがあれば、エラー内容が表示され、再編集を促される
    ```

## オプション説明 (`sudoers` の構文)

`sudoers` の各行は、`who where = (as_whom) what` という基本ルールに基づいています。

| 構文要素 | 説明 |
| :--- | :--- |
| **`who` (誰が)** | ⭐️ ユーザー名 (`john`)<br> ⭐ `%groupname` (グループ名, 例: `%admin`)<br> `User_Alias` (ユーザーエイリアス) |
| **`where` (どこから)** | ⭐ `ALL` (全てのホスト)<br> `Host_Alias` (ホストエイリアス) |
| **`(as_whom)` (誰として)** | ⭐ `(ALL)` (全てのユーザー, 主にroot)<br> `(root)` (rootとして)<br> `(www-data)` (特定のユーザーとして) |
| **`what` (何を)** | ⭐ `ALL` (全てのコマンド)<br> ⭐ `/bin/systemctl restart nginx` (コマンドのフルパス)<br> `NOPASSWD:` (パスワードなしで実行)<br> `Cmnd_Alias` (コマンドエイリアス) |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: `admin` グループに所属するユーザーに、完全な管理者権限を与える。
* **組み合わせ**: `%admin ALL=(ALL) ALL`
* **解説**: システム管理を行うユーザーを `admin` グループ (または `wheel` グループ) に追加するだけで、`sudo` 権限を付与できる、一般的で効率的な設定です。
* **例 (`/etc/sudoers.d/admin` ファイルに記述)**:

    ```bash
    # admin グループのメンバーは、全てのホストから、全てのコマンドをrootとして実行できる
    %admin ALL=(ALL) ALL
    ```

### 2. ブルーチーム視点

* **タスク**: `sudoers` ファイルの設定を監査し、過剰な権限が付与されていないか確認する。
* **組み合わせ**: `cat /etc/sudoers && cat /etc/sudoers.d/*`
* **解説**: 定期的にこのファイルの内容を監査し、組織のセキュリティポリシーに準拠しているか、特に `NOPASSWD: ALL` のような危険な設定がないかを確認します。
* **例**:

    ```bash
    # パスワードなしで実行できるコマンドがないか確認
    sudo grep -r "NOPASSWD" /etc/sudoers /etc/sudoers.d/
    ```

### 3. レッドチーム視点

* **タスク**: 侵入したユーザーの権限を調査し、権限昇格の足がかりを探す。
* **組み合わせ**: `sudo -l`
* **解説**: `sudo -l` は、`sudoers` ファイルに定義された、現在のユーザー向けのルールを表示します。もし、`find`, `vi`, `less`, `nmap` といった、シェルを起動したり任意のファイルを読み書きしたりできるコマンドが `sudo` で許可されていれば、**GTFOBins** を参照しそのコマンドを悪用してroot権限のシェルを奪取します。
* **例**:

    ```bash
    # 現在のユーザーがsudoで何を実行できるか確認
    sudo -l
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `>>> /etc/sudoers: syntax error near line 25 <<<`
    * **考えられる原因**: `visudo` を使わずにファイルを編集した、または `visudo` の編集中に構文を間違えました。
    * **解決策**: `visudo` はエラーがあると保存を中止します。再度 `sudo visudo` を実行し、指定された行の構文を修正してください。

## 環境変数と設定ファイル

* `sudoers` は設定ファイルそのものです。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

### 関連する設定ファイル

* ⭐ **`/etc/sudoers`**: `sudo` のメイン設定ファイル。
* ⭐ **`/etc/sudoers.d/`**: このディレクトリに置かれたファイルも `sudoers` の設定として読み込まれます。ユーザーごとや役割ごとの設定を分割して管理するのに便利です。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **`sudoers` の設定ミス**が、Linuxにおける権限昇格の最も一般的な原因の1つです。
* **悪用シナリオ**: 最小権限の原則に反した、過度に寛容なルール（例: `user ALL=(ALL) /usr/bin/tar`）は、GTFOBins に記載されているような手法で権限昇格に悪用されます。

### GTFOBins / LOLBAS における利用例

`sudoers` は、GTFOBins にリストされているほぼ全てのコマンドの権限昇格（Privilege Escalation）ベクトルです。

* **機能**: **Privilege Escalation**
* **解説**: GTFOBins は、あるコマンドに `sudo` 権限が与えられていた場合、どのようにしてそこからシェルを奪取したり、任意のファイルを読み書きしたりできるかの具体的な手法をリストアップしています。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **`visudo` を必ず使用する**。**最小権限の原則**を徹底し、`ALL` の使用は極力避け、ユーザーには本当に必要なコマンドのフルパスのみを許可する。
* **Detection (検知)**: `sudo` の実行ログ (`/var/log/auth.log`など) をSIEMに連携し、不審な特権コマンドの実行（特に認証失敗）を監視する。FIMツールで `sudoers` ファイルの変更を監視する。

## 注意点・補足

* **コマンドのフルパス指定**: セキュリティのため、`sudoers` で許可するコマンドは `/bin/systemctl` のように必ずフルパスで指定してください。これにより、`PATH` ハイジャックによるなりすましを防ぎます。
