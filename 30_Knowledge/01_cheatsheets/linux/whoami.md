# `whoami` - 現在の実効ユーザー名を表示する

## 概要

`whoami` は、「私は誰ですか？(who am I)」という問いに答える、非常にシンプルで基本的なコマンドです。現在そのコマンドを実行しているシェルの**実効ユーザー名**を一行で表示します。

複数のサーバーを操作している際や、`su` や `sudo` でユーザーを切り替えた後、自分が今どのユーザーとしてコマンドを実行しているのかを素早く確認するために不可欠です。

(出自: `coreutils` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| ⭐ **`id`** | `whoami` がユーザー名のみを表示するのに対し、`id` はUID, GID, 所属する全グループといった**より詳細な情報**を表示します。 | `id` (詳細確認) |
| **`who`** | **現在システムにログインしている**全ユーザーのセッション情報を一覧表示します。`whoami` は**自分自身**の情報のみを表示します。 | - |
| **`logname`** | ログインセッションを開始したユーザー名を表示します。`su` でユーザーを切り替えても元のユーザー名が表示される点が `whoami` とは異なります。 | - |

## よく連携されるコマンド

`whoami` は、スクリプト内で現在のユーザー名を取得し、処理を分岐させる際によく使われます。

### シナリオ例: ユーザー名に基づいたバックアップパスの生成 (インフラ構築・運用視点)

* **目的**: どのユーザーが実行しても、そのユーザーのホームディレクトリ配下にバックアップファイルを作成するスクリプトを作る。
* **連携コマンド**: `tar`, コマンド置換 (`$()`)
* **解説**: コマンド置換 `$(whoami)` を使って現在のユーザー名を取得し、バックアップファイルのパスの一部として動的に組み込みます。
* **コマンド例**:

    ```bash
    # ユーザーのドキュメントディレクトリを、そのユーザー名のバックアップとして保存
    CURRENT_USER=$(whoami)
    tar czf /mnt/backups/${CURRENT_USER}-docs-backup.tar.gz ~/Documents
    ```

## オプション説明

`whoami` は非常にシンプルなコマンドであり、一般的に使われるオプションはほとんどありません。

| オプション | 説明 |
| :--- | :--- |
| ⭐ (引数なし) | 現在の実効ユーザー名を表示します。 |
| `--help` | ヘルプメッセージを表示します。 |
| `--version` | バージョン情報を表示します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: スクリプトが意図したユーザーで実行されているか確認する。
* **組み合わせ**: `whoami`
* **解説**: `sudo -u <user>` で特定のユーザーとしてコマンドを実行した後や、cronジョブのデバッグ中に `whoami` を実行し、スクリプトがどのユーザー権限で動作しているかを確認します。
* **例**:

    ```bash
    # 現在のユーザーが 'root' かどうかを確認する
    if [ "$(whoami)" != "root" ]; then
        echo "This script must be run as root."
        exit 1
    fi
    ```

### 2. ブルーチーム視点

* **タスク**: **インシデントの初動調査**。プロセスの実行コンテキストを確認する。
* **組み合わせ**: `whoami`
* **解説**: 侵害された疑いのあるホストで不審なプロセスを発見した場合、そのプロセスの所有ユーザーを `ps` で特定し、`sudo su - <user>` でそのユーザーになりきってから `whoami` と `id` を実行します。これにより、攻撃者がどのユーザー権限で活動しているかを正確に把握します。
* **例**:

    ```bash
    # Webサーバーの脆弱性を突かれたと仮定し、www-dataユーザーになってみる
    sudo su - www-data -s /bin/bash
    whoami
    # -> www-data
    ```

### 3. レッドチーム視点

* **タスク**: **侵入後の状況確認 (Situational Awareness)**。
* **組み合わせ**: `whoami`
* **解説**: **シェルを奪取した後に実行する最初のコマンドの一つ**。攻撃者は、Webシェルの実行やリバースシェルの確立に成功した後、即座に `whoami` を実行します。`root` であれば権限昇格は完了、`www-data` や `apache` のような低権限ユーザーであれば、そこから `root` を目指す権限昇格のフェーズに移行します。
* **例**:

    ```bash
    # Webシェル経由でコマンドを実行
    # command: whoami
    # output: www-data
    ```

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `whoami` は読み取り専用ツールであり、脆弱性を持つ可能性は低いです。
* **悪用シナリオ**: `whoami` の出力は、攻撃者にとっても防御者にとっても、現在の権限レベルを把握するための**最も基本的な情報**となります。

### GTFOBins / LOLBAS における利用例

`whoami` は情報表示のみを行うコマンドであり、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 最小権限の原則を徹底し、サービスは専用の非特権ユーザーで実行する。
* **Detection (検知)**: `whoami` は非常にありふれたコマンドですが、Webサーバーのプロセス (`www-data`) から `whoami` が実行される、といった**通常ではありえない**コンテキストでの実行をEDRや`auditd`で監視することで、攻撃の兆候を捉えることができます。

## 注意点・補足

* **実効ID (Effective ID)**: `whoami` が表示するのは、SUIDなどで変化する可能性のある**実効ユーザーID**に対応する名前です。`logname` が表示するログイン時のユーザー名とは異なる場合があります。
