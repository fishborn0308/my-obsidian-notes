# `traceroute` - 宛先ホストまでのネットワーク経路を追跡する

## 概要

`traceroute` は、指定したホストまでのIPパケットが、どのような**経路**（ルーター）を通過していくかを一覧表示するためのネットワーク診断ツールです。`traceroute` は、**TTL (Time To Live)** という値が `1` から順にインクリメントされたパケットを送信します。パケットを受け取った各ルーターはTTLを1減らし、TTLが0になると送信元にICMP Time Exceededメッセージを返します。`traceroute` はこの応答を利用して、経路上に存在するルーターのIPアドレスを特定します。

(出自: `traceroute` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| ⭐ **`mtr`** | `traceroute` と `ping` を組み合わせた高機能版。各ホップに対して継続的にパケットを送信し、パケット損失率と応答時間をリアルタイムで表示し続けます。 | `mtr` (インタラクティブ診断) |
| **`ping`** | **最終的な宛先**との疎通と応答時間のみを測定します。途中の経路は分かりません。 | - |

## よく連携されるコマンド

`traceroute` は単体で完結するツールですが、その出力をファイルに保存してレポートとして利用することがあります。

### シナリオ例: ネットワーク問題のレポートを作成する (インフラ構築・運用視点)

* **目的**: ネットワークの問題をISPやデータセンター事業者に報告する際に、客観的な証拠として `traceroute` の結果を添付する。
* **連携コマンド**: リダイレクション (`>`)
* **解説**: `-n` で名前解決を無効にし、迅速に結果を出力してファイルに保存します。
* **コマンド例**:

    ```bash
    # google.com への経路追跡結果をレポートファイルに保存
    traceroute -n google.com > traceroute-report-google-$(date +%F).txt
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **表示制御** | |
| ⭐ `-n` | IPアドレスの名前解決を行わず、数値のまま表示します。実行が高速になります。 |
| **プロトコル指定** | |
| ⭐ `-I`, `--icmp` | (root権限要) プローブにICMP ECHOパケットを使用します。 |
| ⭐ `-T`, `--tcp` | (root権限要) プローブにTCP SYNパケットを使用します。ICMPをブロックするファイアウォール越えの調査に有効です。 |
| `-U`, `--udp` | プローブにUDPパケットを使用します。多くのLinux/Unixでのデフォルトです。 |
| **その他** | |
| `-p <port>` | `-T` や `-U` と併用し、宛先ポート番号を指定します。 |
| `-m <max_ttl>` | 追跡する最大のホップ数を指定します（デフォルトは30）。 |
| `-q <queries>` | 各ホップに送信するプローブパケットの数を指定します（デフォルトは3）。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 「特定のサイトに繋がらない」問題の原因が、どこにあるのかを切り分ける。
* **組み合わせ**: `traceroute <destination_host>`
* **解説**: `traceroute` を実行し、自社ネットワークを出た直後のISPのルーターで応答が途絶えればISP側の問題、相手先のデータセンターのルーターで途絶えれば相手側の問題、といったように責任範囲の切り分けに役立ちます。
* **例**:

    ```bash
    # ICMPがブロックされている可能性を考慮し、HTTPS(443)ポートへのTCPで経路追跡
    sudo traceroute -T -p 443 [www.example.com](https://www.example.com)
    ```

### 2. ブルーチーム視点

* **タスク**: 不審なIPアドレスが、地理的にどこにあり、どのネットワークプロバイダに属しているかを大まかに把握する。
* **組み合わせ**: `traceroute -n <suspicious_ip>`
* **解説**: 経路上ルーターのホスト名（`-n` なしの場合）やIPアドレスから、その通信がどの国や地域、どのISPを経由しているかを推測し、攻撃者の属性を判断する一助とします。
* **例**:

    ```bash
    # 不審なIP 198.51.100.10 への経路を確認
    traceroute -n 198.51.100.10
    ```

### 3. レッドチーム視点

* **タスク**: **ネットワークの内部偵察 (Reconnaissance)**。
* **組み合わせ**: `traceroute -n <internal_target_ip>`
* **解説**: 侵入したネットワークセグメントから、別の内部セグメントにあるサーバーへの経路を `traceroute` で調査します。出力される経路上ルーターのIPアドレスから、組織の内部ネットワークのトポロジー（構造）や、ファイアウォール、ACLの存在を推測します。
* **例**:

    ```bash
    # 内部のDBサーバー (10.10.20.5) への経路を偵察
    traceroute -n 10.10.20.5
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **現象**: **途中のホップが全て `* * *` になる。**
    * **考えられる原因**: 経路上のファイアウォールが、`traceroute` が使用するICMPまたはUDPパケットを完全にブロックしています。
    * **解決策**: `-T` (`--tcp`) オプションを使い、HTTP(80)やHTTPS(443)のような、一般的に許可されているTCPポートで追跡を試みると、経路が表示されることがあります。

## 環境変数と設定ファイル

* `traceroute` の動作に直接影響する環境変数や設定ファイルは通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `traceroute` 自体ではなく、その応答が**情報漏洩**に繋がります。
* **悪用シナリオ**: `traceroute` の応答には、組織の内部ネットワークで使用されているルーターのIPアドレスやホスト名が含まれます。これにより、攻撃者は内部ネットワークの構造を詳細に把握できます。

### GTFOBins / LOLBAS における利用例

`traceroute` はネットワーク診断ツールであり、直接的なコード実行や権限昇格の機能を持たないため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視-点)

* **Prevention (予防)**: 境界ルーターで、外部からのICMP Time Exceededメッセージを要求するパケット（TTLが低いパケット）をブロックまたはレート制限することで、内部ネットワークの偵察を困難にする。
* **Detection (検知)**: 内部から外部、または内部セグメント間で予期せぬ `traceroute` の通信が多発していないかをネットワーク監視ツールで監視する。

## 注意点・補足

* **インストール**: `traceroute` は多くのディストリビューションで標準インストールされていません。`sudo apt install traceroute` や `sudo yum install traceroute` などでインストールが必要です。
