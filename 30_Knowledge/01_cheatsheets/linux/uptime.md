# `uptime` - システムの稼働時間と平均負荷を表示する

## 概要

`uptime` は、システムが最後に起動してから現在までの**稼働時間**、現在ログインしている**ユーザー数**、そしてシステムの**ロードアベレージ**を一行で簡潔に表示するコマンドです。サーバーの状態を素早く把握するための、最も基本的なヘルスチェックコマンドの1つです。

(出自: `procps-ng` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| ⭐ **`w`** | `uptime` と**全く同じヘッダー情報**を表示し、さらに現在ログインしているユーザーの一覧と、彼らが何を実行しているかを詳細に表示します。 | `w` (対話的確認) |
| **`top` / `htop`** | これらの対話的なプロセスビューアも、ヘッダー部分に `uptime` と同じ情報を表示します。`uptime` は、概要だけを素早く知りたい場合に便利です。 | - |
| **`last reboot`** | システムの**再起動履歴**を一覧表示します。`uptime` は**現在のセッション**の稼働時間のみを表示します。 | - |

## よく連携されるコマンド

`uptime` は、その出力をスクリプトで解析し、システムの負荷を監視するために `awk` などと連携させることがあります。

### シナリオ例: ロードアベレージを監視スクリプトで利用する (インフラ構築・運用視点)

* **目的**: 1分間のロードアベレージを取得し、しきい値を超えていたらアラートを出すスクリプトを作成する。
* **連携コマンド**: `awk`
* **解説**: `uptime` の出力は決まった形式なので、`awk` を使ってロードアベレージの部分だけを簡単に抽出できます。
* **コマンド例**:

    ```bash
    # 1分間のロードアベレージだけを抽出する
    uptime | awk -F'load average: ' '{print $2}' | awk -F, '{print $1}'
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **表示形式** | |
| ⭐ `-p`, `--pretty` | `up 4 weeks, 2 days, 3 hours` のように、稼働時間だけを人間が読みやすい形式で表示します。 |
| ⭐ `-s`, `--since` | システムが起動した正確な日時 (`YYYY-MM-DD HH:MM:SS`) を表示します。 |
| **その他** | |
| `-h`, `--help` | ヘルプメッセージを表示します。 |
| `-V`, `--version` | バージョン情報を表示します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: サーバーの負荷状況を素早く確認する。
* **組み合わせ**: `uptime`
* **解説**: 「サーバーの応答が遅い」という通報を受けた際、`uptime` を実行してロードアベレージを確認します。もし15分、5分、1分と右肩上がりに数値が上昇していれば、負荷が急増していることを示唆し、`htop` などで詳細な調査を開始するきっかけとなります。
* **例**:

    ```bash
    # 現在の稼働時間とロードアベレージを確認
    uptime
    # ->  15:00:00 up 30 days,  1:45,  2 users,  load average: 0.00, 0.01, 0.05
    ```

### 2. ブルーチーム視点

* **タスク**: 計画外の再起動がなかったか、システムのパッチ適用状況を推測する。
* **組み合わせ**: `uptime -s`
* **解説**: `uptime -s` の結果と、`last reboot` の履歴、そしてシステムのメンテナンス計画を照合し、計画外の不審な再起動がなかったかを確認します。また、非常に長い稼働時間は、カーネルなどの重要なパッチが長期間適用されていないことを示唆し、システムの脆弱性を評価する指標となります。
* **例**:

    ```bash
    # システムの起動日時を確認
    uptime -s
    # -> 2025-08-29 13:15:00
    ```

### 3. レッドチーム視点

* **タスク**: 侵入後の偵察として、システムのパッチ適用状況を推測する。
* **組み合わせ**: `uptime`
* **解説**: 攻撃者は侵入後 `uptime` を実行し、システムの稼働時間を確認します。もし稼働時間が数百日に及ぶ場合、その間に公開されたカーネルの脆弱性などがパッチ適用されておらず、権限昇格のエクスプロイトが成功する可能性が高いと判断します。
* **例**:

    ```bash
    # 稼働時間を確認
    uptime
    ```

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `uptime` 自体ではなく、その出力が**深刻な情報漏洩**に繋がります。
* **悪用シナリオ**: `uptime` の出力、特に**稼働時間**は、攻撃者にとってそのシステムがどれくらいの間メンテナンス（パッチ適用と再起動）をされていないかを推測するための重要な情報源です。

### GTFOBins / LOLBAS における利用例

`uptime` は情報表示のみを行うコマンドであり、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: システムに定期的にセキュリティパッチを適用し、再起動する運用を徹底する。
* **Detection (検知)**: `uptime` は基本的なコマンドのため検知は困難ですが、Webサーバーのプロセス (`www-data`) から `uptime` が実行されるといった、通常ではありえないコンテキストでの実行をEDRや`auditd`で監視することで、攻撃の兆候を捉えることができます。

## 注意点・補足

* **ロードアベレージの解釈**: ロードアベレージは、CPUコア数と比較して評価します。`nproc` や `lscpu` コマンドでコア数を確認し、ロードアベレージがコア数を大幅に上回っている場合は、システムが過負荷状態にある可能性があります。
