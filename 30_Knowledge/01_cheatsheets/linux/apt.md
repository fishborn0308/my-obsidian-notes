# `apt` - Debian/Ubuntu系パッケージ管理コマンド

## 概要

`apt` (Advanced Package Tool) は、DebianやUbuntuなどのDebianベースのLinuxディストリビューションで使われる、高水準のパッケージ管理コマンドラインツールです。ソフトウェアのインストール、アップグレード、削除、検索といった操作を、依存関係を自動的に解決しながら簡単に行うことができます。従来の `apt-get` や `apt-cache` の主要な機能を統合し、プログレスバー表示など、対話的な利用に適したインターフェイスを提供します。

(出自: `apt` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| **`apt-get`** | `apt` の前身となるコマンド。`apt` はUIがよりユーザーフレンドリー。対話的操作は `apt`、スクリプト内での後方互換性を重視する場合は `apt-get` が使われることがある。 | `apt` |
| **`dpkg`** | 低水準のパッケージ管理ツールで、`.deb` パッケージファイルを直接扱います。依存関係を自動で解決しないため、通常は `apt` を使用します。 | `apt` |
| **`aptitude`** | `apt` 同様の高水準パッケージマネージャ。より複雑な依存関係の解決能力に優れているとされています。 | `apt` |

## よく連携されるコマンド

`apt` はシステムのパッケージ状態を変更するため、`sudo` との連携が必須です。また、パッケージの検索結果を絞り込むためにパイプも多用されます。

### シナリ-オ例: インストール済みパッケージの一覧から不要なものを一括削除 (運用視点)

* **目的**: システムにインストールされている `php8.0` 関連のパッケージを一覧表示し、それらを `xargs` に渡して一括で完全削除する。
* **連携コマンド**: `grep`, `cut`, `xargs`, `sudo`
* **解説**: `apt list` の結果を `grep` で絞り込み、パッケージ名だけを `cut` で切り出し、`xargs` に渡して `apt purge` を実行させることで、効率的にクリーンアップできます。
* **コマンド例**:

    ```bash
    # php8.0関連のインストール済みパッケージを確認後、一括で完全削除
    apt list --installed | grep 'php8.0' | cut -d'/' -f1 | xargs sudo apt purge -y
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **パッケージリスト操作** | |
| ⭐️ `update` | パッケージリストを更新します。インストールやアップグレードの前に必ず実行します。 |
| `upgrade` | インストール済みの全パッケージを、新しいバージョンにアップグレードします。 |
| `full-upgrade` | `upgrade` に加え、依存関係の変更を解決するために、必要であれば古いパッケージの削除も行います。 |
| **パッケージ管理** | |
| ⭐️ `install <pkg>` | 新しいパッケージをインストールします。 |
| `remove <pkg>` | パッケージを削除します（設定ファイルは残ります）。 |
| `purge <pkg>` | パッケージを、設定ファイルも含めて完全に削除します。 |
| `autoremove` | 不要になった依存パッケージを自動的に削除します。 |
| **情報表示** | |
| ⭐️ `list` | パッケージを一覧表示します。`--installed` や `--upgradable` オプションとよく使われます。 |
| `search <keyword>` | パッケージ名や説明文からキーワードに一致するパッケージを検索します。 |
| `show <pkg>` | パッケージのバージョン、依存関係、説明などの詳細情報を表示します。 |
| **共通オプション** | |
| `-y`, `--yes` | 全ての確認プロンプトに対して自動的に "yes" と回答します。スクリプト内で便利です。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: システム全体のパッケージを最新の状態に保つ。
* **組み合わせ**: `update && sudo apt upgrade -y`
* **解説**: パッケージリストを最新の状態に更新し、対話なしでシステム全体のパッケージをアップグレードする、最も基本的なサーバメンテナンスコマンドです。
* **例**:

    ```bash
    # 定期メンテナンスやセキュリティパッチの適用
    sudo apt update && sudo apt upgrade -y
    ```

### 2. ブルーチーム視点

* **タスク**: システムにインストールされているパッケージが、公式リポジトリから提供されたものか確認する。
* **組み合わせ**: `apt-cache policy <package>`
* **解説**: パッケージのインストール元リポジトリ、インストール済みバージョン、利用可能なバージョンなどを確認します。これにより、信頼性の低いサードパーティリポジトリからインストールされたものかを確認し、サプライチェーンリスクを評価します。
* **例**:

    ```bash
    # curlパッケージがどのリポジトリから提供されているか確認
    apt-cache policy curl
    ```

### 3. レッドチーム視点

* **タスク**: 侵入したシステムにインストールされているソフトウェアの一覧を取得し、脆弱性を探す。
* **組み合わせ**: `apt list --installed`
* **解説**: 取得したソフトウェアリストとそのバージョンを元に、既知の脆弱性 (CVE) を持つソフトウェアが存在しないかを検索し、権限昇格やさらなる攻撃の足がかりを探します。
* **例**:

    ```bash
    # インストール済みパッケージ一覧をファイルに出力して、攻撃者マシンに持ち帰り分析する
    apt list --installed > installed_packages.txt
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `E: Unable to locate package <package_name>`
    * **考えられる原因**:
        * `sudo apt update` を実行しておらず、ローカルのパッケージリストが古い。
        * パッケージ名のタイプミス。
    * **解決策**: まず `sudo apt update` を実行します。次にパッケージ名が正しいか再確認します。

2. **エラーメッセージ例 2**: `E: Could not get lock /var/lib/dpkg/lock-frontend ...`
    * **考えられる原因**: 他のプロセス（自動更新など）が `apt` を使用中です。
    * **解決策**: `ps aux | grep -i apt` でプロセスが実行中でないか確認し、実行中なら終了を待ちます。

## 環境変数と設定ファイル

* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

### 影響を与える環境変数

* **`DEBIAN_FRONTEND=noninteractive`**: この環境変数を設定すると、`apt` がユーザーに問い合わせを行う対話的なプロンプトを全て抑制します。シェルスクリプトや自動化ツールで `apt` を実行する際に必須です。

### 関連する設定ファイル

* **`/etc/apt/sources.list`**: APTがパッケージを検索するリポジトリのURLリストが記述されたメインの設定ファイル。
* **`/etc/apt/sources.list.d/`**: サードパーティのリポジトリなど、追加のリポジトリ設定ファイルを格納するディレクトリ。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: 過去に`apt`自体に脆弱性が発見されたことがあります（例: CVE-2019-3462）。HTTPリダイレクトの処理不備を悪用した中間者攻撃を可能にするものでした。
* **悪用シナリオ**: 攻撃者がDNSスプーフィングなどで正規のリポジトリへの通信を乗っ取り、マルウェアが仕込まれた改竄パッケージをインストールさせる（リポジトリハイジャック）。

### GTFOBins / LOLBAS における利用例

* **機能**: **Privilege Escalation** (権限昇選挙)
* **解説**: `sudo` 権限で `apt` を実行できるが、完全なシェルアクセスは許可されていない場合に、`apt` のサブコマンド（`changelog`など）が内部でページャ（`less`など）を呼び出すことを悪用します。ページャのプロンプト内で `!/bin/sh` と入力することで、root権限のシェルを起動できます。
* **参照**: `https://gtfobins.github.io/gtfobins/apt/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: `sudoers` ファイルの設定を厳格にし、ユーザーに不要なコマンドの実行権限を与えない。HTTPSリポジトリを使用し、GPG署名検証を常に有効にする。
* **Detection (検知)**: `/etc/apt/sources.list` や `/etc/apt/sources.list.d/` への変更をファイル整合性監視(FIM)ツールで監視する。

## 注意点・補足

* **`remove` と `purge` の違い**: `remove` はパッケージのみを削除し、設定ファイルは残します。`purge` は設定ファイルも含めて完全に削除します。
