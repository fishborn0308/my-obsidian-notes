# `bzip2` - .bz2ファイルに圧縮する

## 概要

`bzip2` は、Burrows-Wheeler変換とハフマン符号化アルゴリズムを使用してファイルを圧縮するためのコマンドです。一般的に、広く使われている `gzip` よりも高い圧縮率を達成しますが、その分、圧縮・展開にかかる時間は長くなる傾向があります。主に、サイズの大きいファイル（ログファイル、データベースダンプなど）を、速度よりも圧縮率を優先して保管する場合に利用されます。

デフォルトでは、圧縮に成功すると元のファイルは削除されます。

(出自: `bzip2` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| **`gzip`** | `bzip2` よりも**圧縮・展開が高速**ですが、**圧縮率は一般的に劣ります**。速度が重要なら `gzip`、ディスク容量の節約が重要なら `bzip2` と使い分けられます。 | `gzip` (速度) / `bzip2` (圧縮率) |
| **`zip`** | 複数のファイルをまとめるアーカイバ機能も持ちます。`bzip2` は原則として**単一のファイル**を圧縮するためのツールです。 | - |
| **`tar`** | ディレクトリを圧縮する場合、まず `tar` でアーカイブし、それを `bzip2` で圧縮します (`.tar.bz2`)。`tar cjf ...` のように `tar` コマンドから直接利用するのが一般的です。 | `tar` (ディレクトリの場合) |

## よく連携されるコマンド

`bzip2` は `tar` と連携してディレクトリを圧縮したり、コマンドの出力をパイプで受け取って圧縮したりする際に使われます。

### シナリオ例: データベースダンプをオンザフライで高圧縮する (インフラ構築・運用視点)

* **目的**: 巨大なデータベースのダンプファイルを、中間ファイルをディスクに書き出すことなく、直接 `bzip2` で圧縮してバックアップファイルを作成する。
* **連携コマンド**: `mysqldump` (または `pg_dump`), パイプ (`|`)
* **解説**: `mysqldump` の標準出力をパイプで `bzip2` に渡します。`-c` オプションで結果を標準出力に送り、リダイレクションでファイルに保存します。
* **コマンド例**:

    ```bash
    # my_database のダンプを取得し、bzip2で圧縮して保存
    mysqldump -u root -p my_database | bzip2 -c > backup-$(date +%F).sql.bz2
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **基本操作** | |
| ⭐ `-d`, `--decompress` | 展開（解凍）します。これは `bunzip2` コマンドと同じ動作です。 |
| ⭐ `-k`, `--keep` | 圧縮または展開後も、元のファイルを**削除せずに残します (keep)**。 |
| `-c`, `--stdout` | 圧縮または展開結果を標準出力 (stdout) に出力します。 |
| **圧縮レベル** | |
| `-[1-9]` | 圧縮レベルを指定します。`-1` が最も速く（低圧縮）、`-9` が最も遅く（高圧縮）なります。 |
| **その他** | |
| `-f`, `--force` | 同名の出力ファイルがすでに存在する場合でも、警告なしで**強制的に上書き**します。 |
| `-v`, `--verbose` | 実行中の詳細な情報（圧縮率など）を表示します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: ディスク容量を節約するため、巨大なログファイルを高圧縮する。
* **組み合わせ**: `bzip2 -9 -v <large_file>`
* **解説**: `-9` で最高の圧縮率を指定し、`-v` で圧縮結果を確認しながらファイルを圧縮します。頻繁にアクセスしないが長期間保管する必要があるファイルに適しています。
* **例**:

    ```bash
    # 1GBのSQLダンプファイルを最高圧縮レベルで圧縮
    bzip2 -9 -v large_database_dump.sql
    # -> large_database_dump.sql.bz2 が生成される
    ```

### 2. ブルーチーム視点

* **タスク**: 証拠となる大量のファイルを、保管や転送のために1つの高圧縮アーカイブにまとめる。
* **組み合わせ**: `tar cjf <archive.tar.bz2> /path/to/evidence/`
* **解説**: `tar` と `bzip2` を連携 (`j` オプション) させ、証拠となるディレクトリ全体を1つのファイルにまとめます。`bzip2` の高い圧縮率を利用することで、ストレージ容量とネットワーク帯域を節約できます。
* **例**:

    ```bash
    # /var/log ディレクトリ全体を 'logs-archive.tar.bz2' としてアーカイブ・圧縮
    tar cjf logs-archive.tar.bz2 /var/log/
    ```

### 3. レッドチーム視点

* **タスク**: 窃取したデータを高圧縮・暗号化して持ち出す準備をする。
* **組み合わせ**: `tar cjf - /path/to/loot | gpg -c -o loot.tar.bz2.gpg`
* **解説**: 窃取したデータを `tar` と `bzip2` で高圧縮アーカイブし、その結果をパイプで `gpg` に渡してさらに暗号化します。
* **例**:

    ```bash
    # /etc と /home/user の中身を高圧縮し、gpgで暗号化する
    tar cjf - /etc /home/user | gpg -c -o loot.tar.bz2.gpg
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `bzip2: I/O or other error, bailing out. Possible reason: disk full.`
    * **考えられる原因**: 圧縮プロセス中にディスクの空き容量がなくなりました。
    * **解決策**: 不要なファイルを削除してディスクの空き容量を確保するか、`-c` オプションとリダイレクションを使って別のファイルシステムに出力します。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **伸張爆弾 (Decompression Bomb)**。展開すると極端に巨大なファイルになるように細工された悪意のある圧縮ファイル。
* **悪用シナリオ**: `bzip2` で作成された伸張爆弾をサーバ上で展開させてしまうと、ディスクスペースを使い果たし、サービス妨害 (DoS) 攻撃を引き起こす可能性があります。

### GTFOBins / LOLBAS における利用例

`bzip2` はファイルの圧縮に特化しており、直接的なコード実行や権限昇格の機能を持たないため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 信頼できないソースから入手した圧縮ファイルは、隔離されたサンドボックス環境で安全に展開・調査する。
* **Detection (検知)**: ファイルアップロード機能を持つアプリケーションでは、アップロードされた圧縮ファイルのサイズや内容を検証する仕組みを導入する。

## 注意点・補足

* **`tar` との組み合わせ**: 複数のファイルやディレクトリを扱う場合は、`bzip2` を直接使うのではなく、`tar` の `j` オプションを使うのがLinux/Unixの標準的な作法です。

  ```bash
  # アーカイブ・圧縮
  tar cjf archive.tar.bz2 directory/
  # 展開・解凍
  tar xjf archive.tar.bz2
