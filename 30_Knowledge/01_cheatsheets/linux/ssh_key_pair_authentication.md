# SSH公開鍵認証 - パスワードレスで安全なログイン

## 概要

SSH公開鍵認証は、パスワードの代わりに、暗号化された**鍵ペア（公開鍵と秘密鍵）**を使ってサーバーへのログイン認証を行う方式です。

1. **クライアント側**で、自分だけが持つ**秘密鍵 (private key)** と、サーバーに渡すための**公開鍵 (public key)** のペアを生成します。
2. **サーバー側**の `~/.ssh/authorized_keys` ファイルに、クライアントの**公開鍵**を登録しておきます。
3. ログイン時、サーバーは公開鍵を使ってクライアントに「チャレンジ（課題）」を送り、クライアントは秘密鍵を使ってそのチャレンジに「レスポンス（回答）」します。

この仕組みにより、パスワードをネットワークに流すことなく、安全に本人確認ができます。パスワードのブルートフォース攻撃を原理的に無効化できるため、非常にセキュアです。

(出自: `openssh` パッケージの機能)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| ⭐ **パスワード認証** | パスワードは比較的短く、ブルートフォース攻撃で破られるリスクがあります。公開鍵は非常に長く複雑なため、事実上ブルートフォースは不可能です。 | 公開鍵認証 |
| **GPG/PGP** | 同じ公開鍵暗号技術を使いますが、GPGは主にファイルやメールの署名・暗号化に使われます。SSH公開鍵認証は、**セッション認証**に特化しています。 | - |

## よく連携されるコマンド

公開鍵認証のセットアップは、`ssh-keygen` と `ssh-copy-id` という2つのコマンドを連携させて行うのが標準的なワークフローです。

### シナリオ例: 新しいサーバーへの公開鍵認証設定 (インフラ構築・運用視点)

* **目的**: ローカルマシンから新しいサーバーへ、パスワードなしでSSHログインできるように設定する。
* **連携コマンド**: `ssh-keygen`, `ssh-copy-id`
* **解説**: まず `ssh-keygen` で鍵ペアを生成（未作成の場合）。次に `ssh-copy-id` を使って、公開鍵をサーバーの `authorized_keys` ファイルに安全に追記します。
* **コマンド例**:

    ```bash
    # このコマンドシーケンスで公開鍵認証をセットアップする
    # 1. 鍵ペアを生成 (まだ持っていない場合)
    ssh-keygen -t ed25519 -C "your_email@example.com"

    # 2. 公開鍵をサーバーにコピー (パスワード入力を求められる)
    ssh-copy-id user@new-server.example.com

    # 3. パスワードなしでログインできるかテスト
    ssh user@new-server.example.com
    ```

## オプション説明 (関連ファイルとパーミッション)

SSHはセキュリティを確保するため、以下のファイルとディレクトリに厳格なパーミッションを要求します。

| パス | 説明 | 必須パーミッション |
| :--- | :--- | :--- |
| ⭐ `~/.ssh/` | SSH設定ディレクトリ | `700` (`drwx------`) |
| ⭐ `~/.ssh/id_ed25519` | **秘密鍵**。**絶対に他人に渡してはいけません**。 | `600` (`-rw-------`) |
| `~/.ssh/id_ed25519.pub` | **公開鍵**。サーバーや他人に渡す鍵。 | `644` (`-rw-r--r--`) |
| ⭐ `~/.ssh/authorized_keys` | (サーバー側) ログインを許可する公開鍵のリスト。 | `600` (`-rw-------`) |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: サーバーのパスワード認証を無効化し、セキュリティを強化する。
* **組み合わせ**: `ssh-copy-id`, `sshd_config` の編集
* **解説**: 全ての管理者と自動化ツール用に公開鍵を登録した後、サーバーの `/etc/ssh/sshd_config` ファイルで `PasswordAuthentication no` を設定します。これにより、ブルートフォース攻撃のリスクを完全に排除します。
* **例**:

    ```bash
    # (全ユーザーの鍵登録後...)
    # サーバー側でsshd_configを編集
    sudo vi /etc/ssh/sshd_config
    # -> PasswordAuthentication no に変更

    # SSHサービスを再起動
    sudo systemctl restart sshd
    ```

### 2. ブルーチーム視点

* **タスク**: 不正な永続化設定がないか、`authorized_keys` ファイルを監査する。
* **組み合わせ**: `cat ~/.ssh/authorized_keys`
* **解説**: インシデント調査時に、サーバー上の全ての `~/.ssh/authorized_keys` ファイルを監査します。もし見慣れない公開鍵が追加されていれば、それは攻撃者が設置した**永続的なバックドア**である可能性が極めて高いです。
* **例**:

    ```bash
    # 全ユーザーのauthorized_keysをチェックするスクリプト例
    for dir in /home/*/.ssh; do echo "--- Checking $dir ---"; cat "${dir}/authorized_keys"; done
    ```

### 3. レッドチーム視点

* **タスク**: **永続化 (Persistence)** のため、標的ユーザーの `authorized_keys` に自身の公開鍵を追記する。
* **組み合わせ**: `echo <key> >> ~/.ssh/authorized_keys`
* **解説**: 攻撃者はWebアプリケーションの脆弱性などを突いて一般ユーザーとして侵入した後、そのユーザーの `authorized_keys` ファイルに自身の公開鍵を追記します。これにより、いつでもそのユーザーとしてSSHログインできるバックドアを確保し、たとえユーザーのパスワードが変更されてもアクセスを維持できます。
* **例**:

    ```bash
    # 攻撃者の公開鍵をauthorized_keysに追記
    echo "ssh-ed25519 AAAA... attacker@evil.com" >> ~/.ssh/authorized_keys
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **現象**: **鍵を設定したのに、パスワード入力を求められる。**
    * **考えられる原因**:
        * `~/.ssh` や `~/.ssh/authorized_keys` のパーミッションが緩すぎる。
        * 公開鍵が `authorized_keys` に正しくコピーされていない。
        * サーバーの `sshd_config` で `PubkeyAuthentication no` に設定されている。
    * **解決策**: 上記「関連ファイルとパーミッション」のパーミッション設定を確認してください。`ssh -v user@host` で詳細なデバッグログを表示し、どの認証が試行されているかを確認します。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **秘密鍵の漏洩**。公開鍵認証のセキュリティは、秘密鍵がどれだけ安全に管理されているかにかかっています。
* **悪用シナリオ**: 攻撃者は開発者のPCをマルウェアに感染させ、パスフレーズで保護されていない秘密鍵 (`id_rsa`, `id_ed25519`) を窃取しようとします。秘密鍵が盗まれれば、その鍵を使って正規のユーザーとしてサーバーに侵入できます。

### GTFOBins / LOLBAS における利用例

このトピックは認証方式に関するものであり、特定のバイナリではないため、GTFOBinsには直接リストされていません。しかし、リストされている多くのコマンド (`ssh`, `cat`, `echo`) が、この認証方式を悪用するために使われます。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **秘密鍵は必ず強力なパスフレーズで保護**する。可能であれば、秘密鍵をYubiKeyなどのハードウェアセキュリティキーに保管する。
* **Detection (検知)**: FIMツールで `~/.ssh/authorized_keys` ファイルへの全ての変更を監視し、意図しない公開鍵の追加があれば即座にアラートを発報するように設定する。

## 注意点・補足

* **`ssh-agent` の活用**: `ssh-agent` は、一度パスフレーゼを入力すれば、そのセッション中はパスフレーズを再入力することなく秘密鍵を使えるようにしてくれる便利なツールです。これにより、利便性とセキュリティを両立できます。
