# `ip route` - IPルーティングテーブルを管理する

## 概要

`ip route` は、`ip` コマンドスイートの一部であり、Linuxカーネルの**IPルーティングテーブル**を表示・操作するためのコマンドです。ルーティングテーブルとは、IPパケットを宛先に届けるための「経路案内図」のようなものです。このテーブルには、「どの宛先ネットワークへは、どのゲートウェイ（ルーター）を経由して、どのインターフェイスから出ていくか」という情報が記載されています。

`ip route` は、ネットワークの接続性のトラブルシューティングや、複雑なネットワーク構成を構築する際に不可欠なツールです。

(出自: `iproute2` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| **`route`** | **旧世代のコマンド**。`ip route` は、より柔軟な構文と豊富な機能を持ち、現在では `route` コマンドの完全な代替となっています。 | `ip route` |
| **`netstat -r`** | `route` コマンドと同様に、ルーティングテーブルを表示するための旧世代のコマンドです。 | `ip route` |
| **`traceroute`** | 宛先ホストまでの**実際のパケット経路**を追跡します。`ip route` が設定された「地図」を表示するのに対し、`traceroute` はその地図を使った実際の「旅」を報告します。 | - |

## よく連携されるコマンド

`ip route` は、ネットワークのトラブルシューティングにおいて、`ip addr` や `ping` と連携して使われます。

### シナリオ例: ネットワーク接続の基本調査 (インフラ構築・運用視点)

* **目的**: サーバーがインターネットに接続できない原因を調査する。
* **連携コマンド**: `ip addr`, `ip route`, `ping`
* **解説**: まず `ip addr` でIPアドレスが設定されているかを確認し、次に `ip route` でデフォルトゲートウェイが存在するかを確認します。最後に、そのゲートウェイに `ping` が通るかをテストする、という一連の流れで問題を切り分けます。
* **コマンド例**:

    ```bash
    # 1. IPアドレスを確認
    ip a

    # 2. デフォルトゲートウェイを確認 (default via ... の行があるか)
    ip r

    # 3. ゲートウェイ (例: 192.168.1.1) への疎通を確認
    ping -c 4 192.168.1.1
    ```

## オプション説明 (`ip route` のアクション)

| オプション | 説明 |
| :--- | :--- |
| **表示** | |
| ⭐ `show` (or `list`, `ls`) | (デフォルト) ルーティングテーブルを表示します。省略形の `ip r` がよく使われます。 |
| `get <address>` | 指定した宛先IPアドレスへのルートを**単一で**表示します。カーネルが実際にどのルートを選択するかを確認するのに便利です。 |
| **設定** | |
| ⭐ `add <prefix> via <gw>` | 新しいルートを追加します。 |
| ⭐ `del <prefix>` | 既存のルートを削除します。 |
| `flush` | ルーティングテーブルのエントリを削除します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 別のネットワークセグメントへの静的ルートを追加する。
* **組み合わせ**: `sudo ip route add <network>/<prefix> via <gateway_ip>`
* **解説**: 本社のネットワークと、VPNルーターを経由して接続される支社のネットワークがある場合に、本社サーバーに支社へのルートを追加します。
* **例**:

    ```bash
    # 10.10.20.0/24 へのパケットは 192.168.1.254 を経由させる
    sudo ip route add 10.10.20.0/24 via 192.168.1.254
    ```

### 2. ブルーチーム視点

* **タスク**: ルーティングテーブルが改ざんされていないかを確認する。
* **組み合わせ**: `ip route show`
* **解説**: インシデント調査対象のホストで `ip route` を実行し、デフォルトゲートウェイが本来のものから、不審なIPアドレス（攻撃者のマシンなど）に変更されていないかを確認します。これは中間者攻撃の兆候である可能性があります。
* **例**:

    ```bash
    # 現在のルーティングテーブルを確認
    ip r
    ```

### 3. レッドチーム視点

* **タスク**: 侵入後の偵察で、ネットワークトポロジーを把握し、中間者攻撃を仕掛ける。
* **組み合わせ**: `ip route show`, `sudo ip route add/del`
* **解説**: 攻撃者は `ip route` の出力から、デフォルトゲートウェイや他の内部ネットワークの存在を把握し、横展開の計画を立てます。root権限を奪取した後は、デフォルトゲートウェイを自身が制御するマシンに変更し、通信を盗聴します。
* **例**:

    ```bash
    # 既存のデフォルトゲートウェイを削除
    sudo ip route del default
    # 攻撃者のマシン (192.168.1.200) を新しいゲートウェイに設定
    sudo ip route add default via 192.168.1.200
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `RTNETLINK answers: Network is unreachable`
    * **考えられる原因**: ルートを追加しようとしましたが、`via` で指定したゲートウェイが、どのインターフェイスからも直接到達できないネットワークにあります。
    * **解決策**: ゲートウェイのIPアドレスが正しいか、また、そのゲートウェイが所属するネットワークセグメントにホストが正しく接続されているかを `ip addr show` で確認してください。

## 環境変数と設定ファイル

* `ip route` の動作に直接影響する環境変数は通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `ip route` 自体ではなく、**ルーティングテーブルの改ざん**がリスクとなります。
* **悪用シナリオ**: root権限を持つ攻撃者は、ルーティングテーブルを自由に書き換えることができます。これにより、通信の盗聴（中間者攻撃）、特定のサーバーへの通信を妨害（ブラックホール化）、あるいは攻撃者のC2サーバーへの通信経路確保などが可能になります。

### GTFOBins / LOLBAS における利用例

* **機能**: **Privilege Escalation**
* **解説**: もしユーザーが `sudo ip` を実行できる権限を持っている場合、`-f` オプションでスクリプトファイルを読み込ませる機能などを悪用して、任意のコマンドを実行することが可能です。
* **参照**: `https://gtfobins.github.io/gtfobins/ip/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 最小権限の原則に従い、一般ユーザーに `ip` コマンドの `sudo` 権限を安易に与えない。
* **Detection (検知)**: `auditd` を使って、`netlink` 経由でのネットワーク構成の変更を監視する。`ip route` の出力を定期的に取得し、ベースラインと比較して意図しない変更がないかを確認する。

## 注意点・補足

* **設定の永続化**: `ip route` コマンドで行った設定変更は、デフォルトでは**OSを再起動すると失われます**。設定を永続化させるには、各ディストリビューションのネットワーク管理設定ファイル（`/etc/netplan/`など）に記述する必要があります。
