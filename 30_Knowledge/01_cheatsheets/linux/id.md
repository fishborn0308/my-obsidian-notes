# `id` - ユーザーIDとグループID情報を表示する

## 概要

`id` は、指定したユーザー（または引数なしの場合は現在のユーザー）の**実ユーザーID (uid)**、**実グループID (gid)**、そして所属している**全てのグループ (groups)** の情報を表示するコマンドです。「今、自分は誰としてログインしているのか？」「どのグループに所属しているのか？」を正確に確認するための、最も基本的なコマンドの1つです。特に、権限関連のトラブルシューティングや、セキュリティ監査の場面で頻繁に使用されます。

(出自: `coreutils` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| **`whoami`** | 現在の実効ユーザー**名**のみをシンプルに表示します。`id` はUID/GIDの数値や所属グループも表示するため、より詳細です。 | `id` (詳細情報) |
| **`groups`** | ユーザーが所属する**グループ名**のみを一覧表示します。UIDやプライマリGIDは表示しません。 | - |
| **`logname`** | 現在のセッションを開始したユーザーのログイン名を表示します。`su` などでユーザーを切り替えても、元のログイン名が表示される点が `whoami` とは異なります。 | - |

## よく連携されるコマンド

`id` は、スクリプト内で現在のユーザーIDを取得し、処理を分岐させる際によく使われます。

### シナリオ例: rootユーザーでの実行を判定する (インフラ構築・運用視点)

* **目的**: シェルスクリプトが `root` ユーザー（UID 0）で実行されているかどうかを判定し、処理を分岐させる。
* **連携コマンド**: `if` 文
* **解説**: `id -u` は現在のユーザーのUIDを数値でのみ出力します。この結果が `0` と等しいかどうかを `if` 文でチェックすることで、スクリプトがroot権限で実行されているかを確実に判定できます。
* **コマンド例**:

    ```bash
    # スクリプトの冒頭でrootチェックを行う
    if [ "$(id -u)" -ne 0 ]; then
       echo "This script must be run as root"
       exit 1
    fi
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **ユーザー/グループID表示** | |
| ⭐ `-u`, `--user` | 実効ユーザーID (UID) の**数値のみ**を表示します。 |
| ⭐ `-g`, `--group` | 実効プライマリグループID (GID) の**数値のみ**を表示します。 |
| `-G`, `--groups` | 所属する全てのグループID (GID) を数値で表示します。 |
| **表示形式** | |
| ⭐ `-n`, `--name` | `-u`, `-g`, `-G` と組み合わせて使用し、IDの数値ではなく**名前**を表示します。 |
| `-r`, `--real` | 実効IDではなく、実IDを表示します。（SUIDなどで権限が変わった場合に違いが出ます） |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 特定のサービスアカウントのID情報を確認する。
* **組み合わせ**: `id <username>`
* **解説**: ファイルの所有者を `chown` で変更する前や、パーミッション (`chmod`) を設定する前に、対象のユーザー (`nginx` や `mysql` など) がどのグループに所属しているかを正確に確認し、設定ミスを防ぎます。
* **例**:

    ```bash
    # www-data ユーザーの情報を確認
    id www-data
    # -> uid=33(www-data) gid=33(www-data) groups=33(www-data)
    ```

### 2. ブルーチーム視点

* **タスク**: 調査中のプロセスの実行権限を確認する。
* **組み合わせ**: `id`
* **解説**: インシデント調査中に不審なプロセスを発見した場合、そのプロセスの実行ユーザーを `ps` で特定し、`su` でそのユーザーになりきってから `id` を実行します。`uid=0(root)` であれば、そのプロセスがroot権限で動作していることが確定します。
* **例**:

    ```bash
    # 不審なユーザー 'eviluser' に切り替わって権限を確認
    sudo su - eviluser -s /bin/bash
    id
    ```

### 3. レッドチーム視点

* **タスク**: **侵入後の状況確認 (Situational Awareness)** のために、現在の権限を把握する。
* **組み合わせ**: `id`
* **解説**: 攻撃者はシステムに侵入後、即座に `whoami` と `id` を実行します。`uid=0(root)` ならば権限昇格済み、そうでなければ `groups=...` の部分を精査し、`docker`, `sudo`, `lxd` といった権限昇格に悪用できる特権グループに所属していないかを確認します。
* **例**:

    ```bash
    # 侵入後に実行し、権限昇格の足がかりを探す
    id
    # -> uid=1001(dev-user) gid=1001(dev-user) groups=1001(dev-user),27(sudo),999(docker)
    # -> この結果から、`sudo` と `docker` が悪用できると判断する
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `id: ‘<username>’: no such user`
    * **考えられる原因**: 指定したユーザー名が存在しません。
    * **解決策**: `getent passwd` や `/etc/passwd` ファイルで、存在するユーザーの名前を再確認してください。

## 環境変数と設定ファイル

* `id` の動作に直接影響する環境変数は通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `id` は読み取り専用ツールですが、その出力は**情報漏洩**に繋がります。
* **悪用シナリオ**: `id` コマンドの出力は、攻撃者にとって権限昇格のヒントの宝庫です。特に、ユーザーがどの特権グループに所属しているかは、システムの脆弱性を評価する上で非常に重要な情報となります。

### GTFOBins / LOLBAS における利用例

`id` は情報表示のみを行うコマンドであり、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 最小権限の原則に従い、ユーザーを不要な特権グループ（`sudo`, `docker`など）に追加しない。
* **Detection (検知)**: ユーザーのグループメンバーシップの変更を `auditd` で監視する。定期的に全ユーザーの `id` 情報を取得し、ベースラインと比較して意図しない権限が付与されていないか監査する。

## 注意点・補足

* **実IDと実効ID**: 通常、実UIDと実効UIDは同じです。しかし、SUIDビットが設定されたプログラムを実行すると、プロセスの実効UIDがプログラム所有者のUIDに一時的に変わります。`id -ru` (実ID) と `id -u` (実効ID) の結果が異なる場合、そのシェルが何らかの権限昇格状態にあることを示唆します。
