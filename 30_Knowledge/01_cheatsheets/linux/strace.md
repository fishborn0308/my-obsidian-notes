# `strace` - プロセスのシステムコールを追跡する

## 概要

`strace` (System Call Trace) は、プロセスが実行中に、OSの核である**カーネル**の機能を呼び出すためのインターフェイスである**システムコール (System Call)** と、プロセスが受信する**シグナル**を傍受しその詳細をログとして表示する強力なデバッグツールです。

`strace` を使うことで、「アプリケーションがなぜ "Permission denied" で失敗するのか」「どの設定ファイルを読み込もうとしているのか」といった、プログラムの表層的なエラーメッセージだけでは分からない低レベルな挙動を正確に把握できます。

(出自: `strace` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| ⭐ **`ltrace`** | `ltrace` は**ユーザー空間**の**共有ライブラリ**へのコールを追跡します。`strace` は**カーネル空間**への**システムコール**を追跡します。「OSとのやり取り」は`strace`、「プログラムの内部ロジック」は`ltrace`です。 | - |
| **`gdb`** | より高機能な**対話的デバッガ**。プログラムの実行をステップごとに進められます。`strace` はプログラムを実行させつつ、コールを**非対話的**に記録します。 | `gdb` (詳細デバッグ) |
| **`perf`** | Linuxカーネルに組み込まれた**パフォーマンス分析ツール**。システムコールの追跡も可能ですが、CPUのパフォーマンスカウンターに基づいた、より高度な性能解析を得意とします。 | `perf` (性能分析) |

## よく連携されるコマンド

`strace` の出力は非常に膨大になるため、`grep` でフィルタリングしたり、`-o` オプションでファイルに保存して `less` で閲覧するのが一般的です。

### シナリオ例: アプリケーションがハングアップした原因を調査する (インフラ構築・運用視点)

* **目的**: 応答しなくなったプロセスのPIDを特定し、`strace` でアタッチして、どのシステムコールで処理が止まっているのかを確認する。
* **連携コマンド**: `ps`, `grep`, `strace`
* **解説**: まず `ps` と `grep` で問題のプロセスのPIDを特定します。次に `strace -p` でその実行中プロセスにアタッチし、最後のシステムコール出力を調べることで、ハングアップの原因（例: ネットワークの読み取り待ち、ファイルのロック待ちなど）を推測します。
* **コマンド例**:

    ```bash
    # 1. ハングした 'my-app' プロセスのPIDを探す
    ps aux | grep 'my-app'
    # -> user      12345 ... ./my-app

    # 2. 特定したPID (12345) にアタッチしてシステムコールを監視
    sudo strace -p 12345
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **プロセス指定** | |
| ⭐ `-p <PID>` | 既に実行中のプロセスにアタッチして追跡を開始します。 |
| ⭐ `-f` | 子プロセスやスレッドも追跡します (`fork`, `clone`, `vfork`)。**非常に重要です**。 |
| **出力制御** | |
| ⭐ `-o <file>` | 出力を画面ではなく、指定したファイルに書き込みます。 |
| ⭐ `-e <expression>` | **最も重要なオプションの一つ**。追跡するシステムコールをフィルタリングします。<br>`-e trace=open,read`: `open` と `read` のみ追跡。<br>`-e trace=file`: ファイル関連の全システムコールを追跡。<br>`-e trace=network`: ネットワーク関連の全システムコールを追跡。 |
| `-c` | 実行終了時に、各システムコールの統計サマリーを表示します。 |
| `-T` | 各システムコールの実行に要した時間を表示します。 |
| `-s <size>` | `read`/`write` などで表示する文字列の最大サイズを指定します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 「Permission denied」「File not found」エラーの根本原因を究明する。
* **組み合わせ**: `strace -e trace=open,openat,access -f /path/to/app`
* **解説**: `-e` でファイルアクセスに関連するシステムコールに絞り、`-f` で子プロセスも追跡します。出力の中から `EACCES (Permission denied)` や `ENOENT (No such file or directory)` を返している行を見つけることで、問題の原因を正確に特定できます。
* **例**:

    ```bash
    # strace の出力例
    openat(AT_FDCWD, "/etc/myapp/config.conf", O_RDONLY) = -1 EACCES (Permission denied)
    # -> この行から、"/etc/myapp/config.conf" が読み取れずに失敗したことが分かる
    ```

### 2. ブルーチーム視点

* **タスク**: **動的マルウェア解析**。サンドボックス環境で、不審なバイナリを実行してその全てのシステムコールを記録する。
* **組み合わせ**: `sudo strace -f -o trace_output.txt /path/to/malware`
* **解説**: `strace` の出力ログを分析することで、マルウェアがどのような悪意のある活動（情報収集、永続化、通信、破壊活動）を行っているかを把握します。
* **例**:

    ```bash
    # マルウェアのファイル操作とネットワーク活動を追跡
    sudo strace -f -e trace=file,network -o malware_trace.log ./malware_sample
    ```

### 3. レッドチーム視点

* **タスク**: 脆弱性分析のため、攻撃対象のバイナリの内部動作を理解する。
* **組み合わせ**: `strace /path/to/target_app`
* **解説**: 対象のプログラムを `strace` で実行し、そのシステムコールの流れを追跡します。ハードコードされたパスや、予測可能な `/tmp` ファイルの利用（シンボリックリンク攻撃の可能性）、`execve` での外部コマンド呼び出し（コマンドインジェクションの可能性）などを調査し、脆弱性を探します。
* **例**:

    ```bash
    # ユーザー入力を待つプログラムをstraceで起動し、様々な入力を試す
    strace ./vulnerable_app
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `strace: attach: ptrace(PTRACE_ATTACH, ...): Operation not permitted`
    * **考えられる原因**: `root` 権限なしで、自身が所有していないプロセスにアタッチしようとしました。または、`kernel.yama.ptrace_scope` の設定により `ptrace` が制限されています。
    * **解決策**: `sudo strace ...` のように `sudo` を付けて実行してください。

## 環境変数と設定ファイル

* `strace` の動作に直接影響する環境変数や設定ファイルは通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `strace` 自体ではなく、その出力内容が**情報漏洩**に繋がります。
* **悪用シナリオ**: `strace` の出力には、ファイルパス、ネットワークアドレス、読み書きされたデータ（パスワードやキーを含む可能性）など、機密情報が豊富に含まれる可能性があります。

### GTFOBins / LOLBAS における利用例

* **機能**: **File Write**, **Privilege Escalation**
* **解説**: もしユーザーが `sudo strace` を実行できる権限を持っている場合、`-o` オプションを使って任意のファイルを上書きできます。
* **参照**: `https://gtfobins.github.io/gtfobins/strace/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: `sudoers` の設定を厳格にし、`strace` のようなデバッグツールに `sudo` 権限を安易に与えない。`kernel.yama.ptrace_scope` を `1` 以上に設定し、`ptrace` の利用を制限する。
* **Detection (検知)**: `auditd` を使って `strace` コマンドの実行や `ptrace` システムコールの呼び出しを監視する。

## 注意点・補足

* **パフォーマンスへの影響**: プロセスを `strace` でトレースすると、そのプロセスの実行速度は**著しく低下します**。本番環境での常時監視ではなく、一時的なデバッグや分析のためのツールです。
