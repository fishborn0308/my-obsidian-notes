# `netstat` - ネットワーク接続、ルーティングテーブルなどを表示する (旧)

## 概要

`netstat` (network statistics) は、ネットワーク接続（インバウンド、アウトバウンド）、ルーティングテーブル、インターフェイスの統計情報などを表示するための、古典的なコマンドラインツールです。「どのポートがLISTEN（待ち受け）状態か？」「どこに接続しているか？」といった、システムのネットワーク活動を調査する際に広く利用されてきました。

**⚠️ 重要**: `netstat` は `net-tools` パッケージに含まれる**レガシー（旧式）**なコマンドです。多くのモダンなLinuxディストリビューションではデフォルトでインストールされておらず、代わりに `iproute2` パッケージに含まれる **`ss` コマンドの使用が強く推奨されます**。

(出自: `net-tools` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| ⭐ **`ss`** | **`netstat` のモダンな後継**。`ss` はカーネルから直接情報を取得するため、特に接続数が多いシステムで `netstat` よりも**遥かに高速**に動作し、より詳細な情報を提供します。 | `ss` |
| **`lsof -i`** | 開かれているネットワークソケットを一覧表示します。`netstat -p` 同様にプロセス情報も表示でき、非常に高機能です。 | - |
| **`ip route`, `ip link`** | それぞれ `netstat -r`（ルーティングテーブル）、`netstat -i`（インターフェイス統計）のモダンな代替コマンドです。 | `ip route`, `ip link` |

## よく連携されるコマンド

`netstat` の出力はテキストベースなので、`grep` と連携して特定の接続をフィルタリングするのが一般的です。

### シナリオ例: ESTABLISHED状態の接続を絞り込む (ブルーチーム視点)

* **目的**: 現在確立されている（ESTABLISHED）ネットワーク接続の中から、特定のIPアドレスとの通信がないかを確認する。
* **連携コマンド**: `grep`
* **解説**: `netstat -an` で全ての接続を表示し、その結果をパイプで `grep` に渡して `ESTABLISHED` というキーワードでフィルタリングします。これにより、現在アクティブな通信セッションのみを調査できます。
* **コマンド例**:

    ```bash
    # ESTABLISHED状態の接続のみを表示し、さらに特定のIPで絞り込む
    sudo netstat -an | grep ESTABLISHED | grep "198.51.100.10"
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **表示対象** | |
| ⭐ `-t` | **TCP** 接続を表示します。 |
| ⭐ `-u` | **UDP** 接続を表示します。 |
| ⭐ `-l` | **LISTEN** (待ち受け) 状態のソケットのみを表示します。 |
| `-a` | LISTEN状態とそうでないものを含め、**全ての**ソケットを表示します。 |
| `-r` | カーネルのIP**ルーティングテーブル**を表示します。（`ip route` と同義） |
| `-i` | ネットワーク**インターフェイス**の統計情報を表示します。（`ip -s link` と同義） |
| **表示形式** | |
| ⭐ `-n` | ホスト名やサービス名を解決せず、**数値** (IPアドレス、ポート番号) のまま表示します。 |
| ⭐ `-p` | ソケットを使用しているプロセスの**PIDと名前**を表示します。(`root`権限が必要です) |
| `-c` | 1秒ごとに情報を**連続して**表示します (`watch` のような動作)。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: サーバーでどのサービスがどのポートで待ち受けているかを確認する。
* **組み合わせ**: `sudo netstat -tulpn`
* **解説**: 「TCP/UDPの、LISTEN状態の、PID付きの、数値表示」を意味する古典的な組み合わせです。サービス起動後の確認や、ポート競合のトラブルシューティングに使用します。（`ss -tulpn` で代替可能）
* **例**:

    ```bash
    sudo netstat -tulpn
    # Proto Recv-Q Send-Q Local Address           Foreign Address         State       PID/Program name
    # tcp        0      0 0.0.0.0:22              0.0.0.0:* LISTEN      1234/sshd
    # tcp        0      0 127.0.0.1:3306          0.0.0.0:* LISTEN      5678/mysqld
    ```

### 2. ブルーチーム視点

* **タスク**: 侵害されたホスト上で、不審な外部接続がないか調査する。
* **組み合わせ**: `sudo netstat -anp`
* **解説**: **ライブフォレンジックの基本**。全ての接続を数値とプロセス情報付きで表示します。`ESTABLISHED` 状態のリストの中から、見慣れないIPアドレスへのアウトバウンド接続を探し、関連するPID/プロセス名を特定することで、悪意のあるプロセスを突き止めます。
* **例**:

    ```bash
    # 全ての接続を表示し、不審なものがないか確認
    sudo netstat -anp | less
    ```

### 3. レッドチーム視点

* **タスク**: 侵入後の偵察として、システムのネットワーク活動を完全にマッピングする。
* **組み合わせ**: `netstat -anp`
* **解説**: `LISTEN` 状態のポートから攻撃可能な内部サービスを特定したり、`ESTABLISHED` 状態の接続から、このサーバーがどの他のサーバーと通信しているかを把握したりして、ラテラルムーブメント（横展開）の標的を探します。
* **例**:

    ```bash
    # 待ち受けポートと確立済みの接続をリストアップ
    netstat -anp
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `bash: netstat: command not found`
    * **考えられる原因**: `net-tools` パッケージがインストールされていません。モダンなLinuxディストリビューションの最小インストールでは、デフォルトで含まれていないことが多くなっています。
    * **解決策**: `sudo apt install net-tools` や `sudo yum install net-tools` でインストールするか、**代替コマンドである `ss` を使用してください**。

## 環境変数と設定ファイル

* `netstat` の動作に直接影響する環境変数や設定ファイルは通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `netstat` 自体ではなく、その出力が**情報漏洩**に繋がります。
* **悪用シナリオ**: `netstat` の出力は、システムのネットワーク構成、稼働中のサービス、外部との通信相手など、攻撃者にとって非常に価値の高い偵察情報です。

### GTFOBins / LOLBAS における利用例

`netstat` は情報表示のみを行うコマンドであり、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 最小権限の原則を適用し、不要なユーザーに `netstat` の実行権限を与えない。
* **Detection (検知)**: `auditd` で `netstat` コマンドの実行を監視し、不審な偵察活動がないかを確認する。EDR製品は通常、`netstat` のような偵察コマンドの実行を検知・記録します。

## 注意点・補足

* **`ss` への移行を強く推奨**: `netstat` はLinuxネットワークの仕組みを学ぶ上で歴史的に重要ですが、実務では `ss` を使う習慣をつけましょう。`ss` は大規模な接続を扱うサーバーで劇的なパフォーマンスの差が出ます。
