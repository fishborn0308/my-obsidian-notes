# `cryptsetup` - Linuxディスク暗号化 (LUKS) を管理する

## 概要

`cryptsetup` は、Linuxのカーネルモジュールである `dm-crypt` を利用して、ブロックデバイス（ディスクパーティション、LVMなど）を暗号化するための標準的なコマンドラインツールです。主に **LUKS (Linux Unified Key Setup)** という標準フォーマットを扱います。LUKSは、単なる暗号化だけでなく、複数のパスフレーズやキーファイルの管理、メタデータの保持といった堅牢な機能を提供し、Linuxにおけるディスク暗号化のデファクトスタンダードとなっています。`cryptsetup` を使うことで、データパーティションやシステム全体を暗号化し、物理的な盗難や不正なアクセスからデータを保護できます。

(出自: `cryptsetup` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| **eCryptFS / fscrypt** | **ファイル・ディレクトリ単位**で暗号化します。`cryptsetup` は**パーティション全体**を暗号化するブロックデバイスレベル暗号化であり、メタデータの漏洩リスクが少なく、パフォーマンス面でも有利なことが多いです。 | `cryptsetup` (フルディスク) |
| **VeraCrypt** | クロスプラットフォーム対応のツール。ポータブルな暗号化コンテナファイルの作成に長けています。`cryptsetup` はLinuxネイティブの標準であり、OSとの連携に優れています。 | - |
| **BitLocker** | Microsoft Windows向けのフルディスク暗号化機能。`cryptsetup` はLinuxにおける同等の役割を担います。 | - |

## よく連携されるコマンド

LUKSボリュームの作成から利用までは、ファイルシステムを作成する `mkfs` や、マウントする `mount` といったコマンドと連携させる一連のフローになります。

### シナリオ例: 新しいディスクを暗号化して利用可能にする (インフラ構築・運用視点)

* **目的**: 新しいディスク (`/dev/sdb1`) を、データベースのデータ保存用に暗号化し、マウントして利用できるようにする。
* **連携コマンド**: `cryptsetup`, `mkfs`, `mount`
* **解説**: `luksFormat` で初期化し、`open` で復号化されたデバイスを `/dev/mapper` に作成します。その後、その仮想デバイス上にファイルシステムを作成し、マウントします。
* **コマンド例**:

    ```bash
    # このコマンドシーケンスで暗号化ボリュームをセットアップする
    # 1. LUKSでフォーマット (パスフレーズ入力を求められる)
    sudo cryptsetup luksFormat /dev/sdb1

    # 2. "db_data" という名前で開く (パスフレーズ入力)
    sudo cryptsetup open /dev/sdb1 db_data

    # 3. ext4ファイルシステムを作成
    sudo mkfs.ext4 /dev/mapper/db_data

    # 4. マウントして使用開始
    sudo mount /dev/mapper/db_data /mnt/db
    ```

## オプション説明 (`cryptsetup` のアクション)

| オプション | 説明 |
| :--- | :--- |
| **ボリューム操作** | |
| ⭐ `luksFormat <device>` | デバイスをLUKSで初期化します。**デバイス上のデータは全て破壊されます**。 |
| ⭐ `open <device> <name>` | デバイスを `<name>` という名前で開き、`/dev/mapper/<name>` を作成します。 |
| ⭐ `close <name>` | 開いているボリューム `<name>` を閉じます。 |
| **鍵管理** | |
| `luksAddKey <device>` | 既存のLUKSボリュームに新しいパスフレーズやキーファイルを追加します。 |
| `luksRemoveKey <device>` | LUKSボリュームからパスフレーズを削除します。 |
| **情報表示** | |
| `luksDump <device>` | LUKSヘッダの情報を表示します（暗号アルゴリズム、キースロットの使用状況など）。 |
| `status <name>` | 開いているボリュームの状態を表示します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: サーバ起動時に暗号化ボリュームを自動でマウントする。
* **組み合わせ**: `/etc/crypttab` と `/etc/fstab` の設定
* **解説**: サーバが盗難された場合でも、パスフレーズがなければデータは読み取れません。OS起動時に自動でマウントするには `/etc/crypttab` (復号化) と `/etc/fstab` (マウント) の両方の設定が必要です。
* **例**:

    ```bash
    # /etc/crypttab に追記
    # <name>       <device>                                <keyfile>      <options>
    secret_data    UUID=1234-abcd-5678-efgh-9012    /etc/keys/data.key    luks

    # /etc/fstab に追記
    /dev/mapper/secret_data    /mnt/secret    ext4    defaults    0 2
    ```

### 2. ブルーチーム視点

* **タスク**: 証拠保全したディスクイメージに、LUKSで暗号化されたパーティションを発見し、調査する。
* **組み合わせ**: `cryptsetup luksDump <device>`
* **解説**: まず `luksDump` を実行し、暗号化方式やキースロットの使用状況などのメタデータを収集します。もしボリュームが開いている状態（ライブレスポンス中など）であれば、`cryptsetup status <name>` で確認し、復号された `/dev/mapper/<name>` から直接データを保全します。閉じている場合は、パスフレーズやキーファイルをシステム内から見つけ出す必要があります。
* **例**:

    ```bash
    # ディスクイメージ内のパーティションに対してヘッダ情報を確認
    sudo cryptsetup luksDump /dev/loop0p1
    ```

### 3. レッドチーム視点

* **タスク**: 侵入したサーバのroot権限を奪取し、暗号化されたデータへのアクセスを試みる。
* **組み合わせ**: `cat /etc/crypttab`
* **解説**: 攻撃者は、すでに開かれているボリュームからデータを直接窃取します。また、`/etc/crypttab` の設定から、サーバ起動時に自動でボリュームを復号するためのキーファイルの場所が特定できる場合があります。そのキーファイルが読み取り可能であれば、パスフレーズを知らなくても暗号化データを復号できます。
* **例**:

    ```bash
    # 自動マウントの設定ファイルを確認し、キーファイルのパスを探す
    cat /etc/crypttab
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `No key available with this passphrase.`
    * **考えられる原因**: `cryptsetup open` 時に間違ったパスフレーズを入力しました。
    * **解決策**: 正しいパスフレーズを再入力してください。

2. **エラーメッセージ例 2**: `Device /dev/mapper/secret_data is still in use.`
    * **考えられる原因**: `cryptsetup close` を実行しようとしましたが、ボリュームがまだマウントされています。
    * **解決策**: `sudo umount /dev/mapper/secret_data` を実行して、先にアンマウントしてください。

## 環境変数と設定ファイル

* `cryptsetup` の動作に直接影響する環境変数は通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

### 関連する設定ファイル

* **`/etc/crypttab`**: システム起動時に自動的に開く（復号する）暗号化デバイスを定義します。
* **`/etc/fstab`**: 復号されたデバイスをマウントするための設定を記述します。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `cryptsetup` 自体よりも、**パスフレーズの強度**や**キーファイルの管理**の不備がリスクとなります。
* **悪用シナリオ**: 攻撃者がroot権限を奪取した場合、平文で保存されたキーファイルを探したり、メモリ上に残っている復号鍵を抽出（コールドブート攻撃など）したりして、暗号化データを窃取しようと試みます。

### GTFOBins / LOLBAS における利用例

`cryptsetup` はディスクを管理する防御的なツールであり、直接的なコード実行や権限昇格の機能を持たないため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 強力なパスフレーズを設定する。キーファイルを使用する場合は、そのファイルのパーミッションを厳格に管理 (`600`) する。可能であればTPMなどのハードウェアと連携してキーを保護する。
* **Detection (検知)**: `auditd` を使い、`/etc/crypttab` などの設定ファイルへの変更や、`cryptsetup` コマンドの実行を監視する。

## 注意点・補足

* **ヘッダのバックアップ**: LUKSボリュームのヘッダ領域が破損するとデータが永久に失われる可能性があります。`cryptsetup luksHeaderBackup` でヘッダのバックアップを別途保存しておくことが強く推奨されます。
