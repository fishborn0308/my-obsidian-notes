# `ltrace` - プロセスのライブラリコールを追跡する

## 概要

`ltrace` (library call trace) は、プロセスが実行中に呼び出す**共有ライブラリ**の関数と、プロセスが受信するシグナルを傍受し、その詳細をログとして表示するデバッグツールです。

例えば、アプリケーションが標準Cライブラリ (`libc`) の `printf` や `strcpy` をどのような引数で呼び出しているか、あるいはOpenSSLライブラリの暗号化関数をどのように利用しているかをリアルタイムで確認できます。アプリケーションの内部ロジックの解析や、ライブラリ使用に起因するバグの発見、マルウェアの挙動分析などに利用されます。

(出自: `ltrace` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| **`strace`** | `strace` は**カーネル**への**システムコール**を追跡します。`ltrace` は**ユーザー空間**の**共有ライブラリ**へのコールを追跡します。「OSとのやり取り」は`strace`、「プログラムの内部ロジック」は`ltrace`です。 | - |
| **`gdb`** | より高機能な**対話的デバッガ**。プログラムの実行をステップごとに進められます。`ltrace` はプログラムを実行させつつ、コールを**非対話的**に記録します。 | `gdb` (詳細デバッグ) |

## よく連携されるコマンド

`ltrace` の出力は非常に冗長なため、`grep` と連携して必要な情報だけをフィルタリングするのが一般的です。

### シナリオ例: 特定のライブラリ関数呼び出しを監視する (インフラ構築・運用視点)

* **目的**: アプリケーションが `malloc` (メモリ確保) 関数を何回呼び出しているかを確認する。
* **連携コマンド**: `grep`, `wc`
* **解説**: `ltrace` の出力の中から、`grep` を使って "malloc" というキーワードを含む行だけをフィルタリングし、`wc -l` でその回数を数えます。メモリリークの簡易的な調査などに役立ちます。
* **コマンド例**:

    ```bash
    # applicationの実行中にmallocが呼ばれた回数を数える
    ltrace -f ./application 2>&1 | grep "malloc" | wc -l
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **プロセス指定** | |
| ⭐ `-p <PID>` | 既に実行中のプロセスにアタッチして追跡を開始します。 |
| `-f` | 子プロセスやスレッドも追跡します (`fork`, `clone`, `vfork`)。 |
| **出力制御** | |
| ⭐ `-o <file>` | 出力を画面ではなく、指定したファイルに書き込みます。 |
| ⭐ `-e <expr>` | `malloc,free` や `!strcpy` のように、追跡対象とする（または除外する）関数をフィルタリングします。 |
| `-c` | 実行終了時に、各コールにかかった時間や回数の統計サマリーを表示します。 |
| `-S` | ライブラリコールに加えて、**システムコールも表示**します (`strace` の機能も一部取り込みます)。 |
| `-l <library>` | 指定したライブラリへのコールのみを追跡します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: アプリケーションのクラッシュ（セグメンテーション違反など）の原因を調査する。
* **組み合わせ**: `ltrace /path/to/application`
* **解説**: `ltrace` で実行すると、クラッシュする直前に呼び出されたライブラリコールが判明することがあります。例えば、`strcpy` の引数に異常に長い文字列が渡されているのが観測されれば、バッファオーバーフローが原因であると強く推測できます。
* **例**:

    ```bash
    # セグメンテーション違反で落ちるプログラムを実行
    ltrace ./buggy_app
    # -> ...
    # -> strcpy("...", "A very very long string...") = 0x7ffc12345678
    # -> --- SIGSEGV (Segmentation fault) ---
    # -> +++ killed by SIGSEGV +++
    ```

### 2. ブルーチーム視点

* **タスク**: 動的解析で、マルウェアがどのようなライブラリ関数を利用しているか分析する。
* **組み合わせ**: `sudo ltrace -f -o ltrace_output.txt /path/to/malware`
* **解説**: `ltrace` の出力から、マルウェアのより高レベルな挙動を推測します。`libcrypto.so` の `EVP_Encrypt...` 関数呼び出しがあればランサムウェアの可能性、`libcurl.so` の `curl_easy_perform` 呼び出しがあればC2通信の可能性、といった具合に分析します。
* **例**:

    ```bash
    # マルウェアの挙動をファイルに記録
    sudo ltrace -f -o malware_trace.log ./malware_sample
    ```

### 3. レッドチーム視点

* **タスク**: 脆弱性分析のため、攻撃対象のバイナリが危険な関数をどのように使っているか調査する。
* **組み合わせ**: `ltrace /path/to/target_app`
* **解説**: ユーザーからの入力を受け取った後のライブラリコールの流れを追跡し、`gets`, `strcpy` といった境界チェックのない危険な文字列操作関数が使われていないか（バッファオーバーフロー）、`printf` にユーザー入力が直接渡されていないか（フォーマットストリング脆弱性）などを調査します。
* **例**:

    ```bash
    # ユーザー入力を待つプログラムをltraceで起動し、様々な入力を試す
    ltrace ./vulnerable_app
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。
* `ltrace` の実行には `ptrace` システムコールへのアクセス権が必要です。Dockerコンテナ内などで権限が不足していると、`ltrace: ptrace(PTRACE_ATTACH, ...): Operation not permitted` のようなエラーが出ることがあります。

## 環境変数と設定ファイル

* `ltrace` の動作に直接影響する環境変数や設定ファイルは通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `ltrace` 自体ではなく、その出力内容が**情報漏洩**に繋がります。
* **悪用シナリオ**: `ltrace` の出力には、ライブラリ関数に渡された引数が表示されます。これには、パスワード、暗号鍵、個人情報などの機密情報が含まれる可能性があります。

### GTFOBins / LOLBAS における利用例

* **機能**: **File Write**, **Privilege Escalation**
* **解説**: もしユーザーが `sudo ltrace` を実行できる権限を持っている場合、`-o` オプションを使って任意のファイルを上書きできます。
* **参照**: `https://gtfobins.github.io/gtfobins/ltrace/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: `sudoers` の設定を厳格にし、`ltrace` のようなデバッグツールに `sudo` 権限を安易に与えない。`kernel.yama.ptrace_scope` を設定し、`ptrace` の利用を制限する。
* **Detection (検知)**: `auditd` を使って `ltrace` コマンドの実行や `ptrace` システムコールの呼び出しを監視する。

## 注意点・補足

* **パフォーマンスへの影響**: プロセスを `ltrace` でトレースすると、実行速度は**著しく低下します**。デバッグや分析のためのツールであり、本番環境での常時監視には向きません。
* **`-S` オプション**: `-S` オプションを使うと、システムコールとライブラリコールを同時に表示でき、あるライブラリコールが内部でどのようなシステムコールを呼び出しているか、といったより深いレベルでの分析が可能になります。
