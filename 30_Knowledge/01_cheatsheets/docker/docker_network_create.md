# `docker network create` - 新しいDockerネットワークを作成する

## 概要

`docker network create` は、コンテナが相互に通信したり、外部ネットワークと通信したりするための新しい仮想ネットワークを作成するコマンドです。デフォルトで用意されている `bridge` ネットワークとは別に、**アプリケーションごとに専用のネットワークを作成する**ことが、コンテナ環境におけるセキュリティと管理のベストプラクティスです。カスタムネットワークに接続されたコンテナは、Dockerに組み込まれたDNSによって、互いにコンテナ名で名前解決・通信できるようになります。

(出自: `docker` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| ⭐ **`docker run --network`** | **コンテナの作成・起動時**に、所属するネットワークを指定します。`network create` は、その接続先となるネットワークを**事前に作成**するコマンドです。 | - |
| **`docker network rm`** | `docker network create` の逆の操作。ネットワークを**削除**します。 | - |

## よく連携されるコマンド

`docker network create` は、作成したネットワークにコンテナを接続する `docker run` コマンドと一連のワークフローとして連携します。

### シナリオ例: アプリケーション用のネットワークを作成し、コンテナを接続する (インフラ構築・運用視点)

* **目的**: Webサーバーとデータベースで構成されるアプリケーション用に専用ネットワークを作成し、各コンテナを接続する。
* **連携コマンド**: `docker run`
* **解説**: まず `docker network create` でネットワークを作成し、その後 `docker run` の `--network` オプションで、各コンテナをそのネットワークに接続します。
* **コマンド例**:

    ```bash
    # このコマンドシーケンスで、分離されたアプリケーション環境を構築する
    # 1. 'my-app-net' という名前の専用ネットワークを作成
    docker network create my-app-net

    # 2. DBコンテナを 'my-app-net' に接続
    docker run -d --name db --network my-app-net postgres:15

    # 3. WebAppコンテナを 'my-app-net' に接続
    docker run -d --name webapp --network my-app-net -p 8080:80 my-webapp-image
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **ドライバとトポロジ** | |
| ⭐ `-d`, `--driver <driver>` | 使用するネットワークドライバを指定します。デフォルトは `bridge` です。 |
| ⭐ `--subnet <subnet>` | ネットワークのサブネットをCIDR形式で指定します。（例: `--subnet=172.20.0.0/16`） |
| `--gateway <ip>` | ネットワークのゲートウェイのIPアドレスを指定します。 |
| `--internal` | 外部ネットワークへの接続を持たない、完全に内部的なネットワークを作成します。 |
| **その他** | |
| `-o`, `--opt <key=value>` | ドライバ固有のオプションを指定します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: アプリケーション専用のネットワークを作成する。
* **組み合わせ**: `docker network create <network_name>`
* **解説**: デフォルトの `bridge` ドライバで、アプリケーション専用のネットワークを作成する最も基本的な使い方です。これにより、コンテナ間の通信が他のアプリケーションから隔離されます。
* **例**:

    ```bash
    # アプリケーション 'my-app' 用のネットワークを作成
    docker network create my-app-net
    ```

### 2. ブルーチーム視点

* **タスク**: マルウェア分析用の隔離されたサンドボックスネットワークを作成する。
* **組み合わせ**: `docker network create --internal analysis-net`
* **解説**: `--internal` オプションを付けて、外部への通信が一切できない完全に隔離された内部ネットワークを作成します。マルウェアコンテナと、その通信を監視する調査用コンテナのみを接続することで、マルウェアが外部のC2サーバーと通信するのを防ぎながら、その挙動を安全に監視・分析できます。
* **例**:

    ```bash
    # 外部と通信できない分析用ネットワークを作成
    docker network create --internal analysis-net
    ```

### 3. レッドチーム視点

* **タスク**: 侵入したホスト上で、攻撃ツール用の隠蔽されたネットワークを作成する。
* **組み合わせ**: `docker network create --subnet=10.10.10.0/24 hidden-net`
* **解説**: 既存のネットワークとは重複しないカスタムサブネットを持つネットワークを作成します。攻撃者がC2ビーコン、スキャナーなどの複数の攻撃用ツールをコンテナとして展開する場合、それらを連携させるためのプライベートネットワークを作成し、ホスト上の他のアプリケーションネットワークから攻撃ツールの通信を隠蔽します。

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `Error response from daemon: network with name <network_name> already exists`
    * **考えられる原因**: 指定した名前のネットワークはすでに存在します。
    * **解決策**: `docker network ls` で既存のネットワークを確認してください。別の名前を使用するか、既存のネットワークが不要であれば `docker network rm` で削除します。

2. **エラーメッセージ例 2**: `Error response from daemon: Pool overlaps with other one on this address space`
    * **考えられる原因**: `--subnet` で指定したIPアドレス範囲が、既存のDockerネットワークやホストの物理ネットワークのIPアドレス範囲と重複しています。
    * **解決策**: 重複しない別のプライベートIPアドレス範囲を指定してください。

## 環境変数と設定ファイル

* `docker network create` の動作に直接影響する環境変数は通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `docker network create` 自体ではなく、**不適切なネットワークセグメンテーション**がリスクとなります。
* **悪用シナリオ**: 全てのコンテナをデフォルトの `bridge` ネットワークに接続すると、あるコンテナが侵害された場合に、他の無関係なコンテナにも被害が及ぶ（ラテラルムーブメント）可能性があります。

### GTFOBins / LOLBAS における利用例

`docker network create` は `docker` コマンドの一部として扱われます。`sudo docker` の権限があれば、任意のネットワーク操作が可能です。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **アプリケーションごとに専用のネットワークを作成し、コンテナを論理的に分離する**。データベースコンテナのように外部公開が不要なサービスは、`--internal` なネットワークにのみ接続する。
* **Detection (検知)**: `docker network ls` や `docker network inspect` を定期的に実行し、意図しないネットワークが作成されていないか、コンテナが不適切なネットワークに接続されていないかを監査する。

## 注意点・補足

* **DNSによる名前解決**: カスタムの `bridge` ネットワークに接続されたコンテナは、特別な設定なしに、お互いの**コンテナ名**をホスト名として名前解決できます。
