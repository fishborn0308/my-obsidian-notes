# `Dockerfile` - Dockerイメージの設計図

## 概要

`Dockerfile` は、`docker build` コマンドが読み込み、Dockerイメージを自動的に作成するための**設計図**となるテキストファイルです。ベースとなるOSイメージの指定、パッケージのインストール、ソースコードのコピー、実行するコマンドの定義など、イメージを構成するための**一連の命令 (Instruction)** を上から順に記述します。

`Dockerfile` を使うことで、アプリケーションの実行環境をコードとして管理（Infrastructure as Code）でき、誰がいつビルドしても同じイメージが作成される**再現性**と**移植性**を確保できます。

(出自: Dockerの基本機能)

## 類似技術との差異

| 技術 | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| **シェルスクリプト** | `Dockerfile` はDockerの**レイヤーキャッシュ**を利用できるため、変更がない部分は再利用されビルドが高速です。また、コンテナという隔離された環境の定義に特化しています。 | `Dockerfile` (イメージビルド) |
| **`docker commit`** | 実行中のコンテナの状態からイメージを作成しますが、手順が不明確で**再現性がありません**。**再現性のあるイメージ作成には必ず `Dockerfile` を使います**。 | `Dockerfile` |

## よく連携されるコマンド

`Dockerfile` は、`docker build` コマンドによって解釈・実行されます。

### シナリオ例: 基本的なWebアプリケーションのビルド (インフラ構築・運用視点)

* **目的**: `Dockerfile` を使って、Node.jsアプリケーションのDockerイメージを作成する。
* **連携コマンド**: `docker build`
* **解説**: `Dockerfile` があるディレクトリで `docker build` を実行するのが基本的なワークフローです。
* **コマンド例**:

    ```bash
    # カレントディレクトリのDockerfileを基に、my-nodejs-app:1.0 というイメージをビルド
    docker build -t my-nodejs-app:1.0 .
    ```

## 📜 主要な命令 (Instruction)

| 命令 | 説明 |
| :--- | :--- |
| ⭐ **`FROM`** | ビルドの基盤となる**ベースイメージ**を指定します。全ての `Dockerfile` はこの命令から始まります。 |
| ⭐ **`RUN`** | イメージをビルドする過程で実行されるシェルコマンド。パッケージのインストールなどに使います。 |
| ⭐ **`COPY`** | ホストマシンからファイルやディレクトリをイメージ内にコピーします。 |
| **`ADD`** | `COPY` と似ていますが、URLからのダウンロードや、`tar` ファイルの自動展開機能も持ちます。**通常は `COPY` の使用が推奨されます**。 |
| ⭐ **`WORKDIR`** | これ以降の `RUN`, `CMD`, `COPY` などの命令を実行する**作業ディレクトリ**を指定します。 |
| ⭐ **`CMD`** | コンテナ起動時に**デフォルトで実行されるコマンド**。`docker run` で引数が指定されると上書きされます。 |
| **`ENTRYPOINT`** | コンテナ起動時に**必ず実行されるコマンド**。`CMD` は `ENTRYPOINT` の引数として機能します。 |
| **`EXPOSE`** | コンテナがリッスンするネットワークポートをドキュメント化します。 |
| **`ENV`** | イメージの**環境変数**を設定します。 |
| **`ARG`** | **ビルド時**にのみ有効な変数を定義します。 |
| ⭐ **`USER`** | これ以降の命令を実行するユーザーを指定します。**セキュリティ上、root以外のユーザーを指定することが強く推奨されます**。 |

## よく使われる命令の組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: **マルチステージビルド**を利用して、最終的なイメージサイズを最小化し、セキュリティを向上させる。
* **組み合わせ**: 複数の `FROM` 命令、 `COPY --from=...`
* **解説**: ビルド用のステージと実行用のステージを分離します。最終的なイメージに不要なツール（コンパイラなど）が含まれるのを防ぎます。
* **例 (Go言語)**:

    ```Dockerfile
    # --- ビルドステージ ---
    FROM golang:1.22-alpine AS builder
    WORKDIR /app
    COPY . .
    RUN go build -o my-app .

    # --- 実行ステージ ---
    FROM alpine:latest
    WORKDIR /app
    # ビルドステージから、コンパイル済みのバイナリのみをコピー
    COPY --from=builder /app/my-app .
    USER appuser
    CMD ["./my-app"]
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `Step X/Y: RUN apt-get install ... The command '/bin/sh -c apt-get install ...' returned a non-zero code: 100`
    * **考えられる原因**: `Dockerfile` 内の `RUN` 命令が失敗しました。多くの場合、パッケージが見つからない (`apt-get update` を忘れている)、ネットワークの問題、スクリプトのエラーなどが原因です。
    * **解決策**: エラーメッセージの直前の出力を確認し、`Dockerfile` の該当行を修正します。

## 環境変数と設定ファイル

* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。
* **`.dockerignore`**: `COPY` や `ADD` でイメージに含めたくないファイル（`.git`, `.env`など）を記述します。機密情報の混入を防ぎ、ビルドを高速化するために**不可欠**です。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **サプライチェーン攻撃**。`FROM` で指定するベースイメージが悪意のあるものだったり、`RUN` で信頼できないソースからスクリプトをダウンロード・実行したりすると、イメージ全体が侵害される可能性があります。
* **悪用シナリオ**: 攻撃者が公開リポジトリ（GitHubなど）の `Dockerfile` を分析し、古いベースイメージの脆弱性を突いて、ビルドされたコンテナに侵入します。

### GTFOBins / LOLBAS における利用例

`Dockerfile` は設定ファイルのため、GTFOBinsにはリストされていません。しかし、それがビルドするイメージは、GTFOBinsにリストされている多くのコマンド（`bash`, `curl`など）を含んでいます。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 公式の軽量なベースイメージを使用する (`alpine`, `distroless`)。非rootユーザーでプロセスを実行する (`USER` 命令)。マルチステージビルドを活用する。`.dockerignore` を徹底する。
* **Detection (検知)**: `Trivy` などの脆弱性スキャンツールで、ビルドしたイメージに既知の脆弱性がないかをCI/CDパイプラインでスキャンする。

## 注意点・補足

* **レイヤーキャッシュ**: Dockerは `Dockerfile` の各命令をレイヤーとしてキャッシュします。変更がない部分は再利用されるため、変更頻度の低い命令を先に、高い命令 (`COPY . .`など) を後に配置するとビルドが高速化します。
