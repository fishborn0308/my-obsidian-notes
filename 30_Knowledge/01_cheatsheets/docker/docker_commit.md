# `docker commit` - コンテナの変更から新しいイメージを作成する

## 概要

`docker commit` は、コンテナの現在のファイルシステムの状態（作成元のイメージからの差分）を、新しいDockerイメージとして保存するコマンドです。手動でコンテナ内に入ってパッケージをインストールしたり、設定ファイルを変更したりした後に、その状態をスナップショットとしてイメージ化できます。

**⚠️ 重要**: このコマンドは、どのようにイメージが作成されたかの手順が `Dockerfile` のようにコードとして残らないため、**再現性がなく、"ブラックボックス"なイメージ**を生み出します。本番環境用のイメージ作成には**非推奨**であり、代わりに `Dockerfile` と `docker build` を使用すべきです。

(出自: `docker` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| ⭐ **`docker build`** | `Dockerfile` という**再現可能な**設計図に基づいてイメージをビルドします。`docker commit` は、アドホックな手動変更を**再現性なく**イメージ化します。 | `docker build` |

## よく連携されるコマンド

`docker commit` は、状態を保存したいコンテナを特定するために `docker ps` と連携するのが一般的なワークフローです。

### シナリオ例: 侵害されたコンテナの状態を保全する (ブルーチーム視点)

* **目的**: インシデント調査のため、マルウェアが活動している可能性のあるコンテナの現在の状態（ファイル、プロセス情報などを含む）を、分析用のイメージとして完全に保全する。
* **連携コマンド**: `docker ps`, `docker stop`
* **解説**: `docker commit` は、ブルーチームにとって**フォレンジック調査の際に極めて重要**なコマンドです。実行中のコンテナを `commit` することで、メモリ上の状態は失われますが、ファイルシステムの変更点（マルウェアのバイナリ、攻撃者が作成したファイルなど）を全て含んだイメージを作成できます。
* **コマンド例**:

    ```bash
    # 1. 調査対象のコンテナIDを確認
    docker ps

    # 2. (任意) これ以上状態が変わらないようにコンテナを停止
    docker stop suspicious-container

    # 3. コンテナの状態を新しいイメージとして保存
    docker commit suspicious-container malware-evidence:case-001
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **コミット情報** | |
| ⭐ `<container>` `<repository>[:<tag>]` | コミット対象のコンテナと、作成する新しいイメージの名前・タグを指定します。 |
| ⭐ `-a`, `--author <string>` | イメージの作者情報を設定します。（例: `"John Doe <john.doe@example.com>"`） |
| ⭐ `-m`, `--message <string>` | イメージ作成の経緯などを記録するコミットメッセージを設定します。 |
| **プロセス制御** | |
| `-p`, `--pause` | コミット中にコンテナの全プロセスを一時停止します。デフォルトは `true` です。 |
| `-c`, `--change <instruction>` | 作成されるイメージに対して、`CMD` や `ENTRYPOINT` などの `Dockerfile` 命令を適用します。 |

### 1. インフラ構築・運用視点

* **タスク**: デバッグ中に手動でインストールしたツールを含む、一時的なデバッグ用イメージを作成する。

### 3. レッドチーム視点

* **タスク**: **バックドアを仕込んだイメージの作成**。
* **組み合わせ**: `docker commit <backdoored_container> <legitimate_image_name>:<tag>`
* **解説**: 攻撃者は正規のイメージからコンテナを起動し、`docker exec` で内部にリバースシェルなどのバックドアを仕込みます。その後、`docker commit` を使って、その変更を含む新しいイメージを作成します。さらに、そのイメージに `ubuntu:latest` のような正規のイメージに見せかけたタグを付けることで、管理者の目を欺き、永続化を図ります。

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `Error: No such container: <container_name>`
    * **考えられる原因**: 指定した名前またはIDのコンテナが存在しません。
    * **解決策**: `docker ps -a` を実行して、存在するコンテナの名前とIDを再確認してください。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **再現性の欠如と "ブラックボックス"化**。
* **悪用シナリオ**: `docker commit` で作成されたイメージは、その元となった `Dockerfile` が存在しないため、どのようなソフトウェアが含まれ、どのような設定がされているかが不明確になります。攻撃者は、この性質を悪用してマルウェアを仕込んだイメージを正規のものと偽って配布する可能性があります。

### GTFOBins / LOLBAS における利用例

`docker commit` は `docker` コマンドの一部として扱われます。`sudo docker` の権限があれば、任意の操作が可能です。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **`docker commit` の使用を原則禁止**し、全てのイメージ作成を `Dockerfile` と `docker build` で行うことを徹底する。これにより、イメージの作成プロセスがコードとしてレビュー・監査可能になります。
* **Detection (検知)**: `docker history` コマンドでイメージのレイヤーを調査し、`"created by /bin/sh -c ..."` のような `docker commit` によって作成されたレイヤーがないかを確認する。

## 注意点・補足

* **ボリュームは含まれない**: `docker commit` は、コンテナのファイルシステムの変更のみをイメージに含みます。`-v` や `--mount` でマウントされた**ボリューム内のデータは、コミットの対象外**です。
