# `docker system prune` - 未使用のDockerリソースを一括削除する

## 概要

`docker system prune` は、Dockerホスト上に存在する**未使用の**リソースを一度にまとめて削除するためのコマンドです。デフォルトでは、以下のオブジェクトが削除対象となります。

* 停止しているすべてのコンテナ
* どのコンテナからも参照されていないすべてのネットワーク
* danglingイメージ（タグを持たない中間イメージ）
* 未使用のビルドキャッシュ

ディスク容量を解放し、Docker環境をクリーンな状態に保つための、最も手軽で強力なメンテナンスコマンドです。

(出自: `docker` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| ⭐ **`docker <obj> prune`** | `container prune`, `image prune` など、**特定の種類のリソースだけ**を対象にクリーンアップします。`system prune` はこれらをまとめて実行する包括的なコマンドです。 | - |
| **`docker rm`, `docker rmi`** | **個別の**コンテナやイメージを指定して削除します。`prune` は条件に合うものを**一括で**削除します。 | `prune` (一括削除) |
| **`docker system df`** | Dockerが使用するディスク容量の内訳を表示します。`prune` は、`df` で表示された**解放可能なリソースを実際に削除する**ためのコマンドです。 | - |

## よく連携されるコマンド

`docker system prune` は、実行前に `docker system df` で解放可能な容量を確認するという流れでよく使われます。

### シナリオ例: ディスク使用状況を確認し、クリーンアップする (インフラ構築・運用視点)

* **目的**: Dockerホストのディスク容量が逼迫してきた際に、解放可能な容量を確認し、安全にクリーンアップする。
* **連携コマンド**: `docker system df`
* **解説**: まず `docker system df` を実行して、`RECLAIMABLE` な容量がどれくらいあるかを確認します。この値が大きければ、`docker system prune` を実行することで安全に容量を確保できると判断できます。
* **コマンド例**:

    ```bash
    # 1. Dockerのディスク使用状況を確認
    docker system df

    # 2. (RECLAIMABLEな容量が多いことを確認後) 未使用リソースを一括削除
    docker system prune
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **削除対象の拡張** | |
| ⭐ `-a`, `--all` | **未使用のすべてのイメージ**（danglingイメージだけでなく、どのコンテナからも使用されていないすべてのイメージ）も削除対象に含めます。 |
| ⭐ `--volumes` | どのコンテナからも使用されていない**名前付きボリューム**も削除対象に含めます。**データベースのデータなどが消える可能性があるため、最も注意が必要なオプションです**。 |
| **動作制御** | |
| ⭐ `-f`, `--force` | 対話形式の確認プロンプト (`Are you sure ...?`) を表示せず、強制的に実行します。スクリプト内での使用に適しています。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 開発サーバーやCI/CDのビルドサーバーで、不要なリソースを定期的にクリーンアップする。
* **組み合わせ**: `docker system prune -f`
* **解説**: スクリプト内で `-f` (`--force`) オプションを使い、確認プロンプトなしで未使用リソースを削除します。これにより、CIジョブごとに作成されたコンテナやキャッシュがサーバーに溜まっていくのを防ぎます。
* **例**:

    ```bash
    # 定期的なメンテナンスとして実行
    docker system prune
    ```

### 2. ブルーチーム視点

* **タスク**: インシデント対応後のクリーンアップ作業。
* **組み合わせ**: `docker system prune`
* **解説**: 侵害されたコンテナを停止・調査・証拠保全した後、その残骸（停止したコンテナなど）をシステムから除去するために使用します。ただし、**フォレンジック調査が完全に完了する前に実行すると、重要な証拠を破壊してしまう**可能性があるため、実行のタイミングは慎重に判断する必要があります。

### 3. レッドチーム視点

* **タスク**: **痕跡消去 (Anti-Forensics)**。自身の活動の痕跡を消去する。
* **組み合わせ**: `docker system prune -f`
* **解説**: 攻撃者が、侵入したホスト上で攻撃用ツールをコンテナとして一時的に実行した場合、活動終了後に `docker system prune -f` を実行することで、使用したコンテナ（停止後）や中間イメージの残骸を削除し、調査を困難にします。

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **現象**: **コマンドを実行しても、期待したほどディスク容量が解放されない。**
    * **考えられる原因**:
        * 削除したいイメージが、まだ（停止中の）コンテナから参照されている。
        * デフォルトの `prune` は未使用のイメージ（danglingではない）やボリュームを削除しない。
    * **解決策**: より強力な `-a` や `--volumes` オプションの使用を検討しますが、その前に `docker ps -a` や `docker volume ls` で、どのリソースが残っているのかを個別に確認し、本当に不要なものであるかを判断することが重要です。

## 環境変数と設定ファイル

* `docker system prune` の動作に直接影響する環境変数は通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `docker system prune` の**破壊的な性質**がリスクとなります。
* **悪用シナリオ**:
  * **サービス妨害 (DoS)**: `-a` オプションを誤って使用すると、次のデプロイで必要となるベースイメージまで削除してしまい、サービス復旧が遅延する可能性があります。
  * **データ損失**: `--volumes` オプションは、データベースのデータが格納された名前付きボリュームを削除する可能性があり、回復不可能なデータ損失に繋がります。

### GTFOBins / LOLBAS における利用例

`docker system prune` は `docker` コマンドの一部として扱われます。`sudo docker` の権限があれば、任意の操作が可能です。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 本番環境での安易な `prune` 実行を避け、変更管理プロセスを通じて計画的に行う。重要なデータボリュームにはラベルを付け、`--filter "label!=keep"` のようにして `prune` の対象から除外する。
* **Detection (検知)**: `auditd` で `docker system prune` の実行を監視し、計画外のクリーンアップ操作がないかを確認する。

## 注意点・補足

* **個別 `prune` のすすめ**: `system prune` は強力すぎる場合があるため、慣れないうちは `docker container prune`, `docker image prune -a`, `docker volume prune` のように、リソースの種類ごとに個別の `prune` コマンドを実行する方が、何を削除しているかを意識しやすく安全です。
