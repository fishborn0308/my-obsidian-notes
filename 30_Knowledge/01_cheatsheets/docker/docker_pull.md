# `docker pull` - レジストリからDockerイメージを取得する

## 概要

`docker pull` は、Docker Hubやプライベートレジストリなどのコンテナイメージレジストリから、指定したDockerイメージをローカルホストにダウンロードするためのコマンドです。`docker run` を実行した際にローカルにイメージが存在しない場合、Dockerは暗黙的に `docker pull` を実行しますが、事前に `pull` しておくことで、コンテナの起動時間を短縮したり、使用するイメージのバージョンを明示的に指定したりできます。

(出自: `docker` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| ⭐ **`docker run`** | ローカルにイメージが存在しない場合、自動的に `pull` を実行してからコンテナを起動します。`pull` はイメージの取得だけを行います。 | - |
| **`docker build`** | `Dockerfile` を基に**新しいイメージをローカルで作成**します。`pull` は**既存の完成したイメージをリモートから取得**します。 | - |
| **`git pull`** | `git` のコマンドであり、ソースコードリポジトリから最新の変更を取得します。`docker pull` が扱うのは**コンテナイメージ**です。 | - |

## よく連携されるコマンド

`docker pull` は、ダウンロードしたイメージを元にコンテナを起動する `docker run` や、イメージの脆弱性をスキャンするツールと連携します。

### シナリオ例: イメージを取得して脆弱性をスキャンする (ブルーチーム視点)

* **目的**: デプロイ前に、利用する公式イメージに既知の脆弱性が含まれていないかを確認する。
* **連携コマンド**: `Trivy` (または他のスキャンツール)
* **解説**: `docker pull` で対象のイメージを取得した後、`trivy image` コマンドでそのイメージをスキャンします。これにより、安全なイメージのみをデプロイするワークフローを確立できます。
* **コマンド例**:

    ```bash
    # 1. 最新の Nginx イメージを取得
    docker pull nginx:latest

    # 2. Trivyで脆弱性をスキャン
    trivy image nginx:latest
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **ダウンロード制御** | |
| ⭐ `<image>[:<tag>]` | `nginx:1.25.4` のように、ダウンロードするイメージとタグを指定します。 |
| `-a`, `--all-tags` | 指定したリポジトリに存在する**すべてのタグ**のイメージをダウンロードします。 |
| **その他** | |
| `--platform <os/arch>` | マルチアーキテクチャイメージの場合に、ダウンロードするプラットフォームを指定します。（例: `--platform linux/amd64`） |
| `-q`, `--quiet` | ダウンロードの進捗状況などを表示せず、静かに出力します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 再現性を担保するため、バージョンを固定したイメージをデプロイ前に取得する。
* **組み合わせ**: `docker pull <image_name>:<tag>`
* **解説**: `latest` タグではなく `nginx:1.25.4` のようにバージョンを固定することで、意図しないバージョンのイメージがデプロイされるのを防ぎます。
* **例**:

    ```bash
    # Nginxの特定のバージョンを取得
    docker pull nginx:1.25.4

    # Ubuntu 22.04 のイメージを取得
    docker pull ubuntu:22.04
    ```

### 2. ブルーチーム視点

* **タスク**: 脆弱性レポートで報告された特定のアーキテクチャのイメージを分析用にダウンロードする。
* **組み合わせ**: `docker pull --platform <os/arch> <image>:<tag>`
* **解説**: ある脆弱性が `arm64` アーキテクチャのイメージでのみ再現すると報告された場合に、そのイメージを正確に取得し、脆弱性の有無を検証します。
* **例**: -

### 3. レッドチーム視点

* **タスク**: 侵入したホスト上に、攻撃用のツール群が含まれたコンテナイメージをダウンロードする。
* **組み合わせ**: `docker pull <attacker_tool_image>`
* **解説**: 攻撃者は、`nmap` や `metasploit` などをプリインストールした自前のイメージをDocker Hubにアップロードしておき、侵入後に `pull` することで、ホストにツールをインストールする痕跡を残さずに迅速に攻撃環境を整えます。
* **例**:

    ```bash
    # (攻撃者が準備した) Kali Linuxベースのツールイメージをダウンロード
    docker pull attacker/kali-tools:latest
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `Error response from daemon: manifest for <image>:<tag> not found`
    * **考えられる原因**: 指定したイメージ名またはタグが存在しません。
    * **解決策**: Docker Hubなどで、正しいイメージ名と利用可能なタグを確認してください。

2. **エラーメッセージ例 2**: `Error response from daemon: pull access denied for <image>, repository does not exist or may require 'docker login'`
    * **考えられる原因**: プライベートリポジトリのイメージを取得しようとしていますが、認証(ログイン)が済んでいません。
    * **解決策**: `docker login <registry_host>` コマンドを実行し、認証してから再度 `pull` を実行してください。

## 環境変数と設定ファイル

* `docker pull` の動作に直接影響する環境変数は通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **サプライチェーン攻撃**。
* **悪用シナリオ**: Docker Hubなどの公開レジストリには、誰でもイメージをアップロードできます。悪意のある第三者が作成したイメージには、**マルウェア、クリプトマイナー、バックドアなどが仕込まれている**可能性があります。公式イメージに似せた名前（タイポスクワッティング）の悪意のあるイメージをダウンロードさせてしまう攻撃もあります。

### GTFOBins / LOLBAS における利用例

`docker pull` は `docker` コマンドの一部として扱われます。`sudo docker` の権限があれば、任意の操作が可能です。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **公式イメージ・認証済みイメージを使用する**。組織内で使用を許可したイメージのみを格納するプライベートレジストリを運用する。
* **Detection (検知)**: `docker pull` したイメージは、`Trivy` などのツールを使って必ず**脆弱性スキャン**を行ってから使用する。

## 注意点・補足

* **`latest` タグ**: タグを省略すると、Dockerは自動的に `:latest` タグを補完します。しかし、`latest` は必ずしも最新版を指すとは限らないため、**常に具体的なバージョンタグを指定する**ことが強く推奨されます。
