# `docker rmi` - 1つ以上のDockerイメージを削除する

## 概要

`docker rmi` (または `docker image rm`) は、Dockerホストのローカルストレージから、指定した1つ以上のDockerイメージを削除するためのコマンドです。`docker pull` や `docker build` で作成・取得されたイメージのうち、古くなったバージョンや不要になったものを削除することで、ディスク容量を解放し、イメージリストを整理整頓します。

(出自: `docker` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| ⭐ **`docker rm`** | **コンテナ**を削除します。`rmi` は**イメージ**（コンテナの設計図）を削除します。 | - |
| **`docker image prune`** | **danglingイメージ**（タグを持たない中間イメージ）や**未使用のイメージ**を一括で削除します。`rmi` は削除するイメージを個別に指定します。 | `prune` (一括削除) |

## よく連携されるコマンド

`docker rmi` は、不要なイメージを特定する `docker images` と連携して、クリーンアップ作業で頻繁に使われます。

### シナリオ例: 不要な中間イメージを一括削除する (インフラ構築・運用視点)

* **目的**: 新しいイメージをビルドした際に残った、どのタグからも参照されなくなった古いイメージ（dangling image）をクリーンアップする。
* **連携コマンド**: `docker images`, `docker rmi`, コマンド置換 (`$()`)
* **解説**: `docker images` に `--filter "dangling=true"` を付けてdanglingイメージを絞り込み、`-q` でそのIDのみを取得します。その結果を `docker rmi` に渡して一括で削除します。
* **コマンド例**:

    ```bash
    # danglingイメージを検索し、見つかったものを一括削除
    docker rmi $(docker images -f "dangling=true" -q)

    # 注: この操作は `docker image prune` コマンドでより簡単に行えます。
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **削除対象** | |
| ⭐ `<image...>` | 削除するイメージの名前:タグまたはイメージIDを一つ以上、スペース区切りで指定します。 |
| **動作制御** | |
| ⭐ `-f`, `--force` | 1つ以上のコンテナがそのイメージを使用していても、強制的にイメージを削除します。**コンテナが異常動作する原因となるため、通常は非推奨です**。 |
| `--no-prune` | イメージを削除する際に、その親レイヤー(dangling状態になる場合)を削除しません。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 不要になった古いバージョンのアプリケーションイメージを削除してディスク容量を確保する。
* **組み合わせ**: `docker rmi <image_name>:<tag>`
* **解説**: `docker images` で確認した、不要になったイメージを個別に削除する最も基本的な使い方です。
* **例**:

    ```bash
    # 不要になった古いバージョンのイメージを削除
    docker rmi my-app:1.0
    ```

### 2. ブルーチーム視点

* **タスク**: 脆弱性が発見されたイメージをシステムから完全に削除する。
* **組み合わせ**: `docker rmi <vulnerable_image_id>`
* **解説**: `Trivy` などのスキャナーで深刻な脆弱性が発見されたイメージが、誤って実行されるのを防ぐためにホストから削除します。インシデント対応の一環として、悪意のあるイメージを無力化する際にも使用します。
* **例**: -

### 3. レッドチーム視点

* **タスク**: **痕跡消去 (Anti-Forensics)**。攻撃に使用したツールイメージを削除する。
* **組み合わせ**: `docker rmi <attacker_tool_image_id>`
* **解説**: 攻撃者が、侵入したホスト上で `docker pull` を使って攻撃用ツールのイメージをダウンロードした場合、目的を達成した後にそのイメージを `rmi` で削除します。これにより、`/var/lib/docker` 以下にあるイメージの実体を消去し、調査を困難にします。

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `Error response from daemon: conflict: unable to remove repository reference "<image_name>" (must force) - container <container_id> is using its referenced image`
    * **考えられる原因**: **最もよくあるエラー**。削除しようとしているイメージが、まだ存在するコンテナ（実行中または停止中）によって使用されています。
    * **解決策**: エラーメッセージに表示されたコンテナIDを `docker rm` で先に削除してください。その後、再度 `docker rmi` を実行します。

## 環境変数と設定ファイル

* `docker rmi` の動作に直接影響する環境変数や設定ファイルは通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `docker rmi` は**破壊的なコマンド**です。
* **悪用シナリオ**:
  * **サービス妨害 (DoS)**: 攻撃者が本番環境で必要なイメージを削除すると、その後のデプロイやコンテナの自動復旧が失敗し、サービス停止に繋がります。
  * **証拠の破壊**: インシデント発生時に、攻撃者が使用した悪意のあるイメージを `rmi` で削除されると、そのイメージ自体の分析が困難になります。

### GTFOBins / LOLBAS における利用例

`docker rmi` は `docker` コマンドの一部として扱われます。`sudo docker` の権限があれば、任意のイメージを削除可能です。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: Dockerイメージレジストリで、重要なイメージタグが上書き・削除されないように保護設定を行う。Dockerソケットへのアクセス権を厳格に管理する。
* **Detection (検知)**: `auditd` で `docker rmi` コマンドの実行を監視し、計画外のイメージ削除がないかを確認する。

## 注意点・補足

* **レイヤーの共有**: Dockerイメージは複数のレイヤーから構成されています。`docker rmi` を実行した際、削除対象のイメージにしか使われていないレイヤーは削除されますが、他のイメージと共有されているレイヤーは削除されずに残ります。
