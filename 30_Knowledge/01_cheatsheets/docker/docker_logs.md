# `docker logs` - コンテナのログを取得する

## 概要

`docker logs` は、指定したコンテナのログを取得するためのコマンドです。コンテナ内で実行されているメインプロセスの**標準出力 (stdout)** と**標準エラー出力 (stderr)** に書き込まれた内容を収集・表示します。アプリケーションが正常に動作しているかの確認、エラー発生時の原因調査、パフォーマンス問題のデバッグなど、コンテナのトラブルシューティングにおける最も基本的な手段の1つです。

(出自: `docker` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| ⭐ **`docker compose logs`** | `docker-compose.yml` で定義された**プロジェクト全体の複数コンテナ**のログをまとめて表示します。 | `docker compose logs` (Compose環境) |
| **`docker attach`** | 実行中のコンテナの**メインプロセス(PID 1)**の標準入出力に、現在のターミナルを接続します。過去のログは表示されません。 | `docker logs` |
| **`docker exec`** | コンテナ内の**ファイルに直接ログを書き込んでいる場合**に使います。`exec` でコンテナ内に入り、ログファイルを直接閲覧します。 | - |

## よく連携されるコマンド

`docker logs` は、`docker ps -a` で起動に失敗したコンテナを特定した後に、その原因を調査するために連携して使われます。

### シナリオ例: コンテナ起動失敗の原因調査 (インフラ構築・運用視点)

* **目的**: `docker run` で起動したコンテナがすぐに `Exited` 状態になってしまう原因を調査する。
* **連携コマンド**: `docker ps`, `docker logs`
* **解説**: まず `docker ps -a` で停止したコンテナのIDまたは名前を確認します。次に、そのIDを使って `docker logs` を実行し、コンテナが終了する直前にどのようなエラーメッセージを出力したかを確認します。
* **コマンド例**:

    ```bash
    # 1. 停止したコンテナを確認
    docker ps -a

    # 2. 起動に失敗したコンテナ (例: my-app-container) のログを確認
    docker logs my-app-container
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **表示制御** | |
| ⭐ `-f`, `--follow` | ログの出力をリアルタイムで追跡します (`tail -f` と同様)。`Ctrl+C` で終了します。 |
| ⭐ `--tail <number>` | ログの末尾から指定した行数だけを表示します。（例: `--tail 100`） |
| `-t`, `--timestamps` | 各ログ行の先頭にタイムスタンプを付けて表示します。 |
| **フィルタリング** | |
| ⭐ `--since <timestamp>` | 指定したタイムスタンプ**以降**のログのみを表示します。`"2025-09-22T10:00:00"` や `"10m"` (10分前) のように指定します。 |
| `--until <timestamp>` | 指定したタイムスタンプ**以前**のログのみを表示します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: アプリケーションの動作をリアルタイムでデバッグする。
* **組み合わせ**: `docker logs -f <container_name_or_id>`
* **解説**: コンテナのログをリアルタイムで表示し続けます。アプリケーションにリクエストを送りながらこのコマンドでログを監視し、予期せぬエラーが発生していないかなどを確認します。
* **例**:

    ```bash
    # webappコンテナのログをリアルタイムで監視
    docker logs -f webapp
    ```

### 2. ブルーチーム視点

* **タスク**: 特定の時間帯のログだけをピンポイントで抽出し、不正アクセスの調査を行う。
* **組み合わせ**: `docker logs --since '1h' --until '30m' <suspicious_container>`
* **解説**: `--since` と `--until` を使い、不正アクセスが検知された時間帯に絞ってログを調査します。これにより、ノイズを排除し、攻撃者の活動に関連するログを効率的に発見できます。
* **例**:

    ```bash
    # 2025年9月22日の午前2時から午前3時の間のログを調査
    docker logs --since "2025-09-22T02:00:00" --until "2025-09-22T03:00:00" -t web-server
    ```

### 3. レッドチーム視点

* **タスク**: 侵入したホスト上で稼働しているコンテナのログを偵察し、機密情報を探す。
* **組み合わせ**: `docker logs <container_name> | grep "password\|token\|key"`
* **解説**: 開発者がデバッグ目的で誤ってログに出力してしまったパスワード、APIキー、セッショントークンなどの機密情報を探します。特にWebサーバーやAPIゲートウェイのコンテナログは、貴重な情報が含まれている可能性があります。

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **現象**: **`docker logs` を実行しても何もログが表示されない。**
    * **考えられる原因**:
        * アプリケーションが標準出力/標準エラー出力ではなく、コンテナ内のファイルに直接ログを書き込んでいる。
        * コンテナのロギングドライバが `none` に設定されている。
    * **解決策**: `docker inspect <container>` で `LogPath` や `Driver` を確認します。アプリケーションがファイルにログを出力している場合は `docker exec` でコンテナ内に入り、そのファイルを直接 `cat` や `tail` で確認する必要があります。

## 環境変数と設定ファイル

* `docker logs` の動作に直接影響する環境変数や設定ファイルは通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **ログに含まれる機密情報**。
* **悪用シナリオ**: アプリケーションログには、個人情報(PII)、セッショントークン、APIキーなどが誤って出力されることがあります。`docker logs` を実行できる権限を持つ攻撃者は、これらの情報を窃取できます。

### GTFOBins / LOLBAS における利用例

`docker logs` は情報表示が主目的のためGTFOBinsにはリストされていません。しかし、リストされている `docker` コマンドによる権限昇格攻撃の**偵察段階**で利用されます。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 本番環境では機密情報をログに出力しないようにアプリケーションを実装する。
* **Detection (検知)**: **ログの集中管理**。`fluentd`, `splunk` などのロギングドライバを使い、コンテナログを改ざん困難な中央ログサーバー (SIEMなど) にリアルタイムで転送する。

## 注意点・補足

* **ロギングドライバ**: Dockerはコンテナのログをどのように扱うかを「ロギングドライバ」で設定できます。デフォルトは `json-file` で、ログをホスト上のJSONファイルに保存します。環境に応じて、`journald`, `syslog`, `fluentd` など、様々なドライバを選択できます。
