# `docker images` - ローカルのDockerイメージを一覧表示する

## 概要

`docker images` は、Dockerホストのローカルストレージに保存されているDockerイメージを一覧表示するためのコマンドです。このコマンドを使うことで、利用可能なイメージ、そのバージョン(タグ)、イメージID、作成日、サイズなどを確認できます。コンテナを起動する (`docker run`) 前に目的のイメージがローカルに存在するかを確認したり、不要なイメージを削除 (`docker rmi`) するためにリストアップしたりする際に使用します。

(出自: `docker` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| ⭐ **`docker ps`** | **コンテナ**の一覧を表示します。`docker images` はその元となる**イメージ**（設計図）の一覧を表示します。 | - |
| **`docker image ls`** | `docker images` の**エイリアス(別名)**です。機能は全く同じです。 | - |
| **`docker system df`** | イメージ、コンテナ、ボリュームなどが使用しているディスク容量の**サマリー**を表示します。 | - |

## よく連携されるコマンド

`docker images` は、不要なイメージを特定し、`docker rmi` と連携して削除するクリーンアップ作業で頻繁に使われます。

### シナリオ例: 不要な中間イメージを一括削除する (インフラ構築・運用視点)

* **目的**: 新しいイメージをビルドした際に残った、どのタグからも参照されなくなった古いイメージ（dangling image）をクリーンアップする。
* **連携コマンド**: `docker rmi`, コマンド置換 (`$()`)
* **解説**: `docker images` に `--filter "dangling=true"` を付けてdanglingイメージを絞り込み、`-q` でそのIDのみを取得します。その結果を `docker rmi` に渡して一括で削除します。
* **コマンド例**:

    ```bash
    # danglingイメージを検索し、見つかったものを一括削除
    docker rmi $(docker images -f "dangling=true" -q)

    # 注: この操作は `docker image prune` コマンドでより簡単に行えます。
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **表示対象** | |
| ⭐ `-a`, `--all` | 中間イメージ(ビルド過程で作成されたレイヤーなど)も含め、すべてのイメージを表示します。 |
| **出力制御** | |
| ⭐ `-q`, `--quiet` | イメージIDのみを表示します。他のコマンドと組み合わせる際に非常に便利です。 |
| `--digests` | イメージのダイジェスト値(内容から計算される不変のハッシュ値)も表示します。 |
| `--format <Go template>` | Go言語のテンプレートを使って、出力形式を自由にカスタマイズします。 |
| **フィルタリング** | |
| ⭐ `--filter <key>=<value>` | 特定の条件で表示するイメージをフィルタリングします。（例: `--filter "dangling=true"`） |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 新しいコンテナを起動する前に、ローカルにどのようなイメージが、どのバージョンで存在するのかを確認する。
* **組み合わせ**: `docker images`
* **解説**: オプションなしの最も基本的な使い方。定期的に実行して、古くなった、あるいは不要なイメージがないかを棚卸しします。
* **例**:

    ```bash
    # ローカルにある全イメージを確認
    docker images
    ```

### 2. ブルーチーム視点

* **タスク**: イメージの同一性を厳密に検証する。
* **組み合わせ**: `docker images --digests`
* **解説**: `--digests` オプションで、イメージの `IMAGE ID` だけでなく、内容に基づいた SHA256 ダイジェスト値も表示します。`IMAGE ID` はローカルで重複する可能性がありますが、ダイジェスト値はイメージの内容が1ビットでも異なれば変わるため、より厳密な同一性検証が可能です。
* **例**:

    ```bash
    # 監査のため、イメージのダイジェスト値を確認
    docker images --digests my-app
    ```

### 3. レッドチーム視点

* **タスク**: 侵入したホストにどのような技術スタックが存在するかを偵察する。
* **組み合わせ**: `docker images`
* **解説**: 攻撃者は `docker images` の `REPOSITORY` と `TAG` を見て、`postgres:12`, `wordpress:latest` のように稼働サービスとバージョンを特定し、既知の脆弱性を探します。
* **例**:

    ```bash
    # 偵察のため、ローカルイメージをリストアップ
    docker images
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **現象**: **REPOSITORY と TAG が `<none>:<none>` と表示されるイメージがある。**
    * **考えられる原因**: これは **dangling image** と呼ばれるものです。同じ名前とタグで新しいイメージをビルドした際に、古いイメージがどのタグからも参照されなくなり、この状態になります。
    * **解決策**: `docker image prune` コマンドで安全に削除できます。

## 環境変数と設定ファイル

* `docker images` の動作に直接影響する環境変数や設定ファイルは通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **信頼できないイメージの利用**。
* **悪用シナリオ**: 攻撃者は、公式イメージに似せた名前（タイポスクワッティング）の悪意のあるイメージをDocker Hubにアップロードします。開発者がそれを誤って `pull` して利用すると、マルウェアが実行される可能性があります。

### GTFOBins / LOLBAS における利用例

`docker images` は情報表示のみを行うコマンドであり、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: Docker Hubの `Official Image` や `Verified Publisher` のバッジが付いた、信頼できる発行元のイメージを常に使用する。
* **Detection (検知)**: `Trivy` などの脆弱性スキャンツールで、ローカルに存在するイメージ (`docker images` でリストアップ) に既知の脆弱性がないか定期的にスキャンする。

## 注意点・補足

* **`latest` タグの誤解**: `latest` タグは、必ずしもそのリポジトリの「最新バージョン」を意味するわけではありません。これは単なるデフォルトのタグです。再現性を確保するため、**常に具体的なバージョンタグを指定する**ことが強く推奨されます。
