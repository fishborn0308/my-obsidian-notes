# Windows CMD 一般的なエラーとトラブルシューティング

このドキュメントでは、Windowsのコマンドプロンプト (CMD) でよく遭遇する一般的なエラーメッセージとその原因、基本的なトラブルシューティング方法について解説します。

---

## 1. よくあるエラーメッセージと解決策

| エラーメッセージ | 考えられる原因 | 解決策 | セキュリティに関する考慮事項(ブルーチーム視点) |
| :--- | :--- | :--- | :--- |
| `'command' is not recognized as an internal or external command, operable program or batch file.` | 1. コマンドが存在しない、またはインストールされていない。<br>2. コマンドへのパスが環境変数 `Path` に含まれていない。<br>3. コマンド名のスペルミス。 | 1. コマンド名が正しいか、または実行したいプログラムがインストールされているか確認します。<br>2. `echo %Path%` で`Path`を確認し、必要であればコマンドのディレクトリを`Path`に追加します。システム環境変数を変更するには、システムのプロパティ(`sysdm.cpl` -> 詳細設定 -> 環境変数)から行います。<br>3. コマンド名を再確認します。 | 不審な`command not found`エラーが多数発生する場合、攻撃者が存在しないコマンドを実行しようとしている、または`Path`環境変数を操作して悪意のある実行ファイルをロードさせようとしている可能性があります。Windowsイベントログ(特にセキュリティログ、プロセス作成イベント)でこれらの試行を監視すべきです。AppLockerやWindows Defender Application Control (WDAC) で実行を制限することも有効です。 |
| `Access is denied.` (`アクセスが拒否されました。`) | 1. ファイルやディレクトリへのアクセス権限がない。<br>2. 管理者権限が必要な操作だが、CMDが管理者として実行されていない。<br>3. UAC (User Account Control) によるブロック。 | 1. 対象ファイル/ディレクトリのプロパティから「セキュリティ」タブで権限を確認し、必要であれば変更します。<br>2. CMDを右クリックし「管理者として実行」を選択して再試行します。<br>3. UACのプロンプトが表示された場合は許可します。 | 不正なユーザーによる権限昇格の試行や、保護されたファイルへのアクセス試行を示唆する可能性があります。Windowsセキュリティイベントログ(イベントID 4656: オブジェクトへのハンドルが要求された、4663: オブジェクトにアクセスされたなど)で、失敗したアクセス試行を監視し、異常なアクセスパターンを特定します。ファイル整合性監視ツール(例: Sysmon)も有効です。 |
| `The system cannot find the path specified.` (`指定されたパスが見つかりません。`) | 1. 指定されたファイルまたはディレクトリが存在しない。<br>2. ファイルまたはディレクトリへのパスが間違っている。 | 1. `dir` コマンドでファイル/ディレクトリの存在とパスを確認します。<br>2. 絶対パスまたは正しい相対パスを指定します。<br>3. ドライブレターが正しいか確認します。 | 攻撃者が存在しないファイルやディレクトリにアクセスしようとしている、または不正なファイルを配置しようとしている可能性があります。特に、システムファイルや重要なディレクトリへのアクセス試行は監視すべきです。Sysmonのファイル作成イベントやプロセス作成イベントで、不審なパスへのアクセスがないか確認します。 |
| `The process cannot access the file because it is being used by another process.` (`プロセスはファイルにアクセスできません。別のプロセスが使用中です。`) | 指定されたファイルが別のプロセスによってロックされている。 | Task Manager(タスクマネージャ)で不審なプロセスや関連プロセスを特定し、終了します。または、`handle`(Sysinternalsツール)や`resmon`(リソースモニター)でファイルロックを特定します。 | 攻撃者が特定のファイルをロックして、正規のプロセスが実行できないようにDoS攻撃を試みている可能性があります。または、マルウェアがシステムファイルや重要なファイルをロックして、削除や変更を防いでいる可能性も考慮し、`Process Monitor`(Sysinternalsツール)などで詳細なファイルアクセスアクティビティを調査します。 |
| `The parameter is incorrect.` (`パラメータが間違っています。`) | コマンドの引数やオプションの指定が不正である。 | コマンドのヘルプ (`command /?` または `help command`) を確認し、正しい引数やオプションを指定します。 | コマンドインジェクションの試行や、不正なコマンドオプションの利用を示唆する場合があります。特に、スクリプトや自動化ツールでこのエラーが頻繁に発生する場合は、入力値の検証が不十分である可能性があります。 |
| `Access denied - Error code 5` | アクセス拒否の一種で、特定のAPIコールが権限不足で失敗した際に発生することが多い。 | 管理者権限で再試行するか、対象オブジェクトのセキュリティ設定(ACL)を確認・変更します。 | アクセス権限のないユーザーやプロセスが特権リソースにアクセスしようとしていることを示します。セキュリティイベントログの`Failure Audits`(失敗した監査)イベントID 4656や4663などで詳細を確認します。 |
| `Ping request could not find host google.com. Please check the name and try again.` (`ping 要求はホスト google.com を見つけられませんでした。名前を確認してもう一度実行してください。`) | DNS解決に失敗している(名前解決ができない)。 | 1. ホスト名が正しいか確認します。<br>2. `ipconfig /all` でDNSサーバの設定を確認します。<br>3. `nslookup` や `dig`(Windowsでは`nslookup`)で名前解決を試します。<br>4. ファイアウォールやルータの設定を確認し、DNSリクエストがブロックされていないか確認します。 | DNS設定の改ざん(例: `hosts`ファイルの変更、不正なDNSサーバへのリダイレクト)を示唆する場合があります。これはマルウェアのC2通信やフィッシングサイトへの誘導に利用されるため、DNSクエリの異常なパターンや、`hosts`ファイルの不正な変更を監視すべきです。 |
