# `route` - IPルーティングテーブルを表示・操作する

## 概要

`route` コマンドは、WindowsのIPルーティングテーブルの内容を表示し、手動で静的ルートを追加、削除、変更するために使用されます。ルーティングテーブルは、IPパケットをどのネットワークインターフェイスから、どのネクストホップ（次のルーター）に送信するかを決定するための規則のリストです。

ネットワークのトラブルシューティングで通信経路を確認したり、複数のネットワークインターフェイスを持つホスト（マルチホーム）で特定の宛先への通信経路を明示的に制御したりする際に不可欠なツールです。

(出自: `Windows標準搭載`)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 (あれば) |
| :--- | :--- | :--- |
| **`netstat -r`** | `route print` と全く同じ機能で、ルーティングテーブルを表示します。 | どちらでも可 |
| **`netsh interface ip`** | `netsh` は、IPアドレス設定など、より広範なネットワーク構成の変更が可能です。`route` はルーティングテーブルの操作に特化しています。 | `netsh` (包括的な設定) |
| **PowerShell** | `Get-NetRoute`, `New-NetRoute`, `Remove-NetRoute` などのコマンドレットは、ルーティング情報をオブジェクトとして扱え、スクリプトでの利用がより強力で柔軟です。 | `*-NetRoute` (PowerShell) |

## よく連携されるコマンド

`route` は、`ipconfig` でIPアドレスやゲートウェイを確認し、`ping` で疎通を確認するという一連のネットワークトラブルシューティングフローの中で使われます。

### シナリオ例: 特定の宛先への通信が失敗する原因を調査する (インフラ構築・運用視点)

* **目的**: `10.1.1.100` への通信が失敗する場合に、ルーティングテーブルに適切な経路が存在するかを確認する。
* **連携コマンド**: `ipconfig`, `ping`
* **解説**: まず `ipconfig` で自身のIPアドレスとゲートウェイを確認します。次に `ping` で宛先への疎通を試み、失敗した場合に `route print` を実行します。出力結果を見て、宛先ネットワーク (`10.1.1.0/24` など) への経路が正しいゲートウェイに向けられているか、あるいはデフォルトゲートウェイ (`0.0.0.0`) が正しく設定されているかを確認します。
* **コマンド例**:

    ```cmd
    REM ルーティングテーブルを表示し、10.1.1.0 向けの経路を確認
    route print | findstr "10.1.1"
    ```

## スイッチ/オプション説明

| スイッチ/オプション | 説明 |
| :--- | :--- |
| **主なコマンド** | |
| ⭐ `PRINT` | ルーティングテーブルのエントリを表示します。 |
| ⭐ `ADD` | 静的ルートを追加します。 |
| ⭐ `DELETE` | 静的ルートを削除します。 |
| `CHANGE` | 既存の静的ルートを変更します（主にゲートウェイやメトリック）。 |
| **パラメータ** | |
| `<destination>` | `ADD` や `DELETE` で対象となる宛先のネットワークアドレスを指定します。 |
| `MASK <netmask>` | 宛先ネットワークのサブネットマスクを指定します。 |
| ⭐ `<gateway>` | パケットの転送先となるネクストホップのIPアドレスを指定します。 |
| `METRIC <metric>` | ルートの優先度を示すメトリック値を指定します。数値が小さいほど優先されます。 |
| `IF <interface>` | パケットを送信するインターフェイスの番号を指定します。`route print` の Interface List で確認できます。 |
| **永続性** | |
| ⭐ `-p` | (`ADD` と併用) **永続的 (persistent)** なルートとして追加します。このスイッチがない場合、追加したルートはシステムの再起動後に失われます。 |

## よく使われる組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 2つの異なるサブネットに接続されたサーバーで、特定のネットワーク (`10.20.0.0/16`) への通信を、デフォルトゲートウェイとは別のルーター (`192.168.1.254`) に向ける。
* **組み合わせ**: `ADD -p`
* **解説**: 永続的な静的ルート (`-p`) を追加することで、システムの再起動後もカスタムのルーティング設定が維持されるようにします。
* **例**:

    ```cmd
    REM 10.20.0.0/16 への通信を 192.168.1.254 経由にする永続ルートを追加
    route ADD 10.20.0.0 MASK 255.255.0.0 192.168.1.254 -p
    ```

### 2. ブルーチーム視点

* **タスク**: 侵害されたホストのルーティングテーブルを調査し、不審なルートが追加されていないか確認する。
* **組み合わせ**: `print`
* **解説**: 攻撃者は、通信を傍受したり、特定のトラフィックを自身が制御するホストにリダイレクトしたりするために、ルーティングテーブルを改ざんすることがあります。`route print` で現在のルーティングテーブルをダンプし、意図しないルート（特に永続ルート）がないかを監査します。
* **例**:

    ```cmd
    REM 現在のルーティングテーブルをファイルに保存して監査
    route print > C:\Forensics\routing_table.txt
    ```

### 3. レッドチーム視点

* **タスク**: **中間者攻撃 (Man-in-the-Middle)** の準備、または **C2通信の隠蔽**。
* **組み合わせ**: `add`
* **解説**: 攻撃者は権限昇格後、`route add` を使って特定のサーバー（例: ドメインコントローラー）への通信を、自身が制御するホストを経由するように変更します。これにより、認証情報などを盗聴する中間者攻撃の足がかりとします。また、特定のトラフィックを別の経路に向けることで、ネットワーク監視を回避しようとすることもあります。
* **例**:

    ```cmd
    REM ドメインコントローラー (10.1.1.10) への通信を、攻撃者のマシン (192.168.1.123) にリダイレクト
    route add 10.1.1.10 MASK 255.255.255.255 192.168.1.123
    ```

## エラーメッセージとトラブルシューティング

### よくあるエラーメッセージ

1. **エラーメッセージ**: `The route addition failed: The object already exists.`
    * **考えられる原因**: 追加しようとしたルートが既に存在します。
    * **解決策**: `route print` で既存のルートを確認してください。変更したい場合は `route change` を使うか、`route delete` で一度削除してから `route add` を実行します。

2. **エラーメッセージ**: `The requested operation requires elevation.`
    * **考えられる原因**: ルーティングテーブルの変更には管理者権限が必要です。
    * **解決策**: コマンドプロンプトを「管理者として実行」して、再度コマンドを実行してください。

## 環境変数と設定ファイル

* `route` は、その動作に影響を与える特定の環境変数や設定ファイルを使用しません。
* 一般的な環境変数は、[CMD共通の環境変数・設定ファイル](./environment_and_config.md)で解説。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `route` 自体ではなく、**ルーティングテーブルの不正な変更**がセキュリティリスクとなります。
* **悪用シナリオ**: レッドチームのシナリオで述べた通り、ルーティングテーブルの改ざんは、中間者攻撃、通信の傍受、ネットワーク監視の回避など、様々な攻撃の起点となり得ます。

### LOLBAS (Living Off The Land Binaries and Scripts) における利用例

* **機能**: **偵察 (Reconnaissance)**, **防御回避 (Defense Evasion)**
* **解説**: `route.exe` は、攻撃者がネットワークトポロジーを把握したり、通信経路を操作して検知を回避したりするために利用する、典型的な環境寄生型バイナリです。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 最小権限の原則を徹底し、一般ユーザーにルーティングテーブルの変更権限を与えない。
* **Detection (検知)**: 定期的に `route print` の結果をベースラインと比較し、意図しない変更がないか監査する。`route.exe` の実行、特に `add`, `delete`, `change` といった変更操作を、プロセス監査ログやEDRで監視する。

## 注意点・補足

* **永続ルートの保存場所**: `-p` スイッチで追加された永続ルートは、レジストリの `HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters\PersistentRoutes` に保存されます。
