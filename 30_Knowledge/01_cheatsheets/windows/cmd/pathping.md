# `pathping` - ネットワーク経路上のパケット損失と遅延を診断する

## 概要

`pathping` コマンドは、`tracert` と `ping` の機能を組み合わせた、より高度なネットワーク経路診断ツールです。まず `tracert` のように宛先までのネットワーク経路（経由するルーター）を特定し、その後、特定した各ルーターに対して一定時間（デフォルトで25秒間、100回）`ping` のようなパケットを送信します。最終的に、各ルーターでのパケット損失率と平均応答時間を計算して表示します。

これにより、「ネットワークが遅い」「時々接続が切れる」といった断続的な問題の原因が、経路上のどのルーターにあるのかを特定するのに非常に役立ちます。

(出自: `Windows標準搭載`)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 (あれば) |
| :--- | :--- | :--- |
| ⭐ **`tracert`** | `tracert` は経路を**特定**するだけです。`pathping` は経路を特定した後、各経路上で**統計情報（パケット損失率）**を収集します。 | `pathping` (詳細診断) |
| ⭐ **`ping`** | `ping` は**最終的な宛先**との疎通と遅延のみを測定します。`pathping` は**経路上の中間点**ごとの品質も測定します。 | - |
| **`mtr` (Linux)** | Linuxにおける `pathping` と非常によく似た機能を持つツールです。 | - |

## よく連携されるコマンド

`pathping` は単体で完結する診断ツールのため、他のコマンドと直接連携することは少ないですが、その結果はネットワーク管理者への報告や、ファイアウォール設定変更の判断材料として利用されます。

### シナリオ例: ネットワーク品質問題のレポート作成 (インフラ構築・運用視点)

* **目的**: 特定の宛先へのネットワーク品質に問題があることをISPなどに報告するため、客観的な証拠として `pathping` の結果をファイルに保存する。
* **連携コマンド**: リダイレクション (`>`)
* **解説**: `pathping` の実行には時間がかかるため、結果を直接ファイルにリダイレクトしておくと、後からじっくり分析できます。
* **コマンド例**:

    ```cmd
    REM example.com への経路情報を pathping_report.txt として保存
    pathping example.com > pathping_report.txt
    ```

## スイッチ/オプション説明

| スイッチ/オプション | 説明 |
| :--- | :--- |
| **動作制御** | |
| ⭐ `-n` | ホスト名をIPアドレスに解決しません。各ホップでの名前解決をスキップするため、実行が高速になります。 |
| `-h <hops>` | 経路上で検索するホップの最大数を指定します（デフォルト: 30）。 |
| ⭐ `-q <queries>` | 各ホップへの問い合わせ（ping）の回数を指定します（デフォルト: 100）。 |
| `-p <period>` | 各問い合わせ（ping）の間の待機時間をミリ秒単位で指定します（デフォルト: 250ms）。 |
| `-w <timeout>` | 各応答を待つタイムアウト時間をミリ秒単位で指定します（デフォルト: 3000ms）。 |

## よく使われる組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: リモートオフィスやクラウドサービスへの接続が不安定な場合に、どのネットワーク区間でパケットロスが発生しているかを特定する。
* **組み合わせ**: `-n`
* **解説**: `pathping -n <destination>` を実行し、出力結果の `Lost/Sent` 列を確認します。特定のホップで急にパケットロス率が上昇している場合、そのルーターまたはその直前の区間に問題があると推測できます。
* **例**:

    ```cmd
    REM 8.8.8.8 までの経路でパケットロスが発生している場所を特定する
    pathping -n 8.8.8.8
    ```

### 2. ブルーチーム視点

* **タスク**: 内部ネットワークから不審な外部IPへの通信経路と品質を調査する。
* **組み合わせ**: `pathping <suspicious_ip>`
* **解説**: ファイアウォールログなどで検知した不審なIPアドレスに対し `pathping` を実行します。経路情報から、そのIPがどの国のどのISPに属するかを大まかに把握したり、特定のホップで意図的にパケットが破棄されている（フィルタリングされている）様子を観測したりします。
* **例**: -

### 3. レッドチーム視点

* **タスク**: **偵察 (Reconnaissance)**。ターゲットネットワークの内部構造をマッピングする。
* **組み合わせ**: `-n`
* **解説**: 攻撃者は侵入したセグメントから、別の内部セグメントにあるサーバー (`10.10.20.5`など) に対して `pathping` を実行します。表示される経路上ルーターのIPアドレスから、内部ネットワークのトポロジー（構造）や、ACLでフィルタリングを行っているルーターの存在を推測します。
* **例**:

    ```cmd
    REM 内部の別セグメントにあるサーバーへの経路を偵察
    pathping -n -q 20 10.10.20.5
    ```

## エラーメッセージとトラブルシューティング

### よくあるエラーメッセージ

1. **現象**: 最初の数ホップ以降、全てのホップで `Request timed out.` と表示される。
    * **考えられる原因**: 経路上のファイアウォールが、`pathping` が使用するICMPパケットをブロックしています。
    * **解決策**: これは必ずしもネットワーク障害を意味しません。ファイアウォールがセキュリティ上の理由でICMPに応答しないように設定されていることは非常に一般的です。最終的な宛先からの応答があれば、接続性自体は問題ない可能性があります。

## 環境変数と設定ファイル

* `pathping` は、その動作に影響を与える特定の環境変数や設定ファイルを使用しません。
* 一般的な環境変数は、[CMD共通の環境変数・設定ファイル](./environment_and_config.md)で解説。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `pathping` 自体ではなく、その応答が**情報漏洩**に繋がります。
* **悪用シナリオ**: `pathping` の応答には、組織の内部ネットワークで使用されているルーターのIPアドレスやホスト名が含まれる場合があります。これにより、攻撃者は内部ネットワークの構造を詳細に把握し、ラテラルムーブメントの計画に利用します。

### LOLBAS (Living Off The Land Binaries and Scripts) における利用例

* **機能**: **偵察 (Reconnaissance)**
* **解説**: `pathping` は、攻撃者が侵入後の情報収集フェーズで、ネットワークトポロジーをマッピングするために利用する環境寄生型バイナリです。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 境界ルーターやファイアウォールで、外部からのICMP Time Exceededメッセージを要求するパケットをブロックまたはレート制限することで、内部ネットワークの偵察を困難にする。
* **Detection (検知)**: 内部ネットワークから、あるいは内部の特定セグメント間で予期せぬ `pathping` や `tracert` の通信が多発していないかをネットワーク監視ツール (IDS/IPS) で監視する。

## 注意点・補足

* **実行時間**: `pathping` は各ホップに対して多数のパケットを送信するため、結果が表示されるまでに数分かかることがあります。`-q` スイッチで問い合わせ回数を減らすと、より早く結果を得られます。
