# `auditpol` - Windows監査ポリシーを管理する

## 概要

`auditpol` コマンドは、Windowsの監査ポリシーをコマンドラインから表示・設定するためのツールです。このコマンドを使うことで、「どのユーザーが」「どのような操作を」「成功したか/失敗したか」といったセキュリティイベントをイベントログに記録するかどうかを、カテゴリおよびサブカテゴリ単位で詳細に制御できます。適切な監査ポリシーの設定は、インシデントの検知と調査における最も重要な基盤となります。

(出自: `Windows標準搭載`)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 (あれば) |
| :--- | :--- | :--- |
| **`secpol.msc`** | GUIベースのローカルセキュリティポリシーエディタ。`auditpol` は、その設定をCUIで、より詳細なサブカテゴリレベルで操作できます。 | `auditpol` (スクリプト/詳細設定) |
| **`wevtutil`** | イベントログそのものを照会・管理します。`auditpol` は**何をログに記録するかというポリシー**を設定し、`wevtutil` は**記録されたログ**を扱います。 | - |
| **PowerShell** | `Get-AuditPolicy` や `Set-AuditPolicy` といったコマンドレットで、より高度なスクリプティングが可能です。 | PowerShell (スクリプト) |

## よく連携されるコマンド

`auditpol` でポリシーを設定した後は、イベントビューアー (`eventvwr.msc`) や `wevtutil` を使って、実際にイベントが記録されているかを確認するのが一連のフローです。

### シナリオ例: プロセス作成の監査を有効化し、ログを確認する (ブルーチーム視点)

* **目的**: システム上でどのプロセスがいつ起動されたかを追跡できるように、詳細なプロセス作成の監査を有効にする。
* **連携コマンド**: `wevtutil`
* **解説**: まず `auditpol /set` で「プロセス作成」の監査を有効にします。その後、実際にイベントログ（セキュリティログ）にイベントID 4688 が記録されるかを `wevtutil` やイベントビューアーで確認します。
* **コマンド例**:

    ```cmd
    REM "プロセス作成" の成功と失敗の両方を監査するように設定
    auditpol /set /subcategory:"Process Creation" /success:enable /failure:enable

    REM セキュリティログからイベントID 4688 を照会して確認
    wevtutil qe Security /q:"*[System[(EventID=4688)]]" /c:5 /rd:true /f:text
    ```

## スイッチ/オプション説明

| スイッチ/オプション | 説明 |
| :--- | :--- |
| **ポリシーの表示** | |
| ⭐ `/get /category:*` | 全てのカテゴリの現在の監査ポリシー設定を表示します。 |
| `/get /subcategory:"<name>"` | 指定したサブカテゴリのポリシー設定を表示します。 |
| `/list /subcategory:*` | 利用可能な全てのサブカテゴリの一覧を表示します。 |
| **ポリシーの設定** | |
| ⭐ `/set /subcategory:"<name>"` | 指定したサブカテゴリのポリシーを設定します。 |
| ⭐ `/success:enable` or `disable` | 成功イベントの監査を有効または無効にします。 |
| ⭐ `/failure:enable` or `disable` | 失敗イベントの監査を有効または無効にします。 |
| **バックアップとリストア** | |
| `/backup /file:<filename>` | 現在の監査ポリシーをファイルにバックアップします。 |
| `/restore /file:<filename>` | ファイルから監査ポリシーを復元します。 |

## よく使われる組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: サーバーの現在の監査ポリシー設定を確認する。
* **組み合わせ**: `auditpol /get /category:*`
* **解説**: サーバーが組織のセキュリティベースラインに準拠した監査設定になっているかを確認します。
* **例**:

    ```cmd
    REM 現在の全監査ポリシー設定を表示
    auditpol /get /category:*
    ```

### 2. ブルーチーム視点

* **タスク**: **脅威ハンティングとインシデント対応**のために、ログ取得を強化する。
* **組み合わせ**: `/set /subcategory:"<Subcategory Name>" /success:enable /failure:enable`
* **解説**: 攻撃者の活動を追跡するために不可欠なログを取得するようにポリシーを強化します。特に**"Process Creation" (プロセス作成)**, **"Security Group Management" (セキュリティグループ管理)**, **"Logon" (ログオン)** の監査は極めて重要です。
* **例**:

    ```cmd
    REM コマンドライン引数を含めたプロセス作成の監査を有効化 (要GPO設定)
    auditpol /set /subcategory:"Process Creation" /success:enable

    REM ログオン失敗の監査を有効化
    auditpol /set /subcategory:"Logon" /failure:enable
    ```

### 3. レッドチーム視点

* **タスク**: **偵察 (Reconnaissance)** と**防御回避 (Defense Evasion)**。
* **組み合わせ**: `auditpol /get /category:*`
* **解説**: 攻撃者は侵入後、まず `auditpol /get` を実行して、どのような監査ポリシーが設定されているかを把握します。もしプロセス作成などの詳細な監査が有効であれば、自身の活動がログに記録されやすいと判断し、より隠密な手法を選択します。権限昇格後は、`auditpol /set` を使って監査を無効化し、自身の活動の痕跡を消そうと試みます。
* **例**:

    ```cmd
    REM 現在の監査レベルを偵察
    auditpol /get /category:"Detailed Tracking"

    REM (権限昇格後) プロセス作成の監査を無効化して痕跡を隠蔽
    auditpol /set /subcategory:"Process Creation" /success:disable
    ```

## エラーメッセージとトラブルシューティング

### よくあるエラーメッセージ

1. **エラーメッセージ例 1**: `The requested operation requires elevation.`
    * **考えられる原因**: `auditpol` の設定変更には管理者権限が必要です。
    * **解決策**: コマンドプロンプトを「管理者として実行」して、再度コマンドを実行してください。

## 環境変数と設定ファイル

* `auditpol` は、その動作に影響を与える特定の環境変数や設定ファイルを使用しません。
* 一般的な環境変数は、[CMD共通の環境変数・設定ファイル](./environment_and_config.md)で解説。
* 監査ポリシーは、ローカルセキュリティポリシーまたはグループポリシー(GPO)の一部としてOSに格納されます。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `auditpol` 自体ではなく、**監査ポリシーの不備**がセキュリティリスクとなります。
* **悪用シナリオ**: 監査ポリシーが不十分な場合、攻撃者はシステム内で活動してもその痕跡がイベントログに記録されず、インシデントの検知や調査が非常に困難になります。

### LOLBAS (Living Off The Land Binaries and Scripts) における利用例

* **機能**: **防御回避 (Defense Evasion)**
* **解説**: `auditpol` は、攻撃者が自身の活動を隠蔽するために監査設定を無効化する目的で悪用される、典型的な環境寄生型バイナリです。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: グループポリシー(GPO)を使って、監査ポリシー設定を組織全体で一元的に強制適用し、ローカルでの変更を防ぐ。
* **Detection (検知)**:
  * イベントログの集中管理 (SIEM) を行い、ログが停止していないかを監視する。
  * `auditpol` コマンドの実行 (プロセス作成イベントID 4688) や、監査ポリシーの変更イベント (イベントID 4719) を監視し、意図しないポリシー変更を検知する。

## 注意点・補足

* **GPOとの関係**: Active Directory環境では、監査ポリシーは通常、グループポリシーオブジェクト(GPO)によって集中管理されます。ローカルで `auditpol` を使って設定しても、GPOの適用タイミングで上書きされることがあります。
