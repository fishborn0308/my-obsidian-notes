# `tracert` - 宛先ホストへのネットワーク経路を特定する

## 概要

`tracert` (Trace Route) は、指定した宛先ホストまで、IPパケットがどのようなネットワーク経路（経由するルーターのリスト）を辿るのかを特定するためのネットワーク診断ツールです。`tracert` は、TTL (Time To Live) という値を徐々に増やしたICMPエコー要求パケットを送信し、経路上の中間ルーターから返されるICMP「時間超過」メッセージを利用して経路を特定します。

「`ping` は通るのにWebサイトが表示されない」「特定の宛先への通信だけが遅い」といった問題が発生した際に、どのルーターで遅延が発生しているか、あるいは通信がどこで途絶えているのかを切り分けるのに役立ちます。

(出自: `Windows標準搭載`)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 (あれば) |
| :--- | :--- | :--- |
| ⭐ **`ping`** | `ping` は**最終的な宛先**との疎通を確認するだけです。`tracert` はそこに至るまでの**途中経路**を表示します。 | - |
| ⭐ **`pathping`** | `tracert` の高機能版。経路を特定した後、各ルーターに対して一定時間パケットを送り続け、**パケット損失率**も測定します。断続的な問題の診断には `pathping` がより有効です。 | `pathping` (詳細診断) |
| **`mtr` (Linux)** | `ping` と `traceroute` の機能をリアルタイムで組み合わせたツール。`tracert` や `pathping` よりもインタラクティブに経路品質を監視できます。 | - |

## よく連携されるコマンド

`tracert` は、`ping` で疎通がないことを確認した後、問題の切り分けを行うために実行されるのが一般的な流れです。

### シナリオ例: ネットワーク接続問題の切り分け (インフラ構築・運用視点)

* **目的**: リモートサーバー (`example.com`) への接続ができない場合に、問題が社内ネットワークにあるのか、インターネット経路上にあるのかを特定する。
* **連携コマンド**: `ping`
* **解説**: まず `ping` で疎通がないことを確認します。次に `tracert` を実行し、出力が社内の最後のルーター（ゲートウェイ）で止まってしまうのか、あるいはその先のISPのルーターで止まってしまうのかを確認します。
* **コマンド例**:

    ```cmd
    REM 1. まず疎通確認
    ping example.com
    REM > Request timed out.

    REM 2. 経路を調査
    tracert example.com
    ```

## スイッチ/オプション説明

| スイッチ/オプション | 説明 |
| :--- | :--- |
| **動作制御** | |
| ⭐ `-d` | ホップごとのIPアドレスをホスト名に**名前解決しません (don't resolve)**。実行が大幅に高速になります。 |
| `-h <hops>` | 検索するホップの最大数を指定します（デフォルト: 30）。 |
| `-w <timeout>` | 各応答を待つ**タイムアウト時間 (wait)** をミリ秒単位で指定します。 |
| **IPバージョン** | |
| `-4` | IPv4の使用を強制します。 |
| `-6` | IPv6の使用を強制します。 |

## よく使われる組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: ネットワーク遅延の原因を調査する。
* **組み合わせ**: `-d`
* **解説**: `-d` を付けて実行し、各ホップの応答時間 (ms) を確認します。特定のホップで急に応答時間が跳ね上がっている場合、そのルーターまたはその前後のネットワーク区間で遅延が発生していると推測できます。
* **例**:

    ```cmd
    REM 8.8.8.8 までの経路と各ホップでの遅延を（名前解決なしで）確認
    tracert -d 8.8.8.8
    ```

### 2. ブルーチーム視点

* **タスク**: 内部ネットワークから不審な外部IPへの通信経路を調査する。
* **組み合わせ**: `tracert <suspicious_ip>`
* **解説**: ファイアウォールログなどで検知した不審なIPアドレスに対し `tracert` を実行します。経路情報から、その通信がどのISPや国を経由しているかを大まかに把握し、脅威インテリジェンス情報と照合する手がかりとします。
* **例**: -

### 3. レッドチーム視点

* **タスク**: **偵察 (Reconnaissance)**。ターゲットネットワークの内部構造をマッピングする。
* **組み合わせ**: `-d`
* **解説**: 攻撃者は侵入したセグメントから、別の内部セグメントにあるサーバー (`10.10.20.5`など) に対して `tracert` を実行します。表示される経路上ルーターのIPアドレスから、内部ネットワークのトポロジー（構造）や、ACLでフィルタリングを行っているルーターの存在を推測します。
* **例**:

    ```cmd
    REM 内部の別セグメントにあるサーバーへの経路を偵察
    tracert -d 10.10.20.5
    ```

## エラーメッセージとトラブルシューティング

### よくあるエラーメッセージ

* `'tracert'` is not recognized... は、[CMD共通のトラブルシューティング](./troubleshooting_common_errors.md)を参照。

1. **表示**: `* * * Request timed out.`
    * **考えられる原因**:
        * 経路上のルーターが、セキュリティ上の理由でICMPパケットに応答しないように設定されている。（非常によくある）
        * 実際にそのルーターまたはその先の経路でパケットロスが発生している。
    * **解決策**: 1つや2つのホップでタイムアウトが表示されても、その後のホップで応答が返ってくれば、経路自体は繋がっていると判断できます。最終的な宛先まで全てタイムアウトになる場合は、経路上の問題やファイアウォールによるブロックが考えられます。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `tracert` の応答が**情報漏洩**に繋がります。
* **悪用シナリオ**: `tracert` の応答には、組織の内部ネットワークで使用されているルーターのIPアドレスやホスト名が含まれる場合があります。これにより、攻撃者は内部ネットワークの構造を詳細に把握し、ラテラルムーブメントの計画に利用します。

### LOLBAS (Living Off The Land Binaries and Scripts) における利用例

* **機能**: **偵察 (Reconnaissance)**
* **解説**: `tracert.exe` は、攻撃者が侵入後の情報収集フェーズで、ネットワークトポロジーをマッピングするために利用する環境寄生型バイナリです。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 境界ルーターやファイアウォールで、外部からのICMP Time Exceededメッセージを要求するパケット（`tracert` が利用するもの）をブロックまたはレート制限することで、内部ネットワークの偵察を困難にする。
* **Detection (検知)**: 内部ネットワークから、あるいは内部の特定セグメント間で予期せぬ `tracert` の通信が多発していないかをネットワーク監視ツール (IDS/IPS) で監視する。

## 注意点・補足

* **非対称ルーティング**: 行きと帰りのネットワーク経路が異なる（非対称ルーティング）場合、`tracert` の結果だけでは問題の完全な特定が難しいことがあります。
