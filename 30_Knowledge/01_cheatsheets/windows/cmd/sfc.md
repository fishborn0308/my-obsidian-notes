# `sfc` - システムファイルチェッカー

## 概要

`sfc` (System File Checker) は、Windowsの重要なシステムファイル (`.dll`, `.exe` など) が破損したり、改ざんされたりしていないかをスキャンし、検証するためのコマンドラインユーティリティです。問題が検出された場合、`sfc` は `C:\Windows\System32\dllcache` にあるキャッシュされたバックアップから、ファイルを正しいバージョンに復元しようとします。システムの動作が不安定になったり、特定のアプリケーションが起動しなくなったりした場合の、基本的な修復ツールとして利用されます。

(出自: `Windows標準搭載`)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 (あれば) |
| :--- | :--- | :--- |
| ⭐ **`dism`** | `sfc` が修復に使用するソース（コンポーネントストア）自体が破損している場合、`sfc` は失敗します。`dism` は、そのコンポーネントストア自体を修復できます。`sfc` -> `dism` の順で使うのが一般的です。 | - |
| **`chkdsk`** | `chkdsk` はディスクの**ファイルシステム**のエラー（不良セクタなど）をチェック・修復します。`sfc` は**ファイル自体の中身**の整合性をチェックします。 | - |

## よく連携されるコマンド

`sfc` は、修復に失敗した場合に、より強力な修復ツールである `dism` と連携して使用されます。

### シナリオ例: 破損したシステムファイルの修復 (インフラ構築・運用視点)

* **目的**: システムの動作不良の原因がシステムファイルの破損にあるかを診断し、修復を試みる。
* **連携コマンド**: `dism`
* **解説**: まず `sfc /scannow` を実行します。もし「Windows リソース保護は、破損したファイルを検出しましたが、それらの一部を修正できませんでした」というエラーが出た場合、次に `dism` を使って修復ソースを健全化し再度 `sfc` を実行するというのが標準的な手順です。
* **コマンド例**:

    ```cmd
    REM 1. まずシステムファイルチェッカーを実行
    sfc /scannow

    REM 2. (sfcが失敗した場合) DISMでコンポーネントストアを修復
    dism /Online /Cleanup-Image /RestoreHealth

    REM 3. 再度sfcを実行
    sfc /scannow
    ```

## スイッチ/オプション説明

| スイッチ/オプション | 説明 |
| :--- | :--- |
| **主な操作** | |
| ⭐ `/scannow` | 保護されている全てのシステムファイルを即座にスキャンし、問題があれば修復します。 |
| `/verifyonly` | 全ての保護されたシステムファイルの整合性をスキャンしますが、**修復は行いません**。 |
| `/scanfile=<file>` | 指定したファイルの整合性をスキャンし、問題があれば修復します。 |
| `/verifyfile=<file>` | 指定したファイルの整合性をスキャンしますが、修復は行いません。 |
| **オフライン操作** | |
| `/offbootdir=<dir>` | オフライン修復の際に、ブートディレクトリを指定します。 |
| `/offwindir=<dir>` | オフライン修復の際に、Windowsディレクトリを指定します。 |

## よく使われる組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: Windows Update後やドライバのインストール後にシステムが不安定になった場合に、システムの整合性を確認する。
* **組み合わせ**: `/scannow`
* **解説**: システムファイルが破損していないかをチェックするための、最も一般的で基本的な使い方です。
* **例**:

    ```cmd
    REM システム全体の整合性チェックと修復を実行
    sfc /scannow
    ```

### 2. ブルーチーム視点

* **タスク**: **システムファイルの改ざん検知**。マルウェアによって重要なシステムファイルが変更されていないかを調査する。
* **組み合わせ**: `/verifyonly`
* **解説**: インシデント対応の初期段階で、まず `/verifyonly` を使って修復を行わずにスキャンします。これにより、どのファイルが改ざんされているかのリスト（`CBS.log` に出力される）を得ることができます。改ざんが確認されたファイルは、証拠として保全する必要があります。
* **例**:

    ```cmd
    REM システムファイルの整合性を検証（修復はしない）
    sfc /verifyonly
    ```

### 3. レッドチーム視点

* **タスク**: **偵察 (Reconnaissance)**。システムの保護機構を把握する。
* **組み合わせ**: `sfc`
* **解説**: 攻撃者は、自身が改ざんしようとしているシステムファイル（例: `utilman.exe`などのアクセシビリティ機能）が `sfc` によって保護されているかを意識します。`sfc` による自動修復を回避するために、保護メカニズムを無効化したり、`TrustedInstaller` 権限を奪取したりする手法を検討します。

## エラーメッセージとトラブルシューティング

### よくあるエラーメッセージ

1. **エラーメッセージ**: `Windows Resource Protection could not perform the requested operation.`
    * **考えられる原因**: Windowsがセーフモードで実行されていない場合、または保留中の修復操作がある場合に発生します。
    * **解決策**: PCをセーフモードで再起動し、再度 `sfc /scannow` を実行してください。

2. **エラーメッセージ**: `Windows Resource Protection found corrupt files but was unable to fix some of them.`
    * **考えられる原因**: `sfc` が修復に使用するコンポーネントストア自体が破損しています。
    * **解決策**: `dism /Online /Cleanup-Image /RestoreHealth` を実行してコンポーネントストアを修復してから、再度 `sfc /scannow` を実行してください。

## 環境変数と設定ファイル

* `sfc` のスキャン結果の詳細は、`C:\Windows\Logs\CBS\CBS.log` に記録されます。
* 一般的な環境変数は、[CMD共通の環境変数・設定ファイル](./environment_and_config.md)で解説。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `sfc` 自体ではなく、**`sfc` の保護を回避する手法**が悪用されます。
* **悪用シナリオ**: 攻撃者は、`takeown` や `icacls` を使ってシステムファイルの所有権とアクセス権を奪い、`sfc` による保護を無効化してからファイルを改ざんします。これは、アクセシビリティ機能 (`utilman.exe`, `sethc.exe`) をバックドアに置き換えるなどの古典的な攻撃で利用されます。

### LOLBAS (Living Off The Land Binaries and Scripts) における利用例

* `sfc.exe` 自体は攻撃ツールとして直接利用されることは少ないですが、その保護メカニズムを理解し、回避することが攻撃者にとって重要となります。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: ファイルシステムに適切なACLを設定し、`TrustedInstaller` 以外のユーザーが重要なシステムファイルを変更できないようにする。
* **Detection (検知)**: ファイル整合性監視 (FIM) ツールや `Sysmon` を使って、`C:\Windows\System32` 配下のファイルへの予期せぬ書き込みや所有権の変更を監視する。`sfc.exe` の実行ログ (`CBS.log`) を定期的に監査する。

## 注意点・補足

* **管理者権限**: `sfc` を実行するには、コマンドプロンプトを「管理者として実行」する必要があります。
