# `telnet` - Telnetプロトコルでリモートホストに接続する (非推奨/デバッグ用)

## 概要

`telnet` コマンドは、Telnetプロトコルを使用してリモートホストとの間で対話的なテキストベースの通信セッションを確立するためのクライアントです。

**⚠️ 重要**: Telnetは、ユーザー名、パスワード、セッション中の全ての通信を**暗号化せずに平文で送信します**。そのため、リモート管理目的でこれを使用することは**極めて危険**です。安全なリモート管理には `ssh` が標準です。

現代のWindows環境における `telnet` の主な用途は、リモート管理ではなく、「**特定のホストの特定のTCPポートが待ち受け状態（開いている）かどうか**」を素早く確認するための、単純なネットワーク接続テストツールとしてです。

(出自: `Windows標準搭載` - ただし、デフォルトでは無効化されていることが多い)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 (あれば) |
| :--- | :--- | :--- |
| **`ssh`** | 全ての通信を暗号化する、安全なリモートシェルプロトコル。Telnetの完全な代替です。 | `ssh` (安全なリモート管理) |
| **PowerShell** | `Test-NetConnection -Port <port>` は、`telnet` と同様のポート接続テストを、より詳細な情報と共に、よりモダンな方法で実行できます。 | `Test-NetConnection` |
| **`ping`** | `ping` はICMPを使い、ホストが**ネットワーク的に到達可能か**をテストします。`telnet` はTCPを使い、ホスト上の**特定のポート（サービス）が応答するか**をテストします。 | - |

## よく連携されるコマンド

`telnet` は、`ping` でホストへの疎通を確認した後、特定のサービス（ポート）が稼働しているかを確認する、という流れで使われます。

### シナリオ例: ファイアウォールのポート開放確認 (インフラ構築・運用視点)

* **目的**: Webサーバー (`192.168.1.100`) のファイアウォールで、HTTPSポート (443) が正しく開いているかを確認する。
* **連携コマンド**: (なし)
* **解説**: `telnet <IPアドレス> <ポート番号>` を実行します。画面が真っ暗になるか、何らかの文字が表示されれば接続成功（ポートは開いている）です。「接続中。..」のまま応答がない、またはエラーが出れば接続失敗（ポートが閉じているか、ファイアウォールでブロックされている）と判断できます。
* **コマンド例**:

    ```cmd
    REM 192.168.1.100 のポート 443 への接続を試みる
    telnet 192.168.1.100 443
    ```

## よく使われる組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: データベースサーバーへの接続に失敗する場合、DBサーバーのリスニングポート (例: SQL Serverの1433) が開いているかクライアントから確認する。
* **組み合わせ**: `telnet <server_ip> <port>`
* **解説**: アプリケーションサーバーからデータベースサーバーに対し `telnet` を実行し、接続できなければ、途中のネットワーク機器のファイアウォールや、サーバー自体のWindows Defender ファイアウォールが原因である可能性が高いと切り分けられます。
* **例**:

    ```cmd
    REM DBサーバー 'DB-SRV-01' のポート 1433 への接続テスト
    telnet DB-SRV-01 1433
    ```

### 2. ブルーチーム視点

* **タスク**: 調査対象のホストで、不審な待ち受けポートが実際に接続を受け付けるかを確認する。
* **組み合わせ**: `telnet <target_ip> <suspicious_port>`
* **解説**: `netstat` で発見した不審な待ち受けポートに対し、別のマシンから `telnet` で接続を試みます。接続が成功し、何らかのバナー情報（例: `SSH-2.0-OpenSSH...`）が表示されれば、そのポートで動作しているサービスの特定に繋がります。
* **例**: -

### 3. レッドチーム視点

* **タスク**: **偵察 (Reconnaissance)**。ポートスキャン後、開いているポートでどのようなサービスが稼働しているかを手動で確認する（バナーグラビング）。
* **組み合わせ**: `telnet <target_ip> <open_port>`
* **解説**: `nmap` のような高度なツールが使えない環境で、攻撃者は `telnet` を使って手動でポートに接続し、サーバーから返される初期応答（バナー）を取得します。バナーにはサービス名やバージョン情報が含まれていることが多く、脆弱性特定の手がかりとなります。
* **例**:

    ```cmd
    REM ポート 21 (FTP) に接続してバナー情報を取得
    telnet 10.10.1.5 21
    ```

## エラーメッセージとトラブルシューティング

### よくあるエラーメッセージ

* `'telnet'` is not recognized... は、[CMD共通のトラブルシューティング](./troubleshooting_common_errors.md)を参照。

1. **エラーメッセージ**: `Connecting To <host>...Could not open connection to the host, on port <port>: Connect failed`
    * **考えられる原因**:
        * 宛先ホストのそのポートでサービスが待ち受けていない。
        * 途中経路または宛先のファイアウォールが通信をブロックしている。
    * **解決策**: 宛先ホストで `netstat -an | findstr "LISTENING"` を実行し、ポートが待ち受け状態か確認します。ファイアウォールの設定を確認します。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **平文での認証情報とデータ転送**。これがTelnetの根本的かつ致命的な脆弱性です。
* **悪用シナリオ**: 攻撃者はネットワークを盗聴（スニッフィング）することで、Telnetセッションのユーザー名とパスワードを容易に窃取し、システムに不正アクセスできます。

### LOLBAS (Living Off The Land Binaries and Scripts) における利用例

* **機能**: **偵察 (Reconnaissance)**, **データ窃取 (Data Exfiltration)**
* **解説**: `telnet.exe` は、攻撃者がポートスキャンやバナーグラビングのために利用する環境寄生型バイナリです。また、限定的ながら、外部のサーバーに接続してデータを平文で送信する、簡易なデータ窃取ツールとしても悪用される可能性があります。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **Telnetサーバーの使用を全面的に禁止し、SSHに移行する**。不要なTelnetクライアントはアンインストールする。境界ファイアウォールでTelnet (TCP/23) のインバウンド/アウトバウンド通信をブロックする。
* **Detection (検知)**: ネットワーク侵入検知システム(IDS)でTelnet通信を監視し、アラートを発報する。`telnet.exe` の実行をプロセス監査ログ (イベントID 4688) で監視する。

## 注意点・補足

* **デフォルトでは無効**: 近年のWindowsクライアントOSでは、Telnetクライアントはデフォルトでインストールされていません。「コントロールパネル」->「プログラム」->「Windowsの機能の有効化または無効化」から、「Telnet クライアント」にチェックを入れることで有効化できます。
