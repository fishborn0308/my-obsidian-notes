# `wevtutil` - Windowsイベントログを管理する

## 概要

`wevtutil` (Windows Events Utility) は、Windowsのイベントログをコマンドラインから総合的に管理するためのツールです。GUIのイベントビューアー (`eventvwr.msc`) が提供する機能の多くをCUIで実行でき、ログの一覧表示、特定のイベントの照会（クエリ）、ログのエクスポート、アーカイブ、そしてクリアといった操作が可能です。インシデント調査の自動化や、サーバーのログ管理スクリプトにおいて中心的な役割を果たします。

(出自: `Windows標準搭載`)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 (あれば) |
| :--- | :--- | :--- |
| ⭐ **`eventvwr`** | `eventvwr` は **GUI** ベースのイベントビューアーです。`wevtutil` はその機能を **CUI** で提供し、スクリプトによる自動化を可能にします。 | `wevtutil` (スクリプト/自動化) |
| **PowerShell** | `Get-WinEvent` コマンドレットは、XPathクエリと同様のフィルタリングが可能で、結果をオブジェクトとして取得できるため、後続の処理や分析が非常に強力です。 | `Get-WinEvent` (PowerShell) |
| **`auditpol`** | **何をログに記録するか**という監査ポリシーを設定します。`wevtutil` は**記録されたログ**を照会・管理します。 | - |

## よく連携されるコマンド

`wevtutil` は、特定のイベントを詳細に調査するために、`findstr` や、その強力な `/q` (query) オプションと連携して使用されます。

### シナリオ例: 特定のイベントIDを検索する (ブルーチーム視点)

* **目的**: セキュリティログから、ログオン失敗に関連するイベント (イベントID 4625) を直近10件、新しい順に表示する。
* **連携コマンド**: (なし、XPathクエリを使用)
* **解説**: `/q` (query) スイッチとXPathクエリを使うことで、特定のログから特定の条件に一致するイベントを非常に効率的に抽出できます。これは `wevtutil` の最も強力な機能の1つです。
* **コマンド例**:

    ```cmd
    REM セキュリティログからイベントID 4625 を検索し、最新の10件を新しい順にテキスト形式で表示
    wevtutil qe Security /q:"*[System[(EventID=4625)]]" /c:10 /rd:true /f:text
    ```

## スイッチ/オプション説明

`wevtutil` は `wevtutil <command> [arguments]` の形式で使用します。

| コマンド (Operation) | 説明 |
| :--- | :--- |
| ⭐ **`el`** (enum-logs) | システム上の全てのログの名前を一覧表示します。 |
| **`gl <LogName>`** (get-log) | 指定したログの構成情報（最大サイズ、保存場所など）を表示します。 |
| ⭐ **`qe <LogName>`** (query-events) | 指定したログからイベントを照会します。 |
| ⭐ **`cl <LogName>`** (clear-log) | 指定したログの全てのイベントを**クリア（削除）**します。 |
| **`ep <LogName> <File>`** (export-log) | ログをファイルにエクスポートします。 |
| **`al <File>`** (archive-log) | ログを自己完結型の形式でアーカイブします。 |

| 主なパラメータ (`qe`で使用) | 説明 |
| :--- | :--- |
| ⭐ `/q:<Query>` | XPath形式でイベントをフィルタリングするクエリを指定します。 |
| ⭐ `/c:<Count>` | 表示するイベントの最大数を指定します。 |
| ⭐ `/rd:true` | イベントを新しいものから順に（逆順で）表示します。 |
| ⭐ `/f:<Format>` | 出力形式を指定します (`XML`, `Text`, `RenderedXml`)。 |

## よく使われる組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: システムログに記録されている直近のエラーイベントを確認する。
* **組み合わせ**: `qe /q:"*[System[Level=2]]" /c:5 /rd:true`
* **解説**: `Level=2` は「エラー」レベルを示します。`/c:5` で最新5件に絞り込み、`/rd:true` で新しい順に表示することで、最近発生したシステムエラーを素早く特定できます。
* **例**:

    ```cmd
    REM システムログから最新のエラーを5件表示
    wevtutil qe System /q:"*[System[Level=2]]" /c:5 /rd:true /f:text
    ```

### 2. ブルーチーム視点

* **タスク**: **インシデント調査**。特定の時間範囲内に作成された新しいプロセス (イベントID 4688) を調査する。
* **組み合わせ**: `qe /q:"..."` (XPathクエリ)
* **解説**: XPathクエリを駆使することで、イベントIDだけでなく、時間範囲、ユーザー、プロセス名など、複数の条件を組み合わせてログを詳細に分析できます。これはフォレンジック調査において極めて重要です。
* **例**:

    ```cmd
    REM セキュリティログから、特定の時間範囲のプロセス作成イベントを抽出
    wevtutil qe Security /q:"*[System[(EventID=4688) and TimeCreated[@SystemTime>='2025-10-05T00:00:00.000Z' and @SystemTime<='2025-10-05T01:00:00.000Z']]]"
    ```

### 3. レッドチーム視点

* **タスク**: **痕跡消去 (Anti-Forensics)**。自身の攻撃活動の証拠を隠蔽するためにイベントログをクリアする。
* **組み合わせ**: `cl`
* **解説**: 攻撃者は権限昇格後、ラテラルムーブメントやペイロード実行の痕跡が記録された「セキュリティ」ログや「システム」ログなどを `wevtutil cl` コマンドでクリアしようとします。これは攻撃の最終段階でよく見られる行動です。
* **例**:

    ```cmd
    REM セキュリティログとシステムログをクリア
    wevtutil cl Security
    wevtutil cl System
    ```

## エラーメッセージとトラブルシューティング

### よくあるエラーメッセージ

1. **エラーメッセージ**: `Failed to clear log <LogName>. Access is denied.`
    * **考えられる原因**: イベントログのクリアや設定変更には管理者権限が必要です。
    * **解決策**: コマンドプロンプトを「管理者として実行」して、再度コマンドを実行してください。

## 環境変数と設定ファイル

* イベントログファイルは通常 `C:\Windows\System32\winevt\Logs\` 配下に `.evtx` 形式で保存されます。
* 一般的な環境変数は、[CMD共通の環境変数・設定ファイル](./environment_and_config.md)で解説。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `wevtutil` の**ログクリア機能**が、攻撃者によって悪用されます。
* **悪用シナリオ**: レッドチームのシナリオで述べた通り、`wevtutil cl` は、攻撃者が自身の侵入や活動の痕跡を消去するための常套手段です。

### LOLBAS (Living Off The Land Binaries and Scripts) における利用例

* **機能**: **偵察 (Reconnaissance)**, **防御回避 (Defense Evasion)**
* **解説**: `wevtutil.exe` は、攻撃者がシステムのログを調査して情報を収集したり、ログをクリアして自身の痕跡を消したりするために利用する、典型的な環境寄生型バイナリです。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **ログの集中管理（ログフォワーディング）**。イベントログをリアルタイムでSIEM (Security Information and Event Management) などの外部ログサーバーに転送することで、ローカルホスト上でログがクリアされても証拠を保全できます。
* **Detection (検知)**: **イベントログがクリアされたことを示すイベントID 1102 (The audit log was cleared) を最優先で監視し、即座にアラートを発報するように設定する**。これは攻撃の強力な兆候です。

## 注意点・補足

* **XPathクエリ**: `/q` で使用するXPathの構文は非常に強力ですが、複雑です。イベントビューアーのGUIで「現在のログをフィルター」を使い、目的のフィルターを作成した後、「XML」タブで対応するXPathクエリを確認・コピーするのが簡単で確実です。
