# `Clear-Content` - ファイルの内容を消去する

## 概要

`Clear-Content` コマンドレットは、アイテム（主にファイル）の内容を削除しますが、アイテム自体は削除しません。空のファイルを作成する、あるいは既存のログファイルや一時ファイルをリセットする目的で使用されます。

## 類似コマンドレット/コマンドとの差異

| 類似コマンドレット/コマンド | 差異 / 使い分けのポイント |
| :--- | :--- |
| **`Set-Content <path> ""`** | `Set-Content` に空文字列を渡すことでもファイルの内容をクリアできます。`Clear-Content` はその目的専用のコマンドレットであり、意図が明確になります。 |
| **`Remove-Item`** | `Remove-Item` はファイルやディレクトリ**自体を削除**します。`Clear-Content` は**内容のみを削除**し、ファイルは空の状態で残ります。 |

## よく連携されるコマンドレット

### シナリオ例: スクリプト実行前にログファイルを初期化する (インフラ構築・運用視点)

* **目的**: 定期実行されるスクリプトが、毎回新しいログファイルを作成するように、実行の最初に既存のログファイルの内容をクリアする。
* **連携コマンドレット**: `Add-Content`
* **解説**: スクリプトの冒頭で `Clear-Content` を実行して前回のログを消去し、その後 `Add-Content` で新しいログを記録します。
* **コマンド例**:

    ```powershell
    # ログファイルのパスを変数に定義
    $logFile = "C:\Scripts\Logs\maintenance.log"

    # 既存のログファイルの内容をクリア
    Clear-Content -Path $logFile

    # 新しいログの記録を開始
    Add-Content -Path $logFile -Value "$(Get-Date): Maintenance script started."
    ```

## パラメータ説明

| パラメータ | 型 | 説明 |
| :--- | :--- | :--- |
| ⭐ **`-Path`** | `String[]` | 内容をクリアするアイテム（ファイル）のパスを指定します。ワイルドカードが使用できます。 |
| **`-Filter`** | `String` | `-Path` パラメータの対象をフィルタリングするための文字列を指定します。 |
| **`-Force`** | `SwitchParameter` | 読み取り専用や隠しファイルなど、通常は変更できないファイルの内容も強制的にクリアします。 |
| **`-Stream`** | `String` | (NTFSのみ) ファイルのデフォルトストリームではなく、指定した代替データストリームの内容をクリアします。 |

## よく使われる組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 一時ディレクトリ内の全ての `.tmp` ファイルの内容を空にする。
* **組み合わせ**: `Clear-Content -Path <path_with_wildcard>`
* **解説**: ワイルドカード (`*`) を使って、複数のファイルの内容を一度にクリアします。
* **例**:

    ```powershell
    # C:\Temp 内の全ての .tmp ファイルの内容をクリア
    Clear-Content -Path C:\Temp\*.tmp
    ```

### 3. レッドチーム視点

* **タスク**: **痕跡消去 (Anti-Forensics)**。自身の活動の証拠となるログファイルの内容を消去する。
* **組み合わせ**: `Clear-Content -Path <log_file>`
* **解説**: 攻撃者は権限昇格後、`wevtutil cl` と同様の目的で `Clear-Content` を使用して、特定のアプリケーションログやスクリプトの実行履歴、あるいはセキュリティツールのログファイルの内容を消去し、調査を妨害しようとします。ファイル自体を `Remove-Item` で削除するよりも、空ファイルとして残す方が目立ちにくい場合があります。
* **例**:

    ```powershell
    # IISのアクセスログの内容をクリア
    Clear-Content -Path C:\inetpub\logs\LogFiles\W3SVC1\u_ex*.log -Force
    ```

## エラーメッセージとトラブルシューティング

### よくあるエラーメッセージ

1. **エラーメッセージ**: `Clear-Content: Access to the path '...' is denied.`
    * **考えられる原因**: ファイルに対する書き込み権限がないか、ファイルが別のプロセスによってロックされています。
    * **解決策**: PowerShellを「管理者として実行」します。対象が読み取り専用ファイルの場合は `-Force` スイッチを使用します。ファイルがロックされている場合は、ロックしているプロセスを特定して終了させる必要があります。

## セキュリティに関する考慮事項

### 悪用事例

* **痕跡消去**: レッドチームのシナリオで述べた通り、`Clear-Content` は攻撃者が自身の活動の痕跡（ログファイル、スクリプト履歴など）を消去するための常套手段です。

### LOLBASにおける利用例

* **機能**: **防御回避 (Defense Evasion)**
* **解説**: `Clear-Content` は PowerShell の標準機能であり、`powershell.exe` を通じて実行されます。攻撃者はこの正規の機能を利用して、検知されにくい形でログや証拠ファイルを破壊します。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **ログの集中管理（ログフォワーディング）**。イベントログや重要なアプリケーションログをリアルタイムでSIEMなどの外部ログサーバーに転送することで、ローカルホスト上でログがクリアされても証拠を保全できます。
* **Detection (検知)**: PowerShellのスクリプトブロックロギング (イベントID 4104) を有効化し、`Clear-Content` の実行、特に重要なログファイルに対する実行を厳重に監視する。ファイル整合性監視 (FIM) を導入し、ログファイルの予期せぬ変更（サイズが0になるなど）を検知する。

## 注意点・補足

* `Clear-Content` は強力なコマンドです。重要なファイルに対して誤って実行しないよう、`-WhatIf` パラメータを使って事前に影響範囲を確認するのが安全です。
