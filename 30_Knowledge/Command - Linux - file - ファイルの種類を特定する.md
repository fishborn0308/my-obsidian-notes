# Command - Linux - file - ファイルの種類を特定する

## 概要
`file` は、指定されたファイルの内容を検査し、その種類（テキスト、実行ファイル、画像、圧縮データ等）を特定して表示します。拡張子が間違っていたり、拡張子自体がなかったりする場合でも、ファイルヘッダー内の「マジックナンバー」をデータベースと照合することで、正確な判別を試みます。
(出自: `file` パッケージに含まれる。Ian Darwin 等によって開発)



## 類似・関連コマンド
| コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `stat` | ファイルのサイズ、権限、タイムスタンプなどの「属性」を表示する。中身は見ない。 | 属性情報の確認 |
| `mimetype` | ファイルの MIME タイプ（`image/jpeg` 等）に特化して出力する。 | Webアプリ等のプログラム処理用 |
| `strings` | バイナリファイル内の表示可能な文字列を抽出する。 | バイナリ内のヒント探し |
| `file` | ファイルが「そもそも何者か」を判断する。 | **初期調査・識別** |

## よく連携されるコマンド
### シナリオ例: 拡張子のない大量のファイルの一括仕分け (運用・保全視点)
* **目的**: 復元ソフトで救出した「f001, f002...」のような名前のファイルが、それぞれ何であるか特定する。
* **連携コマンド**: `find`, `awk`
* **解説**: `file` の結果からファイル形式を抜き出し、拡張子を付け直すヒントにします。
* **コマンド例**:
    ```bash
    # カレントディレクトリの全ファイルの種類を一覧表示
    file *
    ```

## オプション説明
| オプション | 説明 |
| :--- | :--- |
| ⭐️ `-b`, `--brief` | ファイル名を表示せず、種類の結果のみを簡潔に表示する。 |
| ⭐️ `-i`, `--mime` | MIME タイプ形式（例: `text/plain; charset=utf-8`）で表示する。 |
| ⭐️ `-L`, `--dereference` | シンボリックリンクの場合、リンク先自体の種類を表示する。 |
| `-z`, `--uncompress` | 圧縮ファイルの中身までチェックして種類を特定しようとする。 |
| `-f <file>`, `--files-from` | 調査対象のファイルリストが書かれたファイルから読み込む。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点
* **タスク**: 破損したと思われるアーカイブファイルの確認。
* **解説**: 拡張子が `.tar.gz` なのに展開できない場合、`file` で見ると実は単なるテキスト（エラーメッセージ）だった、ということがよくあります。
* **例**:
    ```bash
    # 圧縮ファイルが本当に有効か確認
    file backup.tar.gz
    # 結果例: gzip compressed data, from Unix, original size... (正常)
    # 結果例: HTML document, ASCII text (エラーページのHTMLを落としていた)
    ```

### 2. ブルーチーム視点
* **タスク**: 偽装された実行ファイルの検知。
* **解説**: 攻撃者はしばしば、実行ファイル（ELF）の拡張子を `.jpg` や `.txt` に変えて隠蔽しようとします。`file` はこの偽装を瞬時に見破ります。
* **例**:
    ```bash
    # 画像に見えるファイルが本当に画像か確認
    file suspicious.jpg
    # 結果例: ELF 64-bit LSB executable... (偽装発覚)
    ```

### 3. レッドチーム視点
* **タスク**: ターゲット環境のバイナリ特性の特定（フィンガープリンティング）。
* **解説**: 収集したバイナリが 32-bit か 64-bit か、静的にリンクされているか、デバッグ情報が残っているか（stripped かどうか）を確認し、適切なエクスプロイトを準備します。
* **例**:
    ```bash
    # ターゲットのシステムバイナリを調査
    file /usr/bin/python3
    ```

## エラーメッセージとトラブルシューティング

1.  **「cannot open (No such file or directory)」**:
    * **原因**: 指定したパスが間違っているか、ファイルが存在しない。
    * **解決策**: 正しいパスを `ls` などで確認する。

2.  **「data」としか表示されない**:
    * **原因**: マジックデータベースに一致するパターンがなく、バイナリデータであることしか判別できなかった。
    * **解決策**: `strings` や `hexdump` で中身を直接覗き、手動で解析する。

## セキュリティに関する考慮事項

### 脆弱性とリスク
* **マジックデータベースの脆弱性**: `file` コマンド自体がマジックファイルを解析する際のバグ（バッファオーバーフロー等）が過去に報告されています。悪意を持って細工されたファイルを `file` で読み込ませることで、解析者側の端末を攻撃する「カウンター攻撃」の可能性があります。

### GTFOBins における利用例
`file` が `sudo` で実行可能な場合、特権を利用してマジックデータベースの指定（`-m`）を悪用したり、任意のファイルを読み取ったりできる可能性があります。
* **機能**: `File Read`, `Sudo`
* **コマンド例**:
    ```bash
    # sudo file を使って /etc/shadow の一部を読み取る（エラー出力を利用）
    sudo file -m /etc/shadow /dev/null
    ```
* **参照**: `https://gtfobins.github.io/gtfobins/file/`

### 対応策・緩和策 (ブルーチーム視点)
* **Prevention (予防)**: `file` コマンドを常に最新の状態に保ち、既知の解析エンジン脆弱性を修正しておく。
* **Detection (検知)**: `/etc/shadow` などの重要ファイルに対して `file` コマンドが実行されているログを `auditd` で監視する。

## 注意点・補足
* **過信は禁物**: 熟練した攻撃者は、ファイルの先頭数バイトだけを本物の画像ファイルのように偽装（ポリグロットファイル）し、`file` コマンドを欺くことがあります。
* **マジックファイル**: `/usr/share/misc/magic` に判別ルールの実体があります。
