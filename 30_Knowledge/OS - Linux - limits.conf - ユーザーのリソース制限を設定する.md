# OS - Linux - limits.conf - ユーザーのリソース制限を設定する

## 概要

`/etc/security/limits.conf` は、**PAM (Pluggable Authentication Modules)** を通じてログインするユーザーやグループに対して、システムリソースの使用制限（**ulimit**）を永続的に設定するためのファイルです。このファイルで設定できるリソースには、開くことができるファイルディスクリプタの最大数、作成できるプロセスの最大数、CPU時間の上限などがあります。

主に、**リソース枯渇によるシステムの不安定化を防ぐ**目的や、**Fork爆弾のようなサービス妨害攻撃を緩和する**ために利用されます。

(出自: `pam` (Pluggable Authentication Modules) の一部として提供される)

## 類似コマンド/ファイルとの差異

| コマンド/ファイル | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`ulimit` (コマンド)** | **現在のシェルセッション**に対して、一時的にリソース制限を設定・表示します。`limits.conf` は**永続的な**設定を行います。 | - |
| **`systemd` ユニットファイル**| `systemd` で管理される**サービス単位**でリソース制限を設定します (`LimitNOFILE=`など)。特定のデーモンにはこちらで設定するのがモダンな方法です。`limits.conf` は**ユーザー/グループ単位**で影響します。 | `systemd` (サービス単位) |
| **`cgroups`** | プロセスグループに対して、CPU、メモリ、ディスクI/Oなどのリソースをより詳細に、かつ動的に制限する強力なカーネル機能です。コンテナ技術の基盤です。 | `cgroups` (高度な制御) |

## `limits.conf` の書式と主要項目

各行は `<domain> <type> <item> <value>` の4つのフィールドで構成されます。

| フィールド | 説明 | 例 |
| :--- | :--- | :--- |
| **`<domain>`** | 制限を適用する対象。ユーザー名、`@groupname` (グループ)、`*` (全ユーザー) などを指定。 | `john`, `@developers`, `*` |
| **`<type>`** | 制限の種類。`soft` (ユーザーが変更可能な警告レベルの上限) または `hard` (rootしか変更できない絶対的な上限) を指定。 | `soft`, `hard` |
| **`<item>`** | 制限するリソースの種類。 | `nofile`, `nproc`, `cpu` |
| **`<value>`** | 制限値。数値、または `unlimited` を指定。 | `65536`, `1024`, `unlimited` |

### 主要なリソース項目 (`<item>`)

| 項目 | 説明 |
| :--- | :--- |
| ⭐ **`nofile`** | **(Number of Files)** オープンできるファイルディスクリプタの最大数。**最もよくチューニングされる項目**。 |
| ⭐ **`nproc`** | **(Number of Processes)** ユーザーが作成できるプロセスの最大数。 |
| `cpu` | ユーザーが使用できるCPU時間の最大値（分）。 |
| `stack` | プロセススタックの最大サイズ（KB）。 |
| `memlock` | メモリ内でロックできる最大サイズ（KB）。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 高負荷なWebサーバーで、「Too many open files」エラーが発生するのを防ぐ。
* **組み合わせ**: `nofile` の `soft` と `hard` リミットを設定。
* **解説**: WebサーバーやDBは、クライアント接続ごとにファイルディスクリプタを消費します。デフォルトの上限値 (多くの場合1024) ではすぐに枯渇するため、この設定で上限を引き上げ、システムの安定稼働を確保します。
* **例 (`/etc/security/limits.conf` に追記)**:

```bash
# Nginxユーザーのオープンファイル数上限を増やす
nginx soft nofile 65536
nginx hard nofile 65536
```

### 2. ブルーチーム視点

* **タスク**: Fork爆弾のような、無限にプロセスを生成するサービス妨害攻撃の影響を緩和する。
* **組み合わせ**: `*` (全ユーザー) に対して `nproc` の `hard` リミットを設定。
* **解説**: この設定により、どの一般ユーザーも合計で指定した数までしかプロセスを作成できなくなります。攻撃者がFork爆弾を実行しても、プロセス数がこの上限に達した時点でそれ以上プロセスを生成できなくなり、システム全体のダウンを防ぐことができます。
* **例 (`/etc/security/limits.conf` に追記)**:

```bash
# 全ての一般ユーザー (root以外) のプロセス数を1024に制限
* hard nproc 1024
```

### 3. レッドチーム視点

* **タスク**: 侵入したシステムの制限設定を調査し、攻撃活動の制約となるかを確認する。
* **組み合わせ**: `ulimit -a`, `cat /etc/security/limits.conf`
* **解説**: 攻撃者は、`nproc` の制限が厳しいと多数のツールを同時に実行できない、`nofile` の制限が厳しいとスキャンツールなどが正常に動作しない、といった制約を受けます。これらの設定を確認することで、自身の活動が検知・妨害される可能性を評価します。
* **例**:

```bash
# 現在のセッションの全リソース制限値を確認
ulimit -a
```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **現象**: **`limits.conf` を編集したのに、設定が反映されない。**
		* **考えられる原因**:
				* 変更後に**再ログイン**していない。`limits.conf` の設定は、PAMを通じて新しいログインセッションが開始される際に読み込まれます。
				* `sshd` の設定で `UsePAM` が `no` になっている。
				* そもそもそのプロセスがPAMセッションを通じて起動されていない（例: OS起動時に `systemd` が直接起動するデーモン）。
		* **解決策**: まずは一度ログアウトし、再ログインして `ulimit -n` などで設定が反映されたか確認してください。デーモンプロセスに制限をかけたい場合は、`systemd` のユニットファイルで `LimitNOFILE=` などを設定するのが正しい方法です。

## 環境変数と設定ファイル

* `limits.conf` の動作に直接影響する環境変数は通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

### 関連する設定ファイル

* **`/etc/security/limits.conf`**: メインの設定ファイル。
* **`/etc/security/limits.d/`**: パッケージ固有の設定などが、個別の `.conf` ファイルとして配置されるディレクトリ。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `limits.conf` 自体ではなく、**不適切なリソース制限の設定**がリスクとなります。
* **悪用シナリオ**: 緩すぎる制限は、単一のユーザーや暴走したプロセスがシステム全体のリソースを使い果たし、DoS状態を引き起こすリスクを高めます。

### GTFOBins / LOLBAS における利用例

`limits.conf` は設定ファイルであり、直接的なコード実行機能を持たないため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 最小権限の原則に従い、ユーザーやサービスごとに適切なリソース制限を設定する。
* **Detection (検知)**: `/etc/security/limits.conf` や `/etc/security/limits.d/` 配下のファイルの変更をファイル整合性監視(FIM)ツールで監視する。

## 注意点・補足

* **`su` と `su -` の挙動**: `su` コマンドはデフォルトではログインセッションを開始しないため、`limits.conf` の設定が引き継がれないことがあります。`su -` のように `-` を付けてログインシェルを起動すると、設定が正しく読み込まれます。
