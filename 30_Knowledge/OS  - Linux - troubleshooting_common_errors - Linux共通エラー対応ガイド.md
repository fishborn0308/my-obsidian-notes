# OS  - Linux - troubleshooting_common_errors - Linux共通エラー対応ガイド

## 概要

このドキュメントは、Linuxシステムを運用する上で、多くのコマンドで共通して発生するエラーメッセージと、その原因を特定し解決するための基本的な診断ワークフローをまとめたものです。個別のコマンドチートシートから、このページにリンクすることで、情報の重複を避け、一貫したトラブルシューティング手順を提供します。

## トラブルシューティングの基本原則

問題に直面したら、闇雲にコマンドを試すのではなく、体系的なアプローチを取ることが重要です。

1. **エラーメッセージを読む**: 表示されたエラーメッセージを注意深く読み、キーワードを把握します。
2. **ログを確認する**: ほぼ全ての答えはログにあります。`journalctl`, `/var/log/` 配下を確認します。
3. **最近の変更を疑う**: 問題が発生する直前に、どのような設定変更やデプロイが行われたかを確認します。
4. **問題を切り分ける**: 何が**できて**、何が**できない**のかを明確にします。「サーバーAには繋がるが、Bには繋がらない」など。
5. **低レイヤーから確認する**: 物理層 → ネットワーク層 → OS層 → アプリケーション層の順で、問題の範囲を絞り込んでいきます。

## 🛠️ 共通エラーと診断ワークフロー

### 1. シェル/コマンドに関するエラー

#### `command not found` (コマンドが見つからない)

* **意味**: 実行しようとしたコマンドの実行ファイルが、`$PATH` 環境変数で指定されたディレクトリの中に見つかりません。
* **診断ワークフロー**:
		1. **タイプミス確認**: まず、コマンド名を打ち間違えていないか確認します。
		2. **`PATH`の確認**: `echo $PATH` を実行し、コマンドが格納されているはずのディレクトリ（例: `/usr/local/bin`）が含まれているか確認します。
		3. **コマンドの場所を特定**:
				* `which <command>`: `PATH` 内からコマンドを探します。
				* `whereis <command>`: バイナリ、ソース、manページの場所を表示します。
				* `locate <command>`: データベースを検索して、システム全体のどこにあるかを探します。
		4. **インストール状況の確認**:
				* `apt search <pkg>` / `yum search <pkg>`: コマンドを提供しているパッケージがインストール可能か確認します。

### 2. ファイル/パーミッションに関するエラー

#### `Permission denied` (権限がありません)

* **意味**: ファイルやディレクトリにアクセスしようとしましたが、実行ユーザーに必要な権限がありません。
* **診断ワークフロー**:
		1. **誰として実行しているか確認**: `whoami` と `id` コマンドで、現在のユーザー名と所属グループを確認します。
		2. **対象の権限を確認**: `ls -l <file>` または `ls -ld <directory>` を実行し、対象の所有者、グループ、パーミッションモード（`rwx`）を確認します。
		3. **パス全体の権限を確認**: `namei -om /path/to/target/file` を実行すると、目的のファイルに至るまでの各ディレクトリの権限が一度に表示され、途中のディレクトリでアクセスが拒否されている場合などに非常に便利です。
		4. **SELinux/AppArmorの確認**: パーミッションが正しいように見えてもアクセスできない場合、SELinuxやAppArmorによって拒否されている可能性があります。
				* `sestatus` (SELinux) / `aa-status` (AppArmor) で状態を確認。
				* `ausearch -m avc -ts recent` (SELinux) や `journalctl | grep "apparmor="` で拒否ログを確認。

#### `No such file or directory` (ファイルやディレクトリがありません)

* **意味**: 指定したパスにファイルまたはディレクトリが存在しません。
* **診断ワークフロー**:
		1. **タイプミス確認**: パスやファイル名にタイプミスがないか、大文字小文字が正しいかを確認します。
		2. **現在地の確認**: `pwd` を実行して、自分が意図したカレントディレクトリにいるかを確認します。
		3. **存在確認**: `ls -ld /path/to/your/target` のように、絶対パスで存在を確認します。Tabキーによる補完を積極的に利用すると、タイプミスを防げます。

### 3. ネットワークに関するエラー

#### `Connection refused` (接続が拒否されました)

* **意味**: ネットワーク的にホストには到達できましたが、指定したポートでリッスンしているサービス（プロセス）が存在しません。
* **診断ワークフロー**:
		1. **サーバー側で確認**:
				* `sudo ss -tulpn | grep <port>` を実行し、目的のサービスが正しいIP (`0.0.0.0` or `127.0.0.1`など) とポートで `LISTEN` しているか確認します。
				* `systemctl status <service>` でサービスがアクティブ (`running`) であることを確認します。
		2. **クライアント側で確認**:
				* 接続先のIPアドレスとポート番号が正しいか再確認します。

#### `Connection timed out` (接続がタイムアウトしました)

* **意味**: 宛先ホストまたは途中のネットワーク機器から、指定した時間内に応答がありませんでした。
* **診断ワークフロー**:
		1. **ファイアウォールの確認**: クライアント側、サーバー側、そしてその間のネットワーク機器（クラウドのセキュリティグループなど）のファイアウォールが、そのポートへの通信をブロックしていないか確認します。
		2. **L3疎通確認**: `ping <ip_address>` を実行し、ICMPパケットが相手に届くか確認します。（ただし、`ping`がブロックされていることも多い）
		3. **経路の確認**: `traceroute <ip_address>` または `mtr <ip_address>` で、宛先までのネットワーク経路のどこでパケットがドロップしているかを確認します。
