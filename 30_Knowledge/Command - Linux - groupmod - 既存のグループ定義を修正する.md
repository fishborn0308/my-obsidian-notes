# Command - Linux - groupmod - 既存のグループ定義を修正する

## 概要
`groupmod` は、`/etc/group` ファイルを編集し、既に存在するグループの属性を変更します。主な用途はグループ名の変更と、GID の数値変更です。手動でファイルを編集するよりも安全に整合性を保ちながら変更を行えます。
(出自: `shadow-utils` パッケージに含まれる)

## 類似・関連コマンド
| コマンド | 役割 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `groupadd` | 新しいグループを作成する。 | グループ作成 |
| `groupdel` | グループを削除する。 | 不要グループの削除 |
| `usermod` | ユーザーの属性（所属グループ等）を変更する。 | ユーザー側の修正 |
| `groupmod` | グループ自体の名前やIDを書き換える。 | **グループ属性の修正** |

## よく連携されるコマンド
### シナリオ例: チーム名変更に伴うディレクトリ権限の追従 (運用視点)
* **目的**: グループ名を `old_dev` から `new_dev` に変更し、古い名前で設定されていたディレクトリの表示も更新する。
* **解説**: 名前を変えるだけであれば、ファイルが保持している「数値（GID）」は変わらないため、権限自体は維持されます。
* **コマンド例**:
    ```bash
    # 1. グループ名を変更 (old_dev -> new_dev)
    sudo groupmod -n new_dev old_dev
    # 2. 確認 (ls -l の表示が new_dev に変わっているはず)
    ls -l /var/www/html
    ```

## オプション説明
※実行には **root 権限** が必須です。

| オプション | 説明 |
| :--- | :--- |
| ⭐️ `-g <GID>`, `--gid` | グループID（GID）を新しい数値に変更する。 |
| ⭐️ `-n <NEW_NAME>`, `--new-name` | グループ名を新しい名前に変更する。 |
| `-o`, `--non-unique` | 他のグループと重複する GID の設定を許可する（通常は非推奨）。 |
| `-R <prefix>`, `--root` | chroot 環境など、別のルートディレクトリ配下のファイルを修正する。 |

## ⚠️ GID 変更時の重大な注意点
`-g` オプションで **GID を変更しても、既存のファイルやディレクトリの所有 GID は自動的に書き換わりません。**
* **現象**: `groupmod -g 2000 mygroup` とした場合、以前の GID（例: 1000）で作成されたファイルは、所有グループが「1000」という数値のまま残り、`mygroup` のメンバーであってもアクセスできなくなる可能性があります。
* **対策**: 変更後に `find` コマンドで古い ID のファイルを探し、`chgrp` で一括修正する必要があります。

## よく使われる業務シナリオ

### 1. インフラ構築・運用視点
* **タスク**: サーバー移行に伴う GID の正規化。
* **解説**: 複数のサーバーで共有ストレージ（NFSなど）を使っている場合、同じグループ名でも GID が異なると権限トラブルが起きます。これを統一するために `groupmod -g` を使用します。
* **例**:
    ```bash
    # shared_grp の GID を全社共通の 5000 に統一
    sudo groupmod -g 5000 shared_grp
    # その後、古い GID (例: 1001) のファイルをすべて修正
    sudo find / -gid 1001 -exec chgrp shared_grp {} +
    ```

### 2. ブルーチーム視点
* **タスク**: 権限の「隠蔽」の調査。
* **解説**: 攻撃者が既存の無害なグループ名を、管理者が警戒しにくい名前に変えたり、GID を重複（`-o`）させて特権グループと同じ権限を持たせたりしていないか確認します。
* **例**:
    ```bash
    # 重複している GID がないか確認
    cut -d: -f3 /etc/group | sort | uniq -d
    ```

### 3. レッドチーム視点
* **タスク**: 既存グループへの「なりすまし」や権限のすり替え。
* **解説**: 名前の変更を利用して、自分が所属している一般グループを、あたかもシステム標準の重要なグループであるかのように見せかけ、管理者の目を欺きます。

## エラーメッセージとトラブルシューティング

1.  **「groupmod: group 'xxx' does not exist」**:
    * **原因**: 修正しようとしたグループが存在しない。
    * **解決策**: `getent group` で正確なグループ名を確認する。

2.  **「groupmod: GID '1001' already exists」**:
    * **原因**: 変更先の GID が既に使用されている。
    * **解決策**: 未使用の ID を選ぶか、どうしても必要な場合は `-o` を使う。

## セキュリティに関する考慮事項

### 脆弱性とリスク
* **権限の混乱**: GID 変更後のファイル修正漏れは、情報の可用性を損なうだけでなく、後にその古い ID が別の（意図しない）グループに割り当てられた際、情報の漏洩（Information Disclosure）に繋がるリスクがあります。

### 対応策・緩和策 (ブルーチーム視点)
* **監査**: `/etc/group` の変更を `auditd` で監視し、予期せぬ `groupmod` の実行を記録します。
* **一貫性チェック**: 定期的に `find / -nogroup` を実行し、どのグループにも属していない「幽霊ファイル」を撲滅します。

## 注意点・補足
* **ログイン中のユーザー**: グループ名や ID を変更しても、現在ログイン中のユーザーのセッションには即座に反映されない場合があります（一度ログアウトして再ログインが必要）。
