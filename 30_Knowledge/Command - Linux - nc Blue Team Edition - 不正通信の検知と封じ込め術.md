# Command - Linux - nc [Blue Team Edition] - 不正通信の検知と封じ込め術

## 概要

ブルーチームにおける `nc` 対策の核心は、「誰が」「いつ」「どんな引数で」実行したかを可視化することにあります。特にサーバー環境において、対話型ではない `nc` の挙動は、それ自体がインシデントの強力な兆候です。

---

## 1. リアルタイム監視と検知 (Detection)

### 🚨 不審なプロセス引数の監視 (EDR/SIEM)

攻撃者が多用する特定のオプションをキーワードとしてアラートを設定します。

* **検知フラグ**:
    * `-e` または `--exec`: シェル実行の明示的な試み。
    * `-lp`: 待ち受け（バックドア）の作成。
    * `mkfifo` との組み合わせ: パイプを利用した回避型リバースシェル。
* **監視ツール (Auditd) の設定例**:
    ```bash
    # nc の実行を記録するルール
    -a always,exit -F arch=b64 -S execve -F exe=/usr/bin/nc -k susp_netcat
    ```

### 📡 ネットワークレベルの異常検知

標準外のポートを使用した長時間の通信を監視します。

* **チェックポイント**:
    * 宛先が外部 IP で、ポートが 4444, 9999, 1337 などの「攻撃者定番ポート」ではないか。
    * 逆に、443 (HTTPS) や 53 (DNS) ポートを使っているのに、中身が SSL や DNS プロトコルではない通信。

---

## 2. フォレンジックと調査 (Forensics)

### 🔍 実行中の不正通信の特定

「今まさに動いている」 `nc` を特定し、どこに繋がっているかを確認します。

* **コマンド**:
    ```bash
    # ネットワーク接続を持っている nc プロセスを特定
    sudo lsof -i | grep nc
    # または、プロセスツリーを確認して親プロセス（シェル等）を特定
    ps -ef --forest | grep -A 3 nc
    ```

### 🧬 メモリ上の痕跡

攻撃者がバイナリを削除したとしても、メモリ上には `nc` の実行コードや通信バッファが残ります。

* **調査ポイント**: `/proc/<PID>/fd/` を確認し、標準入力・出力がどこにリダイレクトされているか（名前付きパイプ等）を突き止めます。

---

## 3. システムの要塞化 (Hardening)

### ✂️ 不要なバイナリの削除・制限

そもそも `nc` が不要なサーバーからは削除するのが最善です。

* **対策**:
    * 伝統的な `nc`（`-e` オプション付き）は特に危険なため、インストールを禁止する。
    * 一般ユーザーの `execution` 権限を剥奪する（`chmod 700 /usr/bin/nc`）。

### 🛡️ AppArmor / SELinux による封じ込め

もし `nc` が必要であっても、その挙動（ネットワークアクセス先や実行可能なコマンド）をプロファイルで制限します。

* **ポリシー例**: 「`nc` は特定の内部監視サーバー以外への通信を許可しない」というホワイトリスト運用。

---

## 4. インシデントレスポンス (Response)

### ⚡ 即時遮断アクション

不正な `nc` 接続を発見した際の初動です。

1.  **通信の遮断**: `iptables` やクラウドの SG で当該 IP をブロック。
2.  **プロセスの強制終了**: `kill -9 <PID>`。
3.  **隔離**: 該当ホストをネットワークから切り離し、メモリダンプを取得。

---

## 💡 ブルーチームの防衛コンボ

| 対策レベル | 具体的なアクション | 防衛効果 |
| :--- | :--- | :--- |
| **予防** | `alias nc='echo "NC is restricted"'` | 簡易的な嫌がらせ・誤操作防止。 |
| **検知** | `grep -E "nc.*-e|nc.*mkfifo" /var/log/auth.log` | 攻撃の試みをログから事後確認。 |
| **防御** | `egress` フィルタリングの設定 | 許可されたポート以外の外向き通信を全遮断し、リバースシェルを無効化。 |

> [!IMPORTANT]
> 攻撃者は `nc` の代わりに `bash -i >& /dev/tcp/...` や `python`, `perl` を使って同様の通信路を確保することもあります。 `nc` だけを追うのではなく、**「外向きの不自然なコネクション」**という本質を監視することが重要です。
