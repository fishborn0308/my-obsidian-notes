# OS  - Linux - `SUID` と `SGID` - ファイル実行時の権限昇格

# `SUID` と `SGID` - ファイル実行時の権限昇格

## 概要

**SUID (Set User ID)** と **SGID (Set Group ID)** は、Linuxのファイルに設定される特殊なパーミッションビットです。これらのビットが設定された実行可能ファイルは、実行したユーザーの権限ではなく、**そのファイルの所有者またはグループの権限で**プロセスが動作します。

* **SUID**: ファイルの**所有者**の権限で実行される。
* **SGID**: ファイルの**グループ**の権限で実行される。

例えば、一般ユーザーが自身のパスワードを変更するには、rootしか書き込めない `/etc/shadow` ファイルを編集する必要があります。これを可能にするため、`/usr/bin/passwd` コマンドにはSUIDビットが設定されており、一般ユーザーが実行しても一時的に `root` 権限で動作するようになっています。

(出自: Linux/Unixのファイルシステム機能)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`sudo`** | **ユーザー**に基づいて、一時的にコマンド単位で権限を昇格させます。SUID/SGIDは**ファイル**に永続的な特権を付与します。 | `sudo` |
| **ケーパビリティ**| SUID/SGIDの**よりモダンで安全な代替**。`root` の全権限を与えるのではなく、「特権ポートにバインドする権限」のように、必要な権限だけを細かく分割してプロセスに与えます。 | Capabilities |

## よく連携されるコマンド

SUID/SGIDビットが設定されたファイルを探すためには、`find` コマンドが不可欠です。

### シナリオ例: 権限昇格に悪用可能なファイルをシステム全体から検索する (レッドチーム視点)

* **目的**: システムに侵入後、SUIDビットが設定されているファイルを探し出し、権限昇格の足がかりとする。
* **連携コマンド**: `find`
* **解説**: `find` の `-perm /4000` はSUIDビットが立っているファイルを検索するための定型句です。攻撃者はこのリストの中から、外部コマンドを実行できるようなプログラム（`find`, `vim`など）を探します。
* **コマンド例**:

```bash
# このコマンドで、システム上の全SUIDファイルを探し出す
find / -perm /4000 -type f 2>/dev/null
```

## オプション説明 (設定と確認方法)

### 設定方法 (`chmod`)

| モード | SUIDの例 | SGIDの例 |
| :--- | :--- | :--- |
| ⭐ **シンボリック**| `chmod u+s <file>` | `chmod g+s <file>` |
| ⭐ **八進数 (Octal)**| `chmod 4755 <file>` | `chmod 2755 <file>` |

### 確認方法 (`ls -l`)

`ls -l` を実行した際、実行権限 (`x`) の部分が `s` または `S` に変わります。

* `s` (小文字): 実行権限 (`x`) **も**設定されている。（正常）
* `S` (大文字): 実行権限 (`x`) **が**設定されていない。（通常は設定ミス）

```bash

\-rwsr-xr-x 1 root root 68280 May 15 2023 /usr/bin/passwd  \<-- SUIDが設定されている

```

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 複数人で利用する共有ディレクトリのグループ所有権を統一する。
* **組み合わせ**: `chmod g+s <directory>`
* **解説**: ディレクトリにSGIDを設定すると、そのディレクトリ内で作成された新しいファイルやディレクトリは、**親ディレクトリのグループを自動的に継承**します。これにより、複数人での共同作業が容易になります。
* **例**:

```bash
# 共有ディレクトリにSGIDを設定
sudo chmod g+s /srv/shared-project
```

### 2. ブルーチーム視点

* **タスク**: システムを監査し、不審なSUID/SGIDファイルがないか確認する。
* **組み合わせ**: `find / -perm /6000 -type f -exec ls -l {} \;`
* **解説**: SUIDまたはSGIDが設定されているファイルをシステム全体から探し出し、詳細を表示します。標準的なコマンド以外（特に `/tmp` や `/home` 配下など）にSUID/SGIDファイルが発見された場合、それは攻撃者によって設置された**権限昇格用のバックドア**である可能性が極めて高いです。
* **例**:

```bash
# システム上の全SUID/SGIDファイルをリストアップして監査する
find / \( -perm /4000 -o -perm /2000 \) -type f -ls 2>/dev/null
```

### 3. レッドチーム視点

* **タスク**: 権限昇格の足がかりを探す。
* **組み合わせ**: `find / -perm /4000 2>/dev/null`
* **解説**: 攻撃者はシステムに一般ユーザーとして侵入した後、まずこのコマンドでSUIDが設定された実行ファイルを探します。もし、`nmap`, `find`, `vim` といった、外部コマンドを実行できる機能を持つプログラムにSUIDビットが設定されていた場合（重大な設定ミス）、それを悪用して簡単にroot権限を奪取できます。
* **例**:

```bash
# 権限昇格に利用できるSUIDファイルを探す
find / -user root -perm /4000 -type f 2>/dev/null
```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。
* SUID/SGIDは `chmod` コマンドで設定するため、`chmod` のエラーメッセージ（`Operation not permitted`など）に準じます。

## 環境変数と設定ファイル

* SUID/SGIDの動作に直接影響する環境変数や設定ファイルは通常ありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: **最大の権限昇格リスク**。`root` が所有するSUIDファイルは、Linuxにおける最も古典的で危険な権限昇格の温床です。
* **悪用シナリオ**: これらのプログラムにバッファオーバーフローやコマンドインジェクションなどの脆弱性が1つでもあれば、即座にシステム全体の侵害に繋がります。

### GTFOBins / LOLBAS における利用例

* **機能**: **Privilege Escalation**
* **解説**: GTFOBinsの多くのエントリには、「SUID」というカテゴリがあり、権限昇格の手法が記載されています。もし `find` コマンドにSUIDビットが設定されていた場合、`-exec` オプションを使って任意のコマンドをroot権限で実行できます。
* **参照**: `https://gtfobins.github.io/gtfobins/find/#suid`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: `/tmp` や `/home` のようなユーザーが書き込めるパーティションは `nosuid` オプション付きでマウントする。不要なSUIDビットは `chmod u-s <file>` で削除する。
* **Detection (検知)**: `find / -perm /6000` でSUID/SGIDが設定されたファイルを定期的に監査し、ベースラインと比較して意図しない変更がないか監視する。

## 注意点・補足

* **スクリプトへの設定**: セキュリティ上の理由から、ほとんどのモダンなLinuxシステムでは、`bash` や `python` などの**シェルスクリプトにSUID/SGIDビットを設定しても、その権限は無視されます**。
