# Command - Linux - awk - テキストのパターン走査と抽出・加工

## 概要

`awk` は、テキストファイルを1行ずつ読み込み、指定したルール（パターン）に従ってデータの抽出や加工を行うテキスト処理言語です。特に、空白やタブで区切られた「列（フィールド）」を扱うのが非常に得意です。

(出自: `gawk` (GNU awk) パッケージに含まれることが多い。名前は開発者 Aho, Weinberger, Kernighan の頭文字)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `grep` | 特定の文字列を含む「行」を探すだけなら最適。列の入れ替えなどは不可。 | 単純な行検索 |
| `sed` | 文字列の置換（S/old/new/g）や行の削除に強い。数値計算などは苦手。 | ストリーム編集・置換 |
| `cut` | 固定の区切り文字（カンマ等）で列を切り出す。空白の数が変動する場合に弱い。 | 単純な列切り出し |
| `awk` | 列の入れ替え、条件分岐、数値計算、正規表現を組み合わせた高度な処理が可能。 | 複雑な加工・集計 |

## よく連携されるコマンド

### シナリオ例: ログからユニークなIPアドレスと出現回数をカウント

* **目的**: アクセスログから攻撃元のIPアドレスを特定し、頻度を確認する。
* **連携コマンド**: `sort`, `uniq`
* **解説**: `awk` で特定の列を抜き出し、`sort` してから `uniq -c` でカウントします。
* **コマンド例**:
    ```bash
    # アクセスログの第1列（IP）を抜き出してカウント
    cat /var/log/apache2/access.log | awk '{print $1}' | sort | uniq -c | sort -nr
    ```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **基本動作** | |
| ⭐️ `-F <char>` | フィールドの区切り文字（デリミタ）を指定する。デフォルトは空白またはタブ。 |
| ⭐️ `-v <var>=<val>` | プログラム内で使用する変数を外部から渡す。 |
| `-f <script_file>` | 実行する awk スクリプトをファイルから読み込む。 |
| **組み込み変数** | |
| `$0` | 行全体。 |
| `$1, $2, ...` | 第1フィールド、第2フィールド... |
| `NF` | その行のフィールド数（Number of Fields）。 |
| `NR` | 現在の行番号（Number of Records）。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: ディスク使用率が 80% を超えているパーティションの抽出。
* **組み合わせ**: `df` との連携
* **解説**: `df` の出力を受け取り、5列目（使用率）の数字部分を評価します。
* **例**:
    ```bash
    # 使用率が80以上の行を表示
    df -h | awk '0+$5 >= 80 {print $0}'
    ```

### 2. ブルーチーム視点

* **タスク**: `/var/log/auth.log` からパスワード試行に失敗したユーザー名とIPを抽出。
* **組み合わせ**: 正規表現マッチング
* **解説**: 失敗ログの特定のパターンにマッチさせ、後ろから数えて特定の列を表示します。
* **例**:
    ```bash
    # Failed password を含む行から ユーザー名($9) と IP($11) を表示
    sudo grep "Failed password" /var/log/auth.log | awk '{print "User: " $9, "IP: " $11}'
    ```

### 3. レッドチーム視点

* **タスク**: `/etc/passwd` からシェルが `/bin/bash` のユーザー名のみをリストアップ。
* **組み合わせ**: `-F:` (区切り文字指定)
* **解説**: コロン区切りのファイルをパースし、最終列が `/bin/bash` であるか判定します。
* **例**:
    ```bash
    # シェルがbashのユーザーを抽出
    awk -F: '$7 == "/bin/bash" {print $1}' /etc/passwd
    ```

## エラーメッセージとトラブルシューティング

1.  **エラーメッセージ例 1**: `awk: line 1: syntax error at or near {`
    * **考えられる原因**: クォートの閉じ忘れや、アクション `{}` の記述ミス。
    * **解決策**: ワンライナーの場合、全体をシングルクォート `' '` で囲んでいるか確認する。

2.  **エラーメッセージ例 2**: `awk: division by zero`
    * **考えられる原因**: プログラム内で 0 による除算が発生した。
    * **解決策**: `if ($2 != 0)` のように、分母が 0 でないことを確認する条件を追加する。

## 環境変数と設定ファイル

* `awk` は通常、設定ファイルを持ちませんが、環境変数 `LANG` や `LC_ALL` が文字列比較やソート順に影響を与えることがあります。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: ユーザー入力をそのまま `awk` のスクリプトとして評価させると、任意のコマンド実行（RCE）につながる可能性があります（OSコマンドインジェクション）。
* **悪用シナリオ**: 攻撃者が root 権限で動く `awk` スクリプトの一部を改ざんし、`system()` 関数を呼び出してリバースシェルを起動させる。

### GTFOBins における利用例

`awk` は `system()` 関数を内蔵しているため、`sudo` 権限がある場合、簡単に root シェルを奪取できます。

* **機能**: `Shell`, `Sudo`, `File Read`
* **コマンド例**:
    ```bash
    # sudo awk を使って root シェルを取得
    sudo awk 'BEGIN {system("/bin/sh")}'
    ```
* **参照**: `https://gtfobins.github.io/gtfobins/awk/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 不特定のユーザーが `awk` を `sudo` で実行できないよう、`/etc/sudoers` を厳格に管理する。
* **Detection (検知)**: ログ内に `awk` の `system` 関数や `print` を使った `/etc/shadow` へのアクセスがないか監視する。

## 注意点・補足

* **シングルクォート vs ダブルクォート**: シェル上での実行時は、`$1` などがシェル変数として展開されないよう、必ず全体をシングルクォートで囲むのが鉄則です。
* **BEGIN と END**: `BEGIN { ... }` は最初の行を読み込む前、`END { ... }` は最後の行を読み終わった後に一度だけ実行されます。合計値の出力などに便利です。
