# Command - Linux - iostat - CPU 統計とデバイスの I/O 統計を表示する

## 概要

`iostat` は、システムのストレージデバイス（HDD, SSD, NVMe等）の入出力統計と、CPU の使用状況をレポートします。デバイスごとの読み書き速度（KB/s）、リクエストの待ち行列、平均応答時間などをリアルタイムで監視でき、パフォーマンスのボトルネック特定に威力を発揮します。

(出自: `sysstat` パッケージに含まれる)

## 類似・関連コマンド

| コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `vmstat` | メモリ、スワップ、プロセスの統計を総合的に表示する。 | システム全体の俯瞰 |
| `sar` | 過去の統計データを蓄積し、遡って確認できる（最強の履歴管理）。 | 過去のトラブル調査 |
| `iotop` | 「どのプロセス」が I/O を食っているかをプロセス単位で表示。 | **犯人（プロセス）の特定** |
| `iostat` | 「どのデバイス（ディスク）」が限界かを物理単位で表示。 | **ハードウェアの限界調査** |

## よく連携されるコマンド

### シナリオ例: 5秒おきに3回、ディスクの状態を監視する

* **目的**: 一時的なスパイクではなく、継続的な負荷の状態を確認したい。
* **解説**: 第1引数に間隔（秒）、第2引数に回数を指定するのが `iostat` の基本作法です。
* **コマンド例**:
    ```bash
    # 5秒おきに3回、詳細な統計を表示
    iostat -x 5 3
    ```

## 主要なオプション

| オプション | 説明 |
| :--- | :--- |
| ⭐️ `-x`, `--extended` | **必須。** 詳細な統計情報（await, %util など）を表示する。 |
| ⭐️ `-c` | CPU の統計情報のみを表示する。 |
| ⭐️ `-d` | ディスクデバイスの統計情報のみを表示する。 |
| ⭐️ `-k` / `-m` | 単位をキロバイト(KB) または メガバイト(MB) で表示する。 |
| `-t` | レポートの各時刻を表示する。 |
| `-z` | アクティビティがゼロ（アイドル状態）のデバイスを非表示にする。 |

## 出力項目の読み解き（ここが肝心！）

`-x` オプションを付けた際の主要な項目を理解しましょう。

* **`r/s` / `w/s`**: 1秒あたりの読み込み/書き込みリクエスト数。
* **`rkB/s` / `wkB/s`**: 1秒あたりの読み込み/書き込み量（KB）。
* **`await`**: ⭐️ **重要**。I/O リクエストが発行されてから完了するまでの平均時間（ミリ秒）。数値が高い（例: 10.0以上）と、ディスクが追いついていません。
* **`avgqu-sz`**: デバイスに送られたリクエストの平均待ち行列の長さ。
* **`%util`**: ⭐️ **最重要**。CPU が I/O リクエストを処理していた時間の割合。100% に近いほど、そのディスクは飽和状態（フル稼働）です。

## 業務シナリオ別活用法

### 1. インフラ構築・運用視点

* **タスク**: データベースのパフォーマンス劣化調査。
* **解説**: DB の応答が遅い際、`iostat -x` で `%util` を確認します。もし 90% を超えているなら、インデックスの最適化や、より高速な SSD/NVMe への交換が必要です。

### 2. ブルーチーム視点

* **タスク**: ランサムウェアの挙動監視。
* **解説**: 大量のファイルを短時間で暗号化するランサムウェアは、異常な `w/s`（書き込み数）と `%util` を発生させます。平常時のベースラインを知っておくことで、この異常値を検知のフックにできます。

### 3. レッドチーム視点

* **タスク**: ステルス（隠密）活動のタイミング調整。
* **解説**: ターゲット環境で重いコマンドを実行する際、あえてシステムが「忙しい（I/O 負荷が高い）」時間を `iostat` で把握し、自分の活動による負荷の増加を目立たなくさせる「紛れ込み」を検討します。

## セキュリティに関する考慮事項

### 脆弱性とリスク

* **情報の露出**: `iostat` は一般ユーザーでも実行可能です。どのデバイスが頻繁に使われているかを知ることで、攻撃者は「データベースがどのパーティションにあるか」などの内部構成を推測するヒントを得られます。

### 対応策・緩和策 (ブルーチーム視点)

* **リソース監視**: `iostat` 自体を制限するよりも、`Prometheus` や `Zabbix` などの監視ツールで I/O メトリクスを常時収集し、異常な書き込み（データ持ち出しの予兆）を自動検知する体制を構築します。

## 注意点・補足

* **最初の1回目の出力**: `iostat` を実行した直後の最初のレポートは、**「システム起動時から現在までの平均値」** です。現在のリアルタイムな状況を知るには、必ず `iostat 1 2` のように2回以上出力させて、2回目以降の数値を見る必要があります。
