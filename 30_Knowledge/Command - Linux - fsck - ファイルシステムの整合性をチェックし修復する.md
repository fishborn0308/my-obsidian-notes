# Command - Linux - fsck - ファイルシステムの整合性をチェックし修復する

## 概要
`fsck` は、ディスク上のファイルシステムの論理的なエラー（iノードの不整合、ブロックの重複割り当て等）を検査し、可能であれば修復します。実際には、対象のファイルシステム形式（ext4, xfs, vfat 等）に応じた専用ツール（`e2fsck`, `fsck.vfat` 等）を呼び出すフロントエンドとして機能します。



## 類似・関連コマンド
| コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `e2fsck` | ext2/ext3/ext4 形式に特化した修復ツール。 | ext系ファイルシステムの詳細修復 |
| `xfs_repair` | XFS 形式の修復ツール。`fsck` 経由ではなく直接使うことが多い。 | XFSの修復 |
| `badblocks` | デバイス上の物理的な不良セクタを検索する。 | 物理故障の疑いがある場合 |
| `fsck` | ファイルシステムの種類を自動判別して適切なツールを起動する。 | **汎用的なチェック・修復** |

## ⚠️ 最重要ルール
**「マウントされている（使用中の）ファイルシステムに対して fsck を実行してはいけない」**
OSが書き込みを行っている最中に `fsck` が構造を書き換えると、致命的なデータ破壊を引き起こします。修復が必要な場合は、必ず `umount` するか、Live CD/USB から起動して実行してください。

## よく連携されるコマンド
### シナリオ例: 外付けHDDの修復 (運用視点)
* **目的**: 正常に取り外せなかった USB HDD が読み取れなくなったため、修復を試みる。
* **手順**:
    1. デバイス名の特定 (`lsblk`)
    2. アンマウントの確認 (`umount`)
    3. 修復の実行 (`fsck`)
* **コマンド例**:
    ```bash
    # /dev/sdb1 のエラーを自動的に修復する
    sudo fsck -y /dev/sdb1
    ```

## オプション説明
※実行には **root 権限** が必須です。

| オプション | 説明 |
| :--- | :--- |
| ⭐️ `-A` | `/etc/fstab` に記載されている全てのファイルシステムを順次チェックする。 |
| ⭐️ `-y` | 全ての質問に "yes" と答え、自動的に修復を試みる。 |
| ⭐️ `-n` | 実際には修復せず、エラーの有無だけを確認する（ReadOnlyモード）。 |
| `-f` | ファイルシステムが「クリーン（正常）」とマークされていても強制的にチェックする。 |
| `-t <type>` | ファイルシステムの種類を明示的に指定する。 |

## 終了ステータス (Exit Codes)
`fsck` の戻り値はビットマスクになっており、スクリプトでの判定に役立ちます。
* `0`: エラーなし
* `1`: ファイルシステムのエラーが修復された
* `2`: システムを再起動する必要がある
* `4`: ファイルシステムのエラーが未修復のまま残っている
* `8`: 操作エラー

## よく使われる業務シナリオ

### 1. インフラ構築・運用視点
* **タスク**: 起動時の自動チェック設定。
* **解説**: `/etc/fstab` の第6フィールドが `1` または `2` になっていると、起動時に `fsck` が走ります。ルートパーティションは `1`、それ以外は `2` にするのが一般的です。

### 2. ブルーチーム視点
* **タスク**: 不審な強制終了後の整合性確保。
* **解説**: 攻撃者による `reboot -f` や電源断を伴う攻撃のあと、システムが不安定になるのを防ぐために、フォレンジック調査の前にクローンしたイメージに対して実行し、構造の健全性を確認します。

### 3. レッドチーム視点
* **タスク**: データの永続的な破壊 (Sabotage)。
* **解説**: マウント中のファイルシステムに対して `fsck` を強制実行（または壊れた `fsck.ext4` を配置）させることで、ターゲットのデータを修復不能な状態に追い込みます。

## エラーメッセージとトラブルシューティング

1.  **「/dev/sda1 is mounted. e2fsck: Cannot continue, aborting.」**:
    * **原因**: 使用中のデバイスをチェックしようとした。
    * **解決策**: `sudo umount /dev/sda1` を実行する。ルートパーティションの場合は、シングルユーザーモードで再起動するか、Liveメディアを使用する。

2.  **修復後に `lost+found` ディレクトリに大量のファイルがある**:
    * **原因**: 持ち主（ディレクトリ構造上の位置）が分からなくなったデータ断片がここに回収された。
    * **解決策**: `file` コマンドなどで中身を確認し、手動で復元を試みる。

## セキュリティに関する考慮事項

### 脆弱性とリスク
* **特権昇格**: もし `fsck` が SUID 設定されている（稀ですが）場合、特定のオプションを介して任意のブロックを操作することで、特権奪取に悪用されるリスクがあります。

### 対応策・緩和策 (ブルーチーム視点)
* **Prevention (予防)**: ジャーナリングファイルシステム（ext4, XFS等）を使用することで、`fsck` が必要になる頻度を下げ、データの安全性を高める。
* **Monitoring**: 起動時の `fsck` ログ (`/var/log/boot.log` 等) を監視し、ディスク故障の初期兆候（セクタエラー等）を見逃さない。

## 注意点・補足
* **SSD/NVMe 時代**: 現代のファイルシステムはジャーナリング機能により非常に頑丈になっており、手動で `fsck` をかける機会は減りました。しかし、ファイルシステムのメタデータが壊れた際の「最後の砦」であることに変わりはありません。
