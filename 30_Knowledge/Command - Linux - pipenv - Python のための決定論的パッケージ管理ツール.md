# Command - Linux - pipenv - Python のための決定論的パッケージ管理ツール

## 概要

`pipenv` は、`pip` と `virtualenv` の機能を一つにまとめ、さらに `Pipfile` と `Pipfile.lock` という仕組みを導入したツールです。これにより、「自分の PC では動くのにサーバーでは動かない」という依存関係の不一致（Dependency Hell）を解消し、誰でも同じ環境を完全に再現できるようになります。

## 🛠 パッケージ管理の進化

| ツール | 管理ファイル | 仮想環境 | 特徴 |
| :--- | :--- | :--- | :--- |
| `pip` | `requirements.txt` | 手動 (venv等) | シンプルだが、依存関係の依存先まで固定しにくい。 |
| **`pipenv`** | **`Pipfile`** | **自動生成** | **直感的。ロックファイルによる厳密な再現性。** |
| `Poetry` | `pyproject.toml` | 自動生成 | 現代の標準。ビルドや公開機能も備える多機能派。 |

## 主要なコマンド

| コマンド | 説明 |
| :--- | :--- |
| ⭐️ `install <pkg>` | パッケージを導入し、`Pipfile` に記録する。 |
| ⭐️ `shell` | プロジェクト専用の仮想環境の中に入る（アクティベート）。 |
| ⭐️ `run <cmd>` | 仮想環境に入らずに、環境内の Python でコマンドを実行する。 |
| ⭐️ `lock` | 現在の環境を解析し、厳密な `Pipfile.lock` を生成する。 |
| ⭐️ `sync` | `Pipfile.lock` に基づいて、環境を寸分違わず再現する。 |
| ⭐️ `check` | インストール済みのライブラリに既知の脆弱性がないかスキャンする。 |

---

## 💡 決定論的（Deterministic）ビルドの力

`pipenv` の真価は `Pipfile.lock` にあります。ここには各ライブラリのバージョンだけでなく、**ハッシュ値（指紋）**も記録されます。これにより、ダウンロード中にファイルが改ざんされていないか、あるいは別の意図しないバージョンが紛れ込んでいないかを保証できます。

## 業務シナリオ別活用法

### 1. インフラ構築・運用視点

* **タスク**: 本番サーバーへのデプロイ。
* **解説**: 開発者が作成した `Pipfile.lock` をサーバーに持っていき、`pipenv sync` を実行します。これにより、ライブラリのバージョンのズレによる「深夜のデバッグ作業」を撲滅します。

### 2. ブルーチーム視点

* **タスク**: サプライチェーン攻撃の防御。
* **解説**: 定期的に `pipenv check` を実行することで、プロジェクトが依存している数多くのライブラリの中に、CVE（既知の脆弱性）が報告されているものが含まれていないかを自動で監視します。

### 3. レッドチーム視点

* **タスク**: ターゲット環境の脆弱性特定。
* **解説**: 侵入後、もしプロジェクトディレクトリに `Pipfile.lock` が落ちていればラッキーです。その中身を解析することで、そのシステムがどのライブラリのどのバージョンを使っているかが手に取るようにわかります。そこから「古いライブラリの脆弱性」を突いて、さらなる権限昇格を狙います。

## セキュリティに関する考慮事項

### 脆弱性とリスク

* **ロックファイルの信頼性**: `Pipfile.lock` 自体が改ざんされると、偽のライブラリをインストールさせられる可能性があります。
* **環境変数の露出**: `pipenv` は `.env` ファイルを自動で読み込む機能があります。ここに API キーを置いたままファイルを共有してしまうミスに注意が必要です。

### 対応策・緩和策 (ブルーチーム視点)

* **CI/CD でのチェック**: コードをプッシュするたびに CI 上で `pipenv check` を走らせ、セキュリティ基準を満たさないコードが本番に行かないようにします。
