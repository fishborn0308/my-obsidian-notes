# Command - Linux - auditd - Linux監査システムのデーモンと管理

## 概要
`auditd` は Linux カーネルの監査サブシステム（Audit Framework）からの情報を受け取り、ログファイル（`/var/log/audit/audit.log`）に記録する常駐デーモンです。ファイルアクセス、システムコール、ネットワーク接続などを詳細に追跡できます。
(出自: `auditd` パッケージに含まれる。管理用ツールの `Auditctl`, `ausearch`, `aureport` 等とセットで使用される)

## 類似コマンドと差異
| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `syslog / journald` | 一般的なシステムイベントを記録。詳細な「誰がどのファイルに何をしたか」の追跡には不向き。 | システムログ全般 |
| `auditd` | セキュリティ監査に特化。システムコールレベルで監視可能。改ざん耐性が高い。 | セキュリティ監査 |
| `EDR (Agent)` | より高度な相関分析や自動応答が可能だが、導入コストが高い。 | 商用セキュリティ |

## よく連携されるコマンド
### シナリオ例: 攻撃者の活動を時系列で解析 (ブルーチーム視点)
* **目的**: 特定の不審な実行ファイルがどのような動作をしたかを調査する。
* **連携コマンド**: `ausearch`, `aureport`
* **解説**: `auditd` 自体はログを吐くだけのため、解析には `ausearch`（検索）や `aureport`（集計）を使用します。
* **コマンド例**:
    ```bash
    # 特定の実行ファイルに関するログを抽出
    sudo ausearch -i -p <Process_ID>
    # 実行されたすべてのシステムコールのサマリーを表示
    sudo aureport -s -i
    ```

## オプション説明（管理用ツール：auditctl）
| オプション | 説明 |
| :--- | :--- |
| **ルール操作** | |
| ⭐️ `-l` | 現在適用されているすべての監査ルールを表示する。 |
| ⭐️ `-w <path> -p <perms> -k <key>` | 指定したパスを監視。`-p`は権限(rwxa)、`-k`は検索用のタグ（キー名）。 |
| `-a <action>,<list> -S <syscall>` | 特定のシステムコールを監視対象に追加。 |
| `-D` | すべての監査ルールを削除する。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点
* **タスク**: 設定ファイルの改ざん監視。
* **組み合わせ**: `auditctl -w /etc/shadow -p wa -k shadow_mod`
* **解説**: `/etc/shadow`（パスワード情報）が書き換え(w)られたり、属性変更(a)されたりした際にログを残し、`shadow_mod` というタグで後から検索しやすくします。
* **例**:
    ```bash
    # shadowファイルの変更を監視
    sudo auditctl -w /etc/shadow -p wa -k password_change
    ```

### 2. ブルーチーム視点
* **タスク**: 権限昇格の痕跡（PrivEsc）の発見。
* **組み合わせ**: `-a always,exit -F arch=b64 -S execve`
* **解説**: 全てのコマンド実行（execve システムコール）を記録します。攻撃者が持ち込んだツールや不審なシェル操作をすべて捕捉可能です。
* **例**:
    ```bash
    # 全てのコマンド実行を記録する（負荷に注意）
    sudo auditctl -a always,exit -F arch=b64 -S execve -k cmd_audit
    ```

### 3. レッドチーム視点
* **タスク**: 監視のバイパスとログの隠蔽。
* **組み合わせ**: `service auditd stop` (権限がある場合)
* **解説**: 攻撃者はまず `auditctl -l` で何が監視されているかを確認します。その後、ログ出力を停止させるか、監視対象外のディレクトリで活動を試みます。
* **例**:
    ```bash
    # 監査ルールを確認し、自分が監視されているか知る
    auditctl -l
    # ログ記録を一時停止（要権限、即座に検知されるリスク大）
    auditctl -e 0
    ```

## エラーメッセージとトラブルシューティング

1.  **エラーメッセージ例 1**: `Error sending add rule data request (Exceeds existing limit)`
    * **考えられる原因**: ルールの数が上限に達している、または構文に誤りがある。
    * **解決策**: 不要なルールを削除するか、設定ファイル `/etc/audit/audit.rules` を見直す。

2.  **エラーメッセージ例 2**: `auditd: can't open log file`
    * **考えられる原因**: ディスクフル、またはログディレクトリの権限異常。
    * **解決策**: `/var/log/audit/` の容量を確認し、ログのローテーション設定を調整する。

## 環境変数と設定ファイル

### 関連する設定ファイル
* **`/etc/audit/auditd.conf`**: デーモン自体の動作（ログの保存先、ローテーション、ディスク容量閾値など）の設定。
* **`/etc/audit/rules.d/audit.rules`**: 永続的な監査ルールの記述場所。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例
* **脆弱性**: ログがローカルに保存されている場合、root権限を奪取された攻撃者によってログ自体を改ざん・消去されるリスクがあります。
* **悪用シナリオ**: 攻撃者は `auditd` を停止させるのではなく、`audit.rules` を書き換えて、自分の活動に関連するタグだけを記録させないように工作します。

### GTFOBins / LOLBAS における利用例
`auditctl` 自体で権限昇格はできませんが、情報の収集（機密ファイルの閲覧監視の回避など）に利用されます。

### 対応策・緩和策 (ブルーチーム視点)
* **Prevention (予防)**: 監査ルールを不変（immutable）にするため、ルールファイルの末尾に `-e 2` を追加します。これにより、リブートするまでルールの変更が不可になります。
* **Detection (検知)**: ログを外部の SIEM（Splunk, ELK等）にリアルタイム転送し、ローカルでのログ消去を無効化します。

## 注意点・補足
* **パフォーマンスへの影響**: 全てのシステムコールを監視すると、特にI/O負荷の高いサーバ（DBサーバ等）では大幅なパフォーマンス低下を招くため、フィルタリングが不可欠です。
* **ログの解釈**: `audit.log` はそのままでは読みづらいため、`ausearch -i` を使って数値を人間が読める形式（UIDをユーザー名にする等）に変換するのが基本です。
