# Command  - Linux - service - SysVinitサービスを管理する (旧)

## 概要

`service` は、SysVinitやUpstartといった旧世代のinitシステムにおいて、サービスの**起動 (start)**、**停止 (stop)**、**再起動 (restart)**、**状態確認 (status)** を行うためのラッパースクリプトです。`/etc/init.d/` ディレクトリに置かれた実際の起動スクリプトを、統一的で分かりやすいインターフェイスで操作できるようにします。

**⚠️ 重要**: `service` コマンドは**レガシー（旧式）**です。モダンな `systemd` システムでは、`service` コマンドは `systemctl` への互換性ラッパーとして動作しますが、**サービスの管理には `systemctl` を使用することが強く推奨されます**。

(出自: `sysvinit-utils` (Debian系) または `initscripts` (RHEL系) パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`systemctl`** | **`service` のモダンな後継**。`systemd` を制御するための中心的なコマンドであり、サービスの起動/停止だけでなく、自動起動の有効化/無効化、ログの確認、依存関係の管理など、遥かに高機能です。 | `systemctl` |
| **`/etc/init.d/<script>`**| `service` が内部的に呼び出す起動スクリプトを直接実行する方法です。`service` は、このスクリプトをより標準的な方法で扱うためのラッパーです。 | `service` (旧環境) |
| **`chkconfig` / `update-rc.d`**| サービスの**自動起動**を設定するコマンドでした。`service` は、**今すぐ**サービスを起動/停止するためのコマンドです。 | - |

## よく連携されるコマンド

`service` コマンドは、設定ファイルを編集した後に、その変更を適用するために使われます。

### シナリオ例: 設定変更後のサービス再起動 (インフラ構築・運用視点)

* **目的**: Apacheの設定ファイル (`httpd.conf`) を編集した後、サービスを再起動して変更を反映させる。
* **連携コマンド**: `vi` (または他のエディタ), `service`
* **解説**: レガシーシステムにおける基本的な運用フローです。設定ファイルを編集し、`service ... restart` でサービスを再起動します。
* **コマンド例**:

    ```bash
    # (CentOS 6 などのレガシーシステムでの例)
    # 1. 設定ファイルを編集
    sudo vi /etc/httpd/conf/httpd.conf

    # 2. サービスを再起動して変更を適用
    sudo service httpd restart
    ```

## オプション説明 (`service` のアクション)

| オプション | 説明 |
| :--- | :--- |
| **サービス操作** | |
| ⭐ `start` | サービスを起動します。 |
| ⭐ `stop` | サービスを停止します。 |
| ⭐ `restart`| サービスを再起動します。 |
| `reload` | サービスを停止せずに、設定ファイルのみを再読み込みさせます（サービスが対応している場合）。 |
| **状態表示** | |
| ⭐ `status` | サービスの現在の状態（実行中か、PIDなど）を表示します。 |
| `--status-all` | (Debian系) システム上の全てのSysVinitサービスの状態を一覧表示します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: レガシーシステムで、設定ファイルを変更した後にサービスを再起動して変更を適用する。
* **組み合わせ**: `sudo service <service_name> restart`
* **解説**: 上記シナリオ例の通り、`httpd.conf` や `my.cnf` などの設定ファイルを編集した後に、サービスを再起動して変更を反映させます。
* **例**:

    ```bash
    # MySQLサービスを再起動
    sudo service mysqld restart
    ```

### 2. ブルーチーム視点

* **タスク**: インシデント対応中に、悪意のあるサービスを停止して封じ込める。
* **組み合わせ**: `sudo service <malicious_service> stop`
* **解説**: 調査の結果、`/etc/init.d/backdoor` という悪意のあるサービスが永続化のために設置され、実行されていることを発見した場合、このコマンドでサービスを即座に停止させ、攻撃者の活動を無力化します。
* **例**:

    ```bash
    # 不審なサービス 'evil-svc' を停止
    sudo service evil-svc stop
    ```

### 3. レッドチーム視点

* **タスク**: 侵入したレガシーシステムで、稼働中のサービスを偵察する。
* **組み合わせ**: `service --status-all` (Debian系)
* **解説**: このコマンドで、稼働中のサービス (`[ + ]` で表示) と停止中のサービス (`[ - ]`) を一覧で把握します。これにより、システムの役割や、攻撃対象となりうる脆弱なサービスがないかを探します。
* **例**:

    ```bash
    # 全サービスのステータスを確認
    service --status-all
    ```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `<service_name>: unrecognized service`
    * **考えられる原因**: 指定したサービス名の起動スクリプトが `/etc/init.d/` ディレクトリに存在しません。
    * **解決策**: サービス名が正しいか、パッケージが正しくインストールされているかを確認してください。

2. **メッセージ (systemd環境)**: `Redirecting to /bin/systemctl restart <service_name>.service`
    * **考えられる原因**: これはエラーではありません。`systemd` 環境で `service` コマンドが実行されたため、コマンドが自動的に `systemctl` にリダイレクト（転送）されたことを示すメッセージです。
    * **解決策**: -

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `service` 自体ではなく、それが管理する**SysVinitスクリプト**の仕組みが**永続化**に悪用されます。
* **悪用シナリオ**: 攻撃者がroot権限を奪取した後、`/etc/init.d/` 配下に悪意のあるスクリプトを設置し、`chkconfig` や `update-rc.d` でサービスとして登録します。

### GTFOBins / LOLBAS における利用例

* **機能**: **Privilege Escalation** (権限昇格)
* **解説**: `service --status-all` の出力が長い場合、内部的に `less` などのページャを呼び出します。もしユーザーが `sudo service --status-all` を実行できる権限を持っている場合、ページャの画面で `!/bin/sh` と入力することで、ページャのシェルエスケープ機能を悪用し、**root権限のシェル**を起動できてしまいます。
* **参照**: `https://gtfobins.github.io/gtfobins/service/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: `sudoers` の設定を厳格にし、一般ユーザーに `service` コマンドの `sudo` 権限を安易に与えない。
* **Detection (検知)**: `/etc/init.d/` 配下のファイルの変更や追加をファイル整合性監視(FIM)ツールで監視する。

## 注意点・補足

* **`systemctl` への移行**: モダンなシステムではサービスの管理は全て `systemctl` で行うべきです。`systemctl` は、より詳細な状態管理、ログ連携、依存関係の制御など、遥かに優れた機能を提供します。
