# Command - Windows - cmd - perfmon - パフォーマンス モニターの起動

## 概要

`perfmon` は、Windows システムのパフォーマンスを詳細に分析するための強力な管理ツールです。リアルタイムのグラフ表示だけでなく、特定の「カウンタ」（CPU、メモリ、ディスク、ネットワーク等の詳細な指標）を一定期間記録する「データ コレクター セット」を作成し、パフォーマンスの傾向分析や障害原因の特定に使用します。
(出自: Windows NT 以降標準搭載)



## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 (あれば) |
| :--- | :--- | :--- |
| `resmon` | リソース モニター。リアルタイムの「プロセス」と「リソース」の紐付けに特化。 | `resmon` (即時調査) |
| `logman` | **CLI版パフォーマンスモニター。** パフォーマンスカウンタの収集をコマンドラインやスクリプトから制御できる。 | `logman` (自動化) |
| `taskmgr` | タスク マネージャー。最も簡易的で直感的な概要把握。 | `taskmgr` (簡易確認) |

## よく連携されるコマンド

`perfmon` は主に GUI として操作されますが、CLI から特定のコンフィグファイルを指定して起動したり、収集したデータを他のツールで解析したりします。

### シナリオ例: パフォーマンスカウンタ設定の読み込み (運用自動化)

* **目的**: 定義済みの設定ファイルを使用して、特定の監視項目を即座に表示する。
* **連携コマンド**: `/settings` オプション
* **解説**: 複雑なカウンタ設定を保存した XML ファイルを指定して起動します。
* **コマンド例**:
    ```cmd
    REM 保存済みの設定ファイルを使用して起動
    perfmon /settings "C:\Configs\ServerHealth.xml"
    ```

## スイッチ/オプション説明

| スイッチ | 説明 |
| :--- | :--- |
| **表示制御** | |
| `/res` | リソース モニター (`resmon`) を直接起動する。 |
| `/report` | 指定したパスのレポート（収集済みデータ）を表示する。 |
| ⭐ `/sys` | システム全体のパフォーマンスを監視するモードで起動する。 |
| **管理操作** | |
| `/rel` | 信頼性モニター (Reliability Monitor) を起動し、システムの安定性履歴を表示する。 |
| `/settings <file>` | XML 設定ファイルを読み込んで起動する。 |

## よく使われる組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 24時間のサーバー負荷測定（データ収集）
* **解説**: 「データ コレクター セット」を作成し、1日を通した CPU やディスク I/O の推移をログファイル（.blg）に記録します。これにより、夜間のバッチ処理中に発生するスパイクなどを特定できます。
* **例**: `perfmon` > 「データ コレクター セット」 > 「ユーザー定義」から新規作成。


### 2. ブルーチーム視点

* **タスク**: ステルス性の高いマルウェアの活動検知
* **解説**: `resmon` に現れないような微細な CPU 消費や、特定のシステムコール、割り込み（Interrupts）の増大を監視します。不審なタイミングでネットワークカウンタ（Bytes Sent/sec）が上昇していないか、ベースラインと比較します。
* **例**: カウンタに「Process」->「% Processor Time」を選択し、特定の疑わしいプロセスを詳細に追跡。

### 3. レッドチーム視点 (偵察)

* **タスク**: アンチ・サンドボックスおよび監視環境の検知
* **解説**: 実行環境に `perfmon` や `logman` によるデータ収集が走っているかを確認します。高度な環境では、監視ツールのオーバーヘッドを測定することで、自分が「観察されている」かどうかを判断します。
* **例**: 
    ```cmd
    REM 現在実行中のデータコレクターセットを確認（logmanを使用）
    logman query
    ```

## エラーメッセージとトラブルシューティング

### よくあるトラブル

1.  **現象**: カウンタが一覧に表示されない、または「0」のまま動かない。
    * **考えられる原因**: 
        - パフォーマンス カウンタのライブラリが破損している。
        - 実行ユーザーに `Performance Log Users` または `Administrators` 権限がない。
    * **解決策**: コマンドプロンプトから `lodctr /R` を実行してカウンタを修復する（要管理者権限）。

2.  **現象**: リモートコンピュータのカウンタを取得できない。
    * **原因**: リモートマシンの「Remote Registry」サービスが停止している、またはファイアウォールで遮断されている。
    * **解決策**: 対象マシンのサービス状態を確認する。

## 環境変数と設定ファイル

* **ログファイル形式**: `.blg` (Binary Log) 形式で保存されます。これは `perfmon` 自身や、Excel (CSV変換後) で解析可能です。
* **設定ファイル**: XML 形式でインポート/エクスポートが可能です。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **悪用シナリオ**: 
    - **Information Disclosure**: パフォーマンスカウンタを通じて、他のユーザーが実行しているプロセスの名称やリソース使用状況を推測される（サイドチャネル攻撃の足がかり）。
    - **Denial of Service (DoS)**: 非常に短い間隔（例: 1ms）で大量のカウンタを収集し続ける設定を強制することで、システム全体の負荷を不当に高める。

### LOLBAS (Living Off The Land Binaries and Scripts) における利用例

* **機能**: `Information Requirements`
* **参照**: [LOLBAS - perfmon.exe](https://lolbas-project.github.io/lolbas/Binaries/Perfmon/)

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 
    - ユーザーを `Performance Log Users` グループに限定し、最小権限を徹底する。
* **Detection (検知)**: 
    - **イベントID 4688**: `perfmon.exe` の不審な親プロセス（例: Webサーバープロセス）からの起動を監視。
    - 大量のレポート生成ファイルが作成されていないかファイルシステムを監視。

## 注意点・補足

* **信頼性モニター (`perfmon /rel`)**: 実務で非常に役立ちます。過去に発生したアプリのクラッシュ、Windows Update の成功/失敗、ドライバーのインストール履歴をカレンダー形式で確認でき、トラブルの「起算点」を特定するのに最適です。
* **オーバーヘッド**: 監視項目を増やしすぎると、監視自体がシステム負荷の原因になるため、本番環境での長期収集は項目を絞ることが推奨されます。
