# Command - Windows - cmd - gpupdate - グループ ポリシーの更新

## 概要

`gpupdate` は、ローカルおよび Active Directory ベースのグループ ポリシー設定を更新（リフレッシュ）します。

通常、グループ ポリシーはバックグラウンドで一定の間隔（デフォルトでは 90 分 + ランダムなオフセット）で自動更新されますが、`gpupdate` を実行することで、管理者が変更したポリシー設定を待機することなく即座にクライアント コンピューターへ反映させることができます。

(出自: `Windows 標準搭載 - 外部コマンド (gpupdate.exe)`)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い使い分けのポイント | 推奨 (あれば) |
| :--- | :--- | :--- |
| `gpresult` | 現在適用されているポリシーを「確認」する。 | 確認用 |
| `Invoke-GPUpdate` | PowerShell 版。リモート コンピューターに対して一括で更新を指示できる。 | リモート一括更新 |
| `secedit /refreshpolicy` | 古い Windows バージョンで使用されていたポリシー更新コマンド。 | レガシー環境 |

## よく連携されるコマンド

### シナリオ例: セキュリティ設定変更後の即時強制適用 (運用・防御)

* **目的**: AD 上で変更した「パスワードの有効期限」や「USB メモリの使用制限」を、全ユーザーに今すぐ適用させる。
* **連携コマンド**: `gpupdate`, `gpresult`
* **解説**: `/force` スイッチを使用して、変更の有無にかかわらずすべての設定を再適用し、その後 `gpresult` で確認します。
* **コマンド例**:
    ```cmd
    REM すべてのポリシー設定を強制的に再適用
    gpupdate /force
    REM 反映結果を簡易確認
    gpresult /r
    ```

## スイッチ/オプション説明

| スイッチ | 説明 |
| :--- | :--- |
| ⭐ `/force` | **【重要】** 変更のあったポリシーだけでなく、**すべての**ポリシー設定を再適用する。 |
| `/target:{computer | user}` | 「コンピューター構成」または「ユーザー構成」のいずれか一方のみを更新する。 |
| `/wait:value` | ポリシー処理が完了するまで待機する時間を秒単位で指定する（デフォルトは 60 秒）。 |
| ⭐ `/logoff` | 更新の反映にログオフが必要な場合（一部のユーザー設定など）、自動的にログオフを促す。 |
| ⭐ `/boot` | 更新の反映に再起動が必要な場合（一部のシステム設定など）、自動的に再起動を促す。 |
| `/sync` | 次のフォアグラウンド更新時に、同期的にポリシーを適用させる（起動時やログイン時と同様の挙動）。 |

## よく使われる組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 新しく配布したソフトウェアのショートカットや、ネットワークドライブの割り出しが反映されていないユーザーに対し、手動更新を指示する。
* **解説**: ユーザーに「一度ログアウトして」と頼む代わりに、`gpupdate /force` を実行させることで解決するケースが多いです。

### 2. ブルーチーム視点 (修復と防御)

* **タスク**: 侵害された可能性のある端末において、攻撃者によって緩和された可能性があるセキュリティ設定を、AD 正規定義に基づき強制的に「上書き」して復旧させる。
* **例**:
    ```cmd
    REM 悪意ある変更を上書きするために強制適用
    gpupdate /force
    ```

### 3. レッドチーム視点 (永続化の発動 - T1484.001)

* **タスク 1: Triggering Malicious GPO (悪意ある GPO の即時発動)**
    * **目的**: ドメイン管理者権限を奪取し、AD 上で作成した「全端末でのバックドア実行」用 GPO を、ターゲット端末で即座に実行させる。
* **タスク 2: Defense Evasion (防御機能の無効化確認)**
    * **目的**: EDR の無効化やログ出力の停止を GPO で設定した後、テスト端末で `gpupdate /force` を実行して、期待通りに防御が沈黙するかを確認する。

## エラーメッセージとトラブルシューティング

1.  **コンピューター/ユーザー ポリシーの更新を正常に完了できませんでした。**
    * **原因**: ドメイン コントローラー (DC) との通信エラー。DNS の問題やネットワークの遮断が考えられます。
    * **解決策**: `nltest /dsgetdc:` や `ping` で DC への疎通を確認してください。

2.  **一部の設定は、次回のログオンまたは起動時に有効になります。**
    * **原因**: ソフトウェアのインストールやフォルダー リダイレクトなど、実行中のセッションでは変更できない項目が含まれている。
    * **解決策**: メッセージに従い `/logoff` または `/boot` を併用するか、手動で再起動します。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **GPO テンプレートの改ざん**:
    * 攻撃者がドメイン上の GPO テンプレート（SYSVOL 内）を書き換えた場合、各端末が `gpupdate` を行うタイミングでその「悪意ある設定」が自動的に全社に拡散されます。

### LOLBAS (Living Off The Land Binaries and Scripts) における利用例

* **機能**: なし（直接的な実行バイパス等には使われないが、攻撃の「トリガー」として利用される）。

### 対応策・緩和策 (ブルーチーム視点)

* **Detection (検知)**:
    * **イベントID 4688**: 短時間に大量の端末で `gpupdate /force` が実行されている場合、ドメインレベルでの攻撃（GPO 改ざんによる一斉感染）を疑う必要があります。
* **Prevention (予防)**:
    * **SYSVOL の保護**: GPO の実体ファイルが格納されている SYSVOL 共有への書き込み権限を、極めて限定された管理者にのみ絞る。

## 注意点・補足

* **ネットワーク負荷**: 全社員が一斉に `gpupdate /force` を実行すると、DC へのトラフィックが急増し、ネットワーク遅延を引き起こす可能性があるため注意が必要です。
* **非対話実行**: バッチファイル内で使用する場合、ログオフや再起動のプロンプトを避けるために `/wait:0` などを検討することもありますが、基本的には `/force` だけで十分なケースがほとんどです。
