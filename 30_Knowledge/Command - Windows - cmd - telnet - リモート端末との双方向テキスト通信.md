# Command - Windows - telnet - リモート端末との双方向テキスト通信

## 概要

`telnet` は、Telnet プロトコルを使用してリモート ホストと通信するためのクライアント ツールです。

元々はリモート管理用（ログイン用）でしたが、現在は特定のポートに対して TCP 接続を試み、その応答（バナー）を確認する **ポートスキャナ代わりの診断ツール** として多用されます。

(注: Windows では「Windows の機能の有効化または無効化」から有効にする必要があります)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 (あれば) |
| :--- | :--- | :--- |
| `nc` (Netcat) | 「ハッカーのアーミーナイフ」。UDP 対応やファイル転送も可能。 | `nc` (ペンテスト) |
| `ssh` | 暗号化通信。リモートログインの標準。 | `ssh` (管理業務) |
| `Test-NetConnection` | PowerShell 版。ポート確認に特化しており、結果をオブジェクトで返す。 | `TNC` (Windows自動化) |

## 有効化コマンド (Windows)

Windows では標準で無効なため、以下のコマンドで有効化します。

```cmd
REM 管理者権限で実行
dism /online /Enable-Feature /FeatureName:TelnetClient
````

## よく連携されるコマンド

### シナリオ例: メールサーバー (SMTP) の生存確認 (インフラ運用)

- **目的**: 25番ポートが外部から到達可能か、正しく応答を返すかを確認する。
- **解説**: 接続後、SMTP プロトコルのコマンドを直接打ち込んで応答を見ます。
- **コマンド例**:

    DOS

    ```
    telnet mail.example.com 25
    ```

## 構文と操作

|**形式**|**説明**|
|---|---|
|`telnet <Host> <Port>`|指定したホストのポートに TCP 接続を試みる。ポートを省略すると 23 (既定)。|
|`Ctrl + ]`|**エスケープ文字。** 接続中に Telnet プロンプトに戻る。|
|`quit`|接続を終了して Telnet を閉じる。|

## よく使われる組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

- **タスク**: ファイアウォール許可設定の動作確認
- **解説**: 新しく構築した Web サーバーや DB サーバーに対し、クライアント端末から `telnet` でポートが開いているか叩きます。
- **例**: `telnet 192.168.1.50 80` （真っ暗な画面になれば接続成功 = ポート開放中）

### 2. ブルーチーム視点

- **タスク**: 暗号化されていない管理アクセスの検知
- **解説**: ネットワーク内で 23番ポート (Telnet) が使われていないか監視します。ID/PW が平文で流れるため、レガシーデバイス（古いスイッチ等）の放置を特定します。

### 3. レッドチーム視点 (OSCP / Reconnaissance)

- **タスク**: バナーグラビングによる OS/サービス特定
- **解説**: ポートに接続した際にサーバーが返すメッセージから、ソフトウェア名やバージョンを特定します。
- **例**:

    DOS

    ```
    telnet 10.10.10.5 80
    HEAD / HTTP/1.0
    [Enterを2回押す]
    ```

    → 応答の `Server: Apache/2.4.41 (Ubuntu)` といった情報から攻撃手法を絞り込みます。

## エラーメッセージとトラブルシューティング

### よくあるトラブル

1. **現象**: `'telnet' は、内部コマンドまたは外部コマンド...として認識されていません。`
    
    - **原因**: Telnet クライアントが有効になっていない。
    - **解決策**: 前述の `dism` コマンドで機能を有効にする。
        
2. **現象**: `接続中: <Host>...ホストに接続できませんでした。`
    
    - **原因**: ポートが閉じている、サーバーがダウンしている、またはファイアウォールで遮断されている。
    - **解決策**: サーバー側のリスニング状態 (`netstat -ano`) と FW ルールを確認する。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

- **悪用シナリオ**:
    - **Sniffing**: Telnet 通信を Wireshark 等で傍受することで、管理者のパスワードを容易に窃取できる。
    - **Information Disclosure**: 不要なバナー情報を公開し続けることで、脆弱なバージョンの特定を容易にしてしまう。

### 対応策・緩和策 (ブルーチーム視点)

- **Prevention (予防)**:
    - リモート管理には必ず **SSH** (Port 22) を使用し、Telnet (Port 23) は無効化する。
    - サーバー設定でバナー情報を隠蔽（ServerTokens Prod 等）する。

## 注意点・補足

- **エコーなし**: Telnet で HTTP や SMTP を叩く際、自分が打った文字が表示されないことがありますが、そのまま打ち込んで Enter を押せばコマンドは送信されます。
- **HTTP/1.1 以降**: 最近の Web サーバーは `Host` ヘッダーを必須とするため、`telnet` での疎通確認が難しくなっています（`curl` や `Invoke-WebRequest` の方が推奨されるケースが多いです）。
