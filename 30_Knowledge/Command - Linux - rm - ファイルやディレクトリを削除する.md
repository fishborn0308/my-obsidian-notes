# Command - Linux - rm - ファイルやディレクトリを削除する

## 概要

`rm` は、指定されたファイルやディレクトリのリンクをファイルシステムから削除します。基本的には削除されたデータが格納されていた領域を「再利用可能」としてマークするだけであり、ディスク上の実データが即座にゼロで上書きされるわけではありません（※フォレンジックの観点で重要）。

## 🛠 `rm` vs `rmdir` vs `shred`

| コマンド | 役割 | 特徴 |
| :--- | :--- | :--- |
| **`rm`** | ファイル・ディレクトリの削除。 | 万能。ディレクトリ削除には `-r` が必要。 |
| `rmdir` | **空の**ディレクトリのみ削除。 | 安全。中身があるディレクトリは消せない。 |
| `shred` | ファイルを上書きして完全に抹消。 | セキュリティ重視。復元を困難にする。 |

## 主要なオプション

| オプション | 説明 |
| :--- | :--- |
| ⭐️ `-r`, `-R` | **再帰的 (Recursive)。** ディレクトリとその中身をすべて削除する。 |
| ⭐️ `-f` | **強制 (Force)。** 存在しないファイルを無視し、確認メッセージを出さない。 |
| ⭐️ `-i` | **対話的 (Interactive)。** 削除前に一つずつ確認を求める。 |
| ⭐️ `-I` | 3つ以上のファイルを消す時、または再帰的削除の時に一度だけ確認する。 |
| ⭐️ `-v` | 何が行われているかを表示する (Verbose)。 |
| `--no-preserve-root` | `/` (ルート) の削除を許可する（※絶対に試さないでください）。 |

## ⚠️ 伝説の禁忌: `rm -rf /`

システムの全ファイルを root 権限で、確認なしに、再帰的に削除するコマンドです。現代の OS では保護機能が働きますが、依然として「壊滅的なミス」の代名詞です。

## 🛡 生き残るための「安全な rm」の作法

1.  **`ls` で確認してから `rm` に変える**:
    ```bash
    # まず対象を確認
    ls *.log
    # 問題なければ履歴を呼び出して ls を rm に書き換える
    rm *.log
    ```
2.  **`-i` オプションをエイリアスにする**:
    `.bashrc` 等に `alias rm='rm -i'` と書いておき、うっかりミスを防ぎます。
3.  **相対パスではなく絶対パスを意識する**:
    `rm -rf ./ bin/` （スペースの打ち間違い）でカレントディレクトリの `bin` を消す事故を防ぎます。

## 業務シナリオ別活用法

### 1. インフラ構築・運用視点

* **タスク**: ディスク容量不足の解消（ログ掃除）。
* **解説**: 肥大化した古いログファイルを削除します。ただし、プログラムが実行中のファイルを消すと、ファイルの実体は消えてもディスク容量が解放されないことがあるため、必要に応じてプロセスの再起動を行います。

### 2. ブルーチーム視点

* **タスク**: マルウェア検体の削除と証跡保護。
* **解説**: 安易に `rm` で消すと、タイムスタンプなどのフォレンジック情報が失われます。削除前に必ずバックアップ（イメージ取得）を行い、機密データであれば `shred` で物理的に抹消します。

### 3. レッドチーム視点

* **タスク**: 痕跡消去（Anti-Forensics）。
* **解説**: 侵入時に使用したツールや一時ファイルを削除します。単純な `rm` では復元される可能性があるため、`-f` を使い、可能であればファイルシステム上のメタデータも考慮した削除手法を検討します。

## セキュリティに関する考慮事項

### 脆弱性とリスク

* **権限管理の不備**: 書き込み権限があるディレクトリでは、自分に所有権がないファイルでも `rm` で消せてしまう場合があります（Sticky Bit が立っていない場合）。

### 対応策・緩和策 (ブルーチーム視点)

* **`chattr +i` (Immutable)**: ⭐️ **プロの技。** 重要な設定ファイルを `sudo chattr +i /etc/passwd` のように保護すると、root であっても `rm` で削除や変更ができなくなります。
