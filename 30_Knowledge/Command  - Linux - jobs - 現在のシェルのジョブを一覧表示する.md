# Command  - Linux - jobs - 現在のシェルのジョブを一覧表示する

## 概要

`jobs` は、現在のシェルセッションから開始され、バックグラウンドで実行中または一時停止している**ジョブ**を一覧表示するためのシェル組み込みコマンドです。ジョブとは、シェルから見たプロセス管理の単位です。このコマンドは、`fg` や `bg`, `kill` といった他のジョブ管理コマンドと連携し、操作対象となるジョブの番号 (`%1`, `%2`など) を確認するために不可欠な役割を果たします。

(出自: `bash`, `zsh` などのシェルの組み込み機能)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`ps`** | システム上で実行されている**全てのプロセス**を一覧表示します。`jobs` は、**現在のシェルに紐付いているジョブ**のみを表示します。 | - |
| **`fg`, `bg`**| `jobs` で表示されたジョブを、それぞれフォアグラウンドやバックグラウンドに移動させるためのコマンドです。`jobs` は**表示**、`fg`/`bg` は**操作**を担当します。 | - |

## よく連携されるコマンド

`jobs` は、`fg` や `bg` でジョブを操作する前に、対象のジョブ番号を確認するためにほぼ常に連携して使われます。

### シナリオ例: 一時停止したジョブをフォアグラウンドに戻す (インフラ構築・運用視点)

* **目的**: `vi`での編集中に一時停止し、他の作業を行った後、どのジョブが`vi`だったかを確認して編集に戻る。
* **連携コマンド**: `jobs`, `fg`
* **解説**: `jobs` を実行して、`vi` セッションのジョブ番号（例: `[1]`）を確認し、`fg %1` でそのジョブをフォアグラウンドに戻します。
* **コマンド例**:

```bash
# (vi編集中に Ctrl+Z で一時停止)
#
# 別のコマンドを実行...
ls -l /etc/

# どのジョブがviだったか確認
jobs
# -> [1]+  Stopped                 vi my_script.sh

# 確認したジョブ番号を指定してフォアグラウンドに戻す
fg %1
```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **表示制御** | |
| ⭐ (引数なし) | 現在のシェルの全てのジョブ（実行中、停止中）を一覧表示します。 |
| `-l` | 通常の情報に加え、プロセスのグループID (PGID) を表示します。 |
| `-p` | 各ジョブのプロセスグループID (PGID) のみを表示します。 |
| `-r` | 実行中 (`Running`) のジョブのみを表示します。 |
| `-s` | 停止中 (`Stopped`) のジョブのみを表示します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: バックグラウンドや一時停止にしたジョブの状態を確認する。
* **組み合わせ**: `jobs`
* **解説**: 設定ファイルを編集中に `Ctrl+Z` で一時停止し、他のコマンドを実行した後、どのジョブをフォアグラウンドに戻すべきか忘れてしまった場合に `jobs` を実行して確認します。
* **例**:

```bash
# 現在のジョブを確認
jobs
```

### 2. ブルーチーム視点

* **タスク**: 調査対象のシェルセッションで隠されているプロセスがないかを確認する。
* **組み合わせ**: `jobs -l`
* **解説**: ライブレスポンス中に、調査対象のシェルで `jobs -l` を実行します。これにより、ユーザーが `Ctrl+Z` で意図的に隠した（一時停止した）不審なプロセスや、バックグラウンドで実行している悪意のあるスクリプトを発見できる可能性があります。
* **例**:

```bash
# プロセスIDも併せてジョブ一覧を表示
jobs -l
```

### 3. レッドチーム視点

* **タスク**: `tmux` や `screen` が使えない環境で、自身が起動した複数の攻撃ツールを管理する。
* **組み合わせ**: `jobs`
* **解説**: リバースシェル用のリスナー (`nc`) と、ペイロード配布用のWebサーバー (`python -m http.server`) などを `Ctrl+Z` と `bg` で管理し、`jobs` を実行して `[1] Running nc...`, `[2] Running python...` のように状態を常に把握します。
* **例**:

```bash
# 複数のバックグラウンドジョブの状態を確認
jobs
```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **現象**: **スクリプト内で `jobs` を実行したが機能しない。**
		* **考えられる原因**: デフォルトでは、シェルスクリプトなどの非対話的なシェルではジョブ管理は無効になっています。
		* **解決策**: スクリプトの冒頭で `set -m` を実行すると、ジョブ管理が有効になります。

## 環境変数と設定ファイル

* `jobs` はシェルの組み込み機能であり、その動作を直接制御する設定ファイルや環境変数は基本的にありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `jobs` 自体はシェルの基本機能であり、脆弱性はありません。
* **悪用シナリオ**: `jobs` の出力は**情報漏洩**に繋がる可能性があります。攻撃者は `jobs -l` を実行して、他の管理者がどのようなプロセスをバックグラウンドで動かしているかを偵察し、システムの運用状況を把握する可能性があります。

### GTFOBins / LOLBAS における利用例

`jobs` は基本的なシェル機能のため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: -
* **Detection (検知)**: ライブ調査中に `ps aux` などでプロセスリストを確認するだけでなく、不審なユーザーのシェルで `jobs` コマンドの実行を試み、そのセッション内で隠されているジョブがないかを確認することが有効です。

## 注意点・補足

* **シェルセッションとの関連**: `jobs` で表示されるジョブリストは、**現在のシェルセッションに固有のもの**です。別のターミナルやSSHセッションで `jobs` を実行しても、そこには表示されません。
