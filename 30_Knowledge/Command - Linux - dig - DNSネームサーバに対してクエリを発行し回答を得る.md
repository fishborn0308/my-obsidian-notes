# Command - Linux - dig - DNSネームサーバに対してクエリを発行し回答を得る

## 概要
`dig` は、DNS ネームサーバに対して問い合わせを行い、得られた回答を表示するコマンドラインツールです。柔軟なオプション、詳細な出力、および標準に準拠した挙動により、古い `nslookup` に代わって事実上の標準ツールとなっています。
(出自: `bind9-host` または `dnsutils` パッケージに含まれる)



## 類似コマンドと差異
| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `nslookup` | 古典的なツール。対話モードがあるが、挙動が OS によって異なり、詳細な情報取得には不向き。 | 互換性維持 |
| `host` | シンプルな出力。IPアドレスからホスト名（逆引き）をサッと確認したい時に便利。 | クイックな確認 |
| `dog` | `dig` のモダンな代替。Rust製で出力が見やすく、JSON出力もサポート。 | 開発者向け |
| `dig` | 最も多機能で、DNS サーバーが返す生の情報を正確に表示できる。 | **詳細な調査・デバッグ** |

## よく連携されるコマンド
### シナリオ例: 外部から見た自社ドメインの全レコード確認 (運用視点)
* **目的**: 権威サーバーに対して直接クエリを投げ、キャッシュの影響を受けずに設定を確認する。
* **連携コマンド**: `grep`, `awk`
* **解説**: 特定のネームサーバを指定してクエリを投げます。
* **コマンド例**:
    ```bash
    # 特定のDNSサーバ（例: Google 8.8.8.8）を使って A レコードを検索
    dig @8.8.8.8 example.com A
    ```

## オプション説明
| オプション | 説明 |
| :--- | :--- |
| **基本クエリ** | |
| ⭐️ `ANY` | 指定したドメインの利用可能な全てのレコードタイプを要求する。 |
| ⭐️ `-x <IP>` | **逆引き**。IPアドレスからドメイン名を探す。 |
| ⭐️ `+short` | 回答部分（IPのみ等）だけを簡潔に表示する。スクリプト利用に最適。 |
| **高度な調査** | |
| ⭐️ `+trace` | ルートサーバから順に再帰的な問い合わせプロセスを追跡表示する。 |
| `AXFR` | **ゾーン転送**を要求する。設定ミスがあると全レコードが漏洩する。 |
| **表示制御** | |
| `+noall +answer` | 余計なヘッダー等を消し、回答（ANSWER）部分のみを表示する。 |
| `+multiline` | レコード情報を人間が読みやすい複数行形式で表示する。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点
* **タスク**: メールサーバ（MXレコード）の伝播確認。
* **組み合わせ**: `dig <domain> MX +short`
* **解説**: 新しく設定した MX レコードが正しく反映されているか、優先順位を含めて確認します。
* **例**:
    ```bash
    # example.com のメールサーバ設定を確認
    dig example.com MX +short
    ```

### 2. ブルーチーム視点
* **タスク**: DNS 設定ミスによるゾーン情報の露出確認。
* **組み合わせ**: `dig @<ns_server> <domain> AXFR`
* **解説**: 権威サーバがゾーン転送を誰にでも許可していないかチェックします。これが成功すると、そのドメイン内の全てのホスト名が露呈します。
* **例**:
    ```bash
    # ゾーン転送の脆弱性テスト
    dig @ns1.example.com example.com AXFR
    ```

### 3. レッドチーム視点
* **タスク**: サブドメインの列挙と情報の収集。
* **組み合わせ**: `+trace` または TXT レコードの確認。
* **解説**: TXT レコードには SPF 設定や API キー、検証用文字列などが含まれていることが多く、攻撃のヒントになります。
* **例**:
    ```bash
    # SPF設定などを含む TXT レコードを詳細に調査
    dig example.com TXT +multiline
    ```

## 出力結果の読み方 (ANSWER SECTION)
```text
;; ANSWER SECTION:
example.com.    86400   IN      A       93.184.216.34
````

- **example.com.**: 問い合わせたドメイン
    
- **86400**: TTL (Time To Live: キャッシュ有効時間/秒)
    
- **IN**: クラス (通常は Internet を示す IN)
    
- **A**: レコードタイプ (IPv4アドレス)
    
- **93.184.216.34**: 回答データ
    

## エラーメッセージとトラブルシューティング

1. **「status: NXDOMAIN」**:
    
    - **意味**: Non-Existent Domain。指定したドメイン名は存在しない。
        
    - **対策**: タイポの確認、または登録が完了しているか確認。
        
2. **「status: SERVFAIL」**:
    
    - **意味**: Server Failure。ネームサーバが回答に失敗した。
        
    - **対策**: ネームサーバ側の設定エラーや、再帰クエリの制限を疑う。
        
3. **「status: REFUSED」**:
    
    - **意味**: 問い合わせが拒否された。
        
    - **対策**: アクセス制限（ACL）やゾーン転送の拒否設定を確認。
        

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

- **悪用シナリオ**: **DNS Tunneling**。攻撃者が `dig` や類似のツールを使い、TXT レコードのクエリを介して、ファイアウォールを通り抜けて外部の C2 サーバーと通信を行う。
    

### GTFOBins における利用例

`dig` はファイル操作を行わないため GTFOBins には登録されていませんが、ネットワーク制限のある環境で外部と「話せる」数少ない手段として重宝されます。

### 対応策・緩和策 (ブルーチーム視点)

- **Prevention (予防)**: 公開ネームサーバでのゾーン転送（AXFR）を厳格に禁止する。
    
- **Detection (検知)**: 異常な数の TXT クエリや、存在しないサブドメインへの大量のクエリ（NXDOMAINフラッド）を監視する。
    

## 注意点・補足

- **ドットの有無**: ドメイン名の最後にドット `.` を付けると「ルートからの絶対パス」として扱われ、OS の検索ドメインリストを無視してクエリを投げられます。
