# Command  - Linux - free - システムのメモリ使用量を表示する

## 概要

`free` コマンドは、Linuxシステムの物理メモリ（RAM）とスワップメモリの総量、使用量、空き容量を一覧表示する基本的なコマンドです。カーネルがディスクI/Oの性能向上のために使用しているバッファやキャッシュの量も表示します。システムのパフォーマンス問題を診断する際の第一歩として、メモリの空き状況を確認するために頻繁に利用されます。

(出自: `procps-ng` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`top` / `htop`** | **リアルタイム**に**プロセスごと**のリソース使用状況を表示します。「どのプロセスが」メモリを使っているかを特定するのに使います。`free` は**システム全体**の合計のスナップショットを表示します。 | `htop` (詳細分析) |
| **`vmstat`** | メモリに加え、プロセス、I/O、CPUアクティビティなど、より広範な仮想メモリ統計情報を表示できます。 | - |
| **`cat /proc/meminfo`** | `free` コマンドが情報を取得している大元のデータソース。`free` よりも遥かに詳細な情報が含まれています。 | - |

## よく連携されるコマンド

`free` は、メモリ状況の変化を監視するために `watch` コマンドとよく連携されます。

### シナリオ例: メモリ使用量の変化をリアルタイムで監視する (インフラ構築・運用視点)

* **目的**: 新しいアプリケーションをデプロイしたり、負荷テストを行ったりする際に、メモリ使用量の変化をリアルタイムで監視し、メモリリークや急激な消費がないかを確認する。
* **連携コマンド**: `watch`
* **解説**: `watch` コマンドと組み合わせることで、指定した秒数ごとに `free -h` を自動実行し、画面を更新し続けます。これにより、メモリ状況の推移を簡単に追跡できます。
* **コマンド例**:

```bash
# 5秒ごとにメモリ状況を更新して表示する
watch -n 5 free -h
```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **表示形式** | |
| ⭐ `-h`, `--human-readable` | サイズを人間が読みやすい単位 (K, M, G) で表示します。 |
| `-b`, `-k`, `-m`, `-g` | サイズをそれぞれバイト、キロバイト、メガバイト、ギガバイト単位で表示します。 |
| **動作制御** | |
| `-s <N>`, `--seconds <N>` | N秒ごとに情報を更新して表示します。 |
| `-c <N>`, `--count <N>` | `-s` と併用し、N回表示した後に終了します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: サーバーの現在のメモリ状態を最も手早く確認する。
* **組み合わせ**: `free -h`
* **解説**: 事実上の標準的な使い方。人間が読みやすい形式でメモリの全体像を表示し、「`available` は十分か？」を一目で判断できます。
* **例**:

```bash
# 読みやすい形式で現在のメモリ使用状況を表示
free -h
```

### 2. ブルーチーム視点

* **タスク**: インシデントの初動調査として、リソース枯渇の兆候がないか確認する。
* **組み合わせ**: `free -h`
* **解説**: システムの動作が遅いといった通報を受けた際に、まず `free -h` を実行します。`available` が極端に少ない、または `Swap` の `used` が急増している場合、リソースを枯渇させるタイプの攻撃（Fork爆弾など）や、メモリを大量に消費するマルウェアの活動が疑われます。
* **例**:

```bash
# メモリとスワップの使用状況を確認
free -h
```

### 3. レッドチーム視点

* **タスク**: 侵入後の初期偵察で、標的システムのスペックを把握する。
* **組み合わせ**: `free -h`
* **解説**: 侵入したサーバーのメモリ総量 (`total`) を確認します。メモリが潤沢であれば、それはデータベースサーバーや重要なアプリケーションサーバーである可能性が高いと推測できます。この情報は、そのサーバーの価値を判断し、次にどのようなペイロードを展開するかを決定するのに役立ちます。
* **例**:

```bash
# システムの搭載メモリ総量を確認
free -h
```

## エラーメッセージとトラブルシューティング

* `free` コマンド自体は単純なため、エラーになることはほとんどありません。トラブルシューティングは、主に出力結果の**正しい解釈**に関連します。
* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **問題**: **`free` の値が非常に小さい。メモリ不足ではないか？**
		* **考えられる原因**: Linuxカーネルがパフォーマンス向上のために、空きメモリを積極的にディスクキャッシュとして利用しているためです。これは正常な動作です。
		* **解決策**: `free` の値ではなく、**`available` の値を確認**してください。`available` が十分に（数百MB以上）残っていれば、通常はメモリ不足ではありません。

2. **問題**: **`Swap` の `used` が増えている。**
		* **考えられる原因**: 物理メモリ (`available`) が不足し、カーネルが使用頻度の低いメモリページをディスク上のスワップ領域に退避（スワップアウト）させています。
		* **解決策**: これはメモリが逼迫している明確な兆候です。`top` や `htop` を実行し、メモリを大量に消費しているプロセスを特定して、対応を検討する必要があります。

## 環境変数と設定ファイル

* `free` は、その動作に影響を与える設定ファイルや環境変数を**使用しません**。データは常に `/proc/meminfo` から直接取得されます。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `free` は読み取り専用の情報表示コマンドであり、脆弱性を持つ可能性は低いです。
* **悪用シナリオ**: `free` は攻撃者にとって重要な**偵察ツール**となります。サーバーのメモリ量からその役割や価値を推測し、攻撃の優先順位を決定します。

### GTFOBins / LOLBAS における利用例

`free` は情報表示のみを行うコマンドであり、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: -
* **Detection (検知)**: `free` コマンドの出力は、Zabbix, Prometheus などの監視システムで常に監視すべき基本メトリクスです。`available` メモリの割合にしきい値を設定し、それを下回った場合にアラートを発報するように設定することで、リソース枯渇型の攻撃やメモリリークを早期に検知できます。

## 注意点・補足

* **`buff/cache` の内訳**: より新しいバージョンの `procps-ng` パッケージでは、`buffers` と `cache` が別々の列で表示されることがあります。`buffers` はメタデータ（ファイル情報など）、`cache` はファイル内容そのものをキャッシュしています。いずれも、必要に応じて解放可能なメモリです。
