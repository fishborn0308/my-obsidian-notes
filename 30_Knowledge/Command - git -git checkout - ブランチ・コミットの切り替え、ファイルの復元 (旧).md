# Command - git -git checkout - ブランチ・コミットの切り替え、ファイルの復元 (旧)

## 概要

`git checkout` コマンドは、Gitのもっとも多機能なコマンドの1つで、主に**ブランチを切り替える**こと、および**作業ディレクトリ内のファイルを特定の状態に復元する**ために使用されます。

**⚠️重要**: Git 2.23以降、このコマンドの機能は `git switch` (ブランチ操作)と `git restore` (ファイル操作)に分割されました。`git checkout` は後方互換性のために残されていますが、**新しいコマンド (`switch`/`restore`) の使用が強く推奨されます**。

(出自: `git` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`git switch`** | **ブランチの切り替えに特化**したモダンなコマンド。`git checkout <branch>` は `git switch <branch>` とほぼ同じです。意図が明確で安全です。 | `git switch` |
| ⭐ **`git restore`** | **ファイルの復元に特化**したモダンなコマンド。`git checkout -- <file>` の機能を引き継いでおり、より安全です。 | `git restore` |
| **`git reset`** | コミット履歴自体を操作する、より強力なコマンドです。`checkout` は作業ディレクトリやHEADポインタを動かしますが、`reset` はブランチポインタ自体を動かします。 | - |

## よく連携されるコマンド

`git checkout` は、新しい機能ブランチを作成し、すぐにそのブランチへ移動する、というワークフローでよく使われます。

### シナリオ例: 新しい機能ブランチを作成し、そこに切り替える (インフラ構築・運用視点)

* **目的**: 新しい機能開発を開始するために、既存のブランチから新しいブランチを作成し、すぐにそのブランチへ移動して作業を始める。
* **連携コマンド**: `git status`, `git branch`
* **解説**: `-b` オプションを使って新しいブランチを作成し、同時にそのブランチへ切り替えます。これは `git branch <name>` と `git switch <name>` をまとめて実行するのと同じです。
* **コマンド例**:

		```bash
    # 'main' ブランチから新しい機能ブランチ 'feature/user-auth' を作成し、切り替える

    git checkout -b feature/user-auth

    ```


## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **ブランチ/コミット操作** | |
| ⭐ `<branch>` | 指定した既存のブランチに作業ディレクトリを切り替えます。 |
| ⭐ `-b <new_branch>` | 新しいブランチを作成し、そのブランチに切り替えます。 |
| ⭐ `-` | 直前に作業していたブランチに切り替えます。 |
| `<commit-id>` | 特定のコミットIDに切り替えます（**detached HEAD** 状態になります）。 |
| **ファイル操作** | |
| ⭐ `-- <file>` | 作業ディレクトリでのファイルの変更を破棄し、最後にコミットされた状態 (`HEAD`) に戻します。 |
| `<commit> -- <file>`| 指定したコミット時点のファイルを、作業ディレクトリに復元します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 誤って変更してしまったインフラの設定ファイル(Kubernetesマニフェストなど)を、直前のコミットされた状態に素早く戻す。
* **組み合わせ**: `git checkout -- <config-file>`
* **解説**: 作業ディレクトリでの変更を完全に破棄し、最後にコミットされた安全な状態に戻します。
* **例**:

		```bash
    # httpd.conf の変更を破棄し、最後にコミットされた状態に戻す

    git checkout -- httpd.conf

    # (Git 2.23以降では git restore httpd.conf が推奨されます)
    ```


### 2. ブルーチーム視点

* **タスク**: 侵害が疑われる期間の直前のクリーンな状態や、特定のマルウェアが導入されたとされるコミット時点のファイルを確認する。
* **組み合わせ**: `git checkout <suspected-commit-id> -- <log-file>`
* **解説**: 特定の過去のコミット時点でのファイルの健全性を確認し、必要であればその状態に復元して比較します。
* **例**:

		```bash
    # コミットID 'abcdefg' 時点の nginx.conf を作業ディレクトリに復元して確認

    git checkout abcdefg -- nginx.conf

    # (Git 2.23以降では git restore --source abcdefg nginx.conf が推奨されます)
    ```


### 3. レッドチーム視点

* **タスク**: 既存のエクスプロイトコードの新しいバリアントを開発する。
* **組み合わせ**: `git checkout -b <exploit-variant> <base-exploit-commit>`
* **解説**: 特定のコミット（ベースとなるエクスプロイト）から新しいブランチを作成し、元のコードベースを汚染せずに独自の変更を加えます。
* **例**:

		```bash
    # ベースとなるC2エージェントの安定版コミットから、新しい開発ブランチを作成

    git checkout -b c2-agent-obfuscation develop-c2

    ```


## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](../linux/troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `error: Your local changes to the following files would be overwritten by checkout: ...`
		* **考えられる原因**: 現在のブランチにコミットされていない変更がある状態で、別のブランチに切り替えようとしました。
		* **解決策**:
				* 変更をコミットする: `git commit -m "WIP"`
				* 変更を一時退避する: `git stash`
				* 変更を破棄する: `git restore .`

## 環境変数と設定ファイル

* `git checkout` に直接影響する独自の環境変数はありません。
* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](../linux/environment_and_config.md) を参照。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: 古いバージョンのGitでは、悪意のあるリポジトリをチェックアウトした際に、`post-checkout` などのクライアントサイドフックが自動的に実行されてしまう脆弱性が存在しました (CVE-2014-9390など)。
* **悪用シナリオ**: 攻撃者が侵害したGitリポジトリの権限を奪取した場合、`git checkout` を使って、既知の脆弱性が存在する古いバージョンや、悪意のあるコードが挿入されたブランチを意図的に本番環境にデプロイする可能性があります。

### GTFOBins / LOLBAS における利用例

`git checkout` は直接的なコード実行機能を持たないため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **常に最新のGitバージョンを使用する**。重要なブランチには**ブランチ保護ルール**を設定し、レビューなしでのマージを防ぐ。
* **Detection (検知)**: Gitホスティングサービスの監査ログを監視し、異常なブランチの切り替えや強制プッシュがないかを確認する。

## 注意点・補足

* **HEAD分離状態**: `git checkout <commit-id>` を実行すると、「detached HEAD」状態になります。この状態でのコミットはどのブランチにも属さず、一時的な調査目的で使うべきです。
