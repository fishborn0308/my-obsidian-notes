# Command - Linux - env - 環境変数の表示および設定した環境でのコマンド実行

## 概要
`env` は、現在のユーザーセッションに設定されている環境変数を一覧表示するほか、特定の環境変数を一時的に追加・変更した状態で別のプログラムを実行するために使用されます。また、実行ファイルの絶対パスを意識せずにスクリプトを起動するための「ポータビリティの向上」にも一役買っています。
(出自: `coreutils` パッケージに含まれるバイナリ。シェル組み込みではない)



## 類似・関連コマンド
| コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `printenv` | 環境変数の表示に特化。特定の変数名だけを指定して表示できる（`printenv PATH`）。 | 変数値のクイック確認 |
| `set` | シェル変数、環境変数、および定義された関数をすべて表示する（非常に量が多い）。 | シェル全般の状態確認 |
| `export` | 現在のシェルおよびその子プロセスに対して変数を「永続的（そのセッション中）」に設定する。 | セッション変数の定義 |
| `env` | **一時的**な環境変更とコマンド実行に特化。Shebangでの利用も多い。 | **使い捨ての環境構築** |

## よく連携されるコマンド
### シナリオ例: スクリプトのポータビリティ確保 (開発・構築視点)
* **目的**: Python や Ruby の実行パスが `/usr/bin/` か `/usr/local/bin/` かを気にせずスクリプトを動かす。
* **連携**: Shebang (`#!`)
* **解説**: `env` がパス（`$PATH`）の中から対象のバイナリを探し出して実行してくれます。
* **コマンド例**:
    ```python
    #!/usr/bin/env python3
    print("Hello, world!")
    ```

## オプション説明
| オプション | 説明 |
| :--- | :--- |
| ⭐️ `(引数なし)` | 現在設定されているすべての環境変数を表示する。 |
| ⭐️ `-i`, `--ignore-environment` | 現在の環境変数を**すべてクリア**した状態でコマンドを実行する（クリーンなテスト用）。 |
| ⭐️ `VAR=VALUE` | 指定した変数を設定した状態でコマンドを実行する。 |
| `-u`, `--unset=NAME` | 特定の環境変数を削除した状態で実行する。 |
| `--debug` | (GNU版) 各ステップの処理内容を詳細に表示する。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点
* **タスク**: 言語設定を一時的に英語に変更してエラーメッセージを確認。
* **組み合わせ**: `LANG=C`
* **解説**: 日本語メッセージだと検索しづらいエラーを、一時的に英語（標準）にして出力させます。
* **例**:
    ```bash
    # 英語環境でコマンドを実行
    env LANG=C ls --help
    ```

### 2. ブルーチーム視点
* **タスク**: 侵害調査における秘密情報の露出確認。
* **組み合わせ**: `env`
* **解説**: 攻撃者が環境変数に API キーやパスワード、秘密のトークンなどを紛れ込ませていないか、あるいは正規のアプリが環境変数に機密情報を保持したままになっていないかを確認します。
* **例**:
    ```bash
    # キーワードで環境変数内の秘密情報を検索
    env | grep -iE "pass|key|token|secret"
    ```

### 3. レッドチーム視点
* **タスク**: `LD_PRELOAD` を利用した共有ライブラリのインジェクションによる権限昇格。
* **組み合わせ**: `env LD_PRELOAD=/tmp/evil.so`
* **解説**: 実行時に特定のライブラリを優先的に読み込ませる環境変数を利用し、プログラムの挙動を乗っ取ります。
* **例**:
    ```bash
    # 悪意あるライブラリを読み込ませてコマンドを実行
    env LD_PRELOAD=/tmp/libhack.so /usr/bin/sudo -l
    ```

## エラーメッセージとトラブルシューティング

1.  **エラーメッセージ例 1**: `env: ‘python3’: No such file or directory`
    * **原因**: `$PATH` の中に `python3` が見つからない。
    * **解決策**: Python がインストールされているか確認し、パスが通っているか `echo $PATH` でチェックする。

2.  **「env で変えたのに、次のコマンドでは戻っている」**:
    * **原因**: `env` はその時実行するコマンドにしか影響を与えないため。
    * **解決策**: 現在のシェル全体で変えたい場合は `export` を使用する。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例
* **情報漏洩**: コンテナ環境（Docker等）では、DB のパスワードなどを環境変数で渡すことが多いため、`env` コマンドが叩ける状態は非常に危険です。
* **LD_LIBRARY_PATH の悪用**: 不適切なライブラリパスを設定させることで、意図しない（攻撃者の）バイナリを実行させる「バイナリ・プランティング」に利用されることがあります。

### GTFOBins における利用例
`env` に `sudo` 権限がある場合、環境変数を自由に操作してシェルを起動できるため、容易に特権を維持・昇格できます。
* **機能**: `Sudo`, `Interactive Shell`
* **コマンド例**:
    ```bash
    # sudo env を使って root シェルを取得
    sudo env /bin/sh
    ```
* **参照**: `https://gtfobins.github.io/gtfobins/env/`

### 対応策・緩和策 (ブルーチーム視点)
* **Prevention (予防)**: 環境変数にパスワードを直接記述する運用を避け、Vault などの秘匿情報管理ツールやファイル経由の読み込みに切り替える。
* **Detection (検知)**: `/proc/<PID>/environ`（他プロセスの環境変数が見える場所）に対する不審なアクセスを監視する。

## 注意点・補足
* **変数の優先順位**: `env VAR=val command` とした場合、コマンド内では既に同名の変数があっても `val` が優先されます。
* **引数のクォート**: 値にスペースが含まれる場合は、`env "MESSAGE=hello world" command` のようにクォートが必要です。
