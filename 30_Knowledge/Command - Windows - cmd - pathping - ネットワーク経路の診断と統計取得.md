# Command - Windows - cmd - pathping - ネットワーク経路の診断と統計取得

## 概要

`pathping` は、宛先ホストに至るまでのネットワーク経路をトレースし、その経路上にある各ルーター（ホップ）でのパケットロスや遅延の統計を算出するコマンドです。

`tracert` のように経路を表示した後、一定時間（デフォルトで250秒間）パケットを送信し続けて各ポイントの品質を測定します。

(出自: Windows NT 4.0 Resource Kit 以降標準搭載)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- | :--- |
| `ping` | 特定の宛先との疎通確認のみ。経路上のどこでロスしているかは不明。 | 簡易確認なら `ping` |
| `tracert` | 経路（ホップ）を表示するが、統計（ロス率）の算出機能はない。 | 経路のみ即知りたいなら `tracert` |
| `mtr` | Linuxで一般的なツール。リアルタイムに統計を更新し続ける。 | Linux環境なら `mtr` |

## よく連携されるコマンド

### シナリオ例: ネットワーク障害の証拠資料作成 (運用/保守)

* **目的**: 通信が不安定な原因が「自社内」か「ISP側」かを切り分け、報告書を作成する。
* **連携コマンド**: `findstr`, リダイレクト (`>`)
* **解説**: `pathping` の結果をファイルに保存し、ロスが発生しているノード（アドレス）を抽出します。
* **コマンド例**:
    ```cmd
    REM 統計情報をファイルに保存
    pathping -n 8.8.8.8 > pathping_report.txt
    ```

## スイッチ/オプション説明

`pathping` は統計を取るために時間がかかるため、オプションで試行回数などを調整することが多いです。

| スイッチ | 説明 |
| :--- | :--- |
| **基本設定** | |
| ⭐ `/n` | **重要。** ホスト名の解決を行わず、IP アドレスのみを表示する。実行スピードが大幅に向上する。 |
| `/h <MaximumHops>` | 宛先を検索する最大ホップ数を指定する（デフォルトは 30）。 |
| ⭐ `/q <NumQueries>` | 各ホップに対して送信するクエリ（エコー要求）の数を指定する（デフォルトは 100）。 |
| **タイミング制御** | |
| `/p <Period>` | 連続する ping 間の待機時間をミリ秒単位で指定する（デフォルトは 250ms）。 |
| `/w <Timeout>` | 各応答のタイムアウト時間をミリ秒単位で指定する（デフォルトは 3000ms）。 |

## よく使われる組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 特定のルーターによる断続的なパケットロスの特定
* **解説**: 100回のクエリを投げ、各ノードでのロス率（%）を算出します。自分の管理外（ISPなど）でロスが発生していることを証明するのに有効です。
* **例**:
    ```cmd
    REM IPアドレスのみ表示し、高速に診断を開始
    pathping /n [www.google.com](https://www.google.com)
    ```

### 2. ブルーチーム視点

* **タスク**: VPN 経由の通信品質低下の調査
* **解説**: リモートユーザーからの「Web 会議がカクつく」といった報告に対し、VPN ゲートウェイまでの経路上のどこにボトルネックがあるかを分析します。
* **例**:
    ```cmd
    REM クエリ数を30に減らし、短時間で概況を把握する
    pathping /n /q 30 10.1.0.1
    ```

### 3. レッドチーム視点 (偵察)

* **タスク**: 内部ネットワークのセグメンテーションとフィルタリングの推測
* **解説**: `pathping` の結果、特定のホップで 100% のロスが発生している場合、そこがファイアウォール（ICMP 遮断）である可能性を推測できます。
* **例**:
    ```cmd
    REM 最大ホップ数を制限し、内部セグメントの入り口を特定する
    pathping /n /h 10 172.16.20.5
    ```

## エラーメッセージとトラブルシューティング

### よくあるトラブル

1.  **現象**: 結果が出るまで非常に時間がかかる。
    * **原因**: `pathping` はデフォルトで 1 ホップにつき 100 回のクエリを投げ、その統計を取るため、総時間が「ホップ数 × 100回 × 250ms」となります。
    * **解決策**: `/q` オプションでクエリ数を減らすか、`/n` で名前解決を無効化する。

2.  **現象**: 統計部分に「*」や「100% ロス」が並ぶ。
    * **原因**: 経路上のルーターが ICMP (ping) への応答を制限（Rate Limiting）または拒否している。
    * **解決策**: そのノード単体の仕様である可能性が高いため、その前後のホップのデータと比較して判断する。

## 環境変数と設定ファイル

`pathping` は設定ファイルを持ちませんが、実行結果はプロトコルスタックのタイムアウト設定（レジストリ）に依存する場合があります。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **悪用シナリオ**:
    - **Information Disclosure**: 内部ネットワークの構成（IP スキーム、ゲートウェイの配置）を外部から推測される。特に、パケットロス率の変化から IDS/IPS の存在を推測される可能性があります。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**:
    - 境界ルーター/ファイアウォールで、外部からの ICMP Time Exceeded や ICMP Echo Request に対する応答を制限する。
* **Detection (検知)**:
    - 短時間に大量の ICMP パケットを送信するホストを監視（特に経路を舐めるような挙動）。

## 注意点・補足

* **実行完了を待つ**: `pathping` は 2 段階のフェーズがあります。第 1 フェーズは `tracert` のような経路表示、第 2 フェーズが統計算出です。第 2 フェーズが終わる前に `Ctrl+C` で止めると、統計情報が得られません。
* **ジッターの把握**: 統計結果の「RTT（ラウンドトリップタイム）」のばらつきを見ることで、回線の安定性（ジッター）を評価できます。
