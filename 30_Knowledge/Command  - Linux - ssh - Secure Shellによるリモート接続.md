# Command  - Linux - ssh - Secure Shellによるリモート接続

## 概要

`ssh` (Secure Shell) は暗号化された安全な通信経路上で、リモートのコンピュータにログインしたり、コマンドを実行したりするための**プロトコル**であり、またその**クライアントコマンド**です。通信は全て暗号化されるため、パスワードや実行したコマンドがネットワーク上で盗聴されるのを防ぎます。

`ssh` は単なるリモートシェルに留まらず、ファイル転送 (`scp`, `sftp`) や、ポートフォワーディング（トンネリング）といった高度な機能も提供し、現代のサーバー管理において不可欠なツールとなっています。

(出自: `openssh-client` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`telnet`, `rlogin`**| **旧世代のプロトコル**。全ての通信が**平文で送信されるため、極めて危険**です。`ssh` はこれらの安全な代替です。 | `ssh` |
| **`mosh`** | `ssh` をベースにした、不安定なネットワーク（モバイル通信など）に強いリモートシェル。接続が一時的に切れてもセッションが維持されるローミング機能などが特徴です。 | `mosh` (不安定な回線) |

## よく連携されるコマンド

`ssh` は、他のコマンドをリモートで実行させたり、ポートフォワーディング（トンネリング）で安全な通信路を確保したりと、様々なコマンドと連携します。

### シナリオ例: ローカル・ポートフォワーディング（SSHトンネリング） (インフラ構築・運用視点)

* **目的**: ファイアウォールの内側にあるデータベースサーバーに、ローカルマシンのGUIツールから安全に接続する。
* **連携コマンド**: `ssh` (`-L` オプション)
* **解説**: 踏み台サーバー (`bastion`) を経由して、ローカルPCの特定ポートへの接続を、DBサーバーのポートに転送します。これにより、DBポートをインターネットに公開することなく、安全にアクセスできます。
* **コマンド例**:

		```bash
		# このコマンドを実行すると、ローカルの5432番ポートへの接続が、
		# bastionサーバー経由でdb.internalの5432番ポートに転送される

		ssh -L 5432:db.internal:5432 user@bastion.example.com

		```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **接続・認証** | |
| ⭐ `-i <key_file>` | 認証に使用する**秘密鍵**のパスを指定します。 |
| ⭐ `-p <port>` | 接続先のSSHポート番号を指定します。デフォルトは22です。 |
| `-l <user>` | 接続するユーザー名を指定します (`<user>@<host>` と同義)。 |
| `-o <option>` | 設定ファイル (`~/.ssh/config`) のオプションをコマンドラインで指定します。 |
| **ポートフォワーディング** | |
| ⭐ `-L <local>:<host>:<remote>`| **ローカル・ポートフォワーディング**。ローカルポートへの接続をリモート側に転送します。 |
| `-R <remote>:<host>:<local>`| **リモート・ポートフォワーディング**。リモートホストのポートへの接続をローカル側に転送します。 |
| `-D <port>` | **ダイナミック・ポートフォワーディング**。ローカルマシン上でSOCKSプロキシとして動作させます。 |
| **デバッグとその他** | |
| ⭐ `-v`, `-vv`, `-vvv`| **詳細表示 (Verbose)**。接続過程のデバッグ情報を表示します。`-v` を増やすほど詳細になります。 |
| `-q` | 静かなモード。警告や診断メッセージを抑制します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 公開鍵認証を使い、リモートサーバーに安全にログインする。
* **組み合わせ**: `ssh -i <key_file> <user>@<host>`
* **解説**: パスワード認証よりもセキュアな公開鍵認証でサーバーに接続する、最も基本的な使い方です。
* **例**:

		```bash

		ssh -i ~/.ssh/id_ed25519 admin@192.168.1.100

		```

### 2. ブルーチーム視点

* **タスク**: `sshd` のログを分析し、不正アクセスの試みを調査する。
* **組み合わせ**: `journalctl -u sshd` または `grep "sshd" /var/log/auth.log`
* **解説**: `sshd` のログは、不正アクセスの調査における最重要の証拠となります。`Failed password for ...` のようなログからブルートフォース攻撃を、`Accepted publickey for ...` のログから不正ログインを追跡します。
* **例**:

		```bash
		# sshdのログをリアルタイムで監視

		sudo journalctl -u sshd -f

		```

## 3. レッドチーム視点

* **タスク**: **横展開 (Lateral Movement)**。侵入したサーバーを踏み台にして、他の内部サーバーへ侵入を拡大する。
* **組み合わせ**: `ssh <internal_user>@<internal_host>`
* **解説**: 攻撃者は侵入したサーバーで入手した秘密鍵や認証情報を使い、`ssh` で他の内部サーバーへ次々と侵入を試みます。また、`-R` (リモート) を使って、ファイアウォールの内側から外部のC2サーバーへリバース接続を確立します。
* **例**:

		```bash
		# 侵害ホストから、盗んだ秘密鍵を使って内部DBサーバーに接続

		ssh -i /tmp/stolen_key.pem dbuser@10.10.20.5

		```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](OS%20%20-%20Linux%20-%20troubleshooting_common_errors%20-%20Linux共通エラー対応ガイド.md) を参照。

1. **エラーメッセージ例 1**: `Permission denied (publickey,password).`
		* **考えられる原因**: SSHの認証エラー。ユーザー名、パスワード、秘密鍵のいずれかが間違っているか、サーバー側で許可されていません。
		* **解決策**: 認証情報を再確認してください。`ssh -v <user>@<host>` で接続を試みると、どの認証方式が試行され、なぜ失敗したかの詳細なログが表示されます。

2. **エラーメッセージ例 2**: `WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!`
		* **考えられる原因**: 接続先のサーバーのホストキーが、以前接続した際に `~/.ssh/known_hosts` に保存されたものと異なっています。これは、サーバーの再インストールか、**中間者攻撃 (Man-in-the-Middle) を受けている**可能性があります。
		* **解決策**: サーバーが正当に再インストールされたことが確実な場合は、`ssh-keygen -R <hostname>` コマンドで古いホストキーを削除してから再接続します。

## 環境変数と設定ファイル

* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

### 関連する設定ファイル

* ⭐ **`~/.ssh/config`**: クライアント側の設定ファイル。ホストごとのエイリアス、ユーザー名、ポート、秘密鍵などを定義し、接続を大幅に簡略化できます。
* **`/etc/ssh/ssh_config`**: システム全体のクライアント設定ファイル。
* **`/etc/ssh/sshd_config`**: **サーバー側**のSSHデーモン設定ファイル。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `ssh` 自体よりも、**弱いパスワードの使用**、**秘密鍵の漏洩**、**古いバージョンの利用**がリスクとなります。
* **悪用シナリオ**: 攻撃者は、漏洩した認証情報やブルートフォース攻撃で特定したパスワード/秘密鍵を使い、`ssh` で最初の足がかりを築きます。その後、`ssh` を使って横展開や永続化を行います。

### GTFOBins / LOLBAS における利用例

* **機能**: **Privilege Escalation**, **File Upload/Download**
* **解説**: `sudo` 権限で `ssh` を実行できる場合、`-o ProxyCommand` を悪用して任意のコマンドを実行したり、`ssh -o "PermitRootLogin yes"` のように設定を上書きしてrootログインを試みたりすることが可能です。
* **参照**: `https://gtfobins.github.io/gtfobins/ssh/`

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: **パスワード認証を無効化 (`PasswordAuthentication no`)** し、強力なパスフレーズで保護されたSSH公開鍵認証のみを許可する。**rootログインを禁止 (`PermitRootLogin no`)** する。
* **Detection (検知)**: `sshd` のログをSIEMなどに集約し、不審なIPアドレスからのログインや、短時間での大量のログイン失敗を監視する。

## 注意点・補足

* **`~/.ssh/config` の活用**: このファイルを活用することで、`ssh prod-db` のような短いエイリアスで、踏み台経由の複雑な接続を簡単に行えるようになります。
