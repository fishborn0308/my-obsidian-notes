# Command - Linux - cd - カレントワーキングディレクトリを変更する

## 概要

`cd` (Change Directory) は、ユーザーがファイルシステム上を移動するために使用する、最も基本的かつ頻繁に使用されるシェル組み込みコマンドです。コマンドを実行する際の基準点となるカレントワーキングディレクトリ（現在地）を指定したディレクトリに変更します。`cd` は外部プログラムではなく、シェル自身の機能として実装されています。なぜなら、子プロセス（外部プログラム）は親プロセス（シェル）の環境（カレントディレクトリなど）を変更できないためです。

(出自: `bash`, `zsh` などのシェルの組み込み機能)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| **`pushd`, `popd`** | ディレクトリスタックという履歴機能を持ち、複数のディレクトリを一時的に行き来する複雑な作業で便利です。`cd` はより手軽な移動に使います。 | - |

## よく連携されるコマンド

`cd` 自体はパイプで何かを受け渡すことはありませんが、他のコマンドと連続して実行することで操作の基本フローを形成します。

### シナリオ例: コマンド実行結果を使って移動する (インフラ構築・運用視点)

* **目的**: `find` コマンドで見つけた特定のディレクトリに直接移動する。
* **連携コマンド**: `find`, コマンド置換 (`$()`)
* **解説**: コマンド置換 `$()` を使うと、あるコマンドの実行結果（標準出力）を別のコマンドの引数として渡せます。これにより、深い階層にある目的のディレクトリを検索し、一発で移動するといった高度なナビゲーションが可能になります。
* **コマンド例**:

```bash
# カレントディレクトリ以下で 'nginx' という名前のディレクトリを探して、最初に見つかったものに移動する

cd $(find . -type d -name "nginx" -print -quit)

```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **特別な引数** | |
| ⭐ `(引数なし)` | ホームディレクトリ (`$HOME`) に移動します。 |
| ⭐ `~` | ホームディレクトリに移動します (`cd ~user` で `user` のホームディレクトリへ移動)。 |
| ⭐ `..` | 親ディレクトリ（1つ上の階層）に移動します。 |
| ⭐ `-` | 直前にいたディレクトリ (`$OLDPWD`) に移動します。 |
| **動作制御** | |
| `-L` | (デフォルト) 論理的なパスをたどります。シンボリックリンクをそのままディレクトリ名として扱います。 |
| `-P` | 物理的なパスをたどります。全てのシンボリックリンクを解決して、実際のディレクトリ構造を追跡します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 設定ファイルやログファイルが格納されているディレクトリへ直接移動する。
* **組み合わせ**: `cd <absolute_path>`
* **解説**: 絶対パスを指定することで、現在地にかかわらず目的のディレクトリへ確実に移動できます。サービスのトラブルシューティングの起点となります。
* **例**:

```bash
# systemdのユニットファイルを確認・編集するために移動

cd /etc/systemd/system/

```

### 2. ブルーチーム視点

* **タスク**: シンボリックリンクの実体を追跡し、ファイルが実際に格納されている場所を特定する。
* **組み合わせ**: `cd -P <symlink_directory>`
* **解説**: インシデント調査中に、攻撃者が作成した可能性のあるシンボリックリンクに惑わされず、物理的なパスを正確に特定するために使用します。
* **例**:

```bash
# /var/wwwが/opt/webappへのシンボリックリンクだった場合、/opt/webappに移動する

cd -P /var/www

pwd

# -> /opt/webapp
```

### 3. レッドチーム視点

* **タスク**: 攻撃ツールやペイロードを一時的に配置するための、書き込み可能なディレクトリへ移動する。
* **組み合わせ**: `cd /dev/shm` または `cd /tmp`
* **解説**: 全てのユーザーが書き込み可能で、監視が手薄になりがちな一時的なディレクトリは、攻撃のステージングエリアとして頻繁に利用されます。
* **例**:

```bash
# ツールをダウンロードして実行権限を付与するための作業場所へ移動

cd /dev/shm

```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **エラーメッセージ例 1**: `bash: cd: <filename>: Not a directory`
		* **考えられる原因**: `cd` の引数として、ディレクトリではなくファイルを指定しています。
		* **解決策**: `ls -F` を実行すると、ディレクトリには末尾に `/` が付与されるため、ファイルと見分けやすくなります。

2. **エラーメッセージ例 2**: `bash: cd: <directory>: Permission denied`
		* **考えられる原因**: そのディレクトリに対する**実行権限 (`x`)** がありません。ディレクトリに入るには、読み取り権限 (`r`) だけでなく実行権限が必要です。
		* **解決策**: `ls -ld <directory>` でパーミッションを確認します。権限がない場合は、`sudo` を使うか、管理者に権限の変更を依頼します。

## 環境変数と設定ファイル

* 一般的な環境変数・設定ファイルは [Linux共通の環境変数・設定ファイル](./environment_and_config.md) を参照。

### 影響を与える環境変数

* **`PWD`**: 現在のワーキングディレクトリのパスが格納されています。`cd` を実行するとこの変数が更新されます。
* **`OLDPWD`**: `cd` を実行する直前のワーキングディレクトリのパスが格納されています。`cd -` はこの変数の値を参照します。
* **`CDPATH`**: `cd` がディレクトリを探す際の検索パスをコロン区切りで指定します。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `cd` はシェル組み込みコマンドであり、それ自体に脆弱性はありません。
* **悪用シナリオ**: `cd` は攻撃活動の基本動作である「**探索 (Discovery)**」で常に使われます。攻撃者はシステム侵入後、`cd` と `ls` を駆使してファイルシステムを渡り歩き、設定ファイル、パスワード、秘密鍵などの機密情報を探します。

### GTFOBins / LOLBAS における利用例

`cd` はファイル操作、コード実行、権限昇格などの機能を持たない純粋なナビゲーションコマンドのため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: -
* **Detection (検知)**: `auditd` やEDRを利用して、ユーザーのファイルシステムアクセスを監視します。特に、通常業務ではアクセスしないはずの機密性の高いディレクトリへの `cd` がサービスアカウントなどから行われた場合は、アラートを発報するよう設定することが有効です。

## 注意点・補足

* **スクリプト内での `cd`**: シェルスクリプト内で `cd` を実行しても、その影響はそのスクリプトのプロセス内にとどまります。スクリプトが終了すれば、呼び出し元のシェルのカレントディレクトリは元のままです。
* **Tab補完の重要性**: `cd` を使う際は、**必ずTabキーによる補完機能を使ってください**。タイプミスを劇的に減らし、作業効率を飛躍的に向上させます。
