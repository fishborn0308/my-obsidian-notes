# Command  - Linux - mtr - ネットワーク経路の診断ツール

## 概要

`mtr` (My Traceroute) は、`traceroute` の経路追跡機能と `ping` の疎通確認・遅延測定機能を組み合わせた、強力なネットワーク診断ツールです。宛先ホストまでのネットワーク経路（どのルーターを経由するか）を一覧表示し、さらに各経由点（ホップ）に対して継続的にパケットを送信して、**パケット損失率 (Loss%)** と**応答時間 (Latency)** をリアルタイムで表示します。「インターネットが遅い」「特定のサイトに繋がらない」といった問題が発生した際に、**ネットワーク経路のどこで問題が発生しているのか**を特定するために使用されます。

(出自: `mtr` パッケージに含まれる)

## 類似コマンドと差異

| 類似コマンド | 差異 / 使い分けのポイント | 推奨 |
| :--- | :--- |:---|
| ⭐ **`traceroute`** | 宛先までのネットワーク経路を表示しますが、各ホップへの遅延測定は通常3回程度で完了します。`mtr` は**継続的に**パケットを送信し続けるため、時間と共に変動するネットワークの状態を捉えるのに優れています。 | `mtr` (インタラクティブ診断) |
| **`ping`** | **最終的な宛先**との疎通と応答時間のみを測定します。途中の経路で何が起きているかは分かりません。 | - |

## よく連携されるコマンド

`mtr` は単体で非常に高機能なため、他のコマンドと直接パイプで連携することは少ないですが、その出力をテキストファイルとして保存し、レポートとして利用することがよくあります。

### シナリオ例: ネットワーク問題のレポートを作成する (インフラ構築・運用視点)

* **目的**: ネットワークの問題をISPやデータセンター事業者に報告する際に、客観的な証拠として `mtr` のレポートを添付する。
* **連携コマンド**: リダイレクション (`>`)
* **解説**: `-r` (レポートモード) と `-c` (カウント) オプションを使い、一定回数のパケットを送信した結果をサマリーとして出力し、ファイルに保存します。`-n` で名前解決を無効にすると、より迅速に結果が得られます。
* **コマンド例**:

```bash
# google.com 宛に100回のパケットを送信し、結果をレポートファイルに保存
mtr -r -c 100 -n google.com > mtr-report-google-$(date +%F).txt
```

## オプション説明

| オプション | 説明 |
| :--- | :--- |
| **動作モード** | |
| ⭐ `-r`, `--report` | **レポートモード**。対話的に実行せず、指定回数のパケットを送信した後にサマリーレポートを表示して終了します。 |
| `-c <count>` | (`--report` と併用) 送信するパケットの総数を指定します。 |
| **表示制御** | |
| ⭐ `-n`, `--no-dns` | IPアドレスの逆引きを行わず、IPアドレスのまま表示します。実行が高速になります。 |
| `-b`, `--show-ips` | ホスト名とIPアドレスの両方を表示します。 |
| **プロトコル指定** | |
| ⭐ `-T`, `--tcp` | (root権限要) ICMPエコーの代わりに**TCP SYNパケット**を送信します。ICMPをブロックするファイアウォール越えの調査に有効です。 |
| `-u`, `--udp` | (root権限要) ICMPエコーの代わりに**UDPパケット**を送信します。 |
| `-P <port>` | `-T` や `-u` と併用し、宛先ポート番号を指定します。 |

## よく使われるオプション組み合わせと業務シナリオ

### 1. インフラ構築・運用視点

* **タスク**: 特定の宛先へのネットワーク経路と品質をリアルタイムで診断する。
* **組み合わせ**: `mtr <destination_host>`
* **解説**: 「Webサイトの表示が遅い」という報告を受け、`mtr` を実行。自社ネットワーク内のルーターで `Loss%` が急上昇していれば社内ネットワークの問題、ISPのルーターで `Avg` (遅延) が急増していればISP側の問題、といったように問題箇所を切り分けます。
* **例**:

```bash
# example.com へのネットワーク経路を診断
mtr example.com
```

### 2. ブルーチーム視点

* **タスク**: 不審なIPアドレスへの通信経路を調査する。
* **組み合わせ**: `sudo mtr -T -P 443 -b <suspicious_ip>`
* **解説**: ファイアウォールログで不審なIPアドレスとの通信を検知したが、そのIPがどこにあるのかを調査したい。ICMPがブロックされていることを想定し、Webでよく使われるTCP/443ポートで経路を追跡することで、通信経路の全体像を把握しようと試みます。
* **例**:

```bash
# 不審なIP 198.51.100.10 へのTCP/443での経路を追跡
sudo mtr -T -P 443 -b 198.51.100.10
```

### 3. レッドチーム視点

* **タスク**: ネットワークの内部偵察 (Reconnaissance)。
* **組み合わせ**: `mtr -n <internal_target_ip>`
* **解説**: 侵入したネットワークセグメントから、別の内部セグメントにあるサーバーへの経路を `mtr` で調査します。出力される経路上ルーターのIPアドレスや命名規則から、組織の内部ネットワークのトポロジー（構造）をマッピングし、ファイアウォールやACLの存在を推測します。
* **例**:

```bash
# 内部のDBサーバー (10.10.20.5) への経路を偵察
mtr -n -r -c 10 10.10.20.5
```

## エラーメッセージとトラブルシューティング

* 一般的なエラーは [Linux共通のトラブルシューティング](./troubleshooting_common_errors.md) を参照。

1. **現象**: **途中のホップが `???` と表示され、Loss% が100%になる。**
		* **考えられる原因**: そのホップのルーターが、セキュリティポリシーなどの理由で `traceroute` が使用するICMP Time Exceededメッセージを返さないように設定されています。
		* **解決策**: **これは必ずしもネットワーク障害を意味しません**。その次のホップ以降で正常に応答が返ってきていれば、通信経路自体は正常である可能性が高いです。

## セキュリティに関する考慮事項

### 脆弱性と悪用事例

* **脆弱性**: `mtr` 自体ではなく、その応答が**情報漏洩**に繋がります。
* **悪用シナリオ**: `mtr` の応答には、組織の内部ネットワークで使用されているルーターのIPアドレスやホスト名が含まれます。これにより、攻撃者は内部ネットワークの構造を詳細に把握できます。

### GTFOBins / LOLBAS における利用例

`mtr` はネットワーク診断ツールであり、直接的なコード実行や権限昇格の機能を持たないため、GTFOBinsにはリストされていません。

### 対応策・緩和策 (ブルーチーム視点)

* **Prevention (予防)**: 境界ルーターで、外部からのICMP Time Exceededメッセージを要求するパケット（TTLが低いパケット）をブロックまたはレート制限することで、内部ネットワークの偵察を困難にする。
* **Detection (検知)**: 内部から外部、または内部セグメント間で予期せぬ `mtr` や `traceroute` の通信が多発していないかをネットワーク監視ツールで監視する。

## 注意点・補足

* **パケットロスの解釈**: 経路の途中の1ホップだけでロスが発生し、それ以降のホップではロスがない場合、それはルーターがICMPの処理を低優先度にしているだけであり、実際の通信品質には問題がないことが多いです。
